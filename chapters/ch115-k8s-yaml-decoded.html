Kubernetes YAML, decoded

Deployment, Service, ConfigMap, Secret — the four manifests behind every agent you run. By the end of this chapter, a wall of Kubernetes YAML reads like a memo you could have written yourself.

~30 minChapter 116 of 117
The story

You run the help desk of a busy airport terminal. You could hire people yourself — interview, train, replace them when they quit at 2 a.m. Instead, you file one standing contract with a staffing agency: "Keep exactly three trained assistants at the desk at all times. Uniform: the blue kit, version 1.4.2. Each assistant gets one desk, one headset, and a fixed snack budget — no more. Every few minutes, check each one is awake; before sending them a customer, check they are actually ready to serve. And give the public a single phone number that rings whichever assistant is free."

Then you go home. Someone resigns overnight? The agency hires a replacement before breakfast — you never get the call. You didn't tell them how to do any of this; you described the world you want and let them keep it true.

That contract is a Kubernetes manifest. The three assistants are replicas, the uniform is your container image, the two checks are probes, the phone number is a Service. The notice board by the desk is a ConfigMap — and the sealed envelope in the drawer is a Secret that is rather less sealed than it looks.

You already read this language

Cast your mind back to chapter 5: YAML is not a programming language, it is dicts and lists in costume. A line like replicas: 3 is a dict entry. Indentation means nesting — a dict inside a dict. A leading dash means a list item. That is the entire grammar; there is nothing else to learn.

Here is the payoff, ten acts later: every Kubernetes manifest is one dict. A 40-line Deployment that makes newcomers' eyes glaze over is a single nested dictionary — the structure you have built in Python since chapter 4. kubectl, last chapter, was you asking the cluster questions; a manifest is how you tell it things. And chapter 111 gave you the idea underneath: declare a desired state, and controllers loop forever making reality match it. The manifest is that desired state, written down.

Better still, every manifest opens with the same four keys — learn these and you can orient yourself in any Kubernetes file ever written:

The Deployment, line by line

A Deployment is the staffing contract itself: "keep N copies of this pod running, looking exactly like this template". It is the manifest you will read most often, so here is a complete, production-shaped one — the kind that runs an agent's API — with no line left unexplained. Click every annotated line until each one is boring. Boring means fluent.

deployment.yaml — a complete apps/v1 Deployment
apiVersion: apps/v1
Workload types — Deployments and their cousins — live in the apps group of the API, stable version 1. This exact string has not changed in years.
kind: Deployment
The form type: "this document declares a Deployment". Together, apiVersion + kind tell the cluster which schema to check the rest of the file against.
metadata:
name: support-agent
The Deployment's own name — what you type in kubectl rollout status deployment/support-agent.
spec:
The contract begins — everything above was the envelope, everything below is desired state.
replicas: 3
"Keep exactly three assistants at the desk." Not "start three" — keep three: crash one and a controller notices 2 ≠ 3 and starts a replacement. Change it to 10 and reapply: that is scaling.
selector:
How this Deployment recognises its pods. It does not track children by name — it claims anything wearing the right label.
matchLabels:
app: support-agent
"My pods are the ones tagged app: support-agent." This line and the label line eleven rows down are a pair — they MUST match. A whole section below covers what happens when they don't.
template:
The pod template — a full pod description nested inside the Deployment, the rubber stamp that prints every replica. Everything you learned about pods in chapter 113 reappears from here down.
metadata:
labels:
app: support-agent
The name tag stamped onto every pod this template prints — the other half of the handshake. The selector above searches for exactly this label; one typo here and the contract is void.
spec:
A second spec! This one is the pod's desired state, nested inside the Deployment's — dicts in dicts, business as usual.
containers:
A YAML list (see the dash below) — a pod can hold several containers, like the MCP sidecar pattern from chapter 113. Ours holds one.
- name: agent
image: myregistry.azurecr.io/support-agent:1.4.2
The uniform: which container image to run, pulled from your Azure Container Registry, pinned to tag 1.4.2. Pin versions — :latest is a different surprise every pull.
ports:
- containerPort: 8000
Documents that the app inside listens on port 8000. Mostly informational — but remember the number: the Service and both probes point at it.
envFrom:
"Take every key in a ConfigMap and inject each as an environment variable." Config flows as it did in chapter 19 — the app just reads its environment, unaware Kubernetes exists.
- configMapRef:
name: agent-config
A reference, by name, to a ConfigMap we will write shortly. The Deployment doesn't contain the config — it points at it, so config changes without rebuilding the image.
env:
Alongside the bulk import, env sets individual variables — literals, or single keys plucked from a ConfigMap or Secret with valueFrom.
- name: AGENT_ENV
value: "production"
resources:
The snack budget. Two numbers per resource — the difference between them is a favourite interview question.
requests:
cpu: "250m"
Requests are a booking. "This pod needs at least a quarter of a CPU core (250 milli-cores) and half a gigabyte." The scheduler uses requests to pick a node with room. Book too much and pods sit Pending with nowhere to go.
memory: "512Mi"
limits:
cpu: "1"
Limits are a ceiling. Exceed the CPU limit and the container is throttled — slowed, not killed. Exceed the memory limit and it is killed on the spot: the OOMKilled state from chapter 114 is this line being enforced.
memory: "1Gi"
livenessProbe:
The awake-check. Kubernetes calls this endpoint on a schedule; if it keeps failing, the container is judged wedged and restarted. Liveness asks one question: "is this process beyond saving?"
httpGet:
path: /healthz
Your app must actually serve this route — a trivial 200 OK handler. The probe is only as honest as the endpoint behind it.
port: 8000
initialDelaySeconds: 10
Startup grace period — don't declare the app dead while it is still lacing its shoes. Agents that warm caches need this.
periodSeconds: 15
readinessProbe:
The ready-to-serve check — NOT a duplicate of liveness. A failing readiness probe restarts nothing; it removes the pod from the Service's routing list until it passes again. Liveness = "restart me"; readiness = "no customers for a moment". A pod waiting on a slow dependency is not ready, but it is very much alive.
httpGet:
path: /ready
port: 8000
periodSeconds: 5
Tip

You never have to memorise field names. kubectl explain deployment.spec.replicas prints the official documentation for any field, straight from the cluster — every path, every kind. It is the dictionary you keep on the desk while reading manifests.

The gotcha: the selector must match the labels

Two places in that file carried the same little dict, app: support-agent: under spec.selector.matchLabels ("these are my pods") and under spec.template.metadata.labels ("stamp this on every pod I print"). This is the classic Kubernetes paper cut: they must match, and nothing auto-corrects them. If the template stamped a label the selector isn't looking for, the Deployment would print pods it could never recognise as its own.

Kubernetes refuses to let that contradiction exist. Apply a manifest where the two disagree and the API server rejects it outright — no pods, no partial state, just an error:

what a mismatch looks like
$ kubectl apply -f deployment.yaml
The Deployment "support-agent" is invalid: spec.template.metadata.labels:
Invalid value: map[string]string{"app":"supprot-agent"}:
`selector` does not match template `labels`

Read that error once now, calmly, so the day it appears at speed you smile instead of panicking. (One more sharp edge: the selector is immutable — on a live Deployment you can only delete and recreate it. Choose a boring label like app: <name> and never touch it.)

And because a manifest is just a dict, you can check the handshake with Python you already own — chapter 5 fluency doing real infrastructure work. Run it, fix the typo, run it again:

What happens when you apply it

Before the other three manifests, watch the whole machine move once. You hand the contract in; three actors do their jobs; the label handshake is the thread connecting them all.

1. You hand in the contract

Your laptopkubectl apply -f deployment.yaml
→
API servervalidates, stores desired state
Controllers

kubectl sends the manifest to the API server — the front desk from chapter 111 — which validates it (where a selector mismatch bounces) and records the desired state. Nothing runs yet; a wish has been filed.

2. The Deployment controller notices the gap

Your laptop
API server
→
Deployment controllerdesired: 3 running, actual: 0

A controller — a loop that never sleeps — compares desired state with reality: three wanted, zero exist. It creates a ReplicaSet, the middle manager whose only job is keeping the pod count right.

3. Three pods, stamped with the label

ReplicaSetneeds 3
→
pod-1app: support-agent
pod-2app: support-agent
pod-3app: support-agent

The ReplicaSet prints three pods from the template, each wearing app: support-agent. The scheduler places each on a node that can afford its requests — too-large requests leave pods stuck Pending.

4. The Service finds them by label

Callershttp://support-agent
→
Serviceselector: app=support-agent
→

pod-2ready
pod-3readiness failing

The Service (next section) runs the same label query and routes traffic to matching pods — but only those whose readiness probe passes. pod-3 is alive but not ready, so it gets no traffic until it recovers.

5. A pod dies at 2 a.m. You sleep through it.

ReplicaSetactual: 2, desired: 3 — fix it
→
pod-1
pod-2
pod-4fresh replacement

pod-3's node fails. The ReplicaSet counts 2 ≠ 3 and prints a replacement; the Service picks it up the moment it turns ready. Nobody paged you. That is what "declarative" buys.

The Service: one phone number for a changing team

Pods are cattle: they die, get replaced, and every replacement gets a fresh IP address — so nothing should ever hard-code one. A Service is the stable front: one fixed internal address and DNS name, load-balancing across whichever pods currently match its label selector. The single phone number that rings whichever assistant is free. Its manifest is refreshingly small:

service.yaml — a v1 Service
apiVersion: v1
Services are older than the apps group — they live in the core API, so the version is just v1. Same for ConfigMaps and Secrets.
kind: Service
metadata:
name: support-agent
This name becomes a DNS entry inside the cluster: any pod in the namespace can call http://support-agent — no IPs, ever. It is how an agent's pod reaches a shared MCP server's Service in chapter 113.
spec:
selector:
The same label-query trick, third appearance — with one small difference: a Service writes its selector directly, no matchLabels wrapper.
app: support-agent
"Route my traffic to every ready pod tagged app: support-agent." The Service does not know or care that a Deployment made those pods — it only sees labels.
ports:
- port: 80
port is the number the Service answers on — what callers dial. Callers use http://support-agent:80 (and port 80 being HTTP's default, they can drop it entirely).
targetPort: 8000
targetPort is where the container actually listens — the containerPort: 8000 from the Deployment. The Service forwards 80 → 8000. Mixing these two up is the other classic paper cut of this chapter.

One thing is conspicuously absent: a type field. Leave it out and you get the default, ClusterIP — a virtual IP reachable only inside the cluster. The right default: your agent's API, MCP servers, and A2A endpoint talk pod-to-pod without touching the internet. Exposing anything outside is a deliberate act (other Service types, or an ingress), never an accident of omission.

ConfigMap: the notice board

Your agent needs settings — log level, max turns, which model to call — and baking them into the image means rebuilding it for every tweak. A ConfigMap is a bucket of plain key/value settings living in the cluster, separate from the image, injected at startup:

configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: agent-config
data:
  LOG_LEVEL: "info"
  MAX_TURNS: "25"
  MODEL_NAME: "claude-sonnet-4-5"

No spec this time — a ConfigMap declares no behaviour, so its content sits under data. Every value is a string (hence "25" in quotes — the chapter 5 rule that unquoted YAML values get guessed at). The name agent-config is what the Deployment's envFrom: configMapRef pointed at: apply this, restart the pods, and your agent finds LOG_LEVEL=info in its environment. The uppercase keys are deliberate — envFrom turns each key into an environment variable, so keys must be legal variable names. ConfigMaps can also be mounted as files, which is how longer artefacts — a system prompt, a SKILL.md — ride along without an image rebuild.

Secret: a sealed envelope, not a safe

A Secret is a ConfigMap's discreet sibling — same shape, same ways of being consumed (env vars via secretKeyRef, or mounted files) — but meant for sensitive values:

secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: agent-keys
type: Opaque
data:
  API_KEY: c2stbGl2ZS1ub3QtYWN0dWFsbHktc2FmZQ==

That scrambled-looking value gives people a warm feeling of security. It should not. It is base64 — an encoding: a way of writing any bytes in 64 safe characters, with no key, no password, no secrecy. Encryption is a locked box; encoding is a costume change. Prove it right now:

Careful

Anyone who can run kubectl get secret agent-keys -o yaml reads every key in seconds — and a Secret committed to git is a leaked credential, full stop. Secrets buy separation and access control, not confidentiality from anyone holding cluster access.

So where do real credentials live? Exactly where chapter 113 put them: outside the cluster, in Azure Key Vault, reached via workload identity — the pod proves who it is with a federated identity token and fetches secrets at runtime, so no API key is ever baked into an image or parked in a manifest. Secret objects still appear in that world (the Key Vault CSI driver can materialise vault values as short-lived Secrets), but the source of truth sits in the vault, behind audit logs and rotation. Rule of thumb: ConfigMap for anything you could say out loud; Key Vault plus workload identity for anything you couldn't.

Shipping it: kubectl apply and kubectl diff

Four files sit in a folder — deployment.yaml, service.yaml, configmap.yaml, secret.yaml. Shipping them takes two commands, one of which you half-know from git:

terminal — preview, then apply
kubectl diff -f deployment.yaml

kubectl apply -f deployment.yaml

kubectl apply -f k8s/

kubectl rollout status deployment/support-agent

Line by line, because each of these earns its place:

Line 1 — kubectl diff -f deployment.yaml. The dry run. It compares your file against what is live in the cluster and prints a git-style diff of exactly what would change; if live state already matches, it prints nothing. The same reflex as reading git diff before committing — on a production cluster, the reflex that keeps you employed.

Line 2 — kubectl apply -f deployment.yaml. The single verb that does everything. There is no separate "create" versus "update": apply means make the cluster match this file. First time, it creates the Deployment; afterwards, it updates only what differs. Run it twice and the second run changes nothing — idempotent, safe to repeat, which is why automation can run it blindly.

Line 3 — kubectl apply -f k8s/. Point -f at a folder and it applies every manifest inside. The whole agent stack ships in one command — precisely the line you will find in CI/CD pipelines.

Line 4 — kubectl rollout status deployment/support-agent. Applying is filing paperwork; this watches the work happen. It blocks until the new pods are up and ready — your readiness probe deciding when a rollout counts as done — and reports failure if they never get there.

Note

Notice what you never wrote: no "stop the old pods", no "start new ones in order", no "switch traffic". You changed a file that describes the destination and let the controllers drive. That is the declarative habit — and why manifests live in git, get code-reviewed like code, and can rebuild a whole environment from scratch.

In the wild

Open any team's repo and look for a folder named k8s/, manifests/, or deploy/ — the four kinds you just read cover most of what lives there. In merge requests, the debates you can now follow are label debates ("this selector won't match after the rename"), resource debates ("512Mi is why it's OOMKilled"), and probe debates ("liveness is restarting pods that were just slow — that belongs in readiness"). When a pipeline log shows kubectl apply -f then rollout status, you are watching this chapter run unattended. One caveat: many shops template their YAML with Helm or Kustomize — the files grow placeholders and overlays, but what lands in the cluster is always the plain manifests you can now read.

Check yourself

1. A teammate edits a Deployment so the selector says app: chat-agent but the pod template labels still say app: support-agent. What happens at kubectl apply?

A Deployment that prints pods its own selector cannot claim is a contradiction, so validation rejects it outright. Nothing auto-corrects; you fix the label pair.

2. A Service declares port: 80 and targetPort: 8000. Which is true?

port is the Service's number (what callers dial); targetPort is where the container listens. The Service adapts between the two.

3. The values under a Secret's data: field are base64-encoded. What does that protect them from?

Base64 is an alphabet change, not encryption — b64decode() reverses it instantly. Real credentials belong in Key Vault, fetched via workload identity.

4. Your agent pod keeps showing OOMKilled. Which manifest line is being enforced?

Requests are a scheduling booking; limits are the enforced ceiling. CPU over its limit is throttled, but memory over its limit gets the container killed — that is what OOMKilled means.

5. A pod's readiness probe starts failing while its liveness probe keeps passing. What does Kubernetes do?

Readiness controls traffic, liveness controls restarts. A pod that is alive but not ready is quietly benched by the Service, then put back the moment it reports ready.

Key takeaways

  • Every Kubernetes manifest is one nested dict — chapter 5 fluency, applied. Four keys open every file: apiVersion, kind, metadata, spec.
  • A Deployment declares desired state — "keep N pods matching this template" — and controllers keep it true, replacing dead pods without you.
  • The label handshake is load-bearing: selector.matchLabels must match template.metadata.labels, or the API server rejects the manifest.
  • Requests book resources for scheduling; limits enforce a ceiling — cross the memory limit and the container is OOMKilled. Liveness restarts a wedged container; readiness only benches it from traffic.
  • A Service is a stable name and internal (ClusterIP) address over changing pods: callers dial port, containers answer on targetPort.
  • ConfigMaps carry sayable settings; Secrets are only base64-encoded — encoding, not encryption — so real credentials come from Key Vault via workload identity.
  • kubectl diff -f previews, kubectl apply -f converges the cluster to your file, idempotently — the two-step in every pipeline.
