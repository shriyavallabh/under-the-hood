Messages and chat models

Four message classes and one factory function — the same raw dicts you hand-built in Act IV, now wearing name badges. Learn this vocabulary and half of every LangChain file becomes instantly readable.

~25 minChapter 53 of 117
The story

Your export business in Pune ships with three courier companies. Each one has its own airway bill: the first wants the sender's address in box 3, the second wants it on page two, the third insists on block capitals and its own reference codes. Same parcel, same addresses — three different forms, three sets of clerical errors waiting to happen. Last month a shipment sat in a warehouse for a week because the declared value was written in the wrong box.

So you hire a freight forwarder. She hands you one standard form — FROM, TO, CONTENTS, DECLARED VALUE — and says: "Fill this in the same way every time. I will translate it into whichever courier's paperwork today's shipment needs." Suddenly, switching couriers is no longer a re-training exercise. It is one word on the booking sheet: carrier = DHL.

LangChain's message-and-model layer is that forwarder. The message classes are the standard form. init_chat_model is the line where you name the carrier. And .invoke() is the moment you hand the parcel over the counter.

You already know the hard part

Back in your first API call, you did all of this by hand: you built a list of plain dicts — each one shaped like {"role": "user", "content": "..."} — posted the whole list to a provider's API, and got a reply back as raw JSON. In tool calling you saw the model answer with a structured request instead of prose, and in the ReAct loop you fed tool results back into that growing list and watched an agent come alive. Nothing in this chapter is a new idea. This chapter is a renaming ceremony.

Why rename something that already works? Because the raw dicts have a dirty secret: every provider wants them shaped slightly differently. OpenAI's chat API takes the system instructions as a message inside the list. Anthropic's API wants them hoisted out into a separate top-level system parameter. Tool results ride in a "tool" role message on one API and inside a content block on another. Model names, parameter names, error shapes — all slightly different dialects of the same conversation.

If you write your application directly against one provider's dict format, you have married that provider. Every experiment with a different model becomes a rewrite. LangChain's answer is the freight forwarder's standard form: you write one neutral format — typed message objects — and a per-provider adapter translates them into whichever dialect the chosen API speaks. That neutral format is the vocabulary of this chapter.

Four envelopes, clearly labelled

A message is one turn of a conversation, wrapped in a small Python object that says who is speaking. LangChain defines a handful of message classes, and four of them carry almost every conversation you will ever read in its code. They import from langchain_core.messages, and each one maps exactly onto a raw dict you have already written by hand:

LangChain objectThe raw dict from Act IVWho is speaking
SystemMessage("Be concise."){"role": "system", "content": "Be concise."}You — standing instructions
HumanMessage("What is 20% GST?"){"role": "user", "content": "What is 20% GST?"}The user
AIMessage("It is 300 rupees."){"role": "assistant", "content": "It is 300 rupees."}The model
ToolMessage("300.0", tool_call_id="call_1"){"role": "tool", "content": "300.0", ...}Your tool code, reporting a result

Read that table twice — it is the entire mapping. HumanMessage is not a new concept; it is {"role": "user"} with a class name. The win is that a typed object can carry extras that a bare dict handles clumsily: an AIMessage knows about its tool calls and token usage, and a ToolMessage insists you supply the tool_call_id that pairs a result with the exact request it answers — the same pairing discipline you met in the tool-calling chapter, now enforced by the constructor instead of by your own care.

init_chat_model: name the carrier, get a counter clerk

Messages are the form; now you need someone to hand it to. A chat model is LangChain's object wrapping one provider's API — it knows the wire format, the authentication, the retries, and how to translate your neutral messages both ways. You could import a provider class directly (ChatAnthropic, ChatOpenAI…), but modern LangChain code overwhelmingly uses the factory function init_chat_model instead:

choose_a_model.py
from langchain.chat_models import init_chat_model
from langchain_core.messages import HumanMessage, SystemMessage

model = init_chat_model("anthropic:claude-sonnet-4-5", temperature=0)

messages = [
    SystemMessage("You are a concise accountant."),
    HumanMessage("What is 20% GST on a 1,500 rupee invoice?"),
]
reply = model.invoke(messages)
print(reply.content)

The string "anthropic:claude-sonnet-4-5" is a provider string — the provider's name, a colon, then the model id. You can also pass the pieces separately as init_chat_model("claude-sonnet-4-5", model_provider="anthropic"), and for famous name prefixes (gpt-…, claude-…, gemini-…) the factory can infer the provider from the model id alone. Extra keyword arguments like temperature pass straight through to the model — the same sampling dial you tuned in the sampling chapter.

And here is the promise from the story, kept. Moving this program to a different provider is a one-line diff:

the entire migration
-model = init_chat_model("anthropic:claude-sonnet-4-5", temperature=0)
+model = init_chat_model("openai:gpt-5.4-mini", temperature=0)

The messages do not change. The .invoke() call does not change. The reply is still an AIMessage with the same attributes. Everything downstream of that one line is provider-neutral — which is exactly what makes "let's benchmark three models on our eval set" an afternoon's work instead of a quarter's. Two small practicalities: each provider needs its adapter package installed (pip install langchain-anthropic, langchain-openai, and so on — the factory tells you plainly if one is missing), and each needs its API key in the environment, the same hygiene you learned in the config chapter.

Tip

Because the swap is one string, teams often lift it into config: init_chat_model(os.environ["APP_MODEL"]). When you see a deployment change a model by editing an environment variable rather than code, this is the mechanism underneath.

What .invoke() actually does

Your code reads .content and .tool_calls without knowing or caring which provider answered. This symmetry — neutral in, neutral out — is the whole trick.

Two conveniences to recognise in other people's code: .invoke("plain string") works — the string is wrapped in a HumanMessage for you — and so does a list of raw {"role": ..., "content": ...} dicts, which LangChain converts to message objects on the way in. Handy for quick scripts; typed objects are clearer in real codebases.

The sibling method is .stream(messages), which you will reach for any time a human is watching the answer appear. Instead of waiting for the full reply, it yields a series of AIMessageChunk objects as tokens arrive — the same incremental delivery you saw in the streaming chapter, dressed in message clothing:

streaming the same conversation
for chunk in model.stream(messages):
    print(chunk.content, end="", flush=True)

Anatomy of the reply: .content and .tool_calls

The AIMessage you get back deserves a closer look, because two of its attributes run the entire agent world. .content is the text — what you print, what the user reads. .tool_calls is the structured version of the ACTION line you parsed by hand in the ReAct chapter: a list of dicts, each with a name, an args dict, and an id. When the model wants to talk, .content is full and .tool_calls is an empty list. When it wants to act, .tool_calls carries the requests — and your code, never the model, does the acting.

Notice one quiet gift: args arrives as an already-parsed Python dict. Raw provider APIs typically hand you the arguments as a JSON string you must decode yourself — LangChain has done the string surgery before you ever see it. And when your tool finishes, you package the result as a ToolMessage whose tool_call_id echoes the request's id, append it to the conversation, and call the model again. That is the ReAct loop, verbatim, in the new vocabulary.

Read this short exchange the way a reviewer would — click the annotated lines:

one_agent_turn.py — the ch045 loop in LangChain vocabulary
from langchain.chat_models import init_chat_model
The factory lives in the main langchain package. This is the modern import — if you see ChatOpenAI imported directly instead, it is the same machinery chosen by hand.
from langchain_core.messages import HumanMessage, ToolMessage
Message classes live in langchain_core — the small foundation package from the ecosystem map chapter that every other LangChain package builds on.
model = init_chat_model("anthropic:claude-sonnet-4-5")
Provider string: provider, colon, model id. Changing providers means changing this string and nothing else in the file.
history = [HumanMessage("What is 20% GST on 1500 rupees?")]
The same history list as your hand-built agent — except each entry is a typed object instead of a labelled string. Still the model's entire memory, still re-sent every turn.
reply = model.invoke(history)
One round trip: translate out, HTTPS call, translate back. In your ReAct loop this line was fake_llm(history) — the interface is identical: history in, one message out.
call = reply.tool_calls[0]
Suppose the model asked to use a calculator (the next chapter shows how it knows one exists). No string parsing, no split("ACTION:") — the request arrives pre-parsed: {"name": "calculate", "args": {"expression": "1500 * 0.20"}, "id": "call_1", ...}.
result = my_tools[call["name"]](**call["args"])
Your code executes the tool — the security boundary from ch045, unchanged. The model asked; your dict of functions decides what is even possible.
history.append(reply)
The AIMessage itself goes into history, tool request and all, so the model can see what it asked for last turn.
history.append(ToolMessage(str(result), tool_call_id=call["id"]))
The OBSERVATION step, formalised. The tool_call_id pairs this result with the exact request — essential when a model fires several tool calls in one turn.
final = model.invoke(history)
Round two: the model sees its own request and the result, and now answers in prose. final.content holds the answer; final.tool_calls is empty — which is precisely how agent loops know they are done.

One more attribute earns its keep in production: reply.usage_metadata reports input and output token counts for the call — the raw numbers behind every cost dashboard you have ever squinted at, straight from the tokens-and-cost chapter.

In the wild

Open any modern LangChain or LangGraph repository and this vocabulary is everywhere. Grep for init_chat_model and you will usually find it once, near the top, often fed from an environment variable — that is the whole model configuration. Graph states carry a messages key holding exactly these objects, and debugging sessions are full of for m in result["messages"]: m.pretty_print(). In a merge request titled "switch to the cheaper model for triage", expect a one-line provider-string diff — now you know why that is safe. And when a stack trace complains that a ToolMessage is missing a tool_call_id, you can hear what it is really saying: a tool result came back without the receipt number that pairs it to its request.

Check yourself

1. What does HumanMessage("hello") actually become when it reaches a provider's API?

Message classes are typed wrappers around the role/content dicts from ch033. The chat model translates them into the chosen provider's exact dialect just before the HTTP call.

2. Your app runs on one provider's model and you want to trial a competitor's. With init_chat_model, what has to change in your code?

The one-line swap is the point of the neutral layer — though operationally you still need the new provider's adapter package installed and its API key in the environment. Option c overshoots: you must still choose the model; LangChain never chooses for you.

3. An AIMessage comes back with an empty .content and one entry in .tool_calls. What is the model telling you?

Tool calls are requests, never actions — the model cannot execute anything. Your code runs the tool, wraps the result in a ToolMessage, and invokes the model again, exactly like the ReAct loop from ch045.

4. Why does ToolMessage require a tool_call_id?

A model can request three tools at once; three results come back. The id — echoed from the tool call's own id field — is the receipt number that keeps the pairings straight.

5. When would you reach for .stream() instead of .invoke()?

.invoke() waits and returns one full AIMessage; .stream() yields AIMessageChunk objects as tokens arrive — the difference between a letter and a live phone call. Both send the same conversation.

Key takeaways

  • LangChain's message classes are the raw role/content dicts of Act IV with name badges: SystemMessage is role "system", HumanMessage is "user", AIMessage is "assistant", ToolMessage is "tool" plus a pairing id.
  • init_chat_model("provider:model") is a factory that returns a provider adapter; swapping providers is a one-line change because translation in both directions is the adapter's job, not yours.
  • .invoke(messages) is one full round trip returning a single AIMessage; .stream(messages) yields AIMessageChunk fragments as the reply is generated.
  • On every AIMessage, .content carries the text and .tool_calls carries pre-parsed action requests — empty tool_calls is how loops know the model is done.
  • The model still executes nothing: tool calls are requests, your code acts, and the result rides back as a ToolMessage — the ReAct loop of ch045 in formal dress.
  • This vocabulary is half the battle for reading LangChain code; the next chapter adds the other half — how the model learns which tools exist at all.
