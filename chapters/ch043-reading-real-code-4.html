The story

There is a tiny diner near the railway station with exactly one counter, one cook, and a metal rail above the pass. When you order, the waiter writes your order on a paper ticket and clips it to the rail — next to every ticket from earlier in your meal. The cook has a strange rule: before making anything, he re-reads your entire rail, first ticket to last. It looks wasteful, but it is why he never forgets that you asked for less salt twenty minutes ago — he has no memory at all, only the rail. And he never plates a dish in one dramatic reveal; he slides it across the pass piece by piece, so you start eating while he is still cooking.

You are about to read the code for that diner. The waiter is a while loop, the ticket rail is a Python list called history, the forgetful cook is an LLM API, and the piece-by-piece pass is streaming. Sixty lines. Let's read the whole kitchen.

The whole program, top to bottom

Here is the complete file. Skim it once now — do not stop at anything confusing. Notice the shape: two small helper functions at the top, then a main() that runs a loop. That is the whole story: set up, then loop forever — read, send, print, remember.

chat.py — a complete chat app in about 60 lines
# chat.py — talk to a model from your terminal
import os
import sys
import anthropic

MODEL = "claude-opus-4-8"
SYSTEM = "You are a concise, friendly assistant."


def make_client():
    """Fail fast, with a clear message, if there is no API key."""
    if not os.environ.get("ANTHROPIC_API_KEY"):
        sys.exit("Set the ANTHROPIC_API_KEY environment variable first.")
    return anthropic.Anthropic()


def stream_reply(client, history):
    """Send the whole history, print the reply as it arrives,
    and return the finished text."""
    parts = []
    with client.messages.stream(
        model=MODEL,
        max_tokens=1024,
        system=SYSTEM,
        messages=history,
    ) as stream:
        for text in stream.text_stream:
            print(text, end="", flush=True)
            parts.append(text)
    print()
    return "".join(parts)


def main():
    client = make_client()
    history = []
    print("Tiny chat. Type 'quit' to leave.")
    while True:
        try:
            user = input("you> ").strip()
        except (EOFError, KeyboardInterrupt):
            print()
            break
        if not user:
            continue
        if user.lower() in ("quit", "exit"):
            print("bye")
            break
        history.append({"role": "user", "content": user})
        print("assistant> ", end="", flush=True)
        try:
            reply = stream_reply(client, history)
        except anthropic.APIError as err:
            print(f"\n[error talking to the API: {err}]")
            history.pop()  # forget the turn that never got an answer
            continue
        history.append({"role": "assistant", "content": reply})


if __name__ == "__main__":
    main()
Note

The model id and the SDK call shapes here are current as of mid-2026 (client.messages.stream(...) is the official streaming helper in the anthropic Python SDK). Model names rotate every few months — which is exactly why this file keeps the id in a single MODEL constant at the top. Upgrading the whole app to a newer model is a one-line edit.

The API call sends the entire history, not just the newest message. The model is stateless — it remembers nothing between calls — so its only knowledge of the conversation is whatever you send. This one fact explains half of all chat-app behaviour.

Instead of waiting for the full answer, the loop prints each chunk the moment it lands — that is the familiar "typing" effect of every chat product. Each chunk is also saved into a list called parts.

The chunks are glued back into one string and appended as an assistant message. Next turn, the model will see its own previous words — which is the only reason it can answer "what did you just say?". Then the loop swings back to the prompt.

MODEL = "claude-opus-4-8"
Configuration lives at the top as a named constant, not buried inside a function call. When the next model generation ships, this file needs a one-line change. Small file, professional habit.
SYSTEM = "You are a concise, friendly assistant."
The standing instructions — the system prompt from chapter 37. It rides along on every request but never appears in history; the SDK sends it as a separate field.

A tiny function with one job (chapter 7): build the client, or die with a useful message. Failing loudly at startup beats a confusing crash mid-conversation.
if not os.environ.get("ANTHROPIC_API_KEY"):
Credentials come from the environment, never from the source code — the config discipline from chapter 19. A key pasted into a file eventually gets committed and leaked.

The client object — your authenticated phone line to the API. Built once, reused for every turn. It finds the key in the environment by itself.
def stream_reply(client, history):
The second helper: make one API call, print the reply as it streams, hand back the finished text. Notice what it does NOT do — it never touches input() or modifies history. Talking to the network and managing the conversation are kept in separate functions.
parts = []
Streaming gives you the reply in fragments, but the history needs the complete text. So every chunk is printed AND collected here, to be reassembled at the end.
with client.messages.stream(
Two old friends in one line: the streaming call from chapter 40, wrapped in a with block — a context manager (chapter 18) that guarantees the network connection is closed properly even if something goes wrong mid-stream.
max_tokens=1024,
The hard ceiling on the reply's length, in tokens (chapter 41). Chat replies are short, so a modest cap doubles as a cost guardrail — a runaway answer cannot bill you forever.
messages=history,
The most important argument in the file. Not the latest message — the WHOLE list, every turn. The server keeps no memory of you between calls; this list IS the conversation.
for text in stream.text_stream:
You iterate over the stream like a list, but values are produced as they arrive over the network — the lazy-iteration idea from generators. Each text is a few characters of reply.
print(text, end="", flush=True)
end="" stops print adding a newline after every fragment; flush=True forces the text onto the screen immediately instead of letting Python buffer it. Delete flush=True and the "typing" effect turns into awkward bursts.
return "".join(parts)
The fragments are glued into one complete string (chapter 3) — this is what gets stored in history as the assistant's turn. Streaming is a delivery mechanism; the transcript still needs whole messages.
history = []
The ticket rail — an ordinary Python list (chapter 4), and the app's entire memory. Close the program and it is gone; real products persist this list in a database, but the shape is identical.
while True:
The heart of every chat app: an infinite loop (chapter 6) that only ends when a break fires. Read a message, answer it, repeat until the user leaves.
except (EOFError, KeyboardInterrupt):
Graceful exits for Ctrl-D and Ctrl-C — the two ways terminal users abandon a program. Without this, quitting with Ctrl-C would splatter a traceback (chapter 9) across the screen.
if user.lower() in ("quit", "exit"):
The quit command. .lower() makes it case-insensitive, and membership in a tuple checks both spellings at once. Every REPL-style tool you have used has a line like this.
history.append({"role": "user", "content": user})
Your words become a message dict — the exact wire shape from chapter 33 — and go onto the rail BEFORE the API call, because the call needs to include them.
except anthropic.APIError as err:
Networks fail, rate limits bite, servers overload. This catches any API failure, prints a short notice instead of a scary traceback, and lets the loop continue — the app survives. Production code would catch specific subclasses (rate limit vs auth) and retry differently for each.
history.pop() # forget the turn that never got an answer
A subtle, professional touch. The user message was appended before the call failed. Leaving it there means the NEXT successful reply would also answer this ghost message — which the user believes failed. Popping it keeps the app's memory consistent with what the user saw on screen.
history.append({"role": "assistant", "content": reply})
The reply joins the rail with the assistant role. This closes the loop of memory: next turn, the model reads its own earlier words. Forget this line and the model would contradict itself constantly — a bug you may now recognise from flaky chat tools.
if __name__ == "__main__":
The import guard from chapter 11: run main() when the file is executed directly, but stay quiet if another file imports it — say, a test that wants to call stream_reply on its own.

Check yourself

1. Why does stream_reply send the entire history list on every single turn?

The API keeps no memory of you between calls. If the earlier messages were not re-sent, the model would meet every message as a total stranger. The history list IS the conversation — which is also why long chats get expensive.

2. When the API call fails, the error handler runs history.pop(). What is that protecting against?

The user message goes into history before the call. If the call fails and the message stays, the transcript no longer matches what the user saw on screen — and the next successful reply would eerily respond to the "lost" question. Popping keeps memory and reality in sync.

3. The streaming loop both prints each chunk and appends it to parts. Why collect the chunks at all?

Streaming is only a delivery mechanism — the transcript still stores whole messages. The chunks are shown live for the user AND glued back together so the assistant's turn can join the history in one piece.

4. You want to upgrade this app with RAG so it can answer questions about your company's documents. Where does the retrieval step go?

RAG (chapter 42) fetches relevant context and puts it in front of the model, so it must happen before the API call that needs it: read the question, retrieve, stuff the findings into the message, then send. The rest of the file does not change.

5. What breaks if you remove flush=True from the streaming print?

Python holds printed text in a buffer and releases it in batches for efficiency. flush=True forces each chunk onto the screen the instant it arrives. Without it the code still works; it just looks like the model is thinking in spasms.

Key takeaways

  • A complete chat app is about sixty lines: a client, a history list, an infinite loop, one streaming API call, a quit command, and an error handler. Everything else in every chat product is decoration on this chassis.
  • The history list is the entire memory of the conversation, and it is re-sent whole on every turn because the model is stateless — that is why costs grow as chats get longer.
  • Streaming is a delivery mechanism, not a different kind of answer: chunks are printed live for the human and reassembled with "".join(parts) for the transcript.
  • Good error handling keeps memory consistent with reality — history.pop() after a failed call removes the question that never got answered.
  • Every other Act IV idea — sampling, tools, structured output, RAG, async — bolts onto a specific, identifiable line of this file.
  • Reading unfamiliar code is a two-pass skill: skim for the story first, then go line by line until every line is boring.
