Tools and the @tool decorator

One line above a plain Python function, and the model can suddenly find it, describe it, and ask for it by name. This is where the decorators of Act II stop being a party trick and start paying rent.

~25 minChapter 54 of 117
The story

Your neighbourhood runs a tool library — a real one, with shelves. Priya donates her tile cutter. The librarian does not just shove it on a shelf. She reads the plate riveted to its side — 230 volts, 600 watts, blade 110 mm — flips through the little manual in the box, and types up a catalogue card: what the machine is called, what it is for, and exactly which details you must fill in to borrow it. Members never wander the shelves. They browse the catalogue, pick a card, and hand in a borrow slip. If a slip arrives with the blade size missing, it bounces back at the front desk before anyone touches the machine.

One librarian, one minute per donation, and every drill, cutter, and sander in the district becomes findable, describable, and safely requestable. In this chapter, the tile cutter is your Python function. The plate on its side is its type hints. The manual in the box is its docstring. And the librarian stamping catalogue cards is a decorator called @tool.

A decorator, earning its keep

Back in the decorators chapter you learned that a decorator is just a function that takes your function and hands back an upgraded version, and that the @ line above a def is shorthand for greet = timer(greet). You watched @lru_cache bolt a memory onto a function without touching its body, and you learned the three-beat rhythm: wrap, enhance, rebind the name. Nothing about the original function changes — what changes is what the name now points to. Hold that thought, because @tool may be the most consequential decorator in this whole course: it is the moment a plain Python function gets a passport into the model's world.

Here is what the librarian needs from you. A function that wants to become a tool must bring two things: type hints on its parameters (the plate on the machine — you met these in the type hints chapter) and a docstring (the manual in the box). Given those, @tool can write the catalogue card entirely on its own:

tools.py
from langchain_core.tools import tool

LEDGER = {"INV-042": 1500.0}

@tool
def get_invoice_total(invoice_id: str) -> float:
    """Return the grand total, in rupees, of one invoice."""
    return LEDGER[invoice_id]

print(get_invoice_total.name)
# get_invoice_total
print(get_invoice_total.description)
# Return the grand total, in rupees, of one invoice.
print(get_invoice_total.args)
# {'invoice_id': {'title': 'Invoice Id', 'type': 'string'}}

Look closely at what happened to the name. After the decorator runs, get_invoice_total no longer refers to your function. It refers to a tool object — an instance of a LangChain class called StructuredTool, which is a function wrapped together with its own catalogue card. It carries a .name (taken from the function name), a .description (taken from the docstring), and an .args view showing the parameters it extracted from your type hints. Your original code is still in there, tucked away as .func, but you now run it the LangChain way, with a dictionary of arguments:

running a tool yourself
get_invoice_total.invoke({"invoice_id": "INV-042"})
# 1500.0

Why a dictionary instead of a normal call? Because that is exactly the shape a model's request arrives in — and because the tool object validates that dictionary against its schema before your function ever runs. Pass an invoice_id of the wrong type, or forget it entirely, and the borrow slip bounces at the front desk. Your function body never sees malformed input.

Careful

Leave the docstring off and @tool refuses the function on the spot, with an error the moment the module loads. This feels strict until you realise what the docstring has become: it is the only thing the model will ever read about your tool when deciding whether to use it. A tool with no description is a machine in the catalogue with a blank card — worse than useless, because it still takes up space in the context window.

What the model actually sees

This is the part worth slowing down for, because it explains half the tool-related bugs you will ever debug. The model never sees your Python. Not the body, not the variable names, not the clever caching. When your tool rides along on an API request, it has been flattened into a schema — a JSON description of the tool's name, its purpose, and the exact shape of its inputs. For the invoice tool above, the schema looks like this:

what travels to the model (JSON, roughly)
{
  "name": "get_invoice_total",
  "description": "Return the grand total, in rupees, of one invoice.",
  "input_schema": {
    "type": "object",
    "properties": {
      "invoice_id": {"type": "string"}
    },
    "required": ["invoice_id"]
  }
}

Three fields. The name is how the model will ask for the tool. The description is how the model decides whether to ask for it — this is your docstring, verbatim. The parameters block tells the model what a valid borrow slip looks like: which fields, which types, which are required. Different providers spell the outer wrapper slightly differently (Anthropic says input_schema, OpenAI says parameters), and LangChain translates for whichever model you bound — one of the small, genuinely useful services the framework provides.

Now the punchline: your docstring is prompt engineering. When an agent "keeps picking the wrong tool" or "never uses the search tool", the fix is very often not in any loop or config — it is a docstring that says "Helper function." where it should say what the tool does, when to use it, and what it returns. You already know from the tool-calling chapter that models choose tools by reading descriptions. The @tool decorator is the pipe that carries your words to that decision.

Tip

Write tool docstrings for the model, not for your colleagues. Say what the tool does, when it should be preferred, and what the arguments mean — in plain sentences. "Look up ONE invoice's grand total by its id, e.g. INV-042. Use this before computing any tax." beats "Fetches invoice data." every single time, and it costs you one minute.

args_schema: taking the wheel

Here is a secret hiding in the miniature you just built: we gave our fake tool an args_schema attribute, and the real one has it too. Every tool made with @tool carries an args_schema — a Pydantic model, auto-generated from your type hints, that formally defines the tool's parameters and validates every incoming call. You met Pydantic in the type hints chapter: classes that turn type annotations into enforced contracts. The decorator has been quietly writing one of those for you all along.

Auto-generation covers most tools. But sometimes you want the wheel: per-argument descriptions the model will read, tighter validation, or names that differ from your Python parameters. Then you write the Pydantic model yourself and hand it over:

explicit args_schema — full control of the catalogue card
from pydantic import BaseModel, Field
from langchain_core.tools import tool

class GstArgs(BaseModel):
    total: float = Field(description="Invoice grand total in rupees")
    rate: float = Field(default=0.18,
                        description="GST rate as a fraction, e.g. 0.18")

@tool(args_schema=GstArgs)
def gst_amount(total: float, rate: float = 0.18) -> float:
    """Compute the GST payable on an invoice total."""
    return round(total * rate, 2)

Those Field(description=...) strings travel into the parameters block of the schema, so the model now gets guidance on each argument, not just the tool as a whole. That is often the difference between a model passing rate=18 and a model passing rate=0.18. A lighter-touch alternative: @tool(parse_docstring=True) reads a Google-style Args: section out of your docstring and turns it into per-argument descriptions — documentation you probably wrote anyway, promoted into the schema. And if you dislike the function's name as a tool name, @tool("lookup_invoice") overrides it.

bind_tools: handing the model the catalogue

A tool object sitting in a file helps nobody. The model needs to receive the catalogue — and that is one method call. bind_tools takes a list of tools and returns a new model wrapper that will include their schemas with every request:

binding tools to a chat model
from langchain.chat_models import init_chat_model

model = init_chat_model("anthropic:claude-sonnet-4-5")
model_with_tools = model.bind_tools([get_invoice_total, gst_amount])

reply = model_with_tools.invoke("How much GST on invoice INV-042?")
print(reply.tool_calls)
# [{'name': 'get_invoice_total',
#   'args': {'invoice_id': 'INV-042'},
#   'id': 'toolu_01Ab...', 'type': 'tool_call'}]

Read that output with last chapter's eyes. The reply is an AIMessage, and instead of prose it carries a tool_calls list: for each request, the tool's name, a dictionary of args, and an id so the eventual result can be matched back to the question that asked for it. This is the structured version of ch045's ACTION: calculate(1500 * 0.20) line — already parsed, already typed, no split("(") in sight.

Now notice what did not happen. No invoice was looked up. No GST was computed. bind_tools attaches the catalogue; the model fills in a borrow slip; and executing the tool is still, always, your code's job. Running the requested tool and sending the result back (as a ToolMessage, keyed by that id) is the loop — the one you hand-rolled in ch045, and the one next chapter's create_agent packages up for you.

Note

A pleasant modern shortcut: invoke a tool with the whole tool-call dictionary — get_invoice_total.invoke(reply.tool_calls[0]) — and it hands back a ready-made ToolMessage, result and matching id included, ready to append to the conversation. The plumbing between "model asked" and "model sees the answer" shrinks to one line.

Five terms from this chapter are worth locking in before the quiz:

In the wild

Open any LangChain-based repository and look for a file named tools.py — you will find a column of @tool-decorated functions, and you can now read each one as name-description-parameters at a glance. In merge requests, watch for diffs that touch only a docstring or a Field(description=...): reviewers who have not read this chapter wave them through as "just docs", but you know they are behaviour changes to a live agent and deserve a test. The same pattern has spread well beyond LangChain: MCP servers register tools with @mcp.tool() from FastMCP, and every provider's native tool-calling API consumes the same name-description-parameters triple underneath. Once you can read one catalogue card, you can read them all.

Check yourself

1. What does @tool read from your function to build the schema?

The decorator inspects the signature (names, hint types, defaults) and takes the docstring as the description. The body is never analysed — and never shown to the model.

2. When your tool travels to the model in an API request, what does the model receive?

The model sees the catalogue card, never the machine. It picks tools by reading names and descriptions, requests them with structured arguments, and your code alone executes anything.

3. After model_with_tools = model.bind_tools([get_invoice_total]), what has actually happened?

bind_tools attaches the catalogue and returns a new wrapper. Execution only ever happens when your code (or create_agent's loop, next chapter) reads reply.tool_calls and invokes the matching tool.

4. Why does @tool refuse a function that has no docstring?

No description means the model has nothing to go on when deciding whether this tool fits the task. The decorator fails fast at import time rather than shipping a blank catalogue card into production.

5. Your agent keeps calling a tool with rate=18 instead of rate=0.18. Which fix attacks the root cause?

The model fills in arguments based on what the schema tells it. Per-argument descriptions via args_schema Fields (or parse_docstring=True) are exactly the channel for this guidance; retries and renames leave the model just as uninformed.

Key takeaways

  • @tool is a decorator in the exact ch017 sense: it wraps your function and rebinds the name to a StructuredTool — the function plus a machine-readable catalogue card.
  • The schema is built from things you write anyway: type hints become the parameters block, the docstring becomes the description. Both are now load-bearing, and a docstring edit is a behaviour change.
  • The model only ever sees name, description, and parameters — never your code. It chooses tools by reading descriptions, so write docstrings for the model.
  • args_schema is the tool's Pydantic contract: auto-generated by default, replaceable with your own model when you need per-argument descriptions or stricter validation.
  • bind_tools attaches the catalogue to a chat model and returns a new wrapper; the model replies with structured tool_calls, and executing them remains your code's job.
  • Compared with ch045's hand-rolled version, the framework replaced the hand-written tool menu, the string parsing, and the unchecked arguments — the loop itself is still yours, until create_agent next chapter.
