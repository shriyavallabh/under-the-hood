Reading a new codebase in 30 minutes

Five questions, one timer, zero fear. You are about to learn the recon method professionals use on every unfamiliar repo — and after today, no wall of folders will ever make your stomach drop again.

~30 minChapter 106 of 133
The story

Your company buys a small office building. The previous facilities manager left overnight with no handover — just a ring of two hundred keys. Tenants arrive Monday at nine; it is now eight-thirty. Do you open all two hundred doors, one by one? Of course not. You read the name board in the lobby to learn what the building is for. You find the main entrance and the loading dock — the only places anything ever comes in. You open the utilities panel to see what the building needs from the outside world: power, water, fibre. Then you follow one tenant from the front door to her desk, noting every turn. Finally you flip through the fire-drill log, because whatever the old manager kept testing is exactly what he was afraid of.

Thirty minutes. Five questions. You can now run a building you had never set foot in. A codebase is an office building, and this chapter hands you the clipboard.

Nobody reads code like a novel

Here is the secret nobody tells newcomers: senior engineers do not read codebases either — not cover to cover. A fifteen-year veteran opening an unfamiliar repo understands maybe two percent of it by lunchtime, but it is the right two percent, found in a deliberate order. You practised the small-scale version in the first reading-real-code chapter and the second: read the imports, the names, the shape — not every line. This chapter scales that habit from one file to a whole repository, as a repeatable drill.

The drill is five questions, asked in a fixed order, against a hard timer:

  1. What is this? — README and pyproject.toml
  2. Where does it start? — the entry points
  3. What does it need? — the config
  4. What is the money path? — one request, followed end-to-end
  5. What matters? — the tests

The order is not decoration. Each question narrows the search space for the next one, the way the lobby name board tells you which floor is worth walking. Skip ahead and you drown; follow the order and a 400-file repo collapses into a one-page map.

Question 1 — what is this? (README and pyproject)

Open exactly two files. The README's first two screens tell you what the authors claim the project does. Then pyproject.toml — which you met in the packaging chapter — tells you what it actually is, because dependencies do not lie. A repo that imports fastapi is a web service. One that imports langgraph is building an agent as a graph, the shape you dissected in the StateGraph chapter. pandas means data crunching; typer or click means a command-line tool. Ten seconds of dependency reading beats ten minutes of prose.

Click through this one the way you would read it on day one:

pyproject.toml — the ten-second fingerprint
[project]
The standard metadata section. Whatever the folder is called on disk, this section holds the project's real name and identity.
name = "support-bot"
The installable name. Watch out: the import name often differs (here it will be support_bot with an underscore) — the layout logic from why-so-many-files applies.
requires-python = ">=3.11"
A maturity clue. A modern floor like 3.11+ suggests active maintenance; a 3.8 floor suggests legacy constraints somewhere.
dependencies = [
The single most information-dense list in the repo. Read it like a detective reads a shopping receipt.
"fastapi>=0.115",
A web framework. So this repo serves HTTP requests — there will be routes, and routes are entry points (question 2 just got easier).
"langgraph>=1.0",
An agent built as a graph. Somewhere in here is a StateGraph with nodes and edges — you already know its anatomy, so you know what to look for.
"httpx",
An HTTP client. The service calls out to some other system — which means there is a URL in the config (question 3 just got easier too).
]
[project.scripts]
Gold. This section is the front door, written down in ink.
support-bot = "support_bot.cli:main"
Read it right-to-left: the function main, in the module support_bot/cli.py, becomes the terminal command support-bot. You now know exactly which file to open first.

One minute in, and you already know the stack, the shape, and the first file worth opening. Write one sentence in your notes — "HTTP service wrapping a LangGraph agent that calls an external refund API" — and move on.

Question 2 — where does it start? (entry points)

An entry point is any place the outside world can start this code running — a terminal command, an HTTP route, a scheduled job, a container boot command. Programs are trees and entry points are the roots: everything else exists because some entry point eventually reaches it, through the import chains you traced in the imports chapter. Find the roots and 400 files stop being a pile and become a hierarchy. Entry points hide in a short, fixed list of places, all sweepable in five minutes:

the front-door sweep — run from the repo root
# 1. installed commands, declared in ink
cat pyproject.toml            # look for [project.scripts]

# 2. "python -m package" support
ls src/*/__main__.py

# 3. runnable scripts
grep -rn "__main__" --include="*.py" . | head

# 4. what the container actually boots
grep -n "CMD\|ENTRYPOINT" Dockerfile

# 5. how CI runs, tests, and ships it -- the team's true habits
cat .gitlab-ci.yml

That last one deserves a highlight: .gitlab-ci.yml is the repo's autobiography — the exact commands the team trusts to install, test, and deploy. If the README and the CI file disagree about how to run the project, believe the CI file: it executes on every push, so it cannot go stale.

Careful

READMEs lie. Not maliciously — they rot, because nothing breaks when they are wrong. Executable artifacts cannot rot: a wrong pyproject.toml fails installs, a wrong .gitlab-ci.yml fails pipelines, a wrong Dockerfile never boots. When prose and executable artifacts disagree, believe the executable one.

Question 3 — what does it need? (config)

The config surface is the complete list of things a program needs from the outside world before it will run — API keys, database URLs, model names, feature flags. It is also, read another way, a map of every external system the repo touches. A config surface with OPENAI_API_KEY, REFUND_API_URL, and PG_DSN tells you this service talks to an LLM provider, a refund system, and a Postgres database — before you have read a single function.

You know the hiding places from the config chapter: a .env.example file (the polite repo's gift — the whole surface in one place), a config.py or settings.py module, or a pydantic settings class where each field is one required input, typed and documented. If the repo deploys to Kubernetes, the deploy/ manifests repeat the same list from the operations side.

Four minutes here, and your notes gain a second line: "needs: LLM key, refund API URL, log level". You now know what this building is plugged into.

Question 4 — the money path

Now the heart of the method. Every repo, however sprawling, exists to serve the request it was funded to serve. The money path is that single most important request, followed end-to-end: from the entry point where it arrives, through every hop, to the response going out. Do not study modules in isolation — trace one real journey, and let it introduce the modules in the order they matter.

This is where your two power tools earn their keep: search to find where the journey starts, and jump-to-definition to follow every hop without guessing. Watch a full trace:

1. Search for a string the user would recognise

The money request here is "a customer submits a refund ticket". Search for the route text, an error message, or a button label — anything user-visible. Such strings are anchors: they appear in exactly one place, and that place is where the journey enters the code.

2. Read the handler — but only its skeleton

app.py@app.post("/tickets") def handle_ticket(...)
→
service.py

The handler is usually short: validate the input, call one function, shape the response. Ignore the details. Your only question is: which one call does the real work? Here it is run_agent(ticket). Write the breadcrumb, keep moving.

3. Jump to definition — never scroll, never guess

app.py
→

In any IDE, place the cursor on run_agent and press the jump-to-definition key (F12 in VS Code). The editor teleports you to the function's source, wherever it lives. This one keystroke is the difference between tracing a path in minutes and drowning in folders for an hour.

4. Recognise a shape you already know

agent.py
→
Agentcreate_agent(model, tools=[refund_lookup])

And here the whole course pays off: inside run_agent sits create_agent — the loop you built by hand in the ReAct chapter and met in framework dress in create_agent. You do not need to read this file carefully. You already know what a model-plus-tools loop does. Recognising shapes is what turns reading into skimming.

Six breadcrumb lines in your notes. That is the spine of the repo — every other file is a rib hanging off one of these vertebrae, and you can now place any file you open later by asking "which part of the spine does this serve?"

Question 5 — what matters? (the tests)

Last stop: the tests/ folder — not to run everything, but to read names. Test names are the most honest documentation in any repo, because a team only writes a test when it fears something breaking. test_refund_over_limit_is_escalated tells you there is a refund limit, that exceeding it must escalate rather than auto-approve, and that someone once got burned here. That is a business rule, an invariant, and a war story in one line — no wiki required.

Three quick reads: skim the test filenames for the repo's areas of concern, skim the test function names in the biggest file for the crown-jewel rules, and check which test files changed most recently (git log, from the git chapter) to see where the action is. If the repo ships an agent, find the eval suite from the previous chapter — where the evals live is where the team's real quality bar lives.

Tools that do the walking for you

The method is yours; the legwork is delegable. Four tool families make each question faster:

Search. grep -rn works everywhere; ripgrep (rg) is the same idea but dramatically faster and skips .git and virtualenvs by default. GitLab's own code search does the same trick across every repo in your group — handy when the money path leaves one repo and enters another.

Jump-to-definition and find-references. Powered by a language server reading the code's structure, not its text. Jump-to-definition follows a call forward to its source; find-references walks backward — "who calls this?" — which is exactly the question you ask before touching anything in a merge request.

Structure and graph tools. tree -L 2 prints the folder skeleton in two seconds. Your IDE's outline panel lists every function in a file so you can read a module's table of contents before its body. And a newer family of code knowledge-graph tools parses a whole repo into functions, classes, and call chains you can query directly — "what calls run_agent?" answered in one lookup instead of twenty file-opens. Whatever the brand, they pre-walk the paths so you can query them.

Git itself. History tells you where a repo is alive:

find the hot files — where the action (and the pain) lives
git log --format="" --name-only | sort | uniq -c | sort -rn | head

The files that change most often are where the features, the bugs, and the team's attention concentrate. If a file has 300 commits and everything around it has 10, you have found the building's busiest corridor.

Note

Yes — an AI assistant with the repo open will give you a guided tour, and you should ask for one. But run the five questions yourself first, then let the assistant fill gaps: the recon is how you verify the tour, and the assistant is occasionally a confident, charming, wrong tour guide. Trust structure over stories — including its stories.

Your practice plan

Like the ReAct loop, this only becomes reflex through reps. Four repos, one per sitting, each a strict 30 minutes with the checklist open:

  1. A repo you already operate. Clone an MCP server or bot you use daily and run the recon. You know its behaviour from the outside, so you can grade your own map — did the money path match what you knew?
  2. The deepagents repository. Small and canonical for this course. Find create_deep_agent, then trace how a task flows into the loop. You will recognise every shape from Act VII.
  3. The MCP Python SDK. Mid-sized. Find where FastMCP lives and how @mcp.tool() registers a function — the machinery behind the server you built.
  4. LangChain. Deliberately huge. The win is not understanding it all — it is proving that the five questions map a monster repo while touching under one percent of its files.

Rules of the drill: hard timer, written answers (one sentence each), and verify at least one answer by doing — run the entry-point command, or run one test file. An unverified map is a guess; the fire-drill log matters because someone actually held the drill.

In the wild

This skill fires constantly. A data scientist opens a forty-file merge request and wants review by Friday — thirty minutes of recon turns you from a rubber stamp into the reviewer who asks "why does the money path skip the validation module?". Your team inherits a service from a departing contractor — the recon is the handover they never wrote. A vendor claims their platform is production-grade — question 5 takes ten minutes, and their empty tests/ folder answers for them. And on day one of any new project, while others wait for the walkthrough meeting, you already have the map.

Check yourself

1. You have just cloned an unfamiliar repo. What do you open first, and why?

The README gives the authors' claim; the dependency list gives the ground truth (fastapi = web service, langgraph = agent, and so on). Together they answer "what is this?" in about a minute — and they tell you where to aim questions 2 and 3.

2. What exactly is the "money path"?

Every repo exists to serve one request well. Tracing that journey — search for where it enters, then jump-to-definition hop by hop — produces the spine of the codebase, and every other file becomes "a rib off some vertebra" instead of a mystery.

3. The README says "run python server.py", but there is no server.py in the repo. Where do you find how it really runs?

Prose rots silently; executable artifacts cannot. The CI file in particular runs on every push — whatever commands it contains are the team's real, working habits.

4. Why read test names before test bodies?

A name like test_refund_over_limit_is_escalated encodes a business rule, an invariant, and a past incident in one line — documentation that stays true because CI keeps it true.

5. The timer rings at 30 minutes and you understand roughly 20% of the repo. What now?

The method's whole premise is that the right 20% — purpose, entry points, config, money path, tests — carries nearly all the working value. Completeness past that point costs hours and returns trivia; the rest you learn the day a task touches it.

Key takeaways

  • Nobody reads codebases cover to cover. Professionals run a recon: five questions, fixed order, hard timer — and so do you now.
  • README + pyproject answer "what is this?"; dependencies are a fingerprint that cannot lie the way prose can.
  • Entry points — [project.scripts], __main__, Dockerfile CMD, the CI file — are the roots of the tree; find them and the file pile becomes a hierarchy.
  • The config surface is both a shopping list (what it needs) and a map (what external systems it touches).
  • The money path — one real request traced end-to-end with search and jump-to-definition — is the spine of the repo; write it down as 5–8 breadcrumbs.
  • Test names are the most honest documentation in the building: teams only test what they fear breaking.
  • When prose and executable artifacts disagree, believe the executable one — and verify at least one recon answer by actually running something.
