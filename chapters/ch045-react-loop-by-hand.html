Build a ReAct loop by hand

The story

A brilliant consultant is locked in a hotel room. No laptop, no phone — only a walkie-talkie to you, standing outside. She can reason about anything, but she cannot touch anything. So you agree on a protocol: she says "ACTION: check the invoice total", you go check it, and you radio back "OBSERVATION: 1,500 rupees". She thinks again, asks for the next thing, and the two of you go back and forth until she says "FINAL ANSWER".

That consultant is the LLM. You — the person outside actually pressing the buttons — are the code you're about to write. The walkie-talkie protocol is called ReAct, and the back-and-forth is the loop. Every agent you have ever operated is this hotel room, scaled up.

The idea: reasoning and acting, taking turns

ReAct — short for Reason + Act — is a simple contract with the model: instead of answering in one shot, it alternates between thinking out loud and requesting actions, and after every action it gets to see the result before deciding what to do next. The name comes from a 2022 research paper, but the pattern is older than the name: it is just how a careful person solves a problem when they can look things up as they go.

Remember from the tool-calling chapter: an LLM can only ever produce text. It cannot run a calculator, query a database, or read a file. What it can do is produce text that describes an action — and then something else, outside the model, must actually perform it. That "something else" is the loop you are about to build.

If the model needs more information it requests another action and the cycle repeats. When it writes FINAL ANSWER, the loop breaks and the answer is returned. Think, act, observe — around and around.

Note

In this chapter the model "requests" actions with plain text like ACTION: calculate(...), and we parse it ourselves. Modern APIs offer a cleaner, structured version of the same handshake — native tool calling — and frameworks build on that. We use the text version here because you can see every moving part with your own eyes. The logic is identical.

The same fifty lines, one line at a time

You just ran it. Now read it the way a reviewer would — click the annotated lines and make sure each one is boring to you. Boring means fluent.

react_agent.py — the heart of it
def run_agent(task, max_turns=5):
The whole agent is one function. It takes the user's task and a safety cap — max_turns=5 is a default argument (ch7), so callers may omit it. Every framework's "agent" object ultimately wraps a function shaped like this.
history = [f"TASK: {task}"]
The conversation history starts as a one-item list. This list is the agent's entire memory — when people say "context", they mean this. The f-string stitches the task text in (ch3).
for turn in range(1, max_turns + 1):
The famous loop, and it is just a for-loop. Bounded, not while-True: if the model dithers, we stop paying after five turns. Runaway agents are a real cost incident — ch48 has the war stories.
reply = fake_llm(history)
The only line that talks to a "model". In production this is an API call: client.messages.create(model=..., messages=history). Note what gets sent: the whole history, every turn. That is why long agent runs get expensive.
history.append(reply)
Whatever the model said becomes part of the transcript it will see next turn. The agent is building its own context as it goes.
if "FINAL ANSWER:" in reply:
The stop condition. The protocol says: when the model writes this marker, it is done. Structured tool-calling APIs signal this more elegantly (a response with no tool calls), but it is the same idea: the model decides when to stop, the loop enforces it.
return reply.split("FINAL ANSWER:")[1].strip()
split cuts the reply at the marker and [1] takes what follows — the answer itself. String surgery like this is exactly what frameworks do for you, hidden behind nicer names.
if "ACTION:" in reply:
The other branch: the model wants something done. We parse its request from plain text here; native tool calling gives you this pre-parsed as JSON — same handshake, less string surgery.
name = call.split("(")[0].strip()
Everything before the bracket is the tool's name — calculate. The model chose it from the menu our system prompt advertised.
result = TOOLS[name](arg.strip())
The most important line in the file. TOOLS is a dict mapping names to functions (ch4) — we look the name up and call the real Python function. The model asked; our code acted. This boundary is where all agent security lives: the model can only ever reach tools we put in this dict.
history.append(observation)
The tool's result re-enters the transcript labelled as an OBSERVATION, closing the circle: think, act, observe. Next iteration, the model sees it and reasons again.
return "Gave up: hit the max-turns limit."
If the loop exhausts its budget, we say so honestly instead of hanging forever. Production harnesses raise alerts here.

In the wild

Open almost any agent repository and search for its loop — you will find it. In LangChain it hides inside create_agent's agent node ("call model, if tool calls then run tools, repeat"). In hand-rolled enterprise code you will see the literal while or for with an LLM call at the top, often within a file named agent.py, loop.py, or executor.py. In a merge request, when someone says "I bumped the recursion limit" or "the agent hit max iterations", they are talking about this exact max_turns guard. You now recognise the heartbeat of every one of these systems on sight.

Check yourself

1. In a ReAct agent, who actually executes the calculator when the model says ACTION: calculate(1500 * 0.20)?

The model only produces text describing what it wants. Your code — the loop — is the only thing that executes anything. That boundary is also why agents are securable at all: the model can only reach tools you registered.

2. Why does the loop send the entire history to the model on every turn?

An LLM remembers nothing between API calls. Whatever you want it to know must be in the messages you send — which is why history grows, why long runs get expensive, and why context management (Act X) is a whole discipline.

3. What is max_turns protecting you from?

If the model never writes FINAL ANSWER, an unbounded loop would keep calling the (paid) API indefinitely. A turn cap is the simplest of the cost guardrails every production harness carries.

4. The observation step feeds the tool result back into the history. What would happen if you forgot to append it?

The model has no eyes. If the result is not placed into the transcript, then from the model's perspective the action never happened — so it re-asks, or worse, fabricates a result. "Agent keeps repeating the same tool call" bugs are often exactly this.

Key takeaways

  • An agent is an LLM, a set of tools, and a loop — you have now built all three, and nothing you meet later is more than an upgrade to one of them.
  • The model never executes anything; it writes requests as text (or structured tool calls) and your code acts. All agent security lives at that boundary.
  • The history list is the agent's entire memory; it is re-sent every turn, which is why context size and cost dominate agent engineering.
  • The loop stops when the model signals completion — and a max_turns cap protects you when it never does.
  • Frameworks (LangChain, deep agents, Claude Code) are this same skeleton with better parsing, persistence, planning, and guardrails bolted on.
