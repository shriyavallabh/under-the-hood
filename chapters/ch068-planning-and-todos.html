The planning tool

A tool that changes nothing in the world — no file, no API call, no side effect — yet reliably makes agents better at long tasks. This chapter explains the trick: the list is not a record of the plan, it is a way of keeping the plan in front of the model's eyes.

~25 minChapter 69 of 135
The story

The best wedding planner in the city carries a battered clipboard, and she rewrites her list constantly. Not once in the morning — constantly. After the flowers arrive: full rewrite. After the caterer calls: full rewrite. Her new assistant finally asks why she wastes time copying out a list she already wrote. She laughs. "The list has never carried a chair or folded a napkin. But a wedding day is loud. Whatever I wrote at seven a.m. is forty decisions behind me by noon, and a person who trusts her morning self gets surprised by her afternoon. Rewriting the list is how I remember what matters now."

Then the florist's van breaks down on the highway. The assistant panics. The planner does not — she uncaps her marker, strikes one line, writes two new ones, and keeps moving.

The clipboard is this chapter. It does no work, and it is the reason all the work gets done. Agents that run for hours carry exactly this clipboard, and in deepagents the tool that holds it is called write_todos.

Picking up from the dissection

In the previous chapter you opened up create_deep_agent and found the familiar ReAct loop at its core, wearing four upgrades: a planning tool, a virtual file system, subagents, and a long system prompt teaching the model how to use all three. This chapter is a deep dive on the first upgrade — the strangest of the four, because when you read its code, it appears to do nothing at all.

The other upgrades earn their keep in obvious ways. Files store things (next chapter). Subagents do work. But the planning tool accepts a list of tasks from the model and hands the very same list straight back. No disk, no network, no calculation. Submitted in a merge request with no explanation, it might be rejected as dead code.

And yet: give a long-running agent this "useless" tool, and it stops wandering off halfway through big jobs. Understanding why is really an idea about attention — and it applies to every model you will ever operate.

A tool that does absolutely nothing

Mechanics first. In deepagents, write_todos is a built-in tool, registered like any tool you have met before — the model sees its name, description, and schema, and calls it when it decides to. The schema asks for one thing: a list of items, each with a content string and a status of pending, in_progress, or completed. On the wire, a call looks like this:

what the model sends when it "plans"
{
  "name": "write_todos",
  "arguments": {
    "todos": [
      {"content": "Survey the three competitor pricing pages", "status": "completed"},
      {"content": "Draft the comparison table",                "status": "in_progress"},
      {"content": "Write the summary with citations",          "status": "pending"}
    ]
  }
}

And here is a conceptually faithful, simplified version of the tool itself. The real implementation stores the list in the agent's LangGraph state — the typed dictionary that flows through the graph — rather than a global variable, but the shape and the point are exactly this:

write_todos — the whole trick, simplified
from typing import Literal, TypedDict
Literal lets a type hint say "only these exact strings are allowed" — the schema generator turns it into a menu of three valid statuses, so the model cannot invent a fourth.
class Todo(TypedDict):
A TypedDict is a plain dictionary with a declared shape (recall type hints). Each item is just two fields. No dates, no owners, no priorities — deliberately minimal.
content: str
status: Literal["pending", "in_progress", "completed"]
Three statuses. If they feel familiar, they should — they are the same three you watch flip past in Claude Code's terminal checklist. Same lineage, as we will see.
def write_todos(todos: list[Todo]) -> str:
Note the argument: the entire list, every time. There is no add_item or check_off_item. To change one entry the model must restate the whole plan — and that restating, not the storage, is the feature.
"""Create or update the task list for this job."""
The real tool's docstring is much longer — it coaches the model on when to plan (multi-step work), when not to (trivial asks), and how to maintain statuses. The description is doing prompt engineering, a trick you saw in the prompting chapter.
state["todos"] = todos # agent state, not your disk
The list lives in run-scoped agent state — memory inside the graph, not a file on your machine. Nothing outside the agent changes. This is the "does nothing" line.
return f"Updated the list to {todos}"
The payoff line, hiding in plain sight. The tool result — containing the full, fresh plan — is appended to the transcript, right at the end, where the model's next turn will see it most clearly.

Read that last annotation again, because the entire chapter turns on it. Every tool result gets appended to the conversation history — that is just how the loop works. So every time the model calls write_todos, a complete, current copy of its plan lands at the newest end of its own context. The tool is not a storage device. It is a delivery mechanism.

Why writing it down makes the model smarter

To see why delivery matters, remember two facts from earlier acts. First, from the memory chapter: a model has no memory between turns — everything it "knows" about the task is rebuilt, every turn, from the transcript you send it. Second, from the context chapter: attention over a long context is not uniform. Models attend most reliably to the beginning and the end of what they are given; material buried in the middle of a very long transcript gets noticeably less pull — researchers call this the lost in the middle effect.

Now picture an agent three hundred turns into a research task. The goal statement — the whole point of the run — was message one. Since then the transcript has filled with tens of thousands of tokens of tool chatter: search results, file contents, half-relevant web pages. The goal has not been deleted. It has done something quieter and worse: sunk into the saggy middle of the context, while the freshest, most attention-grabbing tokens are whatever the last tool spat out. This is why long-running agents drift. Nobody erased the plan; it just stopped being the loudest thing in the room.

The task list is the counter-move. Each time the agent updates its list, the full plan re-enters the transcript at the tail — the position attention favours most. The builders of Manus, a well-known autonomous agent, named this trick recitation: the agent constantly rewrites its own plan so that the global objective keeps appearing in the model's recent attention span, instead of receding into the middle. Their agent famously maintained a markdown checklist file it rewrote as it worked — same clipboard, different stationery.

A well-prompted deep agent updates the list after every meaningful step — so the plan is never more than a few turns from the tail. Each rewrite is also a progress report you, the operator, can read.

There is a second, quieter benefit. Writing a plan forces the model to commit. A vague intention ("research the competitors, then summarise") becomes checkable text with statuses — and once "3 pending, 1 in progress" exists in the transcript, finishing early or skipping a step is visibly inconsistent with the model's own words. The list turns "did I finish?" from a feeling into a lookup.

See it fail, then see it recover

Claims about attention are easy to nod along to and hard to feel, so here is a toy you can poke. The "model" below is a scripted stand-in with one deliberate handicap: it can only see the last four lines of its transcript — a brutally exaggerated version of the fade real attention applies to a long context. We give it a three-item party plan, let tool chatter pile up, and run it twice: once with the plan written only at the start, once with a write_todos-style recitation re-appending the list after every step. Same model, same window, same tasks. Press Run.

Without recitation, the run ends with a confident "looks finished to me" — and the cake never gets ordered. Nothing crashed. The model did not disobey. The plan simply slid out of view under two lines of noise per step, and a model cannot follow a plan it cannot see. With recitation, the identical model completes all three tasks — the list keeps re-entering the window it actually reads.

Tip

Now experiment. Widen WINDOW to 10 and watch the no-recitation run succeed — short tasks do not need planning tools, which is exactly why Claude Code skips the checklist for small asks. Then add three more noise lines per step and watch even the wide window fail. The ratio of noise to plan is the whole game, and real agent transcripts are almost all noise.

Three ways to plan

The living task list is best understood as the middle path between two older patterns. Pure ReAct keeps the plan implicit: the model decides one step at a time, holding its intentions "in its head" — meaning nowhere. It adapts instantly to surprises but drifts badly on long horizons, as you just simulated. At the other extreme, plan-and-execute is a pattern where the model drafts the complete plan up front, then an executor marches through it step by step. It is coherent and cheap (the executor steps can even run on a smaller model), but brittle: when step two's results make step five pointless, a frozen plan executes step five anyway.

The deep-agents list is plan-and-execute with the freeze removed. The plan is explicit — written down, checkable, recited — but it is a living document the agent may rewrite on any turn, because rewriting costs exactly one tool call. You get ReAct's adaptability and plan-and-execute's coherence, paying only a few extra tokens per recitation.

When the plan goes stale

A stale plan is one the world has overtaken: a pending step turns out to be impossible (the API the plan assumed does not exist), pointless (step two's findings already answer step five), or incomplete (the work uncovered three subtasks nobody listed). Staleness is not a failure — on any real multi-hour task it is a certainty. What separates good agents from bad is the recovery move, and here it is beautifully small: call write_todos again with a corrected list. Mark the dead step completed-with-a-note or drop it, split the step that grew, insert the discoveries. Because the tool overwrites the whole list, a replan and a recitation are the same act — recovering from staleness automatically re-anchors attention too.

Two failure modes bracket the healthy behaviour. Plan-clinging: the agent grinds through step four even though step two's results made it moot — doggedly doing work whose premise has collapsed. Plan churn: the agent rewrites its list five times in a row without executing anything — motion on the clipboard, none in the world. The system prompts that ship with these tools push against both, with rules you can almost quote if you have watched Claude Code work: keep exactly one item in_progress at a time; mark an item completed immediately when done, never batching completions; add newly discovered work to the list instead of silently doing it. Each rule has a purpose — one in_progress item makes drift detectable at a glance, immediate completion keeps the transcript's story true, and listing discovered work keeps the plan honest about the real size of the job.

Careful

A task list makes an agent look organised, and looks can be bought cheaply. The list is self-reported: an agent can mark a step completed having done it badly, or having not done it at all. Planning tools fix attention, not honesty — verification (tests, checks, a reviewer agent) is a separate discipline, and the failure-modes chapter is why you never skip it.

You have been watching this tool for months

If you operate Claude Code, none of this is new — you have just been seeing it from the outside. That checklist that appears in your terminal on any multi-step job, with items flipping from open to in-progress to done? That is TodoWrite, Claude Code's planning tool: same three statuses, same overwrite-the-whole-list semantics, same purpose. deepagents' write_todos is built quite openly on that playbook. Every pattern in this chapter is something you have watched happen live: the list appearing a few seconds into a big task (the plan being drafted), an item quietly splitting into three mid-run (stale-plan recovery), the single glowing in-progress item (the one-at-a-time rule).

And this hands you, the operator, a real lever: the task list is the agent's steering wheel, and it is visible. When the plan appears, read it. If step three aims at the wrong target, interrupting now costs seconds; letting the agent find out at step three costs twenty minutes and a pile of tokens. You are being shown the model's intentions before it acts on them — very few systems in your professional life are that courteous.

In the wild

You will meet planning tools everywhere agents run long. In deepagents code, look for write_todos in the tool list and a todos key in agent state. In Claude Code transcripts, the tool is TodoWrite. Manus's engineering write-ups describe the checklist file it rewrites as it works, and coined "recitation" for the practice. In merge requests you will read "gave the research agent a planning tool — long runs stopped losing the thread", and in system prompts you will find whole paragraphs coaching the model on when to plan and how to keep statuses current. When someone says an agent "keeps abandoning tasks halfway", your first diagnostic question writes itself: where is its plan, and when did the model last see it?

Check yourself

1. What does write_todos actually change when the model calls it?

The tool has no external side effect at all. Its entire value is internal: the current plan lands at the newest end of the context, where the model's next turn will attend to it most strongly.

2. Why does rewriting the task list help a model on a 300-turn task?

Models rebuild their understanding from the transcript every turn, and material buried in the middle of a long context gets less attention. Recitation keeps dragging the plan back to the tail, where it is read most reliably. (Option a is exactly what models cannot do — they have no memory between turns.)

3. How does a living task list differ from classic plan-and-execute?

Plan-and-execute freezes the plan up front and marches through it, which breaks when reality disagrees. The living list keeps the explicitness — written, checkable, recited — while making replanning as cheap as any other tool call.

4. Mid-run, the agent discovers that step 3 of its plan is impossible. What is the healthy recovery?

Stale plans are a certainty on long tasks, not a failure. Because the tool overwrites the whole list, the replan doubles as a recitation — the corrected plan re-anchors attention in the same call. Option a is plan-clinging; option b throws away the anchor.

5. Why do these tools' prompts insist on exactly one item being in_progress at a time?

It is a discipline rule, not a technical limit. One glowing item means the list always answers "what is the agent doing right now?" truthfully — which is what lets an operator (or the model itself, on its next turn) spot a wander the moment it starts.

Key takeaways

  • write_todos is deliberately a no-op on the world: it stores {content, status} items in agent state and returns the list as a tool result. Its power is entirely in what that does to the context.
  • Models rebuild their understanding from the transcript every turn, and attention favours the recent end of a long context — so an unrecited plan sinks into the "lost in the middle" zone and the agent drifts.
  • Recitation — rewriting the full plan so it re-enters the tail of the context — is the fix, and the overwrite-the-whole-list design of write_todos makes every update a recitation for free.
  • The living task list is the middle path: pure ReAct adapts but drifts, plan-and-execute stays coherent but clings to stale plans, and a rewritable plan gives you both for one tool call per update.
  • Healthy plan hygiene: one item in_progress, mark items done immediately, add discovered work to the list. Watch for the two bracketing failures — plan-clinging and plan churn.
  • Claude Code's TodoWrite checklist is this exact tool worn on the outside. Read the plan when it appears — it shows you the agent's intentions before it spends your time and tokens on them.
