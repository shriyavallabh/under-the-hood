Tool calling: the trick under every agent

A language model can only ever produce text — so how does it read your files, query your database, book your meetings? This chapter shows you the JSON handshake that makes it all possible, wire message by wire message.

~35 minChapter 36 of 117
The story

The concierge at a grand hotel never leaves her desk. She cannot drive a cab, cook a biryani, or fix a leaking tap. Yet guests swear she can do anything. Ask her for a cab at six and, minutes later, one is idling in the porch.

Here is the trick. Behind her desk sits a tray of printed slips, one for each service the hotel offers. She listens to your request, picks the right slip — CAB REQUEST: pickup time, pickup point, destination — fills it in neat block letters, and drops it through a hatch to the dispatch office. Someone in dispatch actually phones the taxi stand, writes the outcome on the bottom half of the slip — "Cab MH-12 confirmed, 6:00 pm" — and passes it back through the hatch. Only then does the concierge turn to you and say, "Your cab is booked, sir."

She never booked anything. She only ever produced words in a very particular format, and trusted the building around her to act on them. Every AI agent you have ever operated is this desk — the slips, the hatch, the dispatch office, all of it. This chapter opens the hatch.

A text predictor cannot press buttons

Hold on to what you learned earlier in this act. A language model is a next-token predictor: text goes in, the most plausible continuation comes out, one token at a time. In the previous chapter you tuned how it picks those tokens with temperature and friends. But nothing in that machinery can open a file, hit a database, or send an email. The model has no hands. It is the concierge: all voice, no legs.

And yet you watch Claude Code edit files all day. You watch production agents fetch CRM records and fire webhooks. The resolution of this paradox is the single most important mechanism in this entire course, and it is almost embarrassingly simple:

Tool calling is an agreement between you and the model: you tell it what functions your code is willing to run, the model replies with a structured, machine-readable request to run one — and then your code actually runs it and reports the result back. The model "calls" a function the way the concierge "books" a cab: by filling in a form and trusting someone else to act.

That is the whole trick. Everything else in this chapter is just looking closely at the form.

The handshake, in five moves

One complete tool-calling exchange is two API calls and one real function execution, in a fixed order. Step through it before we look at the actual bytes:

1. Your request carries a tool menu

Your codequestion + tool schemas
→
Model API
get_weather()your local function

Alongside the user's question, your request includes a tools list: each tool's name, a description, and a schema of its parameters. This is the tray of slips. The function itself never leaves your machine — only its description travels.

2. The model decides a tool would help

Your code
→
Model"get_weather looks right for this"
get_weather()

The model reads the question, reads the menu, and — purely by predicting text, same as ever — decides that answering well requires a tool. Providers train and constrain the model so this decision comes out as tidy JSON, not free-form prose.

3. The reply is a structured tool call

Model API
→
Your code{"name": "get_weather", "input": {"city": "Pune"}}
get_weather()

Instead of an answer, the response contains a tool-call block — the tool's name and its arguments as parsed JSON — plus a stop reason saying "I am waiting for you". Nothing has executed. The model has written a wish.

4. Your code runs the real function

Model API
Your code
→
get_weather("Pune")= "31 C, humid, rain likely"

Your dispatcher looks the name up in its registry of real Python functions, validates the arguments, and calls it. This is the dispatch office phoning the taxi stand — the only step where anything actually happens in the world.

5. The result goes back as a tool message

Your codetool result: "31 C, humid…"
→
Model"Carry an umbrella in Pune today."

You append the function's output to the conversation as a tool-result message and call the API again. Now the model has what it needed, and it writes the final prose answer. Two API calls, one handshake, done.

Note

Notice who does what. The model chooses and describes; your code executes and reports. When a vendor slide says "the model calls your API", it is marketing shorthand for this handshake. The model never touches your API. It cannot. That distinction is not pedantry — it is where every security guarantee about agents lives.

The wire, message by message

Now the actual bytes. Below is the request a harness sends when it wants tool calling — this one in the Anthropic Messages API dialect, but every provider's version is the same idea with different key names. It is ordinary JSON, nothing exotic. Click the annotated lines:

API call 1 — the request you send
{
"model": "claude-sonnet-4-6",
Any current chat model. The handshake works the same across providers and model tiers — tool calling is a convention of the API, not a feature of one model.
"max_tokens": 1024,
The usual response budget, exactly as in a plain chat call. Nothing about tool calling changes the basic request shape you already know from earlier in this act.
"tools": [
Here is the new part: the tool menu. A list of schemas — one per function your code is prepared to run. This travels up with every single request; the model has no memory of your tools between calls.
{
"name": "get_weather",
The label the model will use to ask for this tool. Your dispatcher will match on this string, so it must be identical on both sides — a mismatch here is a classic agent bug.
"description": "Get current weather for a city.",
Read carefully: this sentence is how the model decides when to use the tool. It is prompt engineering hiding in a schema. Vague descriptions produce agents that ignore their tools or grab the wrong one.
"input_schema": {
A JSON Schema describing the parameters — the blank fields on the slip. Frameworks usually generate this automatically from a Python function's type hints and docstring, which is why type hints matter so much in agent code.
"type": "object",
"properties": {
"city": {"type": "string", "description": "City name"}
One parameter: a string called city, with its own mini description. The model uses these to fill the form correctly — "Pune" goes here, not "tomorrow at 6".
},
"required": ["city"]
Which fields must be filled. The model is trained to honour this, and providers can hard-enforce it if you mark the tool strict — but by default it is a strong hint, not a guarantee, which is why dispatchers still validate.
}
}
],
"messages": [
The familiar conversation array. Tool calling does not replace messages — it rides on top of them.
{"role": "user", "content": "What's the weather in Pune?"}
]
}

The model reads that and — if it decides the tool would help — sends back something like this. Look at what is not here: an answer.

API response 1 — a form, not an answer
{
  "stop_reason": "tool_use",
  "content": [
    {"type": "text",
     "text": "I'll check the current weather in Pune."},
    {"type": "tool_use",
     "id": "toolu_abc123",
     "name": "get_weather",
     "input": {"city": "Pune"}}
  ]
}

Three things deserve a slow look. First, "stop_reason": "tool_use" — the model is explicitly saying "I have stopped because I am waiting for you". Second, the tool_use block: the tool's name and its arguments, already parsed into a JSON object your code can use directly. Third, that id. It looks bureaucratic, but it is what lets one response carry several tool calls at once and still get each result matched to the right request — the slip's serial number.

Your code now runs the real get_weather("Pune"), gets a result, and sends API call number two: the same conversation, extended with the model's tool call and your result:

API call 2 — the tool result goes back
{
  "role": "user",
  "content": [
    {"type": "tool_result",
     "tool_use_id": "toolu_abc123",
     "content": "31 C, humid, rain likely by evening"}
  ]
}

And only now, with the observation in hand, does the model produce what the user actually wanted:

API response 2 — the real answer
{
  "stop_reason": "end_turn",
  "content": [
    {"type": "text",
     "text": "It's 31 C and humid in Pune, rain likely by evening - carry an umbrella."}
  ]
}

stop_reason is back to end_turn: the model is done talking, the handshake is complete. Two API calls, one function execution, one happy user.

One handshake, two dialects

Anthropic and OpenAI implement the same handshake with different key names, and you will meet both in logs within your first week of debugging any real system. The differences are cosmetic, but one of them bites:

Anthropic Messages APIOpenAI chat completions
Tool schema keyinput_schemaparameters (inside a function wrapper)
The model's requesttool_use content blocktool_calls array on the message
Arguments arrive asa parsed JSON objecta JSON string you must parse yourself
Result goes back astool_result block inside a user messagea message with role: "tool"
"Waiting for you" signalstop_reason: "tool_use"finish_reason: "tool_calls"

The one that bites is the arguments-as-a-string row: OpenAI hands you "{\"city\": \"Pune\"}" — a string containing JSON — and your code must json.loads() it before use. A surprising number of "the agent passed garbage to my function" bugs are simply a missing parse on that string. Frameworks hide all of these differences behind one interface, which is precisely why they exist.

When the form comes back wrong

The model fills forms by predicting text, and prediction sometimes goes sideways. Three failure shapes cover most of what you will see. The model may hallucinate arguments — a city that does not exist, a date in the wrong format — so real dispatchers validate inputs against the schema before executing, exactly the way Pydantic validates data at a boundary. It may request a tool that is not on the menu, in which case your registry lookup fails — a plain dict KeyError, nothing mystical. And your function itself may fail: the database is down, the file is missing.

Here is the move that separates people who understand the handshake from people who merely use it: in all three cases, the recovery is the same — send the failure back as the tool result. The result field is just text you choose. Put "ERROR: unknown city 'Punee'. Valid cities: Pune, Mumbai, Delhi" in a tool result and the model reads it, apologises to nobody, and retries with corrected arguments. You are not just piping data back; you are in a conversation, and error messages are how your side of the conversation teaches the model to self-correct.

Careful

The registry is a security boundary — treat it like one. The model can request anything it can imagine, but it can only reach what your dispatcher agrees to run. Never build a "convenient" tool like run_any_shell_command without confirmation gates: you would be handing the concierge the master key to every room. Claude Code's permission prompts are exactly this — the dispatcher pausing at step 4 of the handshake to ask a human before executing.

From one handshake to an agent

Look back at what you now have: the model can ask for a function, your code can run it and report back, and the model can then decide what to do next. What if, instead of stopping after one exchange, your code simply kept going — feeding each result back and letting the model request another tool, and another, until it finally answers with end_turn?

That "simply kept going" is a loop about ten lines long, and it is the entire definition of an agent. Not a metaphorical loop — a literal Python for loop wrapped around everything this chapter taught you. In Build a ReAct loop by hand you will write it yourself, and you will find it almost anticlimactic: the handshake was the hard part, and you now own it. The next chapter pulls a different thread first — once a model can reliably emit structured JSON on demand, you can use that same discipline to get any response in a guaranteed shape, no tools involved. That is structured output, and it is tool calling's quieter twin.

In the wild

This handshake is everywhere once you know its face. In an MCP server, @mcp.tool() above a plain Python function auto-generates the schema and registry entry — MCP is largely a standard for shipping tool menus between processes. In LangChain, the tools list you hand to create_agent becomes exactly the tools array you saw above. In raw API logs, stop_reason: "tool_use" or finish_reason: "tool_calls" tells you the model is mid-handshake. And when a merge request "adds a tool to the agent", expect three things in the diff: a function, its schema (often generated from type hints), and one new line in a registry — the slip, the printed form, and the dispatch office's phone list.

Check yourself

1. When the model "calls" get_weather, what actually happens?

The model only ever produces text — here, text shaped as a JSON tool call. Your dispatcher looks the name up in its registry and runs the real function. The provider never sees your code, and the model never executes anything.

2. What travels up to the API to make tool calling possible?

Only the advertisement travels — never the implementation. The model chooses tools by reading names and descriptions, which is why a sloppy description quietly cripples an agent.

3. Your code ran the tool and has the result. How does the model find out?

The model is stateless between API calls. The result reaches it the only way anything ever reaches it: as a message in the conversation you send. The tool_use_id ties your result to the specific request, which matters when several tools were called at once.

4. A response arrives with stop_reason: "tool_use". What is the model telling you?

A stop reason says why the model stopped talking. tool_use means "the ball is in your court" — nothing has executed yet, and nothing will until your dispatcher acts and sends the result back.

5. The model requests a tool with a made-up argument and your function throws an error. The best-practice response is to…

The tool result is just text you choose, and the model reads it like any other message. A clear error message ("unknown city; valid options are…") routinely produces a corrected retry on the very next turn — self-correction is built out of exactly this.

Key takeaways

  • A language model can only produce text — tool calling is the agreement that lets text describe an action which your code then performs. The model chooses; your code executes.
  • The handshake has five moves: send the tool schemas with your request, receive a structured tool call, execute the real function yourself, return the output as a tool result, and receive the final answer on a second API call.
  • Only schemas travel to the API — name, description, JSON Schema of parameters. The description is prompt engineering in disguise: it is how the model decides when to reach for the tool.
  • stop_reason: "tool_use" (or OpenAI's finish_reason: "tool_calls") means the model is paused, waiting for your side of the handshake.
  • Failures — bad arguments, unknown tools, broken functions — all recover the same way: send the error text back as the tool result and let the model self-correct.
  • The tool registry is the security boundary of every agent, and the whole agent pattern is just this handshake wrapped in a loop — which you will build by hand in chapter 46.
