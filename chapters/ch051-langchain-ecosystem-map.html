The story

Your cousin gets a job at a giant industrial group — one famous brand name painted on the front gate. On day one she expects one building. Instead the campus has five. There is the standards bureau, a small quiet office that decides the shape of every plug, socket, and form used on campus — it makes nothing, but everything is built to its specifications. There are the vendor pavilions, one per electricity supplier, each selling an adapter that converts that supplier's odd connector into the standard plug. There is the big assembly plant where ready-to-use machines get put together from standard parts. Behind it stands the control tower, which choreographs the complicated assembly lines — the ones with branches, loops, and human sign-off stations. And on the hill sits the camera room, recording every station so that when a machine misbehaves, someone can replay exactly what happened.

One brand. Five buildings, each with its own stamp on every crate it ships. LangChain is that campus, and the stamps are import lines. This chapter is your visitor map.

One brand, five packages

Quick recap of where you stand. In the ReAct chapter you built an agent loop by hand — model, tools, history, a for-loop, done. In the last chapter you read a real annotated agent and recognised every part. So you already know what an agent framework must do. What trips people up with LangChain is not the ideas — it is that "LangChain" is one name for a whole family of separately installed packages, and tutorials mix them freely without telling you which is which.

Here is the untangling. LangChain is three things at once: a company, an ecosystem of Python packages that company publishes, and one specific package (langchain) inside that ecosystem. When someone says "we use LangChain", they usually mean the ecosystem. When you read from langchain.agents import create_agent, that is the specific package. The five members of the family you will meet constantly:

  • langchain-core — the standards bureau. Interfaces and message types everything else agrees on.
  • langchain-openai, langchain-anthropic, and friends — the vendor pavilions. One adapter package per model provider.
  • langchain — the assembly plant. The high-level framework: create_agent, middleware, init_chat_model.
  • langgraph — the control tower. Orchestration for flows the standard loop cannot express.
  • langsmith — the camera room. Observability: tracing, evaluation, monitoring.

Remember from the imports chapter that a Python import names the package it comes from, right there on the line. That single fact is about to become your compass: on this campus, every crate carries its building's stamp. You never have to guess which layer you are reading.

Note

Package names on PyPI use hyphens (pip install langchain-anthropic) but Python identifiers cannot contain hyphens, so the import uses an underscore (from langchain_anthropic import ...). Same package, two spellings. This bites everyone exactly once.

Walk the campus, one building at a time

Step through the map. Watch which building lights up, and what it would be doing during one real agent run.

1. langchain-core — the standards bureau

A deliberately small package with almost no dependencies. It defines the shapes: what a message object looks like (HumanMessage, AIMessage, ToolMessage — next chapter's whole topic), what a tool is, what any chat model must accept and return. It contains no agents and calls no APIs. It exists so that every other package can agree on the plugs and sockets.

They are versioned and released separately — when a provider ships a new model or changes a parameter, only its adapter needs an update, not the whole ecosystem.

The main framework package. Its star export, create_agent, is your fifty-line ReAct loop industrialised: native tool calling, retries, streaming, and hook points called middleware where you can intercept each turn. You hand it a model and some tools; it hands you a running agent. Chapters 55 and 56 take it apart.

The orchestration engine, for when one loop is not enough: multi-step workflows with branches, parallel paths, pause-for-a-human, and saved state you can resume next week. Here is the quiet twist: create_agent itself runs on the LangGraph runtime under the hood. The plant's machines are driven by the tower even when you never look at it. 5. langsmith — the camera room

your agentrunning

Observability. It records every model call, tool call, and intermediate step of a run so you can replay it when something goes wrong. Unusually, you often never import it: set the environment variables LANGSMITH_TRACING and LANGSMITH_API_KEY and the other packages start reporting to it automatically. It watches; it does not steer.

The map on one page

Because every crate is stamped, identifying the layer is a mechanical skill. Read the first dotted name after from (or after import) and look it up:

The import starts withLayerWhat it tells you
langchain_core.CoreA shared interface or data shape — messages, tools, prompts. No behaviour of its own worth fearing.

Reading the stamps on a real file

from langchain.chat_models import init_chat_model
Stamp: framework. A convenience factory — give it a model name like "claude-sonnet-4-5" and it finds and configures the right provider adapter for you, so your code is not welded to one vendor.
from langchain_core.messages import HumanMessage
Stamp: core. A data shape from the standards bureau. The whole message vocabulary — HumanMessage, AIMessage, ToolMessage — is the next chapter.
from langchain_core.tools import tool
Stamp: core. The @tool decorator that turns a plain function into a schema the model can call. Note it lives in core, not in langchain — it is an interface, so it belongs to the bureau.

Stamp: orchestration. A checkpointer from the control tower — the thing that lets an agent's state survive between runs. Full story in chapter 62. Its presence tells you this agent remembers.

Carbon-dating a tutorial

Here is the part that saves you real pain. LangChain has existed since late 2022 and has been rewritten more than once, so the internet is full of confident, well-written, completely outdated tutorials. The big line in the sand is LangChain 1.0 (late 2025): agents were rebuilt around create_agent on the LangGraph runtime, and the older machinery was moved aside into a separate legacy package (langchain-classic) so the main package could stay clean. Anything written before that line teaches you a museum.

Three eras, three sets of fingerprints:

  • The legacy era (2023–2024): initialize_agent, AgentExecutor, LLMChain, ConversationBufferMemory, and imports like from langchain.llms import OpenAI — providers bundled inside the main package instead of partner packages. If you see any of these, close the tab.
  • The LCEL era (2023–2025): LCEL — the LangChain Expression Language — is the pipe syntax, prompt | model | parser, for composing steps into a pipeline. It still exists and is fine for simple transformations, but a tutorial that builds agents out of long LCEL chains is from before agents got their own machinery.
  • The 1.x era (current): create_agent, init_chat_model, middleware, partner packages installed separately, LangGraph named openly as the engine underneath. This is the era this course teaches.
carbon dating at a glance — museum out, current in
- from langchain.agents import initialize_agent, AgentExecutor
- from langchain.llms import OpenAI
- from langchain.memory import ConversationBufferMemory
+ from langchain.agents import create_agent
+ from langchain.chat_models import init_chat_model
+ from langgraph.checkpoint.memory import InMemorySaver

The install line is a second fingerprint. Old posts say pip install langchain openai and nothing else. A current setup names the family members it actually uses:

a current-era install line
pip install -U langchain langchain-anthropic langgraph
Careful

Stale docs are not just a beginner hazard — coding assistants trained on years of old tutorials will happily generate AgentExecutor code too. When an AI pair-programmer hands you LangChain code, run the same carbon-dating check on its imports that you would run on a blog post. You now spot the fingerprints faster than it can type them.

Why should you trust the design instead of resenting the churn? Because the split is the same one you would make. Recall why big projects split into many files: things that change at different speeds get separated. Provider APIs change monthly — so adapters are separate packages. Interfaces must almost never change — so core is tiny and frozen. Orchestration is deep infrastructure — so it is its own engine. The ecosystem map is that principle, drawn at package scale.

In the wild

Open the pyproject.toml or requirements.txt of any agent repository (chapter 20 taught you to read these) and the map appears: langchain, langchain-anthropic, langgraph, sometimes langsmith, each pinned to its own version. In merge requests you will see titles like "bump langchain-openai for the new model names" touching one line — the vendor pavilion updated, nothing else disturbed. And in incident channels, "check the LangSmith trace" means: go to the camera room and replay the run. When a teammate asks "is this doc current?", you can now answer from the imports alone, in about four seconds.

Check yourself

1. You read from langchain_core.messages import AIMessage. Which building shipped that crate?

The first dotted name is langchain_core, so it is the core layer: a data shape everyone agrees on, with no behaviour to fear. Read the stamp, know the building.

2. Why do ChatOpenAI and ChatAnthropic live in separate installable packages instead of inside langchain itself?

Provider APIs change constantly; interfaces should not. Splitting adapters into partner packages means a new model name ships as a one-package version bump, without touching core, the framework, or anyone else's code.

3. A tutorial you found builds its agent with initialize_agent and AgentExecutor. What is the correct conclusion?

Those are legacy-era fingerprints. The 1.0 rebuild replaced them with create_agent and moved the old machinery into a separate legacy package. A tutorial teaching them is teaching a museum exhibit.

4. What is the real relationship between create_agent (in langchain) and LangGraph?

The assembly plant's machines are driven by the control tower: create_agent gives you the standard loop ready-made, and when you need custom flow — branches, pauses, multi-agent choreography — you drop down to LangGraph directly.

5. Your agent behaved strangely in production yesterday and you want to replay exactly what it did, step by step. Which layer do you reach for?

LangSmith is the camera room: with tracing switched on (largely via environment variables), it records every model call, tool call, and intermediate step, so yesterday's run can be replayed and inspected today.

Key takeaways

  • "LangChain" names a company, an ecosystem, and one package — the ecosystem is five layers: core (interfaces), integrations (per-provider adapters), langchain (the framework, create_agent and middleware), langgraph (orchestration), and langsmith (observability).
  • Every import line carries its layer's stamp: read the first dotted name — langchain_core, langchain_anthropic, langchain, langgraph, langsmith — and you know exactly which part of the machine you are looking at.
  • Provider adapters are separate packages on purpose: fast-changing vendor APIs get isolated so an update to one never disturbs the rest.
  • create_agent is not an alternative to LangGraph — it runs on the LangGraph runtime, and you drop to LangGraph directly only when the standard loop cannot express your flow.
  • Carbon-date every tutorial before trusting it: initialize_agent, AgentExecutor, LLMChain, and bundled provider imports are pre-1.0 museum pieces; create_agent, init_chat_model, and partner packages mark the current era.
  • LangSmith watches rather than steers — it is usually enabled by environment variables, and "check the trace" is where production debugging starts.
