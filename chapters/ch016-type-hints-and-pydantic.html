Type hints and Pydantic

The little labels after the colons — and the classes that turn those labels into law at your system's borders. After this chapter, def f(x: list[str]) -> dict reads like a plain English sentence, and every tool schema an agent has ever shown you makes sense.

The story

Picture the export desk of a busy courier depot. Every parcel that leaves must carry a declaration form: contents — text; weight — a number in kilograms; fragile — yes or no. The form does not change what is inside the box. A careless sender can write "weight: forty" and tape the box shut anyway, and the parcel will still roll down the belt. But the labels are priceless all the same: sorters route parcels by reading them, new staff learn the whole depot by reading them, and a scanner flags suspicious forms before the truck leaves.

Then there is the customs officer at the border. She does not trust labels. She opens the box, weighs it, and compares reality against the form. "forty" becomes 40 if she can convert it; if she cannot, the parcel is rejected on the spot — with a written list of everything wrong, not just the first thing.

Python type hints are the declaration form: labels describing what should be in the box, enormously useful to readers and tooling, enforced by nobody at runtime. Pydantic is the customs officer: the checkpoint where data actually gets opened, converted, and validated before it enters your system. Every agent you operate runs both.

Reading the labels: annotation syntax

Quick recap before the new material. In the functions chapter you learned that def creates a function, that parameters can carry default values, and that return hands a value back to the caller. In the previous chapter you saw how frameworks hand you base classes to extend. Type hints bolt one more thing onto that machinery: small, optional labels that say what kind of value each name is expected to hold. Here is the same function three ways — bare, labelled, and fully labelled:

annotation_grammar.py
def send_reminder(client, days_overdue):            # bare — you must guess
    ...

def send_reminder(client: str, days_overdue: int):  # labelled parameters
    ...

def send_reminder(client: str, days_overdue: int) -> bool:   # and the return
    ...

retries: int = 3          # variables can be labelled too
last_error: str | None = None

The grammar is tiny. After a parameter name, a colon and then the expected type: client: str reads as "client, a string". Between the closing bracket and the final colon of a def, an arrow: -> bool reads as "and this function returns a True/False". That arrow is just a hyphen and a greater-than sign — it is not an operator, only punctuation for "returns".

The types can be containers from the collections chapter, and the square brackets read as the word of. list[str] is "a list of strings". dict[str, int] is "a dict of string keys to integer values". You can nest them: list[dict[str, int]] — a list of such dicts. Three more shapes complete the everyday vocabulary:

  • str | None — "a string, or nothing". The vertical bar reads as "or". Functions that might not find what they looked for are hinted this way.
  • Optional[str] — the older spelling of exactly the same thing, imported from the typing module. You will meet both constantly in real repositories; they mean the same and the | None form is the modern preference.
  • Callable[[str], str] — "a function that takes a string and returns a string". Remember from ch7 that functions are ordinary objects you can pass around; this is the label for a parameter that expects one.

Flip these until each one reads as a sentence, not a symbol:

The label is not a lock

Here is the fact that surprises everyone coming from operating typed systems: Python ignores type hints when your program runs. A type hint is an annotation attached to a function or variable that describes the intended type — and that is all it is. Pass a number where the hint says string, and Python will not stop you. If the function then does something a number cannot do, you get a crash — from the operation, never from the label. Run it and see:

Two things in that output matter. First, shout(42) was allowed to start — the crash came from .upper() not existing on a number, one line too late. Second, total.__annotations__ printed a dict of the labels. Python did not throw the hints away; it filed them on the function, where any other code can read them. Hold that thought — it is the secret of the whole second half of this chapter.

So if hints stop nothing, why does every serious codebase use them? Because three audiences read them constantly:

  • Humans. def retry(task: str, times: int) -> bool answers, at a glance, the questions you would otherwise dig through the function body to answer. Hints are documentation that sits exactly where you are already looking.
  • Editors. Autocomplete, hover-help, and the red squiggles in an IDE all run on hints. Label a parameter client: str and the editor offers string methods the moment you type the dot.
  • Static checkers. Tools like mypy and pyright — programs that read your code without running it and report every place a value's type contradicts a label. Teams run them in CI, so the mismatch is caught in the merge request, not in production at 2 a.m.
Tip

When you skim an unfamiliar repository, read signatures before bodies. A well-hinted signature like def fetch_invoices(client: str, limit: int = 20) -> list[dict] tells you the inputs, the defaults, and the shape of the output — often you never need to read the body at all. This is the fluency this chapter is buying you.

Pydantic: the customs officer

Labels that nobody enforces are fine inside your own code, where you control both ends of every call. They are not fine at the borders — the places where data arrives from outside: a web request, a YAML config file, a row from a spreadsheet, or (the case this course cares about most) text generated by a language model. Border data is always strings, always optimistic, and frequently wrong. That is where Pydantic comes in — a library that turns type hints into actual runtime validation.

You use it by writing a class that inherits from BaseModel — the framework-style inheritance you met in the previous chapter — and declaring fields with exactly the hint syntax you just learned:

invoice_args.py — real Pydantic (v2)
from pydantic import BaseModel, Field, ValidationError

class InvoiceArgs(BaseModel):
    client: str = Field(description="Registered client name, as it appears in the CRM")
    amount: int = Field(gt=0, description="Invoice value in rupees, before GST")
    urgent: bool = False        # a default makes the field optional

args = InvoiceArgs(client="Meridian Realty", amount="4500")
print(args.amount)              # 4500  — the string was CONVERTED to an int
print(args.model_dump())        # {'client': 'Meridian Realty', 'amount': 4500, 'urgent': False}

try:
    InvoiceArgs(amount=-10)
except ValidationError as e:
    print(e)                    # BOTH problems reported: client missing, amount not > 0

Walk through what happened, because each behaviour is a customs-officer move:

  • Creating an instance is the validation. The moment you write InvoiceArgs(...), every field is checked against its hint. There is no separate "validate" step to forget.
  • Coercion. amount="4500" arrived as a string — as border data always does — and Pydantic converted it to the integer 4500 because the label said int and the conversion was safe. The officer weighed the box and corrected the form.
  • Rejection with a full report. The bad call raised ValidationError, and the error message lists every failing field at once — missing client and non-positive amount — not just the first one it tripped over. In agent systems this whole report is usually sent straight back to the model so it can correct itself.
  • .model_dump() converts the validated object back into a plain dict — clean, converted, ready to log, store, or serialise. Its sibling .model_dump_json() gives you the JSON string directly.
Note

If you meet code calling .dict() and .json() on models, you are reading Pydantic version 1, the older generation. Modern code (v2, current for years now) spells these .model_dump() and .model_dump_json(). Same idea, new names — worth knowing because both generations are still in the wild.

Build a miniature Pydantic yourself

Pydantic can feel like magic until you remember the clue from earlier: Python files the hints away in __annotations__, where code can read them. That is the entire trick. A class's field labels are just a dict of names to types, so a base class can loop over that dict, check each incoming value, attempt conversion, and collect errors. Field descriptions: the sentence the model reads

Look back at the real Pydantic example and notice Field(description="Invoice value in rupees, before GST"). Field is Pydantic's way of attaching extra rules and metadata to one field — constraints like gt=0 (greater than zero), defaults, and, most importantly for us, a human-readable description. Why does a validation library care about prose? Because every Pydantic model can print its own specification:

the model describes itself
print(InvoiceArgs.model_json_schema())
# {'properties': {'client': {'description': 'Registered client name, as it
#      appears in the CRM', 'type': 'string'},
#   'amount':  {'description': 'Invoice value in rupees, before GST',
#      'exclusiveMinimum': 0, 'type': 'integer'},
#   'urgent':  {'default': False, 'type': 'boolean'}},
#  'required': ['client', 'amount'], ...}

That output is a JSON schema — a standard, machine-readable description of what a valid piece of JSON must look like, built automatically from the hints and the Field descriptions. And this is the punchline of the chapter: that schema is exactly what a language model is shown when it is offered a tool. The model never reads your Python. It reads the schema. Which means those description strings are not comments — they are prompt engineering. "Invoice value in rupees, before GST" is the sentence standing between the model and it guessing the amount in dollars, after tax.

A decorator — the @ syntax gets its own chapter next. For now: it registers the function below as a tool AND, at that moment, reads the signature's hints to build the JSON schema from step 2 of the stepper.
def create_invoice(client: str, amount: int, urgent: bool = False) -> str:
The hints do the heavy lifting: client: str becomes "type": "string" in the schema, amount: int becomes "type": "integer", and the default on urgent marks it optional. Delete the hints and the tool's contract collapses to "three mystery arguments".
"""Create a draft invoice for a client, amount in rupees."""
The docstring becomes the tool's description — the sentence the model reads when deciding whether this tool fits the task. Vague docstring, misused tool.
return f"Draft invoice: {client}, {amount} rupees"
Ordinary Python from here on. By the time this line runs, validation has already happened upstream — the arguments are guaranteed to match their labels.

LangChain's @tool decorator plays the identical game — signature hints plus docstring in, schema out — and under the hood it builds a Pydantic model from your signature to do the border check. TypedDict: a shape for plain dictionaries

One last citizen of this family, because you will meet it the moment you open any LangGraph codebase. Sometimes you want to declare the shape of a plain dict — which keys it has, what type each holds — without turning it into a full validated object. That is TypedDict: a way to describe a dictionary's expected keys and value types, for checkers and readers, with zero runtime behaviour.

state.py — the LangGraph pattern, previewed
from typing import TypedDict

class AgentState(TypedDict):
    topic: str
    draft: str
    revision_count: int

state: AgentState = {"topic": "GST filing", "draft": "", "revision_count": 0}
# At runtime this is a PLAIN dict. No validation, no conversion, no methods.
# The class exists purely so humans and checkers know the shape.

Note the difference in temperament. A BaseModel is the customs officer: instantiating it validates, converts, and can reject. A TypedDict is only the declaration form: at runtime it is an ordinary dict and nothing is checked. LangGraph — the graph framework you will spend Act VI inside — asks you to define each graph's shared state exactly like AgentState above, and every node in the graph reads and updates that one described dict. When you see class State(TypedDict): at the top of a graph file, you now know precisely what it is: a labelled shape, not a guarded object. Consider this a preview; the full tour comes with StateGraph later in the course.

In the wild

Open any agent repository and this chapter is everywhere. Tool files are rows of hinted functions, often with a class ...Args(BaseModel) beside each one. LangGraph projects open with a class State(TypedDict). FastAPI services — the most common way agents are wrapped for production — validate every request body with Pydantic models. In merge requests, "add type hints" and "the schema drifted from the model" are everyday comments, and a log line saying ValidationError: 1 validation error for InvoiceArgs — amount: Input should be greater than 0 is the customs officer doing her job: the LLM produced a bad argument and it was stopped at the border, before your code or your database ever saw it.

Check yourself

1. You call shout(42) where the function is declared def shout(name: str) -> str. What does Python do?

Hints are labels, not locks. Python stores them in __annotations__ and otherwise ignores them at runtime — enforcement is the job of editors, static checkers like mypy or pyright, and (at data borders) Pydantic.

2. A Pydantic model declares amount: int and you create it with amount="4500". What happens?

Coercion is a core Pydantic behaviour: border data arrives as strings, so convertible values are converted to match the declared type. Something unconvertible — amount="a lot" — would raise a ValidationError instead.

3. When an LLM decides how to call your tool, what does it actually read?

The model never sees your code. Frameworks read the function's hints (via __annotations__) and build a JSON schema; that schema plus the descriptions is the model's entire briefing — which is why hints and descriptions are prompt engineering, not decoration.

4. What does calling .model_dump() on a validated Pydantic object give you?

.model_dump() converts the object back into an ordinary dict — post-validation, post-coercion — ready to log, store, or serialise. The schema comes from .model_json_schema(); the raw input is gone, replaced by the cleaned values.

5. At runtime, what does a TypedDict such as LangGraph's class State(TypedDict) actually do?

TypedDict is pure description: no validation, no conversion, no methods. It exists so humans, editors, and static checkers know which keys the dict carries — which is exactly the lightweight contract LangGraph wants for graph state.

Key takeaways

  • Annotation grammar is tiny: name: type for parameters and variables, -> type for returns, brackets read as "of" (list[str], dict[str, int]), the bar reads as "or" (str | None, the modern spelling of Optional[str]).
  • Hints are labels, not locks: Python stores them in __annotations__ and ignores them at runtime. Their power is documentation, editor support, and static checkers like mypy and pyright.
  • Pydantic turns those labels into law at data borders: instantiating a BaseModel validates every field, coerces safe conversions ("4500" to 4500), rejects bad data with a complete error report, and .model_dump() returns the cleaned dict.
  • Field(description=...) strings end up in the model's JSON schema — they are the sentences an LLM reads when calling your tool, so write them like instructions, not comments.
  • Agent tool schemas come from hints: frameworks (MCP's FastMCP, LangChain's tool decorator) read __annotations__ plus the docstring to build the schema the model sees, then validate the model's JSON reply against it before your code runs.
  • TypedDict declares a dict's shape with zero runtime behaviour — the pattern LangGraph uses for graph state, coming up in its own act.
