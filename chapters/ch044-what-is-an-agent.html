What is an agent, actually?

An agent is an LLM, a set of tools, and a loop — nothing more. This chapter pins the definition down, walks the spectrum from a single call to full autonomy, and shows you why everything else on the box is packaging.

~25 minChapter 45 of 117
The story

Three ways to get dinner on the table. One: the vending machine in the lobby. You press B4, a packet drops. One request, one fixed answer, no judgement involved. Two: the tiffin service. Every morning a pipeline swings into motion — vegetables chopped, dabbas packed, the courier follows the same route past your building at the same time. The steps never vary; somebody designed them once and the system repeats them faithfully, whatever is in season.

Three: your guests call to say they land at eight, and you tell your house manager one sentence — "Dinner for six, one of them eats Jain." She opens the fridge and takes stock, phones the vegetable vendor, tastes the dal, adjusts the salt, sends the driver out for fresh curd, and stops when the table is ready. You never told her the steps. You told her the goal.

All three put food in front of you. They differ in exactly one thing: who decides the next step. A locked answer, a scripted pipeline, or a mind with hands deciding as it goes. That single question is this whole chapter — and the third one is an agent.

One line, three ingredients

Here is the definition this entire Act rests on. An agent is an LLM, given tools, run in a loop until the task is done — a model that can decide its own next step, act on the world through functions your code is willing to run, and keep going without you between steps. Say it shorter: LLM + tools + loop. There is no fourth ingredient. Every product with "agent" on the label — every framework, every harness, every autonomous coding assistant you have ever supervised — is those three things wearing different clothes.

The good news: you have already met two of the three. In your first API call you sent messages to a model and got text back — that is the LLM ingredient, and it never gets more mysterious than that. In the tool-calling chapter you saw that a model cannot run anything itself; it can only describe a function call, which your code then executes and reports back on. That is the tools ingredient. The only genuinely new thing in this Act is the loop — and a loop is a for statement you have been reading since control flow.

The tiny chat CLI you dissected in the previous chapter is the perfect bridge. That program already had an LLM call inside a loop — but look at who powered that loop. Every single iteration waited for a human to type something. The model answered, then sat idle until you pushed it again. You were the engine. An agent makes exactly two upgrades to that CLI: it gives the model hands (tools), and it lets the model's own output — not your keyboard — decide whether the loop spins again. Take the human out of the crank position, and chat becomes agency.

Note

The word agentic gets stretched by marketing until it covers almost anything with an LLM inside. When you hear it, quietly run the test from the story: who decides the next step? If the answer is "a developer, in advance, in code", you are looking at a pipeline with good PR — not an agent. This is not a put-down; as you will see below, the pipeline is often the better engineering choice.

The agency spectrum: from vending machine to house manager

"Agent or not" is the wrong question; the useful picture is a spectrum, and the dial being turned is always the same one — how much of the control flow the model owns. Step through the four positions. Watch where the highlight sits in each diagram: that is who is deciding what happens next.

1. A single call — the vending machine

Your codebuilds one prompt
→
LLMone transformation
→
Answer

One request in, one answer out. Summarise this email, classify this ticket, translate this line. The model owns nothing but the words; your code decided everything before the call and does everything after it. A huge share of real LLM value ships in exactly this shape.

2. A chained workflow — the tiffin pipeline

Your codeowns the route
→
LLM call 1extract
→
LLM call 2draft
→
LLM call 3polish

Several calls, glued by ordinary code: the output of one becomes the input of the next. The steps and their order were written down before the first request ever ran. The model is brilliant labour on a fixed assembly line.

3. A routed workflow — a fork in the pipeline

LLM"this is a refund case"
→
Your codeif refund: branch A
→
Branch Arefund steps
Branch Bcomplaint steps

Now the model's opinion picks which branch runs — but every branch was authored in advance, and code does the actual switching. The model is a signal into your control flow, not the owner of it. Still a workflow.

4. An agent — the house manager

Goal"handle dinner"
→
LLMdecides the next step, every step
→
Toolsfridge, phone, driver
←
Resultsfed back in

The route is gone. The model reads each result and chooses the following action itself, around a loop, until it declares the goal met. Control flow now lives in the model's judgement — that crossing, and nothing else, is what makes it an agent.

5. Where the systems you operate sit

Agent + harnesscoding assistants, research agents

An autonomous coding assistant such as Claude Code sits at the far right: a model choosing its own next step, a large toolbox, a loop — wrapped in a harness of permission prompts and budgets precisely because that position on the dial is the least predictable.

Anthropic's engineering guide on building effective agents draws this same boundary and it is worth memorising: a workflow is a system where LLM calls are orchestrated along code paths defined in advance, while an agent is a system where the model directs its own process and tool use as it goes. Same model, same API — the difference is entirely about which side owns the steering wheel.

Everything else is packaging

Now the claim in this chapter's title earns its keep. Take any feature from any agent product's launch page, and it will land on one of the three primitives — an upgrade to the LLM call, an upgrade to the tools, or an upgrade to the loop and the history it carries. There is nowhere else for it to land, because there is nothing else in the machine.

Feature on the boxWhich primitive it upgrades
Streaming output, model routing, retriesPackaging around the LLM call
MCP servers and connectorsPackaging around the tools — one standard plug shape so any tool fits any agent
Memory, sessions, checkpointingPackaging around the history the loop carries — saving it, trimming it, restoring it
Planning and to-do listsA tool the model uses to leave notes for its future turns
SubagentsA tool whose implementation happens to be another whole loop
Permission prompts, sandboxes, guardrailsGates at the tool boundary — the one place where anything real happens
Turn limits, cost budgets, timeoutsBrakes bolted onto the loop

This is why the Act ahead will not intimidate you. When you meet LangChain's create_agent, you will recognise the loop with structured tool calling and middleware attached. When you meet deep agents, you will recognise the loop plus a planning tool, a file system, and subagents. The next time a vendor deck says "agentic AI platform", you can play the placing game live in the meeting: LLM, tool, or loop? Every bullet point lands somewhere on that list, and knowing where is the difference between evaluating a product and admiring its fonts.

When not to build an agent

Here is the part the launch pages skip, and the part Anthropic's own guidance leads with: find the simplest solution that meets the need, and only add complexity when it demonstrably pays for itself. An agent is the most flexible thing on the spectrum — and also the slowest, the most expensive, and the least predictable, because every run may take a different route. Those are real engineering costs, paid on every single execution. Autonomy is a price you pay for flexibility, not a badge of sophistication.

Three questions, asked in order, settle most designs:

  • Can one well-crafted prompt do it? Perhaps with some relevant documents pasted in? Then stop there. A shocking fraction of "we need an agent" requests dissolve under this question alone.
  • Can you write the steps as a numbered list before you start? Extract the total from every invoice, convert it, append a row — if the route is knowable in advance, build a workflow. It will be faster, cheaper, testable, and it will fail in ways you can reproduce.
  • Does the route genuinely depend on what each step reveals? Debugging a failing build, researching an unfamiliar market, editing code across a dozen files — only here, where no human can script the path beforehand, does an agent earn its costs.
Careful

The most common failure in this field is not a broken agent — it is a perfectly healthy agent doing a job a fifty-line workflow would have done better. If a reviewer asks "does this need to be agentic?" they are not being cynical; they are asking the simplicity question. Chapter 50 is devoted entirely to this decision — when workflows beat agents, with worked examples — so treat this section as the trailer.

You have been operating this definition for months

One more pass, and this one is personal. You run production agents every day, so every abstract term in this chapter is something you have already watched happen at your keyboard — you just may not have had the name for it. Map your own experience onto the machine:

What you see as an operatorWhat it is underneath
The model narrates "I'll check the config file next"The reasoning half of one loop turn — the model choosing its next step out loud
A permission prompt: allow this command?The loop pausing at the tool boundary. Nothing has run yet; the model only wrote a request, and your side decides whether to execute it
The context meter draining on a long runThe history growing — the whole transcript is re-sent to the model on every turn (tokens and context)
A run stopping with "maximum turns reached"The loop's safety cap firing, exactly like range(1, 4) running out in your pyrun above
The instructions file the assistant reads at start-upThe system prompt — a standing briefing that shapes every turn without appearing in any of them

Read that table twice, because it is the quiet thesis of this Act: you are not learning a foreign machine, you are learning the names of parts you already supervise. When the model asked your permission to run a command, you were standing at the tool boundary — the single most important line in agent security, as you will see when you build tools yourself. When you watched an agent stall and repeat itself, you were watching a loop whose observation never made it back into history. The operator's instincts you already have are the mechanic's instincts; the rest of Act V just hands you the spanner.

In the wild

You will meet this chapter's one question — who decides the next step? — everywhere. In design docs and merge requests it appears as "let's keep this a workflow" or "does this need to be agentic?". In LangGraph code, a graph whose edges are all fixed is a workflow, and a graph with a model-driven conditional edge looping back on itself is an agent wearing graph clothing. In incident reviews, "the agent burned its budget re-running the same search" is a loop-and-observation bug you can now diagnose from the definition alone. And in vendor evaluations, the placing game — LLM, tool, or loop? — will tell you in ten minutes whether a platform contains anything you could not assemble from the three primitives yourself.

Check yourself

1. What is the single distinction that makes a system an agent rather than a workflow?

Chains and routers make many calls with strong models and are still workflows, because the route was authored beforehand. The moment control flow moves into the model's run-time judgement, you have an agent.

2. Your task: "for every new invoice, extract the total, convert it to rupees, and append a row to a spreadsheet." What should you build?

You just wrote the steps as a numbered list, which is the simplicity rule's second question answering itself. A fixed pipeline here is cheaper, faster, and testable; an agent would add cost and unpredictability for zero extra flexibility.

3. In the LLM + tools + loop picture, where does MCP belong?

MCP standardises how tools are described and connected — plumbing for the tools ingredient. It does not change what a model is or how the loop runs, which is exactly why it fits the "everything else is packaging" claim.

4. The tiny chat CLI from the previous chapter had a loop with an LLM call inside. Why wasn't it an agent?

The CLI's loop waited for your keyboard on every turn, and the model could only ever produce words. Give the model hands (tools) and let its own output decide whether the loop continues, and the same skeleton becomes an agent.

5. What does Anthropic's simplicity rule advise?

Agents trade cost, latency, and predictability for flexibility. That trade is worth it for open-ended work like debugging or research — and a waste for tasks whose steps you can write down beforehand.

Key takeaways

  • An agent is an LLM plus tools plus a loop — the model decides its own next step, acts through functions your code runs, and keeps going until the task is done. There is no fourth ingredient.
  • Agency is a spectrum, not a switch: single call, chained workflow, routed workflow, agent. The dial being turned is always who owns the control flow — your code in advance, or the model at run time.
  • Every feature on every agent product maps onto one of the three primitives: memory upgrades the history, MCP upgrades tool plumbing, subagents are tools containing loops, guardrails are gates at the tool boundary.
  • Follow the simplicity rule: one prompt if possible, a workflow if the steps are knowable in advance, an agent only when the route depends on what each step reveals. Autonomy is a cost, not a badge.
  • Your operator experience already contains this whole chapter: permission prompts are the tool boundary, the context meter is the history, "max turns reached" is the loop's brake.
  • Next chapter you build the loop itself, by hand, in about fifty lines — after which no framework will ever look like magic again.
