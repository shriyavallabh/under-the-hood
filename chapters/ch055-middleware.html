Middleware: wrapping the loop

One loop cannot please everyone — so LangChain drilled six standard hook points into its walls. Before-model, after-model, around every tool call: this is the extension mechanism that turned a simple loop into a platform.

~30 minChapter 56 of 123
The story

A new managing director takes over a fast-growing firm. She is superb at decisions — but as the company scales, the board never rewires her. Instead, they build a front office around her door. Inbound, a clerk condenses the towering morning file into a two-page brief before it reaches her desk. Her phone line gets a switchboard that silently redials whenever a call drops mid-sentence. Outbound, a proofreader scans every letter leaving the office and blacks out account numbers before they hit the post. And when she orders something irreversible — "wire the deposit today" — the instruction pauses at a sign-off desk until a human partner initials it.

Notice what never changed: the MD, the door, the rhythm of her day. Every new safeguard was a layer wrapped around the same office, added or removed without touching the person inside. That front office is middleware. The MD is the model, her daily rhythm is the agent loop you built in chapter 46, and each desk — the clerk, the switchboard, the proofreader, the sign-off ledger — is a hook you are about to learn by name.

One loop, and the problem with perfect packaging

Quick recap of where you stand. In Build a ReAct loop by hand you wrote the whole agent yourself: a history list, a model call, an ACTION: parser, a tool dictionary, a turn cap. In create_agent: the packaged loop you watched LangChain hand you that identical loop as a single factory call — model in, tools in, agent out. Tidy, tested, production-ready. And sealed.

Sealed is the problem. The moment a team runs a packaged loop for real, requests arrive: log every model call for the audit trail; trim the history before it blows the context window; make the flaky search tool retry; never, ever let the invoice tool fire without a human's approval. Notice that none of these change what the agent does — they change what happens around each step. Engineers call this kind of requirement a cross-cutting concern: a behaviour that has to touch every pass through the loop rather than living in any one tool or prompt.

Old LangChain answered with a zoo — dozens of pre-built agent variants, and if none fit, you subclassed or forked the loop and maintained your fork forever. LangChain 1.x answers differently: there is one loop, and its walls have standard, documented openings. Middleware is a piece of code that runs at one of those fixed stations inside somebody else's loop — the loop keeps the wheel, your code rides along at agreed points. A hook is the station itself: a named moment (just before the model is called, just after, around a tool run) where the loop pauses and lets your middleware speak.

Note

If you have configured a web server, you have met this word before — Express and Django both call their request-wrapping layers middleware. Same idea, different loop: theirs wraps an HTTP request's journey, LangChain's wraps an agent turn.

The six hooks, on one card

That is the entire surface area. Six named stations, two flavours: the before/after hooks observe and edit the state at a moment in time, while the wrap hooks hold an operation in their hands and decide whether — and how — it proceeds.

HookFiresTypical residents
before_agentOnce, when the run startsLoad user context, start timers
before_modelEvery turn, before the model callLogging, call limits, history summarisation
wrap_model_callAround the model call itselfRetries, model fallback, request rewriting
after_modelEvery turn, after the reply landsOutput logging, redaction, guardrails
wrap_tool_callAround each tool executionHuman approval, tool retries, error shaping
after_agentOnce, when the run endsFlush logs, final accounting

Two mechanics to hold on to. First, the before/after hooks receive the agent's state — the dict carrying the message history — and communicate by returning a dict: return None to change nothing, or return {"model_call_count": 3} and the loop merges it into the state. Some hooks can even return a jump instruction such as {"jump_to": "end"} to end the run early — that is how a call-limit middleware pulls the emergency brake. Second, the wrap hooks receive a handler — a function representing "the rest of the journey inward". Calling it means proceed; not calling it means the wrapped operation never happens. Retry logic is just calling the handler again inside a loop.

One more picture before code: when several middleware are installed, they nest like an onion. With middleware=[A, B], the loop runs A's before_model, then B's; the model speaks; then B's after_model, then A's. First in the list is the outermost skin — last to see the request going in is first to see the response coming out. Keep that image; the runnable demo below prints it in front of you.

Write one: a logging middleware, for real

Here is the smallest useful middleware in real LangChain 1.x — the audit clerk. You subclass AgentMiddleware, override only the hooks you care about, and hand an instance to create_agent. These signatures are verified against the current docs; if your installed version differs slightly, the stable part is the shape — state in, optional dict out.

logging_middleware.py
from langchain.agents import create_agent
from langchain.agents.middleware import AgentMiddleware, AgentState
from langgraph.runtime import Runtime
from typing import Any

class LoggingMiddleware(AgentMiddleware):
    def before_model(self, state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
        print(f"model call coming: {len(state['messages'])} messages so far")
        return None  # observe only, change nothing

    def after_model(self, state: AgentState, runtime: Runtime) -> dict[str, Any] | None:
        last = state["messages"][-1]
        print(f"model replied: {str(last.content)[:80]}")
        return None

agent = create_agent(
    model="claude-sonnet-4-5",
    tools=[lookup_invoice, send_reminder],
    middleware=[LoggingMiddleware()],
)

For a single hook there is an even lighter form: a decorator — remember from chapter 17, a function that wraps another function to extend it. LangChain ships one per hook, and the decorated function goes straight into the middleware list. Here is the switchboard — a retry, built on wrap_model_call and its handler:

retry_middleware.py
from langchain.agents.middleware import wrap_model_call, ModelRequest, ModelResponse
from typing import Callable

@wrap_model_call
def retry_model(
    request: ModelRequest,
    handler: Callable[[ModelRequest], ModelResponse],
) -> ModelResponse:
    for attempt in range(3):
        try:
            return handler(request)   # "proceed" — make the actual call
        except Exception as err:
            if attempt == 2:
                raise                  # three strikes: let it fail loudly
            print(f"retry {attempt + 1} after: {err}")

agent = create_agent(model="gpt-5.5", tools=[...], middleware=[retry_model])

Look at how much power that little handler carries. Call it once: normal operation. Call it in a loop: retries. Catch its exception and call it with a different request: model fallback. Never call it at all: the model call is vetoed. Every built-in you are about to meet is one of these moves, packaged.

The built-ins: an operations toolbox

Because the hook points are standard, LangChain ships ready-made middleware for the problems every agent operator eventually hits. This is the part of the chapter you will use next week — each of these replaces a behaviour you may have hand-rolled or wished for:

production_agent.py — a realistic middleware stack
from langchain.agents import create_agent
from langchain.agents.middleware import (
    SummarizationMiddleware,
    HumanInTheLoopMiddleware,
    ModelRetryMiddleware,
    ModelFallbackMiddleware,
    PIIMiddleware,
)
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model="claude-sonnet-4-5",
    tools=[lookup_invoice, send_reminder],
    checkpointer=InMemorySaver(),        # human-in-the-loop needs pause/resume
    middleware=[
        SummarizationMiddleware(
            model="gpt-5.4-mini",        # a cheap model does the summarising
            trigger=("tokens", 4000),    # act when history passes 4,000 tokens
            keep=("messages", 20),       # always preserve the last 20 messages
        ),
        HumanInTheLoopMiddleware(
            interrupt_on={
                "send_reminder": {"allowed_decisions": ["approve", "edit", "reject"]},
                "lookup_invoice": False,   # reads are safe, no gate
            },
        ),
        ModelRetryMiddleware(max_retries=3, backoff_factor=2.0, initial_delay=1.0),
        ModelFallbackMiddleware("gpt-5.5"),  # if the primary model is down
        PIIMiddleware("email", strategy="redact"),
    ],
)

Note

The roster keeps growing — content-moderation and guardrail layers, context editing, tool selection, planning to-do lists and more ship alongside the ones above. Before you write a middleware, skim the built-in middleware page of the current docs: the layer you need may already exist, tested and maintained by someone else.

One subtlety worth a sentence: HumanInTheLoopMiddleware requires a checkpointer — a save-file mechanism for the run's state, which you will meet properly in the LangGraph act. The reason is physical: an approval might take a human four hours, and the process cannot just hold its breath. The middleware pauses the run, the checkpointer preserves it, and the run resumes at the exact same station when the decision arrives. Our old friend max_turns from chapter 46 has grown up into the call-limit middleware in that table, too — same guard, now composable. And when things still go wrong, the failure signatures you learned in chapter 49 are exactly what these layers exist to catch.

Why this turned a loop into a platform

Step back and ask what actually changed between the fifty-line loop and this chapter. Not the loop — it still thinks, acts, observes. What changed is who can extend it without breaking it.

Before hooks existed, every team's special requirement meant surgery on shared code: fork the loop, patch the fork, drift from upstream, repeat at every release. LangChain's pre-1.0 era showed the failure mode at scale — a catalogue of near-identical agent classes, each one somebody's fork institutionalised. Version 1.x inverted the design: one loop, deliberately small, plus six sanctioned openings. Variation moved out of the engine and into the layers.

That inversion is what the word platform means. An application does what its authors imagined; a platform does what its users imagine, within rails the authors guaranteed. Browsers became platforms through extensions; editors through plugins; web servers through — the ancestors here — request middleware. The rails matter as much as the freedom: because every middleware speaks the same six-hook contract, they compose. Your logging layer stacks under someone else's retry layer, under LangChain's own summariser, and none of them have ever heard of each other. Ordering is the one design decision left to you, and the onion rule makes even that predictable.

It also cleanly splits jobs. Framework authors maintain the loop. Platform teams — people who do what you do — maintain the policy: which gates, which limits, which redactions, in which order. That policy is exactly the middleware=[...] list, which makes it readable, reviewable, and diffable in a merge request. When you read LangChain's own source in the next chapter, you will find the loop startlingly small — because everything interesting has moved into the layers you now know how to write.

In the wild

In any production LangChain 1.x repo, find the create_agent call and read its middleware=[...] list first — it is the team's operating policy in six lines: what gets logged, what needs sign-off, what retries, what gets redacted. Merge-request comments like "add PII redaction before launch", "put HITL on the delete tool", or "the retry middleware masked the outage for two hours" are all conversations about this one list. And enterprise harness teams building on other stacks copy the identical pattern — pre-model hooks, post-model hooks, tool-call wrappers — because the shape, not the brand name, is the idea.

Check yourself

1. SummarizationMiddleware compresses a long history so the model call stays affordable. Which hook must it ride on?

The whole point is that the model receives a trimmed transcript instead of the full one — so the trimming has to happen at the station just before the call, which is before_model.

2. Inside wrap_tool_call, your middleware receives the tool request and a handler. What happens if you return a message without ever calling the handler?

The handler IS the onward journey — skipping it vetoes the operation. That is precisely how approval gates reject a tool call: they return a refusal message, the tool stays untouched, and the model reads the refusal as its observation.

3. You install middleware=[Auditor(), Redactor()]. In what order do their hooks fire around one model call?

The onion rule: first in the list is the outermost layer. Before-hooks run top-to-bottom, the model speaks, then after-hooks unwind bottom-to-top — so the Auditor sees both the very first inbound moment and the very last outbound one.

4. Why does HumanInTheLoopMiddleware insist on a checkpointer?

A human decision arrives on human time. The process cannot hold everything in memory while it waits, so the middleware interrupts the run and the checkpointer preserves the state — messages, pending tool call and all — until the approve/edit/reject decision resumes it.

5. What is the strongest version of the claim "middleware turned LangChain agents into a platform"?

Speed is not the point and tools are untouched. The shift is architectural: one small guaranteed loop plus six standard openings means teams — and the community — extend behaviour by stacking layers instead of maintaining private forks. Users, not just authors, decide what the system can do.

Key takeaways

  • Middleware is code that runs at fixed stations inside the agent loop — the front office wrapped around the model's door, added or removed without touching the loop itself.
  • Six hooks cover the surface: before_agent / after_agent once per run, before_model / after_model every turn, and wrap_model_call / wrap_tool_call holding an operation with a handler that means "proceed".
  • Before/after hooks talk back by returning a state-update dict (or None); wrap hooks gain veto, retry, and rewrite powers by choosing how — or whether — to call the handler.
  • Stacked middleware nest like an onion: first in the middleware=[...] list is outermost, so before-hooks run top-to-bottom and after-hooks unwind in reverse.
  • The built-ins are an operations toolbox — summarisation, human-in-the-loop approval (with a checkpointer so runs can pause), model and tool retries, fallbacks, PII redaction and moderation-style guardrails, call limits.
  • This is why LangChain became a platform: one small guaranteed loop plus standard openings turned every cross-cutting need into a composable layer instead of a fork — and the middleware list became the team's reviewable policy document.
