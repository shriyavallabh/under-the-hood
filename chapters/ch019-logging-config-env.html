Logging, config, and env vars

The unglamorous plumbing that fills half of every real merge request — loggers instead of print, settings that layer neatly, and secrets that appear in code only by name, never by value.

~30 minChapter 19 of 133
The story

A hotel chain opens a new branch. Head office ships every branch the same operations manual — breakfast at seven, checkout at eleven. Each branch keeps a local addendum binder on top of it: our breakfast starts at six thirty, because the airport crowd leaves early. And on any given morning, the duty manager can trump both with a note taped to the front desk: pool closed today. Three layers. The closer a rule sits to today, the more it wins.

Behind that desk sits the logbook. Every entry gets a timestamp, the writer's name, and a severity — routine note, guest complaint, fire alarm. Nobody rips out pages, and when something goes wrong at 2 a.m., the logbook is the only witness that was actually there.

And the safe? Its combination appears nowhere in any manual. The manual says only: "the combination is in the sealed envelope held by the duty manager." A reference by name — never the number itself.

That hotel is every production Python service you have ever operated. This chapter is the logbook, the three layers of rules, and the sealed envelope.

Why print() doesn't survive the night shift

print() writes text to the screen, and that is its whole feature — plenty for a script at your desk, but a service running unattended needs more, and print fails it four ways at once. A print line carries no timestamp, so you cannot tell whether it happened at 2 a.m. or 2 p.m. It carries no severity, so a routine "loaded 3 invoices" looks exactly as urgent as "payment gateway unreachable". It carries no origin, so you cannot tell which of your forty modules said it. And it cannot be turned down: the only way to silence a noisy print is to edit the code and redeploy.

The logging module — Python's built-in logbook — fixes all four. Every message you log gets a timestamp, a severity level, and the name of the module that wrote it, and how much of it actually appears is decided by configuration, not code. That last part is the superpower: when your agent misbehaves in production, you flip one setting from INFO to DEBUG, the logs turn verbose, you find the bug, and you flip it back. No redeploy.

The levels are a threshold, not a menu. When a program is configured at INFO, it shows everything at INFO and above — INFO, WARNING, ERROR, CRITICAL — and hides DEBUG. Configure at WARNING and INFO disappears too. One dial, and the noise level of the whole system moves together.

Three lines to a real logger

Using the logging module properly takes exactly two habits. First, at the top of every module, create that module's named logger:

any module in the project
import logging

log = logging.getLogger(__name__)

getLogger(name) returns the logger with that name, creating it the first time. Passing __name__ — the module's own dotted path, like invoice_bot.gst — means every message is automatically stamped with where it came from, and configuration can later target one module ("turn only the GST module up to DEBUG") because logger names form a hierarchy along the dots.

Second, once, at the program's entry point, configure what happens to all those messages. The simplest tool is basicConfig — a one-call setup that picks the minimum level and the line format:

main.py — the one place configuration happens
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s %(levelname)-8s %(name)s - %(message)s",
)

There is a classic trap hiding here, and it produces the single most-asked logging question on the internet: "why does my logging.info show nothing?" If nobody calls basicConfig, the default threshold is WARNING — so INFO and DEBUG messages are silently dropped. The program is fine; the dial was just left at its factory position.

Note

The etiquette that keeps large codebases sane: applications configure logging, libraries never do. A library just calls getLogger(__name__) and writes messages; whoever runs the final program decides the level, format, and destination. If a library called basicConfig itself, it would be a houseguest rearranging your furniture.

Enough theory — run it. The "model" here is your eyes: watch which lines survive the INFO threshold, then change logging.INFO to logging.DEBUG and run again.

Tip

Try two more experiments. Set the level to logging.ERROR and watch three of the four lines vanish. Then change the format string — delete %(asctime)s, or add %(lineno)d to stamp each message with its line number. The messages in the code never changed; only configuration did. That separation is the whole point.

Reading a log line like a local

Half of operating any production system is reading its logs under pressure. Here is a realistic five-line excerpt from an agent service. Click each line — every one has a story:

app.log — ninety seconds in the life of a production agent
2026-07-02 09:14:03,118 INFO invoice_bot.api - request received: summarise INV-042
Anatomy, left to right: timestamp (the comma introduces milliseconds — 118 of them), level, logger name, message. The name invoice_bot.api tells you which module spoke without opening any code. This line is the system saying "all normal".
2026-07-02 09:14:03,904 INFO invoice_bot.llm - model call 1 finished in 0.78s, 812 tokens
Good INFO messages carry numbers you will want later — durations, counts, token totals. When someone asks "did the agent get slower this week?", lines like this one are the evidence.
2026-07-02 09:14:04,101 WARNING invoice_bot.gst - GST number missing on line 2 - using registry lookup
A WARNING means the code hit something odd and handled it — here, a fallback path. Nothing failed yet. But when an ERROR appears seconds later, the preceding WARNINGs are usually the trail of breadcrumbs that explains it.
2026-07-02 09:14:09,566 ERROR invoice_bot.gst - registry lookup failed after 3 retries
The fallback itself failed — an operation was actually lost. Note the five-second gap in the timestamps: that is the three retries happening. You just diagnosed a timing problem from timestamps alone. This is what "reading logs" means.
Traceback (most recent call last):
When code logs an exception with log.exception("..."), the full traceback — the same one you learned to read in earlier chapters — is written into the log below the message. Multi-line entries like this almost always mark the exact spot the incident report will quote.
ConnectionError: registry.gst.gov.in timed out

Config: where a setting's value really comes from

Now the second half of the plumbing. Every real service has knobs — which model to call, how long to wait, how loud to log. The amateur move is to write those values straight into the code; then every change is a code change. Real projects instead resolve each setting through layers, exactly like the hotel: manual, binder, sticky note. In software the layers are, from weakest to strongest: defaults in code < a settings file < environment variables. Step through one setting — log_level — finding its value:

The sticky note on the desk. An environment variable set on this one machine, for this one run — an engineer is debugging a live incident and cranked the logs up. Environment beats file beats code.

4. Resolved: DEBUG wins

log_level = "DEBUG"source: environment variable

Why this order? Distance. The default was written months ago for everyone; the file was written for this deployment; the env var was set minutes ago, on this exact machine, by the person standing closest to the problem. The nearest layer wins — and when the incident ends, deleting the variable restores everything, with zero code changed.

So what is an environment variable? It is a named piece of text that the operating system hands to a program at launch — settings that live outside the program, attached to the machine or the session instead. Python exposes them as os.environ, which behaves like a dictionary (the same key-lookup you know from the collections chapter):

reading the environment
import os

os.environ["HOME"]                        # KeyError if missing
os.environ.get("APP_LOG_LEVEL", "INFO")   # default if missing — usually what you want
Careful

Environment variables are always strings. TIMEOUT=30 arrives as the text "30", not the number 30, and DEBUG=false arrives as "false" — which, being a non-empty string, counts as true in an if test. Whole outages have been caused by that one. Hold the thought; the fix is two sections away.

The .env file: environment variables you can keep

Typing environment variables into a terminal works, but they evaporate when the window closes. So development settled on a convention: a plain-text file named .env in the project folder, one NAME=value per line, which is loaded into the environment when the program starts. It is the sticky note that survives a restart:

.env — local development only, and ALWAYS in .gitignore
# Settings for my machine
APP_LOG_LEVEL=DEBUG
DATABASE_URL=postgresql://localhost:5432/invoices

# Secrets — values exist ONLY here, never in code
OPENAI_API_KEY=sk-...my-real-key...

The most popular loader is the small python-dotenv package: call load_dotenv() at startup and every line lands in os.environ. Under the hood it simply opens the file with the same with block you met last chapter and parses each line — and it has one thoughtful default: it will not overwrite a variable the real environment already set. The layering survives: real environment beats .env beats everything below.

Two rituals travel with every .env file. First, it is listed in .gitignore, always, because it contains secrets. Second, the project commits a sibling called .env.example containing only the names with the values blanked out — so a new teammate knows which envelopes to fill, without ever seeing what was inside yours.

pydantic-settings: the layering, with a seatbelt

Writing that resolve() function by hand was good for the soul, but nobody does it in production, because a library already does it better. In the type hints and Pydantic chapter you met Pydantic — the library that checks data against declared types the moment it arrives. Its sibling package, pydantic-settings, applies the same idea to configuration: you declare your settings as a typed class, and it fills the values in from the layers — field defaults, then a .env file, then real environment variables, strongest last — and validates every one of them. Click through the file; this exact shape appears in thousands of repositories:

settings.py — the whole configuration story in one class
from pydantic_settings import BaseSettings, SettingsConfigDict
A separate package (pydantic-settings) built on Pydantic. BaseSettings is the class that knows how to read the environment; you inherit from it, just like the models in the Pydantic chapter.
class Settings(BaseSettings):
One class, usually in one file, is the single source of truth for every knob the service has. When a reviewer wants to know "what can we configure?", they open this file and nothing else.
model_config = SettingsConfigDict(env_file=".env")
Wires in the .env file so local development just works. Real environment variables still win over the file's values — the layering you just built by hand, preserved.
app_name: str = "invoice-bot"
A default in code — the weakest layer. Runs fine with no configuration at all.
log_level: str = "INFO"
Set the environment variable LOG_LEVEL=DEBUG (matching is case-insensitive by default) and this field silently becomes "DEBUG". Nobody edits code to change the volume.
timeout_s: int = 30
The seatbelt. The environment can only supply strings, but the type hint says int — so pydantic-settings converts "30" to 30 for you, and if someone sets TIMEOUT_S=banana, the program refuses to start with a clear error. The stringly-typed trap from the warning above: solved.
openai_api_key: str
No default — this field is required, so it MUST come from the environment or the .env file. And look at what is in the repo: only the NAME. The value exists solely in the environment. This one line is the sealed-envelope rule, enforced by a type checker.
settings = Settings()
Constructing the class is the moment all layers get read, merged, converted, and validated. If anything required is missing or malformed, the crash happens HERE — at startup, loudly — not three hours into a run when the key is first used. Engineers call this failing fast, and it is a kindness to your future self.

Secrets: named, never spelled

One rule in this chapter is absolute. A secret — an API key, a database password, a token — appears in code and config files by name only. The code says os.environ["OPENAI_API_KEY"] or declares openai_api_key: str; the actual value is injected from outside — a gitignored .env on a laptop, the deployment platform's secret store or a vault in production. The manual names the envelope; the manual never contains the combination.

Why so strict? Because code travels — committed, pushed, forked, pasted into tickets, reviewed by people who should never hold production keys. And git makes it worse: even after you delete the line, the key lives on in the repository's history, retrievable by anyone with a checkout. Automated bots scan public repositories around the clock and can find a freshly pushed key within minutes. The only real remedy for a committed secret is to rotate it — revoke the old key, issue a new one — not to delete the line and hope.

Tip

This gives you a reviewer's reflex you can apply today, before you write much Python: scan any diff for strings that look like credentials — sk-, token, password, long random-looking text in quotes. If a value is present rather than a name, that is a change-request comment, every single time, no matter how senior the author.

Half the lines of every merge request

Here is the payoff for operators. Open almost any merge request in an enterprise Python repository and count what the diff actually contains. The headline logic is maybe a third of it. The rest is this chapter: log = logging.getLogger(__name__) at the top of a new file, log.info(...) lines narrating each step, a log.warning(...) on the fallback path, a new field on the Settings class, a matching name appended to .env.example, and a review thread saying "please add logging around the retry" or "move that URL into settings". None of it is glamorous. All of it is what makes the feature operable at 2 a.m. by someone who did not write it.

In the wild

You will now recognise this plumbing everywhere. Every framework you operate emits through the same named loggers — LangChain, FastAPI, and friends — which is why one basicConfig level change makes a whole agent stack chattier. Deployment platforms have a screen literally titled "Environment variables" — that is os.environ being filled in; CI pipelines have "secrets" or masked variables — the sealed envelopes. And when a colleague says "it works locally but not in staging", the first professional question is the one this chapter taught you: what is different in the environment? Nine times out of ten, a missing variable or a wrong layer is the whole bug.

Check yourself

1. A program calls logging.basicConfig(level=logging.WARNING). Which messages will appear in the output?

The level is a floor, not a filter for one severity: whatever you set, that level and everything more severe gets through, while DEBUG and INFO are dropped. This is also why an unconfigured program (default threshold WARNING) mysteriously "loses" all its log.info messages.

2. The code defaults log_level to "INFO", settings.yaml says WARNING, and the machine has APP_LOG_LEVEL=DEBUG set. What does the running program use, and why?

The layering is defaults < file < environment, and the nearest layer wins because it was set most recently, most specifically, by the person closest to the problem — and can be removed again without touching code or files.

3. A teammate's diff adds the line api_key = "sk-live-8f3a..." to a Python file. What is the correct fix?

Secrets appear in code by name only; values live in the environment, a gitignored .env, or a secret store. And because the value already appeared in a commit, deleting the line is not enough — git history remembers, so the key must be rotated.

4. Why does every module start with log = logging.getLogger(__name__) instead of just using print?

The named logger is what buys filtering, formatting, levels, and per-module control from the outside. Speed is not the issue — operability is: at 2 a.m. you want to turn one module up to DEBUG through configuration, not edit and redeploy code.

5. The environment sets TIMEOUT_S=30. What does plain os.environ["TIMEOUT_S"] hand your program, and what does the pydantic-settings field timeout_s: int hand it?

Environment variables are always strings. pydantic-settings uses your declared types to convert and validate them at startup, so a malformed value crashes loudly at launch instead of misbehaving quietly at runtime.

Key takeaways

  • print is for scripts; logging is for systems. Every logged message carries a timestamp, a severity level, and the name of the module that wrote it — and how much appears is decided by configuration, not code edits.
  • The two habits: log = logging.getLogger(__name__) at the top of every module, one basicConfig (level + format) at the application's entry point. Libraries write messages; only applications configure. Unconfigured, the default threshold is WARNING — INFO and DEBUG vanish.
  • Settings resolve through layers — defaults in code < settings file < environment variables — and the layer nearest to the running machine wins, so behaviour can change per deployment or per incident without touching code.
  • os.environ is a dict-like view of the environment, and every value in it is a string; a gitignored .env file (plus a committed .env.example of names only) makes local environments repeatable.
  • pydantic-settings turns the whole story into one typed Settings class: layers merged, strings converted to real types, required fields enforced — and any problem crashes the program at startup, loudly, instead of at 2 a.m., quietly.
  • Secrets appear in code by NAME only; values are injected from the environment or a secret store. A secret that ever touches git history is compromised — rotate it.
  • This plumbing is half the lines of every real merge request. You can now read that half fluently — next stop, where the packages themselves come from: environments, pip, and pyproject.toml.
