Context managers: the with block

The with block is Python's built-in promise that whatever your code borrows — a file, a database session, an API connection — goes back on the shelf, even when everything inside blows up.

~20 minChapter 18 of 133
The story

Every Saturday, Meera runs the tool-lending library at a community centre — drills, ladders, soldering irons, exactly one of each. The rule is simple: sign a tool out at her desk, sign it back in when you leave. But Meera has seen everything. Borrowers get urgent phone calls and rush off. They crack a tile, panic, and bolt. Once, the fire alarm went off with nine people mid-project. In the early days, tools vanished every single week — not stolen, just never returned, because returning depended on people remembering during their worst moments.

So the centre rebuilt the room with one clever feature: there is exactly one way out, and it passes her desk. However your visit ends — job finished, job abandoned, alarm ringing — you physically cross the desk on the way out, and the drill comes off your shoulder and goes back on the shelf. Nobody has to remember. Nobody has to be having a good day. Returns are built into the architecture of the room.

Python's with block is that room. The desk on the way in is called __enter__, the desk on the way out is __exit__, and the one-door design is the guarantee: whatever happens inside, borrowed things go back.

The problem: borrowing is easy, returning is hard

Your programs constantly borrow things they do not own. The operating system owns file handles and lends you one when you call open(). A database owns a small, fixed pool of connections and lends you one per query. An API provider holds a network socket open on your behalf. Each of these is a resource — something scarce, borrowed from outside your program, that must be given back when you are done.

Borrowing takes one line. The trap is the return, because the return depends on the future good behaviour of code that has not run yet. Here is the naive version:

the careless version
f = open("state.json", encoding="utf-8")
data = json.load(f)     # ...but what if THIS line raises?
f.close()               # ...then this line simply never runs

Recall from the errors chapter: when an exception is raised, Python abandons the rest of the current block on the spot and goes hunting for a handler. If json.load chokes on a corrupt file, execution never reaches f.close(). The file handle is now signed out forever — a resource leak, meaning a borrowed thing that is never returned and quietly stays counted against you.

One leaked handle is harmless. But agents and services are long-running programs that repeat the same operation thousands of times, and leaks accumulate. This is the anatomy of a whole family of production incidents you may already have met from the operator's side: "Too many open files" after a crash-looping log writer, a database refusing queries with "connection pool exhausted", an HTTP client that slowly starves the machine of sockets. These failures share a signature: the code ran perfectly in every demo, because demos are short. Leaks only show up when nobody is watching.

Python's classic first-aid for this is try/finally — the finally block runs no matter how the try block ends:

the careful-but-noisy version
f = open("state.json", encoding="utf-8")
try:
    data = json.load(f)
finally:
    f.close()           # runs on success AND on failure

This is correct. It is also four lines of ceremony around one line of intent, it doubles when you have two resources, and it relies on every developer remembering to write it, every time, forever. Meera tried "please remember to return the tools" as a policy. It did not work for her either. What you want is the one-door room: a way to make the return structural.

The with block: the return desk built into the syntax

Here is the same code written the way you will see it in essentially every real Python codebase:

the idiomatic version
with open("state.json", encoding="utf-8") as f:
    data = json.load(f)
# by this line, f is ALREADY closed — success or crash

Read it aloud and it almost parses as English: "with this file open, as the name f, do the following." The indented body is the borrowing period. The moment execution leaves that body — by any exit whatsoever — Python closes the file. Block finished normally? Closed. json.load raised? Closed, and then the exception continues on its way. The body hit a return or a break? Still closed. There is no path out of a with block that skips the cleanup. That is the entire pitch: borrow it, then put it back — guaranteed.

An object that can be used in a with statement is called a context manager — an object that knows what "on the way in" and "on the way out" mean for itself. A file's version of "on the way out" is close the handle. A database session's is return the connection to the pool. A lock's is release me so other code can proceed. Same door, different desks.

Note

You can borrow several things in one line — with open(src) as fin, open(dst, "w") as fout: — and Python returns them in reverse order on the way out, like nested doors. Since Python 3.10 you may also wrap the managers in parentheses across multiple lines when the list gets long.

Under the hood: __enter__ and __exit__

In the dunder-methods chapter you learned Python's favourite trick: language features are secretly method calls with double-underscore names. len(x) quietly calls x.__len__(); a for loop quietly calls __iter__ and __next__; a + b is a.__add__(b). Whenever Python syntax appears to "just know" how to handle your object, there is a dunder protocol underneath. The with statement is exactly this trick, one more time, with two methods: __enter__ and __exit__.

Here is what a with block actually expands to, line by line. Click each line to unfold it:

what "with open(...) as f:" really does
mgr = open("state.json", encoding="utf-8")
First, the expression after with is evaluated, producing the manager object. For files that object is the file handle itself — file objects are their own context managers. Nothing special has happened yet; this is an ordinary call.
f = mgr.__enter__()
The sign-out desk. Python calls __enter__(), and whatever it returns is bound to the name after as. For a file, __enter__ returns the file itself, so f and mgr are the same object here — but other managers return something different, like a session or a cursor. The as name is always "what __enter__ handed me", nothing more.
try:
Python wraps your entire block body in a try — this is where the guarantee comes from. You never see this try; the compiler writes it for you.
data = json.load(f)
Your body, exactly as you wrote it. It may finish, raise, or return early — all three paths lead to the same place next.
finally:
The one door out of the room. Every possible exit from the body passes through here.
mgr.__exit__(exc_type, exc_value, tb)
The sign-in desk. On a clean exit the three arguments are None, None, None — "nothing went wrong". If the body raised, they carry the exception's class, the exception itself, and the traceback — so the manager can see how the visit ended before it cleans up. For a file, __exit__ calls close() either way.

One subtlety is worth knowing because you will meet it in code review: __exit__ returns a value. If it returns True, Python treats the exception as handled and swallows it — execution continues after the block as if nothing happened. Returning False (or None, which is what a bare method returns) lets the exception continue outward after cleanup. Almost every context manager you write should return False: cleaning up is your job, deciding whether an error matters is your caller's.

Files were the training wheels — sessions and clients are the cargo

with open(...) is the example every tutorial uses, but in the systems you operate, the pattern earns its keep on far more expensive resources. Three you will meet constantly:

the same shape, wrapped around real infrastructure
# An HTTP/API client - sockets and connection pools cleaned up
with httpx.Client() as client:
    r = client.get("https://api.example.com/health")

# A database session (SQLAlchemy 2.x style) - the connection
# goes back to the pool even if the query raises
with Session(engine) as session:
    session.add(order)
    session.commit()

# A lock - released even if the critical section crashes,
# so the rest of the program is never deadlocked
with lock:
    counter += 1

Look at what varies and what does not. The resource changes — socket, pooled connection, lock. The cleanup changes — close, return, release. The shape never changes: one line to borrow, an indented body to use, and an exit that cannot be skipped. This is why experienced reviewers can spot a leak in seconds: any bare something = acquire_thing() without a with (or a very deliberate try/finally) is a question waiting to be asked.

LLM SDKs lean on the same protocol. The Anthropic Python SDK, for instance, streams a response through a context manager — with client.messages.stream(...) as stream: — so the underlying HTTP connection is torn down properly whether the stream completes or your code stops reading halfway. And the asynchronous flavour, async with, is the identical idea built on __aenter__ and __aexit__; you will see it all over async agent code, including MCP client sessions in the Python SDK. When Act V wires up real agents, the with blocks will already read as old friends.

Careful

"On the way out" means whatever the class decided it means — it is not always close. The classic surprise is the standard library's sqlite3: using a connection in a with block commits the transaction on success and rolls back on error, but does not close the connection. When you meet a new context manager, spend ten seconds finding out what its __exit__ actually does. The guarantee is that it runs, not that it reads your mind.

The shortcut: contextlib.contextmanager

Writing a full class with __enter__ and __exit__ is fine for a library, but heavy for the small before/after brackets you want in everyday code. The standard library offers a lighter path, and it reuses something you learned one chapter ago: a decorator — a function that wraps another function to change how it behaves.

a stopwatch bracket in eight lines
from contextlib import contextmanager
import time

@contextmanager
def timed(label):
    start = time.perf_counter()        # the way IN
    try:
        yield                          # the body runs here
    finally:
        ms = (time.perf_counter() - start) * 1000
        print(f"{label}: {ms:.1f} ms") # the way OUT - guaranteed

with timed("load config"):
    do_expensive_thing()

The one new word is yield. You will study it properly later in the course; for now, read it as "pause this function here and let the with-body run; resume me afterwards". Everything before the yield plays the role of __enter__ (and whatever you yield is what as receives). Everything after it plays __exit__. Wrapping the yield in try/finally is what makes the exit half guaranteed even on error — the decorator does the rest, manufacturing a real context manager out of your little function. Notice also what this example proves: context managers are not only about returning resources. Any paired before/after ritual — start a timer and report it, open a spinner and clear it, begin a transaction and commit it — fits the same door.

In the wild

Once you know the shape, you will see it everywhere. In any Python repository, with open(...) appears wherever configs, prompts, or logs are read; with Session(...) or with engine.connect() guards the database layer; with lock: protects shared state; tests use with pytest.raises(ValueError): — a context manager whose exit desk checks that the right error happened; and tempfile.TemporaryDirectory() hands you a scratch folder that deletes itself on the way out. In merge requests, the review comment "this leaks a session — wrap it in a with" is this chapter in one line, and a diff replacing try/finally ceremony with a slim with block is a reviewer-pleasing classic. Grep any framework you rely on for def __exit__ or @contextmanager and you will find its lending desks.

Check yourself

1. What exactly does a with block guarantee?

A with block never prevents errors — it guarantees the exit desk. Whatever path execution takes out of the body, __exit__ runs first, and only then does any exception continue outward.

2. In with Session(engine) as s:, what is s bound to?

The as name receives __enter__'s return value — nothing more. For files it happens to be the same object, but other managers hand back something different, which is why the distinction matters.

3. An exception is raised in the middle of a with body. What happens next?

The compiler wraps the body in a hidden try/finally, so the failure is routed through __exit__ — which receives the exception type, value, and traceback — before resuming its journey. Cleaned up, not covered up.

4. Why is f = open(path) ... f.close() without a with block considered a bug waiting to happen?

The close line only runs if everything before it succeeds. Short scripts get away with it; agents and services that run for weeks do not — that is where "too many open files" and "pool exhausted" incidents come from.

5. What does the @contextmanager decorator from contextlib do?

The decorator manufactures the __enter__/__exit__ machinery from your little function: it runs it up to the yield on the way in, hands the yielded value to as, and resumes it on the way out — with try/finally making the exit half crash-proof.

Key takeaways

  • A resource is anything scarce your program borrows — file handles, database connections, sockets, locks — and every borrow needs a guaranteed return.
  • with open(path) as f: means "borrow it, use it in this indented block, and put it back on any exit whatsoever" — success, exception, or early return.
  • Under the hood it is a dunder protocol: __enter__ signs the resource out (its return value becomes the as name), and __exit__ signs it back in, receiving the exception details if the body crashed.
  • __exit__ returning False lets errors continue after cleanup; returning True swallows them — clean up, don't cover up.
  • Files are the tutorial example; DB sessions, API clients, locks, and streaming LLM responses are where the guarantee actually saves your 3 a.m. — and async with is the same idea for async code.
  • @contextmanager from contextlib builds a manager from a small function with one yield: before the yield is the way in, after it is the way out.
  • Review heuristic: any bare acquire() without a with or a deliberate try/finally is a leak until proven otherwise.
