Migrating to AG-UI: before, after, and the pitfalls

Re-fronting a LangGraph or deep-agent agent behind AG-UI — a before/after rebuild, the best practices, the common traps, and a glossary that closes the handbook.

~24 minChapter 123 of 133
The story

A busy restaurant decides to move its front-of-house online. The kitchen does not change — the same chefs, the same order tickets moving down the same rail from starter to main to dessert. What changes is the window: instead of a waiter walking over now and then to mutter "it's coming", a screen at every table now lights up the moment your ticket is fired, shows each course being plated as it happens, and — when the chef needs to know "medium or well done?" — puts the question on your screen and holds the dish until you tap an answer.

The owner faces one honest choice. Either keep the existing kitchen exactly as it is and add a translator who watches the order rail and pushes updates to the screens — cheap, and the kitchen keeps its rhythm. Or, if the kitchen was always overkill for a place that serves four dishes, tear out the elaborate ticket rail and let each chef post updates to the screen directly. Same diners, same live window; two very different amounts of machinery behind it. That is exactly the migration to AG-UI — and this final chapter is about making that choice well.

Where we are, and what AG-UI actually is

A quick recap of the chapter you just finished. In the previous chapter we weighed AG-UI against LangGraph and deep agents and drew the line clearly: those are orchestration engines — they run and control the agent — while AG-UI (the Agent-User Interaction Protocol) is only the streaming contract between an agent backend and a user-facing frontend. AG-UI does not run your agent. It standardises how your agent narrates its work to a screen, and how the screen talks back.

That distinction is the whole of this chapter. "Let's migrate to AG-UI" never means "replace LangGraph"; it means standardise the window onto whatever engine you already have. So the question is never "AG-UI or LangGraph?" — it is "keep the engine and add the window, or was the engine overkill all along?" Two honest paths follow, and picking the right one is the skill.

Note

Both paths end at the same place the diner sees: a screen streaming AG-UI events. The protocol does not care which engine sits behind the adapter — it can front a raw LLM loop, CrewAI, Mastra, Agno, Pydantic AI, LangGraph, or the Microsoft Agent Framework equally. AG-UI is framework-agnostic by design. Your choice of path is an engineering call about the engine, not a requirement of the protocol.

Before: a small LangGraph agent

Here is the "before" — a deliberately tiny StateGraph (a graph whose nodes read and write one shared dictionary of state) with two nodes and a .stream() call. It researches a topic, then writes it up. This is the engine we are about to re-front or replace.

before_graph.py — the existing LangGraph agent
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    topic: str
    draft: str
    approved: bool

def research(state):
    return {"draft": "notes on " + state["topic"]}

def write(state):
    return {"draft": state["draft"] + " -> full article", "approved": True}

builder = StateGraph(State)
builder.add_node("research", research)
builder.add_node("write", write)
builder.add_edge(START, "research")
builder.add_edge("research", "write")
builder.add_edge("write", END)
graph = builder.compile()

# The graph streams state updates, one chunk per node:
for chunk in graph.stream({"topic": "AG-UI", "draft": "", "approved": False}):
    print(chunk)

Look honestly at that graph. It is a straight line — research then write, no branches, no loop, no checkpointer, no interrupt(). Every box in step 1's inventory is empty. That is the textbook case for Path B: the graph is ceremony. So let us rebuild it as a direct loop that speaks AG-UI — and see exactly which concept maps to which event.

After: a direct loop that emits AG-UI events

Now the "after". Instead of a graph, a plain generator walks the same two steps and yields AG-UI events — dictionaries whose type is one of the protocol's SCREAMING_SNAKE_CASE names. (Generators and yield are from the generators chapter; streaming from the streaming chapter.) Read it as a reviewer — click the annotated lines:

No graph, no framework — just a generator. In production it would sit behind a FastAPI endpoint returning a StreamingResponse, and each event would be serialised to SSE by the AG-UI EventEncoder.
yield {"type": "RUN_STARTED",
Every AG-UI run opens with RUN_STARTED carrying threadId and runId. The frontend uses these to attach the stream to the right conversation. Field names are camelCase on the wire.
"threadId": thread_id, "runId": run_id}
yield {"type": "STATE_SNAPSHOT",
The first state event is a full snapshot — the complete shared-state object. This is the graph's State dict, now sent to the UI so useCoAgent can render it.
"snapshot": {"topic": topic, "draft": "", "approved": False}}
yield {"type": "STEP_STARTED", "stepName": "research"}
The old research node becomes a named step. STEP_STARTED / STEP_FINISHED are how the UI draws a progress spine — this is the diner watching each course fire.
yield {"type": "STATE_DELTA",
After the first snapshot, never resend the whole state. A STATE_DELTA carries only what changed, as a JSON Patch — an RFC 6902 array of tiny operations. This is the single biggest bandwidth win of the migration.
"delta": [{"op": "replace", "path": "/draft", "value": "notes on " + topic}]}
One patch operation: replace the value at /draft. The UI applies it to its local copy of state. Malform this path and the UI silently diverges — the classic delta bug.
yield {"type": "STEP_FINISHED", "stepName": "research"}
yield {"type": "TEXT_MESSAGE_START", "messageId": "m1", "role": "assistant"}
The write node's prose now streams. A message is bracketed by START / CONTENT… / END, so the UI can open a bubble, fill it token by token, then close it.
for tok in ["AG-UI ", "fronts ", "the ", "agent."]:
yield {"type": "TEXT_MESSAGE_CONTENT", "messageId": "m1", "delta": tok}
Each TEXT_MESSAGE_CONTENT carries one delta — a fragment to append. This is what makes the answer appear to type itself, exactly like the model's own token stream.
yield {"type": "TEXT_MESSAGE_END", "messageId": "m1"}
yield {"type": "RUN_FINISHED",
The single most important line to never forget. Without RUN_FINISHED, the frontend believes the agent is still working forever — the spinner never stops. This is the number-one AG-UI migration bug.
"threadId": thread_id, "runId": run_id}

The frontend side: where the graph's interrupt() lands

On the React side, CopilotKit is the reference frontend. Two hooks carry almost everything. useCoAgent gives the UI a live copy of shared state — the agent's STATE_SNAPSHOT and STATE_DELTA events update it, and setState pushes changes back into the agent's next turn. useCopilotAction renders a custom component from an agent tool call — and with renderAndWaitForResponse it becomes the new home of the graph's interrupt(): the component stays mounted until you call respond(value), and that value is handed back to the agent as the tool result. React code cannot run in the runner above, so read it as static reference:

ResearchPanel.jsx — shared state and human-in-the-loop
import { useCoAgent, useCopilotAction } from "@copilotkit/react-core";

function ResearchPanel() {
  // Shared state: STATE_SNAPSHOT / STATE_DELTA events land right here.
  const { state, setState, running } = useCoAgent({
    name: "researcher",
    initialState: { topic: "", draft: "", approved: false },
  });

  // Human-in-the-loop: this replaces the graph's interrupt().
  // The component stays mounted until respond() is called.
  useCopilotAction({
    name: "approve_draft",
    description: "Ask the human to approve the draft before publishing",
    parameters: [{ name: "draft", type: "string" }],
    renderAndWaitForResponse: ({ args, respond, status }) => (
      <div>
        <p>{args.draft}</p>
        <button onClick={() => respond("approved")}>Approve</button>
        <button onClick={() => respond("revise")}>Revise</button>
      </div>
    ),
  });

  return <pre>{running ? "working…" : "idle"} {JSON.stringify(state)}</pre>;
}

That component is wrapped in a provider that points at the agent endpoint — <CopilotKit runtimeUrl={…} agent="researcher"> — with a prebuilt <CopilotChat /> for the conversation itself. The diner's screen, in nine lines.

Best practices, learned the hard way

A handful of habits separate a migration that ships from one that limps:

  • Emit STEP_STARTED / STEP_FINISHED around real units of work. They are free, and they turn a blank spinner into a progress spine the user trusts.
  • Prefer STATE_DELTA over resending snapshots. Send one STATE_SNAPSHOT to seed the UI, then only patches. Resending the whole state on every token is the bandwidth mistake that makes agents feel heavy.
  • Keep event ordering strict. A message is START → CONTENT… → END; a run is RUN_STARTED → … → RUN_FINISHED. Out-of-order events confuse every client.
  • Version your shared-state shape. The UI and agent both depend on it; a field renamed on one side and not the other is a silent break. Treat the state schema like an API contract.
  • Keep the agent loop testable apart from the transport. Your loop should yield plain event dicts — as in the runner above — so you can unit-test it with zero SSE and zero React, wiring the encoder on only at the edge.
Careful

The common traps, in order of how often they bite: (1) forgetting RUN_FINISHED — the UI hangs forever. (2) A malformed JSON Patch in STATE_DELTA — a wrong path fails silently and the UI drifts out of sync. (3) Re-emitting full state on every token instead of a delta — needless bandwidth and jank. (4) Assuming AG-UI gives you durability or orchestration — it gives you neither; a dropped connection loses the run unless you persisted it. (5) Tight-coupling UI components to the internal state shape — refactor the agent's state and every component breaks. Loosen that coupling with a small mapping layer.

Who runs the standard

Provenance closes the trio. AG-UI was created and open-sourced by CopilotKit, the company behind the React copilot toolkit of the same name, and released as an open, event-based protocol with its spec at docs.ag-ui.com. Unlike LangGraph, it is not a runtime you install to run agents — it is a wire contract, deliberately thin, with a Python SDK (ag_ui.core for the event classes, ag_ui.encoder for SSE) and a React frontend (CopilotKit). Adapters exist to front LangGraph, CrewAI, Mastra, Agno, LlamaIndex, Pydantic AI and the Microsoft Agent Framework, which is the whole point: one UI contract, many engines. It is the youngest of the three protocols in this handbook, and its event set settled at version 1.0 — where, for instance, the newer REASONING_* events replaced an older THINKING_* set.

An honest maturity check

Now the part a launch deck skips. AG-UI is real and useful, but it is narrow on purpose, and pretending otherwise causes the migrations that fail. Say the limits out loud: AG-UI gives you the streaming contract between agent and UI and nothing else. No orchestration, no checkpointing, no control flow — and no durability, so pull the network cable mid-run and, unless you persisted state yourself, that run is gone. When you rebuild lighter (Path B), every one of those is now your job, and it is easy to under-estimate how much the framework was quietly doing.

So hold both truths, the same way we did with A2A. The vocabulary — events, snapshots and deltas, shared state, generative UI — is durable and worth learning now, and every serious framework ships an adapter. But adapter completeness varies by version, the ecosystem is younger than MCP's or A2A's, and "migrate to AG-UI" never means "delete your orchestration". If a teammate proposes ripping out a genuinely branchy, checkpointed graph "to be AG-UI-native", ask the only question that matters — are we prepared to re-own orchestration and persistence by hand? If not, re-front, and keep the engine.

And the plain-language definitions, for the terms that carry this chapter:

Re-front
Keep the LangGraph graph as the engine and add an adapter that translates its stream into AG-UI events. You gain the standard UI contract without giving up the graph's orchestration and checkpointing.
Rebuild lighter
Replace the graph with a direct loop that emits AG-UI events itself. Simpler, but you take on orchestration, ordering and persistence by hand — right only when the graph was overkill.
STATE_SNAPSHOT
A full copy of the shared state, sent once to seed the UI. After it, send only STATE_DELTA patches.
JSON Patch
The RFC 6902 format inside a STATE_DELTA: an array of small operations (op, path, value) describing exactly what changed.
In the wild

You will meet this migration as a merge request titled "expose the research agent over AG-UI": it adds an ag_ui dependency, a FastAPI endpoint returning a StreamingResponse, and either an adapter around graph.stream() (Path A) or a rewritten loop that yields events (Path B). In the frontend diff you will see useCoAgent and useCopilotAction appear and an interrupt() disappear. And when an operator reports "the agent UI is stuck spinning after the answer finished", you now know exactly which line went missing: RUN_FINISHED.

Check yourself

1. A team says "let's migrate to AG-UI" for their LangGraph agent. What does that actually mean?

AG-UI is a streaming contract between agent and UI, not an orchestration engine. Migrating means putting a standard window in front of your existing engine — re-front it, or rebuild it lighter — never "let AG-UI run the agent".

2. Your graph is a straight line — two nodes, no branches, no checkpointer, no interrupt(). Which path fits?

When the orchestration inventory comes back empty, the graph is overkill. A direct loop that yields AG-UI events does the same work with less machinery — Path B.

3. After the first STATE_SNAPSHOT, why send STATE_DELTA events instead of more snapshots?

The snapshot seeds the UI once; deltas carry only the change as an RFC 6902 JSON Patch. Re-emitting full state on every token is one of the named migration pitfalls.

4. The agent finishes but the UI keeps spinning forever. What is the most likely cause?

RUN_FINISHED closes the run on the wire. Forget it and the client stays in the "running" state indefinitely — the single most common AG-UI bug.

5. The graph relied on a checkpointer for durability. After rebuilding lighter, who provides that?

AG-UI is only the streaming contract; it gives you no orchestration, checkpointing or durability. Replace the graph and you inherit all three — persistence becomes your job.

Key takeaways

  • Migrating to AG-UI means standardising the window onto your agent, not replacing your orchestration — it is a streaming contract, not a runtime.
  • Two honest paths: re-front (keep the LangGraph graph, add an adapter that translates its stream to AG-UI events) when the orchestration earns its keep; rebuild lighter (a direct loop that yields events) when the graph was overkill.
  • The concept map is fixed: graph state → STATE_SNAPSHOT then STATE_DELTA; node text → TEXT_MESSAGE_CONTENT; node runs → STEP_STARTED/FINISHED; interrupt() → renderAndWaitForResponse; checkpointer → your own persistence.
  • Best practice: emit steps, prefer deltas over snapshots, keep event ordering strict, version the shared-state shape, and keep the loop testable apart from the transport.
  • Top pitfalls: forgetting RUN_FINISHED (UI hangs), malformed JSON Patch (silent drift), resending full state (bandwidth), and assuming AG-UI gives you durability or orchestration — it gives you neither.
  • Honest status: the vocabulary is durable and every serious framework ships adapters, but the ecosystem is the youngest of the three protocols — "migrate to AG-UI" is never "delete your engine".
