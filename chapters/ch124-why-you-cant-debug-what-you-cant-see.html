Why you can't debug what you can't see

An operations team says your agent is the reason everything got slow. You believe them or you don't - but either way, you have no data to answer with. This is the chapter where that stops happening.

~25 minChapter 124 of 135
The story

Thursday, 4pm. A message lands from the operations team of a large bank: "Since your agent went live, the shared reporting system is slow. Can you turn it off while we investigate?" You do not think it is you. But when the product owner asks the obvious question - "how much time is the agent actually adding?" - you open your mouth and nothing comes out. You have logs, technically: a wall of print statements that say things like calling model and done. None of them have a timestamp you trust. None of them connect to each other. You cannot say whether one user question took two seconds or twenty, or which step inside it was slow, or whether the slow thing was even yours.

So you argue from feelings, and you lose. Not because you were wrong - later it turned out to be a database index, nothing to do with you - but because the person with the data wins the meeting, and that person was not you. That helpless afternoon is where every engineer discovers the same law: you cannot manage, defend, or fix what you cannot see.

The thing you were missing has a name

Observability is the practice of emitting enough signal from a running system that you can answer questions about it after the fact, without redeploying or adding new print statements. The word sounds grand; the idea is humble. It just means: leave a trail. When something goes wrong at 4pm on Thursday, the trail is already there, and you read it instead of guessing.

You have already met the atom this is built around. Back in the ReAct loop, one user question spun the agent through several turns - think, call a tool, observe, think again. Each of those turns did real work that took real time and could fail in a real way. Observability is simply the discipline of recording that work as it happens, in a form you can query later.

The trap in the story was believing that "we have logs" is the same as "we can see." It is not. Raw logs are one signal out of three, and on their own they answer only one kind of question. To run an agent in production you need all three.

The three signals

Every observability system, no matter the vendor, is built from the same three kinds of data. People call them the three signals (or, older word, the three pillars). Each answers a different question, and the whole art is knowing which one to reach for.

Hold the difference in your head with one sentence: logs say what, metrics say how much, traces say where. The Thursday meeting needed a trace, and a trace was the one thing you did not have.

Why agents need traces more than most systems

A plain web server does one small thing per request: receive, look up, reply. A flat log is almost enough for it. An agent is the opposite. As you saw in the loop, a single user question fans out: the model plans, then a tool runs, then a search returns documents, then the model reads them and answers - and any of those steps might repeat two or three times. One question can easily become a dozen internal operations across several services.

A flat log of that is chaos - twelve unrelated lines, no order, no parent, no idea which tool call belongs to which user. A trace takes those same twelve operations and arranges them as a tree: the user turn at the root, each model call and tool call a branch beneath it, each with its own duration. Now the question "which step was slow?" has an answer you can point at. Watch one user turn become a trace:

1. A user question opens the root span

user_turnspan opens, clock starts
llm.plan
tool.search
llm.answer

The moment the request arrives, we open a root span and stamp it with a fresh trace id. Everything that happens next will carry that same id, so it can all be gathered back together.

2. The model call is a child span

user_turn
→
llm.plan520 ms
tool.search
llm.answer

The first model call opens a child span nested under the root. We record how long it took and how many tokens it used. If the whole turn is slow, we will be able to see exactly how much of the blame lands here.

3. The slow step reveals itself

user_turn
llm.plan
→
tool.search900 ms
llm.answer

The search tool takes 900 milliseconds - nearly twice the model call. Without a trace this would be invisible; with one, it is the obvious suspect. This is the "where" that logs and metrics cannot give you.

4. The turn closes and the tree is complete

user_turn1840 ms total
llm.plan
tool.search
llm.answer400 ms

The root span closes. You now hold a small tree: one turn, three children, each with its own duration. Sum the children, spot the outlier, answer the meeting. The whole turn took 1.84 seconds, and now you can say which 0.9 of it was the search.

Build a tiny tracer, so it stops being magic

A trace is not a mysterious cloud product. At heart it is a list of spans, each with a name, a depth in the tree, and a duration. You can build the whole idea in a dozen lines. Press Run - this records a fake agent turn and prints its trace as a little waterfall, the same shape a real tool would show you.

Tip

Change the ms numbers and re-run: make llm.plan the slow one and watch the "slowest child" line move. That single line - find the biggest child and start there - is the entire debugging instinct that the Thursday meeting was missing. A real tracing tool does exactly this, for millions of traces, with a nicer chart.

What to record on an agent's spans

Traces are only as useful as the attributes you hang on each span. For agents, a small set of fields turns a trace from "it was slow" into "here is the bill." On each model span, record the attributes - the labelled facts attached to a span - that matter: the model name, the number of input and output tokens, the latency, and the cost. On each tool span: the tool name, whether it succeeded, and how long it took. Do this and a trace answers not just "which step was slow" but "which step was expensive," "which model version was used," and "did that tool error and get retried."

These are also the numbers every other chapter in this act builds on. You cannot cap a runaway loop you cannot measure, and you certainly cannot tell an operations team the agent added 1.8 seconds if you never recorded the 1.8 seconds.

In the wild

You will meet these three signals the moment you touch a production agent. When a colleague says "add a trace id to the logs so I can correlate them," they are asking you to stitch logs onto traces. When an on-call engineer pastes a "p95 latency" graph into a channel, that is a metric. When someone opens a tool like Grafana, Datadog, or Honeycomb, clicks one slow request, and expands a waterfall of nested bars - that is a trace, exactly the tree you just built by hand. And when a platform team at a large bank asks every agent team to "emit standard telemetry," they mean: give us all three signals, in an agreed format, so we can see across all of you at once. The next chapter is about that agreed format.

Check yourself

1. An operations team claims your agent added latency to a shared system. Which signal most directly lets you prove how much time the agent added to one request?

Logs are unordered detail and metrics are aggregates; only a trace shows one request's steps and their individual durations, which is exactly "how much time did the agent add, and where."

2. What is a span?

A trace is a tree of spans; each span is one operation (one tool call, one model call) with its own clock and its own labelled attributes.

3. Why do agents rely on traces more than a simple web server does?

The ReAct loop turns one question into a burst of nested operations. A flat log of that is chaos; a trace arranges the same operations into a tree you can actually read.

4. You have logs, metrics, and traces all wired up. Someone asks "what was the exact error message the search tool returned at 16:04?" Which do you reach for?

Logs say WHAT happened, verbatim. Metrics only count, and while a span can carry an error flag, the full human-readable message is the log's job. The three signals are complementary - pick by the question.

Key takeaways

  • You cannot manage, defend, or fix what you cannot see - observability is the discipline of leaving a queryable trail so you read the past instead of guessing about it.
  • There are three signals: logs say WHAT happened, metrics say HOW MUCH and the trend, traces say WHERE in the causal chain the time or failure was.
  • A trace is a tree of spans sharing one trace id; each span is a single timed operation with attributes. It is just a list of timed units - you built one in a dozen lines.
  • Agents need traces especially, because one question fans out into many nested model and tool calls that a flat log cannot untangle.
  • Record the useful attributes on each span - model, tokens, latency, cost, tool name, success - and a trace turns "it was slow" into "here is exactly where, and here is the bill."
  • "We have logs" is not "we can see." Production agents need all three signals, in an agreed format - which is the next chapter.
