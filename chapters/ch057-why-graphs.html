Why graphs?

Your fifty-line loop worked beautifully. Now meet the version a bank will sign off on: every step a named station, every route drawn on a map, every stop recorded — and a gate wherever money moves.

~25 minChapter 58 of 117
The story

You need to cross a vast, unfamiliar city, and you have two options. Option one is a freelance taxi driver — brilliant, knows every back lane, says only "trust me". He usually gets you there. But you cannot see the route before you set off. You cannot say "stop before the toll booth, I want to approve the charge" — by the time you notice, you have been charged. And if the trip is interrupted halfway, there is no record of where you were; you start again from your doorstep.

Option two is the metro. A map on the wall shows every station and every track — the complete set of journeys the system can ever make. Display boards show exactly which station your train is at. Certain stations have ticket gates that will not open without a swipe. Two lines can run at once and meet at an interchange. And when the power fails, the train resumes from the station where it stopped — never from your home.

The taxi is the while-loop agent you built in Act V. The metro map is a graph. This chapter is why the world's most careful organisations are metro people.

The loop you built was a taxi ride

Quick recap of the ReAct loop you built by hand: a for-loop calls the model; the model replies with an action request or a final answer; your code runs the tool and feeds the result back as an observation, around and around until the model says done. The model decides everything — which tool, in what order, how many times. LangChain's create_agent, which you learned to read fluently last chapter, is that same loop with structured tool calling and nicer plumbing.

That design has a name: a free-running loop — an agent whose route is improvised turn by turn by the model, with no externally visible plan. For a demo or a personal assistant it is genuinely great. The trouble starts the day it gets a job at a large company. Five complaints arrive almost immediately:

  • You cannot see the route. "What can this agent do, in what order?" — honest answer: "whatever the model feels like". No diagram, only logs, afterwards.
  • You cannot put a gate between two steps. "A human must approve before the refund tool fires" has nowhere to live — by the time you spot the call, it has run (remember the failure-modes chapter).
  • Nothing is deterministic. Steps that should always run in a fixed order — validate, then draft, then check — are left to the model's mood. Same input, different route on Tuesday.
  • No parallelism on purpose. Two independent lookups that could run at once cannot: one cycle, one action, wait, repeat.
  • A crash means starting over. Turn 14 of 20 dies at 2 a.m.? The history list lived in memory — gone. Re-run all fourteen turns and pay again.

None of these is a complaint about the model. They are complaints about the shape of the program around it — and the shape is what a graph changes.

Stations and tracks: what a graph actually is

A graph, in this context, is a set of named steps plus an explicit map of which step may follow which — stations and tracks, drawn before any train moves. Each station is a node — a plain function that does one unit of work: call the model, run the tools, validate an output. Each track is an edge — a declared, allowed route from one node to another. Nothing outside the drawn edges can ever happen, no matter what the model says.

Riding through this map is the state — one shared bundle of data (in practice, a dictionary) that every node receives, may update, and passes along: messages, flags, partial results. In ch045 the history list played this role informally; a graph makes it official.

Some junctions need a decision: after the model speaks, do we go to the tools station or the exit? That is a conditional edge — a small, ordinary function that reads the current state and returns the name of the next node. It is the interchange where the track forks — the only place routing decisions happen. Everything else is fixed rail.

Put together — explicit steps, declared transitions, one current position — this structure has a classic computer-science name: a state machine, a system always in exactly one known state, moving only along pre-declared transitions. Traffic lights and washing-machine cycles are state machines. Well-run agents, it turns out, are too.

Note

The part people miss: a graph does not remove the loop. The ReAct cycle is still there — drawn as an explicit circle on the map: agent to tools, tools back to agent, until the interchange routes to the exit. The graph did not replace your loop. It made your loop visible, inspectable, and interruptible.

Five things the map buys you

Everything enterprises love about graphs falls out of one fact: the structure exists outside the model, before runtime. Five consequences, one per complaint:

The last row hides the most underrated idea. A checkpoint is a snapshot of state plus position in the graph. In memory, it gives crash recovery within a session. In a database, conversations that survive restarts — the durable cousin of the context tricks from the memory chapter. As a full trail, replay: rewind to any past step, inspect the exact state the system saw, run forward again. Debugging stops being archaeology and becomes time travel.

Tip

Evaluating an agent platform? Ask the metro questions: Can I see the map before it runs? Can I put a barrier before the dangerous station? Where is the trip log, and can I resume from it? If the answers are "no, no, and there isn't one", you are looking at a very charming taxi.

Build the engine yourself — twenty lines

The engine under all that grandeur is small enough to build right here: the calculator and scripted fake_llm() from ch045, two nodes, one interchange, one fixed edge — and an engine that checkpoints before every step and refuses to pass a closed gate. Press Run, then flip APPROVED to False and run again to feel a gate close.

Three things to notice. First, behaviour identical to ch045 — the graph changed the shape, not the result. Second, the replay from checkpoint 1 reruns only the tail and lands on the same answer: time-travel debugging in miniature. Third, with APPROVED = False the run stops before the tools node, state saved, nothing irreversible done. Try bolting that pause onto the ch045 loop; there is no clean place for it to live.

The engine, one line at a time

You just ran it; now read it like a reviewer. The whole trick lives in nine lines — click each one:

graph_engine.py — the parts that matter
NODES = {"agent": agent, "tools": tools}
The stations: a dict mapping names to plain functions (ch4) — ch045's TOOLS dict trick, promoted from single tools to whole steps.
FIXED_EDGES = {"tools": "agent"}
Fixed rail: after tools, the state ALWAYS travels back to agent — the ReAct circle, written down as data. The model cannot reroute it, because it is never consulted.
ROUTERS = {"agent": route_after_agent}
The interchange: after the agent node, a small Python function — not the model — reads the state and names the next station. You can unit-test a routing rule; you cannot unit-test a mood.
GATED = {"tools"}
The ticket barriers: stations that require approval before the train may enter. In production: the refund node, the deploy node, the send-email node.
for step in range(max_steps):
Still a bounded loop — max_turns from ch045 wearing a new name. Frameworks call it a recursion limit. The metro also stops at midnight.
trail.append((node, {"messages": list(state["messages"])}))
The checkpoint: position plus a snapshot of the state, saved BEFORE the step runs. list(...) copies the messages so later steps cannot rewrite history. Resume, persistence, and replay are all built on this line.
if node in GATED and not APPROVED:
The gate check — note WHERE it sits: between the checkpoint and the work. The pause lands before anything irreversible, state already saved — exactly what makes it resumable.
state = {**state, **NODES[node](state)}
Run the station, merge its updates into the shared state — ** unpacking spreads both dicts into a new one, the node's updates winning. Every node gets the state, changes its slice, passes it on.
node = ROUTERS[node](state) if node in ROUTERS else FIXED_EDGES[node]
The traversal itself: if the station has an interchange, ask it; otherwise follow the fixed rail. Everything a graph framework does at runtime is an industrial-strength version of this line.

Why enterprises and regulators are metro people

Now put on the auditor's spectacles. Auditors ask the same questions of an AI system as of any process touching money or customers: What can it do? Who approves the risky parts? Show me exactly what happened on the 14th — and reproduce it.

For a free-running loop, the honest answers are uncomfortable: "any tool, in any order the model chooses"; "approval happens... after we read the logs"; "we have transcripts, but not a reproducible run". For a graph, each answer becomes a document. The rendered map answers what can it do: a finite, reviewable set of stations and tracks. Gates answer who approves: a human stop on the rail before every irreversible action, impossible to bypass because no other track exists. The checkpoint trail answers what happened: a record you can rewind and replay in front of the regulator, with the precise state at every stop.

This is why large organisations increasingly run agents as bring-your-own-graph: instead of handing the platform a free-form agent and a prayer, a team hands over a compiled graph object to run inside centrally managed rails — model gateway, logging, checkpoint database, approval UI. The platform team does not need to trust your prompt engineering; they can see your structure. A harness team at a large bank can reject a graph whose refund node has no gate — at review time, before it ever runs. Try rejecting a while-loop for the same reason; there is nothing to point at.

One honest caveat, so you do not become a graph zealot: the taxi is still the right ride for plenty of trips. Free-running loops shine for open-ended work — research, coding, exploration — where the route cannot be drawn in advance and improvisation is the point. Claude Code is a free-running loop, gloriously so. The more regulated, repeatable, and irreversible the work, the more graph it wants; the more exploratory, the more loop. Many production systems land in between — a graph skeleton with an agent loop inside one node.

The shape of what is coming

LangGraph is the graph framework of the LangChain family; next chapter dissects its central object line by line. To set your eyes on the destination, here is this chapter's agent written with the real API — every concept maps one-to-one:

the same map, in LangGraph (dissected fully next chapter)
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver

builder = StateGraph(AgentState)          # the state's shape
builder.add_node("agent", agent)          # stations
builder.add_node("tools", tools)
builder.add_edge(START, "agent")          # fixed rail
builder.add_edge("tools", "agent")        # the ReAct circle
builder.add_conditional_edges(            # the interchange
    "agent", route_after_agent, {"tools": "tools", "END": END}
)

graph = builder.compile(checkpointer=InMemorySaver())

Nothing in that block should feel foreign now: add_node registers stations, add_edge lays fixed rail, add_conditional_edges installs the interchange, and compile(checkpointer=...) switches on the trip log — InMemorySaver keeps checkpoints in memory; drop-in Postgres, Redis, and SQLite checkpointers persist them across restarts. Gates arrive via interrupt(), and each conversation rides under its own thread_id. All of it, chapter by chapter, from here.

In the wild

Open a serious agent repository and look for a file — often graph.py or workflow.py — full of add_node and add_edge calls: that file is the metro map, and reading it tells you everything the system can do. In merge requests, "add a human-in-the-loop gate before the payout node" and "wire a checkpointer so runs survive restarts" are graph sentences. In incident reviews, "replay the thread from checkpoint 12" is the time travel you just performed above. And "the platform only accepts bring-your-own-graph" means: hand us a map we can inspect and gate, not a taxi we have to trust.

Check yourself

1. What is the most fundamental difference between the ch045 free-running loop and the same agent as a graph?

Same model, same tools, same behaviour — the ReAct loop is still there, drawn as an explicit cycle. What changed: the route map now exists in code, before anything runs, where it can be seen, reviewed, gated, and checkpointed.

2. Your agent crashes at step 14 of 20 during an overnight run. What does checkpointing change about the morning after?

A checkpoint is state plus position, saved at every step. Resume from the last one and the run continues where it died; keep the whole trail and you can rewind and replay any step. Crashes still happen — they just stop being expensive.

3. Why can an enterprise platform team reject a risky graph at review time, when they could not do the same for a free-running loop?

Every station and track is declared up front, so "the refund node has no approval gate" is a concrete, pointable defect. A loop's possible behaviours exist only at runtime, in the model's choices — nothing to inspect, nothing to point at.

4. When is the free-running loop still the better choice?

Research, coding, and exploration resist pre-drawn maps — that is why tools like Claude Code are free-running loops. Regulated, repeatable, irreversible work wants a graph; many real systems put a loop inside one node.

Key takeaways

  • A graph is a metro map for an agent: nodes are stations (units of work), edges are tracks (allowed routes), and one shared state rides through them. The structure exists in code, before runtime.
  • A graph does not remove the ReAct loop — it draws it as an explicit cycle (agent to tools and back), which makes the loop visible, gateable, and resumable.
  • The map buys five powers a loop cannot offer: visibility of the route, gates before irreversible steps, deterministic ordering, deliberate parallel branches, and checkpoint/replay.
  • A checkpoint — state plus position, saved at every step — powers three features: crash recovery, cross-session persistence, and time-travel debugging.
  • Enterprises and regulators prefer graphs because the map is a reviewable artifact: what-can-it-do, who-approves, and what-happened all get documentary answers — hence bring-your-own-graph as the standard enterprise handover.
  • Loops are not dead: exploratory work still wants a taxi. Regulated, repeatable, irreversible work wants the metro — and many production systems run a loop inside one node of a graph.
