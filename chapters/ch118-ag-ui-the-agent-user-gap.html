AG-UI: closing the agent-to-user gap

MCP connects agents to tools, A2A to each other — AG-UI connects them to the human. The event-stream protocol that makes an agent's work visible, live, in a real UI.

~28 minChapter 118 of 123
The story

Two kitchens, same order. In the first, you hand your ticket through a hatch and the shutter comes down. A single word glows above it: preparing. Twenty minutes pass. Is the chef searing your fish or has he wandered off for a smoke? You cannot tell. You cannot even tell if he read the note about the allergy. All you can do is stare at the shutter and hope. When the dish finally slides out, it is a surprise — sometimes a good one.

The second kitchen is a glass counter. You watch the chef pick up your ticket, call out "one grilled fish, no butter", reach for the pan, sear, plate. Halfway through he leans over: "the fish is small today — add prawns?" You nod, he adjusts, and thirty seconds later the plate is in front of you. Same food, same chef, same twenty minutes — but you were with him the whole way. You trusted the meal before you tasted it.

Every AI agent is a chef. The first kitchen is a frozen loading... spinner. The second is a chat window that streams its thinking, shows the tool it just grabbed, and asks you a question mid-task. The glass counter has a name now. It is called AG-UI.

The third protocol — the leg that was missing

Step back and look at the whole stack you have built across this handbook. In chapter 75 you met MCP — the Model Context Protocol — connecting an agent downward to its tools and data. In chapter 81 you met A2A, connecting an agent sideways to other agents, as peers. Both point outward, into the machine world. Neither says a word about the one participant who started all this: the human at a screen, waiting.

That is the gap. Your agent can query a database, hire a colleague across the network, and reason for two minutes straight — and the person who asked sees a spinner. The work is invisible; when it lands, it lands as a wall of text with no story behind it. Trust leaks out of exactly this gap.

AG-UI — the Agent-User Interaction Protocol — is the third leg. It is an open, event-based protocol, created by CopilotKit, that standardises how an agent backend streams its activity to a user-facing frontend, and how that UI sends state and actions back the other way. The one-sentence pitch completes the trio you already know:

MCP connects an agent to its tools. A2A connects it to other agents. AG-UI connects it to the human in front of it. Tools below, colleagues across, the user in front.

What a request/response API cannot say

Why invent a protocol at all? Because the plain request/response shape you would reach for by default — send a request, wait, get one JSON blob back — physically cannot express what a running agent needs to say. An agent's work is not one answer; it is a performance over time. Five things happen during that performance, and a single blob can carry none of them:

  • Streaming text. The answer arrives token by token, the way you met in chapter 41, so the reader starts reading before the sentence is finished.
  • Progress and tool visibility. "I am now searching the invoices" — the little chip that shows which tool the agent grabbed and what it got back, so a two-minute wait feels like watching, not hoping.
  • Shared state. A structured object — a draft document, a form, a plan — that both sides hold and both sides can change, kept in sync as the agent edits it.
  • Generative UI. The agent asks the frontend to render a real widget — a chart, a confirmation card, a date picker — instead of dumping raw text.
  • Human-in-the-loop. The agent pauses mid-run, shows you a card, and waits for your click before continuing — the glass-counter chef asking about the prawns.

A request/response API gives you exactly one moment to speak: the end. AG-UI turns that one moment into a live channel. The mechanism is the humble event stream — instead of one big reply, the backend emits a sequence of small, typed events as things happen, and the UI reacts to each the instant it arrives. If generators and yield from chapter 39 are ringing a bell, hold that thought — you are about to run exactly that idea.

The wire: a stream of typed events

Now the plumbing, and it is refreshingly ordinary. The default transport — the way bytes actually travel — is HTTP plus Server-Sent Events (SSE), a decades-old web standard where the server keeps one connection open and pushes text frames as they occur. Each event is serialised as a line beginning data:, a JSON object, and a blank line. (A high-performance binary HTTP option exists for production, and the protocol is transport-agnostic — WebSockets and webhooks work too — but SSE is the one you will meet first.)

Read a real stream the way the frontend does. Each event has a type in SCREAMING_SNAKE_CASE; the JSON field names are camelCase. Click the annotated lines:

the SSE response body — one agent run, narrated event by event
data: {"type": "RUN_STARTED", "threadId": "t-1", "runId": "r-1"}
The opening bracket of the whole performance. Lifecycle events (RUN_STARTED, RUN_FINISHED, RUN_ERROR) frame every run; the UI flips the spinner on here. Note the field names — threadId, runId — are camelCase on the wire.
data: {"type": "TEXT_MESSAGE_START", "messageId": "m-1", "role": "assistant"}
A new chat bubble is beginning. The message will arrive in pieces, so it is bracketed by a START and an END with content in between — the same shape as the streaming you saw in chapter 41.
data: {"type": "TEXT_MESSAGE_CONTENT", "messageId": "m-1", "delta": "Let me check "}
One token-ish chunk of text. The delta — the little "what changed" fragment — is appended to the bubble. Dozens of these stream by, and the sentence grows on screen in real time.
data: {"type": "TOOL_CALL_START", "toolCallId": "tc-1", "toolCallName": "search_invoices"}
The agent decided to use a tool. This is the "calling search_invoices..." chip lighting up. Tool events come as a START / ARGS / END trio, mirroring the tool-calling handshake from chapter 36.
data: {"type": "TOOL_CALL_ARGS", "toolCallId": "tc-1", "delta": "{\"vendor\": \"Acme\"}"}
The tool's arguments, also streamed as deltas because the model writes them token by token. The UI can show the query forming live.
data: {"type": "TOOL_CALL_RESULT", "toolCallId": "tc-1", "content": "3 invoices found"}
What the tool returned, flowing back so the chip can show its result. The agent will see this too and reason on it — the observation step of the loop, made visible.
data: {"type": "STATE_DELTA", "delta": [{"op": "replace", "path": "/total", "value": 1770}]}
A patch to the shared state — the structured object both sides hold. The delta here is a JSON Patch array (RFC 6902): a list of tiny op / path / value edits, so only what changed travels. A full STATE_SNAPSHOT is sent occasionally to resync.
data: {"type": "TEXT_MESSAGE_END", "messageId": "m-1"}
This bubble is complete. Balanced against the START — the UI now knows the assistant message is final and can stop its typing cursor.
data: {"type": "RUN_FINISHED", "threadId": "t-1", "runId": "r-1"}
The closing bracket. Spinner off, run over. There are ~16 standard event types in all — the lifecycle, text, tool and state families you just saw, plus RAW and CUSTOM escape hatches for anything the standard set does not cover. The next chapter maps the full event model, family by family.
Note

The event set is versioned and it evolves. The newer REASONING_* events (for streaming an agent's chain-of-thought) replaced an older THINKING_* set that was removed at the 1.0 release. When you read a stream in the wild and see a type this chapter did not name, that is expected — check the current event reference at docs.ag-ui.com rather than assuming it is a bug.

The two sides in real code

On the backend, the Python SDK gives you typed event classes and an encoder that turns them into SSE frames. The event classes take snake_case keyword arguments that serialise to the camelCase JSON you saw on the wire. A minimal endpoint is just a generator wrapped in a streaming response:

agent_server.py — the backend, with the ag_ui SDK
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from ag_ui.core import (RunAgentInput, RunStartedEvent,
                        TextMessageStartEvent, TextMessageContentEvent,
                        TextMessageEndEvent, RunFinishedEvent)
from ag_ui.encoder import EventEncoder

app = FastAPI()

@app.post("/")
async def run(input: RunAgentInput, accept: str = "text/event-stream"):
    encoder = EventEncoder(accept=accept)

    def gen():
        # snake_case kwargs -> camelCase JSON on the wire
        yield encoder.encode(RunStartedEvent(
            thread_id=input.thread_id, run_id=input.run_id))
        yield encoder.encode(TextMessageStartEvent(message_id="m-1"))
        for chunk in ["Let me ", "check ", "the invoice."]:
            yield encoder.encode(TextMessageContentEvent(
                message_id="m-1", delta=chunk))
        yield encoder.encode(TextMessageEndEvent(message_id="m-1"))
        yield encoder.encode(RunFinishedEvent(
            thread_id=input.thread_id, run_id=input.run_id))

    return StreamingResponse(gen(), media_type="text/event-stream")

On the frontend, CopilotKit (React) consumes that stream. You wrap your app in a provider, drop in a prebuilt chat, and — this is the powerful part — use hooks to wire shared state and human-in-the-loop widgets. The syntax below is illustrative React; you are reading it, not running it:

App.jsx — the frontend, with CopilotKit
import { CopilotKit, CopilotChat } from "@copilotkit/react-ui";
import { useCoAgent, useCopilotAction } from "@copilotkit/react-core";

function Invoice() {
  // Shared state: the agent's STATE_SNAPSHOT / STATE_DELTA
  // events update `state`; setState() pushes UI edits back
  // into the agent's next turn.
  const { state, setState, running } = useCoAgent({
    name: "invoiceAgent",
    initialState: { total: 0 },
  });

  // Human-in-the-loop: renderAndWaitForResponse keeps this
  // card mounted until the human clicks. `respond(value)`
  // returns that value to the agent as the tool result.
  useCopilotAction({
    name: "confirmPayment",
    description: "Ask the human to approve a payment.",
    parameters: [{ name: "amount", type: "number" }],
    renderAndWaitForResponse: ({ args, respond }) => (
      <div>
        Approve ₹{args.amount}?
        <button onClick={() => respond("approved")}>Yes</button>
        <button onClick={() => respond("declined")}>No</button>
      </div>
    ),
  });

  return <div>Total: ₹{state.total} {running && "…"}</div>;
}

export default function App() {
  return (
    <CopilotKit runtimeUrl="/api/copilotkit" agent="invoiceAgent">
      <Invoice />
      <CopilotChat />
    </CopilotKit>
  );
}

Read what those two hooks buy you. useCoAgent gives the component a live copy of the agent's shared state and a setState to push edits back — that is bidirectional state, the draft document both sides can hold. useCopilotAction with renderAndWaitForResponse is human-in-the-loop: an agent tool call renders a real React card, the run pauses while the card waits, and the human's click becomes the tool's result — the same pause-and-resume idea you met as an interrupt in chapter 63, but drawn on the user's screen instead of a debugger.

Who made it, and what it fronts

Provenance matters for a protocol, so name it plainly. AG-UI was created and open-sourced by CopilotKit, the team behind the React toolkit of the same name, to standardise the agent-to-frontend link they kept rebuilding by hand. Its whole selling point is that it is framework-agnostic: the events are a neutral contract, so the same CopilotKit frontend can sit in front of a raw LLM loop you wrote yourself, or CrewAI, Mastra, Agno, LangGraph, Pydantic AI, LlamaIndex, or the Microsoft Agent Framework. Whatever runs the agent, if it can emit AG-UI events, the UI does not care which engine is behind the glass. That neutrality is the same play that made MCP and A2A stick: a protocol nobody has to marry a single vendor to adopt.

An honest maturity check — protocol, not engine

Here is the part a vendor slide would skip, and it is the single most important sentence in this Act. AG-UI is a protocol, not an orchestration engine. It is the contract between an agent and a UI — the wire format for events flowing both ways. It is emphatically not a runtime like LangGraph. It does not give you the agent loop, the orchestration, the checkpointing, the durable state, or the branching. Those you still build or bring yourself.

Get this backwards and you will be disappointed. AG-UI will not manage a multi-step plan, resume a crashed run, or route between sub-agents. It happily carries the events a LangGraph run emits, or an interrupt a durable engine raises — but the durability and routing live in that engine, not the protocol. It is the glass counter and the order-callouts: it makes the kitchen visible and lets you talk to the chef. It is not the kitchen. Chapter 94's question — do you even need a heavy graph framework? — is a different axis from "do you want your agent's work visible in a UI?" You answer them independently, and chapter 122 takes that comparison head-on.

The ecosystem is younger than the idea. The event set churned on the way to 1.0 (the THINKING_* to REASONING_* rename is one visible scar), integration polish varies across the frameworks it fronts, and much of what teams ship today is still a bespoke SSE stream reinvented per project. But the direction is real: event stream, shared state, generative UI, human-in-the-loop are the durable vocabulary of agent frontends now. Learn the shapes; verify the exact event names against the current docs before you code.

In the wild

You will meet AG-UI first as a dependency — @copilotkit/react-core in a package.json, or ag-ui in a requirements.txt — and as an endpoint that returns Content-Type: text/event-stream instead of application/json. In the browser network tab it looks like a request that never "finishes": a long-lived response with data: {...} lines ticking in. In a merge request, "stream the agent over AG-UI" means: emit the lifecycle/text/tool events from your loop and mount a CopilotKit provider on the frontend. And when a teammate says the chat "renders the tool call live" or "waits on a human approval card mid-run", you now know precisely which events and which hook are doing that work.

Check yourself

1. MCP connects an agent to tools and A2A connects it to other agents. What gap does AG-UI fill?

MCP points down to tools, A2A points across to peers; both face the machine world. AG-UI is the third leg — the standard channel between the agent and the user watching at a screen. Tools below, colleagues across, the user in front.

2. Why can a plain request/response API not express what a running agent needs to show?

One request, one blob back, at the very end. That shape cannot carry token-by-token text, a live tool chip, an evolving shared object, or a mid-run pause for a human decision. An event stream turns the single final moment into a live channel.

3. In an AG-UI stream you see {"type": "STATE_DELTA", "delta": [{"op": "replace", "path": "/total", "value": 1770}]}. What is this?

Shared state is synced with STATE_SNAPSHOT (the whole object) and STATE_DELTA (a JSON Patch array of op/path/value edits, RFC 6902). Sending only the delta keeps a live document in sync without re-transmitting all of it.

4. A colleague says "let's adopt AG-UI so we get orchestration, checkpointing and branching for free." What is the honest correction?

AG-UI standardises the events between agent and UI. It happily carries the output of a LangGraph run, but the durability and routing live in that engine, not in the protocol. It is the glass counter, not the kitchen.

5. Which claim about AG-UI's origin and reach is accurate?

AG-UI was created and open-sourced by CopilotKit. Its whole value is neutrality: the events are a wire contract, so the same frontend sits in front of whatever engine emits them — no vendor marriage required.

Key takeaways

  • AG-UI is the third protocol in the stack: MCP connects an agent to tools, A2A to other agents, and AG-UI to the human's frontend — tools below, colleagues across, the user in front.
  • It exists because a request/response API gives an agent one moment to speak; AG-UI replaces that with a live event stream carrying streaming text, tool progress, shared state, generative UI and human-in-the-loop pauses.
  • The wire is ordinary: HTTP + Server-Sent Events by default, ~16 typed events with SCREAMING_SNAKE_CASE type and camelCase JSON fields; STATE_DELTA carries JSON Patch edits so only changes travel.
  • Two sides: a Python backend emits events (the ag_ui SDK + an EventEncoder, streamed from FastAPI), and a CopilotKit React frontend consumes them — useCoAgent for shared state, useCopilotAction with renderAndWaitForResponse for human-in-the-loop.
  • It was created by CopilotKit and is framework-agnostic — the same UI can front a raw loop, CrewAI, Mastra, Agno, LangGraph, Pydantic AI or the Microsoft Agent Framework.
  • Honest limit: AG-UI is a protocol, not an orchestration engine. It does not give you the loop, checkpointing, durability or branching — you build or bring those, and chapter 122 weighs it against a heavy graph framework directly.
