Deep agents vs Claude Code

Two teams, two codebases, zero coordination — and the same four organs grew anyway. Learn to see the shark and the dolphin in every agent you operate.

~25 minChapter 73 of 117
The story

Put a photograph of a shark next to a photograph of a dolphin. Same torpedo body, same dorsal fin, same crescent tail. Yet one is a fish whose lineage is older than trees, and the other is a mammal whose ancestors walked on land and then, fifty million years ago, waded back into the sea. They never consulted each other. The ocean designed them both — because moving fast through water has exactly one good answer, and any body that commits to that life gets squeezed into the same shape. Biologists call this convergent evolution: unrelated lineages, same problem, same solution.

Now put the deepagents source code next to a Claude Code session. A todo-list tool. A task tool that spawns fresh-context subagents. Files used as memory. A compression trick for when the context window fills. Two teams with completely different starting points — a Python library from the LangChain world, a terminal product from an AI lab — grew the same four fins. This chapter is the dissection lab. We lay the shark and the dolphin side by side, find every organ that matches because the ocean demanded it, and every organ that differs because a fish and a mammal are still different animals.

Two lineages, one ocean

Quick recap of where Act VIII stands. In the anatomy chapter you opened create_deep_agent and found LangChain's create_agent loop with extra equipment bolted on. You then met each piece: the write_todos planning tool, the virtual filesystem behind ls, read_file, write_file and edit_file, and the task tool that spawns subagents. And the previous chapter priced the subagent trick: delegation can roughly double your token bill, because the child runs a whole second metered loop with its own big system prompt and the findings are paid for twice — once inside the worker, once as the report in the parent — what it buys is not a discount but a clean parent context.

Separately, you have almost certainly operated the other animal. Claude Code is the terminal agent from Anthropic that edits real files, runs shell commands, and works on codebases for hours. If you have ever watched it print a little checklist, dispatch a subagent, jot notes into a scratch file, or announce that it is "compacting" the conversation — you have already field-observed every organ this chapter dissects.

Here is the framing that makes the whole comparison click: the two systems are different kinds of thing. Claude Code is a harness — a finished, runnable product that wraps the agent loop in everything a human operator needs: a terminal interface, permission prompts, configuration, session management. You do not assemble it; you drive it. deepagents is a library — code you install with pip and embed inside your own product. You pick the model, supply the tools, choose where files live, and ship the result to your users. One is a vehicle; the other is an engine on a pallet with excellent mounting brackets.

Different kind of artefact, different team, different ecosystem. Yet both swim in the same ocean: tasks that run for hours, touch hundreds of tool calls, and generate far more information than one context window can hold. That ocean is about to explain every matching fin.

One task, two animals

Tables are for reference; motion is for understanding. Step through the same job — "research rooftop solar and draft a report" — running on both systems at once.

1. Both write the plan down first

deepagentswrite_todos(["research", "draft", "polish"])
Claude CodeTodoWrite — a 3-item checklist appears in your terminal

Before touching the task, each agent externalises its plan. The plan is stored outside the raw transcript — in agent state on one side, in the harness's todo state on the other — so it cannot fade or be compressed away.

The subagent reads twenty sources and hits three dead ends — all inside its own context. The parent never sees the mess. The tokens are still paid; what you buy is the parent's clean window.

The subagent's full findings go to a file; only a one-line pointer returns to the parent. Same gesture, different disk: the library's default disk is the state object, the harness's disk is your actual machine.

Tip

Two experiments worth a minute each. First, drop WINDOW from 6 to 3 and re-run — compaction fires earlier and eats more turns, yet the printed plan and the files come through untouched, because they live outside the transcript. That is precisely why both systems store the todo list and the notes outside the message history. Second, delete the write_file line inside the subagent: the parent now holds only "done", and the findings are gone forever. That is exactly what "the agent forgot its own research" looks like in production logs.

Where the animals differ

Convergent evolution never makes two lineages identical — a dolphin still surfaces to breathe. The differences here are just as instructive as the similarities, and they cluster around three things: what kind of artefact each is, who guards the dangerous actions, and which ecosystem each plugs into.

deepagentsClaude Code
What it isA library: pip install, then from deepagents import create_deep_agent inside your own appA harness: a finished terminal product you run as-is
Who assembles itYou — choose the model, the tools, the prompt, the file backend, in codeThe vendor — loop, tools and interface arrive pre-wired; you configure, not construct

The permissions row deserves the most attention, because it falls straight out of the harness-versus-library split. Claude Code ships with a human sitting in front of it, so it can — and does — stop and ask before running a shell command or editing a file, with allow-lists and deny-lists to tune how often it asks. A library cannot assume there is a human, a terminal, or even a screen. So deepagents ships no permission prompt at all: if your deep agent can call a tool that deletes records, nothing will pause for approval unless you build that pause — typically with the human-in-the-loop interrupt pattern you already know. Freedom and responsibility arrive in the same box.

Careful

This is the single most common mistake teams make when they move from operating a harness to building on a library: they assume "the framework" is guarding something. It is not. The harness had guardrails because a product team built them; your deepagents app has exactly the guardrails you wrote, and not one more. Audit your tool list with that sentence in mind.

The filesystem difference is subtler but just as consequential. Claude Code's file tools touch your real disk — that is the whole point of a coding agent, and why its permission system exists. deepagents' file tools are, by default, an illusion: with the StateBackend, write_file("/notes/x.md", ...) stores text in the agent's state object, and your real disk is never touched. Swap in a FilesystemBackend and the identical tool calls hit real files; a StoreBackend lands them in a database-backed store; a CompositeBackend routes different path prefixes to different homes. The model-facing interface never changes — the model just writes files — while you decide what "disk" means. That indirection is something a harness never needed to invent: its disk was always real.

Why everyone keeps inventing the same crab

Biologists joke that nature cannot stop evolving crabs — the flat, armoured, pinchy body plan has emerged independently at least five times, often enough to earn its own name, carcinisation. Agent engineering has its own carcinisation, and you can name the selection pressures precisely, because you met each one earlier in this course:

The model is stateless. From the ReAct chapter: the transcript is the model's only memory, and anything not re-asserted fades. Selection pressure: keep the goal somewhere it can be re-read. Result: a todo tool.

The window is finite. From the context chapter: there is a hard ceiling on how much the model can see at once, and hour-long tasks blow through it. Selection pressure: park information outside the window. Result: files as memory — and compression when even that is not enough.

Tokens cost money, and the whole history is re-sent every turn. From the cost chapter: hoarding detail in context is the most expensive possible place to keep it. Selection pressure: keep the working set small. Result: offloading, summarising, delegating.

Clutter degrades reasoning, not just budgets. From the failure-modes chapter: a context stuffed with dead ends makes the model measurably worse at the task in front of it. Selection pressure: quarantine the mess. Result: fresh-context subagents.

Stateless model, finite window, per-token pricing, clutter-degraded reasoning. Any team building an agent meant to run for hours swims into all four currents, and the design space funnels them to the same four organs — the way water funnels every fast swimmer into a torpedo with fins. It is not copying. It is physics.

The evidence keeps arriving: teams who never talk — open-source coding agents, research-agent startups, enterprise harness builders — keep publishing lessons that read like translations of one another. Plan explicitly, use files as memory, isolate subtasks, compress on overflow. Believe the lesson.

In the wild

You will now spot these organs everywhere. In a trace of a deepagents run, the tool calls read write_todos, task, write_file; in a Claude Code session log the same beats appear as TodoWrite, Task, and file edits. A merge request adding .claude/agents/reviewer.md is someone growing the subagent organ in the harness; a merge request wiring a CompositeBackend with a /memories/ prefix is the same organ sprouting library-side. And when a teammate sighs "it auto-compacted mid-task and lost the thread", you know which organ hiccupped — and that the fix is moving precious material into the plan or a file, where compression cannot reach it.

Check yourself

1. Both systems grew a planning tool — write_todos in deepagents, TodoWrite in Claude Code. What problem does writing the plan down actually solve?

The model is stateless and its attention fades over hundreds of turns. A plan stored outside the fading transcript is something it can keep re-reading to stay on course. Prettier logs are a bonus, not the reason.

2. In both animals, what does a subagent hand back to its parent when it finishes?

The point of delegation is context isolation: the subagent absorbs the twenty-file mess, and the parent receives one clean paragraph (often plus a file path). Returning the full transcript would re-pollute the very window you were protecting.

3. A deepagents agent, running with the default backend, calls write_file("report.md", draft). Where does that file end up?

The default StateBackend makes the filesystem virtual: the model believes it is writing files, but the bytes live in agent state. Swapping in a FilesystemBackend routes the very same tool calls to real disk — the model-facing interface never changes.

4. A teammate is building a customer-facing app on deepagents and asks: "Where do I switch on the permission prompt before dangerous tools run?" What is the honest answer?

Claude Code can prompt because it ships with a terminal and a human in front of it. A library assumes neither, so it ships no gate at all — your app has exactly the guardrails you wrote. The interrupt pattern from the human-in-the-loop chapter is the standard way to add one.

5. Two teams that never spoke shipped near-identical features: plan tool, subagents, files, compression. What is the best explanation?

This is convergent evolution: no spec, no copying, just the same ocean. Any team building hour-long agents hits the same four pressures, and the design space funnels them to the same four organs — which is why yet more agent products keep growing them too.

Key takeaways

  • deepagents and Claude Code grew the same organs independently: write_todos vs TodoWrite for planning, task vs Task for fresh-context subagents, a filesystem (virtual vs real disk plus scratchpads) for offloading detail, durable memory files (/memories/ vs CLAUDE.md), and summarisation vs compaction for full windows.
  • The deep split is harness versus library: Claude Code is a finished product you operate; deepagents is an engine you embed in your own product — you choose the model, the tools, and what "disk" means.
  • deepagents' filesystem is virtual by default (StateBackend); FilesystemBackend, StoreBackend and CompositeBackend let the same file tools target real disk, a store, or a composed mix — with a persistent /memories/ prefix as the memory trick.
  • Permission prompts are harness behaviour. A library ships none: on deepagents, every guardrail is one you built, typically with a human-in-the-loop interrupt.
  • The convergence is not fashion but physics: stateless models, finite windows, per-token pricing, and clutter-degraded reasoning push every long-running agent toward the same four designs.
  • You can now read either animal fluently — and predict the anatomy of the next agent system you meet before you open its docs.
