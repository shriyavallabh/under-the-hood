Building an AG-UI backend

Emit RUN_STARTED, stream text deltas, report tool calls, sync state — a minimal runnable agent backend that any AG-UI frontend can render.

The story

Picture a busy restaurant kitchen. For years your cooks scribbled what was ready on scraps of paper — and every waiter read them differently. One thought "table 4, done" meant serve now; another thought it meant plating had started. Chaos. So you install the standard kitchen ticket-printer: the little machine by the pass that spits out slips in one agreed format. Order fired. Starter up. Chef needs a substitution — waiting. Main up. Table complete. Now it does not matter who is on the floor tonight — a new waiter, a temp, the host covering a shift. Anyone who can read a ticket can run your dining room, because your kitchen speaks the printer's language.

Your cooks did not change how they cook. They just started announcing their work on the standard printer. That is exactly what building an AG-UI backend is: your agent keeps its own brain and its own recipes, but every meaningful moment — it started, it is talking, it is calling a tool, its state changed, it finished — comes out as a standard ticket that any frontend already knows how to read.

Where this Act has taken you

A quick recap before we start printing tickets. In the previous chapter you met the AG-UI event model: the roughly sixteen standard event types an agent can emit, grouped into lifecycle (RUN_STARTED, RUN_FINISHED), text (TEXT_MESSAGE_START / CONTENT / END), tools (TOOL_CALL_START / ARGS / END / RESULT), and state (STATE_SNAPSHOT, STATE_DELTA). You learned that the wire type values are SCREAMING_SNAKE_CASE and the JSON field names are camelCase. This chapter turns that vocabulary into a running program: you are going to build the thing that emits those events.

AG-UI — the Agent-User Interaction Protocol, an open standard from the team behind CopilotKit — is the third protocol in the trio you have been assembling all book. A2A connects your agent to other agents; MCP connects it to tools; AG-UI connects it to the human at the screen. And like those siblings, AG-UI is a contract about messages on a wire, not a runtime. It tells you the shape of the tickets; it does not cook the food.

The shape of a backend

An AG-UI backend is astonishingly small. It is one HTTP endpoint that accepts a RunAgentInput — the request the frontend POSTs, carrying the conversation so far (its fields are threadId, runId, messages, state, tools, context) — and streams back a sequence of events. Two ideas from earlier in the book make this click. First, from the generators chapter: a function with yield hands back values one at a time instead of building a whole list, which is precisely how you stream. Second, from async and streaming responses: the web framework keeps the connection open and flushes each yielded chunk to the browser as it appears.

The Python SDK packages this. You import event classes from ag_ui.core, wrap each one in an EventEncoder — the helper that turns an event object into the exact data: {json}\n\n text a Server-Sent-Events stream expects — and hand the generator to a FastAPI StreamingResponse. Note

That endpoint only streams text. Reporting a tool call and syncing state are two more clusters of events emitted from the same generator, at the moments your agent loop decides. We look at those next — and then run the whole thing.

Reporting a tool call, in four tickets

When your agent decides to call a tool, the UI wants to show that — "looking up invoice INV-204…" — long before the result lands. AG-UI reports a single tool call as an ordered burst of events, mirroring exactly how the model itself streams a tool call: name first, arguments dribbling in as text, then the outcome. A tool call on the wire is four events sharing one toolCallId:

  • TOOL_CALL_START — carries the toolCallId, the toolCallName (which the UI shows immediately), and an optional parentMessageId tying it to the message it belongs to.
  • TOOL_CALL_ARGS — one or many, each a delta string that is a fragment of the arguments JSON. The frontend concatenates the fragments the same way it concatenates text deltas. This is why arguments can stream before they are even valid JSON.
  • TOOL_CALL_END — the arguments are complete; the call is fully described.
  • TOOL_CALL_RESULT — carries the toolCallId plus the content your tool actually returned. (Whether the backend runs the tool or the frontend does is a design choice AG-UI supports both ways — but the result is reported with this event either way.)

Notice the symmetry with native tool calling: the model emits a tool call as start/args/end; AG-UI simply forwards that shape to the UI so the human watches the same handshake the model is having.

Syncing state: the whole photo, or the edit

The last cluster is shared state — the structured data both sides care about (a form being filled, a document being drafted, a running total). AG-UI gives you two ways to ship it, and choosing between them is a bandwidth decision you will make constantly.

A STATE_SNAPSHOT carries the entire state object in one event — the whole photograph. Simple, unambiguous, and exactly what you want at the start of a run or when things have drifted. But if your state is a large document and you changed one field, resending all of it every keystroke is wasteful. So there is STATE_DELTA: its delta is a JSON Patch array (the RFC 6902 standard — a list of tiny {"op", "path", "value"} operations), describing only what changed. The UI applies the patch to the state it already holds. Snapshot is "here is the whole photo"; delta is "on the photo you have, change this one thing." Send a snapshot occasionally to stay in sync; send deltas in between to stay cheap.

The same emitter, in front of anything

The quiet superpower of that generator is what it is not coupled to. fake_agent could be a raw LLM stream, a hand-written ReAct loop, CrewAI, LangGraph, or Pydantic AI. AG-UI sits in front of your agent as a translation layer: your loop does its thing, and at each meaningful moment you emit the matching ticket. This is why a team can standardise on AG-UI without betting on any single orchestration framework — the frontend never learns which engine is behind the curtain. Swap the engine; keep the printer.

Who owns the standard

Provenance matters for a protocol, so here it is plainly. AG-UI is an open, MIT-licensed protocol created by CopilotKit — the company behind the popular React library of the same name for building in-app AI copilots. They shipped the frontend for years, kept re-inventing the agent-to-UI glue for every backend, and eventually extracted that glue into a named, versioned standard so the whole ecosystem could share it. It is deliberately positioned as the third member of a trio: MCP (agent-to-tools, from Anthropic), A2A (agent-to-agent, now under the Linux Foundation), and AG-UI (agent-to-user). Official SDKs exist for Python and TypeScript, and the protocol reached its 1.0 line in 2026 — the release that removed the old THINKING_* events in favour of REASONING_*. Integrations ship for CrewAI, LangGraph, Mastra, Agno, Pydantic AI, LlamaIndex and more, which is the real signal: a protocol is only a standard if more than its author builds on it.

An honest maturity check

Now the part the launch blog skips. The thing AG-UI does, it does well and it does only that: emitting well-formed events is genuinely easy — the SDK is thin, the encoder is a one-liner, and you saw the whole backend fit on one screen. But that ease hides the actual hard problem. AG-UI does not decide when to emit anything. Knowing that the model just started a tool call, that a token arrived, that state changed and by how much — that is your agent loop's job, and AG-UI supplies none of it. It is the ticket-printer, not the kitchen. If your loop never tells the printer a tool call started, no TOOL_CALL_START ticket prints, and the user stares at a frozen screen wondering what the agent is doing.

So be clear-eyed about the boundary. AG-UI gives you the streaming contract between agent and UI: a stable, portable, framework-agnostic wire format any conforming frontend can render. It does not give you orchestration, a control loop, checkpointing, durable state, or branching — you either write those yourself or bring a framework (LangGraph, deep agents) that does. Comparing AG-UI to LangGraph is a category error. The ecosystem is young but the shape is stable, and the reason to learn it now is the reason you install a standard ticket-printer: so any waiter can read your kitchen.

In the wild

You will meet an AG-UI backend first as a single POST route — often literally @app.post("/") — in a file named agent.py or server.py, with ag_ui.core and ag_ui.encoder in the imports and a StreamingResponse(..., media_type="text/event-stream") at the bottom. In the browser dev-tools Network tab, the tell is a request whose response streams data: {"type":"TEXT_MESSAGE_CONTENT",...} lines rather than returning all at once. In a merge request, "expose the research agent over AG-UI" means: wrap the existing loop in this generator, emit lifecycle plus text plus tool events at the right moments, return a streaming response. And when a teammate says "the UI spinner never stops," you now know exactly which ticket nobody printed — the terminal RUN_FINISHED.

Check yourself

1. What is the minimal shape of an AG-UI backend?

An AG-UI backend is just one POST endpoint returning a streaming response of events. Graphs, checkpointers and durable state are orchestration concerns AG-UI deliberately leaves to you or to a framework you bring.

2. To stream a reply token by token, which event do you emit repeatedly?

A message is one START, many CONTENT deltas, and one END. Each CONTENT carries a chunk in its delta field, and they share a messageId so the UI appends them into the same bubble.

3. A tool call is reported to the UI as which sequence?

The four tool events share a toolCallId. START gives the name, ARGS stream the arguments as concatenable JSON fragments, END closes the arguments, and RESULT carries the returned content. This mirrors how the model itself streams a tool call.

4. Your shared state is a large document and one field changed. Which is the efficient choice, and what is it?

STATE_DELTA sends only what changed as a JSON Patch the UI applies to state it already holds — "change this one thing on the photo you have." STATE_SNAPSHOT sends the whole object, which is simple but wasteful for big state with tiny edits.

5. Which statement is the honest description of what AG-UI does and does not do?

Emitting well-formed events is easy; knowing when to emit them is orchestration, which AG-UI does not supply. It gives you the streaming contract to the UI and nothing more — you bring the loop, or a framework that has one.

Key takeaways

  • An AG-UI backend is one HTTP POST endpoint: it accepts a RunAgentInput and streams back a sequence of standard events — usually a FastAPI route returning a StreamingResponse with media type text/event-stream.
  • The Python SDK is thin: import event classes from ag_ui.core, wrap each in an EventEncoder, and yield the encoded frames from a generator. Python takes snake_case kwargs; the wire stays camelCase.
  • Stream text as one TEXT_MESSAGE_START, many TEXT_MESSAGE_CONTENT deltas, one TEXT_MESSAGE_END; report a tool call as START to ARGS to END to RESULT, all sharing a toolCallId.
  • Sync state two ways: STATE_SNAPSHOT ships the whole object; STATE_DELTA ships only what changed as a JSON Patch (RFC 6902). Snapshot to resync, deltas to stay cheap.
  • The emitter is framework-agnostic — the same generator can front a raw LLM loop, a ReAct loop, CrewAI or LangGraph — which is why a team standardises on it instead of betting on one heavy graph framework.
  • Be honest about the boundary: AG-UI standardises the streaming contract to the UI but supplies no loop, orchestration, checkpointing or durable state. Deciding when to emit is your agent's job — AG-UI is the ticket-printer, not the kitchen.
