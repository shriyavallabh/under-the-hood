Subgraphs and multi-agent patterns

A compiled graph is just a node waiting to happen. Once you see that, supervisors, swarms, and whole hierarchies of agents stop being architecture-diagram mysticism and become three arrangements of the same trick.

~30 minChapter 64 of 117
The story

You walk into a large bank branch with a messy request: close a fixed deposit, wire part of it abroad, and update your address. The woman at the front desk does none of these things herself. She listens, then routes you: "FD counter first. Then forex, desk seven." Each officer has their own desk, their own drawers of forms, their own screen — the forex officer never sees the FD officer's paperwork, and would only be confused by it. But they all write into one thing: your customer file, which travels with you from desk to desk.

Sometimes the front desk is not involved at all. Mid-way through your wire transfer, the forex officer realises you need a compliance sign-off — and simply walks you across to the compliance desk herself. Colleague to colleague. No trip back to reception.

And here is the detail everyone misses: each officer at each desk is running a little procedure of their own — verify, stamp, sign, file — a routine inside the bigger routine of the branch. The branch is a graph. Every desk is a graph. Desks pass you around either by a router's rules or by a colleague's judgement. Your file is shared state; their drawers are isolated state. This chapter is that branch, in code.

From one graph to graphs of graphs

Quick recap of where you stand, because this chapter is the payoff for the last five. In StateGraph anatomy you built a graph: a shared state object, nodes that return updates to it, edges that say what runs next. In conditional edges you let the graph choose its own path. In checkpointing you gave it a save file, and in the interrupts chapter you learned to pause it mid-run and resume with a human's answer. Every one of those tricks carries forward unchanged today.

Now the one idea this whole chapter stands on. Ask yourself: what is a node, really? It is anything that takes the current state and returns an update to it. State in, updates out. And what does a compiled graph do when you invoke it? It takes state in and returns state out. The shapes match. Which means a finished, compiled graph can be dropped into a bigger graph as a node — no adapter, no ceremony.

That embedded graph is called a subgraph — a complete graph used as a single step inside a larger one. From the outside, the parent graph sees one box called research. Inside that box, an entire multi-node pipeline runs: search, filter, summarise, loop if needed. The parent neither knows nor cares.

a graph as a node — the two-line version
research = research_builder.compile()   # a whole graph, finished

team = StateGraph(State)
team.add_node("research", research)      # ...used as ONE node

There are two ways to wire a subgraph in, and the difference is exactly the shared-file-versus-private-drawer split from the bank branch:

  • Shared keys. If the subgraph's state schema shares keys with the parent's (say, both have a messages list), you pass the compiled subgraph straight to add_node, as above. They read and write the same customer file.
  • Different schemas. If the subgraph keeps its own private state shape, you wrap it in an ordinary node function: translate the parent's state into the subgraph's input, invoke the subgraph, translate the result back. The drawer stays private; only the distilled outcome lands in the shared file.
Note

This is why LangGraph never needed a separate "multi-agent framework" bolted on. An agent, as you built it in the ReAct chapter, is a loop — and in LangGraph a loop is a small graph. So "multiple agents" is just "multiple subgraphs", and everything you already know about state, edges, and checkpoints applies to teams of agents for free.

The vocabulary of teams

Six terms carry every multi-agent conversation you will ever sit in. Flip until fluent:

Pattern one: the supervisor

The supervisor pattern is hub-and-spoke: one router agent in the middle, specialists around it, and control returning to the middle after every job. The supervisor is usually the cheapest kind of agent imaginable — an LLM call whose only output is a name: "research", "write", or "FINISH". It is a decision, not a worker. (In practice you force that decision through structured output so it cannot ramble.)

Watch one ticket move through the branch:

1. The task lands on the supervisor's desk

The parent graph's entry edge points at the supervisor. The specialists — each a subgraph — sit idle. The shared state so far holds only the task.

One LLM call reads the state and returns a worker's name. No searching, no writing — routing is its entire job. Small job, small prompt, small bill.

3. The specialist runs its own private pipeline

Researchsearch → filter → summarise

Inside the box, a whole graph executes — its own nodes, its own loops, its own scratch state. The parent only ever sees the box. This is the subgraph earning its keep.

4. Findings go into the shared file; control returns

A fixed edge leads from every worker back to the supervisor. The researcher's forty raw web pages stay in its drawer; only the summary enters the shared state.

The supervisor routes to the writer, gets the draft back, and finally answers FINISH — which routes to END. Decide, delegate, collect, repeat. It is the ReAct loop one level up: tools replaced by whole agents.

Pattern two: the swarm — handoffs without a boss

The supervisor pattern has a tax built in: after every job, control returns to the hub, which means one extra LLM call per hop and a hub prompt that must understand everything. The swarm pattern removes the hub. Agents pass the conversation directly to each other using a handoff — a special tool an agent can call whose effect is "transfer this whole conversation to my colleague". Whoever currently holds the conversation owns it, answers the user directly, and decides when a colleague would serve better.

1. The billing agent owns the conversation

User"Why was I charged twice?"
→
Billing agenttools: lookup_invoice, handoff_to_refunds
Refunds agent

No front desk. The user talks straight to a specialist, which carries its normal tools plus one odd one: a handoff tool for each colleague it may pass work to.

2. The job drifts outside its lane

User"...so refund the duplicate"
→
Billing agentcalls handoff_to_refunds
Refunds agent

The billing agent recognises this is refunds territory — by judgement, mid-conversation, exactly like the forex officer walking you to compliance. It calls the handoff tool instead of answering.

3. Control jumps sideways, not upward

Billing agent
→
Refunds agentinherits the conversation

Under the hood the handoff tool returns a routing command naming the destination agent, and the graph transfers control there — sibling to sibling in the parent graph, no supervisor consulted, no extra routing call billed.

4. The refunds agent keeps the chair

Refunds agent"Refund issued. Anything else?"
→
User

The swarm remembers who is active: the user's next message goes straight to the refunds agent, not back to billing. Ownership persists until someone hands off again.

In LangGraph terms, a handoff tool returns a Command — the same routing object you will meet again below — that names the destination node; when the handing-off agent is itself a subgraph, the command targets the parent graph so control can jump to a sibling. LangChain also ships prebuilt libraries for both patterns (langgraph-supervisor and langgraph-swarm) so you rarely wire either by hand — check the current docs for the exact constructors before you build.

Choosing between them is a management-style question. A supervisor gives you predictability: every decision passes one desk, so tracing "why did the writer run twice?" means reading one agent's log. A swarm gives you fluidity and fewer hops — but "who decided that?" now has as many answers as you have agents. And the two compose: a supervisor's specialists can themselves be supervisors of smaller teams, which is all a hierarchical team is — subgraphs inside subgraphs, the branch inside the bank's regional network.

Shared versus isolated state

The customer file versus the desk drawers. Getting this split right is most of multi-agent design, so make it explicit:

Shared state (the file)Isolated state (the drawer)
What lives thereThe task, the conversation, distilled results every agent may needRaw tool output, scratch work, retries, a specialist's inner monologue
Who sees itEvery agent in the teamOnly the subgraph that owns it
How it is wiredSubgraph shares state keys with the parent (same messages key)Subgraph has its own schema; a wrapper node translates in and out
Failure smell when misusedEvery agent drowns in everyone else's clutter; prompts balloon; the writer quotes raw HTMLAn agent is missing facts it genuinely needed; teammates repeat each other's work

The default that serves most teams: share the conversation, isolate the work. A researcher should chew through forty pages in its own drawer and file a one-paragraph summary. If instead every raw page lands in shared state, you have rebuilt the original problem — one giant context — and paid for a team on top. When you meet deep agents in the subagents chapter, you will find this exact principle running Claude-Code-style harnesses: a subagent burns thousands of tokens privately and returns one tidy report.

The real thing: a supervisor team in LangGraph

Here is the pattern in genuine LangGraph 1.x, condensed to its skeleton. Click the annotated lines — by now, most of them should feel like old friends wearing team jerseys.

team.py — a graph made of graphs
from langgraph.graph import StateGraph, START, END
The same three imports as every graph since ch58. Nothing new is needed to go multi-agent — that is the point.
from langgraph.types import Command
Command lets a node return "and go THERE next" along with its state update — routing decided inside the node, no separate conditional edge required. Its cousin Command(resume=...) restarted interrupted runs in ch62.
rb = StateGraph(State)
A builder for the research team's own graph. State here is the shared schema — this subgraph will read and write the same messages key as the parent.
rb.add_node("search", search)
rb.add_node("summarise", summarise)
rb.add_edge(START, "search")
rb.add_edge("search", "summarise")
research = rb.compile()
Compile the subgraph exactly as you would any graph. The result is a runnable: state in, state out. Hold that thought for four lines.
def supervisor(state: State) -> Command:
The supervisor is a plain node function that returns a Command. The type hint advertises the contract: this node's job is a routing decision.
nxt = pick_next(state) # an LLM call returning a name
One structured-output LLM call (ch36) constrained to the worker names or FINISH. The whole intelligence of the hub is this one line.
if nxt == "FINISH":
return Command(goto=END)
Routing to END stops the run — the team's FINAL ANSWER moment, decided by the hub rather than by a worker.
return Command(goto=nxt)
Otherwise: hand control to whichever specialist was named. Because the routing lives in the returned Command, you do not write conditional edges for the supervisor at all.
tb = StateGraph(State)
tb.add_node("supervisor", supervisor)
tb.add_node("research", research) # a compiled graph as a node
The headline line of the chapter. The compiled research graph is passed to add_node like any function — legal because their schemas share keys. The parent will treat the entire pipeline as one step.
tb.add_node("write", write)
Specialists can be subgraphs, single functions, or full create_agent agents from ch54 — a node is a node.
tb.add_edge(START, "supervisor")
tb.add_edge("research", "supervisor")
The hub-and-spoke shape in one line: whenever research finishes, control flows back to the supervisor. Same for every worker.
tb.add_edge("write", "supervisor")
team = tb.compile(checkpointer=saver)
And because a team is just a graph, everything from this act comes free: checkpointing persists mid-team runs, interrupts can pause any specialist for approval, and streaming shows you every hop live.

When a team genuinely beats a solo agent

Now the judgement call this chapter's title promises, because "we made it multi-agent" is somehow both a brag and a confession. There are exactly three honest reasons to split one agent into several:

  • Context isolation. One agent doing research, analysis, and writing carries all three jobs' clutter in every single LLM call — and as you saw in the memory chapter, bloated context degrades quality while raising cost. Specialists each see a small, job-shaped context. This is the big one.
  • Parallelism. Independent work can fan out simultaneously — three researchers on three subtopics, results merged downstream, using the same branching machinery you met in why graphs. A solo agent does everything in single file.
  • Permissions. Different agents can hold different tools. The researcher gets read-only search; only the executor holds the tool that can send email or touch a database — and that one node is where you attach the interrupt. A security boundary you can point to in a review beats a paragraph of prompt begging the model to be careful.

Set against those wins, the costs are just as concrete. Every hop through a supervisor is an extra LLM call. Every handoff risks the telephone game — the writer receives a summary of a summary, and the nuance the client actually cared about dies in transit. And debugging stops being "read one transcript" and becomes "reconstruct a conversation between four transcripts". Illustrative round numbers — not benchmarks — to make the trade-off visible:

Careful

The failure mode to watch for in real systems is the org-chart agent team: one agent per department because the diagram looked impressive, each hop adding latency, cost, and a chance to lose information — for a task a single well-tooled agent handled fine. The honest default is one agent until something measurable forces the split: a tool menu the model fumbles, a context that will not fit or visibly degrades quality, or a permission boundary you must be able to point to. Split along those seams, never along job titles.

In the wild

You will meet subgraphs the moment a LangGraph repo grows past one file: look for compile() calls in modules like research_graph.py whose results get add_node-ed in a top-level team.py or supervisor.py. In merge requests, "extract the retrieval flow into a subgraph" is the agent world's "extract this into a function" — same instinct, one level up. Handoffs appear as tools named transfer_to_* or handoff_to_* in customer-support codebases. And when a colleague at a large bank says "our harness runs a planner that dispatches to worker agents", you can now translate instantly: supervisor pattern, workers as subgraphs, shared messages, isolated scratchpads — and you know exactly which two questions to ask next: what state do the workers share, and who holds the dangerous tools?

Check yourself

1. What makes it possible to use a whole compiled graph as a single node in a bigger graph?

No adapter is needed — add_node accepts anything callable with the node contract, and an invoked graph satisfies it. Composition falls out of the design rather than being a bolted-on feature.

2. In the supervisor pattern, what does the supervisor actually do?

The supervisor routes; it never does specialist work. That is why it can run on a small prompt (often a single structured-output call) and why every worker has an edge leading back to it.

3. What distinguishes a swarm from a supervisor team?

A swarm is built entirely from peer-to-peer handoffs — the forex officer walking you to compliance. Whoever holds the conversation owns it until they hand off, and the user's next message goes to the current owner, not back through a hub.

4. Your writer agent keeps producing confused drafts, and you notice the shared state contains forty raw web-page dumps from the researcher. What is the fix?

This is a state-design bug, not a model or prompt problem. Raw tool output belongs in the specialist's private drawer; only the distilled summary belongs in the shared file. Share the conversation, isolate the work.

5. Which of these is a genuine reason to split one agent into several?

Permissions — along with context isolation and parallelism — are the three honest wins. More agents mean more LLM calls, more latency, and more places to lose information, so each split must earn its keep against a measurable problem.

Key takeaways

  • A compiled graph is state-in, state-out — the same shape as a node — so graphs compose into bigger graphs with a plain add_node, and "multi-agent" is just "multiple subgraphs".
  • The supervisor pattern is hub-and-spoke: a router agent that only decides who works next, with every worker's edge leading back to it. Predictable, traceable, one extra LLM call per hop.
  • The swarm pattern removes the hub: agents transfer the conversation peer-to-peer via handoff tools, and whoever holds it owns it until the next handoff. Fewer hops, fuzzier accountability.
  • Share the conversation, isolate the work: distilled results go in shared state, raw dumps and scratch work stay in each subgraph's private schema.
  • Multiple agents genuinely beat one for exactly three reasons — context isolation, parallelism, and permission boundaries; absent those, every added agent is added cost, latency, and telephone-game information loss.
  • Because a team is just a graph, everything from this act transfers for free: checkpointing persists team runs, interrupts gate the dangerous specialist, and streaming narrates every hop.
