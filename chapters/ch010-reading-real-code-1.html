The story

You are in a taxi from the airport, in a country whose alphabet you have been learning for two months. Flashcards over morning tea. Letter drills on the commute. Vowel marks that hang off consonants like earrings. It has felt, honestly, a little abstract — squiggles on cards, graded exercises, nothing real.

Then, at a red light, a glowing sign above a small shop. You do what learners do: you sound it out. First letter. Second. Third — and the sounds snap together into a word you already know: coffee. Nobody translated it for you. Nothing about that sign was designed for beginners. You read it because you spent two months collecting pieces, and at that traffic light the pieces quietly assembled themselves.

This chapter is that taxi ride. You have spent nine chapters on the alphabet of Python. Below is a real sign — an actual script that sends a question to a commercial AI model over the internet. Same drill: sound it out. You are about to discover you already read it.

Meet the script

Here it is: about twenty lines that send one question to a hosted language model and print the answer. Do not study it yet. Just look at it the way chapter 1 taught you — let your eyes find the shapes. Try to spot: exactly one function definition, two named dicts — plus two more little dicts hiding inside a list — one f-string, and the single line where anything actually leaves your computer.

ask.py — a real script that asks an LLM one question
import os
import requests

API_KEY = os.environ["OPENAI_API_KEY"]
URL = "https://api.openai.com/v1/chat/completions"

def ask(question):
    headers = {"Authorization": f"Bearer {API_KEY}"}
    payload = {
        "model": "gpt-4o-mini",
        "messages": [
            {"role": "system", "content": "Answer in one short paragraph."},
            {"role": "user", "content": question},
        ],
    }
    response = requests.post(URL, headers=headers, json=payload, timeout=30)
    response.raise_for_status()
    data = response.json()
    return data["choices"][0]["message"]["content"]

answer = ask("Why is the sky blue?")
print(answer)
Note

The two import lines at the top are the one thing we have not formally covered — they are the star of the very next chapter. For today, read import os as "bring me the toolbox named os". The second toolbox, requests, is the most popular Python library for talking to the web. Also: the model name is just a string, and providers rotate model names every few months — swap that one string and the rest of the script does not change.

Six words from the internet half

Half of this script is Python you know. The other half is web plumbing, and it needs only six words. Flip each card until they feel obvious — they will follow you through the whole course:

One of those six deserves a second look: the messages list — the list of dicts inside the payload, where each dict is one turn of a conversation with a "role" and a "content". This exact shape appears in every provider's API and every framework you will ever meet. A system turn carries the standing instructions; a user turn carries the question. Learn this shape once and you will recognise it for the rest of your career.

Every line, one at a time

Now the slow tour. Click each line and check that the note matches what you already expected — the goal is not to learn the lines but to notice that every single one is built from chapters 1 to 9. Boring is the target. Boring means fluent.

ask.py — read with nine chapters of Python
import os
Bring in the toolbox named os — Python's operating-system tools. How importing really works is the next chapter's whole business; for now, "fetch that toolbox" is a perfectly good reading.
import requests
A second toolbox: requests, the de-facto standard library for web calls. It is not built into Python — someone installed it — which is exactly why this line can fail with an error you now know how to read (chapter 9).
API_KEY = os.environ["OPENAI_API_KEY"]
A name stuck on a value (chapter 2). The value comes from os.environ, which behaves like a dict of the terminal's environment variables — so the square-bracket lookup is pure chapter 4. The secret was exported in the terminal before the run; it never appears in the file. If the variable is missing, this raises a KeyError and the run dies with a traceback on line 4.
URL = "https://api.openai.com/v1/chat/completions"
Just a string (chapter 3) holding the endpoint address. The ALL-CAPS names are a convention, not a rule: "set once at the top, treat as fixed".
def ask(question):
A signature you can read in your sleep after chapter 7: define a function called ask with one required parameter, colon, body indented below. Nothing inside runs yet — this is a recipe being filed, not cooked.
headers = {"Authorization": f"Bearer {API_KEY}"}
A one-entry dict whose value is an f-string (chapter 3) stitching the key after the word Bearer — the exact wording this provider's door expects. This is the envelope label: "here is who I am".
payload = {
An opening brace with nothing after it: this dict is split across several lines for readability. Multi-line dicts read exactly like one-line dicts — just match the braces, as you practised in chapter 5.
"model": "gpt-4o-mini",
First key-value pair: which model should answer. A plain string. Swapping models is editing this one value.
"messages": [
Second key: the conversation itself. Its value is a list — the bracket opens here and closes three lines down (chapter 4).
{"role": "system", "content": "Answer in one short paragraph."},
One dict per conversation turn. The system role carries the standing instructions — the model's briefing before the user says anything.
{"role": "user", "content": question},
The user's turn. Look closely: question has no quotes. It is the function's parameter (chapter 7) — whatever text the caller passed in lands here. This is the exact spot where your words enter the payload.
],
The list of messages closes.
}
And the payload dict closes. Read the structure back: a dict holding a string and a list of dicts. That is precisely the JSON shape from chapter 5, still wearing its Python clothes.
response = requests.post(URL, headers=headers, json=payload, timeout=30)
The only line that touches the internet. A function call with one positional and three keyword arguments (chapter 7). json=payload converts the dict to JSON text and sends it; timeout=30 means "give up after 30 seconds rather than hang forever". Every line before this was packing luggage. This line is the flight.
response.raise_for_status()
Checks the status code on the reply. If the provider said 401 (bad key), 429 (slow down) or 500 (their side broke), this raises an exception and the chapter 9 traceback machinery takes over. If the status is 200 it does nothing at all — a silent guard.
data = response.json()
The reply arrived as JSON text; .json() parses it back into Python dicts and lists (chapter 5, in reverse). From here on there is no more network — just containers.
return data["choices"][0]["message"]["content"]
Four hops of digging, all chapter 4: the key "choices" holds a list; [0] takes its first item; inside that, "message", then "content" — the model's actual words. return hands them back to whoever called.
answer = ask("Why is the sky blue?")
Back at zero indentation, so this runs at the top level (chapter 1). The recipe finally gets cooked: ask is called with a literal question, and the name answer is stuck on whatever comes back (chapter 2).
print(answer)
The model's words land on your screen. That is the whole script.

Count what just happened. Nineteen lines of production-shaped code, and the notes needed nothing beyond chapters 1 to 9 plus six words of web vocabulary. That foreign sign at the traffic light? You just read it.

Run it — with a stunt double for the internet

Reading is good; running is better. This page cannot call a paid API — there is no network here and no real key — so the internet is played by a stunt double: a function called fake_post that stands in for requests.post and returns a canned reply in the provider's exact JSON shape. Everything else — the key from the environment, the headers, the payload, the digging — is identical to the real script. Notice one more honest touch: raise_for_status() is unmasked into the plain if + raise it secretly is (chapter 6 meets chapter 9). Press Run.

Tip

Now sabotage it — that is where the learning lives. First, change the planted key to "wrong-key" and run: you get a 401, the RuntimeError fires, and you read the traceback bottom-up exactly as chapter 9 drilled. Second, change the question and watch the stunt double improvise. Third, ask two questions by calling ask inside a for loop over a list of questions (chapter 6). Every break you cause here is one you will recognise in real logs later.

Change the provider, keep the skeleton

Here is the quiet superpower you just earned: this skeleton barely changes between providers. Point the same twenty lines at a different company's model and only the labels move — the dict-building, the POST, the status check, and the digging are identical. In the wild

You will meet this exact shape constantly: it is the first code block in every model provider's quick-start guide, the bottom of countless READMEs, and the entire contents of little utility files named ask.py, llm.py or chat.py in real repositories. In larger codebases the skeleton hides inside an SDK call, but the messages list inside is byte-for-byte the shape you read today. And when a script wants all the candidate answers instead of just the first, you will spot a chapter 8 comprehension doing the digging: [c["message"]["content"] for c in data["choices"]]. In a merge request, "the key is hardcoded, move it to an env var" is a reviewer pointing at line 4 of this very pattern.

Check yourself

1. Why does the script read the API key from os.environ instead of typing the key directly into the file?

A key in the file travels everywhere the file travels — into git history, code reviews, and backups. The environment keeps the secret on the machine and out of the code. The provider cannot tell the difference; your security team certainly can.

2. In pure chapter-4 terms, what is the payload the script sends?

The outer braces make a dict with two keys: "model" (a string) and "messages" (a list of dicts, one per conversation turn). It only becomes JSON text at the moment requests.post sends it — chapter 5's costume change.

3. What does response.raise_for_status() do when the call succeeded with a 200?

It is a silent guard: on an error status it raises an exception (cue the chapter-9 traceback), and on success it steps aside and does nothing. Parsing happens on the next line, in response.json().

4. In data["choices"][0]["message"]["content"], what is the [0] doing?

"choices" holds a list because the API can return several candidate answers. [0] takes the first one — plain chapter-4 list indexing, sandwiched between two dict lookups.

5. Which single line of ask.py actually touches the internet?

Everything before it builds local containers; everything after it unpacks local containers. The POST is the one and only flight — which is why timeouts, retries, and network errors all concentrate on that line.

Key takeaways

  • A real LLM call is about twenty lines: fetch a key, name an endpoint, build a dict, POST it, check the status, dig the answer out of the reply.
  • The API key lives in the environment, never in the file — secrets in code get committed and leaked, and reviewers will flag it on sight.
  • The messages list — dicts with a "role" and a "content", one per turn — is the universal conversation shape; every provider and framework you will ever meet uses it.
  • The payload is a dict that becomes JSON on the way out; the reply is JSON that becomes a dict on the way in. Reading API code is mostly reading containers.
  • Exactly one line touches the network; raise_for_status() is an if + raise in a trench coat, and its failures arrive as the tracebacks chapter 9 taught you to read.
  • You read every line with nine chapters of Python and six words of web vocabulary — no new syntax. That is the proof this is working.
