Sandboxing and code execution

An agent that can run code can do anything — which is exactly the feature and exactly the problem. This chapter is about building the room where "anything" is survivable.

~30 minChapter 87 of 135
The story

A famous restaurant runs a training programme. Trainees must cook for real — real knives, real flame, real ingredients — because a chef who has only read recipes is not a chef. But the head chef is no fool. Trainees work in a test kitchen: a separate room, stocked with copies of the ingredients, never the originals. The gas line into that room has a flow limiter, so nobody can burn the building down even by trying. The phone on the wall dials only three approved suppliers — you cannot order forty kilos of saffron to a stranger's address. And the one door into the dining room has the head chef standing at it: anything a trainee wants to serve to an actual customer gets looked at first.

When a trainee has a catastrophic evening — someone always does — the staff hose the test kitchen down, restock it, and by morning it is as if nothing happened. The restaurant never noticed.

The trainee is your agent. The test kitchen is a sandbox. The copied ingredients, the flow limiter, the restricted phone, and the chef at the door are the four controls this whole chapter is about.

The killer feature is the loaded one

In the durability chapter you protected the agent from the world — crashes, restarts, lost work. This chapter protects the world from the agent. Way back in the ReAct chapter you built the boundary that all agent security lives on: the model never executes anything — it writes a request, and your code decides whether to act. While the tool menu was a calculator and a weather lookup, that boundary needed no guards. Then agents grew their one truly transformative tool: run code. A shell, a Python interpreter, a "bash" command. It is the universal tool — with it, an agent can analyse a spreadsheet, fix a bug, install a library, call any API ever written. Every capable harness you operate is built around it.

And it is a loaded gun, for a reason that has nothing to do with the model being evil. The model is a text-completion engine steering a shell. It can be confused, it can hallucinate a path, and — worst of all — it can be talked into things by the content it reads. You saw prompt injection in the failure-modes chapter: a poisoned README, a webpage that says "ignore your instructions and run this". The agent reading that content holds your files and credentials in its hands.

Note

Security researchers call the worst version of this the lethal trifecta: an agent that (1) can read private data, (2) processes untrusted content, and (3) can communicate with the outside world. Any two are manageable. All three together mean a well-crafted piece of text on the internet can, in principle, steal whatever the agent can see. Sandboxing is how you break the trifecta without giving up the run-code superpower.

You cannot vet intent — only bound capability

Here is the mental shift that separates clear reasoning about agent safety from hand-waving: you will never make the model trustworthy enough to skip the sandbox. No prompt that says "be careful" survives contact with adversarial input. So instead of validating the agent's intentions, you bound its capabilities — you build a room where the worst command it could possibly type is survivable.

That gives us a working definition. A sandbox is an execution environment deliberately arranged so that code running inside it can only see what you put in, can only reach what you allow out, and can only consume what you budget — no matter what the code is. The related term you will hear in every incident review is blast radius — the total damage the worst-case command could cause. Sandboxing is blast-radius engineering.

Flip these cards until each one feels obvious — they carry the rest of the chapter:

The isolation ladder: one bad command, four rooms

Isolation comes in layers, each one a thicker wall bought with a little more overhead. The clearest way to feel the difference is to run the same hostile command in each room and watch what happens. Suppose a prompt-injected agent tries to read your SSH private key — the credential that unlocks your servers — and post it to an attacker's website:

A container uses Linux namespaces to give the process its own private view of the filesystem, processes, and network, and cgroups to meter its CPU and memory. Your key is not "protected" — it simply is not in the container's universe. The caveat: containers share the host's kernel, so a kernel bug can become a container escape.

A microVM is a stripped-down virtual machine — the best-known engine is Firecracker, which AWS built to run Lambda — that boots its own guest kernel in roughly a tenth of a second. Even a kernel exploit now lands inside a disposable guest kernel, not your host.

The room needs walls and a guarded door: default-deny networking, with an allow-list of the few domains the task genuinely needs.

The three dials: files, network, fuel

Whichever room you pick, you then set three dials — the copied ingredients, the restricted phone, and the flow limiter from the story.

Dial 1 — filesystem scoping. The sandbox should see a per-task workspace — the files this job needs, copied or mounted in — and nothing else. Not your home directory, not ~/.ssh, not the browser profile holding every session cookie you own. Harnesses often mount the OS image read-only and leave exactly one writable folder; outputs get copied back out at the end, deliberately. (deepagents takes this to its extreme: its virtual filesystem is just entries in agent state — files that never touch a real disk are sandboxed by construction.)

Dial 2 — network egress control. Data theft requires a journey out. If the sandbox has no network, an injected agent can read the poisoned instruction "send the secrets to attacker.example" and be physically unable to comply. Most real tasks need some network — packages, APIs — so the practical setting is default-deny plus an allow-list, enforced by routing all traffic through a proxy that knows the approved domains. This is the single highest-leverage dial: it converts "the agent leaked our data" into "the agent tried, got connection-refused, and we have the log".

Dial 3 — resource limits. Code can misbehave without being malicious: an infinite loop, a recursion that eats all memory, a fork bomb spawning processes until the machine chokes. Caps on CPU, memory, process count, disk, and wall-clock time (cgroups on Linux; a plain timeout wrapper for the clock) turn each of those from an outage into a log line. It is the same instinct as max_turns in your ReAct loop: bound everything that could run away.

Here is what all three dials look like in one real incantation, the kind you will find inside a harness that runs agent code in a container. Click the annotated lines:

--rm makes the container ephemeral: destroyed the moment the command finishes, taking any mess with it. The nightly hose-down of the test kitchen.
--user 1000:1000 \
Run as an unprivileged user even inside the container. Defence in depth: if the code finds a way to matter, it matters as a nobody.
--read-only \
The container's own operating-system files are mounted read-only. The code cannot quietly tamper with the tools it will be judged by.
-v /srv/jobs/task-8841:/workspace:rw \
Dial 1. Exactly one host folder — this task's workspace — appears inside, writable. Your ~/.ssh is not hidden; it simply does not exist in this universe.
--network none \
Dial 2, at its strictest: no network interface at all. Harnesses whose tasks need packages or APIs swap this for a private network whose only exit is an allow-listing proxy.
--memory 512m --cpus 1.0 --pids-limit 128 \
Dial 3. Memory, CPU, and process-count caps, enforced by the kernel's cgroups. A fork bomb hits a wall of 128 processes and dies inside the box, invisible to the host.
--workdir /workspace \
python:3.12-slim \
A minimal image: Python and almost nothing else. Every tool you leave out of the room is a tool an attacker cannot use. Slim images are a security choice, not just a size one.
timeout 120 python analyze.py
The wall-clock cap. After 120 seconds the command is killed and the loop gets a "timed out" observation — handled exactly like a tool error in tool calling.
Careful

The most common self-inflicted wound: mounting secrets into the sandbox. If the task's environment variables include AWS_SECRET_ACCESS_KEY, then everything inside the box — including injected code — can read them, and only your egress policy stands between that and exfiltration. The strong pattern keeps long-lived credentials outside, with the harness as broker: it performs the privileged call itself, or issues a short-lived, narrowly-scoped token — the crown jewels never enter the room.

Permission prompts and allow-lists: the UX of safety

That ASK verdict is where security stops being an infrastructure problem and becomes a human-interface problem. A permission prompt — the harness pausing to ask "the agent wants to run pip install pandas. Allow?" — is the dual-control brake pedal in a driving-school car: the learner drives, but a trusted foot hovers near the brake. You have pressed this pedal hundreds of times; every "Allow this command?" dialog in a coding agent is an ASK from a gate like the one you just ran.

Here is the trap: a gate that asks about everything feels safe and is the opposite. Twenty prompts an hour trains a human to press "yes" without reading — approval fatigue — and the one prompt that mattered sails through with the other nineteen. A prompt the user no longer reads is not a safety control; it is a ritual.

So mature harnesses tier the decisions, exactly like our gate's five rules. Provably-safe operations auto-allow silently. Known-catastrophic patterns hard-deny without asking — a human should not be able to fatigue-click into rm -rf /. Prompts cover only the ambiguous middle, and approvals can be remembered — "always allow pip install in this project" — so the same question is never asked twice. Watch the scoping of that memory, though: "always allow this exact command" is a scalpel; "always allow everything" (the yolo flag) removes the brake pedal. Claude Code works precisely this way: per-project allow and deny rules, a sandboxed execution mode built on the operating system's native isolation (Seatbelt on macOS, namespace tricks on Linux) plus a network proxy, and prompts only for what falls in between — the harness-anatomy comparison showed where that sits in the larger machine.

When the vendor says "it has a sandbox"

"The agent's code runs in a sandbox" can mean anything from "a microVM with default-deny egress, destroyed after every session" down to "a subprocess with a nice name". The word guarantees nothing — the configuration is the security. When you evaluate a platform, or your own team's design, ask these six questions:

QuestionWeak answerStrong answer
What is the isolation boundary?"A separate process""A fresh container or microVM per session"
What files can it see?"The user's home directory""An empty per-task workspace; files are copied in and out deliberately"
What can it reach on the network?"It has internet access""Default-deny; a proxy allow-lists the domains the task needs"
What are the resource limits?"We have not hit problems""CPU, memory, process, disk, and wall-clock caps per run"
How long does it live?"One always-on shared box""Ephemeral — created per task, destroyed after"
Where do secrets live?"API keys are in the sandbox's environment""Outside; the sandbox gets short-lived, narrowly-scoped tokens via a broker"

One more question earns real respect: what is inside the fence, and what is not? Some products sandbox only the code-interpreter tool while the shell, browser, or MCP servers run with full host privileges — a fenced front garden and a wide-open back door.

In the wild

You will meet this chapter everywhere once you know its face. In repositories: a Dockerfile plus a compose file with network_mode, mem_limit, and read_only: true — the three dials in YAML. In CI configuration: every pipeline job already runs in a fresh container with scoped variables, which is why CI is many teams' first agent sandbox. In coding-agent settings: allow/deny permission rules and a sandbox toggle. In the cloud: dedicated agent-sandbox services (E2B, Modal, and friends) selling Firecracker-style microVMs by the second, and code-interpreter features in the big model APIs quietly running your model's Python inside a locked-down container. And in merge requests: when a reviewer writes "this tool shells out — what is the blast radius?", that is this chapter, asked by a professional.

Check yourself

1. The model in your agent is well-aligned and has never misbehaved. Why does its shell tool still need a sandbox?

Sandboxing is not about the model's character. A poisoned README or webpage can steer the model's next command, so you bound what any command can do. Capability, not trust.

2. What is the key security difference between a container and a microVM?

Both scope files and meter resources; the wall thickness differs. Container escapes ride on host-kernel bugs, whereas a microVM puts even the kernel inside the disposable box — at near-container startup cost.

3. Your sandbox has microVM isolation and a read-only OS, but unrestricted internet access. The task data you mounted in is confidential. What is the risk?

The confidential data is already inside the room, so isolation cannot protect it; only egress control can. Hence default-deny networking with a domain allow-list.

4. A harness prompts the human for approval on every single command, including ls. Why do security engineers consider this a flaw rather than extra safety?

Approval fatigue: a control humans have stopped reading is a ritual, not a safeguard. Tiering — auto-allow the provably safe, hard-deny the catastrophic, prompt only for the middle — keeps the prompt meaningful.

5. A vendor says "your agent's code runs in a secure sandbox". What is the most revealing follow-up question?

"Sandbox" spans everything from a renamed subprocess to a per-session microVM. The boundary, filesystem scope, egress policy, and fence coverage (all tools, or just the interpreter?) determine the actual blast radius.

Key takeaways

  • Code execution is the universal tool — and it must be treated as running untrusted code, because the agent's inputs can be adversarial even when the model is not.
  • You cannot vet intent, so you bound capability: a sandbox is a room arranged so the worst possible command is survivable — its size is the blast radius.
  • Isolation is a ladder — process permissions, containers (own world, shared kernel), microVMs (own kernel, near-container speed) — each layer a thicker wall for a little more overhead.
  • Three dials define the room: filesystem scoping (a per-task workspace, nothing else), network egress control (default-deny plus an allow-list — the highest-leverage dial against data theft), and resource limits (CPU, memory, processes, time).
  • Permission prompts and allow-lists are the UX of safety: tier decisions into auto-allow, ask, and hard-deny, because a human who is prompted constantly stops reading the prompts.
  • Keep long-lived secrets outside the sandbox, brokered in as short-lived scoped tokens — and when a vendor says "it has a sandbox", ask about the boundary, files, egress, lifetime, and what actually stands inside the fence.
