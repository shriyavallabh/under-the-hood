Day-2 on AKS: scale, heal, secure

Launching the cluster was day 1. This is everything after: autoscaling, zero-downtime updates, health probes, keyless secrets, cost sanity, and the pipeline that ships your agent — production for real.

The story

Opening night at a new restaurant is pure adrenaline: the sign goes up, the first plates go out, everyone cheers. Then comes Tuesday. The lunch rush hits and you need four extra cooks by noon — but only until two. The menu changes, and you must swap it without closing the doors or serving anyone a half-old, half-new dish. A cook comes in pale and shaky; the head chef pulls him off the line before he touches food, and sends him home only if he actually collapses. The safe no longer has a key taped under the till — staff open it with a fingerprint, checked fresh every time. And the accountant puts each kitchen station on its own budget, because the pastry section keeps burning gas the grill section paid for.

Restaurateurs call this operations — the unglamorous craft that decides whether the place survives. Platform engineers call it day-2. Your cluster is open. Now we learn to run it.

Day 2: the day after the launch party

Quick recap of where you stand. In AKS from scratch you created a real cluster with az aks create and wired kubectl to it. In an agent in a pod your agent got a Deployment, a Service, a ConfigMap, and an MCP sidecar. kubectl fluency taught you to interrogate anything, and Kubernetes YAML, decoded taught you to read the manifests behind it all. That was day 1: making it exist.

Day-2 operations — the industry's name for everything a system needs after it first goes live — is a different discipline: handling load you did not schedule, shipping updates without downtime, keeping traffic off sick pods, holding secrets without writing them down, and paying a sane bill. The lovely thing about Kubernetes: each of these is just more YAML and a few commands — the same declare-and-reconcile idea from Kubernetes in plain terms, aimed at new problems. Let's take them in the order they will bite you.

Scaling: hiring cooks for the lunch rush

Your Deployment says replicas: 2 — a fixed headcount, fine at 9 a.m., hopeless at noon. The HorizontalPodAutoscaler (HPA) is a controller that watches a live metric — classically average CPU or memory across your pods — and adjusts the replica count between a floor and a ceiling you set. "Horizontal" means more pods, like hiring more cooks; vertical would mean giving one cook more arms — harder, and usually worse.

hpa.yaml — scale the support agent on CPU
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: support-agent-hpa
  namespace: agents
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: support-agent
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

Top to bottom: scaleTargetRef names the Deployment to resize; minReplicas and maxReplicas are the floor and ceiling; the metric says "keep average CPU at 70% of what each pod requested". Drift above 70% and pods are added; drift well below and they are removed — patiently, since scale-down waits about five minutes by default so a brief lull does not fire half your kitchen. The maths inside is one honest formula — run it:

Tip

Change TARGET_CPU to 30 and the cluster becomes jumpy and expensive; 95 makes it sluggish and cheap. That one number is the whole tuning conversation.

Now the honest caveat for agent workloads: a pod waiting on an LLM API response is mostly idle — low CPU, low memory — while its backlog of queued requests quietly explodes. For I/O-bound agents, CPU is a bad proxy for load. What you actually want to scale on is queue depth: unprocessed tasks in the queue feeding your fleet. That is what KEDA (Kubernetes Event-Driven Autoscaling) does — an open-source add-on, offered on AKS as a managed option, scaling Deployments on external signals such as Azure Service Bus queue length, even down to zero when the queue is empty. Same mental model as the HPA; the metric changes from "how hard are the cooks working" to "how long is the ticket rail".

Rolling updates: changing the menu without closing

Your agent's new image — say v13, with the improved system prompt — sits in ACR after the build from the containers chapter. Shipping it is deliberately boring: edit one line of the Deployment manifest, the image tag, and re-apply.

ship v13 — three commands, one escape hatch
kubectl apply -f deployment.yaml
kubectl rollout status deployment/support-agent -n agents
kubectl rollout history deployment/support-agent -n agents
kubectl rollout undo deployment/support-agent -n agents

Line by line: kubectl apply submits the edited manifest — Kubernetes diffs it against reality and sees only the tag changed. rollout status blocks until every new pod is up or the rollout stalls — the command your pipeline will gate on. rollout history lists previous versions — the old ReplicaSet is kept like an old menu in a drawer. And rollout undo is the escape hatch: one command, back to the previous version. What happens between apply and done is the rolling update — replacing pods a few at a time so the service never goes dark:

One line changed. Kubernetes compares desired state (v13) with reality (three v12 pods) and starts reconciling — the same loop as always.

With maxSurge: 1, maxUnavailable: 0, an extra pod is added before any old one is removed — capacity never dips. No traffic until its readiness probe passes.

3. Ready — one old pod retires

v13 reported ready, so the Service routes to it and one v12 pod drains and exits. Old and new serve side by side for a moment — your API must tolerate that.

Surge, wait for ready, retire, repeat. rollout status unblocks with a success message. Nobody noticed a thing — the whole point.

If v13 crash-loops, rollout undo replays the rolling dance in reverse, back to the stored v12 ReplicaSet. Roll-back is a first-class verb.

Careful

A rolling update protects you from crashes, not bad answers. If v13 passes its probes but the new prompt makes the agent worse, Kubernetes is delighted with it. Quality gates live in the pipeline — the evals chapter — before the rollout.

Probes: the head chef's two questions

During that rollout, Kubernetes kept asking each pod two different questions. A readiness probe asks "can you take customers right now?" — fail it and the pod is removed from the Service's routing list; no traffic, no restart, until it passes again. A liveness probe asks "are you alive at all?" — fail it repeatedly and the container is killed and restarted. The pale cook gets pulled off the line (readiness); only if he collapses is he replaced (liveness).

deployment.yaml — the probes block, line by line (click)
containers:
- name: agent
image: agentsacr.azurecr.io/support-agent:v13
The one line a release changes. Everything below is operational contract, and it rarely changes.
readinessProbe:
"May I send you traffic?" Checked for the pod's whole life — a pod that loses its database connection mid-afternoon goes unready and stops receiving requests until it recovers.
httpGet:
An HTTP GET your agent's web server answers. Your app decides what "ready" means — typically: config loaded, downstream connections alive.
path: /ready
port: 8080
periodSeconds: 5
Ask every five seconds. Keep the endpoint fast — no LLM calls inside a probe handler, ever.
failureThreshold: 2
Two consecutive failures before pulling the pod out of rotation. One hiccup is forgiven.
livenessProbe:
"Are you alive?" Failing this is fatal by design: the container is restarted. Keep the check dumb — "my process responds" — nothing more.
httpGet:
path: /healthz
port: 8080
initialDelaySeconds: 10
A startup grace period, so a slow-booting agent is not executed before it finishes waking up.
periodSeconds: 15
failureThreshold: 3
Three strikes, then restart — roughly 45 seconds of true deadness before the reboot.
Careful

The classic self-inflicted outage: putting a dependency check (is the LLM API reachable?) in the liveness probe. The dependency blips, every pod "fails", the whole fleet restarts at once, and a thirty-second wobble becomes a ten-minute outage. Dependencies belong in readiness; liveness is only for "my own process is wedged".

Workload identity: the fingerprint safe

Your agent needs secrets — the LLM API key, database passwords. You know from the YAML chapter that a Kubernetes Secret is base64-encoded, not encrypted, and from every security review ever that keys baked into images or manifests eventually leak. The grown-up answer on Azure is Microsoft Entra workload identity: a federated arrangement — two systems agreeing to trust each other's identity documents — where your pod proves who it is with a short-lived token Kubernetes itself issues, and Entra exchanges that proof for access to Azure services like Key Vault. No Azure credential ever exists in your image, manifests, or pipeline. Nothing to steal, rotate, or accidentally commit.

1. One-time setup: three trust anchors

AKS clusterOIDC issuer on
managed identitysupport-agent-id
Key Vaultrole: Secrets User

Enable the cluster's token issuer, create a managed identity (an Azure-side "employee badge" with no password), and grant it read access to Key Vault. Commands below.

2. The federation handshake is registered

Kubernetesservice account: support-agent-sa
→
federated credential"trust tokens from THIS cluster for THIS service account"
→
Entra ID

The fingerprint enrolment: "when this cluster vouches for agents/support-agent-sa, treat it as the managed identity". Issuer and subject must both match — nothing else gets in.

3. The pod starts with a projected token

agent podlabel: azure.workload.identity/use
←
kubeletinjects a short-lived token file

Labelled for workload identity and running as the annotated service account, the pod gets a signed, minutes-lived token mounted in — the fingerprint, refreshed automatically.

4. Token exchange at the door

agent pod"here is my K8s token"
→
Entra IDchecks issuer + subject, issues Azure token

The Azure SDK does this silently: it presents the Kubernetes token, Entra checks it against the federated credential, and returns a short-lived, scoped Azure token.

5. Key Vault opens

agent pod
→
Key Vaultreturns LLM-API-KEY at runtime

The agent reads its LLM key from Key Vault at startup, in memory only. Grep the image, manifests, pipeline: no key anywhere. That empty grep is the achievement.

workload identity — the one-time setup, in order
az aks update --resource-group agents-rg --name agents-aks \
    --enable-oidc-issuer --enable-workload-identity

az identity create --resource-group agents-rg --name support-agent-id

export AKS_OIDC_ISSUER=$(az aks show --resource-group agents-rg \
    --name agents-aks --query "oidcIssuerProfile.issuerUrl" -o tsv)

az identity federated-credential create --name support-agent-fed \
    --identity-name support-agent-id --resource-group agents-rg \
    --issuer "$AKS_OIDC_ISSUER" \
    --subject system:serviceaccount:agents:support-agent-sa \
    --audience api://AzureADTokenExchange

Walking through it: the first command switches on the cluster's OIDC issuer — the thing that signs pod tokens — and the workload-identity machinery. The second mints the badge. The third captures the issuer URL with the --query trick from the AKS chapter. The fourth is the handshake: --issuer names the cluster to trust, --subject pins one service account in one namespace, and the audience is the fixed exchange endpoint. On the Kubernetes side, two touches: the service account gets the annotation azure.workload.identity/client-id with the badge's client ID; the pod template gets the label azure.workload.identity/use: "true". Keyless from then on.

Namespaces and quotas: separate stations, separate budgets

You have been typing -n agents all act. A namespace is a named compartment inside one cluster — separate folders for separate teams, so the support team's pods, Services, and Secrets never collide with the research team's, and access is granted per compartment. Compartments alone don't stop the pastry section burning everyone's gas, though. A ResourceQuota is a per-namespace budget enforced at admission time — a workload that would exceed it is simply refused:

quota.yaml — the support team's budget
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-support-quota
  namespace: agents
spec:
  hard:
    requests.cpu: "8"
    requests.memory: 16Gi
    limits.cpu: "16"
    limits.memory: 32Gi
    pods: "30"

Now the HPA can scale to its heart's content — up to eight CPUs' worth of requests and thirty pods, not a millicore more. This is how enterprises run AKS: the platform team owns the cluster; each product team gets a namespace, a quota, and a dashboard.

Node pools and the bill

Pods run on nodes, and nodes are what Azure actually charges for — by the hour, busy or bored. AKS groups nodes into node pools: sets of identical VMs sized and scaled independently. Two habits keep the bill honest. First, keep the system pool — the small, reliable pool running Kubernetes' own components — separate from the user pools where agents live, so a rowdy workload can never starve the control machinery. Second, for dev and batch work, use spot nodes: Azure's spare capacity at a steep discount, with the catch that Azure can evict them at short notice. For a retry-friendly dev workload, a gift; for the production fleet, no.

add a cheap spot pool for dev workloads
az aks nodepool add --resource-group agents-rg \
    --cluster-name agents-aks --name devspot \
    --mode User --node-count 2 \
    --priority Spot --eviction-policy Delete --spot-max-price -1

Line by line: nodepool add bolts a new pool onto the cluster; --mode User marks it as workload space; --priority Spot asks for the discounted evictable VMs; --eviction-policy Delete says "when Azure reclaims one, just remove it" (pods reschedule elsewhere); --spot-max-price -1 means "charge the going spot rate, up to the regular price". Watching the fleet: top and Container Insights

You already own the observability instincts from the observability chapter — traces, costs, audit trails for what the agent did. Day-2 adds the infrastructure layer: what the pods and nodes are doing. The handheld reading is kubectl top nodes and kubectl top pods -n agents — live CPU and memory, the same numbers the HPA acts on. The permanent, queryable version is Container Insights, Azure Monitor's container add-on: it streams pod logs, restarts, and per-container CPU/memory history into a workspace where you can chart a week of memory growth (hello, slow leak before the OOMKill from kubectl fluency) or alert on restart spikes. You need both layers: Container Insights says the pod restarted at 14:03; your agent traces say it was mid-way through a customer refund when it did.

The pipeline that ships it all

Every command in this act, you have run by hand — deliberately, so nothing is magic. But the pipelines chapter taught the real rule of production: humans review, machines ship. .gitlab-ci.yml — the build and deploy jobs
stages: [test, build, deploy]

build-image:
  stage: build
  image: mcr.microsoft.com/azure-cli
  script:
    - az login --service-principal -u "$AZ_CLIENT_ID"
        -p "$AZ_CLIENT_SECRET" --tenant "$AZ_TENANT_ID"
    - az acr build --registry agentsacr
        --image support-agent:$CI_COMMIT_SHORT_SHA .

deploy-prod:
  stage: deploy
  image: mcr.microsoft.com/azure-cli
  environment: production
  script:
    - az login --service-principal -u "$AZ_CLIENT_ID"
        -p "$AZ_CLIENT_SECRET" --tenant "$AZ_TENANT_ID"
    - az aks get-credentials --resource-group agents-rg
        --name agents-aks
    - az aks install-cli
    - kubectl set image deployment/support-agent
        agent=agentsacr.azurecr.io/support-agent:$CI_COMMIT_SHORT_SHA
        -n agents
    - kubectl rollout status deployment/support-agent
        -n agents --timeout=180s

Worth pausing on: the credentials arrive as masked CI/CD variables (ch101), referenced by name and never printed; az acr build hands the build to Azure, so the runner needs no Docker daemon; az aks install-cli drops kubectl into the azure-cli image; kubectl set image is the one-liner equivalent of editing the tag and applying; and the --timeout converts "deployment quietly stuck" into "pipeline loudly red". Teams often go further with pull-based GitOps tooling — but this push-based job is the honest, readable baseline most fleets run on.

In the wild

Day-2 is where merge requests get small and stakes get big. You will see an MR changing nothing but averageUtilization: 70 to 60 ("lunch spike keeps outrunning the scaler"); an incident channel where the entire resolution is kubectl rollout undo pasted at 14:07; a security review rejecting a manifest for an API key in env: where a workload-identity label should be; and a cost review where moving the dev pool to spot saves more than the last three optimisation projects combined. You can now read every one of those diffs — and write them.

Vocabulary check

Eight terms carried this chapter. Flip until they feel like old friends:

Check yourself

1. Your agent fleet is drowning in queued requests, but the HPA refuses to scale up. The pods spend their time awaiting LLM API responses. Most likely explanation?

An agent blocked on a network response is nearly idle by CPU: the HPA sees calm cooks and ignores the growing ticket rail. Queue depth measures the backlog itself.

2. Ten minutes after a release, v13 pods are crash-looping and users are affected. The fastest safe move is:

Kubernetes keeps the previous ReplicaSet precisely for this moment. rollout undo replays the rolling swap in reverse — one command, no YAML archaeology.

3. Your agent takes 40 seconds after boot to warm its caches. Which probe keeps traffic away during that window without risking a restart?

Readiness controls routing; liveness controls restarts. A warming pod that fails liveness is killed and starts warming again — forever. Failing readiness just means "no customers yet".

4. Why is Entra workload identity preferred over storing an Azure credential in a Kubernetes Secret?

A stored credential can be exfiltrated, committed, or forgotten during rotation. Federation replaces it with a live handshake, and every token dies in minutes.

5. In the GitLab deploy job, what does gating on kubectl rollout status --timeout=180s actually buy you?

Without it, the job "succeeds" the moment the manifest is accepted — even if every new pod then crash-loops. The gate makes green mean the new version is actually up.

Key takeaways

  • Day-2 operations — scaling, updating, healing, securing, budgeting — make a cluster production; each is just more declarative YAML plus a few commands.
  • The HPA resizes replica counts on CPU or memory; agents waiting on LLM responses look idle by CPU, so real fleets scale on queue depth with KEDA.
  • A release is a one-line image-tag change plus kubectl apply; rollout status verifies it, rollout undo is the one-command way back.
  • Readiness probes take sick pods out of traffic; liveness probes restart dead ones. Dependency checks belong only in readiness — in liveness they turn a blip into a restart storm.
  • Entra workload identity replaces stored API keys with a federated token exchange: a short-lived Kubernetes token opens Key Vault. Nothing durable exists to leak.
  • Namespaces plus ResourceQuotas make one cluster safely multi-tenant; a protected system pool and spot nodes for dev are the easiest money you will ever save.
  • The GitLab deploy job closes the circle: test and eval gates, az acr build, az aks get-credentials, kubectl apply, and a rollout-status gate that makes green mean "actually running".
