Git: the time machine

Commits are photographs, branches are sticky notes, HEAD is a you-are-here pin. Once you see the graph underneath, eight commands cover 95% of your working life — and none of them will ever scare you again.

~35 minChapter 97 of 117
The story

An architect in Pune has a ritual. Every evening she photographs her entire drafting table — every sheet, exactly as it lies — prints the photo, writes what changed and why on the back, and clips it to yesterday's photo. Month after month, a chain of photographs grows on the studio wall. When a client sighs, "the balcony was better two weeks ago," nobody panics: walk back along the chain, find that evening's photo, and there is the balcony — exactly as it was, recoverable.

Some photos wear sticky notes: approved design flags one, monsoon experiment another. The notes copy nothing — they are labels that can be peeled off and moved. A single red clothespin marked you are here clips onto whichever note the team is working from today. And before each photograph she uses one more trick: a wooden tray. Only the sheets she deliberately places on the tray get photographed tonight; half-finished scribbles stay off it.

That studio is git. The photos are commits, the chain is the graph, the sticky notes are branches, the clothespin is HEAD, and the tray is the staging area. This chapter gives each its real name.

You already drive git — now open the hood

Back in just enough git you learned the survival kit: git is a save-point system for folders, git pull brings down teammates' work, git add and git commit record yours, git push shares it, and the golden rule is to commit early and often because a committed change is nearly impossible to lose. That kit got you through cloning repos and shipping small fixes. What it deliberately skipped was the model — the picture in a senior engineer's head that makes every git message self-explanatory instead of terrifying.

This chapter installs that picture: snapshots, movable labels, one pin. Everything git ever prints — "detached HEAD", "ahead of origin/main by 2 commits", "changes not staged for commit" — becomes a plain sentence once you hold it. In this Act you ship alongside data scientists as an equal; this chapter makes their git vocabulary your native language too.

A commit is a photograph, not a list of changes

The most common misconception about git is that a commit stores "the lines you changed". It does not. A commit is a complete snapshot of every file in the project at one moment — the whole drafting table, not just the sheet you touched. When git shows you a diff, it computes the difference between two photographs on demand.

Each commit also records one thing about the past: the identity of its parent — the commit just before it. Photo clipped to photo. Follow the parent links backwards and you can reconstruct any moment in the project's life. That is the whole time machine: nothing is overwritten; new photographs are only ever added to the front of the chain.

Every commit is named by a hash — a 40-character fingerprint (like 3f2a1c9…) computed from the snapshot's content, message, author, and parent. Change any of those and the fingerprint changes, which is why a commit id is proof of exactly what it contains. Humans use the first seven characters; that is what you see all over GitLab.

A chain that only grows, where each link points to its parent, and which can fork and later rejoin — mathematicians call that a graph, and "the commit graph" is a phrase engineers use daily. Linear history is just the simplest graph. The next chapter, Feature branches, release branches, is entirely about what happens when it forks.

Note

"A full snapshot per commit" sounds wasteful — it is not. Git stores each file's content once, and every snapshot containing an unchanged file points at the same stored copy. You get the mental simplicity of photographs at the storage price of diffs.

Branches are sticky notes, not copies

Ask five people what a branch is and you will hear "a copy of the codebase". Wrong — and expensively wrong, because it makes branches feel heavy and scary. A branch is a movable name-tag pointing at exactly one commit — literally a tiny text file inside the repository whose entire content is one commit id. Creating a branch copies nothing: git writes ~41 bytes and is done, even in a ten-gigabyte repository.

The tag has one behaviour that makes everything work: when you commit while "on" a branch, that branch's tag slides forward to the new commit automatically. The sticky note rides the front of the chain. main is not special technology — just the note everyone agrees to treat as the trunk. Your fix-totals branch is a note of equal rank.

HEAD — the you-are-here pin

So which branch are you "on"? That is HEAD: git's single you-are-here pin. Normally HEAD points at a branch name, and the branch points at a commit — pin on note, note on photo. git switch fix-totals does two things: moves the pin to the fix-totals note, and rearranges your working folder to match the photo that note points at. That second part is the time-machine lever — your actual files on disk change to match the snapshot.

Occasionally the pin gets clipped straight onto a photo with no note in between — a state git calls detached HEAD: "you are viewing a specific commit, not riding any branch". It sounds like an injury; it is sightseeing. CI systems do it on purpose for every pipeline run, because a build should test one exact snapshot. One caution: commits made while detached hang on no note, so create a branch first if you intend to keep them.

The staging area — the tray before the photograph

Between your editor and the photo album sits git's most misunderstood feature: the staging area (older docs say index) — a holding zone where you assemble exactly what the next commit will contain. git add totals.py saves nothing permanently; it places that file's current content on the tray. git commit photographs the tray, not your folder.

Why the extra step? Deliberate composition. By afternoon your folder holds a genuine bug-fix, a half-finished experiment, and a forgotten debug print. The tray lets you stage only the bug-fix and commit it as one clean, reviewable unit, leaving the mess out of history — stage the fixed pipeline script, not the scratch notebook full of exploration. Three zones, then, and every file wanders between them: the working directory (your real files on disk), the staging area (the tray), and the repository (the album of committed snapshots, living inside the hidden .git folder).

Tip

git add -p ("patch mode") walks you through your edits chunk by chunk, asking yes/no for each — you can stage half a file. It is the fastest route to clean, single-purpose commits, and reviewers will quietly love you for it.

The daily eight

Git has well over a hundred subcommands. Working engineers live on eight moves — a couple come as pairs — and reach for the rest maybe once a month. Here they are, mapped onto the studio:

MoveCommandWhat it really does
Copy the albumgit clone <url>Downloads the entire history — every photo, every note — and pins HEAD to the default branch. Done once per project.
Where am I?git statusReads the pin, the tray, and the table: current branch, staged files (green), unstaged edits (red). Free, safe, run it constantly.
Onto the traygit add <file>Stages a file's current content for the next commit. Composition, not saving.

Get their photosgit pullFetches new commits from origin, then (by default) merges them into your branch. Do it before you start work.

Note

Older tutorials — and most colleagues who learned before 2019 — say git checkout where this chapter says git switch. Checkout still works; it is one old command wearing three hats (switch branches, restore files, detach HEAD), which is why the clearer switch and restore were introduced. Read checkout in the wild without fear; type switch yourself.

Two of the eight deserve one more sentence. git diff with no arguments compares working directory against tray — "what have I not yet staged?" — while git diff --staged compares tray against last commit — "what am I about to photograph?". And the most useful log incantation in team life is git log --oneline --graph --all: one line per commit, drawn as the actual graph, all branches visible — the picture this chapter has been painting, on your own screen.

A working session, annotated

Here is the terminal transcript of the morning from the stepper — the exact keystrokes, in order. Click each line and make sure nothing in it is mysterious anymore.

Tuesday, 9:40 am — one bug-fix, start to finish
$ git status
The opening move of every session: which note is the pin on, and is the table clean? Answer: "On branch main, working tree clean." Safe to proceed.
$ git pull
Bring down whatever teammates pushed overnight before building on stale photos. Two moves fused: fetch (download their commits) then merge (weave them into your branch).
$ git switch -c fix-totals
New sticky note on today's latest commit, pin moved onto it, one stroke. Your work now lives on its own label and main stays pristine — etiquette the next chapter turns into a full workflow.
…edit totals.py in your editor…
$ git diff
Line-by-line view of working directory versus last photo: removed lines prefixed -, added lines +. Reading your own diff before staging is the cheapest code review you will ever get.
$ git add totals.py
Onto the tray. Only this file — the half-finished scratch notebook you also touched stays off tonight's photograph.
$ git status
Trust but verify: totals.py now green under "changes to be committed", the notebook still red. The tray holds exactly what you intend.
$ git commit -m "Fix rounding in quarterly totals"
The photograph. A new commit is born pointing at its parent, and the fix-totals note slides onto it. Message in the imperative — "Fix", not "Fixed" — finishing the sentence "this commit will…".
$ git log --oneline --graph -3
A rear-view glance: your new fingerprint on top, decorated (HEAD -> fix-totals). That decoration is this whole chapter in five characters: pin, on note, on photo.
$ git push -u origin fix-totals
Publish the note and its photos. The -u links your local branch to the remote one, so tomorrow a bare git push or pull knows where home is. GitLab replies with a link to open a merge request — your cue for the next chapter.

Prove it to yourself: a toy git in forty lines

Claims like "a branch is a 41-byte pointer" deserve proof. So let us build git's core model in pure Python: commits as dicts with parent pointers, branches as a name-to-id dict, HEAD as one variable. The toy is honest — real git adds compression, the tray, and networking, but the graph underneath is exactly this.

Read the output slowly. From experiment, the log walks three photos back; from main, only two — same album, different starting note. Nothing was copied when the branch was created; history is simply whatever you can reach by following parent links from your note. Now break things: commit the same message and files twice and compare the fingerprints (they differ — the parent changed). Or add a third branch, commit to it, and predict each log before running. When your predictions hit three for three, the model is yours.

Careful

The time machine protects committed work only. Edits in the working directory — unphotographed — are the one thing git can genuinely lose: switching branches can overwrite them, and a careless git restore discards them silently. Hence the discipline: commit early, commit often, push daily. Photos in the album are close to indestructible; sheets on the table are one gust from gone.

In the wild

This vocabulary is everywhere once you can hear it. A GitLab merge request page is a branch comparison: "3 commits, 14 files changed" means three photos on the source note that the target note cannot yet reach. A pipeline log opening with "checking out a4f9c2e… you are in 'detached HEAD' state" is CI photographing one exact snapshot — deliberate, not broken. A data scientist's "pull main and re-run your notebook" means "get the new photos before comparing results". And "ahead of origin/main by 2 commits" in your terminal is geometry: your note has slid two photos past the server's copy. Push, and the notes align.

Check yourself

1. What does a git commit actually store?

Commits are photographs of the whole project, each clipped to its parent. Diffs are computed between snapshots on demand, and clever storage deduplicates unchanged files so the album stays small.

2. Your repository is 10 GB. How expensive is git branch idea?

A branch is a movable name-tag pointing at one commit — nothing more. That is why engineers create branches for everything: the cost is a rounding error.

3. HEAD points at main, and main points at commit C. You run git commit. What is true afterwards?

Committing on a branch always slides that branch's note to the new photo, and HEAD follows because it points at the note, not the photo. Push only affects the server's copy later.

4. Why does the staging area exist at all — why not commit the working directory directly?

The tray separates "everything I touched today" from "the one clean change I want in history". Stage the bug-fix, leave the scratch notebook behind, commit a reviewable unit.

5. What does git pull actually do, by default?

Pull is two moves fused: fetch (download the others' new photos) then merge (weave them into your branch). Push is the same river flowing the other way — and clone is the once-per-project full copy.

Key takeaways

  • A commit is a complete snapshot plus a pointer to its parent — photographs clipped into a chain, never a stored list of edits. The chain is the graph, and it only ever grows.
  • A branch is a movable name-tag on one commit — ~41 bytes, free to create — that slides forward automatically when you commit on it. main is convention, not technology.
  • HEAD is the single you-are-here pin: normally on a branch, occasionally clipped straight onto a commit ("detached HEAD" — sightseeing, not injury).
  • Files live in three zones — working directory, staging area, repository — and git status tells you where everything sits. Commit photographs the tray, not your folder.
  • Eight moves cover 95% of daily work: clone, status, add, commit, pull, push, branch/switch, log/diff. Everything else is monthly at most.
  • Committed work is nearly indestructible; uncommitted work is the only thing git can lose. Commit early, commit often, push daily.
