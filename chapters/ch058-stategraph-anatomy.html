StateGraph anatomy

Every LangGraph app — every one — is exactly three declarations: a state schema, some nodes, and some edges. By the end of this chapter you will read them like a floor plan, trace one by hand, and build the whole machine yourself in twenty-five lines of Python.

The story

Walk into a well-run hospital and follow one patient's file. At the admission desk, someone opens a fresh folder with printed sections: name, current ward, medication log, discharge note. The patient then moves — admissions to the ward, the ward to the pharmacy — and the folder travels with them. Each department writes only in its own section; the pharmacist never rewrites the X-ray report. And notice that the sections behave differently. Current ward gets crossed out and replaced at every move. The medication log is sacred: staff may only add lines to it, never blank it and start over — a patient's dosing history must survive every handover. On the corridor wall hangs a floor plan showing exactly which department hands over to which, so a new nurse never has to guess where the folder goes next. At discharge, the last section is signed and the folder is returned.

That hospital is a LangGraph application. The printed folder is the state schema. The departments are nodes. The corridors on the floor plan are edges. The append-only medication log is a reducer. Admission and discharge are START and END.

One idea, three parts

In the last chapter you saw why graphs exist: the hand-rolled ReAct loop is wonderful until the flow stops being a loop — until you need branches, parallel steps, pause-and-resume, or a shape you can actually draw for a colleague. LangGraph's answer is to stop burying the flow inside if statements and instead declare it: here are the workers, here is the shared paperwork, here is who hands over to whom. The class that holds those declarations is StateGraph — a builder object you load up with a state schema, nodes, and edges, and then compile into something runnable.

That is genuinely the whole anatomy. Three parts:

  • The state schema — the shape of the shared folder every node reads and writes.
  • The nodes — plain Python functions that take the current state and return the part they changed.
  • The edges — the wiring that says which node runs after which, bookended by START and END.

Everything else in this Act — routing, checkpoints, human-in-the-loop — is an upgrade to one of these three parts. Get these three cold and the rest of LangGraph becomes furniture arrangement.

Part one: the state — a folder with a printed form inside

Quick recap from the type-hints chapter, because it is about to earn its keep. A TypedDict is a way to declare the shape of a dictionary: which keys exist and what type each value should hold. It enforces nothing at runtime — it is a description that tools (and frameworks) can read. Pydantic models actively validate data at the boundary; TypedDicts just describe it, at zero cost. LangGraph reads that description and uses it as the blueprint for the folder your graph passes around.

state.py — the printed form
from typing import Annotated, TypedDict
import operator

class ResearchState(TypedDict):
    topic: str                                  # plain: last write wins
    notes: Annotated[list[str], operator.add]   # reducer: appended
    draft: str                                  # plain: last write wins

Each key in the state is called a channel — a named slot that nodes can read from and write to, like one section of the hospital folder. When you build the graph you hand this class to the constructor — StateGraph(ResearchState) — and from then on, every node receives a dictionary of exactly this shape and every update is merged back into it.

The interesting question is how updates are merged, and that is where the second line of that snippet gets its moment.

Reducers: sections that append instead of overwrite

By default a channel behaves like current ward on the folder: when a node writes a new value, the old one is simply replaced. Last write wins. For topic and draft that is exactly right.

But some channels must behave like the medication log. If the planning step writes a note and then the research step writes another, you do not want the second note to erase the first — you want both. Declaring that rule is what Annotated is for: it lets you attach extra metadata to a type hint, and LangGraph reads that metadata as the channel's merge rule. The attached function is called a reducer — it takes the old value and the incoming update and combines them into the new value.

operator.add is the workhorse reducer for lists: recall from the collections chapter that + on two lists concatenates them, so old + new means append. For chat history LangGraph ships a smarter reducer, add_messages (imported from langgraph.graph.message): it appends new messages like operator.add would, but if an incoming message carries the same id as an existing one it replaces it instead of duplicating it — which is what makes edits and retries safe. It is so common that LangGraph provides a ready-made MessagesState with a messages channel already wired to it.

Channel declarationMerge ruleFeels like
draft: strLast write wins — new value replaces old"Current ward" — crossed out and rewritten
notes: Annotated[list, operator.add]Old + new — updates are appendedThe medication log — add lines, never erase
messages: Annotated[list, add_messages]Append, but replace when ids matchA log where a correction overwrites the original entry
Careful

With a reducer channel, a node should return only the new items, never the whole updated list. If a node returns state["notes"] + [new_note], the reducer dutifully appends all of it — and every old note is now in the state twice. The infamous "why is my agent's history duplicated?" bug is almost always someone hand-merging a channel that already has a reducer.

Part two: nodes — wards that write in the folder

A node is a plain function that receives the current state and returns a dictionary containing only the channels it changed — a partial update, meaning you mention what you touched and stay silent about everything else. The graph merges your update into the state using each channel's rule. This is the pharmacist writing one line in the medication log and leaving the X-ray report alone.

one node — state in, partial update out
def plan(state: ResearchState) -> dict:
    # read anything you like from state...
    angle = f"PLAN: three angles on {state['topic']}"
    # ...but return ONLY what you changed
    return {"notes": [angle]}

Two things to sit with. First, nodes never call each other. plan does not import research or pass it anything — it drops its update into the shared state and its job is over. Who runs next is the graph's business, not the node's. That separation is what makes graphs rearrangeable: you can reorder, insert, or remove nodes without touching the code inside any of them.

Second, notice what the node returns for notes: a list containing one new note, not the whole updated log. The operator.add reducer does the appending. The node describes the change; the channel decides the merge.

Part three: edges — the corridors on the wall

An edge is a declared handover: "after this node, run that one." You register nodes with add_node(name, function) and wire them with add_edge(source, target). Two special markers bracket the flow: START is the entry — the admission desk where invoke hands the initial state in — and END is the discharge — when the run reaches it, the graph stops and returns the final state. You never write a function for either; they are just names for "the way in" and "the way out".

When the wiring is done, .compile() checks it — every edge must point at a registered node, and there must be a way in from START — and returns the runnable graph object, the one with .invoke() and .stream() on it (both get their own chapter). Compiling runs nothing and calls no model. It is the hospital board approving the floor plan and pinning it to the wall: validation plus freeze, and also the moment where extras like a checkpointer plug in via .compile(checkpointer=...).

research_graph.py — a complete three-node LangGraph app
from typing import Annotated, TypedDict
Both come from the standard library's typing module — nothing LangGraph-specific yet. TypedDict declares the state's shape; Annotated lets us staple a reducer onto one channel.
import operator
from langgraph.graph import StateGraph, START, END
The three imports that define this Act. StateGraph is the builder; START and END are the special entry and exit markers. This exact line appears near the top of almost every LangGraph file ever written.
class ResearchState(TypedDict):
Part one: the state schema. The class body is the printed form — every node will receive a dict shaped like this.
topic: str
notes: Annotated[list[str], operator.add]
The reducer channel. Without the annotation, a second write to notes would erase the first. With operator.add, updates are concatenated — appended, never overwritten.
draft: str
def plan(state: ResearchState) -> dict:
Part two: a node. Just a function (ch7) — no base class, no decorator required. State in, partial update out.
return {"notes": [f"PLAN: three angles on {state['topic']}"]}
A partial update: only notes is mentioned, so topic and draft are untouched. And it returns one new item, trusting the reducer to append it.
def research(state: ResearchState) -> dict:
return {"notes": ["FACT: exports hit a record last year"]}
In a real app this node would call an LLM or a search tool. Structurally, nothing changes — it still just returns the channels it modified.
def write(state: ResearchState) -> dict:
return {"draft": " ".join(state["notes"])}
This node reads the accumulated notes and writes the plain channel draft. No reducer there, so whatever was in draft before is replaced.
builder = StateGraph(ResearchState)
The builder is born holding the schema. From here on it knows every channel and every merge rule.
builder.add_node("plan", plan)
Register a node under a name. The string is what edges refer to; the function is what actually runs. Same dict-of-callables idea as the TOOLS dict in ch45.
builder.add_node("research", research)
builder.add_node("write", write)
builder.add_edge(START, "plan")
Part three: the wiring. This edge is the admission desk — when someone calls invoke, the state enters here.
builder.add_edge("plan", "research")
Plain edges are unconditional: after plan, always research. The next chapter makes these decisions dynamic.
builder.add_edge("research", "write")
builder.add_edge("write", END)
Discharge. Reaching END stops the run and hands the final state back to the caller. Compile will not force you to draw this edge — a node with no outgoing edge simply ends the run — but drawing it keeps the map honest.
graph = builder.compile()
Validation plus freeze: every edge must point at a real node, START must lead somewhere. Returns the runnable object. No model has been called; no node has run.
result = graph.invoke({"topic": "chai exports"})
Now it runs: the initial state enters at START, flows plan to research to write, and the final merged state comes back as a plain dict — result["draft"] holds the report.

In the wild

Open any LangGraph repository and you can navigate it in one pass: look for a TypedDict (often in state.py, often named AgentState or State) — that is the folder; find the block of add_node/add_edge calls (often graph.py) — that is the floor plan, and you can sketch the whole app from it without reading a single node body. A merge request that says "added a summarise node" will touch exactly one new function and two or three edge lines. And when a teammate uses LangChain's create_agent, the object it returns is a compiled StateGraph under the hood — a prebuilt agent graph (model node, tools node) with the loop drawn as edges. Same anatomy, pre-assembled.

Check yourself

1. A node returns {"draft": "Report text"}. What happens to the state's other keys, topic and notes?

Nodes return only what they changed. The graph merges that update into the shared state channel by channel; anything not mentioned simply keeps its current value.

2. notes is declared as Annotated[list[str], operator.add]. The plan node writes ["PLAN"] and later the research node writes ["FACT"]. What does notes hold afterwards?

That is the entire job of a reducer: instead of replacing the old value, it merges old and new. operator.add on lists concatenates, so successive writes accumulate — the medication log, not the whiteboard.

3. What does builder.compile() actually do?

Compile is check-and-freeze: edges must point at registered nodes and START must lead somewhere. Execution only happens when you call .invoke() or .stream() on the compiled graph.

4. What is START in a StateGraph?

START and END are markers, not nodes — you never write functions for them. The edge add_edge(START, "plan") is you telling the graph where the folder enters the building.

5. A node on an operator.add channel returns state["notes"] + [new_note] — the whole old list plus the new item. What happens?

The reducer appends whatever you return. Return old + new and the reducer computes old + (old + new). On a reducer channel, return only the new items. (The add_messages reducer is more forgiving for chat history because it matches message ids, but plain operator.add has no such mercy.)

Key takeaways

  • Every LangGraph app is three declarations: a state schema (the shared folder, usually a TypedDict), nodes (functions that read state and return partial updates), and edges (who hands over to whom), bookended by START and END.
  • Each state key is a channel with its own merge rule: plain channels overwrite (last write wins); Annotated[..., reducer] channels combine — operator.add appends lists, add_messages appends chat history and replaces by id.
  • Nodes never call each other. They communicate only through the shared state, which is exactly what makes graphs rearrangeable and drawable.
  • On a reducer channel, return only the new items — hand-merging old content re-appends it and duplicates your history.
  • .compile() validates the wiring and returns the runnable object; nothing executes until .invoke(). Checkpointers plug in at compile time.
  • The whole machine fits in twenty-five lines of pure Python — you built it in this chapter, and everything LangGraph adds on top is engineering, not new ideas.
