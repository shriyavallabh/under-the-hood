The story

You have driven the same car every day for three years. You know its habits perfectly — where it hesitates on a cold morning, the exact wiggle that opens the boot — but you have never once opened the bonnet. Then a mechanic friend puts it on a lift and starts pointing. "Fuel pump. Alternator. Brake caliper — same part as in any car on this street. Nothing custom here, just very well chosen and very well fitted." And something clicks: you spent the last year at night classes learning engine parts one by one, and here they all are, bolted together inside the machine you already trust with your commute.

That is this chapter. The car is Claude Code — probably the best-known agent harness in the world, and the one you operate daily. The night classes were Acts V through X: loops, tools, subagents, skills, memory, context engineering, guardrails. Today the bonnet comes up, and the pleasure of it is that nothing inside will be new. Every part has a chapter behind it. The famous machine is famous not because its parts are exotic, but because of how they are fitted together.

The loop at the heart

Strip everything else away and Claude Code is the fifty-line loop from chapter 46, wearing a good suit. Your prompt goes into a transcript. The model replies — with tool calls or without. If with, the harness executes them, appends the results, and sends the whole transcript back to the model. If without, the loop ends and you see the answer. Think, act, observe, repeat. The model never touches your filesystem; it writes structured requests (native tool calling, not the string-parsed ACTION lines of our hand-built version), and the harness — an ordinary process on your machine — does the touching.

Two refinements sit on top. First, planning: for multi-step work the model keeps a visible to-do list through a dedicated tool, ticking items off as it goes — the write_todos pattern from planning and to-dos, there for the same reason: a written plan keeps a long run on the rails and the goal inside the context window. Second, statelessness: the model remembers nothing between API calls, so the growing transcript is the agent's working memory — and every other part in this chapter exists to protect, feed, or prune that transcript.

The toolbox: a handful of sharp verbs

For all its reputation, the core toolbox is startlingly small. Read returns a file with line numbers. Write creates or replaces a file. Edit performs an exact string replacement — it must match the old text character for character, or it refuses. Bash runs a shell command. Grep and Glob search file contents and file names. And Task spawns a subagent, which we will open up shortly. Web access and MCP servers extend the set, but those seven verbs do the overwhelming majority of the work.

Notice what kind of tools these are: small, deterministic, honest about failure. Edit is the interesting one. A model asked to "rewrite the file" can hallucinate away code it never read; a model forced to state the exact old text and the exact new text either matches reality or gets an error it must confront. The tool's design compensates for the model's weaknesses — the principle from the tools chapter, applied ruthlessly. deepagents copied this toolbox almost name for name in the virtual filesystem chapter; the difference is that Claude Code's files are your real ones — which is precisely why the next two parts exist.

The permission gate

Everything dangerous about an agent lives at one boundary: the line between the model proposing an action and the harness executing it. You met that line in chapter 46 — it was the TOOLS[name](arg) dictionary lookup — and you fortified it in guardrails. Claude Code fortifies it with two mechanisms: permission modes, coarse dials that set the session's overall trust level, and permission rules, fine-grained allow/ask/deny lists that pre-decide specific calls.

The four modes, from most to least cautious:

ModeWhat it meansWhen you use it
planRead-only. The agent may explore and search, but every mutating tool is refused; it must present a plan and get your sign-off first.Unfamiliar codebase, risky change, or "tell me what you would do first".
defaultReads are free; edits, shell commands, and anything risky pause the loop and ask you.Everyday work.
acceptEditsFile edits are auto-approved; shell commands still ask.You trust the direction and want fewer interruptions.
bypassPermissionsNothing asks. Every proposed call executes.Only inside a disposable sandbox — see below.
Careful

That last mode is the deal you studied in sandboxing: full autonomy is only safe when the blast radius is contained by the environment — a container, a throwaway VM, a branch you can delete — never by the model promising to behave. Teams that run agents unattended in CI do it in exactly such sandboxes, with bypass on and credentials scoped down.

Below the modes sit the rules, in a file you may have scrolled past in a dozen repositories: .claude/settings.json. Click through the annotated lines — this small file is most of the "safety story" of any team's agent setup:

.claude/settings.json — the gate, as configuration
{
"permissions": {
The rules block, checked on every single tool call before anything executes — policy at the boundary, not in the prompt.
"allow": ["Bash(npm test:*)", "Edit(src/**)"],
Pre-approved calls that never interrupt you. Each rule names a tool and a pattern: any npm test variant may run, any file under src/ may be edited, silently. This is how a team tunes "asks per hour" down without going full bypass.
"ask": ["Bash(git push:*)"],
Always pause for a human on these, whatever the mode. Pushing to a remote is a classic one-way door, so it stays behind a yes/no prompt.
"deny": ["Read(./.env)", "Read(./secrets/**)"]
Hard refusals — the model cannot even read these files, so a secret never enters the transcript at all. Recall from MCP security: what never enters the context can never be exfiltrated by a prompt injection.
},
"hooks": {
The second mechanism: your own scripts wired into the loop's lifecycle. Read on.
"PreToolUse": [{
This hook fires before any matching tool call executes — step 3 of the stepper above.
"matcher": "Bash",
Only fire for Bash calls. Other events exist — PostToolUse after a call, Stop when the agent finishes, PreCompact before compaction.
"hooks": [{"type": "command", "command": "./check-command.sh"}]
A real shell script you wrote. It receives the proposed call as JSON on stdin; if it exits with code 2, the call is blocked and the model is shown your script's explanation. Deterministic veto power.
}]
}
}

Hooks: policy as code, not as pleading

A hook is a shell command of yours that the harness runs at a fixed point in the loop's life — before a tool call, after one, when you submit a prompt, when the agent stops, before compaction. The crucial property: hooks are deterministic. A line in CLAUDE.md saying "never edit the .env file" is a request the model will honour ninety-something per cent of the time; a PreToolUse hook that blocks any edit touching .env works one hundred per cent of the time, because it is code, not persuasion. If you remember middleware from the LangChain act — functions intercepting the loop around the model and tools — hooks are the same architectural slot, exposed as plain shell, so any team can wire in linters, audit logs, or company policy without touching the harness itself.

Subagents: a clean room for messy work

The Task tool is where the harness quietly becomes a multi-agent system. When the main agent faces a job that would flood its transcript — search the whole repository, review a huge diff, chase a failing test — it can delegate to a subagent: a fresh loop with its own empty context window, its own system prompt, and often a deliberately smaller toolbox. The subagent burns thousands of tokens rummaging, then returns only its conclusions. The parent pays a paragraph, not a haystack.

This is context isolation, exactly as you studied it in the subagents chapter — the same reason LangGraph teams reach for subgraphs. What Claude Code adds is a pattern worth stealing: subagents are files. Drop a markdown file in .claude/agents/ and every session in that repository gains a specialist:

.claude/agents/code-reviewer.md
---
name: code-reviewer
description: Reviews a diff for bugs and risky changes.
  Use proactively after any large edit.
tools: Read, Grep, Glob
---
You are a meticulous reviewer. Read the changed files,
hunt for logic errors and unhandled edge cases, and
report findings ranked by severity. You cannot edit
anything - report only.

Read that frontmatter with your Act X eyes. The description tells the parent model when to delegate. The tools line is a guardrail: this reviewer physically cannot edit files, so a review can never silently become a rewrite. Least privilege, in three lines of YAML.

The layered briefing: system prompt, CLAUDE.md, skills, memory

Now the context side of the machine — everything already inside the model's briefing before you typed a word. It comes in four layers, cheapest-first, and the layering is the lesson.

Layer one: the built-in system prompt. Several thousand tokens, shipped with the product, identical for everyone: what the model is, how each tool works, when to be terse, what never to do. This is the vendor's context engineering, and a large share of why the same model feels sharper inside this harness than in a raw API call — a point you met in prompting that matters.

Layer two: CLAUDE.md — your standing orders. At session start the harness reads CLAUDE.md from the project root (plus a global one in your home directory) and splices them into the context. Project conventions, forbidden commands, the test command: written once, obeyed in every session — including your teammates' sessions, because the file is committed to the repository. Configuration-as-prose, version-controlled like code.

Layer three: skills — expertise on demand. A skill is a folder with a SKILL.md file: instructions, checklists, sometimes helper scripts for one recurring job. The design move that makes skills scale, straight from the skills chapter: at startup the harness loads only each skill's name and description — one line apiece — and reads the full body only when a task calls for it. That is progressive disclosure — pay a line now, a page only when needed — and it is how a hundred-skill setup stays affordable when inlining them all would drown the context window before you said hello.

Layer four: auto-memory — notes that survive the session. Each project gets a small memory directory that persists across sessions: a short index loads at startup, and the agent reads and writes notes there as it works — "this client's staging server needs the VPN", "the flaky test is a known issue". Next week's session starts already knowing. The write-your-own-notes pattern from long-term memory — and the storage is, once again, plain files you can open, edit, and audit.

Compaction: when the desk overflows

One problem remains, and you have known it since memory and context: the transcript only grows, the context window does not, and — as the token bill made painfully concrete — every turn re-sends everything. A long session would eventually hit the ceiling mid-task.

The answer is compaction: as the transcript nears the limit, the model writes a dense summary of the conversation so far — decisions made, files touched, current state, next steps — and the old turns are replaced by that summary, with only the recent tail kept verbatim. The loop continues as if nothing happened. You can trigger it yourself with /compact, and a PreCompact hook can preserve anything you refuse to lose. Compaction is lossy on purpose — fine detail from an hour ago genuinely disappears — but the design cushions the loss: files are still on disk and can be re-read, standing orders live in CLAUDE.md, durable lessons live in memory. The transcript is scratch space, not the system of record — context engineering's central teaching, implemented in shipping software.

What the dissection teaches

Step back from the lift. Five design lessons generalise to every harness you will ever evaluate — the scorecard axes for the rest of this act:

Small, honest tools beat clever ones. Seven deterministic verbs, each designed so the model fails loudly rather than plausibly. When you assess a harness, read its tool list before its marketing.

Safety lives at the boundary, in code. Permission rules and hooks sit between proposal and execution, where they cannot be talked out of the way. A prompt-level "please don't" is a preference; a gate is a guarantee. Ask any harness: where is your gate, and what can it veto?

Context is spent in layers. System prompt always; CLAUDE.md per project; skills and memory by one-line index; the rest fetched on demand. Progressive disclosure lets capability grow without the token bill growing with it.

Isolation is the multi-agent trick that matters. Subagents exist for context hygiene first, specialisation second. A harness without fresh-context delegation will choke on large codebases.

Everything is a file. Agents, skills, memory, settings, hooks — plain text, inspectable, diffable in version control. Compare that with configuration buried in a vendor dashboard and you have a genuinely useful axis for the comparisons ahead. And the quiet corollary: the model is the replaceable part; the harness is the product.

In the wild

You will meet this chapter every time you open a repository with agents in its workflow. A .claude/ folder with settings.json, an agents/ directory, and a skills/ tree; a CLAUDE.md at the root encoding the team's conventions; merge requests that change a deny rule or add a PreToolUse hook and get reviewed as seriously as production code — because they are the security model. When a colleague says "run it in plan mode first", "the hook blocked it", or "the session compacted and lost the thread", you now see the exact component behind each sentence. And when a vendor demos "our new agent platform", you can quietly ask where its gate, its layers, and its files are.

Check yourself

1. Where does Claude Code enforce that a dangerous command never runs?

The model only ever proposes. Permission rules and hooks sit in harness code between proposal and execution, so a blocked call is blocked deterministically. CLAUDE.md instructions help the model behave, but they are persuasion, not enforcement.

2. What does plan mode actually change?

Plan mode is a permission mode, not a model setting: reads and searches work, edits and shell commands are denied. It is the safe way to let an agent explore an unfamiliar or high-stakes codebase before you approve any change.

3. Why does the Task tool give a subagent a fresh, empty context window?

Context isolation is the point. The subagent can burn thousands of tokens rummaging through files, and all of it is discarded when it finishes — only the conclusions flow back, keeping the main transcript lean.

4. A session hits the context limit and compacts. What happens to the conversation so far?

Compaction is deliberate, lossy summarisation so the loop can continue. The design cushions the loss elsewhere: files stay on disk, conventions live in CLAUDE.md, durable lessons belong in memory — the transcript is scratch space.

5. You install forty skills. Roughly what do they cost the context window at session start?

Progressive disclosure: the index of skills is always present (that is how the model knows what it could reach for), but each costs roughly a line until used. That lets capability scale without the token bill scaling with it.

Key takeaways

  • Claude Code is the ReAct loop from chapter 46 in production dress: the model proposes structured tool calls, the harness executes, results return to a growing transcript, repeat until the model stops asking.
  • The core toolbox is seven small, deterministic verbs — Read, Write, Edit, Bash, Grep, Glob, Task — each designed so the model fails loudly rather than plausibly.
  • Safety is enforced at the proposal-execution boundary: permission modes set the trust dial, allow/ask/deny rules pre-decide calls, and hooks give your own scripts a deterministic veto.
  • Subagents are fresh-context loops with restricted toolboxes, spawned to do messy work and return a summary — context isolation as a first-class feature.
  • Context arrives in layers — built-in system prompt, CLAUDE.md, one-line skill index, memory index — with detail fetched only on demand; compaction prunes the transcript when it nears the limit.
  • The transferable lessons: sharp small tools, gates in code not prompts, layered context, isolation for scale, everything-as-inspectable-files. These are your scorecard axes for every harness ahead.
