Decorators: what the @ really does

The most common line in agent code is not a spell — it is a rename. Once you see the @ as ordinary function-passing, @tool, @dataclass, and @app.route stop being incantations and start being furniture.

~30 minChapter 17 of 135
The story

Meera cooks the best dal in the neighbourhood. That is all she does — she cooks. One day a tiffin delivery service signs her up. From then on, every dish she makes gets collected, sealed in an insulated box, labelled, tracked, and delivered warm. Customers never meet Meera; they meet the box. Open it, and her dal is inside, completely untouched. The service did not change her cooking — it wrapped her cooking in extra behaviour.

Notice the details, because every one of them is about to become Python. Meera signed the contract once, not per meal. The box carries her name on the label — if it said "packed by TiffinCo" on every box in the city, customers could never tell whose food they were eating. And when she upgraded to the "deliver hot by 1 pm" plan, she first chose the plan, and then the configured service started wrapping her dishes.

A decorator is the tiffin service: it takes a function, returns a wrapped version, and from then on everyone deals with the wrapper. The @ is the signature on the contract.

Functions are things you can hold

Before the wrapping makes sense, one fact has to feel completely normal: in Python, a function is a value — the same kind of citizen as a number, a string, or a list. Back in names and objects you saw that a name is just a label tied to an object. A def statement does two things: it builds a function object, and it ties a name to it. Which means you can untie that name, tie on another one, drop the function into a dict, or hand it to another function as an argument — brackets are the call; the bare name is the thing itself, as you first saw in the functions chapter.

Item 3 is the quiet superstar. If a function can be passed in, a function can also be handed back. A decorator is nothing more than a function that does both: function in, function out. And item 4 — the fact that a function carries its own __name__ and docstring around — will matter enormously in a few minutes, when a framework starts reading those labels to build a schema.

You have already relied on this professionally. An agent's toolbox is literally a dict of functions — in the ReAct chapter you will build one called TOOLS and watch a loop look tools up by name. Functions-as-values is not a party trick; it is how every plugin system, router, and tool registry works.

Closures, gently: a function that remembers where it was born

One more ingredient. Define a function inside another function, and the inner one can use the outer one's variables — and, here is the strange and wonderful part, it keeps using them even after the outer function has finished and gone home. That memory is called a closure — a function bundled together with the variables from the place it was created.

closures.py
def make_greeter(greeting):
    def greet(name):
        # 'greeting' is not defined in here, and it is not global.
        # greet REMEMBERS it from the call that created it.
        return f"{greeting}, {name}!"
    return greet

good_morning = make_greeter("Suprabhat")
formal = make_greeter("Dear")

print(good_morning("Asha"))   # Suprabhat, Asha!
print(formal("Ms Rao"))       # Dear, Ms Rao!

Two calls to make_greeter, two separate greeting functions, each carrying its own private greeting in an invisible backpack. Nobody outside can touch that variable; the function alone remembers it. In the tiffin story, this is the packing slip taped inside the box — the wrapper knows which cook's dish it is carrying, without any global noticeboard announcing it.

That backpack is exactly where a decorator will keep your original function.

The wrapper: same dish, new box

Now combine the two ideas. Here is a function that takes a function and hands back a wrapped version — the tiffin service itself, in seven lines:

logged.py — a hand-made decorator (no @ yet)
def logged(fn):
    def wrapper(*args, **kwargs):
        print(f"calling {fn.__name__} with {args}")
        result = fn(*args, **kwargs)
        print(f"{fn.__name__} returned {result!r}")
        return result
    return wrapper

def fetch_rate(currency):
    return 83.2   # pretend this hits a real API

fetch_rate = logged(fetch_rate)   # the entire trick, right here
fetch_rate("USD")

Read the last two lines slowly, because they are the whole chapter. logged(fetch_rate) is called once. Inside, a new function called wrapper is born, holding the original fetch_rate in its closure backpack. Then — and this is the sleight of hand — the name fetch_rate is re-tied to point at wrapper. The original function still exists, but the only path to it now runs through the wrapper. Every future call to fetch_rate("USD") actually enters wrapper, which logs, calls the real function, logs again, and passes the result back out.

The *args, **kwargs pair is the wrapper saying "I will accept whatever arguments arrive and forward them unchanged" — the pass-through plumbing you met in the functions chapter. It is what lets one logged wrap a function of any shape.

The @ is only sugar

Writing fetch_rate = logged(fetch_rate) works, but it is easy to forget and it repeats the name three times. So Python gives you a shorthand. These two are identical — not similar, identical:

The equivalence — memorise this one
@logged
def fetch_rate(currency):
    return 83.2

# ...means exactly, character for character...

def fetch_rate(currency):
    return 83.2
fetch_rate = logged(fetch_rate)

That is the entire meaning of the @ symbol: "as soon as this function is defined, pass it through the function named above, and re-tie my name to whatever comes back." A decorator — the thing after the @ — is just a callable that takes a function and returns its replacement. Watch Python actually do it, step by step:

The name-tag problem, and functools.wraps

There is one flaw in our hand-made service. Ask the wrapped function who it is:

Whose dal is this?
print(fetch_rate.__name__)   # 'wrapper'  -- Meera's name is gone!
print(fetch_rate.__doc__)    # None       -- and so is her docstring

Every function we wrap now claims to be called wrapper. Every box in the city says "packed by TiffinCo". This is not cosmetic: tracebacks in your logs will point at wrapper instead of the real function, debuggers get vaguer, and — crucially for your world — frameworks that read a function's name and docstring to build something will read the wrong ones.

The standard library fix is functools.wraps — a helper that copies the original function's identity (its __name__, docstring, module, and friends) onto the wrapper, and leaves a pointer back to the original in __wrapped__. It is, delightfully, itself a decorator — one that you put on the wrapper:

The professional version
import functools

def logged(fn):
    @functools.wraps(fn)          # copy fn's name-tag onto wrapper
    def wrapper(*args, **kwargs):
        print(f"calling {fn.__name__}")
        return fn(*args, **kwargs)
    return wrapper
Tip

Treat @functools.wraps(fn) as a non-negotiable part of the wrapper pattern — write it on muscle memory, the way you write a closing bracket. Nearly every "why does my tool show up with the wrong name" mystery in agent code traces back to a wrapper that forgot it.

Decorators with arguments: choose the plan first

You will constantly meet decorators wearing brackets: @retry(times=3), @app.route("/health"), @mcp.tool(). The brackets change the order of events, and it is worth getting this crisp because it explains a lot of otherwise-baffling syntax. Bare @logged hands your function to logged. But @retry(times=3) is two steps: first Python calls retry(times=3) — which must return a decorator — and then that returned decorator receives your function. Meera choosing the "deliver hot by 1 pm" plan: configure the service first, then the configured service wraps the dish.

So a decorator-with-arguments is a function that makes decorators — three nested layers, each one remembering things for the layer below via a closure. Click through the lines:

retry.py — a decorator that takes arguments
import functools
def retry(times):
Layer one: the factory. This is NOT the decorator — it is called first, as retry(3), and its whole job is to remember times in a closure and hand back the real decorator.
def decorator(fn):
Layer two: the actual decorator, shaped exactly like logged — function in, replacement out. It receives your function only after the factory has run.
@functools.wraps(fn)
The name-tag, as always. Without it, every retried function in your logs would be called wrapper.
def wrapper(*args, **kwargs):
Layer three: the wrapper that will replace fn. Through closures it can see both fn (one level up) and times (two levels up). Three backpacks, nested.
last_error = None
for attempt in range(1, times + 1):
times travels here from the very first call, retry(3), made possibly seconds — or in a server, days — earlier. Closures do not expire.
try:
return fn(*args, **kwargs)
The happy path: the first success returns immediately, and the remaining attempts never happen.
except Exception as err:
On failure, note the error and go around the loop again. Production versions also sleep with backoff between attempts.
last_error = err
raise last_error
Budget exhausted: re-raise the final error so the caller sees the real failure, not a polite shrug.
return wrapper
return decorator
The factory returns the decorator, completing the chain: @retry(3) means fetch = retry(3)(fetch) — call the factory, then let its product wrap your function.
@retry(times=3)
In use. Read it aloud: "wrap the function below with a three-attempt retry". Any network call in an agent deserves this hat.
def call_model(prompt):
...
Careful

This is why @mcp.tool() has brackets and @tool often does not. Empty brackets still mean "call the factory first, with default settings". Writing @mcp.tool (no brackets) when the framework expects @mcp.tool() hands your function to the factory instead of the decorator — a classic, confusing bug. When in doubt, copy the exact form from the library's docs.

Decoding the ones you meet every day

You now hold the master key. Every famous decorator is one of three moves, and you can usually tell which from context:

FamilyExamplesWhat you get back
Wrap and replace@timed, @functools.lru_cache, @retry(3)A new function that adds behaviour around the original on every call
Register and return@app.route("/health"), @mcp.tool()Your own function, now also listed in a framework's lookup table
Transform@dataclass, @toolSomething rebuilt or upgraded — a class with generated methods, or a Tool object with a schema

@app.route("/health") — Flask's register-and-return classic. The factory is called with the URL; the decorator it returns writes an entry in the app's URL table — "when a request hits /health, call this function" — and then hands your function back unchanged. No wrapper at all. FastAPI's @app.get("/health") plays the same game. A web framework is, at heart, a dict from URLs to functions, and the decorator is how entries get in.

@dataclass — proof that decorators can receive classes, not just functions. It reads your class's type-hinted fields — the hints you learned in the previous chapter — writes __init__, __repr__, and __eq__ for you, and returns the same class, upgraded. With arguments, @dataclass(frozen=True), it is the factory pattern again.

@tool — LangChain's. This one transforms: it reads the function's __name__, its docstring, and its type hints, builds the JSON schema the model will see, and gives you back a Tool object with the original function inside. Now you know why functools.wraps was a whole section: a wrapper without a name-tag would feed the framework the wrong name and an empty description, and your agent would advertise a tool called wrapper that does "no description". The docstring you write above a tool function is not a comment — it is the prompt the model reads when deciding whether to call it, as you will see properly in the tool-calling chapter.

@mcp.tool() — FastMCP's register-and-return, from the MCP Python SDK. Empty brackets, so the factory runs with defaults; the decorator adds your function to the server's tool table so it is advertised over the protocol to any connected client. Same idea as Flask's URL table, different audience: instead of browsers hitting URLs, models are calling tools.

In the wild

Open any agent repository and search for the @ character — it will light up like a runway. Tool files are stacks of @tool or @mcp.tool() over small functions; API layers are @app.get and @app.post; config objects wear @dataclass; network calls wear @retry hats from tenacity; and test files are full of @pytest.fixture. In merge requests, "can we just decorate it?" means "add this behaviour without editing the function", and "the decorator is swallowing the signature" means someone forgot functools.wraps. When a stack trace in your logs points at a function called wrapper in a file you have never opened — you now know exactly what happened, and roughly where to look.

Check yourself

1. What is @timed written above def report(): literally shorthand for?

The @ is pure sugar for calling the decorator with the fresh function and re-tying the name to whatever comes back. It happens once, when the def runs (usually at import) — the wrapping it sets up is what runs on every call.

2. @app.route("/health") has brackets. What happens first?

Brackets mean factory-first: the call runs immediately and must return a decorator, which then receives your function. That is also why @mcp.tool() needs its empty brackets.

3. Your agent's logs show every tool is named wrapper with no description. What is missing?

Without functools.wraps, the wrapper's own identity replaces the original's — so anything reading __name__ and the docstring (loggers, tracebacks, schema builders like @tool) sees "wrapper" and nothing.

4. After decoration, where does the wrapper keep its reference to the original function?

The wrapper was born inside the decorator, where fn was in scope — so it remembers fn in its closure backpack forever. Nothing global, nothing re-imported.

5. What does @dataclass receive, and what does it give back?

Decorators can take classes too. @dataclass reads the type-hinted fields, generates the boilerplate methods, and returns the upgraded class — a transform, not a wrap.

Key takeaways

  • A decorator is a plain function that takes a function and returns its replacement. @name is exactly f = name(f) — nothing more is happening.
  • The @ line runs once, at definition time (usually import), not on every call; the wrapping it installs is what runs on every call.
  • Wrappers remember the original function through a closure, and *args, **kwargs lets one wrapper forward any call shape unchanged.
  • @functools.wraps(fn) keeps the wrapped function's name-tag. Skip it and logs, tracebacks, and schema builders all see an anonymous wrapper.
  • Brackets after a decorator — @retry(3), @app.route("/x"), @mcp.tool() — mean a factory is called first and its product does the decorating.
  • The famous ones are three moves in disguise: wrap-and-replace (@timed, @lru_cache), register-and-return (@app.route, @mcp.tool()), and transform (@dataclass, @tool).
