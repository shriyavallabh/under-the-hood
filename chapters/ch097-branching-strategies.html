Feature branches, release branches

Ten people, one codebase, nobody stepping on anybody. By the end of this chapter you will know exactly which branch to create, from where, what to name it, and every station a change passes through on its way from your keyboard to main.

~30 minChapter 98 of 117
The story

A busy restaurant keeps one master recipe book at the pass. Every dish that leaves the kitchen is cooked from it, so the book has an iron rule: nothing goes in until it is proven. When a chef wants to improve the dal, she does not scribble in the master. She photocopies the book as it stands today, takes the copy to her own station, and experiments there — tasting, adjusting, failing in private. The other five chefs are doing the same with their own copies, on their own dishes, at the same time. Nobody blocks anybody.

When her new dal is ready, the head chef tastes it, and only then is the one changed page copied into the master book. Her photocopy goes in the bin — it did its job. And twice a year something different happens: the kitchen freezes a copy of the book to send to the printers as the seasonal menu. Only spelling fixes may touch that frozen copy; new dishes keep flowing into the master for next season. The master book is main. The photocopies are feature branches. The printer's copy is a release branch. This chapter is how professional kitchens — sorry, teams — run that system every day.

Why teams branch at all

Quick recap from the previous chapter: a commit is a snapshot of the whole project, every commit points at its parent, and a branch is nothing more than a movable label pointing at one commit. Creating a branch costs nothing — Git just writes a 41-byte file with a commit id in it. Merging joins two lines of history back together. That is the machinery. This chapter is about the social contract teams build on top of it.

The problem branches solve is old and painful: many people, one codebase. If everyone edited the same copy directly — the way two people fight over one shared document — you would get half-finished features tangled into each other, one person's broken experiment blocking everyone's release, and no way to say "the version we ship customers is this one". So teams adopt three rules, and essentially every engineering organisation you will work with runs on some flavour of them:

  • main is sacred. The default branch always builds, always passes tests, and is always in a shippable state. Nobody works directly on it.
  • All work happens on short-lived branches. One branch per change, cut from main, merged back within days.
  • The way back into main is gated. A merge request with a human review and a green pipeline — never a direct push.

This is not politeness. On GitLab, main is usually a protected branch — a branch the server refuses direct pushes to, so the branch-and-merge-request path is physically the only door in. If you have ever seen a push rejected with remote: You are not allowed to push code to this branch, you have met the bouncer.

The question everyone is afraid to ask: what exactly am I branching?

People new to Git quietly worry about this, so let us kill the mystery. You do not branch a file, or a folder, or a chosen subset of the code. A branch always carries the entire repository — every file, exactly as it stood at the commit you branched from. Your feature branch for a one-line fix contains the whole codebase; you simply change one line of it, and at merge time Git folds in only what differs.

The real question is not "which files" but "which starting point". A new branch begins life pointing at whatever commit you were standing on when you created it. That gives us the rule that answers ninety per cent of "where do I branch from":

Tip

New work branches from up-to-date main. Stand on main, pull the latest, then cut the branch. Branching from a stale main — or worse, from some other half-finished feature branch — means you start life already behind, and every commit that landed since becomes a merge conflict waiting for you at the end.

The exceptions are few and follow one principle: branch from the thing you intend to change. Here is the whole decision table:

You want to…Branch fromTypical name
Build a new featureup-to-date mainfeature/invoice-rounding
Fix a bug that will ship with the next releaseup-to-date mainfix/duplicate-webhook-calls

One more anxiety worth removing: which repository? The one the change belongs to — if the fix is in the ingestion service, you branch in the ingestion service's repo. Branches never span repositories; a change touching two repos is two branches and two merge requests, linked in their descriptions.

The life of a feature branch, station by station

Here is the full journey — the same seven stations every change on your team passes through, whether it is a one-line prompt tweak or a month of work someone should have split up. Step through it:

1. Stand on fresh main

originthe shared server copy
→
main (local)git switch main, git pull

Before anything else: switch to main and pull. You want your starting point to be what the team has right now, not what it was when you last looked on Tuesday.

2. Cut the branch

mainuntouched from here on
→
feature/invoice-roundinggit switch -c …

One command creates the branch and moves you onto it. It points at the same commit as main — identical twins, until your first commit. Main stays clean behind you.

3. Commit small, focused changes

main
feature branchcommit → commit → commit

Edit, test, commit; repeat. Each commit is a save point you can retreat to. Everything here is invisible to the rest of the team — this is your own station in the kitchen.

4. Push and open a merge request

feature branchgit push -u origin …
→
GitLabmerge request: feature → main

Pushing publishes your branch to the server; the merge request says "I would like this folded into main, please look". Review and the CI pipeline both happen here — chapters 99 and 101.

5. Stay in sync while you wait

mainmoved ahead — teammates merged
→
feature branchgit fetch, then merge origin/main in

While your MR waits, main keeps moving. Pull main's new commits into your branch every day or so — small, frequent syncs mean small, boring conflicts instead of one horror at the end.

6. Merge — the change enters main

mainnow contains your change
←
merge requestapproved + pipeline green

A reviewer approves, the pipeline is green, someone clicks Merge. Your commits become part of main's history. This is the only door into main, and it just closed behind your change.

7. Delete the branch, start clean

feature/invoice-roundingdeleted — job done
maingit switch main, git pull — ready for the next branch

The branch was scaffolding, not architecture. Its commits live safely in main now, so the label is deleted — on the server (GitLab's "delete source branch" tickbox) and locally. Photocopy, bin. Next change, next branch.

Two habits separate pleasant teams from miserable ones. First: one change per branch. A branch called feature/rounding-and-new-logging-and-retry-fix is three branches wearing a trenchcoat — impossible to review well, impossible to revert cleanly if one of the three turns out bad. Notice an unrelated bug while working? Note it, finish your branch, cut a fresh one for the bug. Second: keep branches short-lived. A branch is a photocopy of a book that keeps being edited; every day it lives, it drifts further from reality and the eventual merge gets uglier. Days, not weeks. Genuinely big work lands as a series of small branches that each keep main working.

Careful

The classic beginner accident: you finish one feature, and — still standing on that feature branch — you run git switch -c for the next one. Now branch two silently contains all of branch one, and its merge request shows both changes mixed together. Always return to main and pull before cutting a new branch. If your MR's diff ever shows changes you did not make, this is usually what happened.

The same lifecycle in commands

Here is the whole journey as the actual terminal session — the dozen commands you will type, in this order, hundreds of times in your career. Click each line to unfold what it is doing and why it is there.

one feature, branch to merge — the full command trail
git switch main
Move to the main branch. switch is the modern command for changing branches; you will also see the older git checkout main in teammates' terminals — same effect.
git pull
Fetch the latest commits from the server (origin) and fold them into your local main. Now your starting point matches the team's reality. Skipping this is how people branch from last week by accident.
git switch -c feature/invoice-rounding
The -c means create: make a new branch at the commit you are standing on and switch to it, one move. This is the moment the photocopy is made. (Older form: git checkout -b feature/invoice-rounding.)
git add billing/rounding.py tests/test_rounding.py
After editing, stage exactly the files that belong to this change — remember staging from chapter 96. Naming files beats git add ., which happily scoops up stray debug scripts.
git commit -m "Round invoice totals to two decimals"
Snapshot the staged change with a message that says what it does. Several small commits are fine — they will be reviewed together in the MR anyway.
git push -u origin feature/invoice-rounding
Publish the branch to the server. -u links your local branch to the remote one, so from now on plain git push and git pull know where to go. GitLab replies with a ready-made URL for opening the merge request.
git fetch origin
Days pass; the MR is in review; main has moved. fetch downloads the server's new commits without touching your files — you are just refreshing your local picture of reality.
git merge origin/main
Fold main's new commits into your branch, resolving any conflicts now, while they are small. Some teams prefer git rebase origin/main, which replays your commits on top of new main for a straighter history — ask which convention your team uses.
git push
Update the branch on the server so the merge request — and its pipeline — see the synced version.
git switch main
The MR merged. Your change lives in main now; time to leave the scaffolding behind.
git pull
Bring your own merged change (and everyone else's) down into local main. Satisfying moment: you pull your own work back as part of the shared history.
git branch -d feature/invoice-rounding
Delete the local branch label. Lowercase -d is the safe version — it refuses unless the work is merged. The commits are not deleted; only the sticky note pointing at them is.
Note

If your team merges with squash (GitLab compresses your whole branch into one new commit on main), git branch -d may protest that the branch is "not fully merged" — Git cannot see your original commits inside the squashed one. Once you have confirmed the MR merged on GitLab, deleting with -D is fine. This confuses everyone exactly once.

Naming: boring, predictable, greppable

Branch names are communication. On a team of ten, git branch -r lists dozens of branches; good names let anyone tell at a glance what each is and who cares about it. The near-universal convention is a type prefix, a slash, then a short lowercase description with hyphens — plus the ticket id if your team tracks work in an issue tracker: feature/PAY-482-invoice-rounding. Release branches: freezing a version without stopping the kitchen

Feature branches manage parallel work. A release branch manages parallel time: it lets the team stabilise and ship version 2.4 while main is already becoming version 2.5. It is a long-lived branch, cut from main at the moment of a planned freeze, that accepts only bug fixes from then on. The seasonal menu at the printers: typo corrections yes, new dishes no.

1. Cut release/2.4 from main

mainfeatures keep landing here
→
release/2.4frozen: fixes only from now on

On freeze day, someone cuts the release branch from main. Everything that made the cut is in; anything merged to main after this moment waits for 2.5. The train doors close.

2. Two timelines run side by side

mainnew features → version 2.5
release/2.4testing finds bugs → fix branches target here

QA and pilot users hammer the release branch. Fixes for what they find are small branches cut from — and merged back into — release/2.4 itself. Meanwhile main charges ahead, unbothered.

3. Tag and ship from the release branch

main
release/2.4tag v2.4.0 → deployed

When the release branch is stable, its tip is tagged v2.4.0 — a permanent, immovable label from chapter 96 — and that exact commit is what goes to production. Everyone can point at precisely what shipped.

4. Merge the fixes back to main

mainreceives the 2.4 fixes too
←
release/2.4its fixes must not be lost

Every fix made on the release branch is also merged back to main — otherwise version 2.5 ships with bugs that 2.4 already fixed. Regressions of exactly this kind are why the step is ritual, not optional.

5. Months later: a production emergency

mainalready deep into 2.5 — too risky to ship
release/2.4hotfix branch → tag v2.4.1

A crash appears in production long after the freeze. You cannot ship main — it is full of unreviewed 2.5 work. So the hotfix branches from release/2.4, ships as v2.4.1, and merges back to main as well. This is the moment release branches exist for.

Who actually uses them? Teams that ship versions: mobile apps waiting on store review, libraries with numbered releases, on-premise software, and regulated environments — a large bank cannot deploy main forty times a day, so it ships audited, tagged releases cut from release branches. At the other end sits trunk-based development: everyone merges tiny branches into main daily and main itself deploys continuously, tags marking what went out — no release branches, because there is no gap between "done" and "shipped" to manage. You inherit your team's choice rather than make it: if the repo has release/* branches, follow the freeze rules; if not, short feature branches into main is the whole game.

In the wild

Open any active GitLab project and the strategy is visible in the furniture. The branch dropdown shows the prefixes: feature/…, fix/…, maybe a release/3.2. Settings → Repository lists the protected branches — main, and usually release/* — with "Allowed to push: No one", which is why everything travels by merge request. Every MR names a source and target branch at the top: source feature/x, target main for normal work; target release/2.4 for a stabilisation fix — the target line is the fastest way to read what kind of change you are looking at. The "Delete source branch when merge request is accepted" tickbox is the after-merge cleanup from the lifecycle, automated. And when a teammate says "that missed the 2.4 cut" or "cherry-pick it onto the release branch", you now know they are talking about the freeze in step one of the release stepper.

Check yourself

1. You are about to start a new feature. Where does the branch come from?

Switch to main, pull, then git switch -c feature/x. Branching from a stale main means starting life already behind; branching from another feature branch drags that branch's changes into yours.

2. Your feature branch is for a one-line fix in billing/rounding.py. Which files does the branch contain?

A branch is never a selection of files — it always carries the whole repository snapshot. You change one line; at merge time Git folds in only what differs.

3. Why do teams insist on one change per branch?

Git has no such limits — the rule is for humans. A branch mixing three changes is hard to review, and if one of the three breaks production you cannot revert it without dragging the innocent two along.

4. What is a release branch for?

A release branch like release/2.4 is cut from main at a planned freeze, accepts only fixes, gets tagged and shipped — and its fixes are merged back to main so the next version does not regress.

5. Your merge request just merged. What happens to the feature branch?

Deleting a merged branch removes only the label; the commits are safe in main's history. Reusing old branches is how unrelated changes leak into your next merge request.

Key takeaways

  • Branches exist so many people can change one codebase in parallel: main stays always-shippable, all work happens on branches, and merge requests are the only door back in — enforced by protected branches, not politeness.
  • You branch the whole repository, never individual files — and you branch from up-to-date main (switch, pull, then git switch -c feature/x) unless you are hotfixing a shipped version, in which case you branch from its release branch or tag.
  • The lifecycle is always the same seven stations: fresh main, cut branch, small commits, push and open a merge request, sync with main while you wait, merge, delete the branch and pull.
  • One change per branch, and short-lived — days, not weeks. Small branches review well, revert cleanly, and merge without drama; long branches rot as main drifts away beneath them.
  • Name branches with the house convention — type prefix, slash, lowercase-hyphenated description, ticket id if the team uses one: feature/PAY-482-invoice-rounding. Thirty seconds of git branch -r reveals the local style.
  • Release branches freeze a version (release/2.4) so it can be stabilised, tagged, and shipped while main becomes the next version; fixes made there merge back to main. Teams that deploy main continuously skip them entirely — follow whichever system your repo already uses.
