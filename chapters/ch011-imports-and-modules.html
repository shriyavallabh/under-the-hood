What import actually does

The lines at the top of every Python file look like boilerplate. They are not. Each one is a courier with a strict search route, a one-time reading ritual, and a superb memory. By the end of this chapter you will know exactly what happens — step by step — when a file says import.

The story

It is your first week at a big law firm, and you need the Stamp Duty Ready Reckoner. You do not go hunting for it yourself — you ask the archives clerk, a man of iron routine. First he checks the trolley beside his desk, where every book fetched earlier that day already sits. If your book is on the trolley, you get that same copy, instantly. If not, he walks his fixed route, always in the same order: the firm's own shelf first, then the standard reference wall, then the bought-in collection at the back. The first place that has the title wins; he never keeps searching after a match.

And here is his one eccentric habit: the very first time a book is fetched, he reads it cover to cover and sticks an index tab on every section — so that everyone after you gets a fully tabbed copy in seconds. Back at your desk, you write a short label on a slip of paper: "Reckoner". That slip is yours alone; the book is shared.

That clerk is import. The trolley, the route, the one-time reading, the label on your desk — Python does all four, every time, in exactly that order.

You have been taking these lines on faith

In the last chapter you read a real script that called a live LLM, and its first two lines were import os and import requests. You could read every other line by then, but those two you took on trust: "this makes os and requests available, somehow." And back in names and objects you learned that Python's assignment is just sticking a name label on an object. Hold on to that idea, because it is about to pay off: import is an assignment in disguise. It finds a file, runs it once, and binds a name to the result. Three moves. That is the whole trick — the rest of this chapter is watching each move in slow motion.

First, meet the three spellings you will see at the top of nearly every file:

the three faces of import
import json                    # fetch the whole book; label it "json"
from pathlib import Path       # fetch the whole book; label ONE chapter: "Path"
import numpy as np             # fetch the whole book; label it with a nickname

The differences are smaller than they look. import json brings the whole module to your desk under its own name — you then reach into it with a dot: json.loads(...). from pathlib import Path does exactly as much work — the whole pathlib file still runs, cover to cover — but only one name, Path, lands on your desk. It is the photocopied chapter, not a lighter fetch. And as is nothing but a relabel: the book is the same, your slip of paper says np. That is it. Three keywords, one mechanism, different labels.

Note

A common early misconception: "I used from pathlib import Path so Python only loaded the Path part — that must be faster." It is not. The module always runs in full; from ... import ... only changes which names get copied onto your desk afterwards. Choose between the forms for readability, never for speed.

A module is a file. A package is a folder.

Time to name the machinery properly. A module is simply a .py file — one file, one module, no ceremony. The file tools.py is the module tools. A package is simply a folder that groups modules together, so that related files can travel under one name. When you see a dotted name in an import, you are reading a folder path with the slashes swapped for dots:

a small project, and the imports that reach into it
# the folder tree on disk          # the import that reaches it
#
# agents/                          import agents
#     __init__.py
#     tools.py                     import agents.tools
#     prompts.py                   from agents.prompts import SYSTEM_PROMPT
#     memory/
#         __init__.py
#         store.py                 from agents.memory.store import save

So agents.memory.store is not an incantation — it is a street address: folder agents, subfolder memory, file store.py. Once you see dotted imports as file paths, unfamiliar codebases become navigable: every from x.y.z import thing at the top of a file tells you exactly which drawer to open.

Which leaves that odd little file, __init__.py — the index card taped to the front of the cabinet. It does two jobs. First, it marks the folder as a package: "these files belong together, treat this folder as importable." Second — and this surprises people — it is a real module that runs whenever the package is first imported. Most __init__.py files are empty, and empty is perfectly respectable. But library authors often use it as a front desk, lifting the best names up to the surface so users can write from agents import Researcher instead of memorising the internal file layout:

agents/__init__.py — a front desk for the package
# Lift the names users need up to the package's front door.
from agents.researcher import Researcher
from agents.tools import web_search
Tip

Modern Python can technically import from a folder with no __init__.py (a "namespace package"), but nearly every real project still includes the file — it makes intent explicit and keeps tools happy. When you browse a repository and see empty __init__.py files everywhere, nothing is missing. They are load-bearing blanks.

The search route: how Python finds things

When Python meets import agents, where does it look? Not "everywhere" — that would be slow and unpredictable. It follows one ordered list, called sys.path — a plain Python list of folder paths that Python walks from front to back, taking the first match it finds. And before it takes even one step, it checks the trolley. Step through the route:

1. Check the trolley first: sys.modules

Every import begins with a lookup in sys.modules, a dict (the very kind you met in collections) mapping module names to already-loaded module objects. Cache hit? What actually happens at import time

Here is the part that changes how you read every Python file from now on. When the clerk fetches a book for the first time, he reads it cover to cover — and Python does this literally: on first import, the module's code runs, top to bottom, exactly once. Every line. def lines execute and build function objects, just as you saw in functions — a def is not a passive declaration, it is a statement that runs and creates something. Top-level assignments run. And any loose print(...) or network call sitting at the top level of the file? It fires the moment someone imports the module — which is why well-behaved libraries keep their top level quiet: define things, do nothing.

The finished module object — with all its functions and variables attached — then goes on the trolley, and only then does your name get bound. Three moves, remember: find (the route), run once (the reading), bind (the label). This one fact explains several things you may have noticed operating real systems: why the first request to a freshly started service is slow (imports are running), why a config value read at the top of a module ignores changes made later (it ran once, at import), and why restarting a process is the only way to truly re-import everything.

Careful

"Runs once per program" cuts both ways. If a module's top level does something heavy or has side effects — opens a connection, reads a large file — every importer pays that cost the first time, and no one can opt out. When you see agent frameworks lazily importing things inside functions, this is what they are dodging: they want startup fast and side effects deferred.

Reading a real import block — and the two dialects

You now know enough to read the top of any professional Python file the way a reviewer does. Below is the header of a typical file inside an AI project — click each annotated line:

agents/researcher.py — the import block, annotated
import os
Standard library, whole-module form. Found on the reference wall (stepper step 3). The convention: stdlib imports come first, as their own group.
import json
Also stdlib. Note both lines use plain import — the file will say json.loads(...) later, and the json. prefix tells every reader exactly where that function came from. Explicit beats short.
from pathlib import Path
The photocopied chapter: all of pathlib runs, but only Path lands in this file's namespace. Used when one name is needed often and its origin is obvious.
import requests
The blank line above is deliberate — group two: third-party libraries, fetched from site-packages because someone ran pip install requests. Same library your chapter 10 script used to call the LLM API.
from agents.prompts import SYSTEM_PROMPT
Group three: the project's own code. This is an absolute import — it spells out the full address from the top-level package down: folder agents, file prompts.py, name SYSTEM_PROMPT. Anyone reading it knows precisely which file to open.
from .tools import web_search
A relative import — the leading dot means "in MY folder": since this file lives in agents/, .tools is agents/tools.py. Two dots (..) would mean the parent folder. Compact, but it only works for files that are part of a package.

Those last two lines are the two dialects for importing your own code, and teams argue mildly about them. Absolute imports (from agents.tools import web_search) are the widely preferred default: unambiguous, greppable, and they read the same from anywhere in the project. Relative imports (from .tools import web_search) shine inside libraries — if the package is ever renamed, none of its internal imports need to change. The one sharp edge worth remembering: relative imports only make sense inside a package. Run a file directly as a script — python researcher.py — and Python does not consider it part of any package, so its relative imports die with ImportError: attempted relative import with no known parent package. If you ever hit that message, you now know it is not the code that is broken; it is how the file was launched.

Circular imports, briefly

One last failure mode, because you will meet it in the wild and it looks scarier than it is. Suppose a.py imports b, and b.py imports a. Each needs the other. The clerk does not deadlock — the cache saves him from that — but he does something subtler and sneakier:

1. a.py starts running — and is cached immediately

ahalf-finished, but already on the trolley
bnot started

The moment Python begins running a.py, it places the (still mostly empty) module a into sys.modules. The book goes on the trolley before the reading is finished — normally harmless, and the key to what happens next.

2. Line 1 of a.py: import b — so b.py starts running

apaused at line 1
→
bnow running top to bottom

a pauses while b is read cover to cover. So far, business as usual — modules importing modules is the normal state of every project.

3. Line 1 of b.py: from a import helper — cache hit on a half-book

basks the trolley for a.helper
→
aon the trolley, but helper is defined LOWER in the file — it does not exist yet

b asks for a. The trolley has it — cache hit! But a.py is paused at line 1: everything below that line, including def helper(), has not run yet. b receives a half-finished module.

4. ImportError: cannot import name 'helper'

ImportErrorcannot import name 'helper' from partially initialized module 'a'

Python even says "partially initialized module" in modern error messages — the diagnosis is in the traceback. The cure is structural: move the shared thing into a third module both can import, or (the pragmatic patch) move one import inside the function that uses it, deferring it until call time, when both modules are finished.

You do not need to master circular imports today — just recognise the smell. Two modules reaching for each other usually means one concept is living in two files, and the tidy fix is a third file both depend on. The next chapter is about exactly that kind of layout decision.

In the wild

Open any file in any serious Python repository — an agent framework, a data pipeline, a web service — and the first screen is an import block in the three-group convention you can now read: stdlib, third-party, own code. A line like from langchain.agents import create_agent is a street address into the installed langchain package in site-packages. Empty __init__.py files dot every folder tree on purpose. In CI logs and merge requests, ModuleNotFoundError: No module named 'x' almost always means "the pipeline's environment never installed x" — an environment problem, not a code problem — and when a reviewer comments "this will be a circular import", they mean two files reaching for each other's names before either has finished running. All of it is the clerk: the route, the trolley, the one-time read.

Check yourself

1. Your file says from pathlib import Path. How much of pathlib actually runs?

Every import form does the same work: find the module, run its file once in full, cache it. from ... import ... only changes which names are copied into your file's namespace afterwards — the photocopied chapter, not a lighter fetch.

2. Twelve files in your project all say import json. How many times is the json module's code executed?

The first import runs the file and puts the finished module on the trolley (sys.modules). The other eleven imports find it there and simply bind a name to the same object. That is why cs is colorsys printed True in the runner.

3. In the import from agents.memory.store import save, what is agents.memory.store, physically, on disk?

Dots in an import are folder separators: packages are folders, modules are files. Reading dotted imports as file paths is how you navigate an unfamiliar repository.

4. Inside agents/researcher.py, what does the dot in from .tools import web_search mean?

A leading dot is a relative import: "in my folder". Two dots mean the parent folder. It only works when the file is being treated as part of a package — run the file directly as a script and this line raises an ImportError about a missing parent package.

5. a.py imports b, and b.py starts with from a import helper — which crashes with "cannot import name 'helper' from partially initialized module". Why?

Modules go into sys.modules the moment they start running, not when they finish. b received a genuine cache hit — on a half-finished module whose lower lines (including def helper) had not executed yet. The fix is structural: a shared third module, or deferring the import into a function.

Key takeaways

  • Import is three moves: find the module (walk sys.path), run its file once top-to-bottom, and bind a name in your file — an assignment in disguise.
  • A module is a .py file; a package is a folder of them; dotted import paths are folder paths. __init__.py marks the folder as a package and runs first when it is imported — empty is normal.
  • from x import y and import x as y do exactly as much work as import x — they only change which labels land in your namespace.
  • sys.modules caches every imported module for the life of the process: the code runs once, everyone shares one object, and only a restart truly re-imports.
  • Absolute imports (from agents.tools import ...) are the readable default; relative imports (from .tools import ...) work only inside packages and break when a file is run directly as a script.
  • Circular imports fail because a module is cached before it finishes running — "partially initialized module" in a traceback is that exact story, and the cure is restructuring, not retrying.
