Tests: why they gate merges

A test is a promise written in code that the machine checks forever. Learn to write one, run one, and read the ones in every merge request first — that is where the real specification lives.

~30 minChapter 104 of 123
The story

At the pass of a busy restaurant kitchen — the counter where finished plates wait to go out — hangs a laminated checklist. Every line on it has a scar behind it. "Taste the dal before plating" was added the night a new cook grabbed sugar instead of salt. "No whole cardamom on top of the biryani" commemorates a customer who bit one. "Cold plate, cold dessert" — you can guess.

No plate leaves the kitchen without passing that checklist. Not because the head chef distrusts her cooks, but because on a Friday night, with six pans going, anyone can grab the wrong jar. The checklist is the reason the kitchen can move fast. And when a new cook joins, the fastest way to learn what this kitchen cares about is not the recipe book — it is that laminated card. Every line is a lesson someone paid for.

A project's test suite is that checklist. Each test is one line, added after a real scar or before an anticipated one. The pipeline is the pass. And "the merge button is greyed out until tests go green" is simply the rule that no plate leaves the kitchen unchecked.

A test is just a function that complains

Strip away the tooling and a test is a plain Python function that calls your code and then checks the result. The checking is done by assert, a one-word Python statement meaning "this must be true, or stop everything": if the condition holds, nothing happens and the line is silent; if it fails, Python raises an AssertionError — the same exception machinery you met in the errors chapter, just triggered deliberately.

tests/test_calculate.py — your first test, complete
from agent import calculate

def test_calculate_multiplies():
    assert calculate("1500 * 0.20") == "300.0"

Read it as a sentence: calculate, given "1500 * 0.20", must return "300.0". That is the whole art. pytest — the test runner nearly every Python team uses — finds functions like this by naming convention, no registration required:

  • Files named test_*.py, usually in a tests/ folder.
  • Functions inside them whose names start with test_.
  • Plain assert for the checks — no assertion library to learn. pytest's signature move: it rewrites your asserts so a failing report shows the actual values on each side, not just "False".

Notice what a test is not: proof of correctness, or paperwork. It is an executable promise. A wiki page quietly rots; a test that stops being true turns red on the very next run — which is why engineers trust suites more than READMEs.

Running them: pytest at the command line

From the project root, one word runs everything:

the commands you will actually type
pytest                       # discover and run every test
pytest -v                    # verbose: one named line per test
pytest -k "calculate"        # only tests whose NAME matches this expression
pytest -k "agent and not slow"   # -k understands and / or / not
pytest tests/test_agent.py   # just one file
pytest tests/test_agent.py::test_calculate_multiplies   # one exact test
pytest -x                    # stop at the first failure
pytest --lf                  # re-run only what failed last time

The -k flag is the one you will use daily. Real suites take minutes; while iterating on one function you filter to the tests that mention it by name — pytest -k "calculate" — and only run the full suite before pushing. The double-colon form (file.py::test_name) pins one exact test: it is what you paste from a CI failure log to reproduce it locally.

The vocabulary

Fixtures, gently: mise en place for tests

Before service, every cook lays out their station — chopped onions here, spice box there — so each dish starts from a clean, known setup. Kitchens call it mise en place; pytest calls it a fixture: a function that prepares something a test needs, so the test body can stay about the behaviour, not the setup.

The mechanism is charmingly odd at first. You mark a setup function with the @pytest.fixture decorator (a decorator being a label that changes how a function is treated). Any test that wants it simply names it as a parameter: pytest matches the name, calls the fixture, and passes in the result — a fresh copy per test, so tests can never contaminate each other.

pytest also ships fixtures ready-made: tmp_path hands your test a throwaway folder, cleaned up afterwards; monkeypatch lets a test temporarily replace something — an environment variable, or a real LLM call swapped for a scripted fake, the fake_llm trick this course uses. Shared fixtures live in conftest.py, visible to every test in the folder.

Reading a real test file — the reviewer's pass

Here is a fuller tests/test_agent.py, the kind you will actually meet in a merge request. Click the annotated lines. This is also where parametrize appears — a decorator that runs one test body over a whole table of cases, so ten checks cost three lines instead of ten copies.

tests/test_agent.py — as a reviewer reads it
import pytest
Only needed for the decorators below. The plain tests further down never touch pytest directly — that is the point of the plain-assert design.
from agent import calculate, run_agent
Tests import the real production code and exercise it — same import system as any other module. Nothing is mocked here except, later, the model.
def test_calculate_refuses_non_arithmetic():
The name is the specification. Read it aloud: "calculate refuses non-arithmetic". When this fails in CI, the failure line alone tells you which promise broke before you read any implementation.
assert calculate("__import__('os')").startswith("ERROR")
A security promise, pinned forever. If someone "simplifies" calculate next year and drops the character allowlist, this line turns red the same minute. Regression tests are institutional memory that never resigns.
@pytest.fixture
The decorator that turns an ordinary function into a fixture — a named piece of mise en place that pytest will serve to any test that asks.
def gst_task():
The fixture's name is its handle. Tests request it by using this exact word as a parameter name.
return "What is 20% GST on a 1500 rupee invoice?"
Whatever the fixture returns is what the test receives. Here it is just a string; in real suites it is often a configured agent, a fake model, or a temporary database.
def test_agent_answers_the_task(gst_task):
Magic, explained: the parameter name matches the fixture name, so pytest calls gst_task() and passes the result in. No import, no wiring — name-matching is the whole trick.
assert "300" in run_agent(gst_task)
Note the loose check: "300" appears somewhere in the answer, not one demanded exact sentence. Testing the behaviour you care about — not the incidental wording — keeps tests from breaking on every harmless rephrase. With LLMs in the loop this instinct becomes essential; the next chapter builds a whole discipline on it.
@pytest.mark.parametrize("expression,expected", [
Parametrize: the first argument names the test's parameters, the list supplies one tuple per case. pytest runs the body once per row and reports each row as its own test.
("1 + 1", "2"),
("10 / 4", "2.5"),
Each row is a promise. Adding a newly discovered edge case costs one line — which is why reviewers love seeing a parametrized table grow in a bug-fix MR: the bug arrives with its own tombstone.
("1500 * 0.20", "300.0"),
])
def test_calculate_arithmetic(expression, expected):
The parameter names match the string in the decorator. Three rows, one body: three tests.
assert calculate(expression) == expected
One assert, table-driven. If only the "10 / 4" row fails, the report says exactly that — you know instantly the bug involves division, not multiplication.

Tests are the first thing to read in an MR

Here is the habit that quietly separates confident reviewers from overwhelmed ones. When a merge request arrives — twelve files, four hundred changed lines — do not start at the top of the diff. Open the test files first. The implementation tells you how; the tests tell you what and why, in sentences:

test_agent_retries_once_on_tool_timeout. test_retry_gives_up_after_second_failure. test_timeout_is_logged. You now know precisely what this MR sets out to do — before reading one line of implementation. Then you read the code asking a sharper question: does it honestly earn those green ticks? And you review the tests themselves with three checks:

  • Do the tests match the story? If the MR description says "adds retry logic" and no test mentions retry, the central claim of the change is unverified — saying so is the most useful review comment a non-expert can make.
  • Do the names read as promises? test_retry_gives_up_after_second_failure is documentation; test_2 is a shrug.
  • Would the test catch the bug it claims to? A test that asserts result is not None passes for almost any behaviour, right or wrong. Green paint is not a checklist.

The same standards, condensed into a table you can review against:

Good tests…Weak tests…
Have names that read as behaviour: test_unknown_tool_returns_errorHave names that read as filler: test_case_3
Check one behaviour each, so a failure points at one causeAssert twenty things, so a failure points everywhere
Follow arrange, act, assert — set up, do the thing, check itInterleave setup and checks until nobody can follow the plot
Test observable behaviour ("the answer contains 300")Test internal wiring ("the private helper was called twice"), shattering on every refactor
Run fast and deterministic — no network, no real LLM, no sleepingDepend on live services, so failures mean "the wifi", not "the code"
Note

That last row explains a pattern you have seen all course: tests replace the real model with a scripted fake_llm. A unit test asserting on a live LLM's exact words would flicker red and green at random — the model is non-deterministic and the network is a lottery. Deterministic fakes keep this layer honest; judging the real model's quality is a different craft called evals, which is exactly where the next chapter picks up.

The gate: how GitLab turns red into "cannot merge"

Now connect this to the previous chapter's pipelines. Teams configure the project so a merge request cannot be merged unless its pipeline — which runs pytest, among other jobs — succeeds. In GitLab this is a project setting ("Pipelines must succeed"), and its effect is visceral: the merge button greys out. Walk the flow as the reviewer sees it:

The author fixes the code (or the wrong expectation), pushes, the suite reruns, and the button comes alive. A human approval plus a green verdict from the machine — the merge needs both.

Why teams block merges on red — the economics

The gate can feel bureaucratic until you price the alternative. The main branch is shared plumbing: everyone branches from it, every deploy is cut from it. Merge one silently-broken change and the breakage does not stay yours: everyone who branches from main inherits your bug, their tests fail for reasons they did not cause, and an hour of shortcut becomes a day of team-wide archaeology. Keeping main green is not perfectionism; it is cheap insurance, paid in seconds per merge.

The gate also does something subtler: it depersonalises quality. Without it, "your change broke things" is an accusation between colleagues, and a risky merge can go in because of who asked. With it, the pipeline is the referee — tireless, unbribable, and equally rude to everyone. A red X on a principal engineer's MR blocks exactly as hard as one on an intern's. Teams that block on red argue less, not more.

And the gate compounds. The suite is the team's laminated checklist: every incident that mattered got a test, so the whole history of past failures is re-checked on every push, for free. This is why reviewers grumble when a bug fix arrives without a test — the fix repairs today; only the test prevents the sequel. You saw agent failure modes catalogued in chapter 49; a mature agent codebase has a test pinning each one it has survived.

Careful

Green means "every promise we wrote down still holds" — nothing more. A suite full of weak asserts is a checklist that says "plate exists" instead of "taste the dal": comfortably green, catching nothing. When you review, weigh the tests, not the tick. And be suspicious of an MR that "fixes" a red pipeline by deleting or loosening the failing test: sometimes the expectation really was wrong, but that deletion deserves an explaining comment, every time.

In the wild

Open any serious Python repository and the shape is instantly recognisable: a tests/ folder mirroring the package layout, conftest.py files holding shared fixtures, and a CI config running pytest on every push. In merge requests the giveaways are everywhere: a reviewer commenting "can you add a test that pins this?", a table of parametrize rows growing by one in a bug-fix MR, a red "1 failed" badge next to a greyed-out merge button, and someone pasting pytest tests/test_agent.py::test_retry_gives_up -x into the discussion so others can reproduce a failure locally. When a data scientist's MR changes behaviour but touches no test file, the most valuable review comment you can make costs one line: "which test would catch this if it regressed?"

Check yourself

1. A 400-line merge request lands in your review queue. Where do you look first, and why?

Descriptions can be stale and diffs tell you how — only the tests state what must now be true, in names that read as sentences and in asserts the machine has already verified. Read them first, then check the implementation earns them.

2. What happens when the condition in a plain assert is false during a pytest run?

assert raises AssertionError on a false condition — the exception machinery from chapter 9. pytest's assertion rewriting turns that into the helpful report that shows the real values on both sides.

3. You are iterating on the calculator tool and the full suite takes four minutes. What do you type?

-k filters by test name expression (it even understands and/or/not), giving you seconds-fast feedback on the code you are touching. There is no --fast flag, and commenting out tests is how promises quietly stop being checked.

4. How does a pytest test get hold of a fixture?

Name-matching is the whole trick: mark the setup function with @pytest.fixture, and any test that lists that name as a parameter receives what the fixture returns — freshly made per test, so tests cannot contaminate each other.

5. Why do teams disable the merge button on a red pipeline instead of trusting authors to fix failures after merging?

Main is shared plumbing — one silently-broken merge becomes everyone's mystery failure. The gate is policy, not a technical limit: it makes the pipeline a referee that blocks a principal engineer exactly as hard as an intern — which is why it works.

Key takeaways

  • A test is a plain function whose name starts with test_, in a file named test_*.py, using plain assert — pytest discovers it by convention and, on failure, shows the real values on both sides.
  • pytest -k "expression" runs a named subset for fast iteration; file.py::test_name pins one exact test — the thing you paste from a CI log to reproduce a failure locally.
  • Fixtures are mise en place: declare setup once with @pytest.fixture, and tests receive it by naming it as a parameter. Parametrize turns one test body into a table of promises, one row per case.
  • Tests are executable documentation — the first thing to read in any merge request. Names that read as behaviour tell you what the change intends before you open the diff.
  • Good tests check one observable behaviour each, run fast and deterministic (scripted fake_llm, never the live model), and pin every bug that ever mattered so it can never quietly return.
  • Merges block on red because main is shared plumbing and the pipeline is an impartial referee: it depersonalises quality, protects everyone downstream, and costs seconds per merge against days of archaeology.
