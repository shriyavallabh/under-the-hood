The checkpoint graph: branching and time-travel

A conversation's history is not a line - it is a graph, exactly like Git. Parent pointers, branches, a head to mark where you are, and the split that keeps light references in a fast database while the heavy state rests in a blob.

~35 minChapter 135 of 135
The story

Think of a choose-your-own-adventure book. Most of the time you read straight through, page after page. But the magic is the moment you go back to page 40, take the other door, and a whole new storyline unfolds from that same point - without erasing the ending you already read. The book is not a single line of pages; it is a tree of possibilities, and a bookmark tells you which path you are currently on. Every page quietly remembers the page it came from, which is the only trick you need to walk any storyline backwards.

An agent's conversation history works the same way, and you already felt it in the time machine chapter when you branched a conversation to try a different path. Now we look at how that is actually stored - and it turns out to be the exact model behind a tool you will meet in the Git chapter. Each checkpoint remembers its parent. Follow parents backwards and you get a history; attach a new checkpoint to an old one instead of the newest and you get a branch. Master this one idea - a graph of checkpoints, each pointing at its parent - and time-travel, undo, and what-if all fall out for free.

History is a graph, not a line

The naive picture of a conversation is a straight list: turn one, turn two, turn three. The real structure is richer. Each checkpoint carries three things: its own id, the state it captured, and a parent pointer - the id of the checkpoint it came from. When every new checkpoint points at the most recent one, you get a straight line. But nothing forces that. Point a new checkpoint at an older one and the history forks into a branch. The whole thing becomes a directed graph of checkpoints - the same shape, and very nearly the same vocabulary, as Git.

Head: which storyline am I reading?

Once branches exist, "give me the latest checkpoint" is ambiguous - the latest on which branch? That is what a head answers. A head is a pointer to the tip of one branch, exactly like a bookmark or a Git branch name. When you fetch the latest state you pass a head, and you get the newest checkpoint on that storyline; when you load history, you pass a head and walk that branch's parents back to the root. Time-travel is then just choosing an earlier checkpoint as your head and carrying on from there - the future you had already written is still on disk, on its own branch, untouched.

3. Head decides what "latest" means

head = c4
→
latest on this branch: c4

Ask for the latest with head at c4 and you get c4 and its history c1, c2, c4. Point head back at c3 and you would get the original storyline instead. One graph, many readable paths.

Pending writes: the pause between checkpoints

Real graphs stop in the middle. When an agent interrupts for a human approval, it has done some work but not reached a clean checkpoint. That half-finished work is saved as pending writes - separate little objects attached to the last checkpoint. When you load state, you get a pair: the checkpoint plus its pending writes, so the agent can resume exactly where it paused rather than redoing the step. It is the difference between "saved the page" and "saved the page plus the sentence I was halfway through writing in the margin."

Where the state actually lives

A subtle, important design choice closes the loop. The graph of references - ids, parent pointers, pending writes, the partition key of user, conversation, and agent - is small and is what you query and traverse, so it lives in a fast, indexed database. But the state payload itself can be large, and it is only ever fetched whole, never queried field by field. So many systems store just the reference in the database and put the heavy state in blob storage - structure-agnostic, as long as it is valid JSON. The backend looks up the reference by its partition key, then fetches the exact state from the blob. Two happy side effects: you dodge the document database's object-size limits, and because a blob is not a queryable database, there is no place for a SQL-injection attack to land.

In the wild

You will see a checkpoint record carrying a parent_id, a "get latest" call that takes a head or branch argument, and a store split into a small reference table plus a blob per state. A product's "fork this chat" or "edit and rerun" button is a branch from a checkpoint; "resume where it paused" is pending writes being replayed. When someone says the history is "a DAG, not linear" or "it works like Git", this is the chapter they are describing - and the reference-in-database, state-in-blob split is a pattern you will meet far beyond agents.

Check yourself

1. What single piece of data turns a list of checkpoints into a branchable graph?

Follow parent pointers backwards to get a history; attach a new checkpoint to an older parent instead of the newest and you get a branch. That one link is the whole graph, exactly like Git.

2. Once branches exist, what does a "head" give you?

A head is a branch pointer, like a bookmark. Fetch the latest relative to a head to get that storyline's newest checkpoint; move the head to an earlier checkpoint to time-travel.

3. What are pending writes?

An interrupt (say, for human approval) leaves work that is not yet a full checkpoint. Saved as pending writes and loaded as a pair with the checkpoint, they let the agent continue without redoing the step.

4. Why do many systems store the checkpoint reference in a database but the state itself in blob storage?

Traversal needs only the light references, which suit a fast indexed database; the heavy state is fetched whole from a blob. Bonus: no document-size ceiling, and a blob is not a queryable database, so there is nowhere for an injection attack to land.

Key takeaways

  • A conversation's history is a graph, not a line: each checkpoint stores its id, its state, and a parent pointer - the same model as Git.
  • Attaching a new checkpoint to the newest gives a straight history; attaching it to an older one forks a branch, non-destructively, leaving the original intact.
  • A head is a pointer to a branch's tip; you pass it to get the latest state or history for that storyline, and time-travel is just moving the head to an earlier checkpoint.
  • Pending writes capture partial work from an interrupt and travel with the checkpoint, so an agent resumes exactly where it paused instead of redoing the step.
  • State is scoped by a partition key - user, conversation, and agent ids - which isolates it and makes lookups fast.
  • Store the light reference (ids, parent, pending writes) in a fast database and the heavy state in a blob: it dodges object-size limits and removes the injection surface.
