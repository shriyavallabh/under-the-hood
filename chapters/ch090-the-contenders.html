The contenders

Six serious agent frameworks, one fair hearing each — what it actually is, where it shines, where it bites — and a method for choosing one without joining a religion.

~35 minChapter 91 of 117
The story

You are fitting out a restaurant kitchen, and six respected contractors come to bid. The first insists on full blueprints before a single pipe is laid — if the site loses power mid-job, his crew resumes from the exact marked step. The second is the in-house crew of a single big supplier: astonishingly fast, provided you buy that supplier's fittings, warranty, and finance plan. The third is the very crew that built the most famous kitchen in the city — they arrive with their own van, their own tools, and their own strict site rules. The fourth is a giant conglomerate with compliance folders, insurance certificates, and a liaison for your facilities department. The fifth is a staffing agency that assembles a named team — a head builder, a plumber, an electrician, each with a role card. The sixth is a lone master craftsman with a small toolbag who improvises brilliantly.

Every one of them can build you a kitchen. Asking "which contractor is best?" is meaningless. Asking "which is best for this site, this budget, this team?" — that has an answer. This chapter introduces the six, fairly, and teaches you how to run the bid.

Six names, one atom

First, a deep breath. Back in the ReAct chapter you built the atom of every agent yourself: a model, a set of tools, and a loop that alternates thinking and acting until the task is done. In the harness chapter you saw what production wraps around that atom — context engineering, memory, skills, durability, sandboxing, guardrails, observability, and, in the last chapter, watchdogs. Every contender in this chapter contains that same fifty-line heartbeat. Not one of them is magic.

What they differ in is everything around the loop: how much production armour ships in the box, how the control flow is expressed, which model provider the framework quietly pulls you toward, and whose problems it was designed to solve. So before the profiles, load the vocabulary you will use to compare them. Flip each card:

One rule for what follows: every framework here demos beautifully in twenty lines. Judge them on week three, not day one — debugging at 2 a.m., upgrade churn, the workflow that must pause four days for a signature. Each profile therefore ends with its honest weakness, because a comparison with no weaknesses is an advertisement.

LangGraph — the blueprint firm

What it is. LangGraph, from the LangChain team, is durable graph orchestration. Instead of one implicit loop, you draw the control flow explicitly — remember why graphs? — as nodes (steps) and edges (routes) over a shared, typed state. The shape you already know from the StateGraph chapter:

the LangGraph shape
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver

builder = StateGraph(TriageState)          # TriageState: your typed state
builder.add_node("classify", classify)
builder.add_node("resolve", resolve)
builder.add_edge(START, "classify")
builder.add_conditional_edges("classify", route_next)
graph = builder.compile(checkpointer=InMemorySaver())

That checkpointer argument is the firm's whole philosophy: every step is saved, so a run can crash, pause on an interrupt() for human approval, and resume days later with Command(resume=...) — exactly the checkpointing machinery you have already studied. Swap InMemorySaver for a Postgres or Redis checkpointer and durability becomes a database row, not a hope.

Sweet spot. Long-running, auditable, resumable workflows where you need to own the control flow: loan processing with approval gates, multi-day research pipelines, anything a regulator might ask you to replay. It is model-agnostic, and the surrounding tooling (LangSmith tracing, LangGraph Studio, a managed deployment platform) is mature.

Honest weakness. Ceremony. For a simple tool-calling assistant, defining state schemas, nodes, and edges is scaffolding you did not need — which is why LangChain's own create_agent exists as the quick path on top. The ecosystem has also churned across versions, so tutorials age fast. You will run a full scorecard on it in chapter 94.

OpenAI Agents SDK and AgentKit — the supplier's fast-track crew

What it is. The OpenAI Agents SDK is a deliberately light Python (and TypeScript) library — the production successor to OpenAI's earlier Swarm experiment. Its atoms are Agent (instructions + tools), Runner (the loop), guardrails — validation checks that run alongside the agent — and its signature idea, the handoff: an agent transferring the whole conversation to a specialist.

the Agents SDK shape
from agents import Agent, Runner

triage = Agent(
    name="Triage",
    instructions="Route the customer to the right specialist.",
    handoffs=[billing_agent, refunds_agent],
)
result = Runner.run_sync(triage, "I was charged twice this month.")
print(result.final_output)

Around the SDK sits AgentKit, OpenAI's platform layer: Agent Builder (a visual drag-and-drop canvas for agent workflows), ChatKit (an embeddable chat UI), a connector registry, and built-in evals. Together they are the fastest route from idea to deployed agent if you live in the OpenAI ecosystem — tracing is on by default and shows up in the OpenAI dashboard with zero setup.

Sweet spot. Teams already committed to OpenAI models who want multi-agent products shipped this quarter. The handoff pattern maps beautifully onto customer-service-style routing, and the SDK stays small enough to read in an afternoon.

Honest weakness. Model gravity. Other providers are usable (via its LiteLLM integration), but the polish, the tracing dashboard, and all of AgentKit point homeward to OpenAI's platform. Durability is also thinner than LangGraph's — long-running resumable workflows typically mean bolting on an external engine rather than a built-in checkpointer.

Claude Agent SDK — the famous kitchen's crew, for hire

What it is. The engine inside Claude Code — the best-known agent harness in the world, dissected fully in the next chapter — packaged as a Python and TypeScript library. It was renamed from "Claude Code SDK" in late 2025 precisely because people were building far more than coding agents with it. Where the other contenders give you parts, this one gives you the whole assembled machine: the tool loop, file and shell tools, the permission system, hooks, subagents, MCP client support (chapter 75), skills, and automatic context compaction.

the Claude Agent SDK shape
import asyncio
from claude_agent_sdk import query, ClaudeAgentOptions

async def main():
    options = ClaudeAgentOptions(
        allowed_tools=["Read", "Grep", "Bash"],
        permission_mode="acceptEdits",
    )
    async for message in query(prompt="Find and fix the failing test.",
                               options=options):
        print(message)

asyncio.run(main())

Notice what you did not write: no loop, no tool registry, no permission checks. They ship in the box, pre-hardened by millions of Claude Code sessions. You met this build-vs-buy tension before in deep agents versus Claude Code.

Sweet spot. Agents that live on a real computer — reading files, running commands, working a repository or an operations runbook. If your job is "a production-grade autonomous worker by Friday, with permissions and guardrails I do not have to invent", this is the shortest honest path.

Honest weakness. The strongest model gravity on the list — it is Anthropic's harness for Anthropic's models — and the most opinionated. You inherit its worldview: its tools, its permission model, its context management. When those opinions fit, they are a gift; when they do not, you are renovating someone else's famous kitchen.

Microsoft Agent Framework — the conglomerate

What it is. Microsoft's open-source framework for agents and multi-agent workflows in both Python and .NET, announced in late 2025 as the convergence of two older projects: AutoGen (the research-grade multi-agent library) and Semantic Kernel (the enterprise SDK). One thing to fix in your mental map right now: if a tutorial teaches AutoGen as the current thing, it is stale — AutoGen's orchestration ideas and Semantic Kernel's enterprise plumbing now live on inside Agent Framework, which is where Microsoft's investment goes.

the Agent Framework shape (Python)
from agent_framework import Agent
from agent_framework.openai import OpenAIChatClient

agent = Agent(
    client=OpenAIChatClient(model="gpt-4o"),
    name="expense-checker",
    instructions="Check expense claims against policy.",
    tools=check_policy,          # a plain Python function
)
# inside async code (ch39: async/await)
response = await agent.run("Is a 4,000 rupee team dinner claimable?")
print(response.text)

Two layers matter: Agent (formerly ChatAgent) for single agents, and typed, graph-based workflows (with checkpointing support) for orchestrating several — a philosophy closer to LangGraph than to CrewAI. It speaks the open standards you know: MCP for tools, A2A for agent-to-agent calls, OpenTelemetry for tracing, and it plugs natively into Azure AI Foundry for hosted deployment.

Sweet spot. Enterprise estates — especially .NET shops and Azure-centric organisations — that need identity, telemetry, and compliance to be first-class citizens rather than afterthoughts. It is the only contender treating .NET as an equal partner to Python.

Honest weakness. Youth and gravity. It is the newest major entrant, so patterns and docs are still settling, and Microsoft's naming history (Semantic Kernel, AutoGen, now this) makes teams reasonably wary of churn. And while it runs anywhere, the whole experience quietly points toward Azure.

CrewAI — the staffing agency

What it is. A standalone Python framework (independent of LangChain) whose core metaphor is an org chart: you declare agents with a role, a goal, and a backstory, give them tasks, and assemble them into a Crew that collaborates on the work.

the CrewAI shape
from crewai import Agent, Task, Crew

researcher = Agent(
    role="Market researcher",
    goal="Find out what rival cafes charge",
    backstory="A meticulous, sceptical analyst.",
)
report = Task(
    description="Compare pricing across rival cafe menus.",
    expected_output="A one-page summary with a table.",
    agent=researcher,
)
crew = Crew(agents=[researcher], tasks=[report])
print(crew.kickoff())

Read that aloud and it almost is not code — which is exactly the appeal. For deterministic control, CrewAI added Flows: event-driven, step-by-step orchestration with persistable state, its answer to the criticism that autonomous crews are hard to steer.

Sweet spot. Fast, readable multi-agent prototypes — content pipelines, research-and-summarise chains, back-office assistants — built by teams who think in roles and responsibilities rather than graphs. The community is large and the gallery of examples enormous.

Honest weakness. The abstraction leaks under pressure. Role and backstory strings become hidden prompt engineering, and when you need precise control over exactly what enters the context on turn four (the token bill taught you why you will), the friendly metaphor can start fighting you. Demos scale to production less smoothly than they demo.

smolagents — the master craftsman

What it is. Hugging Face's minimal agent library — the core agent logic is around a thousand lines you can genuinely read in one sitting. Its distinctive idea is the CodeAgent: instead of emitting JSON tool calls, the model writes its actions as short Python snippets, which are executed in a sandbox. One snippet can call three tools, loop, and combine results — often fewer round-trips than classic tool calling (a plain ToolCallingAgent is there when you want the classic style).

the smolagents shape
from smolagents import CodeAgent, InferenceClientModel, WebSearchTool

agent = CodeAgent(
    tools=[WebSearchTool()],
    model=InferenceClientModel(),   # any provider or open model
)
agent.run("Which of these two laptops is lighter, and by how much?")

Sweet spot. Research, experimentation, and teams committed to open models. It is the most model-agnostic contender by design — Hugging Face hub models, local models, or any API — and the most hackable: when you need to bend the agent's behaviour, you read the source and change it, because you can.

Honest weakness. Minimalism is the product. Durable state, enterprise observability, permissions, guardrails — largely yours to build. And letting a model execute the code it just wrote makes sandboxing non-negotiable, not optional: the library integrates Docker, E2B, and similar isolated executors, and the sandboxing chapter is your briefing on why you must use them.

In the wild

You can identify a codebase's contender from one file. Open pyproject.toml or requirements.txt and look for the telltale line: langgraph, openai-agents, claude-agent-sdk, agent-framework, crewai, or smolagents. In meetings, the tells are verbal: "we checkpoint every node" means LangGraph; "the triage agent hands off to billing" means OpenAI's SDK; "we allowed the Bash tool but gated edits" means Claude Agent SDK; "it plugs into our Azure telemetry" means Agent Framework; "the researcher agent passes its task to the writer agent" means CrewAI; "the model writes Python instead of JSON" means smolagents. And when a vendor slide says "we support all frameworks", your first scorecard question is: which one do your engineers actually use?

Check yourself

1. Your workflow must pause for a compliance officer's approval and resume three days later, surviving a server restart in between. Which contender treats exactly this as its home turf?

Durable execution is LangGraph's founding philosophy: every step is checkpointed to a store, interrupt() pauses for the human, and Command(resume=...) picks up from the saved state — restarts included. Microsoft Agent Framework's workflows are the nearest rival on this ground.

2. A tutorial from 2024 teaches AutoGen as Microsoft's current agent framework. What is the accurate 2026 picture?

Microsoft Agent Framework, announced in late 2025, is the successor that merges AutoGen's multi-agent orchestration ideas with Semantic Kernel's enterprise plumbing. Teaching AutoGen as current is the stale-tutorial trap this chapter warned you about.

3. What actually makes smolagents' CodeAgent different from classic tool calling?

Instead of emitting one JSON tool call per turn, the model writes a short program — call two tools, loop, combine — often finishing in fewer round-trips. The price is that executing model-written code makes sandboxing mandatory.

4. Which is the most accurate one-line description of the Claude Agent SDK?

It is batteries-included by design: you get the assembled, production-hardened machine that powers Claude Code, not neutral plumbing. That is its greatest strength (day-one armour) and its honest weakness (you inherit its opinions and its Claude gravity).

5. Your team asks "which framework is best?" What does this chapter's method say to do first?

Constraints are facts that filter the field before any opinion enters. Most six-way debates collapse to two survivors once the model contract, cloud mandate, and team language are on the table — then you weight, score, and prototype only the finalists.

Key takeaways

  • All six contenders wrap the same ReAct atom you built by hand; they differ in what surrounds the loop — armour, control flow, and model gravity — not in kind.
  • LangGraph owns durable, explicit, resumable workflows; the OpenAI Agents SDK plus AgentKit is the fastest path inside the OpenAI ecosystem; the Claude Agent SDK ships Claude Code's production harness as a library.
  • Microsoft Agent Framework is the AutoGen-plus-Semantic-Kernel successor built for enterprise estates; CrewAI makes multi-agent prototypes read like an org chart; smolagents is the tiny, hackable, open-model-first craftsman whose CodeAgent writes its actions as sandboxed Python.
  • Every framework has model gravity and an honest weakness; a comparison that lists no weaknesses is an advertisement, and demo-day impressions predict nothing about week three.
  • Shortlist by method, not fashion: hard constraints eliminate, workload shape matches, a weighted scorecard ranks, and identical one-week prototypes of the two finalists decide — in writing.
  • The names on this table will churn; the scorecard method survives the churn. Memorise the method.
