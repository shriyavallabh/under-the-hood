Anatomy of a sandbox service

You know the code needs a hardware-isolated box. Now build the service that manages those boxes - sessions, file projection in and out, guardrails, and why the execute tool is a plain agent tool, not an MCP call.

~35 minChapter 130 of 135
The story

A chemistry lab does not let researchers work at their desks. It has a fume hood: you carry in exactly the samples you need, do the reaction behind the glass with the extractor running, and carry the results back out. The hood is booked as a session, used, and cleaned. Nobody mounts their entire office into the hood; they bring the three vials that matter and take back whatever they produced. And when they leave, the hood is reset for the next person.

That is a sandbox service. Last chapter gave you the fume hood itself - the hardware-isolated box. But a lab is more than one hood; it is the booking desk, the trolley that carries samples in and results out, the timer that clears an idle station, and the rule that you only bring what you need. This chapter builds that surrounding machinery: the small service the agent talks to whenever it wants to run code, and the careful choreography of getting files in, running safely, and getting results back without ever handing the stranger the keys to the whole building.

What the service exposes

A sandbox service is a small control plane over disposable isolated compute. The agent's harness talks to it through a handful of endpoints - a booking desk for fume hoods. The core API is short:

Put together, the endpoints are: create a session (spawns an isolated pod), project files in, execute a command (bash or Python), snapshot / project files out, and delete the session. Five moves, and the agent can safely run whatever it writes.

Projection: copy in, copy out

The subtle, important choice is projection. Rather than mounting your whole workspace into the sandbox - which would give hostile code a path back to everything - the service copies a defined set of files in, and copies results back out. Because it is a copy, the sandbox physically cannot reach beyond what you handed it. And not everything comes back: files live in namespaces, and the temporary scratch namespace is deliberately left behind. Follow the round-trip:

1. Create a session

harnesscreate session
→
sandbox podfresh, isolated
workspace

The harness asks the service for a session; the service spawns a hardware-isolated pod and hands back a handle. The box is empty and walled.

2. Project the workspace in

harness
→
sandbox poddata.csv, notes.txt copied in
result

Only the conversation's files are copied into the pod. Not your whole disk - the specific workspace for this run. The sandbox now has exactly what it needs and nothing else.

3. Execute the code

harness
→
sandbox podruns; writes result.txt + scratch/tmp
result

The agent's command runs behind the glass. It reads the inputs, writes a result, and drops a throwaway file in scratch. No network, only the projected files.

4. Snapshot back, drop scratch, destroy

sandbox pod
→
harnessresult.txt returned
→
session deleted

New and modified workspace files are snapshotted back to the harness; the scratch namespace is dropped; the session is destroyed. The harness now holds the results, and nothing of the sandbox remains.

The execute tool is a plain agent tool - not MCP

Here is a design point that surprises people. You met tool calling, and later the idea of MCP as the way agents reach external capabilities. So you might expect "run code in a sandbox" to be an MCP tool. It usually is not. The execute tool is a plain structured tool implemented directly in the harness's own business logic - it sits alongside read, write, and list - and its body does the whole dance: connect to the sandbox service, project files in, run the command, rehydrate the results. It does not go through MCP, because sandboxed execution is core harness plumbing, not a pluggable external service. Read its shape:

execute_tool.py - a structured tool in the harness
def execute(command: str, language: str = "python") -> str:
A normal typed tool the model can call, just like a calculator. Its signature is simple; all the machinery hides in the body. To the model it is one more tool on the menu.
session = sandbox.create_session()
Book a fume hood: the service spawns an isolated pod and returns a handle. This is business logic in the harness, not a call out over MCP.
sandbox.project_in(session, workspace_files())
Copy the conversation's workspace into the pod. The tool decides what to send - the model never gets to point the sandbox at arbitrary paths.
result = sandbox.run(session, command, language)
Run the command behind the glass and capture stdout and errors. No network reaches out; only the projected files are visible.
save(sandbox.project_out(session))
Snapshot new and modified workspace files back into the conversation. Scratch is dropped by the service. The harness now owns the results.
sandbox.delete_session(session)
Tear the sandbox down. Ephemeral by default - the next execution starts from a clean, isolated box.
return result.stdout
Hand the output back to the model as the tool's observation, closing the loop. From the model's side it simply called execute(...) and got text.

Lifecycle, guardrails, and the sharp edges

Sessions are ephemeral: the service sets a max lifetime and/or idle timeout, and the orchestrator (Kubernetes, or a microVM manager) reclaims the pod. Sandbox pods live in a separate namespace from the app - recall from day-two on the cluster that a namespace is a walled section of the cluster - so a burst of sandbox sessions never crowds the agent itself. The guardrails from last chapter still apply: no outbound network, strict resource limits, only the projected files. Two sharp edges are worth knowing. First, authorization: a session must be scoped to a user and run - an unscoped session id that anyone could reuse is a real security hole to close before multi-tenant use. Second, human-in-the-loop: when a tool call is paused for approval, the harness typically kills the session and recreates it on resume, so nothing lingers half-run. And a practical quirk - execution policies can force patterns like "write a temp file, then run it," turning one logical action into two tool calls.

Note

One more decision the platform owns: what is in the sandbox image. A fully curated image ships only the Python standard library - predictable and locked down. A tenant-extensible one lets teams add packages like pandas. It is a genuine trade-off between safety and usefulness, and, like most things in this act, it is a choice you make deliberately, not a default you inherit.

In the wild

You will meet a "sandbox" or "code-exec" microservice with a small OpenAPI page (create-session, execute, delete), ephemeral pods appearing and vanishing in a sandbox-sessions namespace, and an execute tool sitting right next to read_file and write_file in an agent's toolset. In a review you will hear "the session isn't scoped to the user yet" (the authorization hole) and "it needed two calls because the policy blocks direct script exec" (the quirk). When a platform team at a large bank builds this, the debate is almost always projection rules and image contents - exactly the choices above.

Check yourself

1. Why does a sandbox service copy (project) files in rather than mounting the whole workspace?

Projection is a security choice: copy in exactly the needed files, so hostile code has no route to the rest of your system, and copy results back out explicitly.

2. What happens to files the agent writes into the scratch namespace?

Scratch is for throwaway files; only new and modified workspace files are projected back. Scratch evaporates with the ephemeral sandbox.

3. Is the execute tool typically an MCP call?

MCP is for pluggable external capabilities. Sandboxed execution is core harness machinery, so the execute tool sits alongside read/write/list and implements the projection dance itself.

4. What is the authorization risk to close before a sandbox service is multi-tenant?

If a session is not tied to a user and run, a leaked or guessed session id could let one caller reach another's sandbox. Scoping sessions is the fix before opening it to many tenants.

Key takeaways

  • A sandbox service is a small control plane over disposable isolated compute: create a session, project files in, execute, project results out, delete.
  • Projection copies (never mounts) a chosen set of files in and copies new/modified workspace files back - so the sandbox only ever sees what you handed it.
  • Files live in namespaces: the workspace round-trips, but the temporary scratch namespace is deliberately dropped and never returned.
  • The execute tool is a plain structured tool in the harness's own logic - not an MCP call - because sandboxed execution is core plumbing, not a pluggable external service.
  • Sessions are ephemeral (max lifetime / idle timeout) and run in a separate cluster namespace, with the same no-network, resource-limited guardrails.
  • Close the sharp edges: scope sessions to a user and run (authorization), kill and recreate on human-in-the-loop pauses, and decide curated vs tenant-extensible images.
