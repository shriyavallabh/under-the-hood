Build an MCP server

Two decorators, four type hints, one docstring — and any AI host on Earth can call your code. This is the shortest production-grade integration you will ever write, and you already learned every ingredient chapters ago.

~30 minChapter 78 of 133
The story

Meena makes the best lemon pickle in her neighbourhood, and three delivery apps want it. Each app demands its own onboarding: one wants forms in triplicate, another a phone interview, the third insists on its own packaging labels. Three integrations for one pickle. She gives up. Then the market association introduces the standard kitchen window: one spec for every home kitchen in the city. Mount the standard hatch, clip in a menu card that lists each dish — its name, what a rider must specify when ordering (jar size, spice level), and a one-line description — and any app that speaks "window" can order from her. She never meets a single app engineer. Best of all, the hatch ships as a window-in-a-box: she screws it to the frame, clips her existing recipe cards into the menu slot, and she is live before lunch.

In the previous chapter you toured these windows from the street side — what tools, resources, and prompts look like to the apps knocking on them. Today you cross to Meena's side of the hatch and build one. Your Python functions are the recipes. FastMCP is the window-in-a-box. And the menu card, delightfully, writes itself — from labels you already learned to put on your functions back in the type-hints and decorators chapters.

You already know both tricks

Quick recap, because this chapter is really a reunion. In the decorators chapter you learned that a decorator is a function that takes your function, does something useful with it — wraps it, registers it, times it — and hands it back, and that the @name line above a def is just tidy syntax for that handoff. You also met the decorator factory: @app.route("/home") style, where the trailing brackets mean "call this first, get the real decorator back". In the type-hints chapter you learned that annotations like amount: float are machine-readable labels — Python does not enforce them, but libraries such as Pydantic read them at runtime to build validators and schemas. Two chapters, one punchline each: decorators register, hints describe.

FastMCP is what happens when someone combines those two punchlines with a wire protocol. You write an ordinary function with type hints and a docstring. You put @mcp.tool() above it. The decorator registers the function in a catalogue, reads the hints and the docstring, and generates the JSON schema that any MCP host — the application driving the model, such as Claude Code or a desktop chat app — will use as the tool's menu entry. There is no separate integration file. Your function's own labels are the integration.

One more recall before we build: from tool calling and the tools chapter, remember that a model never executes anything — it emits a request, and code outside the model runs the real function. MCP does not change that physics. It just standardises the plumbing between the host that holds the model and the process that holds your functions, so the two can be written by strangers.

The whole server — nineteen lines

Here is a complete, real MCP server: a little finance desk with two tools, GST and loan EMI. Not a sketch, not pseudocode — save this as gst_server.py on a machine with pip install "mcp[cli]" done (that installs the official MCP Python SDK plus its command-line helpers), and every claim in the rest of this chapter works on it. Click the annotated lines; make each one boring to you.

gst_server.py — the entire server
from mcp.server.fastmcp import FastMCP
The official MCP Python SDK. FastMCP is its high-level, decorator-driven front door — there is a lower-level API underneath for exotic cases, and you will probably never need it.
mcp = FastMCP("finance-desk")
One object is your whole server. The string is the server's name — it is what hosts display in their tool menus and logs, so pick something a teammate would recognise, not test123.
@mcp.tool()
The decorator factory from ch17 — note the brackets: calling it returns the real decorator. It registers the function in the server's catalogue and generates its schema from the hints and docstring below. The function comes back unchanged, so your tests can still call gst(1500) directly.
def gst(amount: float, rate: float = 0.18) -> float:
The hints from ch16 do double duty here: they document the function for humans AND become the JSON schema for machines. rate has a default, so the schema marks it optional; amount has none, so it lands in the schema's required list.
"""Calculate the GST due on an invoice amount.
The docstring becomes the tool's description — the sentence the model reads when deciding whether to call this tool. Write it for that reader: say what the tool does and when to use it. A lazy docstring here is a tool the model will misuse or ignore.
rate is a fraction: 0.18 means 18 percent."""
Spell out unit conventions. "Is rate 18 or 0.18?" is exactly the kind of ambiguity that makes a model pass garbage arguments.
return round(amount * rate, 2)
A plain return. FastMCP serialises whatever you return — strings, numbers, dicts, lists — into protocol content and ships it back to the host, where it re-enters the model's context as a tool result.
@mcp.tool()
Growing the server costs exactly one decorated function per tool. No registry file to edit, no schema to hand-write, no config to touch.
def emi(principal: float, annual_rate: float, months: int) -> float:
"""Monthly EMI for a loan (reducing-balance method)."""
r = annual_rate / 12
The body is ordinary Python — nothing MCP-flavoured about it. Whatever your team's real logic is (query a database, call an internal API, read a spreadsheet), it goes here unchanged.
return round(principal * r * (1 + r) ** months
/ ((1 + r) ** months - 1), 2)
if __name__ == "__main__":
The main guard from ch11: this block runs only when the file is executed as a script — which is precisely how a host will launch it — and not when it is imported by your tests.
mcp.run()
Starts serving and blocks forever, waiting for a host to talk to it. With no arguments it uses the stdio transport — JSON-RPC messages over stdin and stdout. For a networked server you would write mcp.run(transport="streamable-http") instead; same tools, different wire.

That is the whole thing. Count the lines that are actually about MCP rather than about GST: the import, the FastMCP(...) line, two decorators, and mcp.run(). Five lines of protocol, fourteen lines of your business logic. That ratio is the entire sales pitch.

Where the menu card comes from

When a host asks this server "what have you got?", the server answers with a schema per tool. You never wrote it; the decorator derived it. Here is what gst's entry looks like on the wire (trimmed for readability):

what the host receives for gst — generated, not written
{
  "name": "gst",
  "description": "Calculate the GST due on an invoice amount.\nrate is a fraction: 0.18 means 18 percent.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "amount": { "type": "number" },
      "rate":   { "type": "number", "default": 0.18 }
    },
    "required": ["amount"]
  }
}

Trace every field back to the source: name is the function name, description is the docstring, each property's type comes from a hint (Python's float maps to JSON Schema's number), and required holds exactly the parameters without defaults. If this shape looks familiar, it should — it is the same schema-for-tools idea you met in tool calling and the same describe-data-with-JSON-Schema machinery from structured output. MCP did not invent a new wheel; it standardised who sends the wheel to whom.

Feels like magic? It is about thirty lines of inspect. Prove it to yourself: below is a toy FastMCP built from nothing but the standard library — a class with a decorator factory that reads annotations and docstrings, exactly like the real one. Press Run, then play the host.

What mcp.run() actually starts

The default transport, stdio, means the host launches your script as a subprocess and speaks JSON-RPC 2.0 — a simple request/response format in JSON — over the pipes every process already owns: standard input and standard output. No port, no URL, no firewall ticket. Your server is not "running somewhere"; it is started fresh by the host, lives for the session, and dies with it. Step through what happens the moment a host boots your file:

Careful

The classic way to kill a stdio server is print("got here"). Stdout is not a debug console any more — it is the protocol wire, and one stray line of non-JSON corrupts the stream, usually surfacing as the host reporting the server "failed to connect" or "returned an invalid response". Debug to stderr instead, ideally with the logging module from ch19 — hosts typically collect a stdio server's stderr into their own logs, which is exactly where you want your breadcrumbs.

Plug it into a real host: Claude Code

A server nobody has registered is a kitchen window on a street no rider walks down. Registration is host configuration, not code. In Claude Code it is one command from your project directory:

terminal — register the server, then verify
claude mcp add finance-desk -- python /absolute/path/to/gst_server.py
claude mcp list

Everything after the -- is the launch command — the exact subprocess the host will spawn in step 1 of the stepper above. Claude Code stores this at one of three scopes: local (default — just you, just this project), user (you, everywhere), or project, which writes a .mcp.json file at the repo root so the whole team inherits the server when they pull. That file is worth recognising on sight, because you will meet it in real repositories constantly:

.mcp.json — project-scope registration, committed to the repo
{
  "mcpServers": {
    "finance-desk": {
      "command": "python",
      "args": ["/absolute/path/to/gst_server.py"]
    }
  }
}

Inside a Claude Code session, the /mcp command shows each configured server's connection status and its tools. Other hosts differ only in where this block lives — the desktop Claude app has a settings JSON with the same mcpServers shape — and that is rather the point of a standard: your nineteen lines did not change between hosts, only one config entry did.

Note

Use absolute paths and the right Python. The host spawns your server with the command you gave it, from its own working directory, with no idea that your project has a virtual environment (ch20). "Works when I run it, fails when the host does" is almost always a relative path or a system Python that has never heard of the mcp package — point command at your venv's interpreter and the mystery evaporates.

Test it before any model touches it

You would not hand a new tool to an intern without trying it yourself first; do not hand one to a model that way either. The MCP Inspector is the official test bench — a small local web app that connects to your server, lists its tools, and lets you call them by hand, with no LLM anywhere in the loop. The SDK's CLI launches it in one line:

terminal — open the test bench
mcp dev gst_server.py

That command boots your server, starts the Inspector, and prints a localhost URL. Open it and the session looks like this:

One habit worth stealing from experienced MCP authors: Inspector first, host second, model last. Most "the agent can't use my server" reports collapse at step one (server crashes on launch — check stderr), a few at step two (tool missing or badly described), and only the rare remainder are genuinely about the model.

Beyond tools: the same trick, twice more

Tools are the star, but the previous chapter's other two primitives are built the same way in FastMCP: @mcp.resource(...) exposes read-only data at a URI, and @mcp.prompt() exposes a reusable prompt template — each just a decorated, hinted, docstringed function. Newer spec additions such as tasks and elicitation extend the protocol further; hosts are still catching up to those, so treat them as ones to watch rather than ones to build on today. The mental model does not change: functions in, standard window out.

In the wild

You will meet this pattern everywhere now that you can see it. A .mcp.json at a repo root means "this project ships team-wide MCP servers" — open it and read the launch commands. A file named server.py containing FastMCP( and a handful of @mcp.tool() functions is the standard shape of "we wrapped our internal API for the agents" at any large company — a bank wrapping its ticketing system, a retailer wrapping inventory lookups. On PyPI and npm, packages named mcp-server-something are exactly this chapter published for strangers. And when a colleague ends a debate with "just expose it as an MCP server", you now know they are proposing roughly nineteen lines, and you could have a draft up before the meeting ends.

Check yourself

1. You put @mcp.tool() above a typed, docstringed function. Where does the JSON schema that hosts receive come from?

FastMCP inspects the function at registration time: hints become property types, defaults decide what is required, the docstring becomes the description. Your labels are the integration — which is why sloppy hints and lazy docstrings turn directly into bad tool calls.

2. Your stdio server worked yesterday. Today, after you added print("got here") for debugging, the host says it failed to connect. Why?

In a stdio server, stdout carries the wire protocol. One line of non-JSON breaks the host's parser. Send debug output to stderr — for example with the logging module — where hosts collect it harmlessly.

3. What does claude mcp add finance-desk -- python gst_server.py actually arrange?

Registration is just recorded launch instructions. Your code never leaves your machine; the host runs it locally as a child process and the model reaches it only through the protocol — via the host.

4. What is the MCP Inspector for?

The Inspector is a local test bench: it speaks real MCP to your server and lets you fire tools/list and tools/call by hand. It splits "is my server broken?" from "is the model confused?" — two very different bugs.

5. What does mcp.run() with no arguments do?

No arguments means the stdio transport: the process sits waiting on stdin for a host's messages and answers on stdout, for as long as the session lives. The networked alternative is opt-in: mcp.run(transport="streamable-http").

Key takeaways

  • An MCP server is your ordinary Python functions plus FastMCP — about twenty lines buys a real server that any MCP host can discover and call.
  • @mcp.tool() is the decorator pattern from ch17 doing registration, and the type hints from ch16 doing description: hints become the schema's types, defaults decide what is required, and the docstring becomes the description the model reads.
  • mcp.run() defaults to stdio: the host launches your script as a subprocess and speaks JSON-RPC over stdin/stdout — so never print to stdout in a stdio server; log to stderr instead.
  • Registration is host configuration, not code: claude mcp add or a .mcp.json with an mcpServers block records the launch command; use absolute paths and the right interpreter.
  • Test with the MCP Inspector (mcp dev server.py) before any model is involved — list the tools, call them with real arguments, read the descriptions a host will see.
  • The same decorated-function trick extends to resources and prompts, so you now hold the builder's side of the previous chapter's entire trio.
