YAML and JSON: dicts in costume

Every config file you have ever squinted at — compose files, CI pipelines, API payloads — is the lists and dicts of the last chapter, written down as text. Once you see the costume, you can never unsee the dict.

~30 minChapter 5 of 117
The story

Walk into a busy restaurant and watch one order live two lives. At the till, a printer spits out a receipt: rigid columns, every item in quotes, every price in its slot, commas exactly where the accounting software demands them. One misplaced character and the machine refuses the whole thing. Meanwhile in the kitchen, the very same order goes up on a whiteboard: table 12 underlined at the top, the dishes indented beneath it, a dash in front of each one, "extra spicy" scribbled beside the biryani. No quotes, no commas — just neat indentation that any cook can read at a glance.

The receipt is for machines talking to machines. The whiteboard is for humans who need to write fast and read faster. But underneath, it is the same order — the same table number, the same list of dishes, the same little notes. That is this entire chapter. JSON is the till receipt. YAML is the kitchen whiteboard. And the order underneath both of them is exactly the dicts and lists you met last chapter.

You already know this — that was the point of chapter 4

A quick recap from the four containers, because this chapter stands entirely on it. Lists hold things in order; dicts label things by name; both nest freely inside each other; and nearly every structure an agent touches — configs, tool arguments, API responses — is dicts and lists nested a few levels deep. That was the punchline of chapter 4: agent plumbing is containers all the way down.

Here is the one new idea this chapter adds. Those containers live in a running program's memory, and memory vanishes the moment the program stops. For a structure to survive — saved in a file, sent over a network, committed to a git repo — it has to be written down as plain text. Turning live containers into text is called serialisation — writing data out in an agreed, machine-readable format. Reading that text back into live containers is called parsing — the reverse trip. JSON and YAML are simply the two most common agreements about what that text should look like.

Watch the same little structure wear all three costumes. First, as Python in memory — pure chapter 4:

In memory — a Python dict
order = {
    "table": 12,
    "dishes": ["dal tadka", "veg biryani"],
    "extra_spicy": True,
    "delivered_at": None,
}

Now the till receipt. Braces for dicts, square brackets for lists, double quotes on every string — and note the small wardrobe changes: True becomes lowercase true, and None becomes null:

order.json — the same thing in JSON
{
  "table": 12,
  "dishes": ["dal tadka", "veg biryani"],
  "extra_spicy": true,
  "delivered_at": null
}

And the kitchen whiteboard. The braces, brackets, and most of the quotes evaporate. Indentation says what belongs inside what; a dash starts each list item:

order.yaml — the same thing in YAML
table: 12
dishes:
  - dal tadka
  - veg biryani
extra_spicy: true
delivered_at: null

Three costumes, one body. If you can read the Python version, you can already read the other two — the rest of this chapter is just learning each costume's dress code and the places where it pinches.

From characters on disk to a living dict

Before the dress codes, fix the journey in your head, because it explains why any of this matters. A file on disk contains no dicts and no lists — only characters, one after another. Something has to read those characters and build the containers. That something is a parser — a piece of code that walks through text, obeying a format's rules, and constructs live data from it. JSON: the strict costume

JSON — JavaScript Object Notation, a text format for data that machines exchange — is the till receipt: rigid on purpose. Its whole value is that a JSON document written by a server in one language parses identically in every other language on Earth. The dress code is short and absolutely non-negotiable:

  • Dicts wear { }, lists wear [ ] — always.
  • Every string and every dict key wears double quotes. Single quotes are rejected.
  • The only value types are: string, number, true, false, null, object (dict), and array (list). Nothing else exists.
  • No comments. None. Not even a little one.
  • No trailing commas. {"a": 1,} is an error, full stop.

In Python, the translator is the built-in json module, and its two verbs are worth memorising now because you will see them in every codebase for the rest of this course: json.loads parses a string into containers (the s means string), and json.dumps serialises containers back into a string. YAML: the human costume

YAML — a text format designed for humans to write configuration in — keeps the same underlying shapes but drops the ceremony. Three words carry its grammar. A mapping is YAML's name for a dict: lines of key: value. A sequence is YAML's name for a list: lines starting with - . A scalar is any single value — a string, number, bool, or null — the leaves of the tree. And instead of braces, indentation is the syntax: whatever is indented under a key is inside it.

agent.yaml
# comments exist in YAML -- a luxury JSON never allows
agent:
  name: shift-summariser
  tools:
    - calculator
    - web_search
  limits:
    max_turns: 5
    budget_usd: 1.50

Read it with chapter-4 eyes: the whole file is one dict with a single key agent; its value is another dict; tools holds a list of two strings; limits holds a dict with an int and a float. In Python, after parsing: cfg["agent"]["tools"][1] is "web_search". Strings usually go unquoted — the parser guesses the type of each bare scalar, which is delightful right up until it guesses wrong (a famous ambush we will spring in a minute). Booleans are true/false, and null is written null, ~, or by leaving the value empty.

One more comforting fact: YAML is very nearly a superset of JSON. You can paste a JSON object into most YAML files and it parses fine — tools: ["calculator", "web_search"] is legal YAML, just in its inline "flow" style. The two costumes share a wardrobe.

Two conveniences JSON cannot offer

YAML earns its keep in config files with two features worth recognising on sight. The first is multiline strings. A | after the key starts a literal block that keeps every line break exactly as written — perfect for embedding a script or a prompt. A > starts a folded block that joins the lines into one long line with spaces:

multiline scalars
release_note: |
  Fixed the login bug.
  Upgraded the parser.
  No downtime expected.

summary: >
  These folded lines will be joined
  into one single line, with spaces
  where the breaks used to be.

The second is anchors. An anchor — written &name — labels a block of YAML so an alias, written *name, can reuse it later in the same file; the popular <<: merge key copies an anchored mapping in and lets you override individual keys. It is how CI files avoid repeating the same settings for ten jobs:

anchors and aliases — write once, reuse
defaults: &defaults
  retries: 3
  timeout: 30

staging:
  <<: *defaults          # copy everything from defaults
  url: https://staging.example.com

production:
  <<: *defaults
  url: https://example.com
  retries: 5              # ...then override just this key

You do not need to write anchors this month. You do need to not panic when you meet & and * in a pipeline file — now you won't.

The gotchas that bite in production

YAML's friendliness has a price: because bare scalars are unquoted, the parser must guess what each one is — and its guesses are governed by rules written down years ago. Three ambushes account for almost every YAML war story you will ever hear.

Tabs are forbidden. YAML indentation must be spaces — a tab character anywhere in the indentation is an immediate parse error. This is the number-one "it worked on my machine" YAML failure: an editor silently inserts a tab, the file looks identical to the eye, and the parser explodes. Two spaces per level, and match whatever the file already uses.

The Norway problem. Under the older YAML 1.1 rules — which the most widely used Python parser, PyYAML, still follows — the bare words yes, no, on, and off (in their usual capitalisations) are booleans. So a perfectly innocent country list containing country: NO quietly delivers country: False to your program, and Norway vanishes from the map. The YAML 1.2 specification dropped those extra spellings, keeping only true and false, but 1.1-style parsers are still everywhere — so the professional habit is to quote anything that even smells like a boolean.

Overeager numbers. The version string 3.10, unquoted, is the float 3.1. A value with a leading zero like 010 is, under 1.1 rules, an octal number — it parses as 8. And pairs of small numbers separated by colons, like the port mapping 22:22, can be read as a base-60 ("sexagesimal") number — 1342. Careful

The universal defence is boring and beautiful: when in doubt, quote it. Quote country codes, version numbers, port mappings, anything starting with a special character, and any string containing : or # (an unquoted # starts a comment and silently eats the rest of the line). Double quotes also let you write escapes like \n; single quotes keep everything literal. Quoting is never wrong — guessing sometimes is.

Reading real specimens

Time to cash in. Below are two files shaped exactly like ones you have deployed against for years. First, a compose-style file — the kind that defines a small stack of services. Click the annotated lines and translate each one back to containers as you go:

docker-compose.yml — a two-service stack
services:
The top-level mapping's first key. Everything indented beneath it is its value — a dict whose keys will be the service names. In Python terms, you are looking at cfg["services"].
web:
A key inside services, so: cfg["services"]["web"]. The service's whole definition is the mapping indented under it. The name web is yours to choose — it is data, not a keyword.
image: nginx:1.27
A plain scalar containing a colon. Safe unquoted, because YAML only splits a key from a value at a colon followed by a space — nginx:1.27 has none, so it stays one string.
ports:
- "8080:80"
A sequence with one item — a quoted string. The quotes are the habit that saves you: port pairs where both numbers are under 60 (like 22:22) can be swallowed as base-60 numbers by 1.1-style parsers. Quote every port mapping, always, and the ambush can never fire.
depends_on:
- api
Another sequence — dash means list item. One item, the bare string api, naming the other service. So cfg["services"]["web"]["depends_on"] is ["api"].
api:
Back out to two spaces of indentation — so this is a sibling of web, the second key in the services dict. Indentation level IS the nesting level; your eye can walk the tree.
build: ./api
environment:
Note what is missing: dashes. The lines under this key have no - prefix, so this is a nested mapping, not a sequence — a dict of variable names to values.
LOG_LEVEL: info
DEBUG: "false"
Quoted on purpose. Environment variables must be strings, but a bare false would parse as a boolean — and some tools then crash or, worse, coerce it in surprising ways. This is the Norway problem's little brother, defused with two quote marks.
restart: unless-stopped
volumes:
Fully unindented — a second top-level key, a sibling of services.
pgdata:
A key with nothing after the colon: its value is null. In compose files this idiom means "create it with default settings". You will see the bare-key-meaning-defaults trick all over real-world YAML.

Second specimen: a CI pipeline file, in the style GitLab and similar systems use. Same grammar, different vocabulary:

.gitlab-ci.yml — a two-stage pipeline
stages:
- test
A sequence of stage names, and here order is meaning: the CI system runs the stages top to bottom. Reordering these two lines reorders your pipeline — a one-character diff with real consequences.
- deploy
default:
image: python:3.12
Settings under default apply to every job. And notice: python:3.12 is safely one string (colon, no space), but if you ever write a bare version like 3.10 on its own, quote it — or the float-guesser will trim it to 3.1.
run-tests:
Any top-level key that is not a reserved word is a job name. The whole file is one big dict, and the CI runner reads it exactly the way you would: cfg["run-tests"]["script"] and so on. No magic — dict access.
stage: test
script:
The heart of every job: a sequence of shell commands. Each dash is one list item is one command, run in order. When a pipeline log says "step 2 failed", it means item index 1 of this list.
- pip install -r requirements.txt
- pytest -q
rules:
- if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
Two costumes at once: a sequence of mappings (dash, then key: value on the same line), and a value wrapped in single quotes because it is full of characters — $, ==, inner double quotes — that beg to be quoted. The CI system, not YAML, gives the expression its meaning; YAML just delivers it as a string.
deploy-prod:
stage: deploy
script:
- ./deploy.sh
when: manual
A plain scalar that tells the CI system to wait for a human click before running this job. Every "manual gate" you have ever pressed in a pipeline UI traces back to one little line like this.

How code eats these files

The last piece is the two lines of Python that perform the trick. JSON support ships inside Python itself; YAML needs the most popular third-party parser, installed with pip install pyyaml and imported as yaml:

loading configs — the shape you will see everywhere
import json
import yaml                       # from: pip install pyyaml

with open("settings.json") as f:
    cfg = json.load(f)            # load reads a file; loads reads a string

with open("settings.yaml") as f:
    cfg = yaml.safe_load(f)       # ALWAYS safe_load, never plain load

Two notes to file away. The with open(...) line is a context manager — chapter 18 unpacks it; for now read it as "open the file, and tidy up automatically afterwards". And the word safe_load is not decoration: full YAML can encode instructions to build arbitrary objects, so the unsafe loader can be tricked into running code from a hostile file. safe_load builds only plain containers and scalars — which is all a config should ever be. If you remember one API name from this chapter, make it that one.

Note

Either way, what lands in cfg is the same thing: dicts, lists, strings, numbers, bools, and None. There is no "YAML object" or "JSON object" living in your program afterwards. The costume never makes it past the parser.

In the wild

You now hold the skeleton key to most files in a modern repo. Compose files, CI pipelines, Kubernetes manifests, and GitHub Actions workflows are YAML mappings and sequences. Every API response, every .mcp.json, every package.json, and every tool call an LLM makes travels as JSON — when an agent "calls a tool", the arguments arrive as a JSON object that json.loads turns into a dict before any code runs (you will build exactly that in the ReAct chapter). In merge requests, watch for the classic diffs: someone quoting a port mapping, quoting "NO", or fixing an indentation level — you can now review those in seconds. And when you meet pyproject.toml, that is TOML, a third costume for the same dicts; the wardrobe changes, the body never does.

Check yourself

1. You run json.loads('{"retries": 3, "hosts": ["a", "b"]}'). What do you get back?

Parsing ends at the door: the output is ordinary containers, accessed with ordinary square brackets. Nothing "JSON-ish" survives into your running program.

2. A config file contains country: NO and is read by a YAML 1.1-style parser such as PyYAML. What lands in Python?

Under YAML 1.1 rules, yes/no/on/off are booleans, so Norway's country code becomes False. YAML 1.2 dropped those spellings, but 1.1-style parsers remain everywhere — hence the habit: quote anything that smells like a boolean.

3. You indent one YAML line with a tab character and the parser rejects the whole file. Why?

Indentation is YAML's syntax, and the spec requires it to be spaces. A tab looks identical on screen, which is exactly why this is the number-one "but the file looks fine!" YAML failure.

4. In a multiline YAML value, what does the | block marker do?

The pipe means literal: line breaks survive verbatim, which is why scripts and prompts are embedded with it. The greater-than sign is the folding one that joins lines into a single line.

5. Compose files conventionally write port mappings quoted, as "8080:80". What is the quoting defending against?

Under YAML 1.1 rules, colon-separated numbers whose parts fit under 60 are sexagesimal integers — 22:22 becomes 1342 and your port mapping silently vanishes. Quoting every port mapping, whatever the numbers, means the ambush can never fire.

Key takeaways

  • JSON and YAML are not new kinds of data — they are the dicts, lists, strings, numbers, bools, and None of chapter 4, written down as text so they can live in files and cross networks.
  • JSON is the strict costume: braces, brackets, double quotes everywhere, no comments, no trailing commas. Its rigidity is the feature — it is the wire format of APIs and of every tool call an LLM makes.
  • YAML is the human costume: indentation for nesting, dashes for list items, key: value for mappings, real comments, multiline blocks, and anchors for reuse.
  • Parsing (json.loads, yaml.safe_load) turns text into live containers; serialising (json.dumps) goes back. After parsing, it is all ordinary dict access — the costume never enters your program.
  • YAML's type-guessing is the source of its gotchas: never indent with tabs, and quote anything that resembles a boolean (NO, off), a version (3.10), or a port pair (22:22). When in doubt, quote.
  • Use yaml.safe_load, never plain load — configs should only ever become plain containers, and the safe loader guarantees it.
