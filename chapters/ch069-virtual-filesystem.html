The virtual file system

Deep agents survive hours-long runs by writing findings to files and reading back only what they need — and a pluggable backend decides whether those "files" live in agent state, on real disk, in Postgres, or in all three at once.

~30 minChapter 70 of 133
The story

Detective Ambika runs the busiest desk in the station, and her corkboard holds exactly twenty index cards. Not twenty-one — that is the rule. Every case she works produces crates of material: bank statements, call logs, witness transcripts. None of it goes on the board. Instead, each crate gets a label and slides through the evidence-room counter, and a single index card goes up in its place: "Bank records — box /cases/mehta/bank". Weeks later, when the whole case turns on one withdrawal, she asks the counter for that one box, copies a single line onto a card, and sends the box straight back.

Here is the part she never thinks about. The clerk behind the counter reads labels, not contents. Crates labelled /memories/ go to the permanent vault across the courtyard — they survive transfers, retirements, even the station changing buildings. Crates labelled /workspace/ go onto the station's own shelves, where any officer can pull them. Everything else sits on a trolley that is emptied the day the case closes. Same counter, same labels, three different fates — and when two labels could both apply, the more specific one always wins.

Her corkboard is a context window. The counter is a virtual file system. The clerk's sorting rule is this chapter.

A desk, not a warehouse

First, place yourself in Act VIII. In the anatomy chapter you opened create_deep_agent and found the same ReAct loop you built by hand wearing four upgrades: a planning tool, a file system, subagents, and a serious system prompt. The previous chapter gave the agent a to-do list, so a long run cannot lose the plot. This chapter gives it a filing cabinet, so a long run cannot drown in its own research. The next one gives it colleagues.

Why does a long-running agent need files at all? Because of the sting you felt in chapter 46: the conversation history is re-sent to the model on every single turn. When a tool returns something huge — a scraped documentation site, a 300-row query result, three market-research reports — that text lands in the history and is then carried, and paid for, on every turn that follows. You read it once; you pay for it forever. Over a twenty-minute run that is annoying. Over the hours-long runs this Act is about, it is fatal: the context fills, the model starts forgetting the plan, and the token bill climbs like a taxi meter in traffic.

Chapter 48 named this the context problem. Deep agents answer it with a move called context offload — moving bulky information out of the message history into external storage that the agent can reach through tools. The discipline is exactly the detective's: write findings to a file, keep only a one-line stub in context, and read back only the slice you need, only when you need it. Watch one full cycle:

The trick: these files may never touch a disk

Now the part that surprises people. When a deep agent calls write_file("/notes/plan.md", ...), there is — by default — no file called plan.md anywhere on any disk. A virtual file system is an interface that looks and behaves like files and folders — paths, listings, reads, writes — while the actual bytes live wherever a storage layer chooses to put them. The agent sees four core tools (recent versions add glob and grep for searching):

  • ls — list what is on the shelf
  • read_file — bring a file's content into context
  • write_file — create or overwrite a file
  • edit_file — make a targeted change without rewriting the whole file

Behind those four tools sits a backend — the component that maps file operations onto an actual storage location. The tools are the evidence-room counter; the backend is whatever is behind the wall. This split is a pattern you already know from composition: program against an interface, swap the implementation freely. The agent's prompt, tools, and habits never change; where the bytes live is a one-line configuration decision. That indirection is the whole reason the word "virtual" is in the chapter title.

Four places the shelf can live

Now the second half of the promise: the backends. The deepagents library ships four core ones (plus sandbox and shell variants), and choosing between them is really answering one question — how long should these files outlive the conversation?

BackendWhere the bytes liveHow long they lastReach for it when
StateBackend (default)Inside the agent's graph state, right next to the messagesThe thread — persisted per-thread if a checkpointer is attachedScratch work, drafts, research notes for this run
FilesystemBackendReal files on real disk, under a root_dir you chooseUntil something deletes them; writes are permanentLocal dev tools and CI jobs that must touch an actual project
StoreBackendA LangGraph store — in-memory for dev, Postgres or Redis in productionAcross threads, sessions, and restartsDurable memory: preferences, accumulated knowledge
CompositeBackendWherever each route says — it delegates by path prefixMixed, folder by folderOne namespace, different guarantees per folder

StateBackend is what you get if you configure nothing. Files are stored inside the agent's state — the same state object that carries the message history through the graph. That has a consequence worth saying twice: state-backed files have exactly the lifetime of the thread. If you wired up a checkpointer in the checkpointing chapter, the files survive restarts for that thread, because they ride inside the checkpoints. But start a fresh conversation tomorrow — a new thread_id — and ls returns an empty shelf. For scratch work, that is not a bug; it is tidiness. The trolley gets emptied when the case closes.

FilesystemBackend swaps the trolley for your actual disk. Now write_file("/workspace/report.md", ...) creates a genuine file a human can open, a CI job can commit, another program can consume. This is the backend that turns a deep agent into a document-producing or code-writing colleague. It is also the only backend on this list that can hurt you, which earns it a callout of its own:

Careful

FilesystemBackend hands the model real read/write access to a machine's disk, and its safety switch is off by default. With virtual_mode=False (the default), setting a root_dir gives no sandboxing at all — paths containing .., ~, or absolute paths can walk straight out of the root and read whatever the process can read: .env files, SSH keys, other tenants' data. Set virtual_mode=True to jail every path under root_dir. Even jailed, remember two things: disk writes are permanent and no checkpointer will undo them, and anything the agent reads enters its context and can travel anywhere the agent can send text. The sensible rule: real disk is fine for a local CLI or a CI job working on one project, and a poor fit for a multi-tenant web service. For risky writes, you can also demand human sign-off with the interrupt machinery from the human-in-the-loop chapter.

StoreBackend points the file tools at a LangGraph store — the long-term key-value storage that lives outside any single thread. Pass an InMemoryStore while developing; in production you construct a Postgres- or Redis-backed store (shipped in the same langgraph-checkpoint-* family of packages you met for checkpointers), and the agent's files become rows in a real database: durable across restarts, shared across conversations, backed up by whatever backs up your database. This is the backend that makes an agent remember you next week — an idea Act X develops fully as long-term memory.

CompositeBackend is the clerk behind the counter. It holds a default backend plus a routing table mapping path prefixes to other backends, and forwards every file operation using longest-prefix routing — the path is compared against every route, and the longest matching prefix wins. The classic configuration gives the agent one seamless namespace with three different fates, exactly like the records room in the story:

  • /memories/... routes to a StoreBackend — survives forever, across sessions
  • /workspace/... routes to a FilesystemBackend — real deliverables on real disk
  • everything else falls through to the default StateBackend — scratch that vanishes with the thread

The agent does not know or care. It writes to paths; the router sorts out the physics.

Longest prefix wins — prove it to yourself

The routing rule deserves one careful minute, because it is the kind of thing that decides code reviews. Suppose the routes are /memories/ and /memories/archive/. A write to /memories/archive/2025.md matches both prefixes — so which backend gets it? The longest match: /memories/archive/. Specific beats general, exactly like postal sorting: a parcel addressed to a flat inside a building goes by the flat number, not just the building's name.

In the wild

Open any deep-agents codebase and find the create_deep_agent(...) call: the backend= argument is the memory architecture in ten lines — read the routes dict and you know what survives a restart before you have read anything else. System prompts that say "save preferences to /memories/user_preferences.txt" are the write half of that same design. You have also been operating this pattern all along: Claude Code writing notes to its memory files and reading them back at session start is offload-and-route with a real disk behind the counter. And in merge requests, comments like "move agent memory from state to the Postgres store" or "that root_dir is not jailed — set virtual_mode" are precisely the two decisions this chapter taught you to make.

Check yourself

1. A tool just returned a 40,000-token report. Why should the agent write it to a file rather than keep it in the conversation?

The whole history travels to the model on every turn, so a big result kept in context is paid for again and again. Offloading swaps that recurring cost for a one-line receipt plus an on-demand read-back.

2. Your agent uses the default StateBackend and writes /notes/plan.md today. Tomorrow the same user starts a brand-new conversation thread. What does ls show?

StateBackend stores files in the agent's graph state, so their lifetime is the thread's lifetime. A checkpointer preserves them per thread, but a new thread_id means new state — and an empty listing.

3. You want the agent to remember user preferences across sessions and server restarts. Which setup does that?

The canonical pattern: scratch stays thread-scoped in state, while the /memories/ prefix routes to a store that Postgres or Redis makes durable. Disk works on one machine, but a database-backed store survives redeploys and scales across servers.

4. Routes are {"/memories/": store, "/memories/archive/": disk}. The agent writes /memories/archive/2025.md. Where does it land?

CompositeBackend resolves overlaps by longest-prefix routing: the most specific route takes the file. Dict order and route count are irrelevant.

5. A teammate configures FilesystemBackend(root_dir="/srv/app/data") and calls it sandboxed. What is true?

Without virtual_mode=True the root is decorative: the agent can traverse out and read anything the process can read. And no checkpointer undoes a disk write — file modifications are permanent. This is the one-line review comment worth making every time.

Key takeaways

  • Files are the deep agent's answer to a finite context: write findings out, keep a one-line stub in the history, and read back only the slice you need, when you need it.
  • The four core file tools — ls, read_file, write_file, edit_file — are an interface; a pluggable backend decides where the bytes actually live. That indirection is what makes the file system "virtual".
  • StateBackend (the default) keeps files inside the agent's state: thread-scoped, checkpointer-persisted per thread, gone in a new conversation — perfect for scratch.
  • FilesystemBackend touches real disk: essential for real deliverables, dangerous by default. virtual_mode=True jails paths under root_dir; without it, root_dir provides no protection — and disk writes are always irreversible.
  • StoreBackend puts files in a LangGraph store — in-memory for development, Postgres or Redis in production — so they survive restarts and are shared across threads.
  • CompositeBackend routes each path by longest matching prefix, so one namespace can mix guarantees: /memories/ lasts forever, /workspace/ hits disk, everything else evaporates with the thread.
