Why Python looks the way it does

Python threw away the braces and semicolons and made the layout of the page carry the logic itself. Twenty minutes from now you will read block structure at a glance — the way you skim a well-made checklist.

~20 minChapter 1 of 133
The story

It is 6:05 on a Monday morning, and a brand-new duty manager at a busy hotel is holding two versions of the same emergency procedure. The first came from the legal department: one enormous paragraph, every rule wrapped in brackets inside other brackets — "in the event of an alarm (whereas the guest has checked in {deliver notice; escort to assembly point;} otherwise {log the room as vacant;})". Technically complete. Practically unreadable at speed.

The second version was written by the outgoing manager, by hand, the night before she left. It is a checklist. Big steps sit at the left edge of the page. Sub-steps are indented beneath the step they belong to. Here and there a short pencil note explains why a step exists. Nobody has to teach the new manager how to read it — the shape of the page carries the meaning. When the fire panel chirps at 6:07, guess which document she grabs.

Python is the second document. Its whole visual design comes down to one deliberate decision: the shape of the text should BE the structure of the logic. This chapter teaches you to read that shape.

One decision explains everything: layout is logic

Every programming language needs a way to say "these lines belong together". A group of lines that belong together — the body of a loop, the consequence of a decision — is called a block, which simply means a bundle of instructions treated as one unit. Most older languages mark blocks with punctuation: an opening brace { to say "the bundle starts here" and a closing brace } to say "the bundle ends here".

Here is the thing, though. Programmers who write brace languages also indent their code — they push the inner lines a few spaces to the right — purely so humans can read it. The machine reads the braces; the humans read the indentation. Two systems, describing the same structure, that can silently drift apart. A misplaced brace with tidy-looking indentation is one of the classic ways bugs hide in plain sight.

Python's designer, Guido van Rossum, started building the language in late 1989 with a simple bet: since humans already read structure from indentation — the whitespace pushing a line in from the left margin — why not make the machine read it from the same place? Delete the braces. Promote the indentation you were doing anyway from decoration to law. One source of truth, and it is the one your eyes were already using.

This idea has a wonderfully odd name: the off-side rule, coined by computer scientist Peter Landin in 1966 and borrowed from football — a line on the pitch that you may not cross. In Python, the left margin is that line. Where a line starts is not styling. It is meaning.

The same rule, written twice

Look at the identical piece of logic in a brace language (JavaScript here) and in Python. Read both slowly — not for the words, for the punctuation.

The brace-language version (JavaScript)
// JavaScript: punctuation carries the structure
for (const message of inbox) {
  if (message.startsWith("URGENT")) {
    escalate(message);
  } else {
    park(message);
  }
}
The Python version
# Python: layout carries the structure
for message in inbox:
    if message.startswith("URGENT"):
        escalate(message)
    else:
        park(message)

Same shape, same logic. But count what Python deleted: six braces, two semicolons, and two pairs of parentheses around the conditions. Nothing was added except three colons — and the indentation, which the JavaScript version had anyway. That is why people say Python "reads like English": it is not that the words are special, it is that the noise is gone. What remains on the page is almost purely the logic.

One vocabulary word before we go further: a single instruction — one line that does one thing — is called a statement, meaning one complete sentence of code. In brace languages a statement ends with a semicolon, like a full stop you must type. In Python, a statement ends where the line ends. Press Enter, the sentence is over. (Python does allow a semicolon to cram two statements onto one line — a = 1; b = 2 — but nobody writes that; it fights the whole design.)

The colon: a small promise

If layout is the structure, the language still needs one tiny signal to say "a block is about to begin". That signal is the colon — the : at the end of a line, which is Python's way of promising "an indented block follows me". Every time you see a line ending in a colon — a loop, a decision, later a function or a class — the very next thing must be at least one line indented beneath it. Colon and indent travel as a pair; a colon with nothing indented after it is an error, and an indent with no colon above it is an error too.

This gives you a lovely reading trick: colon lines are the headings of your code. Skim only the lines that end in colons and you get the outline of a Python file, the way skimming headings gives you the outline of a report.

Careful

Because whitespace is load-bearing, Python is strict about it. The convention — near-universal, and written into Python's own style guide, PEP 8 — is four spaces per level, never a mix of tabs and spaces. Mixing them looks identical on screen but is not identical to the machine, and Python 3 refuses to guess: it stops with an error rather than run a program whose shape is ambiguous. Every serious editor handles this for you (the Tab key inserts spaces); you only meet the problem when pasting code between tools.

Comments: notes to the next human

The pencil notes in the margin of that hotel checklist exist in Python too. A comment is text the machine completely ignores, written purely for the next human reader. It starts with a hash sign # and runs to the end of the line — everything after the # is invisible to Python.

# This whole line is a comment. Python never sees it.
retries = 3  # a comment can also sit at the end of a real line

Good comments do not narrate what the code does — the code already says that. They record why: "the vendor's API times out under 3 retries", "the client insists reports run at 6am". When you read production agent code, the comments are where the scars and the tribal knowledge live. (You will also meet text wrapped in triple quotes sitting under a function's first line — that is documentation with superpowers, and it gets proper treatment when we reach functions.)

print: your window into a running program

One more tool and you can read and run real code. print() is a built-in command that writes text to the screen — it is how a program reports what it is doing while it runs. Give it one thing or several, separated by commas, and it prints them on one line with spaces in between, then moves to the next line:

print("Starting triage")
print("Messages found:", 4)

Do not let its smallness fool you. Nearly every log line you have ever scrolled through in an agent trace started life as a print or its grown-up sibling, a logger call. When engineers debug, the oldest and most honest technique is still to sprinkle print statements through the code and watch what actually happens — you will do exactly that two sections from now.

Seeing structure at a glance

Here is the skill this chapter exists to give you: reading a Python file the way an architect reads a floor plan — shape first, details second. Step through the method:

1. Squint at the left edge

Margin — level 0the spine: setup and top-level flow
One indent — level 1happens inside something
Two indents — level 2inside something, inside something

Blur the words and look only at where lines start. The lines at the margin are the spine of the program — the story read top to bottom. Everything pushed right is detail that happens inside something else. Deeper means more conditional, more nested, more "only sometimes".

2. Find the colon lines — they are the headings

for message in inbox:a colon — "a block follows me"
↓
the indented blockowned by the colon line above

Every line ending in a colon opens a door. Skim only the colon lines and you have the outline: here is a loop, inside it a decision, over there a function. This is how experienced reviewers "read" a 400-line file in ninety seconds.

3. Read each block as a single thought

for message in inbox:
→
everything indented under it= "what happens for each message"

All the lines indented under a colon line are one unit — one paragraph of logic. You can summarise the whole block in a phrase ("decide what to do with this message") without reading any line inside it. Zoom in only when you need to.

4. A return to the margin is the closing brace you never type

park(message)last line inside the loop
print("Triage complete")back at the margin — runs once, after the loop

When the indentation steps back left, the block is over — no closing symbol needed. The dedent IS the closing brace. A line back at the margin after a loop runs once, after the whole loop finishes. This single observation resolves half of all "why did this run five times?" confusions.

Now read a real snippet by shape

Below is a small nightly-check script of the kind that sits behind many an operations dashboard. Before you click anything, try the squint: two colon lines, two levels of indent, a spine of four margin lines. Then click the annotated lines and confirm your reading.

nightly_check.py — read it by shape first
# Nightly check: which agent runs need attention?
A comment — the hash makes the whole line invisible to Python. It answers "why does this file exist?" for the human who opens it at 2am.
inbox = ["ok billing-bot", "FAILED triage-bot", "ok summary-bot"]
A statement at the margin: create a list of three reports and give it the name inbox. What "giving something a name" really means is the whole of the next chapter.
alerts = 0
Another spine statement: start a counter at zero. Notice there is no semicolon — the end of the line ends the statement.
for line in inbox:
A colon line — a heading. It promises an indented block and means "do the following once for each item in inbox". Loops get their full chapter in control flow; today you only need the shape.
if line.startswith("FAILED"):
One indent in: this happens inside the loop, for each line. And it is itself a colon line — a decision that opens its own, deeper block. Deeper indent, deeper question.
alerts = alerts + 1
Two indents in: this runs only for lines that start with "FAILED". Position tells you that before you read a single word.
print("Needs a rerun:", line)
Still at level 2, so still inside the decision: report the failure to the screen. In production this would be a log line; the idea is identical.
print("Checks finished. Alerts:", alerts)
Back at the margin — both doors have closed. This runs exactly once, after the loop is completely done. If it were indented one level, it would print once per message. Same words, different position, different program.

Run it, then break it on purpose

Reading is half the skill; the other half is watching the rules enforce themselves. This is a live Python environment running in your browser. Press Run, read the output next to the code, and then start vandalising — the errors are the lesson.

Tip

Three experiments, in order. One: delete the four spaces before the first print and run — Python refuses with an IndentationError, because the shape no longer makes sense. Two: remove the colon after else and run — a SyntaxError, the promise of a block with no promise made. Three: take the final print("Triage complete", ...) line and indent it to sit inside the loop — no error this time, but it prints after every message instead of once. That third one is the important lesson: indentation mistakes are not always errors. Sometimes they are just a different program.

Note

Those error names are worth a second look, because you will meet them in real logs. IndentationError: unexpected indent means a line is pushed right with no colon line above to own it. IndentationError: expected an indented block means the opposite — a colon made its promise and no block followed. Both are Python defending the one rule everything else stands on.

In the wild

You will use this skill constantly, and not only on Python. Open any agent framework's source and skim just the colon lines — you will map the file in a minute. In a merge request, watch for diffs where a line's only change is its indentation: in Python that is a logic change, not a formatting change, and reviewers who miss it approve bugs. Editors help you see structure too — thin vertical indent guides mark each level, and clicking the fold arrow in the margin collapses a whole block to one line, which is exactly the "read each block as one thought" trick, automated. And the same layout-is-structure idea governs the YAML configuration files that define most agent deployments — indent a key one level too deep and it silently belongs to the wrong section. That story gets its own chapter: YAML and JSON.

Check yourself

1. In Python, what does indentation actually do?

In brace languages indentation is decoration and the braces carry the structure. Python deleted the braces and made the indentation itself load-bearing — one source of truth, the one your eyes already use.

2. A line ends with a colon. What must come next?

The colon is a promise: "an indented block follows me". A colon with nothing indented after it is an IndentationError — the promise was broken.

3. Python has no semicolons at line ends. How does it know where a statement stops?

Press Enter, the sentence is over. This is why Python code has so little punctuation noise: the line break you were typing anyway does the semicolon's job.

4. A print line sits at the left margin, directly after an indented loop. When does it run?

The return to the margin — the dedent — is the closing brace you never type. Back at level 0 means outside the loop, so the line runs once, after the loop completes. Indent that same line one level and it would run once per item: same words, different program.

5. What does Python do with everything after a # on a line?

A comment is invisible to the machine. The best ones record why the code is the way it is — the tribal knowledge the next reader will be grateful for.

Key takeaways

  • Python's one big design decision: the layout IS the logic. Indentation defines block structure — it is law, not styling.
  • A colon at the end of a line promises an indented block. Colon lines are the headings of your code: skim them and you have the file's outline.
  • No semicolons — the end of the line ends the statement. No braces — the return to the previous margin (the dedent) closes the block.
  • To see structure at a glance: squint at the left edge, find the colon lines, read each block as one thought, and treat every dedent as a closed door.
  • Comments (# to end of line) are invisible to Python and precious to humans; print() is the program's voice and the ancestor of every log line you have ever read.
  • Indentation mistakes are not always errors — moving a line one level in or out can silently produce a different, valid program. That is why Python reviewers read whitespace like logic. Because it is.
