A2A: when agents talk to agents

MCP gave your agent hands; A2A gives it colleagues. Agent cards, tasks, messages and artifacts — the standard for one agent hiring another it will never see the inside of.

~25 minChapter 81 of 133
The story

Your interior design studio just landed a hotel project. Your own team handles the drawings, the colours, the site visits — but the custom furniture needs a specialist workshop across town. You have never seen inside that workshop. You do not know their staff, their machines, or how they organise their work. You do not need to. Their brochure — published where anyone can find it — says what they make, what formats they accept ("send drawings as PDF"), and how to place an order. You place one and receive an order number. Over the next two weeks that order has a life of its own: acknowledged, in progress, then a phone call — "which fabric for the chairs?" — and everything waits until you answer. Finally a truck arrives with the furniture and a handover file of care instructions.

Notice what made this work. Not friendship. Not shared software. A standard way of doing business between two firms that stay opaque to each other: the brochure, the order number, the status calls, the delivery. A2A is exactly that standard for agents. The brochure is an agent card, the order is a task, the phone calls are messages, and what the truck delivers are artifacts.

One protocol for tools, one for colleagues

Quick recap of where this Act has taken you. MCP exists to solve the M-by-N integration explosion: an agent's client speaks one standard protocol to any MCP server, which offers tools, resources and prompts. You built a server, and in the last chapter learned why plugging strangers' servers into your agent needs governance. Through all of it, one shape held: an agent on one side, capabilities on the other. The tool waits. The tool gets called. The tool answers in seconds and forgets you.

But another agent is not a tool. It has its own model, its own loop, its own tools behind the curtain. It may take minutes or days, come back with a clarifying question halfway through, or refuse the job. Squeeze all of that behind a tool-call interface — one request in, one result out — and you lose everything that makes it an agent. You would be phoning the furniture workshop and refusing to hang up until the chairs were finished.

A2A — the Agent2Agent protocol — is the missing counterpart: an open standard for one agent to discover, hire and collaborate with another agent over the network, as a peer rather than as a tool. The two protocols are deliberately complementary, and the industry pitch is exactly one sentence long: MCP connects agents to tools; A2A connects agents to each other.

The agent card: a brochure at a fixed address

Delegation starts with a question: who out there can do this, and how do I reach them? A2A answers with the agent card — a small JSON document an agent publishes to describe itself: its name, what it can do, where to send work, what formats it accepts, and how callers must authenticate. It sits at a well-known URL — a standardised path on a domain where machines know to look — and for A2A that path is /.well-known/agent-card.json.

Read one the way a client agent would. Click the annotated lines — the card is trimmed to the fields that matter (the full schema lives in the spec):

agent-card.json — served at https://agents.example-law.com/.well-known/agent-card.json
{
"name": "contract-review-agent",
A stable identity. Your logs, dashboards and audit trails will refer to remote agents by this name, so it is machine-friendly rather than pretty.
"description": "Reviews vendor contracts and flags risky clauses.",
This sentence is not marketing copy for humans — it is written for the calling agent's model to read when deciding whom to delegate to. Exactly like a tool description, but describing a whole colleague instead of a single function.
"url": "https://agents.example-law.com/a2a/v1",
The service endpoint — the front door where tasks are actually sent. The brochure and the workshop are different addresses.
"version": "1.4.2",
"capabilities": { "streaming": true, "pushNotifications": true },
Can you watch progress live (streamed events), and can it call you back over a webhook when a long job finishes? Vital for tasks that outlive a single connection.
"defaultInputModes": ["text/plain", "application/pdf"],
Accepted formats, as media types — the workshop saying "send drawings as PDF". There is a matching defaultOutputModes for what it produces.
"skills": [
The menu. Each skill is one advertised competence with an id, a name and a model-readable description. A client picks a skill the way your agent picks a tool.
{ "id": "review_contract",
"name": "Contract review",
"description": "Flags indemnity, lock-in and auto-renewal clauses." }
],
"securitySchemes": { "bearer": { "type": "http", "scheme": "bearer" } }
How callers must authenticate — declared here, enforced at the endpoint. A2A reuses standard web auth (API keys, OAuth and friends) rather than inventing its own. The card declares the scheme; the credentials themselves are exchanged out of band, and every warning from the security chapter applies doubly when the thing you are trusting is a whole opaque agent.
}
Note

The card is capability discovery, not capability proof. Anyone can publish a brochure claiming to be excellent at contract review. Deciding whether to believe it — vetting, allowlists, signed cards, contracts between companies — sits outside the wire protocol, exactly like trusting an MCP server did in the previous chapter.

Tasks: work orders with a lifecycle

Once a client agent has read the card, it sends a message to the endpoint, and the remote agent typically opens a task — a unit of delegated work with its own id and its own lifecycle. This is the protocol's centre of gravity. A tool call is over in seconds; a task is a work order: acknowledged, progressing, sometimes paused to ask you something, eventually ended — and both sides can look up its state at any time by its id.

Three more words complete the vocabulary. A message is one conversational turn inside the task — from the client ("audit this invoice") or from the remote agent ("which GST rate applies?"). Messages are built from parts — typed chunks of content such as text, a file, or structured data — which is how one message can carry both a question and a PDF. And an artifact is a deliverable: the task's actual output — report, file, dataset — also built from parts, kept separate from the chit-chat so consuming the result does not mean re-reading the whole conversation.

The lifecycle states will show up in your logs and dashboards, so know them by name: submitted, working, then either the terminal states — completed, failed, canceled, or rejected (real colleagues can say no) — or the two paused states: input-required, when the remote agent needs an answer before continuing, and auth-required, when it needs credentials. The paused states are the protocol's superpower — a remote agent can behave like the workshop that phones you about fabric, instead of failing the whole order.

The task lands in a terminal state and its artifacts — the deliverables — travel back. For jobs that take hours, the card's pushNotifications capability means the remote side can webhook you instead of making you poll.

Under the hood this rides on ordinary web plumbing: JSON-RPC requests over HTTPS (with gRPC and plain REST bindings added as the spec matured), server-sent events for live streaming, webhooks for push. The core requests have names like message/send, tasks/get and tasks/cancel — check the current spec for the full set, but the shapes are exactly the five steps above.

Who runs the standard — and who signed up

Provenance matters for protocols, because a standard owned by one vendor is not really a standard. Google announced A2A in April 2025 with over fifty launch partners, then two months later handed the whole project to the Linux Foundation — the neutral non-profit that also stewards Linux and Kubernetes — precisely so competitors could adopt it without strengthening a rival. It worked: the backers now include Amazon, Microsoft, Cisco, SAP, Salesforce and ServiceNow, more than a hundred and fifty organisations are involved, and the big agent platforms — Google's own developer kits, Microsoft's agent services, integrations for frameworks like LangGraph — ship A2A support. After a fast-moving series of 0.x releases, the spec reached version 1.0 in 2026.

Notice the symmetry with the start of this Act: Anthropic seeded MCP, everyone's tool protocol; Google seeded A2A and gave it to a foundation to become everyone's agent protocol. Both camps pitch them as complementary layers of one stack.

Do you actually need it? A2A versus a subgraph

Here is the question that separates architecture from fashion. You already know two ways to make agents cooperate without any network protocol: subgraphs — a supervisor graph invoking specialist graphs that share state in one process — and subagents — a deep agent spinning up scoped helpers inside one harness. Both give you multi-agent behaviour with one codebase, one debugger, one trace, zero network hops.

So when does A2A earn its keep? One test: is the boundary between the agents one you could refactor away? If both agents belong to your team, live in your repo, and trust each other completely, the boundary is artificial — collapse it into a subgraph and keep life simple. A2A pays for its complexity only when the boundary is real and immovable:

Reach for a subgraph or subagent when…Reach for A2A when…
All agents live in one codebase, owned by one teamThe other agent belongs to a different company or department — you will never see its code
Agents benefit from shared state and shared toolsThe agents are built on different vendors' stacks and cannot share a process
You want one trace and one debugger for the whole runWork must survive across systems for hours or days, with pause-and-resume
Latency matters — in-process calls are effectively freeThe relationship needs auth, audit and formal handoffs — a contract, not a function call

The pattern in practice is layered, not either/or: inside each company, teams build their agents with whatever framework they like, wiring tools over MCP and internal collaborators as subgraphs. Then they expose one polished agent at the edge, with a card at the well-known URL — the way a firm has many employees but one reception desk.

An honest maturity check

Now the part vendor decks skip. As of mid-2026, the wire protocol is in decent shape — v1.0, neutral governance, official SDKs — but it got there through churn: method names were renamed between 0.x releases, transports were added, and even the brochure's filename changed (early drafts used agent.json before settling on agent-card.json). Code written against early versions needed rework. That churn has calmed, which is genuinely good news.

The ecosystem, though, is younger than the spec. The grand vision — your procurement agent discovering a supplier's sales agent and negotiating a purchase, cross-company — mostly does not happen yet. Public directories of agent cards are embryonic. And the hard questions are not technical: should your agent believe a stranger's brochure? Who is liable when a delegated task fails? Who pays, and how? Those are contract-and-trust problems, and they move slower than specs. Meanwhile, most "multi-agent systems" in production today remain what they were in Act VII: one team's subgraphs in one process.

So hold both truths. The concepts — cards, tasks, paused states, artifacts — are durable and worth learning now: every serious platform has adopted the vocabulary, and inside large enterprises (where departments genuinely are separate trust domains) A2A is already useful glue. But if someone proposes wrapping your team's five internal agents in A2A servers "to be standards-compliant", ask the only question that matters: which boundary here could we not refactor away?

Note

Two protocols down, one to go. MCP connects your agent to tools; A2A connects it to other agents. Both point the agent outward — but nothing yet connects it back to the human at a screen, waiting to watch it think. That third protocol — AG-UI, the agent-to-user layer — gets its own act at the end of this handbook, and it completes the trio: tools below, colleagues across, the user in front.

In the wild

You will meet A2A first in platform settings screens ("Connect an external agent — paste its agent card URL") and in architecture decks as an arrow labelled "A2A" between company boxes. In code, the tells are a /.well-known/agent-card.json file in a repo, an A2A SDK in the dependency list, and task ids like task-7f3a… threading through logs. In a merge request, "expose the research agent over A2A" means: add an HTTP endpoint, publish a card, map the agent's states onto the task lifecycle. And when an operator says a delegated job is "stuck in input-required", you now know exactly which phone call nobody has answered.

Check yourself

1. MCP already lets your agent call capabilities over the network. What does A2A add that MCP does not cover?

MCP's far side is a passive capability that answers in seconds. A2A's far side is a whole agent that works over time, asks clarifying questions, and can refuse — so it needs cards, task lifecycles and artifacts. The two are complementary layers: tools below, colleagues across.

2. Your agent wants to know whether a remote agent can review contracts, and how to reach it. Where does it look?

The card at /.well-known/agent-card.json is the discovery mechanism — a machine-readable brochure. Remote agents stay opaque by design; you never see their code, only their advertised capabilities and their task states.

3. A task you delegated is sitting in input-required. What is actually happening?

The paused states are the point of having a lifecycle at all: instead of failing when information is missing, the remote agent parks the task and asks — the clarifying phone call from the workshop, encoded as a state. You answer with a message carrying the same task id, and work resumes.

4. Your team owns five agents, all in one repo, all built on one framework, all trusting each other. Should you connect them over A2A?

A network protocol between your own agents adds serialisation, auth, and distributed debugging for no gain. Use subgraphs inside the boundary, and expose one agent at the edge with a card — many employees, one reception desk.

5. Why did Google donate A2A to the Linux Foundation just two months after announcing it?

A cross-company protocol only works if every company can adopt it without strengthening a competitor. Moving A2A to a neutral foundation is what let Amazon, Microsoft, Cisco, SAP and others sign on — the same neutrality play that made Kubernetes universal.

Key takeaways

  • MCP connects agents to tools; A2A connects agents to each other — one protocol for hands, one for colleagues, deliberately complementary.
  • Discovery runs on agent cards: JSON brochures at /.well-known/agent-card.json declaring skills, endpoint, formats and auth — written for a calling model to read.
  • The unit of collaboration is the task: a work order with an id and a lifecycle (submitted, working, paused in input-required or auth-required, then completed, failed, canceled or rejected), carrying messages along the way and artifacts as deliverables.
  • Governance is neutral — Google donated A2A to the Linux Foundation in 2025, over a hundred and fifty organisations backed it, and the spec reached v1.0 in 2026 after real 0.x churn.
  • Reach for A2A only across boundaries you cannot refactor away — other companies, other vendors, other trust domains. Inside your own codebase, a subgraph is simpler, faster and easier to debug.
  • Honest status: the wire protocol is stable and the vocabulary is industry-wide, but cross-org agent ecosystems are still young — the concepts are durable even where the marketing is ahead of reality.
