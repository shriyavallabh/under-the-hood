The story

You have just hired a brilliant analyst, and she needs somewhere to work. Option one: rent her a bare room. She is smart, so she copes — but every plan lives in her head, every document is balanced in her arms, and the moment the pile grows taller than she can carry, things start slipping onto the floor. You watched exactly this happen to plain agents on long tasks in the last chapter.

Option two: you sign one contract with a serviced-office company. The very same room now arrives fitted out. A whiteboard on the wall for her plan. A filing cabinet so nothing has to be carried. A phone with specialists on speed dial for work she should not do herself. And a quiet assistant who condenses yesterday's meeting minutes into one tidy page before the paperwork drowns her.

Same analyst. Same brain. One signature. That signature is create_deep_agent — one call that takes the plain agent you already know and moves it into a fully serviced office before the first task even arrives.

Four pillars, one signature

In the previous chapter you saw what separates a deep agent from a plain tool-caller: it plans before acting, it parks bulky material in a filesystem instead of its own context, it delegates messy research to fresh-context subagents, and it manages its own context so an hours-long run does not drown in its own transcript. Four pillars. What that chapter deliberately skipped is how much wiring those pillars need — and how little of it you have to do yourself.

The answer is the deepagents library, and its answer is one function:

one import, one call
from deepagents import create_deep_agent

agent = create_deep_agent(
    tools=[search_invoices],
    system_prompt="You are a meticulous finance researcher.",
)

If that shape looks familiar, it should. It is almost exactly create_agent — model, tools, system prompt — which gave you a native tool-calling version of the fifty-line ReAct loop you built by hand. create_deep_agent starts from that same core and then keeps going: it wraps the loop in a stack of middleware — remember, middleware is code that hooks into the agent loop at defined points, able to add tools, extend the prompt, and intercept messages on the way in and out. Each pillar of deepness is one middleware layer, pre-wired and pre-tested, installed for you by this single call. That makes create_deep_agent a factory function — a function whose whole job is to assemble and return a configured object, so you never touch the assembly line yourself.

Here is the fuller signature with the parameters you will actually meet in real code. Every one of them is optional except the ones you care about that day:

the knobs you will meet in the wild
agent = create_deep_agent(
    model="openai:gpt-5.5",   # a model string or model object
    tools=[search_invoices],  # your tools, same as create_agent
    system_prompt="...",      # your standing instructions
    subagents=[...],          # specialist workers        (ch70)
    middleware=[...],         # your own extra layers     (ch55)
    backend=...,              # where "files" really live (ch69)
    checkpointer=...,         # pause, resume, threads    (ch61)
    interrupt_on={...},       # approval gates per tool   (ch62)
)
Note

The model accepts the same shorthand strings as init_chat_model — "openai:gpt-5.5", "anthropic:claude-sonnet-4-6", a local model — or a ready-made model object. Leave it out and the library falls back to its own default; production code almost always passes it explicitly, so check the current docs for what your version defaults to.

What comes back: a graph you already know

Here is the part that makes this chapter short on new machinery: create_deep_agent does not return some exotic DeepAgent object with its own rules. It returns a compiled LangGraph graph — the finished, runnable object of the same kind you get when you call .compile() on a StateGraph you built yourself. The factory assembles nodes, edges, and middleware on your behalf, compiles the result, and hands it over.

That one fact carries the whole of Act VII with it, unchanged:

  • You run it with .invoke() or watch it live with .stream(), exactly as in the compile-and-invoke chapter.
  • Give it a checkpointer and a thread_id and it can pause, resume, and survive restarts.
  • Human-in-the-loop works too: pass interrupt_on a mapping like {"write_file": True} and the graph pauses for approval before that tool runs — a checkpointer is required for this, since pausing means saving state.

Invoking it looks like invoking any other graph, because it is any other graph:

invoke it like any graph
result = agent.invoke(
    {"messages": [{"role": "user",
                   "content": "Research our March invoicing."}]},
    config={"configurable": {"thread_id": "run-42"}},
)

The state that comes back, though, is richer than a plain agent's. Alongside result["messages"] you will find a "todos" key holding the plan the agent wrote for itself, and — with the default backend — a "files" key holding everything it filed away during the run. Those two keys are the visible fingerprints of the middleware stack, which it is now time to open up.

Picture the assembly line inside the factory. It starts with the bare chassis — the loop — and each station bolts on one layer. Step through it:

2. TodoListMiddleware: the whiteboard

The first layer bolts a planning tool called write_todos onto the tool belt and appends instructions to the system prompt telling the model to plan multi-step work before diving in. The plan it writes lands in graph state under "todos" — visible to you, revisable by the agent. The next chapter lives entirely inside this layer.

3. FilesystemMiddleware: the filing cabinet

FilesystemMiddlewarels, read_file, write_file, edit_file, glob, grep

Six file tools arrive: ls, read_file, write_file, edit_file, plus glob and grep for finding things. (A seventh, execute, runs shell commands — but it appears only when you swap in a sandbox backend that supports it; on the default virtual filesystem the library hides it.) By default they operate on a virtual filesystem living in graph state — paths like /notes/march.txt that never touch your disk. Swap the backend to reach a real disk or a store. One tool called task lets the main agent hand a subtask to a fresh-context worker and receive only the final report back — a hundred noisy searches happen in the worker's context, one clean paragraph returns. Specialists you declare via subagents= get their own name, description, and system prompt. Chapter 71 goes deep here.

The quiet assistant. It watches the growing transcript, and when it gets too long it condenses the older turns into a summary while keeping recent messages intact — so an hours-long run never hits the context ceiling. This is the context-management problem handled as a standing service.

6. The quiet layers, then compile

the whole stackplus PatchToolCallsMiddleware (+ SkillsMiddleware if you pass skills=)
→

One quiet layer always rides along: PatchToolCallsMiddleware repairs tool calls left dangling by an interrupted run. A second, SkillsMiddleware — which exposes reusable skill instructions to the agent — is a conditional append: it slots in just after planning only when you pass skills=..., and is absent otherwise. Any middleware=[...] you pass is slotted in near the tail. Then the whole assembly is compiled into a single LangGraph graph — the return value. One call in, one serviced office out.

Tip

A lovely detail hides in the subagent layer: workers spawned through task get almost the same serviced office themselves — planning, files, summarization — but no SubAgentMiddleware of their own. Subagents cannot spawn subagents by default, so a confused agent cannot recurse into an org chart of infinite interns.

In the wild

Open the trace of almost any deepagents-based run and the fingerprints are unmistakable: the first tool call is nearly always write_todos, long research bursts appear as single task calls whose inner storm you can expand, and file paths like /notes/findings.md flow through write_file. In code review you will meet create_deep_agent( with a subagents=[...] list of dicts — name, description, system prompt — and someone asking "should this backend persist?". And when you watch a production coding agent print a checklist and then spawn a subtask, you are watching this exact architecture wearing a different badge; chapter 72 puts the two side by side.

Check yourself

1. What kind of object does create_deep_agent return?

It is a compiled graph. That is why nothing from Act VII changes: invoke, stream, checkpointers, thread ids, and interrupts all work exactly as you learned them.

2. You pass tools=[search_invoices] — one tool. What menu does the model actually see?

The middleware layers each bolt their tools onto the belt: planning adds write_todos, the filesystem layer adds ls / read_file / write_file / edit_file / glob / grep, and the subagent layer adds task. Your tool rides alongside the built-ins on one flat menu.

3. With the default backend, where does write_file("/notes/plan.txt", ...) actually put the file?

The default is the state backend: files are entries in graph state, gone when the run ends unless checkpointed. Swap in a filesystem or store backend when you want real persistence — that trade-off is the whole of chapter 70.

4. An agent has been running for an hour and its transcript is enormous. Which layer stops it drowning?

Summarization is the standing context-management service: it watches transcript size and condenses the older turns while keeping recent ones intact, so long runs never slam into the context ceiling.

5. You print the deep agent's final system prompt and it is far longer than the system_prompt you passed. Why?

Your prompt is the opening chapter, not the whole book. Planning, filesystem, and subagent layers each append their tool manuals — how to plan, when to file, how to delegate — so the model knows how to use the office it was given.

Key takeaways

  • create_deep_agent(tools=..., system_prompt=...) is a factory: one call assembles the four pillars of deepness around a standard tool-calling loop and returns the result.
  • What it returns is a compiled LangGraph graph — invoke it, stream it, checkpoint it, interrupt it exactly like any graph you built by hand in Act VII.
  • The assembly is a middleware stack, in order: planning (write_todos), then — only when you pass skills=... — skills, then filesystem (ls, read_file, write_file, edit_file, glob, grep), subagents (task), summarization, and tool-call repair (the last always on).
  • You supplied one tool; the model sees about nine — and your system prompt gains each layer's usage manual automatically.
  • By default files and todos live in graph state: virtual, inspectable in the result, gone unless checkpointed or given a persistent backend.
  • Subagents get the same equipment minus the task tool — delegation goes one level down, never infinitely.
