Inheritance, composition, and framework style

Why framework code says class MyAgent(BaseAgent), what super() actually does, and a five-step method for reading any class hierarchy without drowning in it.

~30 minChapter 15 of 117
The story

Twenty years ago a woman in Pune perfected a dosa. Today her name is on three hundred franchise outlets, and every new franchisee receives the same fat operating manual from head office: how to open, how to clean, how to bill, how to greet, how to close. A new outlet does not write its own manual. It inherits head office's — and then rewrites exactly one page, because their neighbourhood likes a spicier chutney. The staff rule is simple: if your outlet has its own version of a page, follow yours; for every other page, follow head office's.

The smartest franchisees who rewrite a page still open it with "first do everything head office says, then add our step". And notice what the outlet does not inherit: the billing machine. Nobody writes "how to be a billing machine" into the manual — the outlet simply has one, bought off the shelf, replaceable tomorrow without retraining a single cook. Hold those three moves in your head — inherit the manual, override one page, own the machine — and you already understand inheritance, method overriding, and composition. The rest of this chapter is just spelling.

The parentheses in a class line are a family tree

You have just spent two chapters inside classes — building objects with __init__ and meeting the dunder handshakes Python uses behind the scenes. One small piece of syntax was quietly waiting: the parentheses in a class definition. When you read

the single most common line in framework code
class InvoiceAgent(BaseAgent):
    ...

the parentheses do not mean "call BaseAgent" the way they do after a function name (chapter 7). Here they declare parentage: InvoiceAgent is a kind of BaseAgent. This is inheritance — a class receives every method and attribute of another class automatically, without copying a single line. BaseAgent is the base class (also called the parent or superclass) — the head-office manual. InvoiceAgent is the subclass (the child) — the new outlet, which gets the whole manual for free and only writes down what it does differently.

That is the entire reason frameworks are usable at all. The authors of an agent framework write thousands of lines of hard machinery — retries, streaming, callbacks, message formatting — once, in a base class. You subclass it, write the ten lines that are specific to your situation, and inherit the rest. When you see a tiny class in a repository doing apparently enormous things, this is the trick: the class is small because its parents are large.

Run it: one base class, two personalities

Here is a miniature version of what every model-wrapper base class in every framework does. The base class owns the machinery every model shares — counting calls, tidying output. Each subclass overrides exactly one method, _reply, the way an outlet rewrites one page. Press Run, then read the code top to bottom.

Three things to notice, because they are the shape of every framework you will ever read:

First, the subclasses never define invoke, yet you called it on both. When Python cannot find a method on an object's own class, it climbs to the parent and looks there, then the grandparent, all the way up. The formal name for that climbing order is the method resolution order — the queue Python consults, child first, then ancestors — but "look upwards until you find it" is ninety percent of the truth.

Second, ScriptedModel.__init__ begins with super().__init__(name). super() means "the version of this method from my parent". Overriding __init__ replaces the parent's setup entirely — so if you forget that line, self.name and self.calls simply never get created, and some innocent line far away crashes later with AttributeError. This is one of the most common bugs in subclass code, and now you can spot it in review: an overridden __init__ with no super().__init__ call is a raised eyebrow.

Third, the base class's _reply exists only to raise NotImplementedError. That is a deliberate design move called a hook — a method the base class declares but expects children to fill in. The base owns the skeleton (invoke), the child fills one hole (_reply). Framework authors call this the template-method pattern; you can just call it "the manual and the chutney page".

Tip

Break it on purpose. Delete the super().__init__(name) line and run again — read the AttributeError carefully, because you will meet it in real logs. Then add a third subclass, PoliteModel, whose _reply returns "With respect: " + prompt. If it works on the first try, inheritance has clicked.

Reading a framework subclass in the wild

Now let's read the real thing — the kind of file you will find in production agent repositories everywhere. A team wants to use its company's internal model gateway with a LangChain-style framework, so they wrap it in a custom model class. This is lightly simplified from real code (hook names vary slightly between framework versions — the pattern is the lesson). Click the annotated lines:

chat_acme.py — a custom model, framework style
from langchain_core.language_models.chat_models import BaseChatModel
The import tells you the family this class is marrying into. BaseChatModel is the framework's head-office manual for "things that behave like a chat model".
class ChatAcme(BaseChatModel):
The headline. Read it aloud as "ChatAcme IS A BaseChatModel". Everything the framework knows how to do with a chat model — invoke it, stream it, retry it, plug it into an agent — it can now do with ChatAcme, sight unseen.
model_name: str = "acme-large"
Configuration declared as class-level fields — this base class uses the style you will study properly next chapter (type hints and Pydantic). For now: these are the knobs a user can set when creating the object.
temperature: float = 0.2
@property
def _llm_type(self):
A tiny required hook: the framework asks every model "what kind are you?" for logging and tracing. One honest string and you are done.
return "chat-acme"
def _generate(self, messages, stop=None, **kwargs):
The chutney page. The base class's public invoke handles callbacks, retries, and message plumbing, then calls THIS hook to do the actual model call. Override one underscore method and the whole framework machine runs through your gateway.
payload = to_acme_format(messages)
Ordinary code — translate the framework's message objects into whatever the internal gateway expects. Helper functions like this live in the same file, and they are just functions.
raw = acme_gateway_call(self.model_name, payload)
The only line that touches the outside world. Note it reads self.model_name — the field declared at the top, inherited machinery filling it in from whatever the user passed.
return to_chat_result(raw)
Translate the gateway's answer back into the framework's expected return shape. Fifteen lines total, and ChatAcme now works everywhere the framework works.

Here is the punchline, and it resolves a mystery you have probably shrugged past a hundred times: this class has no invoke method, yet every tutorial calls ChatAcme().invoke(...). The invoke everyone calls lives on an ancestor. The subclass is not where the action is — it is where the differences are. Which raises the practical question: when you need to find where the action actually lives, how do you climb a five-storey class hierarchy without getting lost?

How to read a class hierarchy without drowning

A real BaseChatModel-like chain is several classes deep: your class, on top of a base model class, on top of a general language-model class, on top of a runnable-things class, on top of object — the silent ancestor of every Python class. Newcomers open the base class file, see two thousand lines, and despair. Don't. You are never supposed to read a hierarchy top to bottom like a novel. You read it like a detective with one question. Note

Two power moves for step 3. In any editor, "Go to definition" on a method call jumps you straight to the level that defines it — the climb, automated. And in a pinch, print(ChatAcme.__mro__) prints the full parent chain in resolution order, one of those dunder handshakes from last chapter doing honest work. You may also meet a class line with several parents — class ChatAcme(SerialisableMixin, BaseChatModel). Those extra parents are usually mixins: small bolt-on ability packs. Read them left to right and do not panic; the hunt-one-method technique works unchanged.

Composition: the billing machine, not the manual

Inheritance answers "what kind of thing is this?" But look back at the story: the outlet's billing machine was never part of the manual. The outlet simply has one. That is composition — building an object by giving it other objects as attributes and delegating work to them, rather than by being a subclass of them. Inheritance is is-a; composition is has-a. Run the difference:

The Agent class never inherits from ScriptedModel. It receives a model in __init__, stores it on self, and calls self.model.invoke(...) when needed — exactly the shape of the ReAct loop you will build by hand in chapter 45. Because the agent only cares that its model has an invoke method, you can hand it any object that does: a real API-backed model, a cached one, a scripted fake for tests. Swapping takes one line at construction time. With inheritance, changing your parent means changing your class definition — surgery. With composition, changing a part means passing a different argument — a plug.

This is why the design proverb says prefer composition over inheritance. Inheritance is a powerful, rigid bond: the child is coupled to every detail of its ancestors forever. Composition is a loose, swappable bond. Modern agent frameworks lean heavily on it — when LangChain code calls create_agent with a model and a list of tools, nothing there subclasses anything of yours. The agent has a model and has tools, and you compose the parts like Lego. The rule of thumb worth memorising: subclass when the framework explicitly asks you to fill in its hooks; compose for everything you might ever want to swap.

Abstract base classes: forms with mandatory fields

One refinement and the toolkit is complete. Our ChatModel._reply raised NotImplementedError when called — a polite hint, but it only fires if and when someone calls the missing method, possibly hours into a run. Python offers a stricter contract: the abstract base class (ABC) — a base class that declares certain methods as mandatory and refuses to create any instance of a subclass until they are all implemented.

abc — the strict version of a hook
from abc import ABC, abstractmethod

class BaseNotifier(ABC):
    @abstractmethod
    def send(self, message):
        """Every notifier must know how to send."""

class TelegramNotifier(BaseNotifier):
    def send(self, message):
        print("sending:", message)

TelegramNotifier().send("Invoice 42 is paid")   # works

class LazyNotifier(BaseNotifier):               # forgot send()
    pass

# LazyNotifier()  -> TypeError, immediately, at creation time:
# "Can't instantiate abstract class LazyNotifier..."

The @abstractmethod marker (a decorator — the machinery behind that @ sign gets its own chapter in chapter 17) turns "please override this" into "you cannot even construct the object until you do". The failure moves from some random moment at runtime to the instant of creation — loud, early, and easy to fix. When you meet ABC and @abstractmethod in a framework's source, read them as a signpost, because that is what library authors use them for: these are exactly the methods you must implement; everything else is optional. An abstract base class is the framework handing you the form with the mandatory fields highlighted.

In the wild

Grep any serious Python repository for class and watch the three moves of this chapter appear everywhere. Framework-facing code subclasses to fill hooks: a custom chat model overriding _generate, a Pydantic class Settings(BaseModel) in every config file you will ever open (next chapter's star). Application code composes: an agent constructed from a model, a tool list, and a checkpointer — three swappable parts, no subclassing. deepagents even ships composition as a feature: its file backends (StateBackend, FilesystemBackend, CompositeBackend) are interchangeable parts you hand to create_deep_agent. And in merge-request comments you will now decode the dialect instantly: "push this up into the base class" (three children duplicate a method — move it to the parent), "prefer composition here" (you subclassed something you should have stored as an attribute), and "you forgot to call super()" (the AttributeError you met earlier, caught in review before it hit production).

Check yourself

1. A repository contains class InvoiceAgent(BaseAgent):. What does this line mean?

Parentheses in a class line declare a parent, not a call. InvoiceAgent gets BaseAgent's whole manual for free and writes down only its differences. Option c describes composition — a different, looser relationship.

2. Inside a subclass's __init__, what is super().__init__(name) for?

Overriding a method replaces the parent's version entirely. super() reaches back up and runs the parent's version too — skip it in __init__ and the parent's attributes silently never exist, crashing later with AttributeError.

3. You call model.invoke("hi") on a ChatAcme object, but the ChatAcme class contains no def invoke. What happens?

Method lookup walks the family tree — child first, then each ancestor in order (the method resolution order) — and the first match runs. That is why a fifteen-line subclass can respond to dozens of methods it never wrote.

4. An agent is built as Agent(model=ChatAcme(), tools=[search, calculator]) and stores both on self. Which relationship is this?

Receiving parts in the constructor and storing them as attributes is has-a, not is-a. That looseness is the point: hand the same Agent a fake scripted model in tests and a real one in production, zero code changes.

5. A base class marks send with @abstractmethod. A subclass forgets to implement it. When does Python complain?

That is the whole value of ABCs: the failure fires at construction time, loud and early, instead of lurking until some code path finally calls the missing method mid-run. Option c describes the weaker raise-NotImplementedError style.

Key takeaways

  • class Child(Parent) means is-a: the child inherits every method and attribute of the parent and overrides only what it does differently — the franchise manual with one rewritten page.
  • super() means "also run the parent's version". An overridden __init__ without super().__init__(...) silently skips the parent's setup — a classic bug you can now catch in review.
  • Frameworks follow the hook pattern: a big public method in the base class (invoke) calls small underscore hooks (_generate) that your subclass fills in. Small subclass, large inherited machinery.
  • To read a hierarchy: read your class fully, treat the parent's name as a contract, hunt one method at a time up the chain, find the hooks — then stop climbing.
  • Composition (has-a) beats inheritance for anything you might swap: store parts as attributes and delegate, the way an agent has a model and has tools. Subclass only when a framework asks you to fill its hooks.
  • Abstract base classes turn "please override this" into "you cannot construct the object until you do" — the framework's way of highlighting the mandatory fields.
