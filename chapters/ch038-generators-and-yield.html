Generators and yield

A function that can pause mid-sentence, hand you one piece, and resume exactly where it stopped. Master this one Python trick and streaming — the typewriter effect you watch every day — stops being magic.

~25 minChapter 39 of 135
The story

Every night at nine, a grandmother settles into her chair and continues the great family story — one instalment at a time. She never tells it all in one sitting; nobody could stay awake that long. When the clock strikes ten she stops mid-scene — "and just as the merchant lifted the lid of the box…" — and sends everyone to bed. Here is the remarkable part: the next night she resumes from that exact half-sentence. Not a recap from the beginning. Not roughly where she was. Exactly there — every character's name, every plot thread, every unfinished sentence held perfectly in her head between sittings.

Python has a kind of function that works precisely like this grandmother. An ordinary function is a printed book: the whole thing is produced up front and delivered in one heavy lump. This other kind pauses mid-story, hands you tonight's instalment, keeps all its memories, and resumes on request. It is called a generator, the pause is spelt yield — and it is the exact mechanism by which an LLM's reply appears on your screen word by word.

A function that can pause

Everything you know about functions so far comes from the functions chapter: you call one, it runs top to bottom, it hits a return, it hands back a value — and then it dies. All its local variables vanish. Call it again and it starts from scratch, remembering nothing. That is the printed-book model: one delivery, then it is over.

yield changes the deal. A generator function is any function with the keyword yield in its body — and that single keyword rewires how the whole function behaves. When execution reaches a yield, the function hands out a value and pauses right there, mid-body, with every local variable frozen in place. Ask for the next value and it wakes up on the very next line, memories intact, and runs until it reaches another yield. Return says "here is everything, goodbye". Yield says "here is the next piece, I will wait".

There is a second surprise, and it trips up almost everyone once: calling a generator function runs none of its code. Instead it hands you a generator — a small object that holds the paused function, ready to start. Think of it as the grandmother sitting down in her chair: the story exists, but not a single word has been spoken yet. Code only runs when someone asks for a piece.

You will almost never catch it by hand the way we just did, and the next section explains why.

What a for loop really does

Here is the payoff of that odd StopIteration ending: it is exactly the signal a for loop listens for. Ever since the control-flow chapter you have written for item in something: without asking how Python actually pulls items out of the something. The answer is a tiny, universal handshake called the iteration protocol — the agreed set of moves by which any loop asks any object for its next item. Lists, dicts, files, and generators all speak it, which is why one loop syntax works on all of them. Step through the handshake:

1. The loop asks for an iterator

for loopfor course in meal:
→
iter(meal)
generatorpaused at the top

Behind the scenes, for first calls iter() on the object to get an iterator — the thing that will actually be asked for items. A generator is already its own iterator, so it just says "that's me".

2. The loop calls next()

for loop
→
next()"give me one"
→
generatorruns to the first yield

Each turn of the loop is secretly a next() call. The generator's body runs from wherever it was paused until it hits a yield, hands out that value, and freezes again.

3. Your loop body uses the value

for loopcourse = "starter: tomato soup"
next()
generatorfrozen, locals intact

While your loop body works with the value, the generator sits paused. No wasted work: piece two is not cooked until someone actually asks for it.

4. Repeat: next(), yield, freeze

for loop
→
next()"another, please"
→
generatorresumes AFTER the last yield

The resume point is the line after the previous yield, with every local variable exactly as it was — the grandmother picking up mid-sentence.

5. StopIteration ends the loop, silently

generatorbody finished
→
StopIteration
→
for loopends cleanly, no error shown

When the generator's body runs out, next() raises StopIteration. The for loop catches it for you and simply stops. That is the entire reason your loops end gracefully instead of crashing.

Note

This protocol is why for line in open("agent.log"): works, why for row in cursor: works, and why — in the very next chapters — for chunk in stream: works on a live LLM response. Anything that answers next() and eventually raises StopIteration can be looped over. Generators are just the easiest way to build such a thing yourself.

The streaming mental model: a sentence, piece by piece

Now the demo this whole chapter has been building towards. When you watch an AI assistant "type" its answer, here is what is actually happening: the server sends the reply in small chunks over a long-lived connection, the SDK wraps those chunks in an iterator, and the chat UI runs a plain for loop — appending each chunk to the screen as it arrives. Producer pauses, consumer asks, producer resumes. It is exactly the generator handshake you just learned.

Why not just build the whole list?

Fair question. Lists have served you well for thirty-seven chapters — why bother with a function that dribbles values out one at a time? Two reasons, and both matter daily in agent work.

Reason one: memory. A list must hold every item at once. A generator holds one — plus a bookmark. This is lazy evaluation: values are produced only at the moment they are asked for, never stockpiled. For ten items the difference is nothing. For ten million log lines, it is the difference between a script that hums along and one that eats your machine's RAM and gets killed. The numbers below are illustrative round figures, but the shape is honest:

Reason two: time-to-first-item. A list-builder must finish everything before you see anything. A generator hands over piece one immediately. That is why streaming an LLM reply feels fast even when the total generation time is identical — the first word lands in half a second instead of the whole answer landing after twenty. You already know this trade-off as a user; now you know its name.

You have been using lazy iteration without knowing it. File objects speak the same protocol:

reading a huge log, the lazy way
# The eager way: pulls the ENTIRE file into memory at once
lines = open("agent.log").readlines()   # 2 GB file -> ~2 GB of RAM

# The lazy way: the file object is an iterator -- one line at a time
for line in open("agent.log"):          # ~one line of RAM, total
    if "ERROR" in line:
        print(line.rstrip())

(In real code you would wrap that open in a with block — context managers — so the file closes itself. Kept bare here so the laziness is the only idea on the page.)

Careful

Generators are single-use. Once a generator has raised StopIteration, it is exhausted — loop over it a second time and you get nothing, instantly, with no error. If you see a "my loop ran zero times" bug where the data definitely exists, check whether the iterable was a generator that something else already consumed. When you need to go over the data twice, either recreate the generator or collect it into a list first with list(gen) — deliberately paying the memory cost.

Generator expressions: the lazy sibling of comprehensions

Cast your mind back to comprehensions: [w.upper() for w in words] — the one-line recipe that builds a whole list. Swap the square brackets for round ones and you get a generator expression — the same recipe, but lazy: it builds a generator that produces each item on demand instead of a list that holds them all.

brackets decide: eager list or lazy generator
words = ["stream", "chunk", "token", "yield"]

eager = [w.upper() for w in words]   # list: all 4 exist right now
lazy  = (w.upper() for w in words)   # generator: nothing computed yet

print(eager)      # ['STREAM', 'CHUNK', 'TOKEN', 'YIELD']
print(lazy)       # <generator object ...> -- a paused recipe

# Feed a generator expression straight into a consumer:
total = sum(len(w) for w in words)   # no throwaway list ever built

That last line is the idiom you will meet constantly in real codebases: sum(...), max(...), any(...), and "".join(...) all happily drink from a generator expression, item by item, without a temporary list ever existing. When the expression is the only argument, you may even drop the inner brackets — sum(len(w) for w in words) — which is why the pattern looks so bare in the wild. If you can read a comprehension, you can read this: same recipe, different delivery schedule.

Without today's chapter, that line looks like magic. With it, you can see the machinery: a producer pausing at a yield, a consumer calling next, and StopIteration ending things politely when the model finishes.

In the wild

Search any serious Python codebase for yield and you will find the load-bearing walls. LangChain's .stream() methods hand back exactly this kind of lazy iterator of chunks, consumed with a plain for loop. Web frameworks stream responses by accepting a generator and sending each yielded piece to the browser as it is produced. Data pipelines chain generators so a terabyte flows through a laptop one record at a time. Test suites use yield inside fixtures to run setup, pause for the test, then resume for cleanup. And in a merge request, when someone writes "made the export lazy so it stops loading everything into memory", the diff is usually three characters: a return that became a yield.

Check yourself

1. You write stream = fake_llm_stream("hello"), where the function body contains a yield. How much of the function's code has run?

Calling a generator function runs zero lines of its body. It returns a generator, frozen at the top. Code only executes when something asks for a value — a next() call or a for loop turn.

2. What does yield do that return does not?

Return ends a function and destroys its locals. Yield suspends it — position and variables frozen — so the next request resumes on the following line, memories intact. That pause-and-resume is the entire trick.

3. How does a for loop know when a generator has no more values?

When the generator's body finishes, next() raises StopIteration and the for loop treats that as "we are done" — no error shown. There is no advance count; a generator does not know its own length, which is exactly what lets it be endless or network-fed.

4. You loop over a generator once, then loop over the same variable again. What does the second loop see?

Generators are single-use. Once exhausted, further loops end immediately and silently — a classic source of "my loop mysteriously did nothing" bugs. Recreate the generator, or collect it with list() if you truly need two passes.

5. Your script must scan a 2 GB log file for errors on a small machine. Why is for line in open(path): better than open(path).readlines()?

readlines() builds a full list — the entire file in RAM before any work starts. The file object itself is a lazy iterator: one line at a time, constant memory, and the first line is available immediately.

Key takeaways

  • A generator function — any function containing yield — pauses at each yield with all local state frozen, and resumes exactly there on the next request. Return ends; yield waits.
  • Calling a generator function runs none of its code: it returns a paused generator object. Values are produced only when asked — by next(), or by a for loop calling it for you.
  • The iteration protocol is the universal handshake behind every for loop: get an iterator, call next() per turn, stop silently on StopIteration. Lists, files, and live LLM streams all speak it.
  • Generators beat full lists on memory (one item plus a bookmark, versus everything at once) and on time-to-first-item — the exact reason streaming replies feel fast.
  • Generator expressions are comprehensions with round brackets: the same recipe from chapter 8, delivered lazily instead of built eagerly.
  • Generators are single-use — an exhausted one silently yields nothing. Recreate it or collect it into a list when you need a second pass.
  • Streaming an LLM reply is this chapter wearing a network cable: the SDK hands you an iterator of chunks, and your code is just for chunk in stream:.
