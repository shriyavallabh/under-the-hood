The story

Walk into a well-run customer support office. At the front desk, a clerk reads each incoming letter and stamps it: billing, technical, or refund. On the wall behind her hangs a chart saying which desk each stamp goes to. Ordinary questions go to the desk with the product manuals, where someone looks up the answer and writes a kind reply. Refund requests go to a different desk — the one with the locked cash drawer. The clerk there fills in the slip, then stops. She cannot open the drawer. Only the floor manager's signature can, and the manager might be in a meeting until Monday.

Here is the quiet genius of the office: after every step, each case file is photocopied into the records room. If the power fails on Friday night, Monday's shift pulls file 4712 and finds it exactly where it paused — stamped, routed, waiting on one signature. Nothing is lost, nothing is redone from scratch.

You have spent seven chapters learning each station in this office one at a time. Today you read the blueprint of the whole building — in Python.

In StateGraph anatomy you learned that a graph app has exactly three ingredients: a state schema (the shared case file), nodes (plain functions that read the state and return updates), and edges (who runs after whom). In conditional edges you met the router — a small function that returns the name of the next node, letting the graph branch. In compile, invoke, stream you saw the lifecycle: build, .compile(), .invoke(). Then checkpointing gave the graph its records room, interrupts gave it the manager's signature gate, and bring your own graph gave you the conviction that control flow should be deterministic even when an LLM fills in the data.

What you are about to read is a triage app — triage simply means sorting incoming work by what kind of help it needs. A support ticket comes in; an LLM classifies it; deterministic code routes it; ordinary questions get answered with a knowledge-base tool; refunds hit a human approval gate and pause — durably — until someone decides. This is not a toy shape. Swap the nouns and it is a loan-approval flow, an expense-claim flow, a content-moderation flow. The refund gate is the same approval gate a bank's risk team insists on.

The whole app, cold

triage_app.py — the complete listing
"""Support-ticket triage: a complete LangGraph application.

Flow: classify, then route, then one of two desks, then END.
Refunds pause for human approval; a checkpointer makes the pause safe.
"""
from typing import TypedDict, Literal

from langchain.chat_models import init_chat_model
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import interrupt, Command

llm = init_chat_model("anthropic:claude-sonnet-4-5")

# ---------- 1. The state schema ----------
class TicketState(TypedDict, total=False):
    ticket_id: str
    customer: str
    text: str          # what the customer wrote
    category: str      # filled in by classify()
    amount: float      # filled in by handle_refund()
    approved: bool     # filled in by the human gate
    reply: str         # what we finally send back

# ---------- 2. Plain-Python tools ----------
KB = {
    "sync":    "KB-17: sign out and back in to reset the sync token.",
    "invoice": "KB-31: invoices regenerate within 24 hours of a plan change.",
}

def search_kb(text: str) -> str:
    hits = [note for key, note in KB.items() if key in text.lower()]
    return "\n".join(hits) or "No matching notes."

def issue_refund(customer: str, amount: float) -> None:
    print(f"[payments] refunded {amount} to {customer}")  # imagine a real API

# ---------- 3. Nodes: plain functions over the state ----------
def classify(state: TicketState) -> dict:
    prompt = (
        "Classify this support ticket as exactly one word - "
        f"billing, technical, or refund:\n\n{state['text']}"
    )
    category = llm.invoke(prompt).content.strip().lower()
    return {"category": category}

def answer_with_docs(state: TicketState) -> dict:
    notes = search_kb(state["text"])          # the tool call
    prompt = (
        f"Using ONLY these notes:\n{notes}\n\n"
        f"Write a short, kind reply to:\n{state['text']}"
    )
    return {"reply": llm.invoke(prompt).content}

def handle_refund(state: TicketState) -> dict:
    prompt = f"Extract the refund amount as a bare number:\n{state['text']}"
    amount = float(llm.invoke(prompt).content.strip())
    decision = interrupt({                    # the graph pauses HERE
        "question": f"Approve refund of {amount} for {state['customer']}?",
        "ticket_id": state["ticket_id"],
    })
    if decision == "approve":
        issue_refund(state["customer"], amount)  # side effect AFTER the gate
        return {"amount": amount, "approved": True,
                "reply": f"Done - {amount} is on its way back to you."}
    return {"amount": amount, "approved": False,
            "reply": "We reviewed your request and cannot refund this plan."}

# ---------- 4. The router ----------
def route_ticket(state: TicketState) -> Literal["handle_refund",
                                                "answer_with_docs"]:
    if state["category"] == "refund":
        return "handle_refund"
    return "answer_with_docs"

# ---------- 5. Wire the graph ----------
builder = StateGraph(TicketState)
builder.add_node("classify", classify)
builder.add_node("answer_with_docs", answer_with_docs)
builder.add_node("handle_refund", handle_refund)

builder.add_edge(START, "classify")
builder.add_conditional_edges("classify", route_ticket)
builder.add_edge("answer_with_docs", END)
builder.add_edge("handle_refund", END)

# ---------- 6. Checkpointer + compile ----------
checkpointer = InMemorySaver()   # swap for Postgres in production
app = builder.compile(checkpointer=checkpointer)

# ---------- 7. Run it ----------
config = {"configurable": {"thread_id": "ticket-4712"}}
state = app.invoke(
    {"ticket_id": "4712", "customer": "Asha",
     "text": "Please refund my 999 annual plan."},
    config,
)
print(state["__interrupt__"])    # paused: waiting for a human

# ...minutes or days later, possibly in a different process...
state = app.invoke(Command(resume="approve"), config)
print(state["reply"])

One LLM call, one word back. The node returns {"category": "refund"} and LangGraph merges it into the shared state — then writes a checkpoint. The photocopier hums.

The conditional edge runs plain Python and returns a node name. The LLM supplied the data; deterministic code makes the control-flow decision. That split is the whole philosophy of bring your own graph.

Mid-node, interrupt() halts the graph. invoke returns early, and the returned state carries an __interrupt__ entry describing what it is waiting for — the payload your approval screen displays. The server can now redeploy, crash, or sleep for the weekend. The case file is safe in the records room.

A new invoke with the same thread_id reloads the checkpoint and re-enters the node from its first line. This time interrupt() does not pause — it returns the resume value, "approve", and execution flows on past the gate.

Only now — behind the signature — does money move. The node returns its final updates, the edge leads to END, and the last checkpoint records a finished thread. Asha gets her reply.

Annotated, zone one: state, classifier, router

Now read like a reviewer. Click each annotated line; the goal, as always, is for every note to feel slightly unnecessary by the time you reach it.

triage_app.py — the brain
class TicketState(TypedDict, total=False):
The shared whiteboard, as a TypedDict (ch16). total=False means keys are allowed to be missing — and they are: a fresh ticket has only three fields, and each node fills in more as the run progresses. The schema is a contract about what may exist, not what must.
category: str # filled in by classify()
Read those trailing comments — they name which node writes each field. Real teams do exactly this, because in a graph app the state schema is the one page everyone must agree on. A reviewer checks these comments against the nodes like an auditor checks a ledger.
def classify(state: TicketState) -> dict:
A node is just a function: full state in, partial update out (ch58). No base class, no decorator, no framework ceremony. If you can read a Python function, you can read a LangGraph node.
category = llm.invoke(prompt).content.strip().lower()
The LLM call, plus two seatbelts. .content pulls the text out of the message object (ch52); .strip().lower() defends against the model replying "Refund." with a capital letter and a full stop. Production code is full of small distrust like this — the router downstream needs an exact match.
return {"category": category}
Nodes return only what changed — one key here — and LangGraph merges it into the state. Returning the whole state would work but invites accidental overwrites; returning the delta is the house style you will see everywhere.
def route_ticket(state: TicketState) -> Literal["handle_refund",
The router from ch59. It returns the name of a node, never an answer for the customer. The Literal type hint lists every destination — tooling uses it to draw the graph, and reviewers use it to see all possible branches at a glance.
if state["category"] == "refund":
The most important if in the file, and the LLM does not get a vote in it. The model produced the data (the category); this deterministic line makes the decision. Anything touching money routes through code you can unit-test.
builder.add_conditional_edges("classify", route_ticket)
The wiring: after classify, call route_ticket and jump to whichever node it names. Because the router returns real node names, no extra mapping dict is needed. This one line is the wall chart from the story.

Annotated, zone two: the gate, the checkpointer, the resume

Zone two is where this app earns its keep. Everything here exists so that a pause of three days costs the same as a pause of three seconds.

triage_app.py — the gate and the records room
amount = float(llm.invoke(prompt).content.strip())
A second LLM call extracts the amount. Note the naked float(...) — if the model replies "999 rupees", this raises ValueError. A reviewer would flag it and ask for structured output instead. Reading real code includes spotting what you would push back on.
decision = interrupt({
The gate from ch62. interrupt() stops the graph mid-node and hands this payload out to whoever is listening — a review dashboard, an approvals queue, a chat message to the duty manager. The run is now frozen, not failed.
"question": f"Approve refund of {amount} for {state['customer']}?",
Write interrupt payloads for the human who will read them. This dict becomes the text on someone's approval screen; a lazy payload like {"node": "handle_refund"} forces the approver to go read the code. Kindness here is an operational feature.
if decision == "approve":
On resume, interrupt() returns whatever value the human sent back. The node cannot tell the difference between a three-second pause and a three-day one — the code reads as if the answer arrived instantly.
issue_refund(state["customer"], amount) # side effect AFTER the gate
The only line in the whole app that touches the outside world with consequences — and it sits strictly behind the approval. The LLM never calls this function; it cannot. Your code does, and only after a human said yes. This placement is the security model.
checkpointer = InMemorySaver() # swap for Postgres in production
The records room (ch61). InMemorySaver keeps checkpoints in the process's memory — perfect for demos, gone on restart. Production swaps in a Postgres or Redis checkpointer from the langgraph-checkpoint-* packages; the rest of the file does not change by a single character.
app = builder.compile(checkpointer=checkpointer)
.compile() (ch60) turns the builder's wiring into a runnable app, and handing it the checkpointer is what upgrades interrupt() from "crash politely" to "pause durably".
config = {"configurable": {"thread_id": "ticket-4712"}}
One thread per ticket. The thread_id is the case-file number: it tells the checkpointer where to file this run's state, and later, where to find it again. Two tickets on two threads never see each other's state.
print(state["__interrupt__"]) # paused: waiting for a human
When a run pauses, invoke returns early and the result carries a special __interrupt__ entry holding the gate's payload. In a real service this is what you would render into the approvals queue instead of printing.
state = app.invoke(Command(resume="approve"), config)
The wake-up call. Same thread_id, but instead of new ticket data we pass Command(resume="approve"). LangGraph reloads the checkpoint, re-enters handle_refund, and this time interrupt() returns "approve". This line can run in a different process, on a different machine, days later — as long as it reaches the same checkpointer.
Careful

On resume, the paused node replays from its first line — LangGraph does not freeze Python mid-function; it re-runs the node and feeds the saved answer into interrupt(). So in this app the amount-extraction LLM call runs twice: once before the pause, once on resume. Harmless here, but if the code before an interrupt() sent an email or charged a card, it would happen twice. The rule reviewers enforce: side effects go after the gate, or in a separate node that runs before the gating node.

In the wild

Open any LangGraph repository and you will find this chapter's zones scattered across files instead of stacked in one: the state schema in state.py, nodes in nodes.py, wiring in graph.py, and a langgraph.json at the root if the team deploys to a managed platform. In merge requests, the phrases to recognise are "added a node for X" (a new desk), "route on Y" (a new stamp on the wall chart), "wired up the Postgres checkpointer" (the records room went from demo to durable), and "put an interrupt before the write" (someone installed a signature gate). When an incident review says "the run resumed after the deploy and double-sent the email", you now know exactly what happened: a side effect sat before an interrupt(), and the node replayed. You can read all of it now — and you can push back on the parts that deserve it.

Check yourself

1. A refund ticket pauses at the gate on Friday evening. Over the weekend the server is redeployed. On Monday the approver clicks approve and the run finishes correctly. What made that possible?

Nothing survives in process memory across a redeploy. The checkpointer wrote the state to storage after every step, filed under the thread_id — so any process that can reach that storage can pick the run up exactly where it paused.

2. What does route_ticket return?

Routers answer exactly one question: where does control go next? They return a node name as a string, and the conditional edge jumps there. The customer-facing answer is some node's job, never the router's.

3. A ticket arrives reading "My invoice looks wrong for last month." Which path does it take through the graph?

The classifier will stamp it "billing", the router's if only diverts "refund", so everything else flows to answer_with_docs — which finds KB-31 via the invoice keyword and drafts a reply. No gate, no pause.

4. Inside handle_refund, the line decision = interrupt({...}) eventually gives decision a value. Where does that value come from?

The payload travels outward, to the human's screen; the resume value travels inward, from the human. When the thread is re-invoked with Command(resume="approve"), the node replays and interrupt() returns that value.

5. Why must issue_refund(...) sit after the interrupt() call rather than before it?

LangGraph cannot freeze Python mid-function; resuming means re-running the node and feeding the saved answer into interrupt(). Code before the gate therefore runs on both visits — fine for reading state, dangerous for moving money.

Key takeaways

  • A full LangGraph app is five familiar parts assembled: a state schema, node functions, a router, edge wiring, and a compiled graph with a checkpointer. You had learned every one before this chapter began.
  • Nodes return partial updates and LangGraph merges them — the whole app communicates through one shared, checkpointed state.
  • Routers return node names, and the decisions that matter are plain if statements: the LLM fills in data, deterministic code steers control flow.
  • interrupt() plus Command(resume=...) make "wait for a human" a first-class, durable graph state — and the checkpointer plus thread_id are what let that pause survive restarts and redeploys.
  • On resume, a node replays from its first line — so real side effects belong after the gate, or in an earlier node. Reviewers check this placement the way auditors check signatures.
