The frontend: CopilotKit, generative UI, shared state

Rendering the stream with React, useCoAgent for bidirectional state, useCopilotAction for generative UI, and renderAndWaitForResponse for human-in-the-loop.

~28 minChapter 121 of 133
The story

Sit in the co-pilot seat of a modern airliner. The captain — quick, tireless, a little too eager — is flying. In front of you the instruments are alive: the altimeter ticks up as she climbs, a moving-map redraws the route, a weather panel blooms over the next waypoint the instant she looks at it. You are not reading a report she wrote an hour ago. You are watching her fly, live, dial by dial.

And the panel is not a one-way screen. When she proposes a course change to dodge a storm, a switch on your console lights amber and the aircraft waits — she will not bank until you flip it to confirm. You can also reach up mid-flight and nudge the target altitude yourself; on her next scan she sees your number and adjusts. Two hands on one machine, each seeing what the other does.

That cockpit is a good AG-UI frontend. The captain is the agent; the glass is CopilotKit, a React library; the live dials are the streamed events; the amber confirm-switch is human-in-the-loop; and the altitude you dial back is shared state flowing the other way. This chapter builds that glass.

Where we are: a stream with no windscreen yet

Quick recap of the chapter before this one. In building an AG-UI backend you wired a FastAPI endpoint that accepts a RunAgentInput and streams back a sequence of AG-UI events — RUN_STARTED, a flurry of TEXT_MESSAGE_CONTENT, some TOOL_CALL_*, a STATE_DELTA, then RUN_FINISHED — each one serialised over Server-Sent Events as a data: {json}\n\n line. You had a firehose of typed events leaving the server. What you did not have was anything catching them and turning them into pixels.

That catcher is the frontend, and its job is unglamorous but exact: consume the event stream, keep a little state machine in sync, and paint. You could hand-write that reducer in raw JavaScript — almost nobody does, because the reference frontend already exists and is very good.

CopilotKit: the React library that also authored the protocol

CopilotKit is an open-source React library for building agent-facing UIs — chat surfaces, in-app copilots, agent-driven dashboards. It matters here for one reason worth saying plainly: CopilotKit is the team that created AG-UI. The protocol was extracted from their product, not bolted on afterwards. So your frontend is the reference implementation of the standard, by the people who wrote the standard — unusually clean provenance.

Before any hook, there is a wrapper. The provider — a React component you put near the root of your app that hands shared context down to everything inside it — is <CopilotKit>. It needs two things: where the agent lives, and which agent to talk to.

App.tsx — wrapping the app in the provider
import { CopilotKit } from "@copilotkit/react-core";
CopilotKit ships as a few packages. react-core holds the provider and the hooks; a separate react-ui package holds prebuilt components like the chat panel.
import { CopilotChat } from "@copilotkit/react-ui";
 
export function App() {
return (
<CopilotKit runtimeUrl="/api/copilotkit" agent="flightAgent">
runtimeUrl is where the browser POSTs its RunAgentInput — the endpoint that fronts your AG-UI backend. agent="flightAgent" names which registered agent this UI drives, so one app can host several. Everything nested inside can now see the stream.
<FlightDeck />
Your own custom instruments — the moving map, the altimeter. These read agent state through hooks you will meet in a moment.
<CopilotChat />
The prebuilt chat surface. It subscribes to the run automatically and renders the streamed TEXT_MESSAGE_* tokens as they arrive, plus tool calls as they happen — a whole working chat with zero event-handling code from you. This is the "just give me a copilot" path.
</CopilotKit>
);
}

That is the whole "hello world". <CopilotChat /> already turns the event firehose into a scrolling conversation, streaming word by word, showing tool calls as they run. If all you want is a chatbot glued to an AG-UI agent, you are done. The rest of the chapter is the three things that make a copilot feel like a cockpit rather than a chat box: shared state, generative UI, and human-in-the-loop.

Note

CopilotKit is React-only. AG-UI the protocol is framework-agnostic — the wire format is just JSON over SSE, so a Vue, Svelte or plain-JS client could consume the same stream. But the batteries-included frontend this chapter teaches is React. That is why the runnable exercise later is a Python simulation of the reducer rather than the React itself — React cannot run in our in-browser Python sandbox.

Shared state: instruments that read the agent, and a dial you can turn back

The chat panel is one-directional narration. A cockpit is not: the dials reflect the agent's live state, and you can reach in and change that state. AG-UI carries this with two event types you met in the event model chapter — STATE_SNAPSHOT (here is the whole state object) and STATE_DELTA (here is a small JSON Patch to apply to it). CopilotKit exposes both directions through one hook.

useCoAgent is the hook that binds a React component to an agent's shared state. You give it the agent's name and an initial state; it gives you back the live state, a setter, and a running flag.

FlightDeck.tsx — bidirectional shared state
import { useCoAgent } from "@copilotkit/react-core";
 
const { state, setState, running } = useCoAgent({
Three things come back. state is the current shared object, re-rendered whenever the agent sends a snapshot or delta. setState pushes a change from the UI into the agent. running is true while a run is in flight — handy for spinners and disabling controls.
name: "flightAgent",
Must match the agent named on the provider — this is how the hook knows whose state to mirror.
initialState: { altitude: 30000, heading: 90, autopilot: true },
The shape the UI assumes before the first event lands, so the dials render sensibly on cold load. Once a STATE_SNAPSHOT arrives it is replaced by the real thing.
});
 
// AGENT -> UI: state updates itself as events stream in.
<Altimeter value={state.altitude} live={running} />
Read-only direction. When the agent emits STATE_DELTA with a patch on /altitude, CopilotKit applies it, state.altitude changes, and React re-renders the dial. You wrote no event listener — the hook did it.
 
// UI -> AGENT: your change rides along on the next turn.
<button onClick={() => setState({ ...state, altitude: 35000 })}>
Write direction. setState updates the local copy immediately and hands the new state to the agent as part of the next RunAgentInput, so on its next turn the agent sees that you dialled 35,000 and can act on it. This is the amber-lit dial you turned yourself.
Climb to FL350
</button>

As the agent climbs it sends a tiny STATE_DELTA (a JSON Patch) instead of the whole object. CopilotKit applies the patch and React re-renders only what changed.

On the next run the new state is packed into the RunAgentInput the browser POSTs. The agent reads it as fact — the human moved the target — and reasons from there.

Generative UI: when a tool call becomes a component

A chat that can only emit text is a teletype. The cockpit blooms a weather panel because the agent asked it to. That is generative UI — the agent, mid-run, causes a rich custom component to appear, rather than describing it in words. The mechanism rides on tool calls you already understand.

Recall from the ReAct chapter that an agent "does" things by emitting a tool call — a name plus arguments — which your code then executes. Generative UI adds a twist: some tools do not run backend logic at all. Their entire job is to render something on screen. When the agent streams TOOL_CALL_START for such a tool, CopilotKit does not look for a Python function — it looks for a React component you registered under that name, and mounts it with the tool's arguments as props.

useCopilotAction is the hook that registers one of these frontend actions. You declare a name, a description (written for the agent's model to read, exactly like a tool description), typed parameters, and a render function that returns JSX.

WeatherAction.tsx — a tool call that renders a panel
import { useCopilotAction } from "@copilotkit/react-core";
 
useCopilotAction({
name: "show_weather_card",
The tool name the agent will "call". To the agent this is just another tool in its list; it has no idea the result is pixels rather than data.
description: "Display a weather panel for a waypoint.",
Read by the model when it decides whether to call this. Write it for the agent, not for a human — same discipline as any tool description.
parameters: [
The arguments the agent must supply, with types. CopilotKit turns these into the JSON schema the model sees, so the agent knows to send a waypoint and a condition.
{ name: "waypoint", type: "string" },
{ name: "condition", type: "string" },
],
render: ({ args }) => (
The heart of generative UI. Instead of returning data, this returns a React element. The agent's arguments arrive as args; you spread them into your own component. When the tool call streams in, this panel appears.
<WeatherCard waypoint={args.waypoint} condition={args.condition} />
),
});

So "the agent shows the pilot the weather" is, under the hood, "the agent emits a show_weather_card tool call and the UI mounts <WeatherCard/>". No new protocol — generative UI is tool calling pointed at the screen instead of at a function. The same TOOL_CALL_* events you already stream do the whole job.

Human-in-the-loop: the tool call that waits for a hand

Now the amber switch. Sometimes the agent should not proceed until a human decides — approve this refund, confirm this reroute, pick which fabric. You met this exact idea in LangGraph as interrupt(): the graph pauses mid-run, surfaces a question, and resumes only when you pass a value back with Command(resume=…). AG-UI expresses the very same pattern, but at the UI layer, and CopilotKit gives it one small, memorable switch.

Take the generative-UI action from above and swap render for renderAndWaitForResponse. Everything else stays. The difference is enormous: render mounts a component and moves on; renderAndWaitForResponse mounts a component and pauses the run, handing your component a respond function. The agent's turn does not complete until your UI calls respond(value) — and whatever value you pass is delivered back to the agent as that tool's result.

RerouteApproval.tsx — human-in-the-loop with respond()
useCopilotAction({
name: "confirm_reroute",
description: "Ask the pilot to approve a new route before turning.",
parameters: [{ name: "reason", type: "string" }],
renderAndWaitForResponse: ({ args, respond, status }) => (
The one word that changes everything. Note the extra render prop: respond. The component now stays mounted — the run is parked — until you call it. status shows whether the ask is still pending or answered.
<ApprovalDialog
Your own dialog. It could be one Approve/Reject pair, or a whole form — a fabric picker, an edited paragraph, a corrected address. Anything the human returns becomes the tool result.
reason={args.reason}
onApprove={() => respond("approved")}
Calling respond("approved") does two things at once: it unmounts the wait (the run continues) and sends "approved" back as the result of the confirm_reroute tool. To the agent it looks exactly like any tool returning a value.
onReject={() => respond("rejected")} />
The other branch. The agent gets "rejected" and reasons accordingly — perhaps proposing an alternative. A real colleague can be told no.
),
});
Tip

Hold the symmetry in your head: LangGraph's interrupt() pauses a graph on the server and resumes on Command(resume=value); CopilotKit's renderAndWaitForResponse pauses a run at the UI and resumes on respond(value). Same shape — pause, ask a human, resume with their answer — expressed once for graph authors and once for interface builders. If your backend is a LangGraph agent, an interrupt() on the server can surface as a renderAndWaitForResponse dialog on the client. The two ends of the same wire.

Who owns this frontend standard

Provenance is simpler here than for the other two protocols in the trio. Where MCP came from Anthropic and A2A came from Google and moved to the Linux Foundation, AG-UI came from CopilotKit — a company whose whole product is agent frontends. They open-sourced the protocol their React library already spoke, published it at docs.ag-ui.com, and kept the library as the reference client. Because the same team owns both, spec and library move together: a new event type shows up as a new hook or render prop, not six months later. The flip side of single-vendor origin is the usual one — a protocol is only as neutral as its governance, and this one is younger and more centralised than A2A's foundation model.

An honest maturity check

Now the part the demo videos skip. The chat path — provider plus <CopilotChat /> — is genuinely mature; you can wire a streaming copilot to an agent in an afternoon. Shared state, generative UI and human-in-the-loop all work, and are the reason to reach for CopilotKit over hand-rolling a reducer.

But keep three truths in view. First, this is a React story. If your app is Vue, Svelte, SwiftUI or a terminal, you get the protocol but not the batteries — you consume the SSE stream and write your own reducer, which is real work. Second, AG-UI gives you the streaming contract, not an agent. It does not orchestrate, checkpoint, branch, or run a control loop — you bring that from LangGraph, a deep agent, or fifty lines of your own, and AG-UI only carries the events between it and the glass. Confusing "I adopted AG-UI" with "I have an agent architecture" is the classic mistake, and the next chapter is entirely about that boundary. Third, the ecosystem is young: the protocol reached its 1.0 shape recently, deprecated event families (the old THINKING_* set, replaced by REASONING_*) were removed along the way, and APIs still move. Learn the durable concepts — provider, shared state, generative UI, respond() — and check docs.copilotkit.ai for exact current prop names before you ship.

In the wild

You will meet this frontend as a @copilotkit/react-core line in a package.json, a <CopilotKit runtimeUrl=…> wrapper near the top of an app, and useCoAgent / useCopilotAction calls scattered through the components. In a merge request, "add a generative-UI card for the analytics tool" means a new useCopilotAction with a render; "gate the delete behind an approval" means switching that action to renderAndWaitForResponse. When a teammate says the copilot "hangs on the confirm dialog", you now know exactly what is happening: the run is parked in a renderAndWaitForResponse waiting for a respond() that never fired — the amber switch nobody flipped.

Check yourself

1. What does useCoAgent give you that a plain chat panel does not?

The chat panel narrates one way. useCoAgent binds a shared state object that flows both directions: deltas from the agent re-render your dials, and setState packs your change into the next RunAgentInput so the agent sees it. Two hands, one machine.

2. The agent emits a show_weather_card tool call, but there is no backend function by that name. What happens?

Generative UI is tool calling pointed at the screen. A useCopilotAction with a render function registers a frontend "tool" whose result is a mounted component, not backend data. The agent does not know or care that the result is pixels.

3. You change render to renderAndWaitForResponse on an action. What is different?

render mounts and moves on; renderAndWaitForResponse mounts and freezes the run, handing your component a respond prop. The run resumes only when respond(value) fires, and that value becomes the tool's result — the human-in-the-loop switch.

4. How does CopilotKit's renderAndWaitForResponse relate to LangGraph's interrupt()?

Both are pause-ask-resume. LangGraph interrupts on the server and resumes with Command(resume=value); CopilotKit pauses at the interface and resumes with respond(value). A server-side interrupt() can even surface as a client-side renderAndWaitForResponse dialog — two ends of one wire.

5. A colleague says "we adopted AG-UI, so our agent architecture is handled." What is the honest correction?

AG-UI standardises how an agent streams to a frontend and how the UI answers back — no more. The reasoning engine is still yours to bring. Mistaking the transport for the architecture is exactly the trap the next chapter unpacks.

Key takeaways

  • CopilotKit is the React library that consumes AG-UI — and the team that created the protocol, so the library and the spec move as one. Wrap the app in <CopilotKit runtimeUrl agent> and drop in <CopilotChat /> for a streaming copilot with zero event-handling code.
  • Shared state is bidirectional through useCoAgent: agent STATE_SNAPSHOT/STATE_DELTA events update state and re-render, while setState pushes UI changes into the agent's next turn — the same object synced from both ends.
  • Generative UI is tool calling aimed at the screen: useCopilotAction with a render function mounts a custom React component from an agent tool call, using its arguments as props.
  • Human-in-the-loop is one word away — swap render for renderAndWaitForResponse: the component stays mounted, the run pauses, and respond(value) both resumes the run and returns that value to the agent as the tool result. It is LangGraph's interrupt() expressed at the UI layer.
  • Be honest about the edges: CopilotKit's batteries are React-only, and AG-UI is the streaming contract, not an agent — orchestration, checkpointing and the loop still come from your backend. The concepts are durable; check current docs for exact prop names.
