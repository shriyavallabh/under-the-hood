Variables: names stuck on objects

One picture — sticky notes stuck on boxes — and suddenly assignment, copying, and some of the strangest bug reports in Python all make perfect sense.

~25 minChapter 2 of 117
The story

A shared storeroom in an office building. Cardboard boxes on steel shelves, and a pad of sticky notes by the door. The facilities manager packs fairy lights and lanterns into a box and slaps a note on it: FESTIVE. A month later the events team needs supplies for a party. They find the same box, decide it is exactly what they need, and stick their own note next to the first one: PARTY-KIT. They take the fairy lights out, use them, lose half of them, and drop a packet of balloons in.

Come the festival, the facilities manager pulls down the box labelled FESTIVE and finds balloons where his lights should be. He is furious: "Nobody touched my box!" And he is right — nobody touched his note. But there was never a "his" box. There was one box, two notes, and a change made through either note shows up under both. Hold that storeroom in your head. It is, honestly, the whole chapter.

A name is a sticky note, not a box

In the previous chapter you learned to read Python's shape — indentation shows structure, colons open blocks. Now the first sentence of meaning: what the humble = actually does.

Most people arrive with a picture from school or from other languages: a variable is a little box with a name engraved on the side, and assignment pours a value into it. In Python that picture is wrong, and the wrongness costs real money in production bugs. Here is the true picture. An object is a value living in memory — a number, a piece of text, a list — think of it as a box sitting on Python's shelf. A name (what people casually call a variable) is just a sticky note. And an assignment like x = 5 is a two-step act called binding: first Python builds or finds the object on the right-hand side, then it sticks the note x onto that box. The note goes on the box; nothing is ever poured into anything.

Three consequences fall straight out of the sticky-note picture, and each one will earn its keep before this page is done:

  • Notes are free and boxes are real. You can stick as many notes on one box as you like. Two names, one object — our storeroom exactly.
  • The note has no type; the box does. That is why the same name can label a number this line and a list the next. Python does not mind; the note simply moved.
  • Reading a variable means following its note. When you write print(x), Python walks to whatever box x is currently stuck on and looks inside. The note itself holds nothing.

Assignment moves notes; mutation opens boxes

Steps 3 and 5 above are the two fundamentally different kinds of change in Python, and telling them apart is the skill this chapter exists to give you.

Rebinding is what plain assignment does: x = something peels the note x off whatever it was stuck on and sticks it on a new box. The old box is untouched — and if no notes remain on it at all, Python's housekeeping quietly discards it (that cleanup is called garbage collection, and you get it for free). Mutation is changing the contents of an existing box: orders.append("chai"), config["retries"] = 3, team.remove("intern"). No note moves; the box changes — and every name stuck on that box sees the new contents instantly.

Happily, the syntax usually tells you which is which. A bare name on the left of = is rebinding. A dot-method that changes something, or square brackets on the left of =, is mutation — you are reaching through a note into a box.

There is one more twist: some boxes are factory-sealed. An immutable object is one that cannot be changed after it is built — there is no way to open the box, ever. A mutable object can be opened and modified freely.

This is why x = x + 1 never surprises anyone: integers are sealed, so "adding one" can only mean building a new box holding the result and moving x's note to it. Whoever else had a note on the old 5 still sees 5. Strings work the same way — every "change" to text quietly builds new text, which matters enormously for prompt-building and is where the next chapter picks up. All the drama in this chapter comes from the openable column: lists, dicts, and sets are the boxes that can change under your feet.

Careful

Two lines that look like twins behave differently on lists: x = x + [4] builds a brand-new list and moves the note, while x += [4] opens the existing box and extends it in place. If x was the only note on the box, you cannot tell the difference. If the box was shared, one of these changes what your colleagues see and the other does not. When it matters, say what you mean explicitly.

Two notes, one box: why b = a then b.append() surprises people

Now you can name the storeroom incident precisely. When two names are stuck on the same box, each is an alias of the other — alternative names for one object. And the single most-reported "bug" in beginner Python is aliasing working exactly as designed:

the classic surprise
a = ["idli", "dosa"]
b = a              # feels like "make me a copy"...
b.append("vada")   # ...but this opens the ONE shared box
print(a)           # ['idli', 'dosa', 'vada']  -- "who changed a?!"

Nobody changed a. a is a sticky note; notes do not have contents. The box changed, and a happens to be stuck on it. The line that misleads people is b = a, because in everyday English "b equals a" sounds like duplication. In Python it is never duplication — assignment copies notes, full stop. Why was the language designed this way? Because boxes can be enormous — a list of a million log lines, a dict holding an entire model configuration — and silently duplicating them on every = would be slow and wasteful. Python refuses to guess. Notes are free; copies are something you must ask for by name:

asking for a real copy
b = a.copy()      # a new box with the same contents
b = list(a)       # same thing, different spelling
b = a[:]          # same thing again, older spelling

One honest caveat: these are shallow copies — Python builds a new outer box and puts notes to the same inner objects inside it. Copy a list of dicts this way and the list is yours, but the dicts inside are still shared. When you need the whole nested structure duplicated, boxes all the way down, the standard library has copy.deepcopy().

Aliasing also hides in a place people rarely think to look: function calls. When you call process(orders), Python does not ship a copy of your list into the function — it sticks the function's parameter name onto your box. If the function mutates it, your data changes. Sometimes that is the entire point: in the agent loop you will build later, history.append(reply) deliberately mutates one shared transcript that the whole run keeps re-reading. Deliberate sharing is a feature. Accidental sharing is a bug report.

is vs ==: same contents, or same box?

Once you know one box can wear many notes, an obvious question follows: given two names, how do you find out whether you are looking at one box or two? Python gives you two different questions to ask, and they are not the same question.

== asks: do these boxes hold equal contents? Two separate shopping lists that both read "milk, bread" are equal. is asks: are these literally the same box? — the same object, the same shelf position. That sameness is called identity, and Python exposes it through id(), a built-in that returns the box's serial number: a number guaranteed unique to that object for as long as it lives. (In CPython, the standard interpreter, the serial happens to be a memory address — but treat it as an opaque serial. All that matters is: same id, same box.) a is b is simply shorthand for "do a and b have the same serial?"

The professional rule of thumb is short: use == for comparing values, and reserve is for one job — checking for None, as in if response is None:. There is exactly one None box in the whole program, so identity is the correct and idiomatic test for it.

Careful

Never use is to compare numbers or strings. As a speed trick, Python pre-builds one shared box for each small integer (-5 through 256) and reuses it everywhere, so is happens to say True for some numbers and False for others depending on size and how they were made. Code that relies on this passes tests with small values and fails mysteriously with big ones. == is what you mean.

The bug you will actually meet

Here is the shape this chapter takes when it costs someone an afternoon. A module keeps a default list of tools; a factory function builds agents; and one innocent-looking line quietly welds every agent — and the default itself — to a single shared box. Read it like a reviewer would: click each annotated line and make sure the note matches what you now know.

agent_setup.py — spot the shared box
DEFAULT_TOOLS = ["calculator"]
A module-level list: ONE box, built once when the file loads. Every part of the program that touches this name is touching the same box.
def make_agent(name, tools=None):
A factory that builds an agent. Callers may pass their own tool list; if they do not, we fall back to the defaults. So far, reasonable.
if tools is None:
The one blessed use of is — there is a single None box in the whole program, so identity is the right test for "the caller gave us nothing".
tools = DEFAULT_TOOLS
THE BUG. This reads as "give me my own set of defaults", but it only sticks the local note tools onto the one shared box. No copy exists anywhere. The fix is to ask for one: tools = list(DEFAULT_TOOLS).
tools.append("web_search")
Mutation — opens whatever box tools is stuck on. When the bug line ran, that box IS DEFAULT_TOOLS. The first call pollutes the module's defaults; every later call inherits the leftovers and adds more.
return {"name": name, "tools": tools}
The returned dict carries yet another note to the same box. After two calls there are notes everywhere — DEFAULT_TOOLS, both agents' "tools" entries — all stuck on one list.
support = make_agent("support-bot")
Looks fine afterwards: tools are ['calculator', 'web_search']. The damage is invisible — the defaults box now contains the extra tool.
billing = make_agent("billing-bot")
The second victim starts from the polluted box and appends again: ['calculator', 'web_search', 'web_search']. Worse — because it is one box, support's tools silently changed too. By agent fifty, the list is absurd.

Note

A close cousin of this bug is writing the default straight into the signature: def make_agent(name, tools=[]). That single list box is built once, when the function is defined, and then shared by every call that omits the argument. Linters flag it as the "mutable default argument" trap, and the standard cure is exactly the pattern above: default to None, then build a fresh box inside the function.

In the wild

Once you see sticky notes, you see them everywhere. Code reviews are sprinkled with list(items), dict(config), .copy(), and copy.deepcopy(state) — all of it is note-versus-box management, someone deliberately paying for a new box so a caller's data cannot change under them. Linters ship a dedicated rule for the mutable-default trap you just fixed. Agent frameworks care deeply too: LangGraph's state updates work by returning new values for the graph to merge rather than mutating shared state in place, precisely so that checkpointing and replay stay trustworthy. And when a teammate reports "my prompt template changed between runs and nobody edited the file" — you now know the first question to ask: how many notes are stuck on that box?

Check yourself

1. You run a = ["invoice.pdf"], then b = a, then b.append("receipt.pdf"). What does a hold now?

b = a copies the note, never the box. The append opened the single shared list, and both names see the new contents.

2. Which line changes an existing object, rather than moving a name to a (possibly new) object?

Both assignments are rebinding — they stick the note x on a box. Only .append is mutation: it opens the box orders is stuck on and changes its contents for every alias.

3. a == b is True but a is b is False. What do you know for certain?

== compares contents; is compares identity (the serial from id()). Equal-but-not-identical means two separate boxes that happen to hold matching values — mutate one and the other stays put.

4. You want team_b to start with the same members as team_a but grow independently from now on. Which line is right?

Copies must be asked for by name: .copy() (or list(team_a)) builds a new box with the same contents. Plain assignment would alias, and option b is a question, not an action — it just returns True or False.

5. A colleague's function is def add_tool(tools): tools.append("search") and you call it with your own list. After the call, your list...

Calling a function binds the parameter name to the very object you passed — no copy is made. The function opened your box. Deliberate sharing like this powers agent history lists; accidental sharing powers bug reports.

Key takeaways

  • A variable is a sticky note on an object, never a container. Assignment builds or finds the box first, then attaches the name — nothing is ever poured into anything.
  • b = a copies the note, not the box. Two names, one object — and any mutation through either name is visible through both.
  • Rebinding (x = ...) moves a note; mutation (.append, key assignment) opens a box. Immutable objects (int, str, tuple) are sealed boxes and never surprise you; lists, dicts, and sets are where aliasing bites.
  • == asks "equal contents?"; is asks "same box?" (same id() serial). Reserve is for None — never for numbers or strings.
  • Copies must be requested explicitly: a.copy(), list(a), or copy.deepcopy() for nested structures. Function arguments are notes on your boxes, so functions can mutate what you pass them.
  • When data "changes by itself" in any Python system — an agent's tool list, a prompt template, a config dict — the first suspect is always two notes on one box.
