Compile, invoke, stream

A blueprint cannot wash a car. Three method calls take your graph from drawing to running machine — and the one you choose decides whether you see only the finished result or every station along the way.

~25 minChapter 61 of 133
The story

A brand-new automated car wash, the night before opening day. For weeks it has existed only as a drawing: foam station, brushes, rinse, dry, and one junction near the exit — still dirty? loop back to the brushes. Tonight the commissioning inspector walks the tunnel with the drawing in her hand. Does every conveyor lead somewhere real? Is the junction's lever wired to actual track? Does the entrance connect at all? She washes nothing. She signs the certificate, flips the master switch, and hands over a machine you can finally use.

Next morning, two customers. The first drops his keys at the counter, goes for coffee, and collects a gleaming car twenty minutes later — he wants the end result and nothing else. The second stands at the long glass wall beside the tunnel, watching her car change at every station: foam on, foam off, water sheeting away. Same tunnel, same wash — two different ways of consuming it. And stapled to every order is a job ticket carrying the manager's one iron rule: maximum 25 station visits per car, so a hopeless wreck can never circle the brushes forever.

The inspection is .compile(). The keys-drop customer is .invoke(). The glass wall is .stream(). The job ticket is the config dict — and the manager's rule is recursion_limit.

Two chapters of blueprint, zero actual runs

Quick recap, because this chapter is where the previous two pay off. StateGraph anatomy gave you the three parts of a LangGraph app: a state schema (the shared clipboard), nodes (functions that do work and return updates), and edges (the wiring from START to END). Conditional edges added junctions — routing functions that read the state and pick the next stop — and rebuilt the ReAct loop as a two-node graph with one signal box. But notice what never happened: nothing ran. Every call you made — add_node, add_edge, add_conditional_edges — was a call on the builder, and the builder is a drawing. You cannot hand a drawing a question.

That separation is deliberate. LangGraph splits the life of a graph into a build phase (declare the stations and the track) and a run phase (send actual state through it), with exactly one doorway between them. The doorway is .compile().

.compile(): the commissioning inspection

Despite the heavyweight name, builder.compile() does not translate anything to machine code, and it takes milliseconds. It does two jobs. First, the inspection: it walks your wiring and fails loudly, right now, on structural mistakes — an edge pointing at a node you never added, a junction mapping that names a destination that does not exist, no entrypoint from START. These errors surface before a single model is called and before a single rupee is spent, which is exactly where you want them. Second, the handover: it returns a new object — a compiled graph — that is runnable, meaning it speaks the standard run-me interface: .invoke() and .stream(). You met these verbs in messages and models — a compiled graph answers to the same interface a chat model does, and LangChain leans on that symmetry everywhere.

the doorway between phases
graph = builder.compile()   # inspect the wiring, hand over the machine

# The builder was a drawing. `graph` is a machine:
# it has .invoke(), .stream(), and friends.
# No node has run yet. No model has been called. Nothing has cost anything.

Two small things worth knowing about the handover. The compiled graph can draw its own map — graph.get_graph().draw_mermaid() prints diagram text you can paste into any Mermaid viewer, which is how the tidy graph pictures in documentation get made. And compile() takes optional arguments, the most important being checkpointer= — the plug for persistence, which is the entire subject of the next chapter. Today we compile bare.

Both verbs take a config dict as their second argument. Today's key: recursion_limit — the cap on steps per run before LangGraph blows the whistle. In LangGraph 1.x the built-in default is enormous (~10,000, up from 25 in the old 0.x line), so any serious run sets its own tight cap.

.invoke(): drop the keys, collect the car

graph.invoke(state) is the keys-drop customer. You hand in an initial state — usually just the fields you care to set, like one user message — and the call blocks while the entire run happens inside: every node, every junction decision, every trip around the cycle. When the graph routes to END, the call returns. What you get back is not "the answer". It is the final state — the whole clipboard, shaped exactly like your schema, after every node has finished writing to it:

what .invoke() returns
final = graph.invoke(
    {"messages": [{"role": "user", "content": "20% GST on 1500?"}]}
)

# `final` is the FINAL STATE -- the full schema-shaped dict:
#
# {"messages": [
#     HumanMessage("20% GST on 1500?"),
#     AIMessage("", tool_calls=[{"name": "calculate", ...}]),
#     ToolMessage("300.0"),
#     AIMessage("GST at 20% on 1,500 rupees is 300 rupees."),
# ]}

print(final["messages"][-1].content)   # pluck the answer yourself

Read that return value slowly, because it is the run's whole biography: the question, the model's tool call, the tool's result, the final answer — every message the run produced, in order, as real message objects (your plain dict was coerced into a HumanMessage by the add_messages reducer from ch058). Any other schema fields — a counter, a list of documents — would be there too, at their final values. The answer your user wants is just the last message's content, and plucking it out is your job, not LangGraph's.

Note

Blocking is the price of convenience. A serious agent run can take thirty seconds of model calls, and .invoke() gives you silence the whole way — nothing to show a waiting user. Fine for a batch job or a test; miserable for a chat interface. That misery is what the second verb fixes.

.stream(): the glass wall

graph.stream(state) runs the very same tunnel but returns a generator — remember from generators and yield: a function that hands you values one at a time, pausing in between, instead of one big result at the end. You consume it with a for-loop and receive a chunk after every step — one step being, in a straight-line graph, one node finishing its work. What each chunk contains is your choice, made with the stream_mode argument. Two modes carry almost all real usage:

stream_mode="values" gives you the whole car after each station: a full snapshot of the state as it stands after that step. Each chunk looks exactly like .invoke()'s return value, just earlier and still growing.

stream_mode="updates" gives you the station's work order instead: a small dict of {node_name: what_that_node_returned} — nothing but the delta. This is the mode you reach for when you want to see the graph think: which node ran, in what order, and what each one changed. (It is the default if you pass nothing — but production code names the mode explicitly.)

the same run through both lenses
inputs = {"messages": [{"role": "user", "content": "20% GST on 1500?"}]}

for chunk in graph.stream(inputs, stream_mode="updates"):
    print(chunk)
# {"agent": {"messages": [AIMessage("", tool_calls=[...])]}}
# {"tools": {"messages": [ToolMessage("300.0")]}}
# {"agent": {"messages": [AIMessage("GST ... 300 rupees.")]}}

for chunk in graph.stream(inputs, stream_mode="values"):
    print(len(chunk["messages"]), "messages so far")
# 1 messages so far    (the input state, before any node runs)
# 2 messages so far    (after agent: + the tool-call reply)
# 3 messages so far    (after tools: + the tool result)
# 4 messages so far    (after agent: + the final answer)

Two more modes deserve a nod. stream_mode="messages" streams token-by-token output from the LLM calls inside your nodes — the word-at-a-time effect from streaming responses, piped out of a whole graph; it is how LangGraph chat UIs type their answers live. Ask for several modes at once by passing a list — stream_mode=["updates", "messages"] — and each chunk arrives as a (mode, data) tuple so you can tell them apart. Both verbs also have async twins, ainvoke and astream, for the async world.

And here is the punchline that makes the whole API click: .invoke() is not a separate engine. In LangGraph's own source, invoke simply consumes the values stream and keeps the last chunk. The keys-drop customer's car went through the identical glass-walled tunnel — he just chose not to look. One engine, two ways of watching it.

The job ticket: config and recursion_limit

Both verbs accept a second argument: config, a plain dict of settings about this run — as opposed to state, the data inside it. Two compartments matter. "configurable" is a free-form bag of run-scoped values; its star tenant is thread_id, the key that gives a run a durable identity — the doorway to checkpointing, shut for one more chapter. Today's compartment is "recursion_limit", sitting at the top level of the dict: the cap on how many steps a run may take. Mind the version here: LangGraph 0.x shipped a deliberately modest default of 25, but LangGraph 1.x raised it to a very loose ~10,000 (env-overridable via LANGGRAPH_DEFAULT_RECURSION_LIMIT) — a backstop so wide a runaway cycle would rack up thousands of steps, and a real bill, before it ever fired. Which is why serious runs set their own tight cap explicitly rather than leaning on the default.

setting your own cap for an agent run
config = {"recursion_limit": 50}          # top level, NOT inside configurable

final = graph.invoke(inputs, config)      # config is the second argument
# or:
for chunk in graph.stream(inputs, config, stream_mode="updates"):
    ...

Why does a graph even need this? Because ch059 taught you to build cycles on purpose — agent → tools → agent — and a cycle plus a router that never picks END equals a machine that runs, and bills you, forever. recursion_limit is max_turns from your hand-built loop reborn at the engine level: when a run exceeds the cap, LangGraph raises a GraphRecursionError whose message you will eventually meet in a log — "Recursion limit of 50 reached without hitting a stop condition." Each pass around the ReAct cycle costs two steps (agent, then tools), so a cap of 50 allows roughly two dozen tool rounds before the whistle. Plenty for a healthy agent; a mercy for a stuck one.

Careful

The classic mistake: tucking it into the bag, as {"configurable": {"recursion_limit": 100}}. The configurable compartment accepts any key without complaint, so LangGraph will not raise an error — the setting is simply ignored and the default cap still applies. If a changed limit seems to change nothing, check which compartment it landed in.

Build all three verbs yourself

The toy graph engine from ch058 and ch059 grows up today: its bare run() function becomes a proper compile() that inspects the wiring, and the object it hands back offers real .invoke() and .stream() methods — with stream_mode, a config dict, and a working recursion limit. The graph inside is the same two-node ReAct graph, with the same scripted fake_llm standing in for a paid API. Watch for one method above all: invoke is four lines long, because it is just the stream, drained. Press Run.

Tip

Break it three ways. Change the junction mapping's "done" destination to "finish" and re-run — the compile-time inspection catches the unknown destination before anything runs, which is the whole argument for a compile step. Put it back, then switch stream_mode="updates" to "values" in the loop and print len(chunk["messages"]) to watch the snapshot grow. Finally, make should_continue always return "act" — an agent that never finishes — and watch the whistle end what would otherwise loop forever.

Note

Two honest simplifications, in the same spirit as ch059's: real LangGraph's values mode also emits the input state as a first chunk before any node runs (our toy starts at step one), and its step counter counts "super-steps", so nodes running in parallel count as one step together. Neither wrinkle changes the mental model.

Reading a run like a log file

Here is why the updates mode earns a section of its own: in production you will rarely watch an agent live. You will read, after the fact, what somebody's stream() loop — or a tracing tool wired to the same events — printed into the logs. An updates-mode trace is the graph's execution log, and reading one is a skill. A healthy run of our graph logs like this:

a healthy trace — three steps, clean exit
{"agent": {"messages": [AIMessage("", tool_calls=[{"name": "calculate", ...}])]}}
{"tools": {"messages": [ToolMessage("300.0")]}}
{"agent": {"messages": [AIMessage("GST at 20% on 1,500 rupees is 300 rupees.")]}}

Read it in two passes. First pass, keys only: agent, tools, agent — that is the route the state took, and it should match the drawing in your head (in, around the cycle once, out). Second pass, values: what did each node actually write? The agent's first entry carries a tool call, the tools entry carries a result, the agent's last entry carries plain text — which is precisely the condition that made the router choose END. Now compare with a sick run:

a sick trace — the same pair, forever
{"agent": {"messages": [AIMessage("", tool_calls=[{"name": "search", "args": {"q": "gst rate"}}])]}}
{"tools": {"messages": [ToolMessage("no results")]}}
{"agent": {"messages": [AIMessage("", tool_calls=[{"name": "search", "args": {"q": "gst rate"}}])]}}
{"tools": {"messages": [ToolMessage("no results")]}}
# ... the identical pair, ten more times ...
# GraphRecursionError: Recursion limit of 25 reached
#     without hitting a stop condition.

The route never varies and the payloads never change: the model asks the same question, gets the same unhelpful answer, and learns nothing that would make the router pick END. The GraphRecursionError at the bottom is the symptom; the repeating pair above it is the disease. Raising the limit to 100 would buy seventy-five more identical steps and a bigger bill. The real fixes live where the loop lives: a tool that returns something more useful than "no results", a prompt that tells the model what to do when a search comes back empty, or a router that counts attempts and bails out. Notice that you diagnosed all of this from node names and deltas alone — which is why experienced reviewers ask for the updates trace before they ask for anything else.

In the wild

Search any LangGraph service for .compile( and you will find the seam where the app is assembled — usually once, at startup, with the compiled graph reused across requests. Around it: graph.invoke(...) inside API handlers and batch jobs, async for chunk in graph.astream(..., stream_mode="messages") inside anything with a chat window, and updates-mode traces feeding the log pipeline. In merge requests you will meet "bump the recursion limit" (someone's cycle outgrew its cap — sometimes legitimate, sometimes the sick trace above) and "stream this instead of invoking" (a user-facing path that sat silent for thirty seconds). And when an incident channel pastes "Recursion limit of 25 reached without hitting a stop condition", you now know to scroll up from the error to find the repeating pair that caused it.

Check yourself

1. You run graph = builder.compile(). What has happened?

compile() is the commissioning inspection plus handover: structural checks (unknown destinations, missing entrypoint) and a runnable compiled graph in return. It costs milliseconds and zero tokens — running anything is what invoke and stream are for.

2. What does graph.invoke(inputs) hand back?

invoke() blocks until the run routes to END, then returns the whole clipboard — for a messages schema, the entire conversation including tool calls and results. The answer is its last message, and extracting it is your job.

3. A log line reads {"tools": {"messages": [ToolMessage("300.0")]}}. Which stream mode produced it, and what does it show?

The {node_name: delta} shape is updates mode's signature. A values chunk would be the whole state (every message so far, all fields); a messages chunk would be token-level LLM output.

4. You pass config={"configurable": {"recursion_limit": 100}} to cap a runaway agent, yet the run blows straight past 100 steps. Why?

configurable accepts any key without complaint, so the misplaced setting raises no error and does nothing. The correct shape is config={"recursion_limit": 100} at the top level, passed as the second argument to invoke or stream.

5. A trace shows the identical agent-tools pair repeating twelve times, then GraphRecursionError. The right response?

Unchanging payloads mean the model learns nothing new each lap, so the exit condition can never trigger. More steps just means more identical laps at higher cost. The engine did its job — the whistle blew; the loop's contents are yours to fix.

Key takeaways

  • A StateGraph builder is a drawing; .compile() is the commissioning inspection that validates the wiring (unknown destinations, missing entrypoint) and hands back a runnable graph — in milliseconds, with no nodes run and no models called.
  • .invoke(state) is one blocking call: the whole run happens inside, and you get the final state back — the full schema-shaped dict, from which you pluck the answer yourself.
  • .stream(state) is a generator over the same run: stream_mode="values" yields a full state snapshot after each step, "updates" yields just {node: what_it_returned}, and "messages" yields token-level LLM output. invoke() is literally the values stream, drained to its last chunk.
  • The config dict rides along as the second argument: recursion_limit at the top level (LangGraph 1.x defaults to a very loose ~10,000 steps — up from 25 in 0.x — so set your own tight cap; exceed it and GraphRecursionError fires), and the configurable bag for run-scoped values like thread_id — next chapter's key.
  • An updates-mode trace is your execution log: the keys are the route, the values are the work, and an identical repeating pair is a stuck cycle — fix the loop's contents, not the limit.
  • ainvoke and astream are the async twins; same behaviour, awaitable — the shape every production chat service actually uses.
