Tools, resources, prompts — and tasks

An MCP server can offer your agent three classic kinds of gifts — and the spec keeps adding new ones. The trick to never mixing them up is a single question: who pulls the trigger?

~25 minChapter 77 of 123
The story

You run a travel agency. Out front sits a brilliant junior assistant who plans trips; behind a door marked Back Office sit three counters. The first is the ticketing desk — check a fare, hold a seat, issue a refund. The assistant walks over whenever the trip demands it; nobody tells her when. The second is a filing cabinet of brochures, fare rules, and past itineraries. The assistant never rummages in it — the office manager decides which folders land on her desk before each meeting. The third is a laminated menu bolted to the customer-facing counter: "Honeymoon package", "Visa checklist", "Senior-citizen pilgrimage". Customers point at one to start a well-worn kind of request.

One afternoon a customer asks for a round-the-world itinerary — far too big to wait for. The ticketing desk hands over a numbered token: "Come back later, we'll keep working." Halfway through, a runner jogs out from the back office with a question for the customer: "Window or aisle?"

That back office is an MCP server. The three counters are its three classic offerings, the token is a task, and the runner is elicitation. This chapter is a tour of all five.

Fifteen seconds of recap

Two chapters ago you saw why MCP exists: without a standard, every agent product must hand-build an integration to every system — the M-by-N explosion. In MCP architecture you met the cast: a host application (Claude Code, an IDE, your harness) runs one client per server, they speak JSON-RPC over stdio or streamable HTTP, and the conversation opens with an initialize handshake where each side declares its capabilities. So far, so much plumbing.

This chapter answers the obvious next question: once the handshake is done, what is actually on offer? What can a server put on the table for your agent?

The sorting question: who pulls the trigger?

A server's three classic offerings — tools, resources, prompts — confuse newcomers because all three ultimately deliver the same substance: text and data that end up in the model's context. The difference is not what they carry. The difference is who decides when they are used. The spec even names them this way: tools are model-controlled, resources are application-controlled, prompts are user-controlled. Hold on to that axis and everything in the MCP universe files itself neatly.

Flip these until the control column feels automatic:

Tools: the model decides

A tool is an action the server is willing to perform when asked — in plain words, a function with a name, a description, and a schema for its arguments, executed on the server's side of the wall. This should feel like home turf: it is the same handshake you met in native tool calling and hand-rolled in the ReAct loop. The only novelty is where the function lives. In Act V your tools were functions in your own process; an MCP tool lives inside somebody else's server, and your host discovers it at runtime by calling tools/list.

What comes back from that call is worth reading slowly, because it is the join point between two worlds — click the annotated lines:

Tools are model-controlled: once the host has translated this list into the model's native tool-calling format, the model chooses when to invoke one, mid-reasoning, without asking anyone. That autonomy is their power and their danger — which is why tool calls are also where approval prompts, allowlists, and the whole of MCP security concentrate.

Resources: the application decides

A resource is a piece of readable context the server exposes under a URI — a file, a database schema, a log, a policy document — meaning data you can look at, addressed like a web page. resources/list enumerates them; resources/read with a URI like billing://policies/gst or file:///project/README.md returns the content. Servers can also publish resource templates (URI patterns with blanks to fill, such as invoice://{id}) and let clients subscribe to a resource so they are notified when it changes.

Here is the part people miss: the model does not decide to read a resource. Resources are application-controlled — the host, or the human driving it, chooses which ones to attach to the conversation. Think of the office manager placing folders on the assistant's desk: reading them costs attention (and, as the token bill chapter drummed in, tokens are money), so the party that curates context is the app, not the model. Resources are also expected to be safe to read — fetching one should not send an email or delete a row. Context in, no side effects out.

Note

"But my file server exposes read_file as a tool!" — yes, many do, and that is a deliberate design choice, not a mistake. Publishing it as a tool hands the read decision to the model; publishing it as a resource keeps that decision with the app. Same data, different trigger-puller. Choosing between them is the design act.

Prompts: the user decides

A prompt is a reusable message template the server offers to humans — a pre-written, parameterised way to start a task that the server's author knows their system handles well. prompts/list advertises them with names, descriptions, and argument lists; prompts/get fills in the arguments and returns ready-to-send messages. A code-review server might ship a review_merge_request prompt; a billing server, a monthly_report prompt taking a month argument.

Prompts are user-controlled: they surface in the host's interface as slash commands or menu entries, and a human picks one. If you have ever typed a slash command in Claude Code that came from an MCP server, you have used this primitive — the server defined the template, the client rendered it as a command, and you pulled the trigger. It is the laminated menu: the expertise of the person who built the back office, packaged so customers ask for the right thing in the right words.

Before we run all three, here is how little code they cost a server author in the official Python SDK — a teaser for the next chapter, where you will build and wire one of these for real:

server.py — three counters, three decorators
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("billing")

@mcp.tool()
def get_invoice_total(invoice_id: str) -> dict:
    """Return the total of an invoice, by id."""
    return {"invoice_id": invoice_id, "total": 1500, "currency": "INR"}

@mcp.resource("billing://policies/gst")
def gst_policy() -> str:
    """The current GST policy text."""
    return "GST is applied at 18 percent for services."

@mcp.prompt()
def monthly_report(month: str) -> str:
    """Draft the monthly billing report."""
    return f"Write the monthly billing report for {month}. Flag anything overdue."

Why tools hog 90 percent of the attention

Read about MCP anywhere and you will mostly read about tools. Browse real servers and many expose only tools. The skew is real, and it has mechanical reasons, not fashion:

Tools plug into a socket that already exists. Every agent host is already a tool-calling loop — the one you built by hand in Act V and met again in LangChain's create_agent. Supporting MCP tools costs a host almost nothing: fetch tools/list, translate the schemas into the model's native tool format, route tools/call. Resources and prompts, by contrast, need product surface — a resource picker, an @-mention affordance, a slash-command menu. Each client must design and build that UI, and many simply have not.

Tools can impersonate the other two. A server author in a hurry can wrap any document in a get_document tool and any template in a run_playbook tool. The reverse is not true. So the path of least resistance funnels everything into tools — even when handing the model control of what enters the context is exactly wrong for the job.

Should you care? Yes, twice over. As an operator, when a server misbehaves you now know to ask which primitive is in play — a runaway context bill from model-triggered document fetches is a "should have been a resource" bug. As a future author, the three-way split is a gift: it lets you place each capability with the party best placed to decide. The next chapter puts that to work.

The newer additions: tasks and elicitation

The three classic primitives assume a tidy world: every operation finishes quickly, and the server never needs to ask anybody anything. Real back offices are messier — some jobs take an hour, and some need a mid-job decision from the customer. Two newer spec additions patch exactly those holes; here they are in one flow. Step through it:

1. A big job arrives — as a task

Agenttools/call: archive_reports — run it as a task
→
Server
You

Archiving a year of reports will take minutes, not milliseconds. So the request is made as a task — a call-now, fetch-later arrangement for long-running work, added (as experimental) in the 2025-11-25 revision.

2. The server hands back a receipt

Agent
←
Servertask-42, status: working
You

No result yet — just a task handle and a status. The agent is free to do other things and poll later. This is the numbered token from the story, and the same call-now-collect-later shape you saw with async and await.

3. Mid-job, the server needs a human

Agent
Serverelicitation: "Overwrite last year's archive?"
→
Youa small form appears in your client

The server sends an elicitation request — a structured question for the human, delivered through the client's UI, added in the 2025-06-18 revision. The task's status flips to input_required. You can accept (and answer), decline, or cancel — the server never talks to you directly, only through the client.

4. You answer; the job resumes

You"Keep last year's archive"
→
Servertask-42, status: working
Agentpolls now and then

Your answer flows back and the status returns to working. If this pause-for-a-human pattern feels familiar, it should — it is the same human-in-the-loop idea as LangGraph's interrupt and resume, standardised at the protocol layer.

5. Done — collect on the receipt

Agentfetches the result for task-42
←
Serverstatus: completed

The status reaches completed (the other endings are failed and cancelled) and the agent retrieves the result. One long job, one human question, zero connections held hostage.

Keep both at "recognise it when you see it" depth for now. Elicitation is deliberately narrow — simple, schema-described fields, so a server can ask "which region?" but cannot render an arbitrary app inside your client. And tasks are still stamped experimental, which is the spec's way of saying: useful today, details may still shift, check before you build on them.

The spec is a living document — read it like one

Everything above describes a protocol that is still growing. MCP revisions are named by date — 2025-03-26, 2025-06-18, 2025-11-25 — and client and server negotiate which revision they share during the initialize handshake, alongside capability flags declaring which primitives each side actually supports. That negotiation is why a two-year-old server still works with a brand-new client: neither side assumes features the other never claimed.

So when something you read contradicts something you observe, do what working engineers do: open the changelog. The official spec site (modelcontextprotocol.io) publishes a "Key Changes" page per revision, listing exactly what was added, changed, or deprecated since the previous one — elicitation appears in the 2025-06-18 page, tasks in 2025-11-25. Skimming those pages when a new revision lands takes ten minutes and quietly puts you ahead of most people arguing about MCP online.

In the wild

Open any MCP server's README and look at what it exports — you will now read the list differently: a wall of tools and nothing else tells you the author took the path of least resistance. In Claude Code, run /mcp and you can see each connected server's tools; server-provided prompts appear in the slash-command menu, and resources can be pulled in with @-mentions. In a merge request, "expose the schema as a resource instead of a tool" is a reviewer moving the trigger from the model to the app — you now know exactly what that sentence costs and buys. And when a teammate says "that only landed in the November spec revision", they are citing the Key Changes page you just learned to read.

Check yourself

1. A server exposes last quarter's sales report. Who decides whether it enters the model's context — if it is exposed as a resource?

Resources are application-controlled: the host curates which ones are attached. Had the same report been wrapped in a get_report tool, the model would hold that trigger instead — same data, different decider.

2. Why do tools get the lion's share of MCP attention?

All three are core spec, and tools are not inherently cheaper. The skew is mechanical: hosts already run a tool-calling loop, so supporting tools is nearly free — and tools can impersonate the other two, so lazy designs funnel everything into them.

3. What is elicitation?

Elicitation (2025-06-18 revision) reverses the usual direction: the server needs input from the user and asks via the client's UI. It is deliberately limited to simple, schema-described fields — a question form, not an app.

4. An agent starts a long-running operation as a task. What does the server return immediately?

Tasks (experimental, 2025-11-25 revision) are call-now, fetch-later: the server hands back a receipt, the status moves through states like working, input_required, and completed, and the agent collects the result when it is done.

5. A server's prompt named review_merge_request will typically show up in your client as what?

Prompts are user-controlled templates: the server defines them, the client surfaces them as commands or menu entries, and a human pulls the trigger — the laminated menu on the front counter.

Key takeaways

  • An MCP server's three classic offerings differ by who pulls the trigger: tools are model-controlled, resources are application-controlled, prompts are user-controlled.
  • Tools are the familiar tool-calling handshake with the function living in someone else's server — discovered via tools/list, invoked via tools/call, described by JSON Schema, and the natural home of approval gates.
  • Resources are URI-addressed context with no side effects; prompts are parameterised templates that surface as slash commands. Wrapping either in a tool is possible — but it silently hands the model a decision that belonged to the app or the user.
  • Tools dominate the ecosystem because every host already has a tool-calling socket, and tools can impersonate the other two — a skew of convenience, not of merit.
  • Newer spec additions fill real gaps: tasks give long-running work a poll-able receipt (working, input_required, completed, failed, cancelled), and elicitation lets a server ask the user a structured question mid-job.
  • The spec evolves in dated revisions negotiated at initialize; when in doubt, read the Key Changes page for the revision in question rather than arguing from memory.
