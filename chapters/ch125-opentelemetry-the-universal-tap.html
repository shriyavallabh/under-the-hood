OpenTelemetry: the universal tap

You know you need traces, metrics, and logs. OpenTelemetry is the one open standard that lets you emit all three once and send them anywhere - instrument the code, not the vendor.

~30 minChapter 125 of 135
The story

Imagine a house where every appliance needs its own shape of wall socket. The kettle plugs into a round hole, the lamp into a triangle, the toaster into a star. Change electricity supplier and every socket in the house has to be re-cut, because the new company ships its own plug. That was monitoring for years: each vendor shipped its own library, so the code that emitted your telemetry was welded to whichever dashboard you happened to buy. Want to try a different tool? Rip out every measuring line in your codebase and rewrite it against the new library.

Then the industry did the sensible thing a house does: it agreed on a standard socket. Now the kettle, the lamp, and the toaster all end in the same plug, and the supplier is just whatever you connect at the wall. That standard socket, for telemetry, is OpenTelemetry. You wire your code to it once. What you plug in at the other end - which dashboard, which store, which vendor - becomes a config line you can change on a Tuesday afternoon without touching a single measuring statement.

One standard for all three signals

OpenTelemetry - almost always written OTel - is an open, vendor-neutral standard (and a set of SDKs) for producing the three signals you met last chapter: traces, metrics, and logs. It is a project of the Cloud Native Computing Foundation, the same neutral home as Kubernetes, which is why practically every observability vendor now speaks it. The promise is exactly the standard-socket promise: instrument once, export anywhere.

OTel has four pieces worth naming. The API and SDK are what you call in code to start a span or record a number. Instrumentation is the code that does that calling - it can be automatic or manual. OTLP (the OpenTelemetry Protocol) is the agreed wire format the data travels in. And the Collector is a separate little program that receives your telemetry, batches it, optionally cleans it, and forwards it onward. Keep those four in view and nothing about OTel is mysterious.

The mental model: emit, collect, fan out

Every OTel setup, however large, is the same three-station pipeline. Your app emits signals through the SDK. They travel over OTLP to a Collector. The Collector fans them out to one or more backends - a traces store here, a metrics store there. The beauty is that your code only ever knows about the first hop. Where the data ultimately lands is the Collector's business, changed in config, not in code.

Turning it on: one setup call

In practice you write a small setup_telemetry() function, call it once at program start, and forget it. Read it the way a reviewer would - each line is boring, and boring is the point.

telemetry.py - called once at startup
from opentelemetry import trace, metrics
The two entry points you touch most: trace for spans, metrics for counters and histograms. Logs have their own module; many teams keep logs on their existing logger and just attach the trace id.
from opentelemetry.sdk.trace import TracerProvider
The provider is the factory that hands out tracers. You create one, configure where it exports to, and register it as the global default. Everything downstream just asks the global for a tracer.
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
The exporter that speaks OTLP. This is the ONLY line that knows about a destination. Point it at a Collector or a backend; change the URL and you change vendors - no other line moves.
def setup_telemetry(endpoint):
One function, called once. Real projects wrap all of this so an agent author writes a single line at the top of their program and gets the whole pipeline.
provider = TracerProvider()
provider.add_span_processor(exporter_to(endpoint))
Wire the exporter into the provider through a span processor (usually a batching one, so spans are shipped in efficient groups rather than one network call each).
trace.set_tracer_provider(provider)
Register it globally. From now on any code, anywhere, that calls trace.get_tracer(__name__) gets a real, exporting tracer. This global handoff is why one setup call lights up the whole app.

With that in place, emitting is tiny. To trace a block of work you wrap it in a span; to count something you create a counter once and add to it:

using it anywhere in the agent
tracer = trace.get_tracer(__name__)
meter = metrics.get_meter(__name__)
requests = meter.create_counter("agent.requests")

with tracer.start_as_current_span("llm.plan") as span:
    span.set_attribute("gen_ai.request.model", "claude")
    span.set_attribute("gen_ai.usage.input_tokens", 180)
    requests.add(1)
    # ... the real model call happens here ...

Notice the attribute names: gen_ai.request.model, gen_ai.usage.input_tokens. Those are the semantic conventions - agreed names so that your spans line up with everyone else's, and a dashboard built for one agent understands the next. When you are unsure of an exact attribute or method name, check the current OpenTelemetry docs rather than inventing one; the shape above is stable, the precise strings evolve.

Feel the shape by running it

The real SDK ships this data to a backend. A stand-in that prints instead lets you see the exact shape: a span opens, does work, records attributes, and closes with a duration; a counter accumulates. Press Run.

Note

You did not have to edit the model call to trace it - you wrapped it. That is the seed of auto-instrumentation: libraries that wrap the popular HTTP and LLM clients for you, so spans appear the moment you switch them on, with no changes to your business logic. Manual spans like the ones above are for the work only you know is worth measuring.

In the wild

You will see OTel as a cluster of opentelemetry-sdk and opentelemetry-instrumentation-* packages in a requirements file, an OTEL_EXPORTER_OTLP_ENDPOINT environment variable in a deployment config, and a otel-collector-config.yaml in the repo. When a platform team at a large bank says "just emit OTLP to our Collector," they mean: use this standard, point your exporter at our address, and we handle the rest. When someone runs their app with an opentelemetry-instrument launcher, they are turning on auto-instrumentation without changing a line of the program. And a merge request titled "add gen_ai span attributes" is someone adopting the semantic conventions so token counts show up on the shared dashboard.

Check yourself

1. What problem does OpenTelemetry primarily solve?

OTel is the universal socket: one standard way to emit traces, metrics, and logs, decoupled from any particular vendor. Storing and drawing the data is the backend's job, covered next chapter.

2. In the setup, which line is the only one that knows about a specific destination?

That decoupling is the whole point: only the exporter carries the address. Change vendors by changing the endpoint; every measuring line in your code stays exactly the same.

3. What does the OpenTelemetry Collector do?

The Collector is the middle station: receive, batch, optionally redact or route, fan out. Your app talks only to it, which is why swapping or adding backends never touches your code.

4. Why do semantic conventions like gen_ai.usage.input_tokens matter?

Conventions are about agreement, not enforcement. When everyone names the token count the same way, one dashboard can read every agent's spans - the difference between a private hack and a platform.

Key takeaways

  • OpenTelemetry (OTel) is a vendor-neutral standard plus SDKs for all three signals - the universal socket that lets you instrument once and export anywhere.
  • Its four parts: the API/SDK you call, instrumentation (auto or manual), OTLP (the wire format), and the Collector (receive, process, fan out).
  • The pipeline is always emit to Collector to backends; your code only knows the first hop, so changing or adding a destination is a config line, not a code change.
  • One setup_telemetry() call registers a global provider; after that, wrapping work in start_as_current_span and ticking a counter is all the emitting you do.
  • Semantic conventions (the gen_ai.* attributes) are agreed names so token counts and model names line up across every agent on a shared dashboard.
  • OTel gives you the pipes and the format, not the picture - where the data lives and how you look at it is the next chapter.
