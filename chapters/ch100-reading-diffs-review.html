Reading diffs and reviewing like a pro

Red and green will read as fluently as prose, you will know exactly what to hunt for, and you will leave comments authors are glad to receive. A booby-trapped diff waits at the end — and you are going to catch every trap in it.

~30 minChapter 101 of 117
The story

You email the rental agreement for your new flat to a lawyer friend. Fourteen pages of dense legalese. It comes back an hour later — but she has not rewritten it. She has marked it up: every deleted phrase struck through in red, every new phrase in green, and in the margins, short notes. You do not reread fourteen pages. You read only the markings, and in five minutes you know everything that changed.

Then, on page nine, a red-green pair with no explanation: the notice period, quietly changed from sixty days to thirty. The landlord's cover note never mentioned it. Your friend's margin comment sits right beside it: "Was this discussed? It isn't in the covering letter." Polite. Precise. Devastating.

That marked-up contract is a diff. The margin conversation is a code review. And the most valuable thing a reviewer ever catches is exactly what your friend caught: the change nobody mentioned.

A diff is a redline, and you already half-read them

Back in the git chapter you learned that a commit is a snapshot of the whole project. A diff is the subtraction of two snapshots — a machine-generated redline showing only the lines that differ, so a reviewer never rereads the whole file. Every merge request from the previous chapter had one waiting under its Changes tab; now you learn to actually read it.

The colour code has exactly two rules. A line prefixed with - (shown red) existed in the old version and is being removed. A line prefixed with + (shown green) is being added in the new version. That is the whole alphabet. There is no such thing as a "modified line" in a diff — a changed line always appears as a pair: the red line as it was, directly above the green line as it will be. Read every pair as a before-and-after photograph.

Lines with no prefix (just a leading space) are context lines — unchanged lines included around each change purely so you can orient yourself. They are the unmarked sentences your lawyer friend left visible around each strikethrough.

Here is the smallest interesting diff in the world:

a one-line change, as git shows it
--- a/greeting.py
+++ b/greeting.py
@@ -1,4 +1,4 @@
 def greet(name):
-    return "Hello " + name
+    return f"Hello, {name}!"

 print(greet("Asha"))

One function, one changed line, shown as a remove-add pair with context for orientation. Now let us name every part, because each line is load-bearing.

The anatomy, line by line

Click each annotated line and make sure it feels boring. Boring means fluent — same trick as the ReAct chapter.

the same diff, dissected
diff --git a/greeting.py b/greeting.py
The envelope. Git names the file being compared, in both its old (a/) and new (b/) incarnations. A multi-file diff is just several of these envelopes stacked; GitLab turns each one into a collapsible file section.
--- a/greeting.py
The old version — the "a" side, the minus side. If a file is brand new, this line reads --- /dev/null: it came from nothing. That is how you spot file creations at a glance.
+++ b/greeting.py
The new version — the "b" side, the plus side. A deleted file has +++ /dev/null: it goes to nothing. Renames show a different path on each side.
@@ -1,4 +1,4 @@
The famous hunk header, decoded in the next section. For now: it is a map reference telling you where in the file you are standing.
def greet(name):
A context line — note the single leading space instead of + or -. Unchanged, shown only so you know which function you are inside.
- return "Hello " + name
Red. This exact line exists in the old version and will not exist in the new one. On its own it would be a deletion; paired with the next line, it is the "before" half of an edit.
+ return f"Hello, {name}!"
Green, the "after" half. Reading red-green pairs top-to-bottom — was this, becomes this — is the core fluency skill. Practise it until your eye does it automatically.
print(greet("Asha"))
Trailing context. The diff shows a handful of unchanged lines after the change too, so the edit never floats in a void.

Hunks and the strange @@ line

Diffs do not show whole files; they show hunks — contiguous patches of change, each wrapped in a few context lines, like paragraphs in the redline. A big merge request is a list of files, each file a list of hunks. The @@ -1,4 +1,4 @@ header on each hunk is its map reference: -1,4 means "in the old file, this hunk starts at line 1 and spans 4 of its lines"; +1,4 says the same for the new file. If a hunk adds more than it removes, the second number grows: @@ -10,9 +10,17 @@ tells you nine old lines became seventeen new ones before you have read a single line of code.

Nobody computes these numbers by hand. But being able to read them means you always know where you are — and any text after the closing @@ (often a function name) is git being helpful about which function the hunk lives inside.

Unified or side-by-side — and the lines you cannot see

Everything so far has been the unified view: old and new interleaved in one column, red directly above green. It is compact, it is what git diff prints in a terminal, and it is ideal for small, surgical changes. The alternative is the side-by-side view: old file left, new file right, changed regions aligned across the gutter. When a function has been substantially rewritten — when the red and green blocks are too large to pair up in your head — side-by-side lets you read the new version as flowing code with the old one in peripheral vision. In GitLab the toggle lives in the viewer settings on the Changes tab. Rule of thumb: unified for tweaks, side-by-side for rewrites.

Now the trap every new reviewer falls into. A diff shows you a keyhole: each hunk plus three-or-so context lines. The truth often lives outside that keyhole — the variable defined forty lines up, the function's other caller, the import block at the top of the file. GitLab (like GitHub) puts fold bars between hunks; clicking them expands the hidden, unchanged lines right there in the review. Expanding context is not a power-user trick. It is basic diligence, and the booby-trapped diff at the end of this chapter will punish anyone who skips it.

Tip

Adopt this as a reflex: never review a change to a function you cannot see the whole of. If the hunk shows the middle of something, expand up and down until you have the full body on screen. Half the "how did that get merged?" bugs in the wild slipped through a keyhole.

What a good reviewer actually looks for

Here is the liberating part: reviewing is not "could I have written this?" It is "can I verify this?" — a different skill, and one you already have from years of operating production systems and reviewing what agents propose before approving it. Work down this checklist, in order, every time:

  1. Does it do what it says? Read the title and description first, then hold every red-green pair up against them. Every change should be accounted for. A change the description never mentions — the sixty-to-thirty-days clause — is finding number one, every time. It may be innocent; it still gets a question.
  2. Are the tests there — and honest? New behaviour should come with new or updated tests (pytest fluency arrives in the pytest chapter, but you can count test functions today). Watch especially for tests deleted or loosened in the same MR that changes behaviour: sometimes legitimate, often a way of making a red pipeline green without making the code right.
  3. Names and readability. Would the next reader — six months from now, at 2 a.m., mid-incident — understand this? scored_batch beats data2. Naming comments feel small; they are compound interest.
  4. Edge cases. Empty list, zero, None, the failure path. What happens when the network call fails, when the retry runs out? Happy-path-only code is half-written — you learned to read the unhappy path in the tracebacks chapter.
  5. Security smells. Hardcoded secrets, credentials in fixtures, SQL built by string concatenation, TLS verification switched off, secrets written to logs, surprise new dependencies. Any of these blocks the merge — the instinct the guardrails chapter drilled for agents applies to human diffs too.
Tip

You never need to out-code the author to add value. "What happens when this list is empty?" is a first-class review comment, and either the author answers convincingly — in which case the answer usually belongs in the code or tests — or they discover a bug you just prevented. Questions are findings.

How to say it: ask, don't accuse

A review comment lands in another human's day. Two habits keep the temperature low and the quality high. First, address the code, never the coder: "this function returns an empty dict on failure", not "you swallowed the error". Second, when something looks wrong, ask — a genuine question is both more polite and more accurate, because sometimes the surprising thing is deliberate and the real bug is a missing explanation in the code.

"You broke the timeout.""Was the timeout change from 30 to 3 intentional? It isn't mentioned in the description."
"This is wrong.""I think this returns an empty dict on failure — how will callers tell an error from a genuinely empty batch?"
"Bad variable name.""nit: could data2 be scored_batch? Fine to skip."
"Why are there no tests???""Could we add a test for the retry path — say, two failures then a success?"

Third habit: suggest concretely. When you know the fix, do not describe it — write it. GitLab lets any diff comment carry a suggestion: a fenced block of replacement code the author can apply with one click. It is the most generous move in reviewing:

Check yourself

1. Warm-up, mechanics: in the header @@ -10,9 +10,17 @@, what does -10,9 tell you?

The minus side describes the old file, the plus side the new: start line, then span. Here nine old lines became seventeen new ones — the hunk header told you the change grew the file before you read any code.

2. "Does it do what it says": the description promises "No behaviour change otherwise." Which red-green pair breaks that promise?

The timeout was cut tenfold and the description never mentions it — this diff's sixty-to-thirty-days clause. Slow-but-healthy responses that used to succeed will now time out. The retry parameter and header are what the description advertises; the timeout is the stowaway.

3. Security: which line should block this merge until it is fixed, no matter how good the rest is?

A live credential hardcoded in source control is a blocking finding, full stop — once merged it lives in git history forever, even if a later commit removes it. The fix is the suggestion you saw earlier: read it from the environment, and rotate the leaked token. The backoff line and the params are fine.

4. Tests: what does the deleted test_fetch_scores_raises_on_500 tell a sharp reviewer?

Follow the logic: the new except Exception swallows every error and finally returns an empty dict, so callers can no longer tell "server down" from "batch genuinely empty". The old test asserted that failures raise — the new code fails that test, and it vanished in the same MR. A green pipeline is not the same thing as correct behaviour; this deserves a firm, blocking comment.

5. Etiquette: you decide to raise the timeout change. Which comment is the professional one?

Ask, don't accuse — and suggest concretely. Option (b) addresses the code, leaves room for a good answer you have not thought of, and proposes a specific alternative. Option (a) attacks the person; option (c) is how sixty days quietly becomes thirty.

Careful

There was a fifth trap, hiding outside the keyhole. The new code calls time.sleep(...) — but is time imported? The hunk cannot tell you: the only import visible is the import requests peeking through the hunk header. Expanding context is the only way to know. And the twist worth savouring: if the import is missing, the crash happens inside the retry path — the exact path whose test this MR deleted. The deleted test would have caught the missing import. In reviews, the traps hold hands.

Key takeaways

  • A diff is a machine-made redline: red - lines are the old version, green + lines the new, and a changed line is always a red-green pair — there is no "modified line".
  • Hunks are the diff's paragraphs, and the @@ -start,span +start,span @@ header is a map reference: old-file position on the minus side, new-file position on the plus side.
  • Use unified view for tweaks and side-by-side for rewrites — and always expand context until you can see the whole function, because the bug often lives just outside the keyhole.
  • Review with a checklist, in order: does it do what the description says, are the tests present and honest, names, edge cases, security smells. Unexplained changes and deleted tests are the two loudest alarms.
  • Comment on the code, not the coder: ask genuine questions, mark tiny things as nit:, say plainly when something blocks, and use GitLab suggestions to hand the author a one-click fix.
  • Batch comments with "Start a review" and publish once with "Submit review" — one coherent package beats a drip-feed of notifications.
  • You do not need to out-code the author to review well; verify claims and ask the questions the diff raises. You can do that now.
