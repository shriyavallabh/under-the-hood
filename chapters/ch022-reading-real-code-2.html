The story

You have been asked to assess a small, busy restaurant before investing in it. You have never set foot inside. The owner gives you the tour. The menu in the window tells you, in one glance, what this place offers the world — not how anything is cooked, just what you can order. In the kitchen, the pantry is a wall of labelled jars: salt, chilli, rice, each with a sensible amount, refilled from outside the kitchen. The head chef's station is where orders actually become food — the chef reaches into the pantry for settings and into a drawer of small sharp tools for the fiddly jobs: peeling, chopping, portioning. And taped inside the back door you find the best document in the building: the inspection log. "The dal must arrive warm. The bill must add up. An empty order is politely refused." Each promise has a tick against it, dated yesterday.

You never watched a single dish being cooked, yet you now understand the restaurant. A Python package is this building. This chapter is that tour.

From one page to a whole building

In the first reading-real-code chapter you learned to read a single Python file like a page: imports at the top declaring what it borrows, constants next, then function and class definitions, and any actual "do something now" code at the bottom. That skill still applies — but real projects are almost never one file. And in the previous chapter you picked up just enough git to clone a repository and wander around inside it. So today we stop practising on pages and read a whole building.

A package is simply a folder of Python files that Python treats as one unit, marked by a special file called __init__.py. Each file inside it is a module — a single .py file, exactly the kind you already know how to read (that vocabulary comes from the imports chapter). Nothing new is happening at the level of Python itself. What is new is layout: the code has been split across files, and each file has a job. Learn the jobs and an unfamiliar repo stops being a wall of filenames and becomes a floor plan.

Here is the package we will tour — a tiny library called digest that turns long text into short summaries by calling a language model. Five files. You will see every line of all five.

Stop 1: __init__.py — the front door

Open an unfamiliar package here first, always. It is usually short, and it answers the most valuable question you can ask about a stranger's code: what does this thing want to be used for?

digest/__init__.py
"""digest -- turn long text into short summaries via a language model."""

from .client import DigestClient
from .tools import chunk, clean

__all__ = ["DigestClient", "chunk", "clean"]

Three things to notice. First, the docstring at the top is the package's one-line pitch — the sign in the window. Second, those imports start with a dot: from .client import ... is a relative import, meaning "from the module named client inside this same package". The dot is a location, like saying "the kitchen down our own corridor" rather than a street address. Third, __all__ is the package's declared public API — the short list of names it promises to outsiders. Everything else inside the folder is wiring, free to change without warning.

This file is why users get to write from digest import DigestClient instead of the clunkier from digest.client import DigestClient. The front door re-exports the good stuff so nobody has to memorise the floor plan. When you read any library's import line in production code, this is the machinery behind it.

Stop 2: config.py — the pantry

Recall from the logging and config chapter: well-behaved code never hard-codes the values an operator might want to change, and it never, ever hard-codes secrets. This file is that principle given a room of its own.

digest/config.py
"""All the knobs, read once from the environment."""

import os

MODEL = os.getenv("DIGEST_MODEL", "small-fast-1")
MAX_CHUNK = int(os.getenv("DIGEST_MAX_CHUNK", "1000"))
API_KEY = os.getenv("DIGEST_API_KEY")   # a secret gets NO default


def require_api_key() -> str:
    """Fail early and loudly if the key is missing."""
    if not API_KEY:
        raise RuntimeError(
            "DIGEST_API_KEY is not set -- export it before using "
            "the real transport."
        )
    return API_KEY

Read the three constants like pantry labels. MODEL and MAX_CHUNK come with sensible defaults, so the package works out of the box; an operator overrides them by exporting an environment variable, no code edit required. API_KEY pointedly has no default — a baked-in fallback secret would sooner or later be committed to git and leaked. And require_api_key() converts a vague failure later ("401 Unauthorized, somewhere") into a loud, plain-English failure now. When a production incident ends with "the env var wasn't set on the new machine", this file is the scene of the crime.

Stop 3: tools.py — the drawer of small sharp things

digest/tools.py
"""Small, pure text helpers. No API calls, no stored state."""


def clean(text: str) -> str:
    """Collapse runs of whitespace into single spaces."""
    return " ".join(text.split())


def chunk(text: str, size: int) -> list[str]:
    """Split text into pieces of at most `size` characters."""
    if size < 1:
        raise ValueError("size must be at least 1")
    return [text[i:i + size] for i in range(0, len(text), size)]

Both are pure functions — same input, same output, touching nothing outside themselves (the shape you met in the functions chapter). The type hints (chapter 16) state the contract at a glance, and the comprehension in chunk is the compact way to say "take a slice every size characters". Files like this are deliberately boring, and that is a compliment: no moving parts means reviewers trust them and tests nail them down in one line.

Tip

You have now read three of five files and none contained the "real work". That is normal: most files in a healthy package are front doors, settings, and small helpers. Read them first, and the one genuinely complex file arrives with everything it leans on already familiar.

Stop 4: client.py — the head chef

Now the main event. This file holds a class — the pattern from the classes chapter: data and behaviour bundled together, with __init__ stashing what each instance needs onto self. Click through the annotated lines; the goal is for every one of them to feel boring by the end.

That design decision in __init__ deserves its name: dependency injection — instead of a class creating the things it depends on, the caller hands them in. Because DigestClient accepts any object with a complete method, a test can hand it a scripted fake and check the class's behaviour without spending a rupee on API calls or touching the network.

Stop 5: the tests — promises, executable

Here is the habit that separates people who skim repos from people who understand them: read the tests as documentation. Prose documentation drifts out of date; a test cannot, because the moment it stops being true the suite fails and someone must deal with it. Every assert — a statement meaning "this must be true, or crash with a complaint" — is a promise about behaviour that is still being kept, re-verified mechanically on every change.

Almost every test has the same rhythm: arrange (set up the objects), act (call the thing), assert (check the promise). And each test's name is a full sentence with underscores for spaces — read just the names below and you get a specification for free.

tests/test_client.py

pytest is the near-universal Python test runner: it finds files named test_*.py, runs every function named test_*, and reports which asserts held. No registration, no ceremony — naming is the wiring.

Note

Your default route for any unfamiliar package, worth writing on your wall: front door first, tests second, then inward. __init__.py says what it offers; the tests say what it promises and how it is meant to be called; only then read the core files, with config and helpers as reference. You will understand a repo in a quarter of the time of someone reading files alphabetically.

In the wild

This five-file shape scales up but never really changes. Open any serious Python repo and you will find it: an __init__.py re-exporting the headline names (when you write from langchain.agents import create_agent, an agents/ folder's front door re-exported that function for you), a config.py or settings.py reading environment variables, one or two core classes, a utils.py drawer, and a tests/ tree mirroring the package. Seasoned reviewers read a merge request's changed tests first — the diff of the promises — before the implementation. And when a colleague says "that's not part of the public API, don't depend on it", they mean exactly the __init__.py / __all__ boundary you toured today.

Check yourself

1. You have just cloned an unfamiliar Python package. Which file do you open first, and why?

The front door is short and answers the highest-value question: what is this thing for, and which names does it want you to use? Logic and settings make far more sense once you know that. (Tests are the strong second stop.)

2. In client.py you read from .tools import clean. What does the leading dot mean?

The dot is a relative location: "the tools module down our own corridor". It is how files inside one package refer to their siblings without caring where the package is installed.

3. Why does DigestClient accept its transport as a constructor argument instead of creating the real API caller internally?

Dependency injection: the caller hands in the dependency, so a test can hand in FakeTransport. That single design choice is why the whole suite runs free, offline, and deterministically — and it is the same fake-the-model trick agent test suites use everywhere.

4. A test contains assert len(fake.prompts) == 1, "short text should cost ONE call". What is this line really documenting?

The assert pins down a cost behaviour of the client. Unlike a comment or a wiki page, it cannot silently drift: the moment someone makes short text cost two calls, this line fails and names the broken promise.

5. In config.py, MODEL has a default value but API_KEY does not. Why the difference?

Tunables get sensible defaults so the package works out of the box. Secrets must only ever arrive from the environment — the only safe default for a credential is "absent, and fail loudly".

Key takeaways

  • A package is a folder of modules with an __init__.py front door; the layout — public face, config, core class, helpers, tests — repeats across the entire Python ecosystem.
  • Read in this order: __init__.py for what it offers, tests for what it promises, then the core files — never alphabetically.
  • Relative imports (from .tools import ...) wire sibling modules together, and dependencies flow from complex files towards simple ones.
  • Config lives in one file, read from environment variables: tunables get defaults, secrets get none, and missing keys fail early and loudly.
  • Tests are executable documentation — each assert is a promise that cannot drift out of date, and test names read as a free specification.
  • Dependency injection (handing the transport in from outside) is what makes the package testable with a scripted fake — the same pattern used to test agents without paying for model calls.
