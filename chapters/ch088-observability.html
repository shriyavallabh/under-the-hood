Observability: traces, costs, audit

An agent you cannot see is an agent you cannot trust, debug, or bill. This chapter gives you the three instruments every serious harness carries: the trace, the meter, and the tamper-evident record.

~30 minChapter 89 of 123
The story

Friday night at a packed Pune restaurant. Table nine waves the manager over: the paneer arrived forty minutes late, it was the wrong dish, and the bill looks inflated. In a badly run kitchen this becomes a shouting match — the waiter blames the chef, the chef blames the tandoor, and nobody knows what the plate actually cost to make.

But this kitchen runs on tickets. Every order prints a chit, timestamped as it enters, stamped again at each station it passes — cold prep, tandoor, plating, pass. The costing sheet tracks the ingredient cost of every plate, sliced by table and by waiter. And as each order completes, its chit is pressed onto the steel spike by the pass: order after order, in strict sequence, impossible to reorder or quietly slide one out. The manager pulls the chit for table nine: ordered 8:14, fired 8:19, plated 8:52 — the delay was the tandoor queue, plain as ink. The costing sheet shows the wrong dish was rung in by a trainee. And when the food-safety inspector asks "who handled the chicken on the 14th?", the spike answers, because the spike never forgets and never lies.

The ticket that follows the dish is a trace. The costing sheet is metering. The spike is the audit trail. Your agent needs all three.

Three questions, three instruments

The previous chapter, guardrails, was about deciding in the moment: is this input safe, is this action allowed, does this need a human? Guardrails are the bouncer at the door. This chapter is the security camera, the till roll, and the visitor register — the layer that lets you answer questions after the fact. Observability is the property of a system that lets you work out what happened inside it purely from what it emitted while running — without stopping it, guessing, or re-running the failure and hoping.

For agents, every question an operator ever asks turns out to be one of three:

  • What did it do? Which model calls, which tool calls, in what order, with what inputs and outputs, and how long did each take? That is the trace.
  • What did it cost? How many tokens, at what price, attributed to which agent, which user, which customer? That is metering.
  • On whose authority? Who asked for this, under which permissions, and can we prove the record has not been edited since? That is the audit trail.

They sound similar, and lazy systems mash them into one log file. They are not the same. A trace is rich and disposable — you might keep it for thirty days and sample only a fraction of runs. A meter is a running total — you keep the aggregates forever but not the detail. An audit trail is sparse and sacred — every consequential action, kept for years, tamper-evident, because a regulator may one day read it back to you.

Traces: the ticket that follows the dish

Remember the loop you built by hand in the ReAct chapter: model call, tool call, observation, repeat. A trace is the complete recorded story of one run of that loop — every model call and tool call captured as a timeline, with timings, token counts, inputs and outputs attached. Each individual operation inside the trace is a span — one timed unit of work with a start, an end, and a bag of attributes. Spans nest: the run is the root span, each turn of the loop is a child, each tool call a child of its turn. The result is a tree, and the tree shape is the whole point — a flat log tells you that things happened; a trace tells you which run they belonged to and what caused what.

Two names dominate this space. LangChain's hosted trace viewer is LangSmith — set its environment variables (an API key and a tracing flag) and every agent run is captured automatically, no code changes; its traceable decorator (a decorator, wrapping a function to add behaviour) lets you add spans around your own functions too. The vendor-neutral standard is OpenTelemetry — an open specification for emitting traces from any software, which your platform team almost certainly already uses for ordinary services; agent frameworks increasingly emit OpenTelemetry spans so agent traces land in the same dashboards as everything else. Langfuse, Arize Phoenix and friends play in the same league. The concepts are identical everywhere; only the logos differ.

Metering: the bill, sliced the way you need it

You met token pricing in tokens and cost and watched deep agents multiply it in the token bill. Metering is the harness discipline built on top of that arithmetic: recording usage per call, then rolling it up by every dimension someone will eventually ask about. "How much did the agent cost today?" is the easy question. The real ones are: which agent (the invoice-chaser or the researcher?), which user (is one analyst running fifty times more jobs than the rest?), and which tenant — which customer, in a multi-customer product — so you can charge back, cap, or investigate. If cost is not attributed at write time, it cannot be reconstructed later; the meter must stamp every span with agent, user and tenant the moment the call happens.

On top of attribution sit budgets — pre-agreed spending limits that the harness enforces rather than merely reports. A sane setup has three rungs. A soft alert fires at, say, 80 percent of budget: a message to the operators, nothing stops. A hard cap stops new runs when the budget is spent — the agent politely refuses rather than silently overdrafting. And a per-run circuit breaker kills any single run that exceeds its own allowance, because the most common cost incident is not a thousand users — it is one confused agent looping, exactly the failure the max_turns guard in chapter 46 protected against, now measured in currency instead of iterations. Budgets are guardrails' financial cousins: the guardrail says "you may not call that tool"; the budget says "you may not spend a rupee more".

Note

Meter tokens, not just money. Prices change, cached input tokens are billed at a discount, and different models cost wildly different amounts per token — so the durable record is tokens per call per model, with price applied at read time. Store the raw measurement; derive the invoice.

The audit trail: memory that cannot be edited

Traces and meters serve you. The audit trail serves people who do not trust you — auditors, regulators, courts, and your own future self during an incident review. An audit trail is an append-only record of every consequential action, designed so that its completeness and integrity can be proven. Financial regulators have demanded this of trading systems for decades (records kept for years, on storage that cannot be rewritten), and the EU AI Act extends the same expectation to high-risk AI systems: automatic event logging, retained, reviewable. When an agent moves money, sends email, or changes customer data, "we think it was fine" is not an answer. The spike must answer.

Three properties separate an audit trail from a fancy log. It is append-only: records are added, never updated, never deleted within the retention window — often enforced with WORM storage ("write once, read many"). It is tamper-evident: each record carries a cryptographic fingerprint chained to the previous one, as in the runner above, so any edit is detectable. And it is complete for the questions that matter — every record answers who asked, what the agent did, and on whose authority. That last one is the piece newcomers miss: an agent acts with delegated permission, so the record must capture the delegation — which human or service initiated the run, which role and policy version authorised the action, and whether a human approval was in the chain. Click through a real-shaped record:

audit-record 214 — one line in the trail, unfolded
{
"seq": 214,
A strictly increasing sequence number. If the trail ever jumps from 213 to 215, something was deleted — gaps are themselves evidence.
"ts": "2026-07-02T09:14:07Z",
Timestamp in UTC, from a trusted clock. Auditors reconstruct timelines across systems, so everyone logs in the same timezone: none.
"actor": {"user": "priya.k", "channel": "web", "session": "s-88f2"},
Who asked. The human (or upstream service) that initiated the run — not the agent's name. Agents are never the actor of record; they are the instrument.
"authority": {"role": "ops-analyst", "policy": "refunds-v3", "approval": "auto"},
On whose authority. The role that permitted this action, the exact policy version in force, and whether approval was automatic or given by a named human. When the policy changes next month, this record still shows which rules applied today.
"action": {"type": "tool.call", "name": "issue_refund", "args_digest": "sha256:9c1f..."},
What the agent did. Note the digest: a fingerprint of the arguments rather than the raw arguments, so the trail proves what was sent without storing card numbers or personal data inside an immutable record you can never redact.
"agent": {"run_id": "run-7f3a", "model": "claude-x", "prompt_version": "12"},
The bridge to the trace: the run id links this legal-grade record to the rich debugging detail, and the model + prompt version pin down exactly which brain, with which instructions, acted.
"outcome": {"status": "ok", "amount": 1500, "currency": "INR"},
What actually happened — including failures. An audit trail that only records successes is a marketing document.
"prev_hash": "sha256:41ab...",
The previous record's fingerprint, folded into this one. This is the chain from the runner above: edit record 150 and records 151 onward all stop verifying.
"hash": "sha256:77d0..."
This record's own fingerprint. Periodically, the latest hash is written somewhere external (another system, even a printed register) — anchoring the whole chain against wholesale replacement.

Debugging from a trace: reading a failed run

Now the payoff. Observability earns its keep on the bad morning: an invoice-chasing agent has emailed a payment reminder to a customer who already paid, the customer is annoyed, and you were not in the room — the run happened at 2 a.m. Here is how an operator actually reads the failure, step by step. This exact walk — symptom, trace, suspicious span, attributes, root cause — is the same every time; only the details change.

1. Find the run

Trace searchtenant=acme · agent=invoice-chaser · last 24 h
→
run-4c9dstatus: success

You filter thousands of traces by the metadata tags — tenant, agent, time window — and find run-4c9d. First surprise: the run reports success. The agent believes it did its job. Whatever went wrong, the model never saw it as an error.

2. Open the trace tree

run-4c9droot span · 41 s · $0.09
model.call #1plan
tool.get_unpaid_invoices
model.call #2draft email
tool.send_email

The root span unfolds into its children: two model calls, two tool calls, all green, latencies unremarkable. Nothing is flashing red — which tells you this is a data problem or a reasoning problem, not a crash. Time to read contents, not just statuses.

3. Read what the tool returned

model.call #1
→
tool.get_unpaid_invoicesoutput: 12 rows... INV-208 "unpaid"

Traces record inputs and outputs, so you open the tool span's recorded output — and there it is: INV-208, the invoice the customer already paid, listed as unpaid. The tool told the agent this invoice was outstanding. The suspect list just changed.

4. Clear the model

tool outputINV-208 unpaid
→
model.call #2reasoned correctly from what it saw

Read model.call #2's input and output: the model was shown a list containing INV-208 and correctly drafted a reminder for it. Given its inputs, the reasoning is flawless. This was never a hallucination — the model was fed a wrong world and behaved perfectly inside it.

5. Drop one level: the span attributes

tool.get_unpaid_invoicescache_hit: true · data_as_of: 22:10
→
payment landed22:41

The tool span's attributes hold the confession: cache_hit: true, data as of 22:10. The customer paid at 22:41. The run happened at 2 a.m. against a cache that never refreshed. Root cause: a stale cache TTL in the invoice tool — three levels below anything an eyeball on the agent's chat output could have caught.

6. Fix, then answer the hard question

Fixcache TTL + freshness check
→
Audit trailwho was emailed, what was sent, on whose authority

The engineering fix is a one-liner. But the customer-facing question — "what exactly did you send, to whom, authorised by what?" — is answered from the audit trail, verbatim and provable. Trace for the diagnosis, audit for the accountability. That division of labour is the whole chapter.

Careful

The most dangerous failed run is the one that reports success. Error dashboards only catch crashes; the wrong-refund, wrong-email, wrong-customer class of failure sails through green. This is why serious teams sample successful traces for review and run evals against recorded runs — the trace store doubles as a test-case mine.

In the wild

In the wild

You will meet this layer everywhere once you know its face. In LangChain projects, a couple of LangSmith environment variables in the deployment config mean every run is being traced — follow the run id from a bug report straight into the trace viewer. In enterprise repos, look for an OpenTelemetry collector config (a YAML file routing spans to the company's dashboards), middleware named tracing.py or metering.py wrapped around the agent loop, and merge-request comments like "add a span around the retriever" or "tag traces with tenant id". Cost work shows up as dashboards titled "spend by agent" and incident channels lighting up with budget alerts. And in any bank-adjacent setting, the audit trail has its own review: append-only tables or WORM buckets, hash chaining, retention policies measured in years — because when the regulator asks "who authorised this action?", the answer must come from the spike, not from memory.

Check yourself

1. What does a trace give you that a plain log file does not?

Logs are flat lines from everywhere at once. A trace groups every model call and tool call under the run that caused them, nested parent-to-child — so you can follow one request through the whole loop instead of grepping and guessing.

2. Why must cost be attributed to agent, user, and tenant at the moment each call happens?

Once usage lands in one undifferentiated total, no amount of analysis recovers who caused it. The meter must stamp each span with its dimensions at write time; roll-ups, chargeback and per-tenant caps all depend on that.

3. What does hash-chaining the audit trail actually guarantee?

A hash chain is tamper evidence, not tamper prevention and not encryption. Anyone can still read the records, and a bad action still lands in the trail — but nobody can quietly rewrite history, because each record's fingerprint is folded into the next.

4. In the failed-run walkthrough, the model drafted a reminder for an invoice that was already paid. What did the trace reveal?

Reading the spans cleared the model: given its inputs it behaved perfectly. The tool span's attributes (cache_hit: true, stale timestamp) held the root cause. A huge share of "the AI messed up" incidents are really "the AI was fed a wrong world" — and only traces can tell the difference.

5. A regulator asks: "Prove who authorised the refund your agent issued in March." Which instrument answers, and why?

Traces are rich but disposable — sampled, expired after weeks, and editable in principle. The audit trail exists precisely for this question: who asked, what was done, on whose authority, append-only and tamper-evident across the retention window.

Key takeaways

  • Observability answers three distinct questions with three distinct instruments: the trace (what happened), the meter (what it cost), and the audit trail (who asked, on whose authority).
  • A trace is a tree of spans — every model call and tool call with latency, tokens and recorded inputs/outputs. LangSmith and OpenTelemetry are the two names you will meet; the concepts are identical.
  • Meter tokens per call, stamped with agent, user and tenant at write time — attribution cannot be reconstructed later. Budgets turn the meter into a guardrail: soft alert, hard cap, per-run circuit breaker.
  • An audit trail is append-only, tamper-evident (hash-chained), and captures actor, action and authority — the record regulators require when an agent acts with delegated permission.
  • Debugging a failed run is a repeatable walk: find the run by its tags, open the tree, read tool outputs before blaming the model, then drop into span attributes for the root cause.
  • The scariest failures report success. Sample and review green traces too — the trace store is also your eval data.
