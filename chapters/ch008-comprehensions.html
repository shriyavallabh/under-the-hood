Comprehensions: loops folded flat

The cryptic one-liners that fill framework code — [x for x in ...] — are ordinary loops, folded like a shirt. Learn the fold, and they read like English sentences.

~25 minChapter 8 of 135
The story

A wedding kitchen, half past noon. The head caterer does not have time for paragraphs, so she gives her junior one sentence: "A plate of jalebi for every guest at table four who asked for dessert." Then she is gone, back into the steam.

The junior unfolds that sentence in his head without even noticing. Take an empty tray. Walk around table four, guest by guest. For each guest, check: did they ask for dessert? If no, move on. If yes, plate some jalebi and set it on the tray. When the table is done, the tray is done.

One instruction, two shapes. The folded shape — a single sentence naming what to collect, from where, on what condition — and the unfolded shape, a step-by-step procedure with a tray, a walk, a check, and an add. They are the same plan. Nobody at the wedding thinks the sentence is "advanced" and the procedure is "basic". They are just the compact and the expanded spelling of one idea.

Python has both spellings too. The unfolded one you already know: the for-loop. The folded one is called a comprehension, and it is the single most common reason framework code looks cryptic to a newcomer. Today we learn the fold.

One pattern, two spellings

Quick recap of where you stand. In the control-flow chapter you met the for-loop — do this once per item. In the four containers you met lists, dicts, and sets — the trays everything gets collected into. And in the functions chapter you learned to read any signature without flinching. Put the first two together and you get the most common four lines in all of Python, a shape worth naming: the collect pattern.

the collect pattern — unfolded
plates = []                        # 1. take an empty tray
for guest in table_four:           # 2. walk the table, guest by guest
    if guest.wants_dessert:        # 3. check the condition
        plates.append(make_jalebi(guest))   # 4. add to the tray

Empty container, loop, optional check, append. You will see this shape hundreds of times: collecting the user turns out of a message history, collecting the file names that end in .yaml, collecting the tools an agent is allowed to call. Python's designers noticed that too, and gave the pattern a folded spelling — one line that says the same four things:

the collect pattern — folded flat
plates = [make_jalebi(guest) for guest in table_four if guest.wants_dessert]

That is a list comprehension — a single expression that builds a whole list by transforming and filtering the items of some source. Nothing new happens inside it. It is the caterer's sentence: a plate of jalebi, for every guest at table four, who asked for dessert. Read it back to yourself with those commas and the syntax dissolves.

Note

The name "comprehension" comes from set-builder notation in mathematics, not from "understanding". Unfortunate, because the first time you see one, understanding is exactly what is missing. It comes fast, though — there are only three slots to learn.

The grammar: three slots and a bracket

Every list comprehension has the same anatomy, in the same order:

the anatomy
[ EXPRESSION   for ITEM in SOURCE   if CONDITION ]
#  what to        where the items      keep only the
#  collect        come from            ones that pass  (optional)

The reading trick — and this is the one thing to memorise in this chapter — is that you do not read it left to right. You read the middle first, then the end, then the front: "for each ITEM in SOURCE... keeping the ones where CONDITION... collect EXPRESSION." The middle tells you the walk, the end tells you the check, the front tells you what lands on the tray. Watch one execute, item by item:

Unfold it, fold it back

Here is the mechanical recipe, and it works on every comprehension you will ever meet. To unfold: the for part becomes the loop header, the if part becomes an if-statement inside it, and the expression at the front becomes the thing you append. To fold: run the recipe backwards. No judgement, no cleverness — pure origami. Prove it to yourself by running both spellings side by side:

Tip

When a comprehension in real code refuses to make sense, unfold it on paper. Write for, copy the middle. Write if, copy the end. Write append, copy the front. Thirty seconds, and the mystery is a four-line loop you have been reading since chapter 6. Fluent readers eventually stop needing the paper — but every fluent reader started with it.

Two ifs, two different jobs

Here is the one genuine trap in this chapter. There are two places an if can appear in a comprehension, and they do different jobs.

The if at the end is the one you have met: a filter. It decides which items get processed at all, and it never takes an else — a gate either lets you through or it does not.

But an if can also appear at the front, inside the expression slot. That one is a conditional expression — Python's one-line "this if that, otherwise the other", which picks between two values. In the expression slot it is not filtering anything; every item still comes through. It is choosing what to collect for each item, and it must have an else, because an expression has to produce something either way:

filter at the back vs. chooser at the front
vips = {"asha", "arjun"}

# FILTER — fewer items out than in; no else allowed
[g for g in guests if g in vips]
# ['asha', 'arjun']

# CHOOSER — same number of items out; else required
["VIP" if g in vips else "regular" for g in guests]
# ['VIP', 'regular', 'regular', 'VIP', 'regular', 'regular']

# BOTH at once — chooser at the front, filter at the back
["VIP: " + g if g in vips else g for g in guests if len(g) > 4]

Read the position, and you know the job: back of the line means who gets in, front of the line means what each one becomes. When you hit SyntaxError stories about comprehensions in code review, nine times out of ten someone put an else on a trailing filter, or forgot the else on a front chooser.

The whole family: dict, set, and the lazy sibling

Once the fold is in your fingers, Python lets you aim it at every container from chapter 4. Swap the brackets and you swap the tray:

The fourth card deserves a story of its own. A list comprehension is the halwai frying a full tray of jalebis in advance — all the memory is spent up front, whether or not anyone eats. A generator expression is the halwai frying one jalebi as each guest walks up — same recipe, round brackets instead of square, and nothing exists until something asks for the next item. That "something" is usually a function that consumes items one by one: sum(), max(), any(), ", ".join(). For a six-guest wedding the difference is invisible; for a six-million-row log file, the square brackets fill your memory and the round ones sip it.

Two small courtesies of the syntax: when a generator expression is the only argument to a function, you may drop its own parentheses — sum(len(m) for m in messages) rather than sum((len(m) for m in messages)) — which is exactly the form you will see in the wild. And a generator can be consumed only once; it is a conveyor, not a tray. If you need the items twice, you wanted a list.

Nesting: the cliff edge of readability

Comprehensions can nest. Two for clauses in one comprehension walk a list of lists, and the reading rule is mechanical: the clauses run left to right exactly as they would stack top to bottom in the unfolded loop. Left is the outer loop, right is the inner one:

flattening a grid — folded and unfolded are the same plan
tables = [["asha", "rohan"], ["meera"], ["arjun", "divya"]]

# folded
everyone = [guest for table in tables for guest in table]

# unfolded — read the clauses in the SAME order, top to bottom
everyone = []
for table in tables:          # first 'for' in the comprehension
    for guest in table:       # second 'for' in the comprehension
        everyone.append(guest)
# ['asha', 'rohan', 'meera', 'arjun', 'divya']

That flatten-a-nested-list shape is the one nested comprehension worth knowing on sight, because it appears everywhere — flattening batches of results, lists of pages of documents, rows of cells. Beyond it lies the cliff. Add a second condition, or a chooser, or a third for, and the sentence stops being a sentence:

technically legal
out = [f(x) if p(x) else g(x) for row in grid if row for x in row if x > 0]
Careful

The rule good teams actually enforce: one for plus at most one if reads as a sentence — keep it folded. Two fors is acceptable only for the plain flatten. Anything past that, write the loop. A comprehension exists to be easier to read than the loop; the moment it is harder, it has failed at its only job. Reviewers who reject a five-clause comprehension are not being pedantic — they are protecting the person reading it at 2 a.m. during an incident.

Real framework one-liners, decoded

Time to cash in. Each line below is the kind of one-liner that fills real agent and pipeline code — the sort of thing that used to read as hieroglyphics. Click each line and unfold it:

five one-liners from the wild
tools_by_name = {tool.name: tool for tool in tools}
Dict comprehension. "For each tool in the list, file it under its own name." Turns a list of tool objects into a lookup table, so the agent loop can later grab tools_by_name["calculate"] in one step. You will build exactly this dict yourself in the ReAct chapter.
context = "\n\n".join(doc.page_content for doc in docs)
Generator expression feeding join (the string glue from chapter 3). "For each retrieved document, take its text, and glue the lot together with blank lines." This is the line that stuffs retrieved documents into a prompt in nearly every retrieval pipeline. Round brackets omitted because it is the only argument.
messages = [m for m in messages if m["role"] != "system"]
Pure filter — the expression slot just passes m through untouched. "Keep every message that is not a system message." Note it builds a new list and re-points the name at it, rather than surgically deleting from the old one.
names = [t["function"]["name"] for t in response["tool_calls"]]
Transform with no filter, digging into nested dicts. "For each tool call the model returned, collect just its function name." Chained square brackets are dict lookups inside dict lookups — chapter 4, twice over.
flags = {k: v for k, v in config.items() if k.startswith("agent_")}
Dict comprehension with a filter. .items() yields key-value pairs; the comprehension keeps only the keys with the right prefix and rebuilds a smaller dict. The standard move for "give me just this component's settings out of a big config".

Notice what just happened: none of those five needed new knowledge. Each was the collect pattern wearing one of the outfits from this chapter — a filter here, a dict there, a generator feeding join. That is the whole trick. There is no sixth secret variant waiting to ambush you.

In the wild

Once you can read comprehensions, you will find them on nearly every screen of framework source: building message lists before an API call, filtering tool schemas, flattening batch results, prefix-filtering environment variables and config dicts. In merge requests they are a reliable little drama: someone posts a four-clause comprehension, a reviewer comments "please unfold this", and both of them are right — the author that it works, the reviewer that it should not need a whiteboard. And when you see sum(... for ...) or any(... for ...) with round brackets, you now know the author was being deliberately frugal: counting a million items without ever storing them.

Check yourself

1. What does [m for m in messages if m["role"] == "user"] produce?

The expression slot is just m — items pass through untouched — and the trailing if is a filter. So: same items, fewer of them. A comprehension always walks the whole source and always hands back a whole new collection.

2. In [cell for row in grid for cell in row], which for is the outer loop when you unfold it?

The clauses sit in exactly the order the unfolded loops would stack: for row in grid: on the outside, for cell in row: nested inside. That is also why row is already defined by the time the second clause uses it.

3. What is the practical difference between [len(d) for d in docs] and (len(d) for d in docs)?

Square brackets fry the whole tray up front. Round brackets set up a conveyor that produces each value only when something — like sum() — asks, and it can be consumed only once. Same recipe, very different memory bill on large data.

4. Why is [g for g in guests if g in vips else "regular"] a syntax error?

Two ifs, two jobs. The back-position if only decides who gets in — no third option, no else. To choose between two values per guest, move the conditional to the front: ["VIP" if g in vips else "regular" for g in guests].

5. What does {t.name: t for t in tools} build?

Curly brackets plus a colon in the expression slot means dict comprehension: key on the left of the colon, value on the right. It is the one-liner agent frameworks use so the loop can find a tool by the name the model asked for.

Key takeaways

  • A comprehension is the collect pattern — empty container, loop, check, append — folded into one line. Nothing new executes; it is the same plan in a compact spelling.
  • Read the middle first: "for each ITEM in SOURCE, keeping ones where CONDITION, collect EXPRESSION." Unfolding one on paper takes thirty seconds and never lies.
  • Two ifs, two jobs: a trailing if filters items and never takes else; an if/else at the front is a chooser inside the expression and must have else.
  • Brackets pick the tray: [ ] list, {k: v} dict, { } set, ( ) a lazy generator that produces items on demand and can be consumed once.
  • One for plus one if reads like a sentence; two fors is fine only for the plain flatten; anything denser deserves to be a loop again.
  • Framework one-liners — tool registries, prompt-stuffing joins, message filters, config slices — are all this one pattern in different outfits. There is no sixth secret variant.
