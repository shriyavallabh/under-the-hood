What a harness actually is

Model, framework, harness, platform — four words that get swapped for each other in every meeting you attend. One nested diagram untangles them for good, and after this chapter you will never mix them up again.

~25 minChapter 82 of 135
The story

At a family dinner, your uncle announces he has bought "a car". Everyone congratulates him. Then you visit his garage and find a gleaming V8 engine, on a wooden pallet, going nowhere. It is a magnificent brain of a machine — but an engine moves nothing by itself.

Over the next year the garage fills up. First a drivetrain kit: gearbox, axles, a frame that connects engine to wheels. Now it moves — but it is a go-kart, and nobody sane commutes on a go-kart. So next come the cabin, seatbelts, mirrors, a fuel gauge, door locks, headlights, a horn: everything that makes raw motion safe and usable by a person. Finally your uncle reveals the actual plan: an airport taxi business. Twenty cars, a dispatch desk, night-shift supervisors, a maintenance rota, a booking system.

Four purchases. Engine, drivetrain, car, fleet. Nobody at that dinner table would confuse them — yet in AI conversations, people use one word for all four. The engine is the model. The drivetrain kit is the framework. The whole car is the harness. The taxi company is the platform.

Four words, one confused conversation

Quick recap before the new material, because this whole Act stands on it. In the ReAct chapter you built an agent in fifty lines: a stateless model that only emits text, a dict of tools, and a loop that ferries tool results back into the transcript. In the LangChain chapters and the LangGraph chapters you saw frameworks package that same loop as importable code. And in deep agents vs Claude Code you glimpsed something bigger wrapped around the loop — planning, files, permissions, compaction. This Act names that bigger thing and dissects it organ by organ. Its theme is the loop around the loop.

Here is the confusion this chapter kills. You have heard sentences like these, possibly in the same meeting: "We're switching from LangChain to Claude." "Is Claude Code a model?" "Our platform uses GPT." Each one grinds different layers together, and the grinding has real costs — teams benchmark models when their actual problem is a missing permission system, or rip out a perfectly good framework because the harness around it was badly configured.

The fix is one picture: four layers, each wrapped around the previous one like an onion. Step through it.

1. Innermost: the model — a brain in a jar

Modeltext in, text out · stateless · no hands

Weights behind an API. It completes text and does absolutely nothing else: no memory between calls, no tools, no permissions, no opinion about where its output goes. This is the engine on the pallet — from chapter 24, remember, it is literally a file of numbers.

2. Wrap it in a framework — the loop you import

Frameworkloop + graph code you import
Model

Library code that turns one-shot completions into an agent: the ReAct loop, tool dispatch, graph wiring, checkpointing. LangChain's create_agent, LangGraph's StateGraph, deepagents' create_deep_agent. Crucially, a framework is inert — it runs only when your program calls it. The drivetrain kit.

3. Wrap that in a harness — the running system

Now it is a product that runs: something assembles the context each turn, decides which tools exist and which are allowed, persists memory across sessions, and gives humans a doorway in — a terminal, a chat window, an editor. This is the whole car, and it is what you actually operate every day.

One car is transport; twenty cars with dispatch is a business. The platform layer runs many harnessed agents: cron-style schedulers, watchdogs that restart crashed runs, per-user authentication, billing, dashboards. The taxi company.

Four layers, four different purchase decisions, four different failure modes. When something goes wrong in production, your first diagnostic question should be: which ring is misbehaving? A wrong answer is usually the model; a missing tool result is the framework loop; a permission prompt or amnesia is the harness; a run that never started is the platform.

Layer 1 — the model: the brain

Everything you learned in Acts III and IV applies here and only here. A model is stateless text-in, text-out — you met the raw API in your first API call. It cannot execute a tool, remember yesterday, or refuse to be swapped out. GPT-5, Claude Opus, Gemini, Llama: these names belong to this ring and no other. When someone asks "which model should we use?", they are asking about capability, cost per token, latency, and context window. Nothing else.

Layer 2 — the framework: the loop you import

A framework (or library — the words blur) is pre-written loop and graph code that you import into your own program: the agent loop with native tool calling, retries, streaming, state graphs, checkpointing. You have already used three: LangChain's create_agent, LangGraph's StateGraph with its nodes and edges, and deepagents' create_deep_agent from chapter 68. A framework ships no user interface, holds no API keys, runs on no schedule. It is scaffolding for your code.

Layer 3 — the harness: the body around the loop

Here is the word this Act is named for. A harness is the complete running system wrapped around the agent loop — the software that turns "a loop I could run" into "a system I do run, all day, safely". Concretely, a harness owns five jobs:

  • Context management — deciding what actually goes into the model's window each turn: system prompt, instructions files, trimmed history, compaction when the transcript bloats. The next chapter, context engineering, is entirely about this job.
  • Tool inventory — which tools exist at all: file access, shell, search, MCP servers plugged in from outside.
  • Permissions — which of those tools may run right now, and which need a human to say yes first. The approval prompt you click every day lives in this ring.
  • Memory — what survives between sessions: instruction files, notes the agent writes to itself, project state. (Recall memory vs context: context is the working window, memory is what persists outside it.)
  • Channels — the doorways humans use to reach the agent: a terminal, a chat app, an editor pane, a voice interface. Same loop, different front doors.

Claude Code is a harness. So is the ChatGPT app — around the same GPT models you can call raw over the API, it adds conversation history, memory, tools, and a chat channel. So is an AI-native code editor's agent mode. None of these are models, and none are frameworks: they are cars, not engines or drivetrains.

Layer 4 — the platform: the fleet

A platform is what you need the day one agent becomes twenty: schedulers that launch runs at 6 a.m. without a human, watchdogs that notice a hung agent and restart it, authentication so a hundred users each get their own sandboxed harness, quotas and billing, observability dashboards, and — increasingly — A2A endpoints so the fleet's agents can talk to each other. The hosted services that deploy LangGraph agents for you sit here, as does any enterprise setup where agents run in Kubernetes with cron jobs and log pipelines. Platform problems are fleet problems: "the nightly run didn't fire", "who approved this agent's spend", "restart the stuck ones".

Put the products you know on the map

The fastest way to make the layers stick is to sort familiar names into them. Illustrative, not exhaustive:

Things you hear aboutLayerThe tell
GPT-5, Claude Opus, Gemini, LlamaModelIt is priced per token and does nothing without a caller.
LangChain, LangGraph, deepagents, Microsoft Agent FrameworkFrameworkYou pip install and import it; it ships no running product.

Two wrinkles keep this interesting. First, one product can span rings: the ChatGPT app is a harness, but the company behind it also sells the raw model underneath, and its team-management features reach into platform territory. Second, a harness does not have to be built on a public framework — Claude Code implements its own loop rather than importing LangGraph. The rings describe roles, not vendor boundaries.

Careful

Marketing loves the word "platform" — every vendor with a landing page claims to be one. Ignore the label and run the test: does it complete text (model), do you import it (framework), do you work inside it (harness), or does it run fleets of agents for many users (platform)? What a thing does places it on the diagram; what its pricing page says does not.

The wrapping, line by line

The Harness class is small, but every line maps to something you click or configure daily. Read it like a reviewer:

harness.py — what each line is in real life
class Harness:
The loop is not the product; this wrapper is. Everything Claude Code does that a bare LangGraph script does not lives in classes shaped like this one.
self.allowed = allowed
The permission policy — the harness's most important organ. In Claude Code this is your settings allowlist plus the approval prompt; in an enterprise harness it is a policy file with an audit trail.
self.max_context = max_context
The context budget. Real harnesses do far more than slice a list — trimming, summarising, compacting — but the decision is the same: the window is finite, something must choose what fills it. That choosing is context engineering, next chapter.
self.memory = []
State that outlives one task. In real harnesses: instruction files, memory directories, project state. The model still remembers nothing — the harness remembers on its behalf.
def gate(self, name):
Every tool gets wrapped in a permission check before the loop ever sees it. The model cannot tell a gated tool from a bare one — the enforcement is invisible from inside the jar.
if name not in self.allowed:
The actual gate. One line here is the difference between "the agent read the invoice" and "the agent ran a destructive command at 2 a.m." Harness bugs at this line are security incidents, not glitches.
context = self.memory[-self.max_context:] + ["TASK: " + task]
Context assembly: memory plus fresh task, trimmed to budget. Note the order of authority — the harness decides what the model gets to know before the model gets a say in anything.
answer = agent_loop(self.model, guarded, context)
The entire framework layer is this one line. That proportion is honest: in production harnesses, the loop is a small fraction of the code, and the wrapping is the rest.
self.memory.extend(context)
Persistence after the run — what makes tomorrow's session smarter than a goldfish. Frameworks offer this too (recall checkpointing); the harness decides the policy of what is kept.

"Which harness?" is a different question from "which model?"

Now the payoff, and the sentence to carry into your next architecture meeting. When teams evaluate "AI", they usually reach for model benchmarks — but most of what they will live with day to day is decided by the harness. The two questions have almost no overlap:

Questions that pick a modelQuestions that pick a harness
How capable is it on our kind of task?Which tools can the agent reach, and how do new ones get added?
What does it cost per million tokens?Who approves a dangerous action, and is there an audit log?
How fast is the first token?How is the context window managed when the transcript bloats?
How large is the context window?Where does memory live, and what survives a restart?
Open weights or API-only?Which channels can our people use — terminal, chat, editor?

Two consequences follow. First, the same model behaves like two different employees in two different harnesses — one harness gives it your codebase, a scratchpad, and a shell; the other gives it a chat box and nothing else. When a colleague says "Claude was useless for this", the honest question is: in which harness, with which tools and which context? Second, harnesses generally let you swap the model inside — a config line, because the model is a component of the harness and never the other way round. Committing to a harness is the sticky decision: your permissions, memory, channels, and habits all grow into it. Committing to a model is, increasingly, a dropdown.

This is also why the chapters ahead matter more than another benchmark chart. The model race is loud, but your token bill, your safety, and your agents' apparent intelligence are mostly determined in the quiet ring around the loop. Act X walks that ring organ by organ: context engineering, memory, skills, durability, guardrails.

In the wild

You will use this diagram weekly to decode sentences. "We're standardising on one model" — an inner-ring decision that leaves every harness question open. "The platform team owns the schedulers" — outermost ring. A vendor pitching an "agent platform" that turns out to be an importable library — second ring wearing a fourth-ring costume; now you can say so. In repositories, harness code is everything around the loop: permission configs, context-compaction modules, memory directories, channel adapters — often outweighing the agent logic ten to one. And in job postings, "harness team" and "agent infrastructure" roles at AI labs are precisely layer-3 engineering: not making the brain smarter, but building it a better body.

Check yourself

1. A colleague says: "We're replacing LangChain with Claude." What is muddled about that sentence?

LangChain is loop code you import (ring 2); Claude is a model (ring 1). A LangChain agent can call Claude — the sensible follow-up question is which layer actually has the problem.

2. Which layer decides whether the agent may actually run a destructive shell command right now?

A model can only emit text, and refusal is not enforcement. The gate in the harness is the enforcement point — like the if name not in self.allowed line you just walked through, but production grade.

3. You run the identical model in two harnesses and get wildly different quality. Why is this the expected outcome?

The brain was identical; the bodies were not. One harness fed it your codebase and gave it hands; the other gave it a chat box. Same weights, different employee.

4. Nightly scheduled runs, a watchdog restarting crashed agents, and per-user authentication belong to which layer?

Schedulers, watchdogs, and multi-user auth are what you need when one agent becomes twenty — the taxi company, not the car.

5. What exactly does the framework layer ship?

A framework is inert scaffolding — create_agent, StateGraph, create_deep_agent. No UI, no schedule, no keys. The moment something is running with context management and permissions around it, you are looking at a harness.

Key takeaways

  • Four nested layers: the model (weights: text in, text out), the framework (loop and graph code you import), the harness (the running system around the loop: context, permissions, tools, memory, channels), and the platform (fleet machinery: schedulers, watchdogs, multi-user).
  • Claude Code and the ChatGPT app are harnesses; LangChain, LangGraph, and deepagents are frameworks; GPT-5 and Claude Opus are models; the fleet services that schedule and supervise many agents are platforms.
  • The rings describe roles, not vendors — one company can sell several rings, and a harness may implement its own loop instead of importing a framework.
  • Permissions, context assembly, and memory are harness jobs. When an agent misbehaves, first ask which ring failed before blaming the model.
  • "Which model" is about capability, cost, and latency; "which harness" is about tools, permissions, context management, memory, and channels — the sticky, everyday decision, since models are usually swappable inside a harness.
  • In production harnesses the loop is one line and the wrapping is the rest of the codebase — which is why this Act spends its remaining chapters walking the wrapping, organ by organ.
