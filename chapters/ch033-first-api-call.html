Your first API call

One HTTP POST, one JSON body, one JSON reply. That is the entire conversation between your code and a frontier model — and every framework you have ever operated is secretly built on this one exchange.

~30 minChapter 34 of 133
The story

There is a legendary kitchen in your city. The chef is a genius, but you never see her — the entire restaurant is one wall with a small hatch in it. You slide an order form through the hatch; a tray comes back. That is the whole interface.

The form is strict. Top corner: your membership number, because the kitchen bills by the word. Next line: tonight's house rules — "vegetarian, no cream, explain each dish briefly". Then the strange part: you must write out your entire conversation with the chef so far, labelled who-said-what, because the chef has no memory whatsoever. Every form starts her from zero. Forget to copy yesterday's exchange onto today's form, and yesterday never happened.

The tray comes back with the dish, plus a printed slip: how many words you sent in, how many words came back out. That slip is your bill.

The hatch is an HTTP POST. The form is JSON. The membership number is your API key, the house rules are the system prompt, and the who-said-what transcript is the messages array. Every SDK, every agent framework, every harness you have ever run is just a well-trained assistant filling in this form for you.

It was HTTP all along

The previous chapter toured the model landscape — who makes the frontier models and how to choose between them. This chapter is about the wire: what actually travels between your machine and theirs. And the answer is refreshingly boring. There is no magic tunnel, no special protocol, no persistent psychic link. Talking to a large language model is an ordinary web request — the same kind your browser makes when it loads a page.

Four words carry the mechanics. An API (Application Programming Interface) is a service designed to be called by programs instead of clicked by humans. HTTP is the delivery protocol of the web — a way to send a request to an address and get a response back. A POST is the flavour of HTTP request that carries a payload with it ("here is some data, do something with it"), as opposed to GET, which only fetches. And the payload itself is JSON — the curly-brace text format for structured data you met in the YAML and JSON chapter: keys, strings, lists, and nested objects.

Put together: your code builds a JSON document describing the conversation, POSTs it to a URL the provider publishes — called an endpoint, the specific address where a service listens — and receives a JSON document back containing the model's reply. That round trip is the whole thing. Claude Code does it. LangChain does it. Your production agents do it hundreds of times a day. Watch one call travel:

The messages array: who said what

The heart of the request body is a list called messages — a plain JSON array of the conversation so far, in order, oldest first. Each entry is a little object with two jobs: who said it and what they said. The "who" is called a role — a label telling the model which voice each message speaks in. Three roles do almost all the work:

a messages array with all three roles (OpenAI dialect)
[
  {"role": "system",    "content": "You are a concise assistant for a small business."},
  {"role": "user",      "content": "What is 20% GST on a 1,500 rupee invoice?"},
  {"role": "assistant", "content": "GST at 20% on 1,500 rupees is 300 rupees."},
  {"role": "user",      "content": "And on 4,000?"}
]

system is the house rules card: standing instructions about how to behave — persona, tone, format, boundaries. The model treats it with more authority than ordinary conversation, which is why every serious product buries its real behaviour in the system prompt. user is whatever the human (or the calling program) says. assistant is the model's own past replies.

That last one deserves a double take. The model does not remember saying those words — remember step 5, the server forgets everything between calls. The API is stateless. When you send an assistant message, you are handing the model a transcript and saying "this is what you said earlier". It rereads its own lines like an actor picking up a script. This has a delightfully strange consequence: the history is just data you construct, so you could edit it, trim it, or fabricate it entirely — and the model would be none the wiser. Chat products feel continuous only because their code faithfully replays the whole transcript on every single call, appending one new user message each time. The transcript grows, the request bodies grow, and the bills grow with them — the root economic fact of agent engineering.

Structurally, notice there is nothing exotic here: the array is a Python list of dictionaries the moment your code touches it — the same collections you already know — serialised to JSON for the trip across the wire.

Six words carry the rest of this chapter. Flip until each feels obvious:

Your key stays in the environment

The API key is a long secret string — sk-ant-api03-... for Anthropic, sk-proj-... for OpenAI — that identifies your account and authorises billing. It is, in the most literal sense, money: anyone who has it can spend on your account. Which is why there is one iron rule about where it lives.

It lives in an environment variable — a named value the operating system holds for a running program, which you met in the logging, config and env chapter. Your code asks the environment for the key at runtime; the key itself never appears in the source file:

the right way: key from the environment
# once, in your shell or your server's secret store
export ANTHROPIC_API_KEY="sk-ant-api03-..."
import os
api_key = os.environ["ANTHROPIC_API_KEY"]   # read at runtime, never written in code
Careful

Never paste a key directly into a source file — not even "temporarily". Code gets committed, commits get pushed, and a key that touches git history is compromised even if you delete it in the next commit, because history keeps the old version forever. Automated scrapers watch public repositories and will find a leaked key in minutes. If a key ever leaks: revoke it in the provider's console and mint a fresh one. The official SDKs cooperate with the rule — construct them with no arguments and they quietly read ANTHROPIC_API_KEY or OPENAI_API_KEY from the environment themselves.

A complete request, line by line

Time to read a real order form. This is a complete request body in the OpenAI chat-completions dialect — the shape most of the industry standardised on. It POSTs to https://api.openai.com/v1/chat/completions with two headers: Authorization: Bearer $OPENAI_API_KEY and Content-Type: application/json. Click each annotated line:

POST /v1/chat/completions — the request body
{
One JSON object is the entire request. Everything the model will know about this conversation lives inside these braces.
"model": "gpt-4o",
Which model answers. This is a routing label, not magic — the same endpoint serves many models, and providers rename them every few months, so real code keeps this in config, not hardcoded. Check the provider's docs for current names.
"max_tokens": 200,
A hard cap on the length of the reply, in tokens. A cost seatbelt: the model physically cannot bill you for more output than this per call.
"messages": [
The transcript begins. Oldest first, every turn, every time — because the server remembers nothing from your last call.
{"role": "system",
The house rules. In the OpenAI dialect the system prompt is just the first message in the array, wearing the system role.
"content": "You are a concise assistant for a small business."},
{"role": "user",
The human's first question — this already happened, one turn ago. We resend it so the model can re-establish context.
"content": "What is 20% GST on a 1,500 rupee invoice?"},
{"role": "assistant",
The model's own previous answer, played back to it. It has no memory of writing this — the transcript IS its memory, and you are the one holding it.
"content": "GST at 20% on 1,500 rupees is 300 rupees."},
{"role": "user", "content": "And on 4,000?"}
The one genuinely new line in the whole request. "And on 4,000?" is meaningless alone — it only makes sense because the three messages above it travelled too. This is why context matters and why long conversations cost more.
]
}

And the tray that comes back — a JSON response, roughly one second later:

the response body
{
"id": "chatcmpl-8x2Kq...",
A unique receipt number for this exchange. Production systems log it — when something goes wrong, this ID is what you quote to the provider.
"model": "gpt-4o",
Confirmation of which model actually answered.
"choices": [
An array of candidate replies. In practice it almost always holds exactly one — but the wrapper means the text you want is always one level deeper than you expect.
{
"message": {
"role": "assistant",
The reply arrives pre-labelled with the assistant role — deliberately shaped so your code can append it straight onto the messages array for the next turn. The loop closes itself.
"content": "GST at 20% on 4,000 rupees is 800 rupees."
The payload. The path your code walks to reach it — choices[0].message.content — is worth memorising; you will type it hundreds of times.
},
"finish_reason": "stop"
Why generation ended. "stop" = finished naturally. "length" = hit your max_tokens cap, so the reply is truncated — always worth checking. "tool_calls" = the model wants an action performed, the hook the whole of tool calling hangs on.
}
],
"usage": {
The bill. prompt_tokens counts everything you sent (and it grows every turn, because the transcript grows); completion_tokens counts the reply. Multiply by the per-token price and that is your cost for this call.
"prompt_tokens": 61, "completion_tokens": 14, "total_tokens": 75
}
}

Two dialects of the same language

Anthropic's API speaks the same fundamental grammar — POST, JSON body, messages array, roles — with a different accent. The same conversation, in the Anthropic dialect, POSTs to https://api.anthropic.com/v1/messages:

Anthropic request — POST /v1/messages
{
  "model": "claude-opus-4-8",
  "max_tokens": 300,
  "system": "You are a concise assistant for a small business.",
  "messages": [
    {"role": "user", "content": "What is 20% GST on a 1,500 rupee invoice?"}
  ]
}
Anthropic response
{
  "id": "msg_01XFDUDYJgAACzvnptvVoYEL",
  "type": "message",
  "role": "assistant",
  "content": [
    {"type": "text", "text": "GST at 20% on 1,500 rupees is 300 rupees."}
  ],
  "stop_reason": "end_turn",
  "usage": {"input_tokens": 32, "output_tokens": 17}
}

Spot the differences: the system prompt is a top-level system field instead of a message in the array; max_tokens is required, not optional; and the reply's content is a list of typed blocks rather than a single string — future-proofing for replies that mix text with tool calls and other block types. Note

Nobody hand-writes these POSTs in production. Each provider ships an SDK — an official library that wraps the raw HTTP: pip install anthropic gives you client.messages.create(model=..., max_tokens=..., messages=[...]), and pip install openai gives you client.chat.completions.create(...). Under the hood they do exactly what you just read — build the JSON, attach the headers, POST, parse — plus retries and error types. OpenAI also offers a newer Responses API alongside chat completions; different wrapper, same wire idea. Learn the wire format once and every SDK becomes transparent.

In the wild

This chapter's shapes are everywhere once you know them. Framework configs with model:, base_url:, and api_key: keys are just filling in this POST's blanks — LangChain's init_chat_model is a coat of paint over it. CI pipelines and deploy scripts export OPENAI_API_KEY / ANTHROPIC_API_KEY as environment secrets, never in code. Ops runbooks carry curl https://api.anthropic.com/v1/messages ... one-liners for smoke-testing whether "the model is down or just our code". Log lines shouting 401 mean a bad or missing key; 429 means rate-limited; finish_reason: "length" in a trace means someone's max_tokens was too tight and a reply got cut mid-sentence. And when you reach the ReAct loop, the mysterious line that "calls the model" will be exactly the call_model() you just ran.

Check yourself

1. When a framework "calls the model", what actually travels over the network?

One POST, one JSON body, one JSON reply — and the body must carry the entire conversation, because the server keeps nothing between calls. Every framework is a wrapper around this exchange.

2. Which role carries the standing instructions — persona, tone, rules of behaviour?

The system role is the house rules card, and the model treats it with more authority than ordinary conversation. OpenAI puts it inside the messages array; Anthropic gives it its own top-level field — same idea, different placement.

3. Why does turn two of a chat resend turn one's messages?

The server wipes the slate after every call. If turn one is not in turn two's messages array, then as far as the model is concerned, turn one never happened. That is also why input tokens — and cost — grow every turn.

4. Where should your API key live?

The key is spendable money. It lives in the environment (or a secrets manager) and the code reads it at runtime — never in source, because anything committed to git is in history forever and scrapers find leaked keys in minutes.

5. Your code just received a response from the Anthropic messages API. Where is the reply text?

Anthropic returns content as a list of typed blocks, so the text sits at content[0].text. The choices[0].message.content path is the OpenAI dialect — same language, different accent.

Key takeaways

  • Talking to an LLM is one ordinary HTTP POST: a JSON body goes to an endpoint, a JSON reply comes back. Every framework and harness wraps this single exchange.
  • The messages array is the conversation — a list of role-plus-content objects, where system sets the rules, user asks, and assistant is the model's own past replies played back to it.
  • The API is stateless: the server forgets you after every call, so your code resends the whole growing transcript each turn — which is exactly why long conversations cost more.
  • API keys are spendable money. They live in environment variables and are read at runtime; a key in source code or git history is a leaked key.
  • OpenAI and Anthropic are two dialects of one grammar: same POST-and-JSON idea, different endpoints, auth headers, system-prompt placement, and reply paths (choices[0].message.content vs content[0].text).
  • The usage section of every response is the bill — input and output token counts — and the finish/stop reason tells you whether the reply ended naturally or was cut short.
