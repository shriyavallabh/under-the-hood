A guided tour of GitLab

Groups, projects, the sidebar, the home screen — by the end of this chapter you will walk into any GitLab project your team owns and know exactly where everything lives.

~30 minChapter 99 of 123
The story

It is your first Monday at a huge office campus. Reception gives you a badge and a laminated map. The campus has dozens of buildings, one per department, and some buildings contain smaller wings for sub-teams. But here is the thing that makes the place navigable: inside every building, every office follows the same floor plan. Filing room on the left, where all the documents live. A front desk with a register of requests and complaints. A tray of proposals waiting for a manager's sign-off. A workshop humming behind glass, where work is assembled and tested. And a manager's cabin at the back, where the rules of the office are set.

By lunch you have visited exactly one office — yet you could now find the filing room in any of the four hundred others, because the layout never changes. GitLab is that campus. Buildings are groups, offices are projects, and the identical floor plan is the left sidebar. Learn one office and you have learned them all.

From your laptop to the shared home

Quick recap of where we are. In Git: the time machine you learned that git records snapshots of your work as commits, entirely on your own laptop. In Feature branches, release branches you learned that branches are parallel drafts of the same history. And from just enough git you know that git push sends commits to a remote — a shared copy the whole team can reach.

This chapter answers the obvious next question: where does that shared copy actually live? For most large organisations — and almost every bank — the answer is GitLab. Run this on any work laptop and you will see the connection:

terminal — the remote is just a GitLab address
$ git remote -v
origin  https://gitlab.yourbank.internal/data-science/agent-platform/fraud-triage-agent.git (fetch)
origin  https://gitlab.yourbank.internal/data-science/agent-platform/fraud-triage-agent.git (push)

Every git push you will ever run at work ends its journey inside a GitLab project. The rest of this chapter is a tour of what surrounds that landing spot.

Why GitLab, and not GitHub?

You have almost certainly seen GitHub — the public square of open source, home of LangChain, LangGraph and every framework this course has touched. GitLab is the same species of tool: a website wrapped around git remotes, with reviews, issue tracking and automation attached. GitHub even ships a self-hostable edition, GitHub Enterprise Server. So why do so many large regulated organisations — most banks among them — reach for GitLab instead? Three reasons.

First, self-hosting. GitLab is software a company can install on its own servers, behind its own firewall. Your employer's GitLab lives at an internal address — something like gitlab.yourbank.internal — and the source code never leaves the building. For a regulated business, "our code never touches someone else's cloud" is not a preference, it is a compliance requirement. A company-run copy of GitLab is called an instance — one installation of the software, wholly owned and administered by the company.

Second, built-in CI. GitLab shipped with a native automation engine — pipelines — from its early days. Push code, and the same application that stores it can automatically test, lint, build and deploy it, no add-on required. You will spend two whole chapters on this (Pipelines: .gitlab-ci.yml decoded), because pipelines are where your tests and agent evals actually run.

Third, one application for everything. Code, review, issues, milestones, CI, container registry — one login, one sidebar, one place for auditors to look. Enterprises like fewer moving parts.

Note

Everything you learn here transfers. GitHub and GitLab agree on about ninety percent of concepts and differ mostly in vocabulary — the famous one being that GitHub says pull request (PR) where GitLab says merge request (MR). Same object, different name on the door. If a colleague says "raise an MR", they mean exactly what a GitHub tutorial calls opening a PR.

Groups and projects: reading the campus map

GitLab organises everything into a hierarchy of exactly two kinds of thing. A group is a folder that holds projects and other groups — the building on our campus, usually named after a department. A group inside a group is a subgroup — a wing of the building. And a project is the office itself: one git repository plus everything that orbits it — issues, merge requests, pipelines, settings. When people casually say "the repo", in GitLab terms they nearly always mean the whole project.

The beautiful part: the hierarchy is spelled out in every URL you visit, readable like a postal address. Click through the lines below — the address of a single merge request, dissected:

anatomy of a GitLab URL — click each segment
https://gitlab.yourbank.internal
The instance — your company's own installation of GitLab. On the public internet this would be gitlab.com; at work it is an internal hostname reachable only from the office network or VPN.
/data-science
The top-level group. The building. Every project the department owns lives under this path, and permissions granted here flow down to everything inside.
/agent-platform
A subgroup — a wing of the building for one team. Groups nest several levels deep; big organisations mirror their org chart this way.
/fraud-triage-agent
The project. One git repository plus its issues, merge requests, pipelines and settings. This is the level your git remote points at.
/-/
A quiet separator. Everything before it names where you are; everything after it names which feature page you are viewing. It exists so a project named, say, issues can never collide with the issues feature.
merge_requests/42
The feature page: merge request number 42. Swap this segment for issues/7 or pipelines and you are in a different room of the same office.
Tip

This means you can navigate GitLab by editing the address bar, the way power users do. Lost deep inside a pipeline log? Delete everything after the project name and press Enter — you are back at the project's front door.

Careful

Your company's instance may hide entries an administrator has switched off, and version differences move the odd item around. Do not panic if a screenshot from a blog post does not match your screen pixel-for-pixel — the section names and the URL patterns are the stable part, and they are what this chapter teaches.

Tip

Press ? anywhere in GitLab to see the full keyboard-shortcut list. Even learning just two — g then m jumps to a project's merge requests, g then p to its pipelines — makes you look like you have worked there for years.

Everything so far was about one project. But on a workday morning your first question is personal: what needs me? GitLab answers it at the very top of the sidebar, next to your avatar, with three small counters — assigned issues, merge requests, and the To-Do List. Step through the morning routine:

The words on the doors

GitLab has its own dialect, and using it correctly is half of sounding fluent in your first week. Flip each card until the vocabulary feels natural:

In the wild

Listen for this dialect in your team's chat. "Can you look at !87?" means review my merge request. "It's tracked in #431" means the discussion lives on that issue. "The runner queue is backed up" explains why pipelines sit in pending before running. "It's slated for the March milestone" tells you when work is due. And when someone posts a bare GitLab URL, you can now read it before clicking: instance, group, subgroup, project, /-/, feature page. The moment you reply "approved !87, pipeline's green" without pausing to translate, you have stopped being a visitor.

Check yourself

1. A colleague says "raise an MR when you're ready". What are they asking for?

MR = merge request, GitLab's name for GitHub's pull request. Same object: your branch, proposed for merging, wrapped in a review conversation. It is written as !42, while issues are written as #42.

2. Why do banks and other regulated companies run their own GitLab instance instead of using a cloud service?

The decisive feature is self-hosting: the instance runs on company servers behind the company firewall, so code never leaves the building. Built-in CI and the single-application design sweeten the deal, but the compliance story is what closes it.

3. You want to see the open bugs and task discussions for a project. Which sidebar section do you open?

Plan is the front desk: issues, issue boards, milestones. Code holds the repository and merge requests; Build holds pipelines and jobs; Settings holds the rules. Same map in every project.

4. A pipeline has been sitting in "pending" for ten minutes. Which piece of the machinery is the bottleneck?

GitLab schedules jobs, but runners — the actual machines — execute them. When every runner is busy, jobs queue up as pending. "The runner queue is backed up" is a sentence you will hear, and now understand.

5. What is the relationship between a group and a project?

Groups are the buildings, projects are the offices. The hierarchy is visible in every URL: instance, then group and subgroups, then the project, then /-/ and the feature page.

Key takeaways

  • GitLab is the shared home your git push lands in — a website wrapped around remotes, plus reviews, issues and automation. Enterprises choose it because it can be self-hosted inside their own network and ships with built-in CI.
  • The hierarchy is groups (folders, often nested as subgroups) containing projects (one repository plus its issues, merge requests, pipelines and settings) — and every URL spells the whole path out, with /-/ separating the place from the feature page.
  • The left sidebar is the same in every project: Plan for issues and milestones, Code for the repository and merge requests, Build for pipelines and jobs, Settings for the rules. Learn it once, use it everywhere.
  • The project home screen is five bands — nameplate, branch switcher, last-commit line with its pipeline verdict, file list, README — and the little green or red pipeline icon is the fastest health-read of any project.
  • Your personal work lives in the counters by your avatar: assigned issues, merge requests (split into "assigned to you" and "review requests"), and the To-Do List — GitLab's inbox. Clear review requests first; they block other people.
  • Speak the dialect: MR not PR, !42 for merge requests, #218 for issues, runners execute pipeline jobs, milestones bundle work toward a date.
