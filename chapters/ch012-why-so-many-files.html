Why so many files?

One script was fine last week. Today the repo you opened has forty files and eleven folders. Here is the map — why engineers split code up, how the pieces link back together, and the five-minute routine that makes any strange repo navigable.

~30 minChapter 12 of 135
The story

You step off a night train in a city you have never visited, and you need milk, bread, and a toothbrush. You walk into a supermarket you have never seen — and ninety seconds later you are holding all three. Nobody gave you a tour. You simply knew: produce near the entrance, dairy along the back wall, toiletries in a middle aisle, checkout by the door. Every supermarket on Earth follows roughly the same layout, so a total stranger can navigate any of them.

Now picture the other shop — the tiny one from your childhood where everything lived in one glorious heap behind the counter, and only the owner knew that the batteries were under the biscuit tins. Brilliant while it was small. Unworkable the day the owner hired a second person.

A Python repository with forty files is the supermarket. A single do-everything script is the heap behind the counter. This chapter teaches you the aisle layout — so that any repo, from any team, becomes a place you can walk into cold and find things fast.

You already know the physics

Quick recap of the previous chapter, because this one is built directly on top of it. Every .py file is a module — a bag of names that Python fills by running the file top to bottom, exactly once. The import statement finds that file, runs it, caches the result in sys.modules, and binds a name in your namespace so you can reach inside. Import the same module ten times and it still only runs once, because the cache answers first. That is the entire physics of this chapter.

What was missing from that picture is the town planning: if every file can import every other file, how do real teams decide what goes where? Why does a production repo have a src folder, a tests folder, a pyproject.toml, three config files and a README, when the whole thing could technically be one long script? The answer is not about what Python demands — Python barely cares. It is about what humans and their tools need. Layout is a message from the authors to you.

The anatomy of a real project

Here is a small but completely realistic project — an email-drafting agent called agentmail. Read the tree slowly; the rest of the chapter keeps returning to it.

the agentmail repository, from the root
agentmail/                  # the repo root
  README.md                 # the welcome mat: what this is, how to run it
  pyproject.toml            # the signboard: name, dependencies, commands
  .gitignore                # files git must never track
  config/
    settings.yaml           # knobs and settings — data, not code
  src/
    agentmail/              # THE package: name matches the project
      __init__.py           # marks this folder as importable
      cli.py                # entry point: reads the command line
      agent.py              # the core loop
      tools.py              # functions the agent may call
      config.py             # loads settings.yaml into Python
  tests/
    test_agent.py           # proves agent.py behaves
    test_tools.py           # proves tools.py behaves

Four zones, four jobs. The root holds paperwork: the README that explains the project to humans, and pyproject.toml that explains it to tools. The src/ folder holds exactly one thing — the package, meaning a folder of modules that Python treats as one importable unit. The tests/ folder holds code whose only job is to check the package's behaviour. And config/ holds settings as data — YAML, dicts in costume — kept away from logic so an operator can change a knob without touching code.

Two details in that tree earn their strange looks. First, the file __init__.py: an often-empty file whose presence is the signal. It tells Python "this folder is a package — importing agentmail is allowed, and start here". When it does contain code, that code runs on the first import of the package, so teams use it to publish the package's public face. Second, the doubled name src/agentmail/: the outer src is a fence, not a package. It exists so that when you run tests, Python cannot accidentally import the loose files sitting in your working directory instead of the properly installed package — a classic source of "it works on my machine". You will see this src layout — code fenced one level down inside src/ — in most modern Python repos, and now you know it is deliberate, not decoration.

Now the signboard itself. pyproject.toml is the single most information-dense file in any modern Python repo — click through the lines that matter:

pyproject.toml — the project's signboard
[project]
A TOML section header — like a heading in a form. Everything under it describes the project itself. TOML is another dicts-in-costume format, a cousin of the YAML you met in chapter 5.
name = "agentmail"
The project's official name. Convention: the package folder under src/ carries the same name. When the two match, you always know where the real code lives.
version = "0.4.2"
requires-python = ">=3.11"
The minimum Python this project promises to work on. If your environment is older, installers refuse early instead of failing weirdly later.
dependencies = [
The shopping list: every outside library this project needs. Installers read this list and fetch each one. When a reviewer asks "why did we add a dependency?", this is the diff line they are staring at.
"httpx>=0.27",
"pydantic>=2.7",
]
[project.scripts]
The treasure map. This section creates real terminal commands when the project is installed.
agentmail = "agentmail.cli:main"
Read it right to left: in the package agentmail, in the module cli, there is a function main — and typing agentmail in a terminal calls it. One line answers the biggest question about any repo: where does this program start?
[build-system]
Plumbing for packaging tools — how to turn this folder into an installable artifact. You can safely skim past it while reading; chapter 20 unpacks it properly.
requires = ["hatchling"]
build-backend = "hatchling.build"

How the files hold hands

A folder of files is just a filing cabinet. What turns it into a program is the import lines at the top of each file: cli.py imports from agent.py, which imports from tools.py and config.py. Those lines are load-bearing — they are the corridors between the aisles, and together they form a one-way chain: files deep in the chain know nothing about the files above them. tools.py has no idea cli.py exists. That one-way flow is what lets you change the command-line handling without ever worrying about breaking a tool.

You can watch the whole mechanism run right here. The sandbox below holds a three-file mini-project — each string is the content of one file — and a tiny load() helper that plays the role of Python's import system, exactly as chapter 11 described it: make a module object, register it in sys.modules, run the file's code into it. Press Run.

Tip

Break it on purpose. Reorder the loading list to ["cli", "invoice", "pricing"] and run again — you get a ModuleNotFoundError, the very same error you met in the tracebacks chapter, because cli asked for invoice before it existed. Then change RATE to 0.05 in one file and watch every invoice line change — that is the point of splitting: one fact lives in one place.

Notice what the demo proves: the three files never mention each other's contents, only each other's names. Change how tax() works internally and neither invoice nor cli needs to know. That boundary — "you may know my name and my functions, never my insides" — is the entire art of splitting code well, and it is the same idea you will meet again next chapter as objects with a job.

The front door: where does it start?

A supermarket has many aisles but one entrance. A project with forty files also has one — the entry point, the function that runs first when a human or a scheduler starts the program. Everything else is only ever reached from there, import by import. Repos advertise their front door in up to three standard ways, and you have already seen the first:

One — the console script. The [project.scripts] line in pyproject.toml turns agentmail.cli:main into a terminal command. This is how installed tools work: when you type a command like pytest or uvicorn, you are calling exactly such a line.

Two — the runnable package. If a package contains a file named __main__.py, then python -m agentmail runs that file. The -m flag means "run this module or package as a program". The file is usually two lines:

src/agentmail/__main__.py
from agentmail.cli import main

main()

Three — the dual-use module. At the bottom of many files you will find the most famous idiom in Python:

src/agentmail/cli.py — the last three lines
def main():
    ...

if __name__ == "__main__":   # True only when run directly
    main()

Here is one piece of the module machinery chapter 11 did not cover: every module carries a __name__. When a file is imported, its __name__ is its module name, so the guard is false and nothing extra runs. When the same file is run directly — python cli.py — Python names it "__main__", the guard is true, and main() fires. One file, two personalities: a quiet library when imported, a program when executed. When you skim a strange repo, searching for these three signatures — project.scripts, __main__.py, and the __name__ guard — finds the front door in seconds.

Why engineers actually split code

None of this is for the computer's benefit. Python would happily run agentmail as one 2,000-line file, at the same speed. Teams split code for three human reasons, and each one will sound familiar from operating production systems.

Cohesion. Cohesion means each file has one job it does completely — pricing logic lives in the pricing file, and nowhere else. High cohesion is why the supermarket works: when everything about dairy is in the dairy aisle, you never have to search the whole shop. The test is brutal and simple: can you summarise the file in one sentence without the word "and"? "Loads settings from YAML" — good. "Loads settings and sends emails and formats dates" — that file is three files in a trench coat.

Testability. A small file with one job can be tested in isolation. tests/test_tools.py can import tools.py alone, feed it crafted inputs, and check outputs — no email account, no live model, no command line needed. This is why the tests/ folder so often mirrors the package structure file for file: each test file is the proof, and the worked example, for exactly one module. In the one-giant-script world there is nothing to import in isolation, so there is nothing to test in isolation.

Review size. Code changes ship through merge requests, and a reviewer's attention is the scarcest resource in a team. When code is split well, a bug fix touches one small file and its test — a diff a reviewer can genuinely hold in their head. When everything lives in one file, every change collides with every other change: bigger diffs, more merge conflicts, slower reviews, and a git history (chapter 21) where the answer to "who changed this and why?" is always "everyone, constantly".

Everyday taskOne 2,000-line scriptWell-split project
Find the pricing logicScroll and prayOpen pricing.py
Test the tax ruleRun the whole programRun one small test file
Review a bug fixDiff buried in a huge fileOne file, one test, done

Note

Splitting has a failure mode too: too many tiny files, each three lines long, so that reading any behaviour means a scavenger hunt across a dozen tabs. Good engineers split along job boundaries, not line counts. If you constantly need four files open to follow one thought, the project is over-split — a real critique you will see in code reviews.

Your first five minutes in a strange repo

Time to put it together. You have just been handed an unfamiliar repository — a client's agent, an open-source framework, anything. Here is the exact walk, the same one experienced engineers do on instinct. Step through it:

1. Start at the root — read, don't open code

README.mdwhat is this, how do I run it
pyproject.tomlname, dependencies, commands
src/
tests/
config/

Resist the urge to click into code. The root paperwork answers "what is this and what does it depend on?" faster than any source file. Two minutes here saves twenty later.

2. Find the entry point in pyproject.toml

README.md
pyproject.tomlagentmail = "agentmail.cli:main"
src/
tests/

The scripts section names the exact package, module, and function where execution begins. No scripts section? Look for a __main__.py, or a __name__ guard in a likely file.

3. Walk to the package that matches the name

pyproject.toml
src/agentmail/the real code — name matches the project
tests/
config/

Everything outside this folder is scaffolding — useful, but not the program. The doubled src/agentmail/ pattern tells you this team uses the standard fenced layout.

4. Open the entry module, then follow its imports

cli.pymain() lives here
→
agent.pyimported by cli
→
tools.pyimported by agent

Read in execution order, not alphabetical order. The import lines at the top of each file are your corridor map: they tell you which file to open next, and which files this one can never affect.

5. Use tests/ as worked examples

src/agentmail/agent.py
tests/test_agent.pyshows agent.py used correctly

Each test file demonstrates one module being called with realistic inputs and shows what output counts as correct. When prose documentation is thin, the tests are the documentation.

6. Consciously ignore everything else — for now

src/ + tests/ + pyproject.tomlyou have the whole picture
.github/, lock files, linter configsmachinery — visit when needed

Automation configs, lock files, and CI pipelines matter, but not on day one. Knowing what you are allowed to skip is half of navigating well.

The words on the door

Six terms from this chapter will come up in every repo and every code review from here on. Flip until each back feels obvious:

In the wild

Open any serious Python repository and this map holds. LangChain's repo carries a pyproject.toml per package with the code fenced below it; the MCP Python SDK keeps its package under src/mcp/ with a mirroring tests/ tree. In merge requests, seasoned reviewers read the file paths first — a diff touching only tools/search.py and its test is a Tuesday; a diff touching cli.py, agent.py, and pyproject.toml at once means the blast radius spans the whole program, so buckle up. And when an AI coding agent explores a repo for you, watch its tool calls: it reads the README and pyproject.toml, finds the entry point, follows imports — the exact six-step walk you just learned. Now you can audit whether it walked the aisles properly.

Check yourself

1. What conventionally marks a folder as an importable Python package?

The presence of __init__.py is the classic signal — and any code inside it runs on the package's first import. (Modern Python can also treat marker-less folders as "namespace packages", but well-run repos keep the explicit file.)

2. You type python -m agentmail in a terminal. Which file runs?

The -m flag runs a package as a program, and a package's program is its __main__.py. (The __init__.py also runs first, as part of importing the package — but __main__.py is the show.)

3. Fastest way to find where an unfamiliar project starts executing?

A line like agentmail = "agentmail.cli:main" names the exact package, module, and function where execution begins. No scripts section? Then check for a __main__.py or a __name__ guard.

4. Why do engineers split code into many files?

The interpreter is indifferent — splitting is for humans and their tools. One job per file means you can find it, test it alone, and review changes to it without wading through everything else.

5. In the sandbox, cli.py said from invoice import line. What actually happened?

Imports link to a module object, loaded once and cached in sys.modules — chapter 11's core lesson. That is why reordering the load list broke things: cli asked the cache for invoice before it existed.

Key takeaways

  • A repo's layout is a message to humans, not a demand from Python — root paperwork, package under src/, mirror-image tests/, settings as data in config/.
  • pyproject.toml is the signboard: the project's name, its dependency shopping list, and — in [project.scripts] — the exact function where execution starts.
  • Import lines are the corridors that turn a folder of files into one program; they flow one way, so deep files never depend on the files above them.
  • Three signatures mark a front door: a console script in pyproject.toml, a __main__.py for python -m, and the if __name__ == "__main__" guard for dual-use files.
  • Engineers split code for cohesion (one job per file), testability (test each piece alone), and review size (diffs a human can hold in their head) — never for speed.
  • Navigating a strange repo is a routine, not a talent: README and pyproject.toml, find the package, enter at the entry point, follow imports, let tests narrate, ignore the rest.
