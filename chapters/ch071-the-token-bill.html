The 2x token bill

Deep agents bill roughly twice what a plain loop does on the same task. This chapter follows the meter turn by turn — where every extra token goes, why the overhead is a feature, and a clean framework for deciding when double is a bargain.

~25 minChapter 72 of 123
The story

Your bathroom tap leaks. A handyman comes over, fixes it in an hour, and bills you 2,000 rupees. Simple job, simple invoice.

Now imagine you had called a full renovation firm instead. A site manager arrives first and walks the house, writing a plan on a clipboard. She opens a job folder and files her notes. She briefs a plumbing crew — and the crew gets its own complete induction: safety rules, site map, the whole folder read aloud. After the crew reports back, she updates the clipboard, checks the folder, and only then signs off. For the same tap, the firm's invoice reads 4,000 rupees — and not one line of it is padding. Every rupee bought coordination: the plan, the folder, the briefings, the reports.

For a leaky tap, hiring the firm is absurd. For gutting and rebuilding the whole house, the handyman is the absurd choice — he will lose track by day three. That is this chapter in one picture. Deep agents are the renovation firm. The question is never "which is cheaper?" It is "which jobs collapse without coordination?"

You have met the machinery — now meet the invoice

Quick recap of where we are in this act. Chapter 67 named the four things that make an agent "deep": a detailed system prompt, a planning tool, a file system, and subagents — and chapter 68 showed create_deep_agent wiring all four into one harness. Chapter 69 showed the write_todos planning tool keeping long tasks on the rails, chapter 70 gave the agent a virtual file system to park its working notes, and chapter 71 introduced the task tool that spawns fresh-context subagents. Each piece earned its keep on capability. This chapter asks the operator's question: what does all of that machinery cost?

The honest answer, and the number this chapter is named after: there is no single authoritative benchmark for "deep agent vs plain ReAct loop," and the real overhead swings hard with task size and subagent count — published head-to-head reports run anywhere from well under 2x to 20x or more, and Anthropic's own multi-agent research clocks in around 15x the tokens of a plain chat. This chapter deliberately picks "roughly 2x" as a modest, round working number — the toy meter further down lands near 1.9x on a mid-size task — because it is the cleanest figure for showing where the extra tokens go, not because it is a measured law. Treat it as an illustration, not a citation. The bars below use round, illustrative figures — the shape is what matters, not the last digit:

Remember the two facts that make any agent bill grow, from the tokens-and-cost chapter and the loop you built by hand: the model is stateless, so the entire transcript is re-sent on every single turn, and you pay for every token in both directions. Everything a deep agent adds — instructions, plans, file contents, subagent briefings — rides that same conveyor belt, again and again. Overhead in an agent is never a one-time fee. It is a per-turn tax.

Where the extra tokens go: an itemised receipt

Take that illustrative extra 100,000 tokens and split it into the four buckets every deep-agent bill contains. Again — round numbers, real proportions:

Bucket 1: the bigger system prompt

A plain loop's system prompt might be a few hundred tokens: "you are a helpful assistant, here are your two tools." A deep agent's system prompt must teach the planning protocol, describe the file tools (ls, read_file, write_file, edit_file), explain when and how to delegate via task, and lay out standards for doing all of it well. That easily runs to a few thousand tokens. Harmless as a one-off — except it is not a one-off. Statelessness means those thousands of tokens are re-sent on turn one, turn two, turn seventeen. A 2,600-token prompt over a 40-turn run is over 100,000 input tokens before the agent has done anything at all.

Bucket 2: planning turns

Every write_todos call is a full round trip that gathers no evidence and produces no answer. The model reads the whole transcript, writes a plan, and the plan itself joins the transcript — where you pay to re-send it on every later turn. Then the plan gets updated as items complete, which is another round trip each time. Chapter 69 showed why this is worth it — plans are the difference between a 100-turn run that finishes and one that wanders. But on the receipt, planning is pure coordination overhead: turns about the work rather than turns of the work.

Bucket 3: file operations

The virtual file system exists to keep bulk out of the context — park the 5,000-token research dump in a file, keep only the filename in the transcript. That genuinely saves tokens. But the plumbing itself costs: every write_file is a turn, every read_file is a turn, and the moment a file is read, its contents pour into the history and get re-billed on every turn after that. A deep agent that writes notes, reads them back twice, and edits them once has spent four round trips on filing. Good filing — but filing.

Bucket 4: subagent spawning — the big one

Here is the line item people miss. When the orchestrator calls task("research X"), the transcript shows one modest tool call and, later, one tidy summary. What actually happened in between is invisible: a whole second agent booted up, with its own copy of a big system prompt, its own loop, its own growing history, its own ten or twenty tool calls — every one of them billed. You pay for the entire hidden loop and see only the summary. Spawn three subagents and you are running four agents' worth of metered conversation to produce one answer. Context quarantine is why this is worth doing — the orchestrator stays lean and focused — but the quarantined context is not free context. It is someone else's bill stapled to yours.

Note

Notice what is not on the receipt: quality overhead. The deep agent does not think "better" per token. It spends its extra tokens on structure — plans, files, briefings — the same way the renovation firm spends money on a site manager who never lifts a wrench. Whether that structure pays for itself depends entirely on the job, which is where the decision framework below comes in.

So when is double a bargain?

Two properties of a task decide almost everything: how long it is (how many turns of real work) and how unpredictable it is (whether you can write the steps down in advance). Cross them and you get the whole decision framework:

Predictable stepsUnpredictable steps
Short task (a few turns)No agent at all — a fixed pipeline or a single prompt. Cheapest and most reliable.Plain ReAct loop. The 2x overhead would dwarf the work, like hiring the renovation firm for the tap.
Long task (tens of turns, hours)A workflow graph — you know the steps, so wire them explicitly and skip paying for improvisation.Deep agent territory. Research, multi-file builds, open-ended investigations. Here the coordination pays for itself.

This is the same instinct as "build the simplest thing that works", now with a price tag attached. And there is a second-order effect the table hides: the cost of failure. A plain loop that loses the plot at turn 40 — forgets the goal, repeats work, drowns in its own history (chapter 49's greatest hits) — does not bill you 1x. It bills you 1x for the failed run, then 1x again for the retry, plus your time diagnosing it. A deep agent that finishes correctly the first time at 2x is cheaper than a plain loop that finishes on the second attempt. On long, messy tasks, reliability is the cost optimisation.

Careful

The framework cuts both ways. If your task reliably completes in six turns of a plain loop, moving it to a deep agent because deep agents are fashionable does not make it better — it makes it slower and twice the price. Overhead you do not need is just overhead. Audit what your agents actually do before upgrading their architecture.

Four dials that pull the bill back down

Choosing the deep agent does not mean accepting the full 2x forever. Production teams routinely claw back a large share of the premium with four dials:

Dial 1 — prompt caching. The deep agent's biggest recurring cost is that huge, identical system prompt re-sent every turn. Provider APIs can cache a stable prompt prefix and bill subsequent sends of it at a steep discount — prompt caching, meaning the provider stores the unchanged opening of your request and charges a fraction (commonly around a tenth) of the normal input price when it repeats; check your provider's current pricing page for exact numbers. Because the prompt is fixed and the history only ever appends, agent traffic is close to the ideal caching customer. This one dial can quietly remove most of Bucket 1.

Dial 2 — model tiering. Nothing says every loop must run on the same brain. Model tiering means matching each loop to the cheapest model that does its job well: keep the frontier model where judgement lives — the orchestrator — and run subagents on a cheaper, faster tier when their work is mechanical (fetching, extracting, formatting, summarising). Since subagent spawning is the other dominant bucket, paying a fraction of the per-token rate inside those hidden loops takes a real bite out of the bill. Reserve same-tier subagents for delegated work that genuinely needs deep judgement.

Dial 3 — budgets. Your hand-built loop in chapter 46 had max_turns; a deep agent needs the grown-up version, at every level. Cap the orchestrator's turns, cap each subagent's turns, cap tokens per run, and alert when a run blows past its typical spend. A budget does not make the average run cheaper — it makes the worst run survivable, and in cost incidents it is always the worst run that writes the story.

Dial 4 — lean returns. Enforce the discipline that makes quarantine pay: subagents return short, structured summaries — never their transcripts; bulk results go to files, and the orchestrator reads files only when it must (every read pours the contents back into the metered history). The cheapest token in any agent system is the one that never enters the context.

In the wild

This chapter is the decoder ring for cost conversations. When a usage dashboard shows one agent run at 40x the median, the trace almost always reveals a subagent that spun in a hidden loop until a budget — or the absence of one — decided the outcome. When a merge request says "moved the tool schemas into the cached prefix" or "downgraded the extraction subagent to the small model", those are Dials 1 and 2 verbatim. In observability tools you will see per-run token columns where deep-agent runs sit at roughly double their plain siblings on the same workload — and now, instead of asking "why is this so expensive?", you can read the receipt: prompt re-sends, planning turns, file traffic, hidden loops.

Check yourself

1. A deep agent's system prompt is 2,600 tokens instead of 400. Why does that difference multiply instead of being a one-time cost?

Statelessness is the multiplier. A 2,600-token prompt over a 40-turn run is over 100,000 input tokens of pure instructions. This is also exactly why prompt caching (Dial 1) works so well — the re-sent prefix is identical every time.

2. In the orchestrator's transcript, a task() call looks like one small tool call and one tidy 350-token summary. Where did the real cost go?

Every subagent is a complete agent: its own big prompt, its own growing transcript, its own metered turns. You pay for the entire hidden conversation and only the summary comes home — that is why subagent spawning is typically the largest line on the receipt.

3. Which task profile most clearly justifies paying the deep-agent premium?

Short tasks do not amortise the fixed overhead, and predictable long tasks are better served by an explicit workflow graph. Long plus unpredictable is where coordination pays: a 2x run that finishes once beats a 1x run that fails and runs twice.

4. Which cost dial most directly attacks the re-sent-system-prompt bucket?

Budgets cap the worst-case run and lean returns keep the history slim, but neither touches the fixed prompt. Caching does: the prompt never changes between turns, which makes agent traffic nearly the perfect caching workload.

5. In the token-meter simulation, shrinking the task to three evidence calls pushed the ratio well past 2x. Why?

Plans, file setup, and subagent briefings cost roughly the same whether the job is tiny or huge. Spread over three working turns, that fixed cost dominates; spread over sixty, it fades — which is why the 2x folk number describes mid-size tasks and bends at both extremes.

Key takeaways

  • Deep agents cost roughly double a plain loop on the same task — not from waste, but from coordination: a bigger system prompt, planning turns, file traffic, and subagent spawning.
  • Because the model is stateless, every token in context is a per-turn tax: the big prompt and the plan are re-sent and re-billed on every turn of the run.
  • Subagent spawning is usually the largest line item — each task() call runs a complete hidden loop with its own prompt and history, and you pay for all of it while seeing only the summary.
  • Decide by task shape: short or predictable work belongs to prompts, pipelines, or plain loops; long and unpredictable work is where the 2x premium buys reliability — and a first-time finish at 2x beats a failed 1x run plus a retry.
  • Four dials claw the premium back: prompt caching for the fixed prefix, model tiering inside subagents, hard turn/token budgets against the catastrophic run, and lean summaries that keep bulk out of the metered context.
  • The ratio bends at the extremes — worse than 2x on tiny tasks (fixed overhead dominates), better on huge ones (fresh subagent contexts beat one ever-growing history).
