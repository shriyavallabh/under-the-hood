The story

Meera has run the output of a bottling plant for years — quotas, dashboards, quality reports. She knows exactly what the plant does. On the day she becomes plant manager, the chief engineer offers her a tour. "We can walk all nine hundred metres of the line," he says, "or I can show you how engineers actually learn a plant." He takes her to three places. First, the receiving dock — the one door every order enters through. Second, the wall map of the line — which stations exist, and in what order a bottle meets them. Third, a single station — the capper — which they shut down, strip to parts, and rebuild until Meera can explain it back to him. "That's it," he says. "You will never need to see every bolt. Every plant is a door, a map, and stations. Learn to find those three and no factory will ever be a mystery to you again."

Today the plant is the deepagents package. You will not read all of it. You will find the door, read the map, and rebuild one station with your own hands.

The manual tells you the promise; the panel tells you the truth

Quick recap of where you stand. In chapter 68 you met create_deep_agent and the powers it installs; chapters 68–70 took planning, the virtual filesystem, and subagents one at a time; chapter 72 priced it all; and last chapter you weighed the open-source blueprint against the commercial harness you use every day. Everything so far came from docs and this course. Today we go to the source itself — and the real prize is not deepagents trivia, it is the skill of reading any framework's insides.

Why bother, when the docs exist? Three reasons that matter to an operator:

First, docs describe intent; source describes fact — and only the source keeps up with what actually runs on your machine tonight.

Second, names drift. The docs talk about deepagents' "planning tool". The source has no class called PlanningMiddleware — the part is TodoListMiddleware, and it is not even defined in deepagents; it is imported from LangChain. Know only the marketing name and you cannot grep for the real one.

Third, incidents end in source code. When a run summarises itself at a strange moment, or a tool call gets mysteriously repaired, the answer is in a file, not a FAQ — and the engineers who get there fastest are the ones who can navigate, not the ones who memorised.

Note

Notice what the three passes deliberately avoid: reading a repository top-to-bottom like a novel. Real packages are mostly plumbing — imports, type declarations, error handling, backwards-compatibility shims. The passes route you around the plumbing, straight to the ideas.

Pass 1 — finding the front door

The quickstart line every deepagents example shares is from deepagents import create_deep_agent. That single line tells you the package name and the public entry point. Two facts from chapter 11 now pay off: a package's __init__.py is its lobby, re-exporting the names meant for the public; and the code you run sits on your own disk, because pip install put it there. You need no permission, and no GitHub, to read it:

terminal — locating the source you actually run
# Where is the installed package?
pip show deepagents
# Name: deepagents
# Version: ...
# Location: /your/venv/lib/python3.13/site-packages

# Which file defines the entry point?
grep -rn "def create_deep_agent" /your/venv/lib/python3.13/site-packages/deepagents/
# deepagents/graph.py: def create_deep_agent(

One grep, and the package snaps into focus: the front door is a function in a file called graph.py. The filename is itself a clue — it tells you the author thinks of the finished agent as a graph, which should ring a bell from Act VII. A quick glance at the folder gives you the floor plan:

site-packages/deepagents/ — condensed floor plan

Tip

Prefer the copy in site-packages over the GitHub default branch. GitHub's main branch shows you the future — whatever has merged for the next release. Your bug lives in the version pip show reports. During an incident, that difference is not academic.

Count what one shallow pass paid you: the default model, the vanish-with-the-run file default, the 85% compaction trigger, a janitor middleware no quickstart mentions, a recursion limit of 9,999. Five operational facts, none on the landing page, all from a single function.

Careful

Condensed means condensed. Most of what was trimmed is one theme: every subagent gets its own middleware stack, built by the same recipe, and the profiles/ folder lets each model family add or veto stations. And the roster drifts: earlier releases had no skills or memory stations, and borrowed their compactor from LangChain rather than shipping their own. That drift is exactly why the durable skill is the reading method, not a memorised list.

Pass 3 — one station, stripped to parts

Now the deep read. Choosing which unit to strip is a skill in itself: pick the smallest complete one. Here that is the planning middleware — the whole file is about 350 lines, and much of that is prompt text and async twins. When you open any middleware file, you are hunting the answers to exactly three questions:

One: what tools does it register? Two: what state does it add? Three: which hooks does it override? Those three answers ARE the middleware; everything else is finish work. Here is the shape you will find, condensed from the real file:

langchain/agents/middleware/todo.py — the shape (~350 lines in full)

Answer to question three. It overrides the switchboard hook from chapter 56 (plus its async twin, awrap_model_call). The real signature carries type generics; the shape is what matters.

This is where the "always keep your plan updated" behaviour comes from — not model magic, just prompt text injected on every model call. Seeing this cures a common superstition: planning is a convention the middleware enforces, not an ability the model was born with.

Hand the modified request onward. The middleware never calls the model itself; it decorates the call and steps aside.

Three answers: one tool, one state field, one hook. A feature that sounded architectural — "deep agents can plan" — turns out to be about thirty lines of live logic wrapped in prompt text. That is the recurring experience of reading source: features shrink to their true size.

Check yourself

1. You read to the bottom of create_deep_agent. What does it actually return?

The source shows a list of middleware being assembled and then handed to the same create_agent from Act VI. A deep agent is configuration, not new machinery — that is the chapter's central sighting.

2. The docs talk about deepagents' "planning tool", but grep finds no PlanningMiddleware. What is the lesson?

Names drift between marketing, docs, and code. The feature is real; its implementing class has a different name and even lives in a different package. Only the source gives you the greppable truth.

3. Inside the factory, TodoListMiddleware() is appended with no condition, while SkillsMiddleware sits inside if skills is not None:. What does this pattern tell you?

Conditional append is the standard shape of an optional feature in a composition-style framework: pass the argument, get the station. Reading the factory's if-statements tells you exactly which features your configuration actually enables.

4. You open an unfamiliar middleware file. What three things do you hunt for first?

Those three answers are the middleware — everything else in the file is prompt text, typing, and finish work. TodoListMiddleware resolved to one tool, one state field, one hook.

Key takeaways

  • Read frameworks in three passes: find the front door (the quickstart import, followed through __init__.py with grep), read the composition shallowly, then strip one small unit completely.
  • create_deep_agent assembles a middleware list — planning, filesystem, subagents, summarization, a tool-call janitor, prompt caching, plus conditional extras — and returns LangChain's create_agent with the recursion limit raised to 9,999. A deep agent is configuration of the standard loop, not a new engine.
  • Names drift: the docs' "planning tool" is the source's TodoListMiddleware, and it lives in LangChain, not deepagents. Grep for the part, not the promise.
  • To read any middleware, answer three questions — what tools, what state, which hooks. The planning feature resolves to one tool, one state field, and prompt text injected via wrap_model_call.
  • The code you are debugging is in site-packages, at the version pip show reports — not on the GitHub default branch.
  • Understanding is proven by reconstruction: if you can rebuild the pattern in sixty toy lines, the full package becomes your toy plus edge cases.
