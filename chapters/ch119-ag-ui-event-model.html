The AG-UI event model

The sixteen event types, the SSE stream, the RunAgentInput lifecycle, and a real event trace read line by line.

~28 minChapter 119 of 133
The story

Two people are following the same cricket match. One waits for the newspaper the next morning and reads a single line: India won by six wickets. True, complete, and utterly dead. The other has the radio on, and hears the whole living thing arrive as it happens — "he's coming in to bowl… full toss… flicked away… they're running… that's four!" — a stream of tiny, ordered events that, stacked up, become the match. Same final score. Wildly different experience.

Every agent you operate can talk to its user in either voice. It can go quiet for thirty seconds and then dump one finished answer — the newspaper. Or it can narrate its work as it unfolds: starting… writing… calling a tool… got the result… updating what I know… done. That live play-by-play is what a user-facing agent needs, and it turns out you cannot improvise it per app — the commentary needs a shared vocabulary of moves. AG-UI is that vocabulary. Roughly sixteen named event types, streamed in order, that let any agent backend narrate its work to any frontend. This chapter reads the commentary line by line.

Streaming, but with a grammar

Recall two ideas you already own. In the streaming chapter you saw that an LLM does not hand back a finished paragraph — it emits tokens (word-fragments) one at a time, and a stream lets the UI show them as they land instead of spinning for ten seconds. And in tool calling you saw the model does not run anything itself; it emits a structured request — a tool name and JSON arguments — and your code executes it. AG-UI takes both of those raw dribbles and gives them a grammar: it says which token belongs to which message, when a tool call has started and finished, and how the shared state changed — all as discrete, typed, ordered events on one stream.

AG-UI — the Agent-User Interaction Protocol — is an open, event-based protocol for exactly this: how an agent backend streams its running activity to a user-facing frontend, and how the frontend feeds state and actions back. It completes the trio you met earlier in the book: MCP connects an agent to its tools, A2A connects an agent to other agents, and AG-UI connects an agent to the human at the screen. Tools below, colleagues across, the user in front.

One honesty note up front, because this act refuses to sell you anything. AG-UI is a protocol, not an engine. It standardises the stream of events between agent and UI — nothing more. It does not give you orchestration, a control loop, branching, checkpointing, or durable state; you still build or bring those yourself. What it buys a team is real but narrow: instead of inventing a bespoke message format for every product and rewriting the frontend each time the agent changes, you emit standard events and any AG-UI-aware UI just renders them — which is why a team reaches for it rather than dragging in a heavy graph framework just to get a chat box on the screen.

Sixteen events, five families

The full protocol is about sixteen standard event types, and nobody memorises them as a flat list. They fall into five families, and once you see the families the individual names are obvious. Every event, whatever its family, shares the same base shape: a type (which event this is), an optional timestamp, and an optional rawEvent (the original provider event it was translated from, kept for debugging). Everything else is family-specific.

A naming rule you will lean on constantly: the wire type value is always SCREAMING_SNAKE_CASE — all capitals with underscores, like TEXT_MESSAGE_CONTENT — while every other JSON field name is camelCase — first word lowercase, later words capitalised, like messageId. So a text chunk is {"type": "TEXT_MESSAGE_CONTENT", "messageId": "…", "delta": "…"}. Types shout; fields whisper. Get that habit and half the protocol reads itself.

Three details worth pinning down before we watch them in motion:

First, the *_CHUNK variants (TEXT_MESSAGE_CHUNK, TOOL_CALL_CHUNK) are a compact convenience: one event that folds start, content and end together for backends that prefer a single combined frame over the three-part sequence.

Second, STATE_DELTA does not carry the new state — it carries a JSON Patch (the RFC 6902 standard): a tiny array of operations describing how to change the state you already hold. Each operation is an object like {"op": "replace", "path": "/cart/total", "value": 4200} — an op (add, replace, remove…), a path (which field, as a slash-separated pointer), and a value. Sending the patch instead of the whole object is what makes live state cheap: you ship the diff, not the document. A periodic STATE_SNAPSHOT ships the whole object to resynchronise.

The lifecycle: one POST, one stream back

Now the shape of a whole exchange. A run in AG-UI is not a chat socket that stays open forever — it is one request and one streamed response, like ordering the commentary for a single innings. The frontend gathers everything the agent needs into a single input object, POSTs it, and the backend answers with a stream of events that ends when the run does.

That input object is the RunAgentInput — the payload the frontend sends to start a run. It carries six fields: threadId (which conversation this belongs to, so turns link up), runId (this specific run's id), messages (the conversation so far), state (the current shared state the UI holds), tools (the tools the frontend is willing to let the agent use), and context (extra situational data). The backend reads it, does its work, and streams BaseEvents — the umbrella type every event inherits from — back until it emits RUN_FINISHED, or RUN_ERROR if something broke.

The transport: SSE by default

How do those events physically travel? By default, over SSE — Server-Sent Events, a plain HTTP mechanism where the server holds one response open and pushes text frames down it as they happen. The response's Content-Type is text/event-stream, and each event is serialised as one line of the form data: {json} followed by a blank line. That is the whole format: data:, the event as JSON, then \n\n to mark the boundary. Your browser's EventSource already speaks it; no special client library is strictly required, though the reference one is called HttpAgent.

AG-UI is deliberately transport-agnostic: SSE is the friendly default, but the same events can ride a high-performance binary HTTP encoding for production throughput, or WebSockets, or webhooks for long jobs. The events are the contract; the pipe is a choice. Now let us read an actual stream. Click the lines that carry meaning — this is one complete run for a support agent asked "where is my order A-204?", exactly as the bytes arrive:

the SSE stream — one run, top to bottom (blank line after each event omitted)
data: {"type":"RUN_STARTED","threadId":"t_42","runId":"r_a2f9"}
The stream's first event, always a lifecycle open. It echoes the threadId and runId from the RunAgentInput, tying this stream to the right conversation.
data: {"type":"TEXT_MESSAGE_START","messageId":"m_1","role":"assistant"}
A new assistant message opens and gets an id, m_1. Every following text delta will carry this same messageId — that is how the UI knows which bubble to append to when several things stream at once.
data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"m_1","delta":"Let me "}
The first token fragment. delta is the new slice of text, not the whole message — the UI concatenates deltas onto message m_1. This is the streaming you met in chapter 41, now with an id attached.
data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"m_1","delta":"check that order."}
Another delta on the same message. On screen the bubble now reads "Let me check that order." — still open, no END yet.
data: {"type":"TOOL_CALL_START","toolCallId":"tc_7","toolCallName":"lookup_order"}
The agent wants a tool. A tool call opens with its own id, tc_7, and the tool's name — a separate lifecycle from the text message, keyed by toolCallId, which is why the two can interleave on one stream without confusion.
data: {"type":"TOOL_CALL_ARGS","toolCallId":"tc_7","delta":"{\"order_id\":"}
The tool's JSON arguments stream in as deltas too, just like text — here the first fragment. A UI can show "looking up order…" the moment it sees the name, before the arguments finish.
data: {"type":"TOOL_CALL_ARGS","toolCallId":"tc_7","delta":"\"A-204\"}"}
The rest of the arguments. Concatenated, the deltas spell {"order_id":"A-204"} — the full request the agent is making.
data: {"type":"TOOL_CALL_END","toolCallId":"tc_7"}
The request is complete. In AG-UI the tool itself is usually run by the backend (or handed to the frontend for a client-side tool); either way, END means "arguments finished, go execute."
data: {"type":"TOOL_CALL_RESULT","toolCallId":"tc_7","messageId":"m_2","content":"status=shipped; eta=Fri"}
The result comes back tied to the same toolCallId, so the UI can pair it with the request it saw earlier. It also gets its own messageId because a tool result is itself a message in the transcript.
data: {"type":"STATE_DELTA","delta":[{"op":"replace","path":"/lastOrderStatus","value":"shipped"}]}
A JSON Patch, not a new state object. This one operation replaces the lastOrderStatus field. The UI applies the patch to the state it already holds — a side panel showing order status updates instantly, without the agent resending the whole object.
data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"m_1","delta":" It shipped and arrives Friday."}
Back to message m_1 — still open from before the tool call. The agent resumes the same bubble now that it knows the answer. This is the payoff of id-keyed streams: the assistant's sentence wraps around the tool call cleanly.
data: {"type":"TEXT_MESSAGE_END","messageId":"m_1"}
Message m_1 is closed. Its final assembled text is "Let me check that order. It shipped and arrives Friday." The UI can now mark the bubble complete.
data: {"type":"RUN_FINISHED","threadId":"t_42","runId":"r_a2f9"}
The terminal lifecycle event. The run is over, the SSE stream closes, and the frontend knows no more events are coming. Had the tool thrown, you would see RUN_ERROR here instead.
Note

See how the text message and the tool call overlapped? Message m_1 opened, streamed, paused while tool call tc_7 ran, then resumed — all on one stream. That only works because each lifecycle is framed by its own id (messageId vs toolCallId). The events are not a single sequential story; they are several interleaved threads that a reducer sorts out by id. Which is exactly what you will build next.

The frontend is a reducer

Here is the mental model that makes AG-UI click. The stream is a flat list of small events; the UI you see is the accumulation of them. A reducer — a function that folds a sequence of events into a running state, one event at a time — is what does the accumulating. Feed it TEXT_MESSAGE_CONTENT, it appends a delta to the right message. Feed it STATE_DELTA, it applies the patch. This is the same shape as the ReAct loop's history-building, just driven by typed events instead of parsed strings.

The real reducer lives in React (via CopilotKit — that is the next two chapters) and React cannot run here, so we will do the honest thing: simulate the reducer in pure Python. It consumes the exact event list from the trace above and reconstructs the assembled message and the final state — proving that the stream really does fold down to a coherent result. Press Run, then change a delta and watch the assembled message change.

Tip

Break it to learn it. Delete the TEXT_MESSAGE_END line — notice the message still assembles fine, because the reducer keys on messageId, not on order. Add a second STATE_DELTA with {"op":"add","path":"/eta","value":"Fri"} and watch the state grow one field. Change a RUN_FINISHED to RUN_ERROR and add a branch that sets status="error" — you have just handled the failure path every real UI must handle.

Who made it, and how mature is it

Provenance matters for a protocol, so be clear-eyed. AG-UI comes from CopilotKit — the company behind a widely used open-source React library for building in-app AI copilots. They extracted the agent-to-frontend streaming contract they had built for their own product and published it as an open, framework-neutral protocol, with a specification, a Python SDK (ag_ui.core and an EventEncoder), and TypeScript packages. The pitch mirrors the rest of the trio: a small, open event vocabulary anyone can implement on either side, so backends and frontends stop being welded together.

And the honest maturity check this act promised you. AG-UI is the youngest of the three protocols in this book. MCP and A2A each have neutral-foundation backing and dozens of independent implementations; AG-UI is newer, more tightly tied to a single company's ecosystem, and its event set is still settling — remember the THINKING_* events removed and replaced by REASONING_* at 1.0. It has genuine traction: official integrations with LangGraph, CrewAI, Mastra, Agno, Pydantic AI, LlamaIndex and more, and it can equally front a raw LLM loop with no framework at all. But "standard" here means "a good, open, adopted spec," not yet "an industry-wide inevitability" the way HTTP is. The concepts — event families, the SSE stream, RunAgentInput, state deltas — are durable even if a field name shifts. Learn the shape; check the current docs for the exact spelling.

In the wild

You will meet AG-UI first in a browser's network tab: open the dev tools on an agent app and watch a single request to the agent endpoint stay pending while data: {"type":"TEXT_MESSAGE_CONTENT"…} lines tick down it — that is the play-by-play, live. In code, the tells are from ag_ui.core import … and an EventEncoder on the backend, a runtimeUrl on the frontend, and functions named reduce or applyEvents that switch on event.type. In a merge request, "emit a STATE_DELTA when the cart changes" means: send a JSON Patch of just the changed field so the side panel updates without a full refresh. And when someone says "the run never sent RUN_FINISHED and the spinner hangs forever," you now know exactly which terminal event went missing.

Check yourself

1. On the wire, a text delta looks like {"type": "TEXT_MESSAGE_CONTENT", "messageId": "m_1", "delta": "hi"}. Why is TEXT_MESSAGE_CONTENT in capitals but messageId in camelCase?

AG-UI's rule is consistent: the type value shouts in SCREAMING_SNAKE_CASE, and every field name (messageId, threadId, toolCallId, delta) is camelCase. It is a spelling convention, not a meaning.

2. A STATE_DELTA arrives carrying [{"op":"replace","path":"/cart/total","value":4200}]. What should the frontend do with it?

STATE_DELTA carries a JSON Patch (RFC 6902): a list of {op, path, value} operations describing how to change existing state. Shipping the diff instead of the whole object is what makes live state cheap. A STATE_SNAPSHOT is the full object.

3. In the trace, a text message and a tool call overlapped on the same stream — m_1 paused, tc_7 ran, then m_1 resumed. Why does that not confuse the UI?

The stream is several interleaved threads, not one linear story. Because every text event carries a messageId and every tool event a toolCallId, a reducer can append each delta to the right place regardless of interleaving.

4. What does the frontend send to start a run, and what are its core fields?

A run begins with the frontend POSTing a RunAgentInput carrying the thread and run ids, the conversation, the current shared state, the permitted tools, and extra context. The backend answers with a stream of events until RUN_FINISHED or RUN_ERROR.

5. A teammate says "AG-UI will replace our LangGraph orchestration." What is the accurate correction?

AG-UI standardises the event stream between an agent backend and a frontend — nothing more. It deliberately does not provide orchestration, branching, checkpointing or a loop. It sits alongside a framework (or a raw loop), it does not replace one.

Key takeaways

  • AG-UI is the agent-to-user protocol — the third of the trio (MCP for tools, A2A for agents, AG-UI for the human at the screen) — and it standardises how a backend streams its live work to a frontend as typed, ordered events.
  • Roughly sixteen events fall into five families: lifecycle (RUN_/STEP_), text (TEXT_MESSAGE_), tools (TOOL_CALL_), state (STATE_SNAPSHOT/STATE_DELTA/MESSAGES_SNAPSHOT), and special (RAW/CUSTOM). Every event shares a base of type + optional timestamp + optional rawEvent.
  • Spelling rule: the type value is SCREAMING_SNAKE_CASE; all other JSON fields are camelCase (messageId, threadId, runId, toolCallId, delta). STATE_DELTA carries a JSON Patch (RFC 6902) of {op, path, value} operations, not a whole new state.
  • A run is one POST and one stream: the frontend sends a RunAgentInput (threadId, runId, messages, state, tools, context); the backend streams BaseEvents over HTTP+SSE (text/event-stream, each event a data: {json} line) until RUN_FINISHED or RUN_ERROR. A binary transport exists for production, and the protocol is transport-agnostic.
  • The frontend is a reducer: it folds the flat event stream into assembled messages and mirrored state, keying on messageId and toolCallId, which is why interleaved text and tool events sort themselves out cleanly.
  • Honest status: AG-UI comes from CopilotKit and is the youngest, most single-vendor-associated of the three protocols, with an event set still settling (REASONING_* replaced the removed THINKING_*). It is a streaming protocol, not an engine — learn the durable shape, and verify exact field names against the current docs.
