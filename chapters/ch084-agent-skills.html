Agent skills: SKILL.md

Tools give an agent hands. Skills give it training. A skill is a folder with a markdown file inside — and that humble format has quietly become the way you teach any agent a procedure, in any harness, without writing a line of code.

~25 minChapter 85 of 117
The story

Six a.m. in the kitchen of a serious restaurant. A new chef starts today. Nobody sits her down to recite three hundred recipes — that would take a week and she would forget most of it by lunch. Instead, the sous-chef points at a box on the shelf: laminated cards, one per dish. Each card has a name on its spine — biryani, consommé, sourdough — and one line beneath it saying when the card matters. She scans the spines in two minutes and gets to work.

When the first biryani order lands, she pulls that one card. Numbered steps. A warning about resting the rice. A note pointing to the spice-blend sheet clipped behind it, which she flips to only when she reaches step four. A small measuring scoop taped to the back — she uses it without needing to know how it was made. She never opens the other 299 cards. The head chef who wrote them left years ago, yet every cook since has cooked exactly like him.

That card box is a skills directory. Each card is a SKILL.md. And "scan the spines, pull one card, flip to the clipped sheet only at step four" has a name: progressive disclosure.

What a skill actually is

A skill is a folder containing a markdown file called SKILL.md — a written procedure that teaches an agent how to do a specific job, loaded into context only when that job comes up. No code is required. If you can write a clear checklist for a new colleague, you can write a skill.

Remember the frame from the harness chapter: the harness is everything wrapped around the model — the loop, the tools, the context plumbing. Skills are the harness's training library. And recall tools: a tool is a function your code is willing to execute on the model's behalf. A skill adds no new capability at all. It is choreography — it tells the model which of its existing tools to use, in what order, with what cautions, to a standard you wrote down once. The recipe card does not invent knives; it tells you where to make the cut.

On disk, a typical skill looks like this:

the folder layout
release-notes/
  SKILL.md               # the card: frontmatter + procedure
  references/
    tone-guide.md        # extra reading, pulled in only if needed
  scripts/
    collect_commits.sh   # runnable helper — executed, not read

Only SKILL.md is mandatory. The references/ and scripts/ folders (the names are convention, not law) hold whatever the procedure leans on: long reference documents the model reads on demand, and helper scripts it can run without ever loading their source into context — the measuring scoop taped to the back of the card.

Anatomy of a SKILL.md

The file itself has two parts. At the top sits frontmatter — a small block of YAML (the human-friendly config format from chapter 5) fenced by --- lines, holding structured metadata. Below it comes the body: ordinary markdown, the procedure itself. The open standard requires exactly two frontmatter fields: name (lowercase, hyphenated — the spine of the card) and description (what the skill does and when to use it — the line beneath the spine). # Writing release notes
The body begins — plain markdown, loaded into context only when the skill is triggered. Headings, lists, and short imperative sentences work best: you are writing for a very fast reader with a very expensive attention span.
Follow these steps in order.
1. Run scripts/collect_commits.sh to list merges since the last tag.
Step one delegates to a bundled script. The model executes it with its shell tool — it never reads the script's source, so a 200-line helper costs zero context. Deterministic work belongs in scripts; judgement belongs in prose.
2. Group the changes into Added, Fixed, and Changed.
3. Rewrite each item for a customer: benefit first, no ticket numbers.
This is the institutional-knowledge part — the judgement call a veteran would make automatically and a newcomer (or a bare model) would fumble. Encoding taste like this is most of what makes a skill valuable.
4. Match the voice described in references/tone-guide.md.
A pointer, not a paste. The tone guide might be two thousand words; the model opens it only if it gets this far. This is the third level of progressive disclosure — more on that in a moment.
5. Show the user the draft. Never publish without approval.
Skills routinely carry guardrails as plain sentences. It is remarkable how much production safety in real harnesses is exactly this: a firm English sentence in a markdown file.
Tip

Keep the body under roughly 500 lines; push anything longer into references/. Some harnesses also read optional frontmatter fields — Claude Code, for instance, understands an allowed-tools list that fences which tools the skill may use, and a license field — but name and description are the only two the open standard requires.

Progressive disclosure: the whole trick

Here is the problem skills solve. You saw in the context-engineering chapter that the context window is the scarcest resource in the building — every token you preload is attention spent before the user even says hello. Now imagine an agent with two hundred procedures: release notes, database runbooks, invoice formats, deployment rituals. Paste them all into the system prompt and you have burned an enormous token budget on material that is 99% irrelevant to any single task.

Progressive disclosure is the answer: reveal information in layers, each layer only when the previous one says it is worth it. Skills have three layers. The name and description of every skill are always in context — a few dozen tokens each, the spines in the card box. The full body of a skill is loaded only when the model judges, from the description, that this skill matches the task at hand. And the bundled files — references, scripts — are touched only if the procedure actually reaches the step that needs them. Watch it happen:

Skills vs tools vs prompts — and vs memory

Four things live near each other in a harness and get muddled constantly. Keep them apart with one question: what does it give the agent?

Examplesend_email()The weekly release ritual"Be concise. Never push to main."

A skill usually orchestrates tools — the recipe uses knives it did not invent. That also separates skills from MCP: an MCP server plugs new capabilities in from outside (more knives); a skill teaches the model to use the knives it already has, in the right order, to your standard. They compose beautifully: many real skills are step-by-step guides to driving a particular MCP server well. And where prompting shapes how the model behaves everywhere, all the time, a skill is a prompt with a name, a trigger, and luggage — it shows up only when called for, and it brings its own files.

What about the previous chapter's subject, long-term memory? Memory is what the agent learned — facts that accumulated from conversations, curated after the fact. A skill is what somebody taught — deliberately authored, versioned in git, reviewed in merge requests like any other artefact. Memory drifts; skills are edited. When a lesson from one session should upgrade every future session, the professional move is to fold it into the owning skill — which is why good teams treat skills as living documents, not write-once cards.

AGENTS.md, the elder sibling — and how markdown won

Skills did not appear from nowhere. Their elder sibling is AGENTS.md — a plain markdown file placed at the root of a repository, holding instructions any agent should follow when working there: how to build, which conventions to keep, what never to touch. Think of it as a README written for agents instead of humans. It began as a convention around one coding agent in 2025 and spread across the ecosystem within months, precisely because it asked nothing of anyone: no SDK, no schema, no vendor — just a file every tool could choose to read. (Claude Code's CLAUDE.md is the same idea wearing a tool-specific name.)

The kinship with skills is obvious once you see it: both bet that markdown is the interface — that the best way to program a language model is well-organised prose in files, checked into version control. The difference is scope and timing. AGENTS.md is ambient: one file, tied to a place (this repo), loaded at session start whether or not it is needed. A skill is named and lazy: tied to a job rather than a place, portable across projects, and loaded only when its description matches the task. The house rules pinned to the kitchen wall versus the recipe cards in the box.

The adoption story rhymed, too. Anthropic introduced Agent Skills in late 2025 for Claude Code, the Claude apps, and the Claude Agent SDK — and then published the format as an open standard: a specification anyone can implement, deliberately owned by no single SDK. Because implementing it is a weekend's work — scan folders, parse two YAML fields, inject an index, load bodies on demand — support spread the way AGENTS.md support had. Within months, dozens of tools across the ecosystem, from the major coding agents to open-source harnesses, could read the same folder of cards. Write a skill once and it travels with you between harnesses; that portability, more than any single feature, is why the format stuck.

Note

A skill is prose that steers an agent, so treat it with the respect you give code: review changes, test the trigger, and be deliberate about what you install. A malicious or sloppy third-party skill is an instruction-injection risk with a friendly filename — the same caution you learned for MCP servers applies to card boxes you did not write yourself.

In the wild

You will meet skills as .claude/skills/ folders inside repositories, as personal libraries in a home directory (~/.claude/skills/, or a shared ~/.agents/skills/ read by several tools), and as installable packs in plugin marketplaces — public galleries carry hundreds of ready-made skills for document formats, spreadsheets, and deployment rituals. In merge requests, watch for diffs that touch only a SKILL.md: someone is changing an agent's behaviour with zero code, and the review question is not "does it compile" but "is this procedure correct, and will the description actually trigger". When a teammate says "the skill never fires", check the description first; when they say "the skill blew the context budget", look for a body that should have been split into references/.

Check yourself

1. Which two frontmatter fields does the SKILL.md open standard require?

Just name and description. Everything else — allowed-tools, license, and friends — is an optional extra that particular harnesses may read. Two required fields is a big part of why the format spread so easily.

2. When does the full body of a SKILL.md enter the model's context?

That is progressive disclosure: names and descriptions are always visible, the body loads on trigger, and bundled files load later still — or never. The other forty-nine skills keep costing two lines each.

3. What does a skill add to an agent that a tool does not?

A skill adds no capability at all — it is choreography for the tools the agent already has. New capabilities come from tools (or MCP servers); learned facts live in memory; skills carry the how-to.

4. Your team wrote a superb 400-line skill, but agents never seem to use it. What is the most likely culprit?

The description is the only part of the skill visible at decision time. If it does not say what the skill does and when to use it — ideally with the phrases users actually say — the model has no reason ever to pull the card.

5. Why bundle a helper as scripts/collect_commits.sh instead of writing those steps as prose in the body?

The model executes the script through its shell tool and sees only the output. Fixed, mechanical steps belong in scripts; prose in the body is for the judgement calls a script cannot make.

Key takeaways

  • A skill is a folder with a SKILL.md: YAML frontmatter (name + description — the trigger) above a markdown body (the procedure), with optional references/ and scripts/ alongside.
  • Progressive disclosure is the engine: descriptions always in context, the body only on trigger, bundled files only at the step that needs them — so two hundred skills cost almost nothing until one is used.
  • The description is the skill's advertisement and its only evidence at decision time; most "the skill never fires" bugs are description bugs.
  • Tools give capability, prompts give disposition, memory gives learned facts — skills give competence: taught procedures, versioned in git and reviewed like code.
  • Scripts beside the skill run without entering context; deterministic steps go in code, judgement goes in prose.
  • SKILL.md follows AGENTS.md's path: a plain-markdown open standard, trivial to implement, now read by dozens of tools — write a procedure once and it travels with you.
