An agent in a pod

The loop, the skills, the MCP servers, the A2A endpoint — everything snaps together here, into one pod: the agent's body, drawn end to end.

~35 minChapter 114 of 135
The story

The consultant from the hotel room — the one you built a walkie-talkie loop with in chapter 46 — finally got hired. On her first morning, facilities walks her to a cabin that is astonishingly complete. On the desk, a laminated card from the front office: working hours, daily budget, useful extensions. In the wall, a safe she cannot open herself — she taps her badge and the building's vault, three floors down, releases exactly the credential she needs, for exactly as long as she needs it. A trolley of procedure manuals was wheeled in overnight. In the corner sits a junior assistant who handles one specific chore; she just turns and speaks to him. For everything else, a phone with two buttons: one rings the shared research desk every team uses, the other reaches any colleague by name. And on her door: a nameplate and extension, listed in the company directory, so colleagues can call her.

Here is the trick: facilities can demolish this cabin tonight and rebuild an identical one on another floor by 9 a.m. Nothing inside is irreplaceable, because everything was delivered by the building. That cabin is a pod. This chapter furnishes it, item by item.

The body plan

Earlier in this act you built the shell: an image in a registry, a cluster, the Deployment (the blueprint that keeps N copies of a pod alive), the Service (the stable name in front of them), the ConfigMap and the Secret. What you have not yet done is put a real agent inside — skills, MCP connections, A2A doorplate and all. That is this chapter's whole job.

Recall the shape of a pod: the smallest thing Kubernetes schedules — one or more containers that live and die together, sharing one network identity. If a container is a process in a box, the pod is a body, and the harness container is its heart: the very loop you hand-built in chapter 46, unchanged. What changes is everything around it — how settings, credentials, skills, and connections reach it. Here is the full anatomy; the rest of the chapter walks through it organ by organ:

Read it in three columns: what the platform injects (the agent never fetches its own config, secrets, or skills — they are simply there when it wakes, like the furnished cabin), the pod itself (two containers, one body), and the network around it. Keep these six words handy as we go:

Configuration: the laminated card on the desk

Back in chapter 19 you learned that good programs read settings from environment variables rather than hard-coding them. Kubernetes industrialises that rule: write the settings once, as a ConfigMap, and the platform injects them into every replica at container start.

research-agent-config.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: research-agent-config
data:
  MODEL_NAME: "big-frontier-model"
  MAX_TURNS: "12"
  LOG_LEVEL: "INFO"
  KB_MCP_URL: "http://kb-mcp.agents.svc.cluster.local/mcp"

Four settings, each earning its keep. MODEL_NAME and MAX_TURNS tune the loop — the chapter-45 max_turns guard, now operator-adjustable without touching code. Every value is a string, even the number: ConfigMaps carry text, your harness parses it. And look at KB_MCP_URL — the address of a shared MCP server is itself just configuration; rewiring to a different knowledge base is a one-line change, not a rebuild. In the Deployment, one envFrom stanza pours all four keys in as env vars.

Note

Env vars are injected when the container starts. Edit the ConfigMap and running pods keep the old values — a classic on-call surprise. kubectl rollout restart deployment/research-agent replaces them with containers that wake up on the new settings.

Secrets: the safe the agent cannot open

Now the part where careers are made or ended. The agent needs an LLM API key, and the one place it must never live is inside the container image. An image is a stack of copyable layers — pulled to every node, cached, exported — and anyone with pull access can unpack it and read whatever you baked in. Baking a key into an image is photocopying your house key and stapling it to every job advert.

Kubernetes offers a built-in Secret object, and earlier in this act you learned its uncomfortable truth: it stores values base64-encoded, not encrypted — an envelope, not a safe. Fine for dev; not where a bank keeps an API key. Production on Azure does one of two better things:

Option one: Key Vault through the CSI driver. A CSI driver — container storage interface, a plugin that mounts external storage into pods — fetches secrets from Azure Key Vault and presents them as read-only files under a path like /mnt/secrets/. The AKS add-on is enabled with az aks enable-addons --addons azure-keyvault-secrets-provider, and a small SecretProviderClass manifest names which vault, which secrets. The key never touches the image, the YAML, or git.

Option two: workload identity — no key at all. With Microsoft Entra workload identity, the pod's service account is federated with an Entra identity: the pod presents a short-lived signed token saying "I am the research agent", and Azure grants access to the vault or model endpoint directly. Nothing long-lived exists to rotate, leak, or bake. When the security review asks "where is the key stored?", the best answer is "there isn't one".

Careful

The hierarchy: image < plain Kubernetes Secret < Key Vault via CSI < workload identity. Each step right removes a place a credential can leak from. An ENV OPENAI_API_KEY=sk-... line in a Dockerfile is not a style nit — that key is in every copy of the image, forever. Flag it.

Skills: the bookshelf wheeled in overnight

In chapter 85 you met Agent Skills: folders of SKILL.md files the harness discovers by name and description, loading the full procedure only when a task matches — progressive disclosure. On your laptop those folders sit next to the code. In a pod, they arrive by one of two roads:

RoadHowBehaviour
Baked into the imageCOPY skills/ /app/skills/ in the Dockerfile at build timeVersioned with the image tag — immutable, reviewable. Changing a skill means rolling a new image.
Mounted as a volumeA ConfigMap volume per skill folder, or an Azure Files share at /app/skills/Updates without a rebuild — faster iteration, but behaviour can drift from the image tag.

Both are legitimate. Teams that treat skills as code — reviewed, versioned — bake them in, so image tag 1.4.2 means exactly one behaviour. Teams iterating on prompts daily mount them, trading drift risk for speed. One quirk on the mounted road: a ConfigMap's keys become flat files, so a skill folder maps naturally to one ConfigMap per skill, each mounted at its own subfolder. A library of fifty skills outgrows that — use a file share or bake them in.

MCP, wired both ways

From chapter 76, recall the two MCP transports: stdio for a server running locally as a child of the harness, streamable HTTP for a server across the network. Both survive the move into the cluster — they just take different seats.

The local server becomes a sidecar. On your laptop, the harness spawned a private MCP server of your own — like the finance desk you built in chapter 78 — as a subprocess and spoke stdio to it. In the cluster, that same private server ships as a sidecar — a second container declared in the same pod. Containers in a pod share one network namespace — the technical way of saying they are the same "localhost" — so the harness reaches it at 127.0.0.1: the assistant in the corner of the cabin, private to this agent, started and stopped as one body. (Some harnesses keep the stdio server as a true subprocess inside the harness container instead — also fine; the point is it stays inside the body.)

The shared server becomes a Service. The team knowledge-base MCP server that ten agents query is the opposite case: a private copy in every pod would mean ten copies to patch. Instead it gets its own Deployment and Service, speaks streamable HTTP, and every agent reaches it at one stable URL — the KB_MCP_URL from the ConfigMap. If the knowledge base becomes the bottleneck, you scale it, not every agent.

The A2A doorplate

So far, everything has flowed into the agent. The last organ faces outward. From chapter 81: an agent publishes an agent card — a JSON self-description at /.well-known/agent-card.json listing its name, abilities, and how to send it tasks — and other agents call it over HTTP. Inside the harness container that endpoint is just a web server on port 8000. But a pod's IP is disposable — pods die and return with new addresses. Nobody may ever hold a pod's number.

The fix is the one Kubernetes always offers: a Service in front, giving the agent a stable DNS name — research-agent.agents.svc.cluster.local — and forwarding traffic to whichever healthy pod currently carries the label. The nameplate on the door, outliving every individual cabin.

Now zoom out, because the multi-agent picture snaps into focus here. Each agent in the fleet is its own Deployment plus its own Service. Planner, research, summariser — three Deployments, three Services, three agent cards. When the planner wants research done, it resolves the research agent's Service name over cluster DNS, fetches the agent card, and posts the task — A2A over the flat cluster network:

planner agentDeployment + Service
→
research agentGET /.well-known/agent-card.json
→
summariser agentA2A task handoff

Inside one cluster, Services are all you need. The moment an agent must be reachable from outside — another cluster, a partner's fleet — you add an ingress (the cluster's front door for external HTTP) in front of its Service. Same body, bigger directory.

The whole body, in YAML

Now the entire anatomy as Kubernetes receives it: one Deployment manifest. Read it like a reviewer — click the annotated lines. Every organ from the diagram is here.

research-agent.yaml — the agent's body as a manifest

And the doorplate — the Service that gives this body its permanent name:

research-agent-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: research-agent
  namespace: agents
spec:
  selector:
    app: research-agent
  ports:
  - port: 80
    targetPort: 8000

Default type, ClusterIP — reachable inside the cluster only, exactly right for fleet-internal A2A. The selector finds every pod labelled app: research-agent; port: 80 is the number colleagues dial; targetPort: 8000 is where the harness listens. From this moment, http://research-agent.agents.svc.cluster.local/ works for every agent in the cluster, however many times the pod behind it is rebuilt.

Ship it and look inside:

deploy and inspect
kubectl apply -f research-agent.yaml -f research-agent-service.yaml
kubectl get pods -l app=research-agent
kubectl describe pod research-agent-7d4f9b6c8-x2p4v
kubectl logs research-agent-7d4f9b6c8-x2p4v -c harness
kubectl logs research-agent-7d4f9b6c8-x2p4v -c docs-mcp

Line by line. Line 1: kubectl apply hands both manifests to the cluster to reconcile — create what is missing, update what changed. Line 2: get pods shows the body being born; READY reads 2/2, and that first 2 is not replicas — it is containers in the pod, harness and sidecar both up. Line 3: describe is the physical exam — volumes attaching, events rolling from Scheduled to Started; when a pod sticks in Pending or ImagePullBackOff, the reason is written here. Lines 4 and 5: with two containers, kubectl logs needs -c to name which story you want — harness for reasoning, sidecar for MCP traffic.

prove the doorplate works
kubectl port-forward svc/research-agent 8080:80
curl http://localhost:8080/.well-known/agent-card.json

Line 1: port-forward tunnels your laptop's port 8080 to the Service's port 80 — a private debugging peephole, no ingress required. Line 2: fetch the agent card exactly as a fellow agent would. If JSON describing your agent's skills comes back, the body is alive, named, and discoverable — the chapter-80 handshake, now answered by a pod.

Hold the body in your hands

Everything Kubernetes injects, your harness experiences as ordinary Python: environment variables, files on disk, URLs that answer. So the whole anatomy fits in a few lines you can run and break. Press Run — then delete the secret entry and see exactly what a pod with a misconfigured SecretProviderClass would log.

Tip

Run the failure drills. Remove the llm-api-key entry: the KeyError is the Python twin of a pod failing because its secret volume would not mount. Point KB_MCP_URL at a wrong Service name: that is a ConfigMap typo as production would log it. The bugs you cause here are the tickets you will triage later.

In the wild

Open any agent platform's deploy folder and this chapter stares back: a deployment.yaml with a suspiciously long containers: list (sidecars), an envFrom block, a secretProviderClass name, a Service per agent. In merge requests you will read "moved the API key into Key Vault", "bumped the skills ConfigMap", or "split the docs MCP into its own Service — three teams want it now". A favourite production trick: a readiness probe pointed at /.well-known/agent-card.json, so a pod only receives traffic once its agent card answers.

Check yourself

1. Why must the LLM API key never be baked into the container image?

An image is pulled, cached, and copied everywhere; whatever is baked in travels with every copy, forever. So secrets arrive at run time — Key Vault via the CSI driver, or no key at all via workload identity.

2. Your agent uses a small private docs MCP server, and the whole team shares a knowledge-base MCP server. Where does each one run?

Private servers belong inside the body — a sidecar sharing the pod's localhost and lifecycle. Shared servers get their own Deployment and Service, so ten agents hit one URL that scales independently.

3. How does the planner agent find and call your research agent?

Pod IPs are disposable. The Service provides the permanent name; the agent card at the well-known path describes what the agent can do. Discovery plus stable addressing is the whole multi-agent wiring.

4. You edit MAX_TURNS in the ConfigMap, but the running agent keeps using the old value. Why?

Env vars were poured in at container start; editing the ConfigMap does not reach into running processes. A kubectl rollout restart replaces the pods with containers that wake up on the new values.

5. kubectl get pods shows your agent pod as READY 2/2. What does that mean for reading logs?

The READY fraction counts containers inside one pod. -c harness gives the agent's reasoning; -c docs-mcp gives the tool server's traffic — two organs, two log streams.

Key takeaways

  • The pod is the agent's body: the harness container (your chapter-45 loop, containerised) at its heart, everything else injected or attached by the platform.
  • Configuration flows in as env vars from a ConfigMap — injected at container start, so config changes need a rollout to take effect.
  • Secrets never ride in the image: Key Vault delivers them at run time via the Secrets Store CSI driver, or workload identity removes the stored key entirely.
  • Skills arrive by two roads — baked into the image (versioned, immutable) or mounted as a volume (hot-swappable, drift-prone); the harness discovers SKILL.md folders as it did on your laptop.
  • MCP is wired both ways: private servers ride as sidecars on the pod's localhost; shared servers sit behind their own Services speaking streamable HTTP.
  • The A2A endpoint hides behind a Service, so other agents find the agent card at a stable DNS name — a fleet is one Deployment plus one Service per agent, talking A2A across the cluster network.
  • Because everything is delivered by the platform, the body is disposable: kill the pod and the Deployment rebuilds it identically — that disposability is the point.
