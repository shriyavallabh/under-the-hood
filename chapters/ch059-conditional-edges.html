Conditional edges and routing

Fixed edges make a pipeline. One routing function at the right junction makes it an agent — and the ReAct loop you built by hand turns out to be exactly two stations and a signal box.

~25 minChapter 60 of 133
The story

A freight yard at night. Nearly all of the track layout is fixed: from the loading dock every wagon rolls to the weighbridge, from the weighbridge every wagon rolls to the junction. No decisions anywhere — just steel rails. But at the junction stands a raised signal box, and in it sits a signalwoman with two levers. She never loads cargo. She never drives a train. Her entire job is one glance and one pull: as each wagon pauses beneath her window, she reads the last line of its manifest. If it says "needs sealing", she pulls lever one and the wagon curves back around the loop to the sealing shed — from where the rails will carry it straight back to her. If it says "sealed", she pulls lever two and the wagon glides off to the dispatch platform, done.

Notice the elegant split of power. The decision is live — made fresh for every wagon, based on whatever the manifest says right now. But the options are fixed — there are exactly two levers, wired to exactly two tracks, and she cannot improvise a third. That signal box is a conditional edge. The woman is a routing function. The lever-to-track wiring is the mapping. And a surprising amount of this act is about putting the right person in that box.

The one piece ch058 left out

Quick recap of StateGraph anatomy, because this chapter stands directly on it. A LangGraph app has three parts: a state schema (a TypedDict describing the shared clipboard every step reads and writes), nodes (plain Python functions that take the state and return only the fields they changed), and edges (the route card wiring nodes together, from START to END). You even traced a three-node pipeline through the engine by hand. But every edge in that chapter was fixed — add_edge("a", "b") means after a, always b, no exceptions, forever.

Fixed edges can only build pipelines: straight lines whose route was settled at design time, before any real data arrived. They cannot build the thing this whole course orbits — the ReAct loop, where the system keeps going around until the model says it is done. "Until" is a decision made at run time, from live data. Pipelines cannot decide. Something at the junction has to look at the state and choose.

That something is a routing function — a plain Python function that receives the current state, inspects it, and returns a string: the name of wherever the graph should go next. Read that sentence again, because it contains the whole chapter. A node returns a dict of state updates — it changes the world. A routing function returns a name — it changes nothing; it only points. The signalwoman never touches the cargo.

Note

This split — nodes do work, routers only point — is a deliberate design rule, not an accident. Because a routing function cannot modify state, you can read any LangGraph app and know that every state change lives inside a node and every decision lives inside a router. Two short lists to audit instead of one tangled loop. Enterprises, as the why-graphs chapter put it, are metro people — this is a big part of why.

add_conditional_edges: wiring the signal box

Here is the API, and then we will take it apart piece by piece:

wiring a junction after the "agent" node
def should_continue(state):
    last = state["messages"][-1]
    if last.tool_calls:          # the model asked for a tool
        return "act"
    return "done"                # no tool calls: it answered

builder.add_conditional_edges(
    "agent",                          # 1. junction sits AFTER this node
    should_continue,                  # 2. the routing function
    {"act": "tools", "done": END},    # 3. mapping: label to destination
)

add_conditional_edges takes three things. First, the source node — the junction is placed on the track leaving this node, so the router runs every single time this node finishes. Second, the routing function itself. Third, the mapping (the docs call it a path map) — a dict translating whatever label the router returns into an actual destination: another node's name, or END to stop the run. Router says "act", the mapping says that means the tools node; router says "done", the mapping says that means END. Levers on one side, tracks on the other, wiring in between.

Two pieces of small print worth knowing. You may skip the mapping entirely and have the router return real node names directly — LangGraph accepts that — but the mapping earns its keep twice over: it keeps the router's vocabulary decoupled from your node names (rename a node, touch one dict), and it declares every possible destination up front, so the picture LangGraph can draw of your compiled graph shows the decision arrows without ever running your code. And if the router returns a label that is in neither the mapping nor the node list — say a typo like "tols" — the run fails loudly at that step with an unknown-destination error. The signalwoman has no third lever; she cannot send a wagon down a track that was never laid. Compare that with the hand-written loop from ch045, where a typo in the branch logic just… did something, silently, and you found out from the logs.

The punchline: ReAct is a two-node graph

Now the promise in this chapter's tagline. Take the fifty-line ReAct loop you built by hand — think, act, observe, repeat until FINAL ANSWER — and redraw it as a graph. It comes out almost embarrassingly small:

  • An agent node: calls the model with the message history, appends the reply. (One node.)
  • A tools node: executes whatever tool calls the model requested, appends the results. (Two nodes. That is all of them.)
  • A fixed edge START → agent; a fixed edge tools → agent, because results must always go back to the model — remember from ch045 what happens when the observation never returns; and one conditional edge after agent: tool calls pending? Go to tools. None? Go to END.

The famous loop is not written as a loop anywhere. It emerges from the cycle in the wiring — agent → tools → agent → tools… — until the router finally pulls the other lever. The while-loop you wrote in ch045 has dissolved into topology. Watch one complete run move through it:

Sit with how little machinery that took, because it is the mental unlock of Act VII: loops are not a special feature — they are cycles in the wiring, and conditional edges are the exit doors. Once you can see the ReAct loop as two stations and a signal box, bigger graphs stop being intimidating. A research agent with a planner, three specialists, and a reviewer is the same picture with more stations — and every "revise or ship?" moment inside it is a junction exactly like this one.

Note

One paragraph on fan-out: the Send API

Everything so far picks one next stop. Sometimes you want many at once — thirty documents to summarise, and you would like thirty parallel copies of a summarise node, each holding its own document. For that, a routing function can return a list of Send objects instead of a label: from langgraph.types import Send, then return [Send("summarise", {"doc": d}) for d in state["docs"]]. Each Send names a node and hands it a private payload as its state; LangGraph runs the copies in parallel, and their returned updates merge back into the shared state through reducers — which is exactly why ch058 made such a fuss about append-style reducers and parallel writes. That is the map-reduce pattern expressed in one routing decision, and it is as far as we take it here; it returns in force with the multi-agent patterns of ch063.

In the wild

Open any LangGraph repository and search for add_conditional_edges — the hits are a complete inventory of every decision the system can make, which is why reviewers read them first. The router is almost always named something like should_continue, route_after_agent, or next_step, and nine times out of ten its body checks tool_calls on the last message, exactly as here. In merge requests you will meet comments like "this belongs in the router, not the node" (someone buried a decision inside work code) or "route this to END instead of raising" (turning a crash into a clean stop). And when a teammate says "the graph has a cycle between agent and tools", you can nod and picture the signal box that keeps that cycle from spinning forever.

Check yourself

1. What does a routing function return, and what may it change?

Routers read state and point; only nodes write. That separation is what makes a graph auditable: all changes live in nodes, all decisions live in routers.

2. In the two-node ReAct graph, what does the conditional edge after the agent node actually check?

The router inspects the last message: pending tool calls mean the model wants to act, so route to the tools node; a plain answer means it is done, so route to END. That one check is the entire stop condition of ReAct.

3. What is the mapping (path map) in add_conditional_edges("agent", router, {"act": "tools", "done": END}) for?

The mapping is the lever-to-track wiring: labels stay decoupled from node names, and the full set of destinations is visible without running any code. It is technically optional — a router may return node names directly — but the decoupling and the drawable diagram make it the usual style.

4. Where is the ReAct loop actually written in the two-node graph?

The while-loop from ch045 dissolves into topology: a fixed edge back from tools to agent creates the cycle, and the junction after agent decides on every pass whether to go around again or leave for END.

5. Your router returns a list like [Send("summarise", {"doc": d}) for d in docs]. What happens?

That is the Send API's fan-out: each Send names a node and hands it a private state, the copies run in parallel, and append-style reducers gather the results — map-reduce inside one routing decision.

Key takeaways

  • A routing function is a plain function that reads the state and returns a string naming the next destination — it points, it never writes. Nodes do the work; routers make the decisions.
  • add_conditional_edges(source, router, mapping) places a junction after a node; the mapping translates the router's labels into node names or END, and declares every possible exit up front.
  • The decision is live but the options are fixed: a router can only choose among destinations that exist in the graph — bounded autonomy, drawn in track.
  • The ReAct loop is secretly a two-node graph: agent and tools, a fixed return edge, and one conditional edge whose tool-calls check is the entire stop condition. Loops are cycles in the wiring, with a junction as the exit door.
  • create_agent from Act VI builds exactly this graph for you — conditional edges are how you build it yourself when the prebuilt loop is not the flow you need.
  • For fan-out, a router can return a list of Send objects — each launches a parallel copy of a node with its own private payload, merged back through reducers.
