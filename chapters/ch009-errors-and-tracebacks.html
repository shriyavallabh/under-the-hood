Errors, exceptions, and tracebacks

A traceback is not a wall of blame. It is a well-organised incident report — and once you learn to read it from the bottom up, errors stop being scary and become the fastest map to the fix.

~25 minChapter 9 of 117
The story

Friday night, a packed restaurant. A customer orders the paneer makhani. The waiter carries the order to the pass; the head chef hands it to the curry station; the curry cook reaches into the fridge for cream — and the tub is empty. Work on that order stops dead. But nobody runs around shouting. The kitchen has a ritual: the cook fills in an incident slip. Bottom line first: "FRIDGE EMPTY: no cream." Above it, the chain of hands the order passed through — curry station, head chef, waiter, table 12.

The manager who picks up this slip has a trick: she reads it from the bottom. The last line says what went wrong; the entry above it says where. Everything else is just the delivery route. Python writes exactly this slip every time something breaks. It is called a traceback, and by the end of this chapter you will read a forty-line one the way that manager reads her slip: calmly, bottom-up, straight to the point.

Two kinds of going wrong

Before any syntax, one distinction that will serve you for the rest of your career: a bug is not the same as a handled error.

An empty cream tub is not a scandal. Fridges run empty; customers mistype their email; a model replies with slightly broken JSON. These are expected failures — things a well-run kitchen has a plan for. Handling them is just another kind of control flow, a sibling of the if/else decisions you already know: if this goes wrong, do that instead.

A bug is different. A bug is the recipe itself being wrong — the code asked for something that can never work. You do not want a plan for bugs. You want them to fail loudly, with a full incident report, so you can fix the recipe. A program that hides its bugs is more dangerous than one that crashes, because it keeps serving wrong food with a smile.

Python's mechanism for both is the same. When something cannot continue, Python raises an exception — it stops normal execution on the spot and sends an error object shooting back up through the chain of function calls, looking for someone with a plan. If some code catches it, that was a handled error: the program carries on. If nothing catches it, the program stops and Python prints the incident slip — the traceback, a listing of every function call the error travelled through. Remember from the functions chapter that functions call functions, building a stack of "who is waiting on whom"? A traceback is that stack, printed at the moment of the crash.

Note

The Zen of Python — the design philosophy you met in chapter 1 — has a line for this: "Errors should never pass silently. Unless explicitly silenced." Loud by default, quiet only on purpose. Hold that thought; it is the moral of this whole chapter.

Every error carries a type

Python does not throw one generic "error". Each exception has a type — a name that tells you the category of failure before you read another word. Seasoned readers get most of their diagnosis from this one word. Flip these cards until each name conjures a picture:

Two things worth knowing. First, the name comes from where the failure happens, not from your intent — asking a dictionary (chapter 4) for a missing key is always a KeyError, whether it holds settings, prices, or tool registrations. Second, the types form families: JSONDecodeError is a specialised kind of ValueError, so a net that catches ValueError catches it too. Wider nets catch more; in a moment you will see why the widest nets are a trap.

try / except / else / finally: the four rooms

The handling machinery is one statement with four rooms, and you will meet all four in real code. try holds the risky work. except is the plan for a specific failure. else runs only if nothing failed. finally runs no matter what — the cleanup room. Read the annotated version first, clicking each line; then run and break the live copy below it.

settings.py — a parser with a plan B
import json
The standard-library JSON tools from chapter 5. json.loads raises JSONDecodeError when the text is not valid JSON — which is exactly the failure we are about to plan for.
DEFAULTS = {"model": "small", "retries": 2}
The fallback. A handled error needs somewhere sensible to land — here, a known-good set of defaults.
def load_settings(text):
try:
"The next lines might fail, and I have a plan." Keep this room small, so you always know exactly which line the plan is for.
settings = json.loads(text)
The one risky line. If text is valid JSON, we fall through to else. If not, execution jumps — mid-statement — to the matching except.
except json.JSONDecodeError as err:
The plan, and only for THIS failure type. as err names the error object so we can read its message. A different exception — say a TypeError — sails straight past, uncaught. Deliberate: that would be a bug, and bugs should crash.
print(f"Bad settings ({err.msg}) - using defaults")
Say what happened. Silent fallbacks are how systems drift into serving defaults for six months without anyone noticing.
settings = dict(DEFAULTS)
Land on the fallback. A copy, not the original — so later edits cannot quietly rewrite the shared defaults (the aliasing lesson from chapter 2).
else:
The rarely-seen room: runs only when the try succeeded. Why not put this inside try? Because the plan should cover the risky line only, not your victory lap.
print("Settings parsed cleanly")
finally:
Runs in every timeline — success, handled failure, even an uncaught exception on its way out. This is where files get closed and connections released, whatever happened above.
print("Settings step finished")
return settings
Tip

Break it three ways. One: pass load_settings(42) and watch a TypeError blow straight past the except — the net only catches what it names. Two: change the except line to except ValueError as err: — it still catches bad JSON, because JSONDecodeError belongs to the ValueError family. Three: delete the finally block and see the goodbye message vanish from both runs.

Careful

The widest net of all is a bare except: — it catches everything, including the typo-induced NameError that was trying to report a bug. Its evil twin, except: pass, catches everything and says nothing. Reviewers call this "swallowing errors"; it turns a ten-minute fix into a three-day mystery. Name the exception you expect; let everything else crash loudly.

raise: sounding the alarm yourself

Everything so far reacts to Python raising exceptions. You can also raise your own — and good code does, early and specifically, the moment it detects a situation it cannot honour:

guardrails.py
def set_retries(n):
    if n < 0:
        raise ValueError(f"retries must be 0 or more, got {n}")
    return {"retries": n}

That raise statement creates an exception and fires it up the call chain, exactly as if Python had raised it. Why crash on purpose? Because a negative retry count accepted quietly here becomes a bizarre infinite-loop bug three files away tonight. Raising at the door — naming the culprit value — moves the crash to where the mistake actually is. There is also the bare form, used inside an except block:

relay.py — look, log, pass it on
try:
    price = rates[symbol]
except KeyError:
    log_incident("rate lookup failed")
    raise    # re-raise the SAME exception, original traceback intact

A bare raise says: I wanted to witness this failure, not handle it. The exception continues upward as if never caught — the honest middle ground between swallowing an error and ignoring it.

Reading a traceback, calmly

Now the superpower this chapter promised. Here is the whole method: a traceback reads bottom-up. Python even tells you so in its first line — "most recent call last" means the freshest, most useful information is printed at the end. Step through a real one, exactly as your eyes should travel:

1. Freeze — then jump to the last line

Traceback (most recent call last):
briefing.py, line 31, in <module>main()
briefing.py, line 27, in mainreport = build_report(today)
report.py, line 14, in build_reportprice = fetch_rate(row)
report.py, line 8, in fetch_ratereturn RATES[symbol]
KeyError: 'SENSEX'start reading HERE

Your eye wants to start at the top, like any other text. Resist. The header is a promise: the most recent call — the scene of the failure — is printed last. Go straight to the bottom line.

2. Read the name and the message

Traceback (most recent call last):
briefing.py, line 31, in <module>main()
briefing.py, line 27, in mainreport = build_report(today)
report.py, line 14, in build_reportprice = fetch_rate(row)
report.py, line 8, in fetch_ratereturn RATES[symbol]
KeyError: 'SENSEX'type + message = the diagnosis

Type first: KeyError — a dictionary was asked for a key it does not hold. Message second: the missing key was 'SENSEX'. Two seconds in, you know WHAT went wrong — half the diagnosis, before reading a single frame.

3. One line up: the scene of the crash

Traceback (most recent call last):
briefing.py, line 31, in <module>main()
briefing.py, line 27, in mainreport = build_report(today)
report.py, line 14, in build_reportprice = fetch_rate(row)
report.py, line 8, in fetch_ratereturn RATES[symbol]
KeyError: 'SENSEX'

The deepest frame — directly above the error — gives you WHERE: file, line, function, and the source line that blew up. That is the curry cook at the empty fridge.

4. Climb the chain of custody

Traceback (most recent call last):
briefing.py, line 31, in <module>main()
briefing.py, line 27, in mainand who asked for the report?
report.py, line 14, in build_reportwho asked for the rate?
report.py, line 8, in fetch_ratereturn RATES[symbol]
KeyError: 'SENSEX'

Each frame above answers one question: "and who asked for THAT?" This is the call stack from the functions chapter, photographed at the instant of the crash. Climb only as far as you need.

5. The top is just the front door

Traceback (most recent call last):
briefing.py, line 31, in <module>where the program began
briefing.py, line 27, in main
report.py, line 14, in build_report
report.py, line 8, in fetch_rate
KeyError: 'SENSEX'

The topmost frame is only where execution started — the waiter taking the order. Untrained readers find tracebacks exhausting because they read them upside down.

6. The verdict — and a twist

Traceback (most recent call last):
briefing.py, line 31, in <module>
briefing.py, line 27, in main
report.py, line 14, in build_reportthe fix probably lives here
report.py, line 8, in fetch_ratewhere it crashed
KeyError: 'SENSEX'what went wrong

What: KeyError 'SENSEX'. Where: fetch_rate, report.py line 8. Why: RATES never loaded that symbol. The twist — the crash line is innocent; the mistake is upstream, where the symbol list was built. Crash frame and fix frame are often neighbours, not the same line.

The forty-line monster

Real production tracebacks are longer, because real code calls libraries and libraries call each other.   File "/home/you/agents/briefing/agent.py", line 33, in run_agent
    action = parse_action(reply)
  File "/home/you/agents/briefing/agent.py", line 21, in parse_action
    data = json.loads(text)
  File "/usr/local/lib/python3.12/json/__init__.py", line 346, in loads
    return _default_decoder.decode(s)
  File "/usr/local/lib/python3.12/json/decoder.py", line 338, in decode
    obj, end = self.raw_decode(s, idx=_w(s, 0).end())
  File "/usr/local/lib/python3.12/json/decoder.py", line 356, in raw_decode
    raise JSONDecodeError("Expecting value", s, err.value) from None
json.decoder.JSONDecodeError: Expecting value: line 1 column 1 (char 0)

Long tracebacks add exactly one new skill: telling your frames from library frames. Look at the file paths. Anything under /usr/local/lib/, site-packages/, or a virtual-environment folder belongs to a library; frames pointing into your own folders are yours. Library frames are almost never the problem — json/decoder.py is battle-tested by millions of programs. The question a long traceback answers is: which of MY lines handed the library something it could not digest?

Tip

The three-move method, for any length. Move one: the last line — JSONDecodeError: Expecting value ... (char 0), the text did not even start like JSON. Move two: scan upward for the deepest frame in a file you own — agent.py, line 21, in parse_action. That is where to look: the model's reply was not JSON. Move three — only if needed — climb further to see how execution got there. Most sessions never need move three.

Chained tracebacks: two reports stapled together

Sometimes a traceback contains two reports glued by one sentence. Do not panic — it is the same skill twice. The deliberate case: code catches a low-level error and re-raises it as something more meaningful, using raise ... from ...:

rates.py — translating an error upward
RATES = {"NIFTY": 25400.0}

def fetch_rate(symbol):
    try:
        return RATES[symbol]
    except KeyError as err:
        raise RuntimeError(f"{symbol!r} is not in today's feed") from err

fetch_rate("SENSEX")

Run mentally, that produces:

terminal — a chained traceback
Traceback (most recent call last):
  File "rates.py", line 5, in fetch_rate
    return RATES[symbol]
KeyError: 'SENSEX'

The above exception was the direct cause of the following exception:

Traceback (most recent call last):
  File "rates.py", line 9, in <module>
    fetch_rate("SENSEX")
  File "rates.py", line 7, in fetch_rate
    raise RuntimeError(f"{symbol!r} is not in today's feed") from err
RuntimeError: 'SENSEX' is not in today's feed

Reading order for any chained traceback: the bottom chunk is what finally surfaced — read its last line first, as always. The top chunk is the original cause, preserved so you do not lose the trail. The glue sentence tells you how they relate. "The above exception was the direct cause" means someone chose to translate the error, as here. Its cousin — "During handling of the above exception, another exception occurred" — means something clumsier: an except block crashed while rescuing. That version usually reveals two bugs for the price of one: the original failure, plus a plan B that was never tested. Either way the method survives: last line of the whole thing first, then upward chunk by chunk.

Errors in agent land

Now connect this to the systems you operate every day. A production agent loop wraps exception handling around two precise places, and the choice of what to catch — and what to let crash — is the design.

First: parsing the model's reply. Models sometimes return chatty, slightly broken output; that is an expected error, so the loop catches it and asks the model to try again. Second: running a tool. Tools fail constantly, and a crashed agent helps nobody, so harnesses catch the tool's exception and feed the error message back to the model as the observation. The model reads its own failure and self-corrects — that is why your agent logs show a tool error followed by the model apologising and adjusting. In the wild

You will meet this chapter everywhere. When an agent run dies, the last screenful of the log is a traceback — the bottom-up method works on it unchanged; find the deepest frame that belongs to the team's own code rather than the framework's. In merge requests, reviewers write "this except is too broad", "don't swallow that", or "raise early with a better message" — you can now hear exactly what they mean. Framework retry options and tool-error hooks are industrialised versions of the catch-and-feed-back pattern you just ran. And when a colleague says "it failed with a KeyError on 'symbol'", notice they handed you the type, the message, and half the diagnosis in one sentence — traceback fluency is a shared language.

Check yourself

Five quick checks

1. A 40-line traceback fills your screen. Where do you look first?

"Most recent call last" is Python's promise that what actually went wrong is printed at the bottom. Type and message first, then the deepest frame in your own code, then climb only if needed.

2. The last line says KeyError: 'symbol'. Before reading any frames, what do you already know?

The exception type carries the category: KeyError is always a failed dictionary lookup. A missing variable would be a NameError; a missing file would be a FileNotFoundError. One word, half the diagnosis.

3. When does a finally block run?

finally is the cleanup room: it runs in every timeline, which is exactly why files get closed and connections released there. else is the room that runs only on success.

4. A traceback contains: "During handling of the above exception, another exception occurred". What happened?

Two reports, stapled together in time order. The rescue attempt itself failed — often revealing an untested plan B. Read the whole thing bottom-up by chunk: last line first, then the earlier report for the root cause.

5. Inside an agent loop, a tool raises FileNotFoundError. What does a well-built harness typically do?

Tool failures are expected errors, so they become information the model can react to. Crashing is reserved for genuine bugs; swallowing silently is the one option that is always wrong.

Key takeaways

  • A traceback is an incident report, printed "most recent call last" — so read it bottom-up: exception type and message, then the deepest frame, then climb only if you must.
  • The exception type is half the diagnosis: KeyError means a failed dictionary lookup, TypeError a wrong kind of thing, JSONDecodeError text that was not JSON — one word tells you the category before you read a single frame.
  • try/except/else/finally are four rooms: risky work, the plan for a named failure, the success-only path, and cleanup that runs in every timeline.
  • Catch narrowly and loudly; raise early, naming the bad value. Bare except: pass hides the report you will desperately need later.
  • In long tracebacks, find the deepest frame in a file you own — library frames are plumbing, and the real mistake often sits one or two frames upstream of the crash line.
  • Chained tracebacks are two reports glued by one sentence: "direct cause" means a deliberate translation with raise ... from; "during handling" means the rescue itself crashed.
  • Handled errors are control flow, bugs deserve loud crashes — and in agent systems, tool exceptions get recycled into observations so the model can read its own failure and self-correct.
