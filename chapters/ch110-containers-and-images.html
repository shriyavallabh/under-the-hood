Containers: shipping your agent in a box

Your agent runs beautifully on your laptop; now it must run identically on a cloud machine you will never touch. Meet the box that makes that possible — the Dockerfile line by line, the build-tag-push commands, and the registry where shipped agents wait.

~30 minChapter 111 of 133
The story

Your aunt's biryani is legendary. Relatives beg for the recipe, she gives it freely — and every single attempt fails. Different rice, a different pressure cooker, harder water, a substituted spice because "surely it's the same thing". The dish that emerges in Delhi tastes nothing like the one from her kitchen in Pune. Her verdict, delivered with a shrug: "It works in my kitchen."

So she changes strategy. She stops shipping the recipe and starts shipping a sealed kit: the exact rice, every spice pre-measured, the precise amount of water in a pouch, even the right pot — plus one instruction taped on top: "heat for 20 minutes". Now any kitchen on Earth produces the identical biryani, because the kit carries its whole world with it. She stacks finished kits in a warehouse; family members order one by name — "biryani, version 3" — whenever they want to cook.

The sealed kit is a container image. The steaming pot cooked from one is a container. The warehouse is a registry. This chapter is about building kits for your agent.

The problem: "works on my machine"

Start with a memory from the environments chapter: a virtual environment freezes your Python packages — the exact versions of langchain, anthropic, pydantic your agent was tested with. That solved dependency chaos on one machine. But a venv freezes only the bookshelf, not the room. It does not carry the Python interpreter itself, the operating system's shared libraries, the system tools your code shells out to, or the OS's certificate store that HTTPS quietly depends on.

The moment your agent leaves your laptop for a cloud machine, all of those loose ends bite. The server has Python 3.10 where you tested on 3.12. It is a different Linux flavour with a different version of the SSL library. Someone forgot one pip install. Your agent — the same loop you built in the ReAct chapter — crashes on a machine you cannot even log in to, at 2 a.m., for a reason that does not exist on your laptop.

A container is the fix: a way to run a program inside a sealed, self-sufficient box that carries everything the program needs — a miniature Linux filesystem, the exact Python, the exact packages, your code — so it behaves identically on any machine that can host the box. The "works on my machine" excuse dies here, because you ship the machine.

Note

You have actually met containers once before, wearing a different hat: in the sandboxing chapter they were a wall that keeps an agent in. Here they are a box that carries an agent out — same technology, two superpowers: isolation and portability.

Image vs container: the recipe kit and the cooked dish

Two words carry this whole act, and people mix them up constantly. An image is the sealed kit: a frozen, read-only snapshot containing the OS files, Python, your dependencies, and your code — a file sitting on disk, doing nothing, costing nothing to keep. A container is what you get when you run an image: a live process cooked from that kit, with real memory, real CPU, real log output.

The relationship is one-to-many. From a single image you can start one container, or fifty identical ones — just as one warehouse of biryani kits can feed fifty kitchens tonight. When a container stops, the image is untouched, ready to be run again. When people say "deploy the agent", they mean: put the image somewhere reachable, then ask a machine to run containers from it. Hold that sentence; it is the entire deployment story of this act.

Flip these until each one feels obvious:

The Dockerfile, line by line

A Dockerfile is a small script — usually six to ten lines, living at the root of the repository — that tells the docker build command how to assemble an image. Each line is one instruction, and the instructions read like a recipe: start from a base, set up the counter, add ingredients, state how to cook.

Here is a complete, production-shaped Dockerfile for the kind of small Python agent you built in Act V. This is not a toy — swap in your own file names and this exact shape ships real agents. Click each line:

Dockerfile — at the root of your agent's repo
FROM python:3.12-slim
Every image starts on the shoulders of another. python:3.12-slim is an official, publicly maintained image containing a minimal Linux plus Python 3.12 already installed — the pre-stocked kitchen you begin from. The -slim variant strips out compilers and extras you rarely need, so your final image is hundreds of megabytes smaller. Pinning 3.12 here is what finally freezes the interpreter version a venv never could.
WORKDIR /app
Sets the working directory inside the image to /app, creating it if needed. Every later instruction — and your agent when it runs — starts from this folder. Think of it as cd /app made permanent.
COPY requirements.txt .
Copies exactly one file from your repo into the image's /app (the . means "here, in the WORKDIR"). Why copy this one file alone, before everything else? Caching — the next section is entirely about this line's placement, and it is the single most-quoted Dockerfile trick in code review.
RUN pip install --no-cache-dir -r requirements.txt
Executes a command during the build, baking its results into the image: pip reads the requirements file — the same pinned list from ch20 — and installs every package permanently into the image's Python. This is usually the slowest line, easily 60–90 seconds. --no-cache-dir tells pip not to keep its download cache, which would only bloat the image.
COPY . .
Now copy everything else — agent.py, your tools module, prompt files — from the repo into /app. A sibling file called .dockerignore (same idea as .gitignore from ch21) lists what to leave out: .git, .venv, __pycache__, and crucially .env — more on that danger shortly.
CMD ["python", "agent.py"]
The instruction taped on top of the kit: when someone runs this image, execute this command. Nothing runs at build time — CMD only records the default start-up command for later. The bracket form is a JSON list: the program and each argument as separate strings, exactly the argument-splitting discipline you know from the command line.

Read the shape once more from a distance: base, workdir, dependencies, install, code, start command. Six lines, two halves — everything above COPY . . is the slow-changing world of dependencies; everything below is your fast-changing code. That split is deliberate, and here is why.

Layers and the cache: why line order matters

Each Dockerfile instruction produces a layer — a saved snapshot of what that step changed — and Docker stacks the layers to form the image. The payoff: when you rebuild, Docker walks the Dockerfile from the top and reuses any layer whose instruction and inputs have not changed. The first change it meets invalidates that layer and every layer below it.

Now the trick clicks. You edit agent.py fifty times a day; you touch requirements.txt once a fortnight. Because the requirements copy and the slow pip install sit above COPY . ., a code edit only rebuilds the cheap final layers — the 90-second install is served from cache. Put COPY . . first instead, and every one-line code tweak would re-run the whole install. Run the simulation and watch the cache do its work:

Tip

Try reordering the steps in the simulation — move the COPY . . tuple above the requirements pair — and rerun the "after editing agent.py" build. Watch the slow pip layer lose its CACHED status. You have just reproduced the most common Dockerfile review comment in existence.

Build, run, tag

Three commands take you from a Dockerfile to a running agent. First, the build:

terminal — build the image
docker build -t agent:v1 .

Walk it word by word. docker build reads the Dockerfile and assembles the image, layer by layer, printing each step as it goes. -t agent:v1 tags the result — names it agent, version v1 — so you can refer to it later without quoting a 64-character id. The trailing . is easy to miss and mandatory: it says "build from the current folder", which is where Docker looks for the Dockerfile and the files that COPY pulls in.

Then, the run:

terminal — cook a container from the kit
docker run agent:v1
docker run --env-file .env agent:v1

Line one: docker run agent:v1 creates a fresh container from the image and executes its CMD — your python agent.py starts, inside the box, and its output streams to your terminal. Line two is the version you will actually use: --env-file .env injects environment variables — your ANTHROPIC_API_KEY and friends, exactly the pattern from the config chapter — into the container at start-up. The key rides in through the door at runtime; it is never baked into the image.

Careful

Never COPY a .env file or hard-code an API key into an image. Layers are inspectable: anyone who can pull the image can read every file inside it, and registries keep old versions around for years. Secrets enter a container at runtime — via --env-file locally, and via proper secret stores and workload identity once you reach the cluster (a later chapter in this act). Put .env in .dockerignore today.

The third command, docker tag, looks anticlimactic and is the hinge of everything that follows:

terminal — give the image its shipping address
docker tag agent:v1 myagentsacr.azurecr.io/agent:v1

docker tag gives an existing image an additional name — no copying, no rebuilding, just a second label on the same kit. But look at the shape of the new name: it now starts with a server address. That prefix is how Docker knows where the image should be shipped. An image's full name has three parts:

PartExampleMeaning
registrymyagentsacr.azurecr.ioWhich warehouse — the server that stores it. Omitted = Docker Hub.
repositoryagentWhich product — all versions of one image share a repository.
tagv1Which version. Omitted = latest, which you should treat with suspicion — it is whatever was pushed last, not necessarily newest or stable.

Registries: the library your images live in

A registry is a server whose only job is to store images and hand them out by name — the warehouse in the biryani story, or a library where every book is a sealed machine. You already used one without noticing: FROM python:3.12-slim quietly downloaded that base image from Docker Hub, the big public registry. Anyone can pull public images from it; your company's agent, though, is not going on a public shelf.

Teams therefore run private registries, and since this act lives on Azure, ours is Azure Container Registry (ACR) — a private image warehouse inside your Azure subscription, next door to the Kubernetes cluster that will eventually pull from it. The full journey from laptop to warehouse looks like this:

1. Build — the kit is assembled on your laptop

The build runs entirely on your machine. The resulting image exists only in your local Docker storage — nobody else can see it yet.

docker tag adds a name that starts with the registry's address. Docker reads the destination straight off the name — there is no separate "where to" flag on push.

Push uploads each layer, skipping any the registry already holds — the same cache thinking as builds. From this moment the image is durable, shared, and independent of your laptop.

Any authorised machine — a teammate's laptop, a test server, and in the next chapters a Kubernetes cluster — pulls the image by name and runs bit-for-bit what you built. This pull is exactly what a cluster does when you deploy.

Azure Container Registry: the commands

Creating and using an ACR is four commands. You have already seen az login and resource groups earlier in this act — the registry is just another resource that lives inside one. Note the naming rule before you run anything: an ACR name must be globally unique across all of Azure, lowercase letters and digits only, because it becomes a public DNS name — <registry-name>.azurecr.io.

terminal — create the registry (once, ever)
az acr create --resource-group agents-rg --name myagentsacr --sku Basic

Piece by piece: az acr create asks Azure to provision a container registry. --resource-group agents-rg files it inside the resource group you made earlier in the act — remember the hierarchy: subscription, then resource group, then resources. --name myagentsacr is the globally unique name; Azure will answer with your login server, myagentsacr.azurecr.io. --sku Basic picks the cheapest tier — Basic, Standard and Premium differ in storage and speed, and Basic is plenty while you learn.

terminal — authenticate, address, push
az acr login --name myagentsacr
docker tag agent:v1 myagentsacr.azurecr.io/agent:v1
docker push myagentsacr.azurecr.io/agent:v1

Line one, az acr login --name myagentsacr: using your already-logged-in Azure identity, fetch a short-lived credential for this registry and hand it to Docker. It prints Login Succeeded and expires after a few hours — if a push suddenly fails with an authentication error tomorrow, rerun this line before suspecting anything deeper. Line two is the addressing step you met above. Line three, docker push myagentsacr.azurecr.io/agent:v1, reads the destination from the name and uploads the layers. When it finishes, your agent exists as a named, versioned artifact in your team's private library — pullable by any machine you authorise, whether or not your laptop is even switched on.

The image IS the artifact

Step back — this is the idea the rest of the act stands on. In the git chapter the thing teams shipped around was source code. From here on, it is not: the deployable unit is the image. Code review approves source; the pipeline then builds an image from it, pushes it to ACR, and everything downstream — test environments, staging, production — deploys that exact image by name and tag. Nobody re-installs packages on a server ever again; they pull the sealed kit.

This is why a CI pipeline's final, triumphant log line is not "tests passed" but docker push …/agent:sha-4f2a91c — teams commonly tag images with the git commit id, so every running container can be traced back to the exact source that produced it. Version v1 misbehaving in production? Run v0.9 again — the old kit is still on the shelf, unchanged, and rollback is just running yesterday's image. And when the next chapters put your agent into a Kubernetes pod, the manifest will contain one line that makes all of this click: image: myagentsacr.azurecr.io/agent:v1. The cluster is simply the fiftieth kitchen ordering the same kit.

In the wild

Open almost any deployable repo and look at the root: a Dockerfile and a .dockerignore sitting beside the code is the tell that "the artifact here is an image". In CI logs you will recognise the build's layer-by-layer output, the cache hits (CACHED), and the final push to a registry URL. In merge requests, "you broke the layer cache" means someone moved COPY . . above the dependency install; "pin the base image" means FROM python:3.12-slim, not FROM python:latest. And in incident channels, "which image is prod running?" is the first question asked — answered with a registry name and a tag, exactly the anatomy you can now read at a glance.

Check yourself

1. You run docker run agent:v1 three times. How many images and how many containers exist afterwards?

The image is read-only and untouched by running it. Every docker run creates a new, independent container from that same image — one recipe kit, many dishes.

2. Why does the Dockerfile copy requirements.txt and run pip install before copying the rest of the code?

Docker reuses cached layers from the top until the first change. With the install above COPY . ., editing agent.py only rebuilds the cheap bottom layers; the 90-second pip layer is served from cache.

3. How does docker push know to send your image to Azure Container Registry rather than Docker Hub?

There is no destination flag. The registry address travels inside the name itself — that is why the seemingly boring docker tag step is mandatory before pushing to a private registry.

4. You already used virtual environments (ch20). What does a container freeze that a venv cannot?

A venv pins packages against whatever Python and OS happen to be on the machine. An image carries a miniature Linux filesystem, the exact interpreter, the packages, and your code — which is why it runs identically anywhere.

5. In a container-based team, what exactly does CI hand over for deployment when a change is approved?

The image is the artifact. CI builds it once, tags it (often with the git commit id), and pushes it to the registry; every environment then runs that identical image — and rollback is just running an older tag.

Key takeaways

  • An image is the frozen, read-only kit — OS files, Python, packages, code; a container is a live process cooked from it. One image, many containers.
  • Containers finish the job venvs started: they freeze the whole room — interpreter, system libraries, everything — so your agent runs identically on any machine.
  • The Dockerfile reads base, workdir, dependencies, install, code, start command — and dependencies go above COPY . . so the layer cache spares you the slow install on everyday code edits.
  • docker build -t makes the image, docker run starts a container from it, and docker tag prefixes the registry address that tells docker push where to ship.
  • On Azure the private warehouse is ACR: az acr create once, az acr login per session, then push to <registry-name>.azurecr.io/agent:v1.
  • Secrets never go inside the image — they enter at runtime via environment variables or secret stores, because image layers are readable by anyone who can pull them.
  • The image is the artifact CI ships: environments pull a named tag, rollback is running an older one, and the Kubernetes manifests ahead will reference exactly this registry path.
