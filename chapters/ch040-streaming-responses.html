Streaming responses

The typing effect in every chat UI is not an animation. It is the model's actual heartbeat, forwarded to your screen one small piece at a time — and the machinery behind it is exactly the yield and await you just learned.

~25 minChapter 41 of 117
The story

There are two ways to follow a cricket match. The first: wait for tomorrow's newspaper. Complete, polished, every scorecard detail in place — but you spend the whole evening knowing nothing. The second: switch on the radio. The commentator cannot tell you the final score, because it does not exist yet. So she gives you the match the only way it exists right now — one ball at a time. "Single taken." "Dropped at slip!" "Bowled him!" Your radio stays tuned to one station for the whole innings. Small updates keep arriving down that single open connection. And you keep the running score in your head, revising it after every ball. When the innings ends, she says so — and the broadcast has nothing more to give.

An LLM answering your question is that match in progress. The answer genuinely does not exist yet; the model is producing it token by token, live. A non-streaming API is the newspaper: silence, then everything. Streaming is the radio: the same match, the same total length — but you hear every ball as it happens. This chapter is that broadcast, taken apart: the open connection (server-sent events), the ball-by-ball updates (chunks and deltas), and the running score you keep (accumulating the reply).

You already own both halves of this machine

Quick recap, because this chapter is where the last two finally pay off. In the generators chapter you met functions that pause: a function with yield in it does not return one big result — it hands out values one at a time, freezing between each, resuming only when the consumer asks for the next. You learned that a generator is a promise of a sequence, delivered on demand. Then in the async chapter you met await: a marker that says "I am waiting on the outside world here — someone else can use the CPU in the meantime", letting one Python process juggle many slow conversations at once instead of blocking on each.

Streaming is where those two ideas shake hands. A streamed LLM reply reaches your code as something you loop over — one piece per iteration, produced on demand, exactly the generator shape from ch038. And between pieces there is dead air on the network — gaps that, in a server handling many users, await converts from wasted time into time spent serving somebody else. Every chat product you have ever used is a generator being consumed inside an event loop. By the end of this page you will have built one.

The problem: the model was never silent — you just couldn't hear it

Recall from the tokens chapter that an LLM is autoregressive: it produces its answer one token at a time, each token conditioned on everything before it. There is no moment when the full reply exists inside the model but hasn't been "sent" — the reply comes into existence gradually, over seconds, token by token by token.

A plain, non-streaming API call — the kind you made in your first API call — hides all of that. The server collects every token into a buffer, waits for the model to finish, then ships you the completed message in one lump. Correct, simple, and agonising: for a long answer you stare at a spinner for ten, twenty, thirty seconds while — behind the curtain — perfectly good words are piling up unread.

Streaming simply stops hiding the progress. You add one flag to the request, and the server forwards each small piece the moment it exists, down a connection it deliberately holds open. Nothing about the model changes. Nothing gets faster in total. What collapses is the wait before the first word — the number engineers call time to first token, the delay between sending your request and seeing the first sliver of the answer.

That gap is why streamed UIs feel alive. Humans forgive slowness they can watch; a wall of text assembling itself is progress, while a spinner is a black box that might be broken. There is a practical bonus too: when you can see an answer going wrong from the second sentence, you press stop — instead of paying for twenty more seconds of a reply you were always going to throw away. (Paying for tokens is the whole of the next chapter.)

Server-sent events: one response that refuses to end

So how does a server keep sending after it has started answering? Not with anything exotic. The standard trick is server-sent events (SSE) — an ordinary HTTP response whose body never quite finishes, so the server can keep appending small text records to it, one per update, until it finally says "done" and closes. One request from you; one long, slow-motion response from them; many little messages inside it.

Each record in the body is a few lines of plain text: an optional event: line naming the kind of update, a data: line carrying a small JSON payload, and a blank line to say "record over". Here is what genuinely crosses the wire when you stream from the Anthropic API (a few bookkeeping events trimmed for space):

What the wire actually carries (trimmed)
HTTP/1.1 200 OK
Content-Type: text/event-stream

event: message_start
data: {"type":"message_start","message":{"role":"assistant","content":[]}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"Str"}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"eaming"}}

event: message_stop
data: {"type":"message_stop"}

Read it like the radio broadcast it is. message_start is the commentator clearing her throat. Each content_block_delta is one ball bowled — and notice its payload carries only the new fragment ("Str", then "eaming"), never the text so far. message_stop is stumps. The OpenAI API speaks a dialect of the same thing — bare data: lines with no event names, closed by a literal data: [DONE] — but the shape of the conversation is identical.

Two vocabulary notes before we go on. A chunk is one of those records as your program receives it — one parcel off the wire. A delta is the payload inside: the increment, the change since last time, named after the Greek letter mathematicians use for "difference". And if you have heard of WebSockets: those are a two-way phone call, overkill here. An LLM reply only flows one direction — server to you — and SSE, riding on plain boring HTTP, is exactly one-way. That is why nearly every LLM API chose it.

Chunks, deltas, and the accumulator

That last step deserves its own heading, because it is the single most common surprise for people meeting streaming in code: during a stream, the complete reply exists nowhere. Not on the wire, not in the final event, not in some hidden field. It exists only if your code builds it — an empty string before the first chunk, grown by += on every delta, complete only after the stop event. That humble variable is called the accumulator: the running total you keep while increments flow past, exactly the running score you kept in your head at the radio.

stream=True: what the real SDKs look like

In production you will never hand-parse data: lines — the SDK does the SSE plumbing and hands you Python objects. The switch is one argument: pass stream=True to the same client.messages.create(...) call you know from ch033, and instead of returning one finished message, it returns an iterable of events — a generator you consume with a plain for loop, each iteration quietly waiting for the next chunk to arrive. The Anthropic SDK also offers a friendlier wrapper, client.messages.stream(...), used as a context manager so the connection is always closed properly. Read it line by line:

stream_reply.py — the real thing, Anthropic SDK
from anthropic import Anthropic
The same SDK, the same import, as your first API call in ch033. Streaming is not a different product — it is a different delivery mode for the same endpoint.
client = Anthropic()
reply = ""
The accumulator, empty before the first chunk. Everything the previous section said in prose is this one variable in code.
with client.messages.stream(
The streaming helper, opened as a context manager (ch18): entering opens the SSE connection, leaving guarantees it closes — even if your code crashes mid-stream. The rawer form, messages.create(..., stream=True), gives you the event objects directly.
model="claude-sonnet-4-5",
max_tokens=500,
messages=[{"role": "user", "content": "Explain SSE briefly"}],
The familiar messages list — nothing about the request body changes for streaming beyond the mode itself.
) as stream:
for text in stream.text_stream:
The payoff line. text_stream is a generator (ch38): each turn of this loop waits for the next SSE event, unwraps it, and yields just the text delta. Your cursor sits on this line during every gap between chunks.
reply += text
Feed the accumulator. Skip this and the text scrolls past your terminal once, unrecoverable — the classic first streaming bug.
print(text, end="", flush=True)
The typing effect in one line. end="" stops print adding a newline after every fragment; flush=True pushes each piece to the terminal immediately instead of letting Python batch output. Forget flush and your "stream" appears in frozen lumps.

Two conveniences worth knowing. The helper keeps its own accumulator too — after the loop, stream.get_final_message() hands you the assembled message with usage counts, so you can stream to the screen and log the complete reply. And the OpenAI SDK follows the same pattern with different spelling — the flag itself is in the call:

openai_stream.py — same idea, OpenAI dialect
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Explain SSE briefly"}],
    stream=True,          # the whole feature is this one flag
)
reply = ""
for chunk in stream:
    delta = chunk.choices[0].delta.content or ""   # some chunks carry no text
    reply += delta
    print(delta, end="", flush=True)

Notice the or "": not every chunk carries text (the first mostly announces the assistant role, others carry tool-call fragments or finish reasons), and gluing None onto a string is a crash. Every streaming client you read will have some version of this guard.

Where await joins the party

Run the walkthrough code above in a plain script and it works beautifully — for one user. But look again at that for text in stream.text_stream line: between chunks, your program just sits there, blocked, doing nothing with the dead air. For a script, fine. For a server relaying streams to two hundred chat users at once, fatal — you would need two hundred blocked programs.

This is precisely the problem ch39 gave you the tool for. Swap in the async client and the loop gains two small words:

relay.py — streaming inside an async server
from anthropic import AsyncAnthropic

client = AsyncAnthropic()

async def relay(question):
    async with client.messages.stream(
        model="claude-sonnet-4-5",
        max_tokens=500,
        messages=[{"role": "user", "content": question}],
    ) as stream:
        async for text in stream.text_stream:
            print(text, end="", flush=True)  # or: push to this user's browser

async for means: "give me the next chunk — and while I wait for it, let the event loop serve everyone else." Each gap between deltas, a few tens of milliseconds of network silence, becomes time in which the same single Python process advances a hundred other users' streams. This is the honest answer to a question you may have been carrying since ch39: why is agent and chat-server code so relentlessly async? Because its defining workload — many simultaneous, slow, drip-fed LLM streams — is the exact shape async was invented for. Generators give streaming its interface; await gives it its scale.

Careful

Three traps that bite in production. One: when a model streams a tool call, the arguments arrive as fragments of JSON — {"loca, then tion": "Pu — and feeding a fragment to json.loads raises instantly; you must accumulate until the block completes (the structured-output chapter feels this pain too). Two: because streaming commits to 200 OK before generation finishes, an overloaded-server error can arrive mid-stream as an error event — a status-code check at the start is not enough. Three: streaming changes nothing about billing; every token costs the same whether it arrived in one lump or five hundred deltas. That meter is the next chapter.

In the wild

Open your browser's developer tools while any chat UI is answering and look at the network tab: one request, type eventsource or fetch, response header Content-Type: text/event-stream, body growing line by line — you can watch the raw data: records arrive. In codebases, grep for stream=True, text_stream, or delta and you will land in the relay code; in agent frameworks the same idea surfaces as LangGraph's .stream(...) with its stream_mode options (one of which yields token-by-token messages), and agent CLIs like Claude Code print their answers through exactly this machinery. In a merge request, "accumulate the deltas before parsing" or "flush per chunk" now reads as plain English to you.

Check yourself

1. What does stream=True actually change about an LLM API call?

The model works at the same speed either way. Streaming only changes delivery: pieces are forwarded the moment they exist, so the wait before the first word collapses while total time stays the same.

2. In server-sent events, how do many small updates reach the client?

SSE is one request and one long response that refuses to end: the server writes an "event:" / "data:" record per update into the open body until a final stop record. One-way is all an LLM reply needs, so no WebSocket is required.

3. During a stream, where does the complete reply text live?

Deltas carry only the new fragment, and the stop event carries no text at all. If your code never does reply += delta, the reply scrolls past once and is gone — which is why SDK helpers keep an accumulator for you.

4. Total generation time is identical with or without streaming. Why do streamed UIs still feel dramatically faster?

Perceived speed is about the wait before the first sign of life. Watching text assemble is progress; a spinner is a black box. And seeing an answer go wrong at sentence two lets you cancel instead of paying for the rest.

5. The SDK's stream.text_stream behaves like which Python idea from two chapters ago?

Each turn of the for-loop asks for the next value and waits until one exists — the exact contract of yield from ch38. In async code the same loop becomes async for, so the pauses between chunks are handed to the event loop (ch39).

Key takeaways

  • The typing effect is real: models generate token by token, and streaming just forwards each piece the moment it exists instead of buffering the finished reply.
  • The wire format is server-sent events — one HTTP response held open, carrying small event:/data: records (chunks) whose payloads (deltas) contain only the new fragment.
  • The full reply exists nowhere until your code builds it: keep an accumulator and += every delta, or the text is gone once it scrolls past.
  • stream=True (or the messages.stream(...) helper) turns the SDK's return value from one finished message into a generator you consume with a for-loop — ch38's yield, wearing production clothes.
  • async for lets one server relay hundreds of streams at once by giving the dead air between chunks to other users — ch39's await, and the real reason chat and agent code is async.
  • Streaming changes delivery, not price: every token bills the same either way — which is exactly where the next chapter picks up.
