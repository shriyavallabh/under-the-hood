Why AG-UI over LangGraph and Deep Agents

An honest trade-off: what a graph framework gave you, what drove the move, and — critically — what you now own yourself once orchestration is no longer in the box.

~28 minChapter 122 of 133
The story

Two restaurants, same street. The first leased a full commercial kitchen: a walk-in fridge that keeps every half-finished dish cold and safe overnight, a ticket rail that routes each order to the right station, an expediter at the pass who holds a plate until the customer confirms "yes, medium-rare", and prep cooks who hand work down the line to each other. A complete operation — but behind a solid wall. Diners never see in. And the whole rig is the landlord's fixed layout on a heavy lease; the year he remodels, every workflow in the place has to be relearned.

The second restaurant knocked down that wall and put in a glass pass. Now every diner watches the chef fire, plate, and call each step out loud, live. That glass counter is wildly popular — but be clear about what it is: a contract about how the kitchen shows its work to the room. Not a fridge, not a ticket rail, not an expediter. Take your food behind that glass into a bare unit, and the fridge, the routing, the overnight safety log are all yours to build. That glass pass is AG-UI. This chapter is what you gain by installing it — and exactly what you now carry yourself.

Where this Act has taken you

A quick recap. This Act opened with the agent-to-user gap — the missing third protocol next to MCP and A2A — then taught the event model (the sixteen-odd SCREAMING_SNAKE_CASE events an agent streams out), built a backend that emits them over server-sent events, and wired the CopilotKit frontend that consumes them. So you know how AG-UI works. This chapter answers the harder question a team argues about in a design review: why adopt this thin streaming contract instead of the heavy graph framework we already know? The honest answer has three parts — what the framework gave us, what pushed us to move, and what the move costs. Getting it wrong is expensive either way, so this is the most even-handed chapter in the Act.

What LangGraph and Deep Agents gave you: a whole runtime

Start by being generous to what you might leave behind, because it is a lot. When you built with LangGraph (chapters 58 onward) and Deep Agents (chapters 67 onward), you were not just getting a loop — you were getting a runtime: a full engine that runs your agent, remembers where it was, and enforces the flow. That runtime handed you real machinery, for free:

  • Durable orchestration. The StateGraph compiles into a real execution engine with nodes and edges — a graph you can draw, replay and reason about.
  • Checkpointing and persistence. Checkpointing (chapter 62) saves the agent's whole state after every step to a checkpointer, keyed by a thread_id. Crash mid-run, come back tomorrow, resume from the exact step. That is durability (chapter 86) you did not have to design.
  • Branching and routing. Conditional edges — add_conditional_edges — let the graph fork: send this state to the refund node, that one to the escalation node, based on a function you write once.
  • Built-in state reducers. A state reducer is a rule for merging each step's output into the running state (append to this list, overwrite that field) so concurrent updates combine sanely instead of clobbering.
  • Interrupts. Human-in-the-loop (chapter 63) via interrupt() + Command(resume=…) pauses a run in place, durably, until a person answers — then continues.
  • Subgraphs and subagents. Multi-agent orchestration (chapter 64): a supervisor graph invoking specialist graphs, or a deep agent spinning up scoped helpers — all in one process, one trace.

That is a serviced apartment, fully furnished. For a genuinely complex, long-running, branch-heavy workflow, throwing it away would be reckless. Keep that firmly in mind — this chapter is not here to sell you a downgrade.

The pain that drove a move

So why do teams move at all? Not because the framework is bad — because for a large class of apps you were paying for a whole kitchen to boil an egg. The pain is real and defensible engineering pain, not vendor-bashing:

  • The graph mental model is overhead for a straight line. Many production agents are just a ReAct loop (chapter 46) — think, call a tool, observe, repeat. Expressing that as nodes, edges, channels and reducers is ceremony around what is honestly a while loop. You write graph boilerplate to describe a straight line.
  • Framework coupling and version churn. Your agent's shape is bound to one framework's abstractions and one framework's release cadence. When the framework reorganises its APIs, your code follows whether you wanted the change or not. That is the landlord's remodel.
  • Getting a rich, streaming, interactive UI out is awkward. The graph runs beautifully server-side, but surfacing its live activity — token-by-token text, which tool is firing, shared state the user can edit — to a real frontend was always a bolt-on. There was no first-class, standard agent-to-user channel.
  • Orchestration you did not need. If your app never branches, never runs for hours, never resumes across a crash, the durable-graph machinery is weight you carry and never use.
  • Lock-in. Rewrite everything as a graph and your UI, your team's skills and your deploy story are all fused to that one runtime. Swapping it later means a rewrite.

What AG-UI gives instead: a streaming contract, not a runtime

AG-UI — the Agent-User Interaction Protocol, created by CopilotKit — is an open, event-based standard for how an agent backend streams its work to a user-facing frontend, and how the UI sends state and actions back. It is the third member of the trio you have met across this handbook: MCP connects an agent to its tools, A2A connects an agent to other agents, and AG-UI connects an agent to the human watching it. Concretely it gives you, first-class and standardised: live streaming of text and tool calls, tool-call visibility, shared bidirectional agent-and-UI state, generative UI (the agent asking the frontend to render a real component), and human-in-the-loop — all as a wire format any frontend can consume.

Here is the sentence that governs the whole chapter: AG-UI is a protocol, not a runtime. It standardises the shape of what you stream out. It does not give you a control loop, orchestration, checkpointing, durable state, or branching. The glass pass shows the kitchen's work; it is not the fridge. That is not a flaw — it is the design decision — but it is exactly the line where your new responsibilities begin.

Note

Because it is a wire contract and not an engine, "AG-UI versus LangGraph" is partly a false binary. They sit on different layers: LangGraph is orchestration (agent↔runtime), AG-UI is presentation (agent↔user). You can run AG-UI in front of a LangGraph agent — an adapter exists. The real decision this chapter frames is narrower and more honest: for this app, do you also drop the heavy graph runtime and drive AG-UI from a light hand-loop instead? Sometimes yes, sometimes emphatically no.

The EventEncoder turns a typed event object into the exact SSE string data: {json}\n\n. It is the only piece that knows the transport; your logic just yields events.
@app.post("/")
A single POST endpoint. The frontend POSTs a RunAgentInput here (fields: threadId, runId, messages, state, tools, context) and the backend streams events back. That is the whole protocol surface.
async def run(input: RunAgentInput, request: Request):
input carries the conversation so far and the current shared state. You decide what to do with it — this function is your control loop, and nothing in AG-UI wrote it.
encoder = EventEncoder(accept=request.headers.get("accept"))
The encoder is created per request from the client's Accept header, so it can pick plain SSE or the high-performance binary encoding the client asked for.
async def gen():
An async generator (chapters 39–40): each yield pushes one encoded event down the open HTTP connection the instant it is ready. This is why the user sees tokens appear live instead of waiting for the whole answer.
yield encoder.encode(RunStartedEvent(
Every run opens with RUN_STARTED and closes with RUN_FINISHED (or RUN_ERROR). Between them you emit whatever text, tool-call and state events describe the work.
type=EventType.RUN_STARTED,
thread_id=input.thread_id, run_id=input.run_id))
# ... your loop: call the model, run tools, emit events ...
This comment is the whole point of the chapter. The loop, the tool execution, the retries, any durable state — all of it lives here, hand-written. AG-UI never fills this in.
yield encoder.encode(RunFinishedEvent(
type=EventType.RUN_FINISHED,
thread_id=input.thread_id, run_id=input.run_id))
return StreamingResponse(gen(), media_type="text/event-stream")
text/event-stream is the default AG-UI transport: HTTP + server-sent events. Transport-agnostic by design (a binary option and WebSockets exist), but SSE is the one you will meet first.

And the freedom this buys: because the contract is standard, the frontend does not care what runtime sits behind it. The same React code below works whether the backend is a raw LLM loop, CrewAI, Mastra, Pydantic AI, or LangGraph — you can swap the engine without touching the UI. That is the anti-lock-in payoff, made concrete:

app.tsx — the CopilotKit frontend (unchanged when you swap the backend)
import { CopilotKit, useCoAgent } from "@copilotkit/react-core";
import { CopilotChat } from "@copilotkit/react-ui";

function BalanceApp() {
  // Bidirectional shared state. The agent's STATE_SNAPSHOT / STATE_DELTA
  // events flow into `state`; setState() pushes edits back to the agent.
  const { state, setState, running } = useCoAgent({
    name: "myAgent",
    initialState: { status: "idle" },
  });

  return (
    <div>
      <p>Agent status: {state.status} {running ? "(working)" : ""}</p>
      <CopilotChat />
    </div>
  );
}

export default function App() {
  return (
    <CopilotKit runtimeUrl="/api/copilotkit" agent="myAgent">
      <BalanceApp />
    </CopilotKit>
  );
}

So which do you actually pick?

This is where honesty earns its keep. There is no universal winner — there is a fit question, and you already have the tool for it: the capability-matrix method (chapter 96) and the is-LangGraph-right test (chapter 94). Score your actual app against what each layer is strong at:

Keep the graph runtime (LangGraph / Deep Agents) when…Go AG-UI + a light loop when…
The workflow is genuinely complex: many steps, real branching, multiple cooperating agentsThe agent is essentially a straight ReAct loop — one tool at a time until done
You need durable, resumable state across crashes and days — built-in checkpointing is a hard requirementRuns are short-lived; if one fails, restarting is fine
Human-in-the-loop must survive a process restart, not just a page reloadThe product is streaming-first and UI-centric — the live view is the value
Conditional routing and state-merge rules are core to the logicYou want to avoid framework lock-in and keep freedom to swap the backend

And the mature answer sits in the middle: use both, on their own layers. Keep LangGraph where you need durable orchestration, and put AG-UI in front of it for a first-class live view. Reach for AG-UI-plus-a-hand-loop only when the app truly does not need the graph's runtime. Choosing the light path because "graphs feel heavy" while quietly needing checkpointing and branching is how teams reinvent LangGraph badly, one bug at a time.

The vocabulary, one more time

Flip until the line between "protocol" and "runtime" is reflex:

Who runs the standard

Provenance, briefly, because it shapes trust. AG-UI was created and open-sourced by CopilotKit — the team behind the React copilot toolkit — to fill the agent-to-user gap that MCP (agent↔tools, seeded by Anthropic) and A2A (agent↔agents, now under the Linux Foundation) left open. It is deliberately framework-agnostic: the same event contract can front a raw LLM loop, CrewAI, Mastra, Agno, Pydantic AI, LlamaIndex, the Microsoft Agent Framework, or LangGraph, and integrations for those ship as adapters. The spec, the Python SDK (ag_ui.core, ag_ui.encoder) and the CopilotKit frontend are all public at docs.ag-ui.com and docs.copilotkit.ai. It has reached a 1.0-era spec — one sign of maturing being that the newer REASONING_* events replaced the deprecated THINKING_* set, which was removed on the way to 1.0.

An honest maturity check

Now the part the launch blog skips. AG-UI is young — it arrived in 2025, and that THINKING_* to REASONING_* removal is the kind of churn a not-yet-settled protocol goes through. The richest path today is CopilotKit's own React stack; the "front any backend" promise is real, but adapters vary in maturity, so verify your framework's adapter rather than assume parity. And the deepest caveat is the one this chapter is built around: AG-UI solves presentation, not orchestration, on purpose. The moment you drop the graph and drive AG-UI from a hand-loop, every runtime concern in that fourth table column becomes yours, forever — a fine trade for a streaming-first app, a bad one for a durable, branch-heavy workflow. The protocol is worth learning now; just size the runtime you are signing up to own.

In the wild

You will meet AG-UI first as a dependency — @copilotkit/react-core in a frontend's package.json, ag-ui / ag_ui in a backend's, and a single @app.post("/") returning a StreamingResponse with media_type="text/event-stream". In the browser dev-tools Network tab, the tell is one long-lived request whose response is a stream of data: {"type":"TEXT_MESSAGE_CONTENT",…} lines. In a design review, "let's put AG-UI in front of the agent" means: expose the backend as an event stream and adopt CopilotKit, without necessarily changing the runtime underneath. And when a teammate says "we dropped the graph and hand-rolled the loop behind AG-UI", your next question — the one this chapter trained — is: "so who owns checkpointing and resume now?"

Check yourself

1. What is the single most important thing to understand about AG-UI relative to LangGraph?

AG-UI standardises the shape of what an agent streams to the user. It is the glass pass, not the kitchen. LangGraph is a runtime that actually runs your agent — a different layer of the stack.

2. Your app is a streaming chat surface: one tool at a time, short runs, and the live view of the agent's work is the whole product. Which fit is most defensible?

A straight ReAct loop with no branching and no need to resume across crashes does not exercise the graph runtime. AG-UI's strengths (streaming, tool visibility, shared state) are exactly this product's value.

3. You drop the graph and drive AG-UI from a hand-written loop. Which of these is now YOUR responsibility that LangGraph used to handle?

Streaming and shared state are exactly what AG-UI gives you for free. Checkpointing, the control loop, branching and retries are runtime concerns AG-UI never provides — they move into your code.

4. In the wire event STATE_DELTA, how is the state update represented?

STATE_DELTA carries just the change as a JSON Patch array ({"op":"replace","path":…,"value":…}). The full-state variant is a separate event, STATE_SNAPSHOT.

5. A teammate argues "AG-UI versus LangGraph — we must choose one". Why is that framing partly wrong?

AG-UI is agent↔user; LangGraph is agent↔runtime. An adapter lets AG-UI front a LangGraph backend, so adopting AG-UI is not automatically leaving LangGraph. The narrower, honest decision is whether this app needs the graph's durable orchestration at all.

Key takeaways

  • LangGraph and Deep Agents gave you a whole runtime — durable orchestration, checkpointing, conditional branching, state reducers, interrupts and subgraphs. For complex, long-running, branch-heavy workflows that is worth keeping.
  • The pain that drives a move is real: graph boilerplate around what is often a straight loop, framework coupling and version churn, an awkward path to a rich streaming UI, orchestration many apps never use, and lock-in.
  • AG-UI is a thin, open, framework-agnostic streaming contract between agent and UI — first-class streaming, tool visibility, shared bidirectional state, generative UI and human-in-the-loop. It is a protocol, not a runtime.
  • The critical honest part: with AG-UI you now own the control loop, orchestration, checkpointing and durable resume, branching, and retries yourself. AG-UI's strengths face the human; the graph framework's strengths face the runtime.
  • There is no universal winner — score your real app with the capability-matrix method (chapter 96) and the is-LangGraph-right test (chapter 94). Keep the graph for durable, branch-heavy, resumable work; go AG-UI-plus-a-light-loop for streaming-first, UI-centric agents.
  • The mature pattern is both, on their own layers: run AG-UI in front of your runtime. "AG-UI vs LangGraph" is partly a false binary — the honest question is whether this app needs the graph's runtime at all.
