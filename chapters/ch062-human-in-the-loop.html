Human-in-the-loop: interrupts

A graph that can stop mid-step, wait a week for a human signature, and pick up exactly where it froze — this is the mechanism that lets agents touch money, production, and anything else you are not willing to let a model do alone.

~25 minChapter 63 of 133
The story

Thursday afternoon at a bank branch. A clerk prepares an urgent transfer: 4,85,000 rupees to a supplier. The rulebook is blunt — anything above fifty thousand needs the branch manager's signature. But the clerk does not stand outside the manager's office holding the form. She writes everything onto it — payee, account, amount, the checks she already did — stamps it PENDING APPROVAL, drops it into the approvals tray, and gets on with her day. The manager, as it happens, is away until Monday.

Monday morning, the manager reads the form, signs it, and drops it back. A clerk — not even the same one — pulls it from the tray and continues from precisely where the work stopped. Nobody re-did the earlier steps. Nobody's memory mattered. The branch could have had a power cut all weekend and the process would still be intact, because the process lived on the form, not in anyone's head.

That tray is interrupt(). The form is a checkpoint. The signature is Command(resume="yes"). This chapter is about teaching your graph to use the tray.

Where we left off — and the door it unlocks

Quick recap of the checkpointing chapter, because everything here stands on it. A checkpointer is a save-game system for a graph: after every node, LangGraph writes the full state to a store — memory, SQLite, Postgres, Redis — keyed by a thread_id, which is just a name for one ongoing run. Pass the same config={"configurable": {"thread_id": ...}} later and the graph reloads that saved state and carries on. We framed it as crash insurance and as conversation memory.

But there is a third use, and it is the most valuable one: if a graph can survive an accidental stop, it can afford to stop on purpose. Deliberately. Mid-task. To ask a human something. The pause costs nothing — no server thread sits blocked, no process waits in memory — because the run is not "waiting" anywhere. It is simply a row in a database, like the form in the tray, until somebody comes back with an answer. That deliberate pause is what LangGraph calls an interrupt: a point where the graph stops itself and hands a question up to whoever is operating it.

What interrupt() actually does

The API is one function and one class, both from langgraph.types: interrupt() and Command. You call interrupt(payload) inside a node, at the exact line where you want a human's input. The payload is whatever the human needs to see — a question, a draft, a payment object. Under the hood, interrupt() raises a special exception. The LangGraph runtime catches it, saves everything to the checkpointer, and ends the run — not with a crash, but with a polite report: the result you get back from .invoke() carries an __interrupt__ entry containing your payload. Your application shows that payload to a human however it likes: a web form, a chat message, an approvals inbox.

Then — minutes, hours, or days later — someone answers. You call .invoke() again on the same thread_id, but instead of fresh input you pass Command(resume=answer). The runtime reloads the checkpoint, re-enters the interrupted node, and this time the very same interrupt() call does not raise — it returns the human's answer, as if the function had simply taken a very long time to come back. Your node code reads the decision and carries on down the graph.

The invoke() call returns. The process is free — it can serve other users, or exit, or the whole server can reboot. The run now exists only as saved state, like the form in the tray.

4. The world takes its time

This pause can last five seconds or five days. Nothing is consuming tokens, nothing is holding a connection open. Durable pauses are the point: real approvals happen on human timescales.

The approvals UI translates the click into a second invoke() on the same thread, carrying the answer. The runtime pulls the frozen state back out of the checkpointer.

6. The gate re-runs, interrupt() returns, money moves

draft nodenot re-run
→
gate nodeinterrupt() returns "yes"
→
send nodepayment executes

The interrupted node runs again from its top; this time interrupt() returns the human's answer instead of pausing. The graph flows on to the send node and finishes. One recorded human decision sits between the draft and the money.

Careful

Step 6 hides the one real gotcha: on resume, the interrupted node is re-executed from its first line, not from the interrupt() call. LangGraph rewinds to the start of the node and replays it; the replayed interrupt() then returns the answer. So any code before interrupt() in that node runs twice. Keep it cheap and repeatable — and never put the irreversible action (the actual bank call) before the interrupt in the same node. Draft in one node, gate in another, act in a third.

Plain edges throughout — the gate is not a conditional edge. Routing decides where to go; an interrupt decides whether the run may continue at all. They compose nicely, but they are different tools.
graph = builder.compile(checkpointer=InMemorySaver())
Compile with the checkpointer attached. Forget this and the pause still fires — invoke() returns an __interrupt__ — but the frozen state has nowhere to live: the resume call fails with RuntimeError: Cannot use Command(resume=...) without checkpointer, and a restart loses the run entirely.

And here is what running it looks like — two invocations, possibly days apart, possibly from different processes:

run_it.py — pause, then resume

result = graph.invoke({}, config)
print(result["__interrupt__"][0].value)
# {'question': 'Approve this payment?',
#  'payment': {'to': 'Meridian Traders', 'amount_inr': 485000}}

# ... hours or days later, after a human clicks a button ...

print(final["receipt"])   # the transfer reference from bank_api
Note

You can also inspect a paused run without resuming it: graph.get_state(config) returns the saved state plus which node is up next — handy for building an approvals dashboard that lists every run currently waiting on a human. And LangGraph still supports an older, blunter mechanism — compile(interrupt_before=["send"]) — which pauses before a named node with no payload. It is useful for debugging, but interrupt() is the one to build products on: it pauses at an exact line, carries a payload out, and carries an answer back in.

Beyond yes and no: three shapes of human input

Approve-or-reject is the headline pattern, but Command(resume=...) carries any Python value, which gives you three distinct conversations between graph and human:

Approve or reject. Resume with "yes" or "no" (or a boolean). The gate node turns it into state; a later node — or a conditional edge — acts on it. This is the pattern for payments, deployments, sending emails to customers, and deleting anything.

Edit, then continue. Resume with a modified version of the payload itself. The human does not just bless the draft — they fix it: correct the amount, soften the wording of an email, trim a tool call's arguments. The node receives the edited object from interrupt() and writes it into state as if the model had produced it. Human judgement becomes part of the pipeline's data, not just its permission slip.

Answer a question. The agent hits a genuine unknown — "two customers share this name; which one?" — and interrupts with the question. The resume value is information the model could never have derived alone. This turns interrupts from a safety mechanism into a collaboration mechanism: the graph does the legwork, the human supplies the judgement calls, each exactly where they are strongest.

One more detail for completeness: if several branches of a graph pause at once — parallel nodes can each call interrupt() — the __interrupt__ list carries one entry per pause, each with its own id, and you can resume them individually by mapping ids to answers in the resume payload. File it under "the docs have you covered when you get there".

Why regulators love this

Now step back from the code and look at what you have built, because this is the part your risk and compliance colleagues will actually care about.

Banks and regulated enterprises have run on the four-eyes principle — also called maker-checker — for a century: the person who prepares a transaction must not be the person who approves it. Every core banking system encodes it; auditors test for it. The uncomfortable question of the agent era is: if a model drafts and executes, where did the second pair of eyes go? An interrupt gate is the literal, code-level answer. The agent is the maker; the human at the interrupt is the checker; and the graph's topology guarantees the send node is unreachable except through the gate. That is not a policy document promising review happens — it is a structural property you can point at in the code.

Three more properties fall out for free. First, the audit trail: because every pause and resume flows through the checkpointer, the database holds the exact state the human saw, what they answered, and when — precisely the evidence an auditor asks for, produced as a side effect of the architecture rather than as extra logging work. Second, accountability lands at the right moment: the human decision is recorded at the instant of the irreversible action, not in a blanket "the team reviews outputs weekly" sense. Third, human timescales are respected: because a paused run is just stored state, approval can take a weekend without any process burning memory or tokens — which means nobody is ever tempted to weaken the control "because holding the run open is expensive". The cheap pause is what makes the strong control sustainable.

This is also why the argument for graphs from the start of this act keeps paying off: a free-running agent loop can be asked to seek approval in its prompt; a graph can be built so that skipping approval is topologically impossible. Regulators, quite reasonably, prefer the second kind of promise.

In the wild

You will meet interrupts wherever an agent touches something irreversible. In LangGraph codebases, search for interrupt( or Command(resume — approval gates cluster around nodes named send_*, execute_*, deploy_*, or apply_*. Agent platforms surface the same idea as "approval policies" or "tool review" settings; coding assistants like Claude Code do it every time they stop to ask before running a shell command — that permission prompt is an interrupt with a nicer UI. And in a merge request, when someone writes "put a HITL gate on the refund tool" or "this needs to go through the approvals inbox", they are asking for exactly the three-node pattern you built in this chapter: draft, gate, act.

Check yourself

1. A graph hits interrupt() and pauses. What is happening on the server while it waits three days for an approval?

The pause is durable, not busy. invoke() returned long ago; the run is a checkpoint row keyed by thread_id. The server can restart mid-wait and nothing is lost — that is why approvals can take a weekend for free.

2. After a human approves, you call graph.invoke(Command(resume="yes"), config). Where does that "yes" end up?

Command(resume=value) feeds the value back into the exact interrupt() that paused the run — the node re-executes, interrupt() returns "yes" instead of raising, and the node code carries on with the answer in hand.

3. A colleague writes one node that charges a card and then calls interrupt() to confirm sending the receipt email. What goes wrong?

Resume replays the interrupted node from the top; only then does interrupt() return the answer. Any side effect placed before the interrupt runs twice. The fix is structural: irreversible actions live in their own node, after the gate.

4. Why does resuming an interrupt require a checkpointer attached at compile time?

An interrupt is a deliberate stop that must survive until someone comes back — possibly after the process itself has exited. That only works if the full state is persisted somewhere, which is exactly the checkpointer's job.

5. What makes an interrupt gate more convincing to an auditor than a prompt instructing the agent to "always ask before paying"?

A prompt is a polite request to a probabilistic system; the model might comply. A gate is topology: there is no edge to the send node except through the interrupt, and the checkpointer records who answered what, when. Structural controls beat behavioural promises.

Key takeaways

  • interrupt(payload), called inside a node, pauses the entire run at that line and surfaces the payload to your application via the __interrupt__ entry in the result.
  • The pause is durable because the checkpointer freezes the full state under a thread_id — no process waits, no tokens burn, and the approval can take days across restarts.
  • graph.invoke(Command(resume=value), config) on the same thread reloads the state, re-runs the interrupted node from its top, and the replayed interrupt() returns the human's value.
  • Because the node re-executes on resume, keep everything before the interrupt cheap and repeatable — irreversible actions belong in their own node, on the far side of the gate.
  • The resume value can be a verdict, an edited draft, or an answer to a question — interrupts are a collaboration channel, not just a permission prompt.
  • Regulators love the pattern because it is maker-checker as topology: the consequential node is structurally unreachable without a recorded human decision — a theme bring your own graph takes to its conclusion.
