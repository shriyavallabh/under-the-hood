MCP architecture

Three roles — host, client, server — one wire format underneath, and a strict handshake before any business. By the end of this chapter you can draw exactly who talks to whom, and read the messages as they fly past.

~30 minChapter 76 of 117
The story

Picture a country's embassy in a busy foreign capital. The ambassador — a brilliant, overcommitted person — never phones a ministry directly. Instead, for every ministry the embassy deals with (trade, visas, agriculture), it appoints one dedicated attaché, and that attaché handles that one ministry and nothing else. Every attaché is trained in the same diplomatic protocol: every cable carries a number, every numbered cable gets a reply carrying the same number, and unnumbered notes are courtesies that expect no answer at all.

And before any business happens — before a single favour is asked — there are formalities. The attaché presents credentials: "Here is who we are, and here is what we are equipped to discuss." The ministry replies in kind: "Understood. Here is what we offer." Only after this exchange does anyone request anything.

One more detail. Some ministries sit downstairs in the same building — the attaché walks a memo along the corridor. Others are across an ocean — cables go over the wire. The protocol is identical either way; only the carrying differs. Hold this embassy in your head: it is the entire architecture of MCP, role for role, message for message.

Three roles, one strict rule

Quick recall before the new material. In the previous chapter you watched the industry drown in the M-by-N problem — every agent wired to every tool with bespoke glue, M × N adapters and counting — until one open agreement, the Model Context Protocol, collapsed the mess the way USB-C collapsed the charger shoebox. That chapter was the why. This chapter is the what: when an app talks to an MCP server, what actually happens, in what order, carried by what?

MCP names exactly three roles, and the embassy maps onto them one-to-one.

The host is the application you actually use — the embassy itself. Claude Code in your terminal, a desktop chat app, an IDE, your company's agent harness. The host owns the user interface, owns the conversation, calls the LLM, and decides — with your consent — what the model is allowed to touch. The host is where the agent loop you built by hand lives.

The server is the tool provider — the ministry. It is a separate program, written by anyone, that exposes capabilities in the standard shape: a git server exposes repository tools, a database server exposes query tools, a browser server exposes navigation tools. The server knows nothing about which app is calling it. That ignorance is the whole point — write it once, and every MCP host on Earth can use it.

The client is the piece people forget, and the piece exam questions love: the dedicated attaché. A client is a connector object inside the host that maintains a session with exactly one server. This is the strict rule of MCP architecture: one client per server, always. A host talking to five servers holds five clients, each with its own private, stateful, one-to-one conversation. The host is the diplomat's brain; the clients are its telephone lines.

Hostthe app: UI, agent loop, LLM calls
→
Client Aconnector, 1:1
Client Bconnector, 1:1
→
Server Agit tools
Server Binvoice tools

Read it left to right: one host, one client per configured server, one private line each. Add a third server to your config tomorrow and the host quietly creates a third client — nothing else changes. That is exactly how last chapter's M × N wall became M + N.

Note

Everyday speech says "I connected Claude to an MCP server" and never mentions the client — fine, since it is built into the host and invisible in daily use. But in protocol docs, error logs, and SDK code, client always means this per-server connector — never "the customer", never the app as a whole.

The vocabulary you need

Six terms carry everything that follows. Flip until boring:

The wire underneath: JSON-RPC

Peel back any MCP connection and you find JSON-RPC 2.0 — a tiny, twenty-year-old convention for making requests as JSON text. It defines just three message shapes. A request carries a method name, optional params, and an id number. A response carries the same id plus either a result or an error — the id is how replies get matched to questions when several are in flight. And a notification is a request with no id at all: fire-and-forget, no reply expected. That is the numbered-cables rule from the embassy, verbatim.

You already speak the payload language — it is plain JSON. Below is a real session transcript, exactly what flows across the pipe, one message per line. Click the annotated lines and read it like a reviewer:

the wire — one JSON-RPC message per line, client to server and back
{"jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {"protocolVersion": "2025-06-18", "capabilities": {}, "clientInfo": {"name": "my-host", "version": "1.0"}}}
The opening cable, client to server. Every message carries the "jsonrpc": "2.0" tag; id: 1 means "this is a request — match your reply to it". The client states which spec revision it speaks (the version is literally a date), its optional features, and who it is. Nothing may be sent before this.
{"jsonrpc": "2.0", "id": 1, "result": {"protocolVersion": "2025-06-18", "capabilities": {"tools": {"listChanged": true}}, "serverInfo": {"name": "invoice-server", "version": "0.1"}}}
The server answers the same id: agreed version, then — the important part — its capabilities: "I serve tools, and I will notify you if my tool list ever changes." Capability discovery in one line. A server offering resources or prompts would declare them here too.
{"jsonrpc": "2.0", "method": "notifications/initialized"}
Look closely: no id. This is a notification — the fire-and-forget kind. The client is saying "formalities complete, ready for business", and expects no reply. Only after this may normal traffic flow.
{"jsonrpc": "2.0", "id": 2, "method": "tools/list"}
First real request: "show me the menu." The client asks because the handshake said tools exist — it still does not know which tools.
{"jsonrpc": "2.0", "id": 2, "result": {"tools": [{"name": "get_invoice_total", "description": "Return the total of one invoice.", "inputSchema": {"type": "object", "properties": {"invoice_id": {"type": "string"}}}}]}}
The menu arrives: name, description, input schema — the exact three fields you met in the tools chapter. The host will hand these straight to the LLM as its tool menu. This is the moment a stranger's server becomes the model's capability.
{"jsonrpc": "2.0", "id": 3, "method": "tools/call", "params": {"name": "get_invoice_total", "arguments": {"invoice_id": "INV-042"}}}
Later, the model has chosen the tool (via native tool calling), and the host routes the call to this server's client, which relays it as tools/call. The model never touches the wire — the client speaks for it.
{"jsonrpc": "2.0", "id": 3, "result": {"content": [{"type": "text", "text": "1500"}]}}
The result comes back wrapped in content blocks, travels up through the client to the host, and lands in the model's context as the observation — closing the same think-act-observe circle you built in the ReAct chapter.

Seven lines of JSON, and you have seen an entire MCP session: handshake, discovery, one tool call. Everything else in the protocol is a variation on these shapes.

The handshake: three messages before any business

The transcript above hid a rule worth stating outright. An MCP session always opens with the same three-beat initialization handshake — a fixed opening sequence both sides must complete before anything else is allowed:

First, the client sends initialize: its protocol version, its capabilities, its name. Second, the server responds: the version it agrees to, its capabilities, its name. Third, the client sends the notifications/initialized notification — the "ready" bell. Then, and only then, normal traffic begins.

Why so formal? Because capability negotiation — each side declaring up front which optional features it supports — is what lets the protocol evolve without breaking anyone. A server that only serves tools says so; a client that supports fancy extras says so; both sides then know exactly what they may ask of each other. Neither ever assumes. Newer spec additions (long-running tasks, servers asking the user follow-up questions) slot in the same way: declared as capabilities during this handshake, ignored by parties that do not speak them.

Two transports: same messages, different pipes

Everything above was about what is said. The transport is how it travels, and MCP standardises two.

stdio — standard input/output — is the corridor inside the embassy. The host launches the server as a local subprocess, then writes JSON-RPC messages into the server's stdin and reads replies from its stdout, one message per line. It is beautifully simple: no network, no ports, no certificates, and the server dies when the host does. This is how most personal-machine servers run — the filesystem server, the git server, the little utilities you will build in the build-a-server chapter.

Streamable HTTP is the overseas cable. The server runs as an independent process — in your cloud, or run by a vendor — and exposes a single HTTP endpoint. The client POSTs each JSON-RPC message to it; the server answers with plain JSON, or opens a stream of server-sent events to push several messages back (progress, notifications) over one request. A session id in an HTTP header tells its many clients apart. One server, many hosts — the transport of shared and remote infrastructure.

Careful

The classic stdio bug, seen in the wild weekly: a server author adds a friendly print("got a request!") for debugging. But in a stdio server, stdout is the wire — that print statement injects non-JSON into the protocol stream and the client chokes with a parse error or hangs. Diagnostics belong on stderr, which stdio transports leave free for logging. If a local MCP server "mysteriously disconnects" the moment it does anything, look for a stray print first.

Note

Older docs mention a separate "HTTP + SSE" transport with two endpoints. That was the first remote transport; the spec replaced it with streamable HTTP, which does the same job over a single endpoint. You will still meet legacy servers speaking it — treat it as the previous model year, not a third architecture.

Where you will actually touch this

The comforting part: you will almost never write the client, and rarely the raw JSON. The host ships the client built in; your touchpoint is a few lines of configuration naming the servers — and the transport is visible right there in the shape of each entry:

a typical host config — one stdio server, one remote server
{
  "mcpServers": {
    "invoices": {
      "command": "python",
      "args": ["invoice_server.py"]
    },
    "team-search": {
      "url": "https://mcp.example.com/mcp"
    }
  }
}

A command entry means stdio — the host launches that program as a subprocess and speaks down its pipes. A url entry means streamable HTTP — the host POSTs to that endpoint. Behind either line: a client created, a transport connected, a three-message handshake, a tools/list, a menu handed to the model. You now know every step that config quietly triggers.

On the server side, SDKs hide the plumbing just as thoroughly. In Python, FastMCP from the official SDK lets you decorate a plain function with @mcp.tool() and run it with mcp.run(transport="stdio") — or flip the same server to streamable HTTP with a different transport string. You will build one end to end in the build-a-server chapter. On the framework side, langchain-mcp-adapters gives LangChain and LangGraph agents a MultiServerMCPClient that manages a fleet of these one-to-one clients for you — covered in the MCP-in-LangGraph chapter.

In the wild

You will meet this architecture as config blocks and log lines. In a repo: a .mcp.json or app settings file whose entries you can now read the transport straight off — command/args is stdio, url is streamable HTTP. In logs: "failed to initialize MCP server" means the handshake never completed (wrong path, crashed subprocess, unreachable URL); "server disconnected unexpectedly" from a local server is very often the stdout-print bug from the Careful box. And in vendor announcements, "we now ship an MCP server" means: their system is a ministry now — your host already has the attaché.

Check yourself

1. Your host app is configured with three MCP servers. How many MCP clients exist?

One client per server is the strict rule. The host creates and owns the clients; each maintains its own private session. The LLM never touches the wire at all — the host relays on its behalf.

2. What must happen before a client may send tools/list?

Every session opens with the same three beats: initialize request, initialize response (capabilities both ways), then the initialized notification. Only then does normal traffic — discovery included — begin.

3. The tool server your team needs runs on a cloud VM, shared by twelve people. Which transport, and why?

stdio literally means the host launches the server and holds its stdin/stdout pipes — impossible across machines. Remote or shared means streamable HTTP: an independent server process, one endpoint, many clients told apart by session.

4. In the wire transcript, notifications/initialized had no id field. What does that tell you?

JSON-RPC uses the id to match replies to requests. A message without one is a notification: the sender expects nothing back. Requests and responses always carry ids; notifications never do.

5. A developer adds print("debug!") inside a tool in their stdio MCP server. What happens?

In a stdio server, stdout carries the protocol itself: one JSON-RPC message per line. A stray print injects garbage mid-stream and the client fails to parse. Diagnostics go to stderr, which the transport leaves free.

Key takeaways

  • MCP has exactly three roles: the host (the app — UI, agent loop, LLM calls), clients (connectors inside the host, strictly one per server), and servers (independent tool providers that know nothing about who calls them).
  • Underneath, everything is JSON-RPC 2.0: numbered requests matched to same-numbered responses, plus unnumbered fire-and-forget notifications.
  • No business before the handshake — initialize (versions and capabilities exchanged both ways), then the initialized notification. Capabilities are always declared, never assumed; that is how the protocol evolves without breaking anyone.
  • Discovery is just a request: tools/list returns names, descriptions, and schemas, which the host hands to the LLM as its tool menu; tools/call carries the model's choice back down the chain.
  • Two transports carry identical messages: stdio for a local subprocess (JSON per line on stdin/stdout — keep prints off stdout!), streamable HTTP for anything remote or shared (POST to one endpoint, streamed replies when needed).
  • In daily life you touch only config: a command entry means stdio, a url entry means streamable HTTP — and behind either line sits the whole handshake you can now draw from memory.
