Just enough git to explore

Four moves — clone, pull, log, and reading a diff — and every codebase on the internet becomes a place you can walk into, wander around, and read like a story. No fear, nothing breakable, today.

~25 minChapter 21 of 133
The story

A senior lawyer keeps the master file for a long-running contract. Every revision that was ever made is in that file — not just the current wording, but every earlier draft, each one stamped with who changed it, when, and a one-line note saying why. Nothing is ever torn out. The current contract is simply the top page; the history underneath is what makes it trustworthy.

A new associate joins. She does not borrow the master file — she gets a complete photocopy of it, history and all, to keep on her own desk. Each morning she collects whatever new pages were added overnight. When she wants to understand a clause, she flips back through the register of revisions. And when she wants to know exactly what changed between two drafts, she reads the redline: the old words struck out in red, the new words inked in green.

That master file is a repository. The photocopy is a clone. The morning collection is a pull. The register is the log. The redline is a diff. You already know how to work this way — you just have not met the commands yet.

A folder that remembers everything

A repository (everyone says repo) is a project folder that carries its own complete history — every version of every file, plus a record of who changed what, when, and why. From the outside it looks like any ordinary folder: the same packages and modules and YAML and JSON config files you already know, plus the pyproject.toml from the previous chapter. The history lives in one hidden subfolder called .git, a small database that the git command reads and writes. Delete that hidden folder and you have a plain folder with amnesia; keep it and the folder remembers everything back to day one.

History is recorded as a chain of commits — snapshots of the whole project at a moment in time, each stamped with an author, a date, and a short message explaining the change. A commit is the lawyer's stamped revision page. Pause on the word snapshot, because it flips how newcomers picture version control: git does not store "the changes". It stores whole snapshots, and when you ask "what changed?", it compares two snapshots on the spot. You will prove this with runnable code in a few minutes.

One more piece completes the map. Repos live in two kinds of places at once: one shared copy on a hosting service — GitLab, GitHub, and friends — called the remote (the master file in the senior lawyer's office), and any number of full personal copies on people's machines. Yours will be one of them. They stay in sync only through explicit commands — which is what makes git feel safe once you see it: nothing moves unless someone asks.

Note

This chapter is deliberately a visitor's pass, not a driving licence. Today you learn the moves that let you read any project: clone, pull, log, diff, plus one compass command called status. Making your own commits, pushing, branching, merge requests, and the whole GitLab workflow get their own full act — Act XII. You do not need any of that to explore, and exploring is the skill this act runs on.

Six words that unlock every repository

Flip each card until the back of it feels obvious. These six terms carry the whole chapter — and most conversations you will ever have near a codebase.

Move one: clone — take your own full copy

Cloning means downloading a complete copy of a repository — every file and every commit ever made — into a new folder on your machine. One command, run once per project:

terminal
$ git clone https://gitlab.com/acme-tools/invoice-agent.git
Cloning into 'invoice-agent'...
remote: Enumerating objects: 1284, done.
Receiving objects: 100% (1284/1284), 2.1 MiB, done.

$ cd invoice-agent
$ ls
README.md   pyproject.toml   src/   tests/

Notice what you did not need: permission, an account on the project, or any understanding of its internals. If a repo is public — and an enormous amount of the AI tooling you operate daily is — cloning takes you from "I saw it on the internet" to "it is on my laptop and I can open every file". The frameworks of later acts, LangChain and LangGraph included, are all just repos you can clone and read.

Two things about a clone surprise people pleasantly. It is complete: you hold the full history offline, so log and diff work on a train with no network. And it is yours: a copy, not a borrowed original. Open files, run the code, scribble experiments — the remote does not feel a thing. The worst case in this entire chapter is deleting your clone folder and cloning again. That is the whole blast radius.

Tip

Where do you find the URL? Every GitLab or GitHub project page has a button labelled Code or Clone that shows a copyable HTTPS address ending in .git. Copy, paste after git clone, done. This is also the same address pip can install from directly — the package world of the previous chapter and the repo world of this one are two views of the same folder.

The compass: status — where am I, is my copy clean?

Before and between the other moves, one command tells you where you stand. git status answers three questions: which line of history am I on (you will usually see main — think "the master file", formally a branch, a word Act XII unpacks properly), have I edited anything locally, and has my copy drifted behind the remote?

terminal
$ git status
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean

"Working tree clean" means your copy exactly matches the last snapshot — no local edits lying around. That sentence is your green light. Status never changes anything; it only reports. Run it habitually — before a pull, after an experiment, in any unfamiliar folder — and you will never be surprised.

Move two: pull — collect the new pages

Your clone was perfect the moment you made it, and started going stale the moment after. Teammates (or an open-source community, or an AI coding agent) keep adding commits to the remote. Pulling downloads those new commits and updates your files to match — the associate collecting the overnight pages. Watch the whole rhythm:

1. Right after cloning, all copies match

Remote on GitLabcommits 1 to 47
→
Your clonecommits 1 to 47
Teammate's clonecommits 1 to 47

Three complete, independent copies of the same history. Nothing syncs on its own — each copy changes only when its owner runs a command.

2. A teammate publishes commit 48

Remote on GitLabcommits 1 to 48
←
Your clonestill 1 to 47
Teammate's clonecommits 1 to 48

The teammate pushes their new commit to the remote (pushing is their move, from Act XII). Your clone is now one commit behind — and completely unaware of it.

3. You run git pull

Remote on GitLabcommits 1 to 48
→
Your clonedownloading commit 48

Git asks the remote "anything new since commit 47?", downloads commit 48, and updates the files in your folder to the newest snapshot.

4. Up to date — until next time

Remote on GitLabcommits 1 to 48
Your clonecommits 1 to 48

Your copy matches the reference again. Clone once, pull often: that is the entire maintenance routine of a reader.

terminal
$ git pull
Updating 9b2d1e4..4f2a9c1
Fast-forward
 config.yaml | 3 ++-
 1 file changed, 2 insertions(+), 1 deletion(-)

Read that output like a receipt: git moved you from one snapshot fingerprint to a newer one, one file changed, two lines added, one removed. Even the summary speaks in pluses and minuses — the redline language you are about to learn properly.

Careful

Of the moves in this chapter, git pull is the only one that rewrites files in your folder — which is exactly its job. It is entirely safe when git status says "working tree clean". If you have scribbled local edits, git will refuse to trample them and print a warning instead of guessing. If that ever happens, nothing is broken: your edits are intact, git simply paused. Handling that gracefully is an Act XII skill; today, the rule of thumb is status first, then pull.

Move three: log — read the story backwards

The log is the register of commits, newest first. It is the most underrated command in this chapter, because it answers the question every operator actually has when something changes behaviour overnight: what happened here recently?

terminal
$ git log --oneline -5
4f2a9c1 (HEAD -> main) Raise the LLM timeout and add retries
9b2d1e4 Fix crash when the invoice has no line items
c37b8a0 Set a calmer default temperature
a1f9d33 Add the agent config file
8e04b71 Initial project skeleton

Each line is one commit: a short version of its unique fingerprint (the hash — a 40-character ID computed from the snapshot's contents, usually shown as its first seven characters), then the author's one-line message. The -5 limits it to the latest five; plain git log gives the full view with authors and dates, one screenful at a time (press q to leave it — the number-one "am I stuck?" moment for newcomers, and now you are immune). HEAD -> main marks the snapshot your folder currently shows.

A well-kept log reads like a project diary, and teams that write good commit messages are handing you free documentation. When you wonder why a config value is what it is, the log often answers before any human does: someone changed it, on a date, and wrote down why. That history-first curiosity is the most professional-looking move a code reader can make.

Move four: reading a diff — red and green

Now the payoff. A diff is the redline between two snapshots: line by line, what was removed and what was added. Removed lines start with - (shown red in every tool); added lines start with + (shown green). A changed line appears as a pair — old version removed, new version added — because git thinks in whole lines, not edits within them.

You will meet diffs in three places: git diff (your local edits, if any), git show <hash> (what one commit changed), and — most often — in a code-review screen, because a GitLab merge request is essentially a diff with a comment thread attached. The skill is identical everywhere. Here is the commit from the top of our log. Click each line to unfold its meaning:

git show 4f2a9c1 — one commit, printed as a diff
commit 4f2a9c1e8b03d7f2a41c96d5e0b8a7c3d2f1e0a9
The commit's full fingerprint. Computed from the snapshot's contents, so it is globally unique — quote the first seven characters (4f2a9c1) in a chat message and a teammate can look at exactly the same change.
Author: Priya Nair <priya@example.com>
Who made it. Git records this on every commit — accountability is built into the data itself, not bolted on by the hosting site.
Date: Tue Mar 3 09:14:22 2026 +0530
When. Together with the author line, this is why git log can answer "what happened here last Tuesday" years later.
Raise the LLM timeout and add retries
The commit message — the author's one-line "why". Everything below shows WHAT changed; this line is the only place that says what it was FOR.
diff --git a/config.yaml b/config.yaml
The change touches one file: config.yaml. By convention "a/" is the before version and "b/" is the after. A commit touching many files repeats this whole block per file.
--- a/config.yaml
Minus signs label the BEFORE side. Any line prefixed with a single "-" below belongs to this older snapshot.
+++ b/config.yaml
Plus signs label the AFTER side. Any "+" line below is new. Red/minus = old, green/plus = new — the entire colour code in one sentence.
@@ -4,7 +4,8 @@ agent:
A hunk header — it locates this excerpt inside the file. Decoded: in the old file this excerpt starts at line 4 and spans 7 lines; in the new file it starts at line 4 and spans 8 (one line longer, because one was added). You never need to compute these; just read them as "we are near line 4".
model: claude-sonnet
No + or -, just a space: a context line. Unchanged, shown only so you can see where you are. Diffs include a few of these around every change for orientation.
- timeout_seconds: 30
Red: this line existed before and is gone after. Paired with the + line below it, this is git's way of saying the value CHANGED from 30...
+ timeout_seconds: 90
...to 90. A changed line always shows as this remove-and-add pair. Reading the pair together — "timeout went from 30 to 90" — is the core skill of diff literacy.
+ retries: 2
Green with no red partner: a genuinely new line. The agent's config gained a retries setting that never existed before.
temperature: 0.2
Another context line closing the excerpt. Total story of this commit, read fluently: "timeout raised 30 to 90, retries: 2 added, nothing else touched." You just reviewed a change.

That is all a diff is. Once the minus/plus pairing clicks, you can skim a colleague's change — or an AI coding agent's proposed change, which arrives in exactly this format — and say what it does in one sentence. That sentence is the beginning of code review.

Prove it in Python: git stores snapshots, computes diffs

Earlier I claimed git does not store the red and green lines anywhere — it stores whole snapshots and computes the diff the moment you ask. Python's standard library ships the same comparison machinery in a module called difflib, so you can watch the trick happen: a toy three-commit repository built from plain data structures, a hand-rolled log, and a diff computed live between the last two snapshots. Press Run, then try the experiments in the comments.

Compare the output with the walkthrough above: same ---/+++ labels, same @@ hunk header, same minus/plus lines. That is not imitation — unified diff is a standard format, and git, difflib, and GitLab's review screen all speak it. You have now produced a diff by hand, which means no diff will ever again look like machine noise.

What you do not need yet

Everything today flowed one way: from the remote, to you, to your eyes. That is deliberate. The writing half of git — making commits, pushing, branching so parallel work does not collide, merge requests, the etiquette of shared history — is a genuinely bigger topic, and it gets the space it deserves in Act XII, the full git and GitLab story, where you run the whole contributor loop yourself with the GitLab screens walked through element by element.

Until then, your safety card is short: clone copies, status and log and diff only read, and pull updates a clean copy safely. Nothing in this chapter can damage a shared project — you hold a photocopy, and the master file is in another office entirely. So explore greedily. Clone the framework you operate. Read its log over coffee. Open a commit that mentions something you recognise and read the redline. The codebases you depend on stop being weather and become geography: fixed, mapped, walkable.

In the wild

These four moves are ambient in every engineering conversation. "Clone the repo and poke around" is the standard onboarding sentence — now a ten-second task. "Did you pull?" is the first question when your copy behaves differently from a teammate's. "Check the log" is how people answer "when did this break?", and git status is the first command a senior engineer types in any unfamiliar folder — a glance at the dashboard before driving. Above all: every GitLab merge request you will ever be asked to look at, including the ones AI coding agents open, is presented as a diff — red and green, exactly as you read it here. "Have a look at the diff" is now an invitation, not a test.

Check yourself

1. You run git clone on a public framework repo. What lands on your machine?

A clone is the associate's full photocopy — every file and every commit ever made, stored locally in the hidden .git database. That is why log and diff keep working with no network at all.

2. In a diff, what does this pair of lines tell you?
- timeout_seconds: 30
+ timeout_seconds: 90

Git thinks in whole lines, so a changed line shows as a remove-and-add pair: red minus is the before, green plus is the after. Reading the pair as one edit — "30 became 90" — is the heart of diff literacy.

3. Your clone is a week old and a teammate says the config changed yesterday. Which command brings your copy up to date?

Pull downloads the new commits from the remote and updates your files to the latest snapshot. Status only reports where you stand, and log only lists the history you already have — neither fetches anything new.

4. Which of this chapter's commands could damage the shared project on the remote?

Clone, status, log, and diff are read-only; pull changes only your local folder, and refuses rather than trample local edits. Commands that write to the shared remote — commit, push, merge — arrive with full training in Act XII.

5. A config value looks strange and you want to know why it was set that way. What is the history-first move?

The log tells you who changed it, when, and — via the commit message — why; git show on that commit gives you the exact redline. Someone already wrote the answer down at the moment they made the change.

Key takeaways

  • A repository is a folder that remembers everything: current files plus the full chain of commits — stamped snapshots with an author, a date, and a "why" message — kept in a hidden .git database.
  • Clone once, pull often: git clone gives you a complete independent copy of any repo, and git pull collects whatever new commits landed on the remote since.
  • git status is your compass — run it before pulling and in any unfamiliar folder; "working tree clean" is the green light.
  • git log --oneline reads the project's diary newest-first, and is usually the fastest answer to "what changed recently and why".
  • A diff is a computed redline between two snapshots: minus/red lines are the before, plus/green lines are the after, and a changed line always appears as a remove-and-add pair.
  • Nothing in this chapter can harm a shared project — these are a reader's moves. Writing history (commit, push, branch, merge requests, the GitLab workflow) is Act XII's full story.
