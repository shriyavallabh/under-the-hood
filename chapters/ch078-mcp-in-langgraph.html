MCP inside LangChain and LangGraph

Three lines of adapter code turn any MCP server into ordinary agent tools — and one strategic idea turns your toolbox into an asset that no framework migration can ever take away from you.

~25 minChapter 79 of 123
The story

Meera runs an equipment-hire house for film crews. For years, her warehouse was a museum of incompatibility: every camera brand had its own battery mount, so a crew that switched from one brand to another had to re-buy an entire wall of batteries, chargers, and rigs. Her capital was locked to whichever brand she had bet on.

Then the industry quietly standardised on one battery mount. Today her single shelf of batteries powers every camera that comes through the door. When a crew arrives with a body she has never seen, they clip on a thin adapter plate — thirty seconds, no rewiring — and her whole inventory just works. Crews now switch camera brands the way you change shoes, because the expensive part, the powered toolbox, no longer cares which body it feeds.

Your MCP servers are that shelf of batteries. LangChain, LangGraph, Claude Code, a hand-rolled loop — those are camera bodies, and they will come and go. This chapter is about the thin adapter plate: langchain-mcp-adapters, the package that clips your standard toolbox onto the LangChain mount. And about why, once you own the batteries, you stop fearing the bodies.

The problem the adapter solves

In the previous chapter you built an MCP server with FastMCP: a handful of Python functions decorated with @mcp.tool(), served over stdio or streamable HTTP, callable by anything that speaks the protocol. You proved it works from an MCP client. Lovely. But your working agents are not generic MCP clients — they are LangChain agents built with create_agent and LangGraph graphs built with StateGraph. Those expect tools the LangChain way: the @tool-decorated Python functions you met in the LangChain tools chapter, with a name, a description, and an argument schema.

So there is a gap. On one side, a server speaking MCP's wire protocol. On the other, a harness expecting a list of tool objects. An adapter — a small piece of code that translates between two interfaces that mean the same thing but speak differently — closes that gap. The official one for this ecosystem is the langchain-mcp-adapters package, and its whole public surface fits in a sentence: give MultiServerMCPClient a dictionary of server configs, call get_tools(), and pass the result anywhere LangChain tools go.

Why does this matter beyond convenience? Count the integrations. If every harness needs custom glue for every toolbox, m harnesses and n toolboxes cost you m times n integrations — every combination hand-wired. With one protocol in the middle, each harness implements MCP once and each toolbox implements MCP once: m plus n. That arithmetic is the entire business case for MCP, and the adapter is where LangChain pays its share of it.

Three lines between your server and your agent

Here is the whole bridge, in real code. One server runs locally over stdio (an invoice server of your own, built the same way as last chapter's finance desk); another is a shared team server reached over streamable HTTP. Read it once, then click through the annotated version below it.

agent_with_mcp.py
from langchain_mcp_adapters.client import MultiServerMCPClient
from langchain.agents import create_agent

client = MultiServerMCPClient(
    {
        "invoices": {
            "command": "python",
            "args": ["invoice_server.py"],
            "transport": "stdio",
        },
        "docs": {
            "url": "http://localhost:8000/mcp/",
            "transport": "streamable_http",
        },
    }
)

tools = await client.get_tools()

agent = create_agent(
    "anthropic:claude-sonnet-4-5",
    tools,
    system_prompt="You are a billing assistant.",
)

result = await agent.ainvoke(
    {"messages": [{"role": "user", "content": "Total for invoice 42?"}]}
)

Note

Two housekeeping details. First, await only works inside an async function, so in a script you wrap this in async def main() and run it with asyncio.run(main()). Second, older tutorials show MultiServerMCPClient used as an async with context manager — current versions let you simply create it and call get_tools(), as above. The package can also load MCP prompts and resources, but tools are the headline act.

And LangGraph? By the time get_tools() returns, the adapter's work is finished and you are holding ordinary LangChain tools. So everything you already know applies unchanged: bind them to a model with bind_tools(tools) inside a hand-built StateGraph, route on tool calls with conditional edges, or hand the same list to create_deep_agent. One toolbox, and every harness in this course — plus the ones outside it — can hold it.

What actually happens, step by step

The code above hides a small choreography. Watch it once in slow motion, because when an MCP integration misbehaves in production, the failure is almost always in one of these five steps.

1. The client reads its map

→
invoice_server.pynot running yet

Nothing has connected yet. The client just knows two nicknames and how each server likes to be reached.

2. Connect and ask: what do you offer?

get_tools() starts the stdio server as a child process and opens an HTTP session to the remote one, then asks each for its tool list — names, descriptions, argument schemas.

3. Each schema becomes a LangChain tool

→
Adapterwraps it
→

This is the adapter plate clicking on. Each wrapped tool remembers which server owns it, but shows the harness a perfectly ordinary face.

4. The agent loop calls a tool

→

→

Mid-conversation, the model requests a tool exactly as in your hand-built loop. The wrapper forwards the call across the transport to the owning server, which executes the real Python.

5. The result rejoins the loop

The server's answer comes back over the wire, lands in the transcript as a tool result, and the loop continues. Think, act, observe — the heartbeat from chapter 46, with a longer arm.

The strategic argument: your toolbox outlives your harness

Here is the part worth repeating in a leadership meeting. Agent harnesses churn. In this course alone you have watched LangChain deprecate one agent API for another, and the wider landscape (chapter 91 maps it) saw two major Microsoft frameworks merge into a third. Betting your company's integrations on any single harness means rewriting them every time the framework world lurches.

But look at what your tools actually contain: access to the invoice database, the CRM query, the deployment script, the document store. That is business logic — the durable, expensive, audited part. The harness is just the loop around it, and you built one yourself in fifty lines. MCP's quiet trick is to move the durable part out of the framework and into servers that you own, speak a public protocol, and answer to any client. The harness stops being a marriage and becomes a rental.

When you migrate harnesses…What happens to it
MCP servers (the toolbox)Unchanged. Not one line.
Tool schemas, auth, audit logging in the serversUnchanged.
Adapter wiring (get_tools() call)Swapped for the new harness's MCP client — a few lines.
Agent loop, prompts, graph shapeRewritten — but this was always the cheap part.

This also reorganises teams. A platform group can own the servers — one hardened invoice server, one CRM server, centrally deployed over streamable HTTP — while every product team picks whatever harness suits them. The same battery shelf, many camera bodies. When someone asks "should we build our tools as LangChain tools or MCP servers?", the honest answer is: build anything shared or long-lived as an MCP server, and keep @tool functions for glue that genuinely belongs to one agent.

The 2026 efficiency turn: code execution with MCP

Success created MCP's next problem. Once connecting servers became trivial, agents started carrying dozens of them — and suddenly hundreds of tool schemas were being stuffed into the model's context before it read the user's first word. Worse, remember from the tokens-and-cost chapter that every tool result flows through the context too: ask a spreadsheet tool for a 2,000-row sheet so you can find five rows, and all 2,000 rows ride through the model — billed, slow, and distracting. You met the same pain from another angle in the token bill.

The pattern the field converged on — Anthropic's engineering team wrote it up in late 2025, and 2026 made it mainstream — is code execution with MCP: instead of loading every tool schema into context and making the model request calls one at a time, present each server as a small code API — a folder of thin wrapper functions — and let the model write a short script that imports and calls them inside a sandbox. Three things improve at once:

  • Progressive disclosure. The model reads only the wrapper files it actually needs, instead of every schema from every server up front.
  • Results stay out of context. The 2,000 rows live inside the sandbox as a variable; only the five filtered rows return to the model.
  • Control flow moves into code. Loops, retries, and if-this-then-that happen in the script — one round trip instead of a chain of model turns.

Feel the difference yourself. The runnable below simulates both patterns against the same 2,000-row "sales sheet" server and counts how many tokens pass through the model's context in each. The token counter is a crude characters-divided-by-four estimate — the standard back-of-envelope from chapter 42 — but the shape of the result is exactly what production teams see.

Careful

Code execution buys efficiency by running model-written code, and that is never free. The script must run in a real sandbox — isolated, resource-limited, with no credentials lying around — because you are now executing whatever the model chose to write. The next chapter, on MCP security and governance, treats this properly. For now, hold one rule: never adopt the code-execution pattern on infrastructure you would not let an intern run arbitrary scripts on.

In the wild

You will spot this chapter everywhere once you know its shapes. In a Python repo, langchain-mcp-adapters in requirements.txt and a MultiServerMCPClient({...}) block — often in a file named tools.py or mcp_client.py — mean the team buys the decoupling argument. Config dictionaries with "transport": "stdio" or "streamable_http" are the same shape you see in a .mcp.json for Claude Code: one toolbox description, many harnesses reading it. In merge requests, "moved the retriever into an MCP server" or "pointed the agent at the shared docs server" is the platform-team pattern from this chapter. And when an engineering blog or an internal design doc says "we let the agent write scripts against generated tool wrappers", that is code execution with MCP — usually justified with a token-bill chart very like the one above.

Check yourself

1. What does await client.get_tools() actually hand back to you?

The adapter wraps every advertised MCP tool as a normal LangChain tool. The harness — and the model — cannot tell it from a hand-written @tool function; only the wrapper knows a server sits at the other end.

2. Your team migrates its agents from LangGraph to a different harness next year. What happens to the MCP servers?

That is the strategic decoupling argument: the durable business logic lives in servers speaking a public protocol, so a harness migration touches only the thin adapter layer, not the toolbox.

3. In the code-execution-with-MCP pattern, why does the token bill drop so sharply?

Classic tool calling routes every intermediate result — all 2,000 rows — through the model's context. When the model writes a script that runs next to the data, the bulky middle stays in the sandbox and context carries only the script and the answer.

4. A shared invoice server must be reachable by every agent in the company, running on a central machine. Which transport does its config need?

stdio launches a subprocess on your machine — fine for local tools, useless for sharing. A centrally deployed server is reached over the network, which is what streamable HTTP is for.

5. Where can the tools returned by the adapter be used?

Once wrapped, they are ordinary tools. The same list feeds a prebuilt agent, a custom StateGraph via bind_tools, or a deep agent — which is the one-toolbox-many-harnesses promise made good inside a single ecosystem.

Key takeaways

  • langchain-mcp-adapters is the bridge: MultiServerMCPClient(configs), then await client.get_tools(), then pass the list to create_agent or any graph — three moves, and MCP servers become ordinary agent tools.
  • One client speaks to many servers at once: stdio configs (command + args) launch local subprocesses; streamable HTTP configs (url) reach shared servers over the network.
  • After get_tools(), the harness cannot distinguish MCP tools from hand-written ones — the adapter routes each call back to its owning server invisibly.
  • The strategic win is decoupling: business logic lives in harness-agnostic servers you own, so frameworks become swappable front-ends instead of marriages. m plus n integrations, not m times n.
  • Code execution with MCP is the 2026 efficiency direction: present servers as importable code APIs, let the model write a script in a sandbox, and keep bulky intermediate results out of context — published examples cut token flow by roughly 98 percent.
  • Model-written code demands a real sandbox; the security half of that story is exactly where the next chapter picks up.
