Environments, pip, and pyproject.toml

Why the same code runs perfectly on your laptop and dies on your teammate's — and the three small artefacts (a venv, a pyproject.toml, a lock file) that make a Python project reproducible on any machine on Earth.

The story

A watch-repair shop has two benches. On bench one sits a 1962 hand-wound classic that needs gear number 7 in the old 2.1 mm cut. On bench two sits a modern diver's watch that needs gear number 7 in the new 2.4 mm cut. The old repairman keeps one big parts drawer for the whole shop. One afternoon he "upgrades" the drawer — throws out the old 2.1 mm gears, stocks the new ones — and the diver's watch is fixed, beautifully. Next morning the 1962 classic, which had kept perfect time for weeks, comes back dead. Nobody touched it. The drawer changed underneath it.

His apprentice runs things differently. Every job gets its own tray: the watch, its exact parts, nothing else. Taped to each tray is a printed list — every part, every size, down to the batch number — so any bench, in any branch of the shop, can rebuild the identical tray from the warehouse.

The tray is a virtual environment. The printed list is a lock file. The warehouse is PyPI. And "it works on my bench" is the disease this whole chapter cures.

One Python, many projects — the collision

Recall from the imports chapter: when your code says import httpx, Python walks a list of folders and, for anything you installed rather than wrote, ends up in a folder called site-packages — the interpreter's parts drawer. Here is the fact that causes decades of pain: one interpreter has one site-packages, and it can hold only one version of any given package. There is a single folder named httpx in there. It is version 0.27 or it is version 0.28. It cannot be both.

Now put two projects on one machine, sharing that one Python. Project A was built against an older langchain; project B needs the newest. The day you upgrade for B, you have silently swapped the gears under A. Nothing announces it — A breaks days later, in a project you never edited. That is the old repairman's drawer, and it is failure mode number one.

There is a sneakier one. Your machine's shared drawer holds fifty packages accumulated over months of experiments. Your new project uses three but you only wrote down two — the third is just there, so everything runs for you. On your teammate's clean machine, the first import fails. The code was never wrong; the environment was different. Hold that thought — it is the anatomy of "works on my machine", dissected fully in a minute.

Note

Installing packages into the operating system's own Python is so risky (some OS tools are written in Python and depend on that drawer) that modern systems flat-out refuse, with an externally-managed-environment error. The tooling itself is pushing you toward the fix you are about to learn.

A virtual environment is a private toolbox

A virtual environment (a "venv") is a folder — conventionally named .venv, sitting inside your project — that contains a private link to a Python interpreter plus its own empty site-packages. Its own drawer. Anything you install while "inside" the venv lands in that drawer and nowhere else; anything installed elsewhere on the machine is invisible to it. One project, one tray.

Creating one is a single command, and the whole daily rhythm is four lines:

terminal — the classic venv workflow
cd invoice-agent
python3 -m venv .venv            # create the tray (a one-time step per project)
source .venv/bin/activate        # step up to this bench (Windows: .venv\Scripts\activate)
pip install httpx                # parts go into THIS tray only
deactivate                       # step away when done

The word "activate" sounds mystical; it is not. Remember environment variables from the last chapter? One of them, PATH, is the ordered list of folders your shell searches when you type a command. Activation simply prepends .venv/bin to PATH, so typing python or pip finds the venv's copies first. That is the entire trick — one environment variable, edited. deactivate puts it back.

pip, PyPI, and what "install" really means

PyPI — the Python Package Index — is the public warehouse: hundreds of thousands of packages, uploaded by their authors, free to download. pip is the fetcher that goes shopping there. But pip install langchain does four jobs, not one, and seeing them is what makes lock files click later.

First, pip asks PyPI what versions of langchain exist. Second, it reads the package's own requirements — packages depend on packages: langchain needs pydantic (from the type hints chapter), which needs typing-extensions, and so on down the tree. A dependency of a dependency is a transitive dependency — you never asked for it by name, but you cannot run without it. Third, pip plays matchmaker: it must pick one version of every package involved that satisfies everybody at once. That job, dependency resolution, is genuinely hard — one package wants http-lib below 2.0, another wants 2.0 or newer, and the resolver must find a combination that keeps both happy or say honestly that none exists. Fourth, it downloads each winner as a wheel — a pre-built zip archive — and unpacks it into the active environment's site-packages.

This is why asking for three packages routinely installs forty. Watch a toy resolver work, right now: the fake warehouse below has three packages, you ask for two, the newest versions clash, and it must back off to the combination that works. Press Run.

The packing lists: requirements.txt vs pyproject.toml

An environment is a folder on your disk. Your teammate does not want your folder — they want the list taped to the tray, so their machine can rebuild it. Python has two generations of that list, and you will meet both constantly.

The elder is requirements.txt — a plain text file, one requirement per line, fed to pip install -r requirements.txt. It has been the convention for twenty years and still works:

requirements.txt
langchain>=1.0,<2.0
pydantic>=2.7
httpx>=0.27

Its weakness: it is just a list, used for two contradictory jobs. Some teams hand-write it loosely, like the example above — a statement of intent. Others generate it with pip freeze > requirements.txt, which dumps every installed package at its exact version — direct and transitive deps jumbled together, no hint of which three you actually chose. Same filename, opposite meanings, and no other information: not the project's name, not which Python it needs, nothing.

The modern answer is pyproject.toml — one standardised file at the project root, the project's identity card. It is written in TOML, a third config cousin of YAML and JSON: key = value lines grouped under [section] headers. It is usually the first file worth reading in any Python repository, so let us read one properly. Click the annotated lines:

pyproject.toml — a project's identity card
[project]
A TOML section header, like a heading in a form. Everything under it describes the project itself. This layout is a Python-wide standard, so every tool — pip, uv, poetry — reads the same fields.
name = "invoice-agent"
The project's official name — what it would be called if published to PyPI, and how other code would pip install it.
version = "0.4.2"
The project's own version number. When a changelog says "released 0.5.0", someone edited this line (or a tool did).
requires-python = ">=3.11"
Which interpreters this project supports. Installers refuse to proceed on an older Python instead of failing mysteriously at runtime — one whole class of "works on my machine" deleted by a single line.
dependencies = [
The heart of the file: the packages this project directly needs. A TOML list, exactly like a Python list of strings.
"langchain>=1.0,<2.0",
A name plus a version constraint: "any 1.x, but not 2.0". Ranges here are deliberate — this file states what is acceptable, not what is exact. Exactness is the lock file's job, coming next.
"pydantic>=2.7",
"httpx>=0.27",
]
[project.optional-dependencies]
Extras that only some users need. Installing the project can opt in to a named group.
dev = ["pytest>=8.0", "ruff"]
The classic example: developers need the test runner and the linter; production servers do not. Keeping them out of the main list keeps deployments lean.
[build-system]
Plumbing: which tool turns this folder into an installable wheel. You will rarely edit it — recognising it is enough.
requires = ["hatchling"]
build-backend = "hatchling.build"
[tool.ruff]
Any tool may claim a [tool.something] section for its own settings — linters, formatters, test runners all park their config here. One file instead of six dotfiles littering the repo root.

Lock files: the receipt, not the shopping list

Look again at that dependencies list. It says httpx>=0.27 — a range. Installing today might get you 0.27.2; your teammate in three months gets 0.29.0, because the range allows it and newer is preferred. Both installs "obeyed" the file, yet your machines now differ. Ranges are the right way to state intent and exactly the wrong way to reproduce a machine.

Enter the lock file: a machine-generated record of one fully-solved installation — every package, direct and transitive, pinned to an exact version, usually with a checksum of the exact file downloaded. pyproject.toml is the shopping list ("some good coffee, around 500 g"); the lock file is the till receipt ("this estate, this batch, paid at 16:04"). It is the apprentice's printed list from the story — and it is simply the resolver's winning answer, written down and committed to git. You watched that answer get computed in the toy above.

uv.lock (excerpt, trimmed)
[[package]]
name = "httpx"
version = "0.27.2"
dependencies = [
    { name = "certifi" },
    { name = "httpcore" },
]

[[package]]
name = "certifi"
version = "2025.1.31"

Different tools name it differently — uv.lock, poetry.lock, a fully-pinned requirements.txt from pip-tools — and a common standard format (pylock.toml) has recently been agreed. The names matter less than the discipline: humans edit the intent file; tools generate the lock file; both are committed; machines install from the lock. When every laptop and CI server installs from the same lock, they end up identical — and an entire species of bug goes extinct. The checksums add a bonus: if anyone tampers with a package on the warehouse shelf, the receipt no longer matches and the install refuses.

The anatomy of "works on my machine"

You now have every organ on the dissection table. When code runs for you and fails for someone else, it is almost always one of five suspects, each with a cure you have just learnt:

SuspectHow it bitesThe cure
No lock fileYou installed in March, they installed in June; the same ranges resolved to different versions and behaviour drifted.Commit a lock file; everyone installs from it, never from raw ranges.
Different Python versionYour 3.12 accepts syntax or stdlib calls their 3.10 has never heard of.requires-python in pyproject.toml, plus a pinned interpreter (uv can even install it, below).
Undeclared dependencyA package you use sits in your global drawer by historical accident, so you never wrote it down. Their clean machine fails on the first import.Always work inside a fresh venv — then the missing declaration fails on your machine first, where it is cheap.
Missing config or env varsYour shell exports an API key or a path theirs does not — the previous chapter's whole territory.A committed .env.example, and config that validates loudly at startup.
Different operating systemPath separators, case-sensitive filenames, or a compiled wheel that exists for your platform and not theirs.Run CI on the same OS you deploy to, so the mismatch surfaces in a robot's logs instead of a customer's.

Notice what the first three cures have in common: none of them is "be more careful". They are all artefacts — files committed to the repository that make correctness the default. That is the engineering layer's recurring move, and you will see it again with tests and CI.

uv: the whole workshop in one fast tool

Everything so far used the classic toolkit: venv for trays, pip for fetching, pip freeze or pip-tools for receipts. Three tools, many chances to forget a step. uv is the modern consolidation: one tool, from the makers of the ruff linter, that creates environments, resolves, installs, locks, and even installs Python interpreters themselves. Written in Rust (a fast systems language), it is typically orders of magnitude faster than pip — installs that took a coffee break take a blink — and since the mid-2020s it has become the default for new projects. Its daily rhythm is delightfully short:

terminal — the uv workflow
uv init invoice-agent        # scaffold a project with a pyproject.toml
cd invoice-agent
uv add httpx                 # add to pyproject, resolve, update uv.lock, install
uv add --dev pytest          # a dev-only dependency
uv run python agent.py       # run inside the project env - no activate dance
uv sync                      # teammate or CI: rebuild the exact env from uv.lock

Read uv add httpx again: it performs this chapter's whole discipline in one stroke — writes the dependency into pyproject.toml (intent), re-runs the resolver (the matchmaking), records the full solution in uv.lock (the receipt), and installs into .venv (the tray). Nothing to forget. uv sync is the other half: point it at a fresh clone and it rebuilds the identical environment from the lock file. Teams run it in CI with a flag that refuses to proceed if the lock is stale (check the uv docs for the exact flag), turning "someone forgot to re-lock" from a mystery into a loud, immediate failure.

Careful

Whichever tool you use, two git rules are non-negotiable: the .venv folder is never committed (it is huge, machine-specific, and rebuildable), and the lock file is always committed (it is the reproducibility itself). A repo with .venv in its .gitignore and uv.lock in its history is a repo run by adults.

In the wild

Open any serious Python repository and this chapter is the first thing you will see: a pyproject.toml at the root (read its dependencies list to learn what the project is built on before reading any code), a uv.lock or poetry.lock or pinned requirements.txt beside it, and .venv in .gitignore. CI logs open with uv sync before any test runs. Merge requests titled "bump langchain 1.1 to 1.2" are edits to exactly these files — the diff worth reviewing is often the lock file, where every transitive package that moved along for the ride is visible. And when an agent harness mysteriously changes behaviour after a redeploy with "no code changes", an unlocked dependency that quietly resolved newer is the first suspect a senior engineer checks.

Check yourself

1. Two projects share one machine's global Python. You upgrade a library for project B, and project A — untouched for weeks — breaks days later. What is the root cause?

A shared interpreter has a single parts drawer. Upgrading for B swapped the gears underneath A — the old repairman's mistake. Per-project venvs make this collision impossible.

2. What does source .venv/bin/activate actually do?

Activation is just an environment-variable edit: .venv/bin is prepended to PATH. Nothing is installed or copied — which is also why deactivating is instant and harmless.

3. Which statement about pyproject.toml dependencies versus a lock file is true?

The shopping list versus the receipt. Intent lives in pyproject as ranges a human maintains; the lock file is the resolver's full, exact, machine-generated answer — and installing from it is what makes two machines identical.

4. Your pyproject.toml declares 3 dependencies, but uv.lock lists 38 packages. Why?

Your 3 direct picks each brought friends, and the friends brought friends. Reproducing an environment requires pinning all of them, which is exactly why the lock file — not the 3-line intent list — is the reproducibility artefact.

5. A teammate clones your repo, installs your declared dependencies, and hits ModuleNotFoundError on a package that runs fine for you. Most likely explanation?

The undeclared-dependency suspect. Had you developed inside a clean venv, the missing declaration would have failed on your machine first — the cheapest possible place to catch it.

Key takeaways

  • One interpreter has one site-packages, and it holds one version of each package — so projects sharing a Python silently break each other. A venv gives every project its own disposable drawer: python3 -m venv .venv.
  • Activation is not magic — it prepends .venv/bin to the PATH environment variable so python and pip mean the venv's copies. Delete the folder to delete the environment.
  • pip install is really four jobs: query PyPI, gather everyone's requirements (transitive dependencies included), resolve one compatible version of every package, and unpack the wheels into site-packages.
  • pyproject.toml is the project's identity card: name, supported Pythons, and dependencies as ranges stating intent. requirements.txt is the older flat-list convention you will still meet everywhere.
  • A lock file is the resolver's exact answer written down — every package pinned, checksums included, committed to git. Installing from the lock is what makes your laptop, your teammate's, and CI byte-for-byte identical.
  • "Works on my machine" is not a mystery; it is five nameable suspects — unlocked versions, mismatched Pythons, undeclared dependencies, missing env vars, OS differences — each cured by an artefact, not by carefulness.
  • uv folds the whole workshop into one fast tool: uv add keeps pyproject, lock, and environment in sync in a single command, and uv sync rebuilds the exact environment anywhere.
