Durability: checkpoints, resume, replay

Your agent's process will die mid-task — a deploy, an out-of-memory kill, a pulled cable. Durability makes that a shrug: write every step down, resume where you stopped, and re-watch any run months later.

~25 minChapter 86 of 123
The story

A monsoon evening in 1987, at a small branch of a state bank. The power fails every hour, and the senior clerk is working through a stack of salary transfers — several steps each: verify the form, check the balance, debit this account, credit that one, stamp, sign. The lights die halfway through transfer forty-one. When they flicker back, a different clerk sits at the desk. And nothing is lost — because the branch runs on one iron rule: ink the register before your hand moves to the next step.

The new clerk reads the register, sees exactly where transfer forty-one stopped — debit done, credit not yet — and continues from that line. She does not debit twice: the register says it already happened. And when an inspector asks, six months later, why account 4402 was debited that evening, nobody has to remember anything. They open the register and re-walk the whole night, step by step.

That register — write each step before the next, resume from the last written line, re-read the whole day whenever anyone asks — is durability. This chapter gives your agent that register.

First, remember the checkpointer

Back in the checkpointing chapter you met LangGraph's answer to amnesia: pass a checkpointer to .compile(), give each conversation a thread_id, and after every node the graph's state is snapshotted into a database — InMemorySaver for development, Postgres/Redis/SQLite backends (the langgraph-checkpoint-* packages) for production. You even built a toy one yourself. That chapter was one framework's implementation.

This chapter zooms out to the harness level — the "loop around the loop" of this act. Whatever framework sits inside, a harness must answer three questions before the first real task runs:

  • Checkpoint — when the process dies, how much work do we lose?
  • Resume — how does a fresh process continue an interrupted task, safely?
  • Replay — when someone asks "why did the agent do that?", can we show them, step by step?

Notice what is missing: "how do we stop the process dying?" You can't. Durability starts from the honest premise that you never will.

Processes die. Being surprised is the real failure

The deaths are mundane. A deploy terminates the old process mid-run. A Kubernetes pod is evicted. The container is OOM-killed — it ate more memory than its limit allowed. An exception goes uncaught and ends the process (ch9). On a busy platform, several of these happen every day.

Die mid-task without durability and you lose two things — the second far worse:

  • The work. Nine steps of an eleven-step run — twenty expensive LLM calls, minutes of tool time — evaporate. Annoying, but recoverable: run it again.
  • Certainty about the world. Step seven sent a payment. Or did it? The process died around then, and the record of what it had done died with it. Running again might pay twice; not running might pay zero times. Both answers are wrong.

That second loss is why durability is a harness concern, not a nice-to-have. An agent that only reads can be restarted carelessly; one whose tools have real side effects cannot. Watch a durable harness handle it:

Checkpoint: write the register before the next step

A checkpoint is a durable snapshot of everything the run would need to continue — written to storage that outlives the process, after every step, before the next begins. That is more than a step counter: a real checkpoint carries the conversation history (the agent's whole memory), the current plan, every tool result so far, and any tool call issued but not yet answered.

Two design rules do most of the work:

  • Durable means outside the process. A Python variable is not a checkpoint; neither is a container's local disk. Production checkpoints live in Postgres, Redis, or an object store — ch61's lesson about checkpointer backends.
  • Frequency follows cost. Checkpoint after every step: a step is expensive (an LLM call costs seconds and money), a checkpoint write costs milliseconds. The most you can lose is one step — and you know precisely where you stopped.

In LangGraph clothing, this section is one argument and one config key:

durability, the LangGraph way (recap of ch61)
from langgraph.checkpoint.memory import InMemorySaver  # dev only
# production backends: langgraph-checkpoint-postgres / -redis / -sqlite

graph = builder.compile(checkpointer=InMemorySaver())

config = {"configurable": {"thread_id": "invoice-8801"}}
graph.invoke({"task": "pay vendor invoice 8801"}, config)

# process died? a fresh process resumes the same thread
# by invoking again with the same thread_id
graph.invoke(None, config)

The same machinery made human-in-the-loop pauses possible: interrupt() parks the run as a checkpoint and Command(resume=...) continues it — days later, in a different process. To the checkpoint store, a pause and a crash are nearly the same event. One is just politer.

Resume: a fresh process picks up the pen

Resume means a brand-new process loads the latest checkpoint and continues the task forward, with real effects. Done right, it is anticlimactic — read the register, find the next undone step, carry on. The subtlety is the gap you saw in the stepper: the crash window — after a tool's effect fires, before its checkpoint lands. A crash there means the resumed process will retry a step whose effect already happened.

Careful

You cannot close the crash window by reordering. Checkpoint before the effect, and a crash between them leaves the register claiming a payment that never happened. Checkpoint after, and the payment happens without the register knowing. Distributed-systems people have proven you cannot get "exactly once" from ordering alone. The real fix is to make the retry itself harmless — the next section.

The payment that must not fire twice

An operation is idempotent if doing it twice leaves the world exactly as doing it once — so retries are free. Reading a balance is naturally idempotent. But the operations agents get hired for — send the payment, email the customer, create the ticket — are naturally not. Do them twice and the world is wrong twice.

The standard cure is the idempotency key: a unique ID attached to the effect, chosen before the first attempt, and honoured by the system that owns the effect. The payment API keeps a ledger of keys it has executed; a retry with the same key moves no money — the API returns the first attempt's recorded result. Stripe's Idempotency-Key header is the best-known example; serious payment, messaging, and ordering APIs all offer the same contract.

Together they give the guarantee production systems actually run on, sometimes called effectively once: retry whenever in doubt, make every effect idempotent, and the extra attempts collapse into no-ops. Checkpoints make retries rare and precise; idempotency keys make them safe. You need both.

Replay: the flight recorder

Resume answers "how do we finish the task?" Replay answers "show me exactly what happened" — re-walking a recorded run step by step without touching the real world. No LLM is called, no tool fires; the recorded outputs are read back in order, like a flight recorder, or the clerk's register under the inspector's lamp.

Replay has two customers, and both will find you eventually:

  • The debugger. "The agent did something weird on Tuesday" is an anecdote. With replay it becomes reproducible: load run 8801, step to turn 6, see which tool result nudged the model down the wrong branch. Without replay you are re-running a non-deterministic system (ch31: same input, different output) and hoping the ghost reappears.
  • The auditor. Months later, someone official asks why the agent approved a transaction. The acceptable answer is a record: every prompt, model response, tool call and result, in order, tamper-evident. "We believe it followed the system prompt" is not a record.

Replay's price is paid up front: record everything non-deterministic as it happens — model outputs, tool results, timestamps, random draws. Checkpoint stores that keep full history give you this for free; it is much of what observability tooling, three chapters from now, is built on.

Durable-execution engines, in one picture

Everything so far you could hand-build. But the pattern is so universal that a product category exists for it: durable-execution engines — systems that run your code as a journaled, crash-proof workflow. Temporal is the best-known; Restate and Azure Durable Functions play in the same space. Hold their one clever idea:

  • You write the workflow as ordinary-looking code; every step with a side effect is marked as an activity.
  • The engine keeps a journal — an append-only log of every activity's result, written durably as the run proceeds.
  • After a crash, the engine re-runs your workflow code from the top on a fresh worker. Each activity call checks the journal first: already recorded? Return the recorded result instantly, executing nothing. The code fast-forwards to the exact point of death in milliseconds and keeps going.

That trick is replay-as-resume, and it explains these engines' strict rule: workflow code must be deterministic — it is re-executed against the journal and must take the same branches every time. Model calls and tool calls — non-deterministic by nature — always live inside journaled activities.

Squint and the family resemblance shows: LangGraph's checkpointer snapshots state and continues forward; Temporal-style engines journal events and recover by re-running against the record. Different mechanics, identical promise — a death costs at most one step, and never costs you certainty about the world.

Before the bank walks in, pin the vocabulary:

Why a bank will not negotiate on this

Four reasons banks treat replayable runs as non-negotiable — only the first is about technology:

  • The effects are irreversible. A payment that fires twice is someone's money in the wrong place, a customer call, a manual reversal, sometimes a regulatory report. Idempotent effects with keys are the only "retry" a payments engineer will sign off on.
  • Regulators demand reconstructable decisions. Supervisors require firms to explain and evidence automated decisions, and to retain the records for years. When an agent touches lending, payments, or trading, "replay run 8801" is the compliance answer; a system that cannot replay does not go live.
  • Operational-resilience rules assume failure. Critical services must survive infrastructure failure; an in-flight transfer that evaporates with its process is a reportable incident, not bad luck.
  • Disputes are settled by records. When a customer says "I never authorised this", the institution with the complete, ordered journal wins the argument. The one with "the logs rotated" writes the cheque.

This is why an enterprise harness team asks durability questions before capability questions. A clever agent that loses in-flight state is a demo. A modest agent that checkpoints every step, resumes cleanly, retries idempotently, and replays on demand is a system a risk committee can approve. In a bank, only the second one ships — an ordering that is correct everywhere; banks just write it down.

In the wild

You will trip over this chapter everywhere. In LangGraph code: the checkpointer= argument to .compile() and thread_id in the invoke config. In enterprise repositories: Temporal or Durable Functions workflows wrapped around agent calls, and retry policies on every job queue. In API integrations: Stripe's Idempotency-Key header, or a homegrown request_id column with a unique constraint. In your own terminal: Claude Code's --continue and --resume flags are exactly this machinery, reviving a session from its saved transcript. And in merge requests, the giveaway phrases are "make this handler idempotent" and "what happens if this retries?" — you now know which clerk's register each one defends.

Check yourself

1. A run crashes after the transfer tool moved the money but before that step's checkpoint was written. On resume, the harness retries the step. What prevents a double payment?

The checkpoint store honestly believes the step never ran — that is the crash window. Only the system that owns the effect can catch the repeat; the idempotency key is how it recognises one. Trusting the model to notice is hope, not engineering.

2. What is the difference between resume and replay?

Resume is for finishing work — a fresh process picks up from the last checkpoint and keeps acting. Replay is for understanding work — debugging and audit read the record back, and nothing real fires.

3. Why checkpoint after every step instead of once at the end of the run?

A checkpoint written only at the end protects nothing — the crash, by definition, arrives before the end. Per-step writes cost milliseconds against LLM calls costing seconds and money.

4. Which of these tool calls is naturally idempotent — safe to retry with no extra machinery?

Reading twice leaves the world identical to reading once. The email arrives twice and the spreadsheet grows two rows — both need an idempotency key (or a duplicate check) before an agent may retry them.

5. A durable-execution engine recovers from a crash by re-running the workflow code from the top. Why doesn't that double-execute steps that already ran?

That is the journal trick: re-execution fast-forwards through recorded results, reaches the point of death, and continues. It is also why workflow code must be deterministic — it must take the same path every replay.

Key takeaways

  • Processes die routinely — deploys, evictions, OOM kills. Durability assumes the death and engineers the aftermath instead of trying to prevent it.
  • Checkpoint after every step, to a store that outlives the process: the maximum loss is one step, and you know exactly where the run stopped.
  • Resume is a fresh process reading the register and continuing forward — boring by design, sharing its code path with normal startup.
  • Checkpoints cannot close the crash window between an effect and its checkpoint; idempotency keys, honoured by the system that owns the effect, make retries safe. Together: effectively once.
  • Replay is the flight recorder: re-walking a recorded run without touching the world — turning "the agent did something weird" into a reproducible bug and an answerable audit question.
  • Durable-execution engines (Temporal and friends) industrialise the pattern with a journal and deterministic re-execution; banks make it all non-negotiable because their effects are irreversible and their regulators demand reconstructable decisions.
