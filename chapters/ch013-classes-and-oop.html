Classes: objects with a job

Every framework you operate is a box of classes. Half an hour from now, class, __init__, and self will read like plain English — and framework source code will stop looking like a wall.

~30 minChapter 13 of 133
The story

Your friend Meera turned one Pune chai stall into a franchise. At head office there is a binder: the playbook. It declares what every outlet must have — a city, a till with opening cash, a cups-sold counter — and what every outlet must be able to do: open in the morning, sell a cup, cash up at night. The binder itself sells nothing. You cannot buy tea from a document.

But each time Meera signs a new outlet, the same ritual runs: premises found, then the opening-day checklist — put the float in the till, set the counter to zero, paint the city's name on the board. From that moment the outlet lives its own life. Pune's till fills at its own pace; Indore's counter climbs on its own schedule. Same playbook, separate realities.

And notice one small word doing quiet, essential work. The playbook says "at closing, count your till" — and every outlet applies the rule to itself. The playbook is a class. Each outlet is an instance. The opening-day checklist is __init__. And that little word your? Python spells it self.

You have been using objects all along

Quick recap from earlier in the course, because this chapter stands on it. In names and objects you learned that everything in Python is an object — a value that carries both data and behaviour, with names merely pointing at it. In the strings chapter you wrote "invoice".upper(), and in collections you wrote history.append(reply) — both times you were calling behaviour that lives on the value itself, reached through a dot. You have been a happy customer of objects for twelve chapters.

Here is the part nobody told you: str is a class. list is a class. dict is a class. Someone, once, wrote down the playbook for what every string has and can do — and every string you have ever created was an outlet opened from that playbook. A class is a blueprint that defines what data every object of a particular kind carries and what that object can do. This chapter, you stop being the customer and step behind the counter: you will write your own.

Why should an operator of AI systems care? Because frameworks are made of almost nothing else. When you write client = Anthropic() or mcp = FastMCP("weather"), you are instantiating classes. When you open framework source to see why something misbehaves, every second line starts with self. — and after today, that will read as effortlessly as your till, your counter.

Writing the blueprint

Here is Meera's playbook, translated. Read it slowly once — every piece gets unpacked below.

outlet.py
class Outlet:
    """One chai outlet in the franchise."""

    def __init__(self, city, opening_float):
        self.city = city
        self.till = opening_float
        self.cups_sold = 0

    def sell(self, cups):
        self.cups_sold += cups
        self.till += cups * 15

The anatomy: the class keyword, a name, a colon, and an indented body — the same block shape you know from functions. Two reading cues are worth locking in immediately. First, class names use CapWords by convention (Outlet, ChatSession, StateGraph) while functions and variables stay lowercase — so when you spot a capitalised name being called in unfamiliar code, you can bet it is a class being instantiated. Second, the body of a class is mostly def statements. A function defined inside a class gets a new job title: it becomes a method — behaviour that every object built from this class will carry around with it.

The blueprint alone brews no tea. To get a working outlet you call the class as if it were a function:

opening two outlets
pune = Outlet("Pune", 500)
indore = Outlet("Indore", 200)

That call is instantiation — asking the class to manufacture one fresh object, called an instance, built to the blueprint's specification. Two details that trip up every newcomer, settled now: you never call __init__ yourself, and you never pass self yourself. Python handles both. Here is the full sequence of what actually happens when that first line runs — step through it:

1. The class statement has already run

class Outletblueprint object in memory
punedoes not exist yet

When Python first reads the class block — typically at import time, as you saw in the previous chapter on files and imports — it builds one blueprint object and binds the name Outlet to it. No outlets exist yet; only the playbook does.

2. You call the class like a function

Outlet("Pune", 500)a call, with arguments
→
class Outlet

Round brackets after a class name mean "manufacture one". The arguments are the details this particular outlet needs — they are on their way to the opening-day checklist.

3. Python creates a blank object

class Outlet
→
(new empty object)no attributes yet

A brand-new, empty object appears in memory — the freshly signed premises, keys in hand, nothing inside. It knows which class built it, and nothing else.

4. __init__ runs, with self = that new object

__init__(self, "Pune", 500)self IS the blank object
→
self.city = "Pune"self.till = 500, self.cups_sold = 0

Python calls __init__ automatically, quietly passing the blank object as the first argument — that is self. Each self.something = value line bolts an attribute onto the new object: the opening-day checklist, executed.

5. The finished object is handed back

punea name
→
Outlet instancecity="Pune", till=500, cups_sold=0

The completed object is returned from the call, and the name pune is bound to it — ordinary name-binding, exactly as in chapter 2. Run the line again with different arguments and you get a second, fully independent object.

Note

Those double underscores around __init__ mark it as one of Python's special hook methods — pronounced "dunder init". The next chapter is entirely about these secret handshakes; for now, just know that the dunders are how your class plugs into machinery Python runs for you, and __init__ is the one that runs at instantiation.

self: the explicit "this"

Look again at def sell(self, cups):. Two parameters — yet you call it as pune.sell(3), passing one argument. Where does the other one come from?

self is the particular instance a method is currently working on, and Python passes it for you, automatically, as the first argument of every method call. The dot-call is really a shorthand. These two lines do exactly the same thing:

the shorthand, unfolded
pune.sell(3)          # what you write
Outlet.sell(pune, 3)  # what Python actually does

The method is written once, in the class. But when Pune sells tea, self is the Pune object, so self.till += cups * 15 tops up Pune's till. When Indore sells tea a moment later, the very same lines of code run with self pointing at the Indore object, and Indore's till changes instead. One recipe, cooked in whichever kitchen invoked it. That is the entire trick of object-oriented programming, and you already understood it the moment the playbook said "count your till".

If you have brushed against JavaScript or Java, this is the same idea as their this — except those languages make it appear invisibly inside methods, while Python puts it right there in the parameter list where you can see it. Explicit beats magic: when you read framework source, every self.history, self.model, self.tools tells you plainly "this lives on the object, not in a local variable". One more demystification: self is not a keyword. It is only an unbreakable convention for naming that first parameter — you could call it outlet, and the code would run, and every Python reader on Earth would frown. Never rename it.

Tip

Now break it on purpose — the errors you cause here are ones you will meet in real logs for years. Delete self from def sell(self, cups): and run: you get TypeError: Outlet.sell() takes 1 positional argument but 2 were given. Baffling until today; obvious now — Python invisibly passed pune as the first argument and found no parameter waiting for it. Attributes and methods: the nouns and the verbs

Everything you reach through the dot falls into two families. An attribute is data parked on the object — a noun you look at. A method is behaviour the object can perform — a verb you invoke. The reading cue is the brackets:

You readWhat it isIn the franchise
pune.tillAttribute — data, no bracketsGlance at the till
pune.sell(3)Method — behaviour, bracketsAsk the outlet to do work

This tiny distinction is a superpower when skimming unfamiliar code. agent.model — reading a setting. agent.invoke(state) — making it do something. No brackets, no side effects (usually); brackets, work is happening.

There is one more split hiding in the pyrun above, and the franchise makes it intuitive. self.till was created inside __init__, so each outlet gets its own — an instance attribute, one copy per object, like each outlet's till. But brand was defined directly in the class body, outside any method — a class attribute, a single copy stored on the blueprint and shared by every instance, like the menu board design head office prints for everyone. When you ask pune.brand, Python looks on the pune object first, finds nothing, then checks the class and finds it there. Instance first, class second — that lookup order is the whole rule.

shared vs. per-outlet
class Outlet:
    brand = "Cutting Chai Co."   # class attribute: ONE copy, on the blueprint
    price = 15                   # change it at HQ, every outlet sees it

    def __init__(self, city):
        self.city = city         # instance attribute: one per outlet
Careful

The classic trap: putting something mutable — a list or dict from chapter 4 — in the class body. Write history = [] at class level and every instance appends into the same shared list; session A suddenly "remembers" session B's messages. It is a famous, painful bug. The rule that prevents it: constants and settings may live on the class; anything an instance will change must be created in __init__ as self.history = [], so each object starts with its own.

When a class beats loose functions

Now the judgement call the tagline promised. Nothing about classes is required — you could run the whole franchise with plain functions and dicts. Plenty of code does, and here is what it looks like:

the loose-functions version
def make_session(name):
    return {"name": name, "history": []}

def ask(session, text):
    session["history"].append(text)
    # ...call the model, append the reply...

def summary(session):
    return f"{session['name']}: {len(session['history'])} messages"

Notice the smell: three functions, and every single one takes session as its first parameter, because they all operate on the same bundle of state. The bundle and its behaviours clearly belong together — but the language is not being told that. Any code anywhere can reach into the dict and mangle it; the related functions drift apart across files; and every caller must remember to pass the right bundle to the right function.

That smell — a bundle of data plus several functions that all take it as their first argument — is precisely the signal that the bundle wants to be an object. A class fuses the state and the behaviour into one thing with a job. And look what happens to that first parameter when you convert: session becomes self. That is all self ever was — the bundle, given its permanent seat.

Two sessions, two histories, zero bookkeeping by the caller — each object guards its own state and carries its own behaviour. Run it, then read it once more the way a reviewer would, line by line:

chat_session.py — read it like a reviewer
class ChatSession:

The opening-day checklist. Runs automatically on every ChatSession(...) call, with self already pointing at the brand-new blank object.

Read the two sides carefully: the right-hand name is the incoming argument, a temporary local; the left-hand self.name bolts it onto the object permanently. Parameter in, attribute out — the most common line in all of Python.

A fresh, private list for each session, created at instantiation time. This is the correct home for mutable state — putting history = [] up at class level would make every session share one list.

A method: a function that lives on the class and receives the instance first. Called as support.ask("..."); Python translates that to ChatSession.ask(support, "...").

The object updating its own state — no bundle passed in from outside, because the bundle IS self. Note this is this session's history; the other session is untouched.
reply = fake_llm(self.history)
In production this line is a real API call — client.messages.create(...). The scripted stand-in keeps the example runnable in your browser; the shape is identical.
self.history.append(reply)
return reply
Methods return values exactly like ordinary functions — because that is what they are, plus a reserved front seat for the instance.

One caution to balance the enthusiasm: do not convert everything into classes. Python is not one of those languages where every line must live inside one. A one-off transformation with no state to guard is happiest as a plain function; a pipeline of steps is often just functions calling functions. Reach for a class when state and behaviour genuinely travel together — and keep this ChatSession in your pocket, because an object holding a conversation history with an ask method is the exact seed that grows into the full agent loop you will build by hand in Act V.

In the wild

Once you see instantiation, you cannot unsee it. client = Anthropic(), mcp = FastMCP("weather"), graph = StateGraph(State) — capitalised name, brackets, object back: classes being instantiated, and the keyword arguments you pass are riding straight into someone's __init__. Open any framework's source file and count the self.s — an agent class reads self.model, self.tools, self.middleware, which you can now translate as "this particular agent's model, tools, and middleware". Two error messages also just became legible: TypeError: __init__() missing 1 required positional argument means you instantiated a class without something its checklist needs, and AttributeError: 'X' object has no attribute 'y' means you asked an object for a noun that was never bolted on. Both point you straight at the class definition.

Check yourself

1. Inside def sell(self, cups):, what exactly is self?

pune.sell(3) is shorthand for Outlet.sell(pune, 3) — Python slides the instance in as the first argument. It is not even a keyword, just an ironclad naming convention for that first parameter.

2. What does the line session = ChatSession("billing") actually do?

Instantiation is manufacture: blank object, opening-day checklist (__init__) attaches the starting attributes, finished object handed back and bound to the name. The other methods sit ready but do not run until called.

3. brand = "Cutting Chai Co." sits in the class body; self.till = 500 sits inside __init__. What is the difference?

Class attributes live once, on the class, shared by all instances (the printed menu from head office). Instance attributes are created per object in __init__ (each outlet's own till). Lookup checks the instance first, then the class.

4. A teammate writes history = [] directly in the class body of ChatSession. What goes wrong?

A class attribute is a single shared object. Every instance appending to it appends to the same list — the classic mutable-class-attribute bug. Mutable state belongs in __init__ as self.history = [], giving each instance its own.

5. You find three functions that all take the same session dict as their first parameter. What is that a signal for?

Data plus several functions that all operate on it is exactly what a class expresses: state and behaviour fused into one object with a job. The tell is beautiful — the parameter they all shared is literally what self becomes.

Key takeaways

  • A class is a blueprint; an instance is one object manufactured from it by calling the class like a function. You have used them since chapter 1 — str, list, and dict are classes too.
  • __init__ is the opening-day checklist: it runs automatically on every new instance and bolts on its starting attributes. You never call it yourself.
  • self is Python's explicit "this" — the instance a method is working on, passed automatically: pune.sell(3) is really Outlet.sell(pune, 3).
  • Attributes are the nouns (data, no brackets); methods are the verbs (behaviour, brackets). A method is just a function with a reserved front seat for the instance.
  • Class attributes live once on the blueprint and are shared; instance attributes are created per object in __init__. Never put mutable state (lists, dicts) at class level.
  • Reach for a class when several functions keep passing the same bundle of state around — and not before. Plain functions remain excellent Python.
  • Framework code is classes all the way down: Anthropic(), FastMCP("weather"), StateGraph(State) are instantiations, and every self. in their source now reads as plain English.
