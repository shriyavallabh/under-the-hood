Pipelines: .gitlab-ci.yml decoded

That intimidating CI file at the root of every repo is a YAML document you could already read three acts ago. Today you decode one keyword at a time — stages, jobs, runners, artifacts — until the whole thing is boring.

~35 minChapter 102 of 133
The story

The night before a big wedding, the head caterer tapes one laminated sheet to the kitchen wall and goes home. The sheet lists stations in strict order — CHOP, COOK, TASTE, PACK — and under each station, the tasks: dice onions, grind masala, simmer dal, taste every pot, seal two hundred boxes. Workers arrive through the night, read the sheet, and pick up whatever task their station allows. Nobody phones the caterer to ask what to do next. The sheet is the boss.

Watch the details. Chopped vegetables leave CHOP in labelled trays that are handed forward to COOK — the next station never re-chops. The masala trolley stays stocked between weddings so nobody grinds from scratch every single event. A margin note says "sugar-free batch: only if the client asked". And one packer is allowed to start folding boxes at midnight, hours before cooking ends, because folding boxes only depends on the guest count, not on the dal.

That sheet is .gitlab-ci.yml. The stations are stages, the tasks are jobs, the workers are runners, the labelled trays are artifacts, the trolley is the cache, the margin note is a rule, and the early-starting packer is needs. You are about to read the sheet.

You already read this language

Back in the YAML chapter you learned the whole grammar: indentation shows what belongs inside what, key: value pairs make a mapping, and lines starting with - make a list. You learned that YAML is just structured data — a shape, not a program — and that most configuration files in this industry are YAML wearing different hats. That chapter is the key that turns in today's lock, because a .gitlab-ci.yml file is nothing but one YAML mapping, sitting at the root of the repository.

Here is the smallest pipeline that does anything at all:

.gitlab-ci.yml — the whole file
stages:
  - test

say-hello:
  stage: test
  script:
    - echo "the pipeline is alive"

Read it with your ch005 eyes. Two top-level keys. stages is a list with one item. say-hello is a mapping with two keys inside it, and script holds a list of one string. That is the entire trick of this chapter: GitLab reserves a handful of top-level keywords (stages, default, variables, workflow, include, a few more) — and every other top-level key you write becomes a job. You do not declare jobs inside some jobs: block. You just name them, the way say-hello is named here.

Commit this file to a repo, push, and GitLab creates a pipeline — one full run of the sheet, triggered by that push — with one job in it. The job prints its line, exits with code 0, and the commit gets a green tick. Every green tick you have ever seen next to a commit in GitLab was earned exactly this way.

Stages and jobs: columns and cards

A stage is a named phase of the pipeline — a station on the kitchen wall. The stages: list at the top of the file declares the stations and their order: lint, then test, then build, then deploy. A job is one unit of actual work — a named block with a script: — and every job says which stage it belongs to. Two rules govern everything you will ever watch in the pipeline UI:

  • Jobs in the same stage run in parallel, if enough workers are free. Unit tests and type checks do not queue behind each other; they are both TEST-station tasks.
  • The next stage starts only when every job in the current stage has passed. One red job in TEST and the BUILD station never opens.

The script is simply a list of shell commands, run top to bottom in a fresh environment — the same commands you could type in a terminal. If the last command exits with status 0, the job passes; any non-zero exit turns it red. There is no deeper magic: CI is a robot typing shell commands and reading the exit code, the same exit-code contract you'll formalise when you meet pytest a couple of chapters from now.

And image:? Each job runs inside a container started from that image — python:3.12-slim hands every job an identical, clean machine with Python pre-installed. You'll see what images are made of in the containers chapter; for now, picture an image as a frozen, ready-made machine — and here is where they quietly earn their keep. A fresh container per job is why "works on my machine" arguments end at the pipeline: there is no my machine any more.

Runners: who actually does the work

Here is the fact that unlocks the whole architecture, and it should feel familiar: the GitLab server never executes your code. GitLab is the caterer's office — it stores the sheet, decides what may start when, and displays the pretty graph. The actual cooking is done by a runner: a small standalone program (called GitLab Runner) installed on some machine — a virtual machine, a container host, a Kubernetes cluster — that polls the server on a loop: "any jobs for me?"

When a job is handed over, the runner clones the repository at exactly that commit, starts a container from the job's image, runs the script lines one by one, streams the log back live (that is the black console you watch in the UI), and reports the exit code. Then it asks for more work. If this reminds you of the ReAct loop, trust the feeling — the YAML file, like the model, only ever describes actions. Something else, standing outside, does the doing.

Runners come in flavours you will hear named in stand-ups. Shared runners belong to the whole GitLab instance — on gitlab.com they are the free machines everyone queues for. Group and project runners are machines your own team registered, and jobs choose them with tags: — a data-science team's training job might say tags: [gpu] so it only lands on the machine with the graphics card. When a colleague sighs "the pipeline is stuck, no runner picked it up", they mean: a job is advertised with tags no available worker carries. Nothing is broken; nobody with the right apron showed up.

Note

This division of labour is why CI can be trusted: the runner starts from a clean container every single time. No leftover files, no half-installed packages from Tuesday, no local hacks. Your repo plus your YAML must contain everything the job needs — which is precisely what makes a green pipeline meaningful.

A real file, decoded line by line

Now the centrepiece. Below is a realistic .gitlab-ci.yml for a small Python project — the lint, test, build, deploy pipeline the stepper just animated. Click any highlighted line and make sure its note feels obvious; as with the ReAct loop, boring means fluent.

.gitlab-ci.yml — a Python project's full pipeline
stages:
The stations, declared once, in order. Any job naming a stage not in this list is a syntax error — GitLab refuses the whole pipeline.
- lint
- test
- build
- deploy
Plain YAML list items — your ch005 fluency at work. Order here is execution order. If a job omits its stage entirely, GitLab quietly defaults it to test.
default:
Settings under default apply to every job that does not override them — the house rules of the kitchen.
image: python:3.12-slim
Every job runs in a fresh container from this image. One job could still say image: python:3.11-slim to test an older version — job-level keys beat default.
variables:
Environment variables visible to all jobs. GitLab also injects dozens of predefined ones — CI_COMMIT_BRANCH, CI_PROJECT_DIR, CI_DEFAULT_BRANCH — which the rules below lean on.
PIP_CACHE_DIR: "$CI_PROJECT_DIR/.cache/pip"
Points pip's download folder inside the project directory — because that is the only place the cache system (next block) is allowed to save and restore.
cache:
The masala trolley. Between pipelines, these paths are zipped and stashed near the runner so repeat runs skip re-downloading packages.
key:
files:
- requirements.txt
The cache key is derived from this file's contents: edit requirements.txt and you get a fresh cache automatically. Stale-dependency bugs, dissolved in three lines of YAML.
paths:
- .cache/pip
What actually gets stashed. Note what is not cached: your installed environment. Each job still runs pip install — it just hits local disk instead of the network.
ruff-lint:
Not a reserved keyword, so it is a job — the name you will see on the pill in the graph. Naming jobs after their tool is a widely loved convention.
stage: lint
script:
- pip install ruff
Remember: fresh container, empty except for Python. Every job installs what it needs. Feels wasteful; it is the price of perfectly reproducible builds.
- ruff check src/
The linter. Exits 0 if the code is clean, non-zero if not — and that exit code alone decides whether this pill turns green or red.
unit-tests:
stage: test
Same stage as type-check below, so the two run in parallel once lint passes — the two cooks at the TEST station.
script:
- pip install -r requirements.txt
- pytest --junitxml=report.xml
Runs the test suite (chapter ch103) and also writes the results as a machine-readable XML file — for the next three lines to pick up.
artifacts:
Declares what this job hands forward after it finishes — the labelled tray leaving the station.
when: always
Save the report even when tests fail — especially when tests fail. Without this, artifacts upload only on success, which is exactly when you least need a test report.
reports:
junit: report.xml
A special artifact flavour: GitLab parses this XML and shows failed tests by name right inside the merge request widget, so reviewers never dig through raw logs.
type-check:
stage: test
script:
- pip install -r requirements.txt mypy
- mypy src/
Static type checking against the hints you met in ch016. A second, independent verdict on the same commit, earned in parallel with the unit tests.
build-package:
stage: build
Only reachable when every test-stage job passed. By the time anything is packaged, it has already survived lint, tests, and types.
script:
- pip install build
- python -m build
Builds the wheel — the installable package format from ch020 — into dist/.
artifacts:
paths:
- dist/
The built wheel is uploaded to GitLab itself. Later jobs receive it automatically, and you can download it from the UI — the parcel, not the trolley.
expire_in: 1 week
Housekeeping: artifacts eat server storage, so give them a shelf life. A week is plenty for anything a human might want to fetch.
deploy-uat:
stage: deploy
needs: ["build-package"]
The queue-jumper. This job waits only for build-package — not for every earlier job — and inherits its artifacts, so dist/ is already sitting in the container when the script starts.
script:
- pip install twine
- twine upload --repository-url "$UAT_INDEX_URL" dist/*
Pushes the wheel to the team's test package index. $UAT_INDEX_URL is a CI/CD variable set in project settings — secrets live there, never in this file, never in git.
rules:
The margin note. Rules are evaluated when the pipeline is created, top to bottom, first match wins — deciding whether this job exists in this run at all.
- if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
Condition: this push landed on the default branch (usually main). On a feature branch this rule does not match, no rule remains, and the job simply is not created — no grey pill, nothing.
when: manual
Even on main, do not fire automatically — render a play button and wait for a human. The whole release gate for many teams is exactly this one line.
Tip

You never have to hand-check your YAML by pushing and praying. GitLab ships a pipeline editor (left sidebar, under Build) that validates the file as you type and previews the graph before you commit. Ten seconds there saves a red pipeline and a noisy notification to the whole team.

Artifacts versus cache: the parcel and the trolley

These two get confused in every team, so pin the distinction now. An artifact is an output — files a job produced that later jobs (and humans) need; GitLab guarantees delivery by storing them on the server. A cache is an optimisation — stuff worth keeping around so the next run is faster; it lives near the runners and comes with no guarantee whatsoever. The kitchen knows the difference instinctively: the tray of chopped onions must reach the next station or dinner fails; the masala trolley being restocked is merely nice.

ArtifactsCache
PurposePass a job's outputs forwardSpeed up repeat runs
Typical contentsBuilt wheel, test report, coverage filepip downloads, dependency folders
Guaranteed?Yes — stored by the GitLab serverNo — may be empty at any time
Visible in the UI?Yes — Browse and Download buttons per jobNo
If it vanishedThe pipeline would breakThe pipeline would just run slower
Careful

The classic mistake: using cache to pass files between stages. It appears to work — until the two jobs land on different runners with different local caches, and the file "randomly" is not there. If a later job needs it, it is an artifact. No exceptions.

needs, rules, and the ghost of only

Two refinements complete your fluency. First, needs replaces the rigid stage-by-stage march with a precise dependency graph: a job listing needs: ["build-package"] starts the moment that one job passes, even while unrelated jobs in earlier stages are still grinding. The extreme case, needs: [], means "I depend on nothing — start immediately", which is how a documentation job can finish before the test suite is halfway done. Big pipelines earn minutes of wall-clock time this way.

Second, rules decide whether a job is created at all. Each rule can test a condition (if:), react to which files changed (changes:), and set the start behaviour (when: on_success, manual, or never). A pattern you will meet constantly in agent and data-science repos — run the expensive evaluation job only when prompts or model code actually changed:

rules in the wild — run evals only when it matters
eval-agent:
  stage: test
  script:
    - python -m evals.run --suite smoke
  rules:
    - changes:
        - prompts/**/*
        - src/agent/**/*
      when: on_success

And the ghost: older files use only: and except: — for example only: [main] — to do a cruder version of the same thing. They still work, but GitLab froze their development years ago and the documentation steers everyone to rules. Read only: fluently in legacy files; write rules: in new ones. (Evaluation pipelines for agents get their own chapter in agent evals.)

In the wild

Open the repo of any team you join and look for .gitlab-ci.yml in the root — it is the single most honest document about how that team really works: what they test, what they deploy, and what still needs a human's play button. In large organisations the file is often startlingly short, because an include: line at the top pulls in shared templates maintained by a platform team — the same decoding skills apply, just across two files. And in merge-request chatter, you can now translate fluently: "pipeline's red" (a job's script exited non-zero), "retry it, that runner is flaky" (re-run the job, blame the worker not the recipe), "just bump the cache key" (force a fresh trolley), "make it manual on main" (add the two rules lines you read today).

Check yourself

1. When a job's script: lines execute, where do they actually run?

GitLab only coordinates and displays. A runner — its own program on its own machine — claims the job, starts a fresh container from the job's image, types the script, and streams the log back. Same division of labour as the ReAct loop: the file describes, the runner does.

2. Your build job creates dist/app.whl and the deploy job needs that file. Artifact or cache?

Artifacts are outputs stored by the GitLab server and handed to later jobs reliably. Cache is a best-effort speed-up that can vanish whenever jobs land on different runners — the classic source of "it worked yesterday" pipeline bugs.

3. A deploy job declares needs: ["build-package"]. What does that change?

needs replaces "wait for every earlier stage" with a precise dependency list — and the needed job's artifacts come along automatically. It is the queue-jump that makes big pipelines fast.

4. A job's only rule is if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH with when: manual. You push to a feature branch. What happens to the job?

Rules are evaluated when the pipeline is created. No rule matches on a feature branch, so the job is simply not created — no pill, no button, nothing. The manual play button appears only on default-branch pipelines, where the rule matches.

5. In a .gitlab-ci.yml file, what makes a top-level key a job?

There is no jobs: block. GitLab reserves a handful of keywords (stages, default, variables, workflow, include…) and treats every other top-level mapping as a job definition — which is why say-hello and ruff-lint just work.

Key takeaways

  • A .gitlab-ci.yml file is plain YAML: a few reserved top-level keywords, and every other top-level key is a job. Your ch005 fluency reads all of it.
  • Stages run strictly in order; jobs inside a stage run in parallel; one red job blocks every stage after it. A job is just shell commands in a fresh container, judged by exit code.
  • GitLab never executes your code — runners do: separate machines that poll for jobs, clone the commit, run the script, and stream the log back. Tags decide which runner gets which job.
  • Artifacts are guaranteed outputs passed between stages and downloadable in the UI; cache is a best-effort speed-up that may be empty. Passing files through cache is the classic pipeline bug.
  • needs turns the stage march into a precise dependency graph so independent jobs jump the queue; rules (successor to the older only/except) decides whether a job exists in a given pipeline and whether it waits for a human's play button.
  • The pipeline graph in the UI is the file rendered live — and the file itself is the most honest document about how a team actually tests and ships.
