Checkpointing and persistence

Right now, your graph's entire memory lives inside a process that can die at any second. One argument to compile() — a checkpointer — turns that fragile run into something that survives crashes, restarts, and deploys. This chapter shows you exactly how, database and all.

The story

A tailor is stitching a wedding outfit — a job far too big for one sitting. She has a habit her whole shop runs on: after every single seam, she pins a small card to the garment. The card says what is done, which thread she used, and what comes next. Then the garment goes back on the rack, on a hanger tagged with the customer's order number.

One Tuesday the power cuts out mid-stitch and everyone goes home. On Wednesday the tailor is ill, so her sister opens the shop instead. The sister has never seen this outfit. She pulls the hanger by its order number, reads the last pinned card — "sleeves done, collar next" — and picks up exactly where the work stopped. No seam is re-stitched. No customer is asked to explain their order again.

The pinned card is a checkpoint. The habit of writing one after every seam is a checkpointer. The order number on the hanger is a thread_id. And the rack — the sturdy thing that holds the work while people and power come and go — is a database. This chapter is that shop, in Python.

A five-minute detour: what even is a database?

Before the LangGraph part makes sense, you need one piece of background that most programming courses assume and never explain. It takes five minutes, and it starts with your own laptop.

A computer keeps information in two very different places. Memory (RAM) is the whiteboard — blazing fast to read and write, but wiped clean the instant the power goes or the program ends. Disk is the notebook — slower, but what you write there is still there tomorrow. Every variable in every Python program you have run in this course lived on the whiteboard. Close the program, and it is gone. That is why your terminal agents forget things when a session ends: nobody wrote to the notebook.

A database is a program whose only job is to be a very good notebook: it takes data from other programs, writes it safely to disk, and hands it back on request — even to a different program, on a different day, after a crash. Two databases matter for this chapter, and they sit at opposite ends of the whiteboard-notebook spectrum.

PostgreSQL (everyone says "Postgres") is the filing cabinet with a meticulous librarian. Data goes into labelled tables — think spreadsheet-like grids of rows — and you talk to the librarian in a small English-like language called SQL: SELECT * FROM checkpoints WHERE thread_id = 'invoice-42' reads almost like the sentence it means. Postgres's obsession is never losing anything: it confirms a write only once the data is truly safe on disk. It has run banks, airlines, and half the internet for three decades.

Redis is the pad of sticky notes on the side of the monitor. It keeps everything in memory — the whiteboard — which makes it absurdly fast, roughly the difference between glancing at a sticky note and walking to the filing cabinet. The trade-off is the whiteboard's trade-off: memory is volatile, so Redis periodically copies its notes to disk as insurance rather than filing every one instantly. You choose Redis when speed matters most; you choose Postgres when losing a single row would be unacceptable.

Note

You will not administer these databases in this course, and you do not need to. You use them the way you use a web API: a server runs somewhere (your laptop via Docker, or a managed cloud service), and your Python code connects to it with one line — a connection string like postgresql://user:password@localhost:5432/agent_memory. That string is just an address plus credentials, and it almost always arrives via an environment variable such as DATABASE_URL.

The amnesia problem your graph has

Back to LangGraph. In the previous chapter you compiled a graph and invoked it: state flowed in, nodes transformed it, state flowed out. Here is the uncomfortable question: where was that state, physically? In a Python object. On the whiteboard. Inside one running process.

So when the process ends — the script finishes, the server redeploys, the pod restarts, someone trips over a cable — every conversation your graph was holding evaporates. The user who spent ten minutes explaining their problem to your support agent is now talking to a goldfish. You met this theme in the memory chapter: an LLM remembers nothing between calls, so memory must be engineered around it. Checkpointing is where that engineering stops being a list and becomes infrastructure.

The fix is a pattern the industry calls durable execution: record your progress in a durable place after every step, so that any crash, restart, or handover can resume from the last record instead of starting over. The tailor's pinned cards. Video games invented the friendly name for it decades ago — the save point. LangGraph implements it with two ideas, and only two:

A checkpoint is a snapshot of the graph's entire state at one moment — every channel of your state object, plus bookkeeping about which node runs next. A checkpointer is the component that writes one of those snapshots to storage after every step of the graph, automatically. You never call it yourself. You hand it to .compile() once, and from then on every node that finishes gets its seam pinned.

Threads: giving a conversation a name

One graph serves many users at once, so the saved snapshots need labels — otherwise whose conversation would you be resuming? That label is the thread_id: a string you choose that names one independent conversation, the way the order number on the hanger names one outfit. It is the dry-cleaner's ticket of agent engineering: present the same ticket, get the same garment back, exactly as you left it.

You pass it at invoke time, in a small config dictionary:

the thread ticket
config = {"configurable": {"thread_id": "customer-8841"}}
graph.invoke({"messages": [("user", "Where is my order?")]}, config)

Same thread_id tomorrow? The checkpointer loads that thread's latest checkpoint and the conversation continues with full history. A different thread_id? A blank slate. This is precisely why every chat product on Earth has that sidebar of past conversations — each entry is a thread id over a store of checkpoints, and clicking one just means "invoke with this ticket".

Watch the whole machine work through one crashy afternoon:

The checkpointer checks the database for thread "invoice-42". Nothing there — so this is a brand-new conversation, starting from your input state.

The draft node finishes and the checkpointer snapshots the whole state to Postgres before anything else happens. Seam stitched, card pinned.

3. Next node, next checkpoint

Review runs; checkpoint 2 lands in the database. Note that checkpoints accumulate — the thread now holds its whole history, not just the latest state.

4. The process dies mid-run

processkilled: deploy / crash / power cut
sendnever ran
Postgrescheckpoint 2 is safe on disk

The whiteboard is wiped: every Python object is gone, including the graph and its in-flight state. The filing cabinet does not care. Checkpoint 2 sits there, patient.

5. A new process presents the same ticket

new processsame graph, recompiled
→
Postgresloads checkpoint 2 for invoice-42
→
graphnext node: send

A fresh process — the sister opening the shop — compiles the same graph, connects the same checkpointer, and resumes ticket "invoice-42" with graph.invoke(None, config) — None meaning "no new input, just continue". The checkpointer restores the state, and execution picks up from where it stopped. (Hand it a fresh input dict instead and you start a new run from the top on top of the saved state, repeating the finished work.)

6. Only the unfinished work runs

draftnot repeated
reviewnot repeated
sendruns now
→
Postgresfinal checkpoint

Draft and review are not re-executed — their results were already filed. Only send runs. That is durable execution: no lost work, no repeated work, no re-asking the user.

The real thing, five lines at a time

Here is a complete persistent chat graph in current LangGraph. Almost all of it you already know from compile, invoke, stream — the new material is three lines. Click through the annotations.

persistent_chat.py
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
A reducer that appends new messages to the existing list instead of overwriting it — so each turn's messages pile up into a conversation. Reducers were the "how state updates merge" idea from the state chapter.
from langgraph.checkpoint.memory import InMemorySaver
New line one: the simplest checkpointer. It keeps checkpoints in a Python dict — the whiteboard. Perfect for development and tests; useless against a restart, as we will see.
from typing import Annotated, TypedDict
class State(TypedDict):
The state schema — a typed dict declaring what flows through the graph. A checkpoint is a snapshot of exactly this, plus bookkeeping about what runs next.
messages: Annotated[list, add_messages]
def chat(state: State):
reply = model.invoke(state["messages"])
The one line that talks to an LLM, exactly as in your hand-built loop. Thanks to the checkpointer, state["messages"] already contains every earlier turn of this thread — loaded from storage, not from luck.
return {"messages": [reply]}
builder = StateGraph(State)
builder.add_node("chat", chat)
builder.add_edge(START, "chat")
builder.add_edge("chat", END)
graph = builder.compile(checkpointer=InMemorySaver())
New line two, and the heart of the chapter. Handing a checkpointer to .compile() switches on persistence for the whole graph: after every step, the state is saved; before every invoke, the thread's history is restored. No other code changes.
config = {"configurable": {"thread_id": "ravi-support"}}
New line three: the ticket. Any string works — user ids, order numbers, UUIDs. Everything invoked with this config belongs to one continuing conversation.
graph.invoke({"messages": [("user", "My name is Ravi.")]}, config)
graph.invoke({"messages": [("user", "What is my name?")]}, config)
Same ticket, so the checkpointer loads the previous turn first and add_messages appends the new question. The model sees both turns and answers "Ravi". Invoke with a different thread_id and it would have no idea.
snapshot = graph.get_state(config)
You can inspect a thread whenever you like: get_state(config) returns the latest checkpoint — the state values plus what would run next. Its sibling get_state_history(config) lists every checkpoint ever taken for the thread, newest first. Debugging gold.
Tip

Read graph.compile(checkpointer=...) the way you would read "hire the scribe". Ordinary functions do not get automatic save points; your graph does, because a component you attached once now files every step. If that move — behaviour bolted on from outside, code unchanged — feels familiar, it should: it is the same shape as decorators.

Choosing a backend: from dev to production

Everything above used one checkpointer. LangGraph ships several, all interchangeable behind the same .compile(checkpointer=...) slot — you pick where the filing happens, and nothing else about your code changes.

InMemorySaver is for development: zero setup, instant, and wiped with the process — a checkpointer made of whiteboard. The production savers live in small add-on packages named langgraph-checkpoint-postgres, langgraph-checkpoint-redis, and langgraph-checkpoint-sqlite, and they file checkpoints into exactly the databases you met in the primer:

production: checkpoints in Postgres
# pip install langgraph-checkpoint-postgres
from langgraph.checkpoint.postgres import PostgresSaver

DB_URI = "postgresql://agent:secret@localhost:5432/agent_memory"

with PostgresSaver.from_conn_string(DB_URI) as checkpointer:
    checkpointer.setup()      # one-time: creates the checkpoint tables
    graph = builder.compile(checkpointer=checkpointer)
    graph.invoke(user_input, config)

Three things worth noticing. The connection string is the only Postgres knowledge required. The with block is a context manager making sure the connection closes cleanly. And checkpointer.setup() runs once to create the tables — after that, the librarian handles the filing. The Redis saver follows the same pattern from langgraph.checkpoint.redis; check the current docs for its exact setup call, as the add-on packages evolve faster than the core.

SaverBacked byReach for it when
InMemorySaverA Python dict (whiteboard)Development, tests, notebooks
SQLite saverA single file on diskLocal tools, demos, one-machine apps
PostgresSaverPostgres (filing cabinet)Production — the boring, correct default
Redis saverRedis (sticky notes)High-traffic state where speed dominates
Careful

The classic mistake: shipping InMemorySaver to production because it worked beautifully in development. It will keep working beautifully — until the first deploy, when every live conversation silently vanishes and your support inbox explains the word "volatile" to you. If conversations must survive a restart, the checkpointer must write to a real database. No exceptions.

Persistence also unlocks the doors you walk through next. The coming chapter's human-in-the-loop interrupts only work because a graph can pause, checkpoint, wait a day for a human's approval, and resume — a pause is just a checkpoint nobody has resumed yet. Time travel (rewinding a thread to an earlier checkpoint and re-running from there) and forking conversations are the same trick read backwards. Checkpointing is not a feature among features; it is the floor the other features stand on.

Check yourself

1. What exactly does a checkpointer do once you pass it to .compile()?

A checkpointer files the state — not the code, and not just the final result — after every single step. That per-step granularity is what makes mid-run crash recovery possible.

2. Two users chat with the same compiled graph. What keeps their conversations from bleeding into each other?

One graph serves everyone; the thread_id in the invoke config is the only thing separating conversations. Same ticket, same garment — different ticket, different garment.

3. Your agent works perfectly with InMemorySaver in development. Why must it not ship to production that way?

InMemorySaver is fast and handles many threads fine — its flaw is volatility. It is a checkpointer made of whiteboard: the first deploy wipes it. Production checkpoints belong in a real database.

4. In the filing-cabinet versus sticky-notes picture, what is the core trade-off between Postgres and Redis?

They are both excellent and both current — they just sit at opposite ends of the memory-versus-disk spectrum. Durability-first is Postgres; speed-first is Redis. Neither handles text versus numbers differently.

5. A graph with a Postgres checkpointer crashes after node 2 of 3. What happens when a new process invokes with the same thread_id?

This is durable execution: the last checkpoint survives the crash on disk, so the new process resumes exactly there — no lost work, no repeated work, no re-asking the user. One catch: to continue rather than restart, invoke with None (graph.invoke(None, config)); passing a fresh input dict instead starts a new run from the top on top of the saved state.

Key takeaways

  • A database is a program whose job is remembering: Postgres is the filing cabinet on disk with a librarian who speaks SQL; Redis is the sticky notes held in memory for speed. You use both through one connection string.
  • Without a checkpointer, a graph's state lives on the whiteboard — inside one process — and every restart wipes every conversation.
  • A checkpointer, attached once via .compile(checkpointer=...), snapshots the full state after every step, automatically. That per-step habit is the whole mechanism.
  • The thread_id in config={"configurable": {"thread_id": ...}} is a conversation's identity: same id resumes with full history, new id starts clean.
  • InMemorySaver is for development; Postgres, Redis, or SQLite savers from the langgraph-checkpoint-* packages are for anything that must survive a deploy.
  • Durable execution — resume from the last checkpoint after any crash — is also the foundation for what comes next: human-in-the-loop pauses are just checkpoints waiting for a person.
