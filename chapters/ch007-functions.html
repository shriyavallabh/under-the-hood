Functions: reading any signature

Every function definition is an order form with five possible zones, always in the same order. Learn the zones once and no signature — stdlib, LangChain, or a two-thousand-line enterprise module — will ever look foreign again.

~30 minChapter 7 of 117
The story

There is a small travel agency near the railway station that has booked tickets the same way for thirty years. On the counter sits a pad of booking forms. Two fields are compulsory — passenger and destination; the clerk will not move without them. Two more arrive pre-filled in faint grey ink: seat: economy, meal: veg. Cross them out and write your own if you care; leave them and the grey ink stands. Regulars rattle off answers in order — "Asha, Jaipur" — and the clerk fills the fields top to bottom. Fussier customers name their fields — "meal: Jain, seat: window" — in whatever order they please. At the bottom sit two catch-all boxes: extra tags — attach as many slips as you like, and other preferences — write each as label = value. When the form is complete, the clerk disappears behind a bead curtain and returns with exactly one thing: your ticket. Taped to the counter is a laminated card explaining, in one friendly paragraph, what this form is for.

That form is a function signature. The compulsory fields are parameters, the grey ink is defaults, naming your fields is keyword arguments, the two catch-all boxes are *args and **kwargs, the ticket is the return value, and the laminated card is the docstring. You have filled this form a thousand times without knowing it. Today you learn to read it at a glance.

The machine behind the counter

Quick recap of where you stand. In chapter 2 you learned that Python variables are names stuck on objects. In the last chapter you learned how if, for and while steer a program — decisions and repetition. But everything so far ran top to bottom, once. Functions are how code becomes reusable: you wrap a block of logic, give it a name, and from then on anyone can trigger it by filling in the form.

A function is a named, reusable block of code that takes some inputs and hands back a result. You create one with the def keyword — short for define. The line that starts with def is the signature: the function's name plus the full list of what it accepts, exactly like the printed fields on the booking form. Two words that sound interchangeable but are not: a parameter is a named slot printed on the form (in the def line), while an argument is the actual value you write into a slot when you call the function. Parameters are the blanks; arguments are the ink.

Here is the agency as four lines of Python. Read it, then click each annotated line in the walkthrough below until every note feels obvious.

booking.py — a complete function, top to bottom
def book_ticket(passenger, destination, seat="economy", meal="veg"):
The signature. def starts the definition, book_ticket is the machine's name, and the brackets hold four parameters. The first two are bare names — compulsory fields. The last two carry = and a value — grey-ink defaults a caller may simply leave alone. The colon at the end promises an indented body, exactly like the if and for lines of the last chapter.
"""Book one ticket and return its summary line."""
The docstring — the laminated card. A plain string sitting as the first statement in the body. Python stores it on the function itself, and tools like help() read it back. Later in this chapter you will see why agent frameworks care about it far more than you would expect.
label = f"{passenger} to {destination}"
Inside the body, the parameters behave like ordinary variables that were pre-loaded with whatever the caller sent. The f-string stitching is chapter 3.
return f"{label}: {seat} seat, {meal} meal"
return is the bead curtain. It hands one value back to whoever called and ends the function on the spot — nothing after a reached return ever runs. The return value is the ticket.
summary = book_ticket("Asha", "Jaipur", meal="jain")
The call. "Asha" and "Jaipur" fill the first two slots by position; meal="jain" crosses out one piece of grey ink by name; seat was never mentioned, so it stays "economy". The returned ticket lands in the variable summary.

What comes back: return, and the print trap

The single most common beginner confusion is print versus return, so let us kill it early. print shows a value on the screen and hands back nothing useful; return hands a value back to the calling code, silently. A function that prints is a clerk who shouts your booking details across the room and then shreds the ticket. A function that returns quietly passes you the ticket so you can do something with it — store it, pass it to another function, test it. Production code returns; printing is for humans watching.

Two more facts complete the picture. First, a function with no return statement (or a bare return with nothing after it) still returns something: the special value None, Python's word for "nothing here". If you have ever seen a mysterious None appear where a result should be, somebody forgot to return. Second, when a function needs to hand back several things, it returns them separated by commas — Python quietly bundles them into a tuple, one of the four containers, and the caller unpacks it: text, cost = call_model(prompt).

The laminated card: docstrings

A docstring is a triple-quoted string placed as the very first statement inside a function body. It documents what the function is for — not how it works, but what a caller may expect. Type help(book_ticket) in any Python session and the docstring comes back, along with the signature. Every serious codebase treats them as mandatory for public functions.

Note

Here is the part that makes docstrings personally relevant to you as an agent operator: modern agent frameworks lift the docstring off a tool function and hand it to the model as the tool's description. In the MCP Python SDK, a function decorated with @mcp.tool() (a decorator — that @-marker line gets its own chapter later) advertises itself to the model using its docstring; LangChain tools do the same. A lazy docstring literally makes your agent worse at choosing tools — the model picks tools by reading those descriptions. The laminated card is not decoration; it is the menu the model orders from.

Say the words before we go deeper

Grey ink: how defaults really behave

A default value turns a compulsory field into an optional one. Write seat="economy" in the signature and every caller who says nothing about seats gets economy. Two rules govern them. First, position: once one parameter has a default, every parameter after it must have one too — def f(a, b=1, c) is a syntax error, because Python could never tell which slot a two-argument call meant to fill. All the grey ink lives at the end of the form. Second, defaults make change cheap: adding a new defaulted parameter to a widely used function breaks nobody, because every existing call simply inherits the default. Adding a new required parameter breaks every caller at once. Reviewers approve one of these in seconds and question the other — now you know why.

Careful

One famous trap: a default is created once, when the def line runs — not freshly on every call. With immutable defaults like 3 or "veg" that is invisible. But def add_task(task, queue=[]) creates one shared list that every call without a queue argument silently reuses — remember names stuck on objects: each call sticks the same name on the same list, and tasks from unrelated calls pile up together. The convention you will see everywhere: default to None, then create the fresh list inside the body. When a signature says tools=None, this is usually why.

By position or by name

Every ordinary parameter can be filled two ways. A positional argument relies on order: book_ticket("Asha", "Jaipur") works because "Asha" is first and so is passenger. A keyword argument names its slot: book_ticket(destination="Jaipur", passenger="Asha") says exactly where each value goes, so order stops mattering. The grammar has three small rules: positional arguments must come before keyword arguments in a call, each parameter may be filled only once, and a name that is not on the form is an immediate TypeError — the clerk hands the form straight back.

Style matters here more than syntax. Real production code leans hard on keyword arguments because they make call sites self-documenting: retrain(data, 0.2, True) tells a reviewer nothing, while retrain(data, test_split=0.2, shuffle=True) reads like a sentence. The two stars: catch-all boxes

Now the part that makes most newcomers' eyes slide off a signature: *args and **kwargs. The names args and kwargs are only conventions — the stars are the machinery. A single star in a signature means: scoop up every extra positional argument into a tuple. A double star means: scoop up every extra keyword argument into a dict — yes, the same dict you met in chapter 4. They are the form's two catch-all boxes: as many extra slips as you like, and any label = value preferences the printed fields did not anticipate.

And here is the elegant part: the stars work in both directions. In a def line they collect; at a call site they unpack. Write call_model(prompt, **config) and Python explodes the config dict into individual keyword arguments, as if you had typed each one by hand. Remember chapter 5, where a YAML settings file became a Python dict the moment it was loaded? This is the payoff: that dict gets starred straight into a function call. The five zones, assembled

Put every piece from this chapter side by side and a full signature turns out to have at most five zones, always in the same order: required parameters, then defaulted ones, then *args, then keyword-only defaults, then **kwargs. Almost no real function uses all five — but every real function uses a subset, in that order. That is why the skill transfers: once the zones are familiar, an unfamiliar signature is just a form with some fields missing.

Zone 1 — required. def create_agent(model, tools, ...) opens with bare names. No grey ink, no stars: the caller must supply a model and a tool list or the clerk hands the form back with a TypeError. Required parameters always tell you what a function is fundamentally about.
Zone 2 — defaults. Next comes prompt=None, temperature=1.0, max_retries=3. Grey ink. Every one of these is a decision the author made for you, which you may override by name. And prompt=None is the mutable-default convention from the warning above: "none supplied, I will build a fresh one inside."
Zone 3 — the single star. A bare * or *args. If extras arrive by position, they land here as a tuple. A lone * with no name is the same fence with the box removed: nothing positional may pass, everything after it must be named.
Zone 4 and 5 — keyword-only, then the double star. Parameters after the star can only be filled by name — frameworks use this to force readable call sites. Finally **kwargs quietly accepts whatever named options the author did not anticipate, usually to pass them through to a lower layer.

Read signatures in that order — left to right, zone by zone — and the intimidation evaporates. The required names tell you the point of the function; the defaults tell you its opinions; the stars tell you how forgiving it is about extras. Three glances, and you know how to call code you have never seen before.

In the wild

Open any agent framework and the form-reading skill pays out immediately. The OpenAI and Anthropic Python clients are called almost exclusively with keyword arguments — client.chat.completions.create(model="gpt-4o", messages=msgs, temperature=0) — because self-documenting call sites survive code review. Config-driven code everywhere ends in a starred dict: create(**payload), where payload came out of a YAML or JSON file. And wrapper functions — retry helpers, logging shims, LangChain's layers over model clients — are built almost entirely from def wrapper(*args, **kwargs): accept anything, pass everything through unchanged. When you meet a signature that seems to accept nothing but stars, you are looking at a polite middleman, not magic.

Check yourself

1. In def book_ticket(passenger, seat="economy") called as book_ticket("Asha"), which is the parameter and which is the argument?

Parameters are the blanks printed on the form — they live in the def line. Arguments are the ink — the actual values a caller supplies. Error messages use both words precisely, so keeping them straight makes tracebacks readable.

2. Why does def f(a, b=1, c): refuse to even run?

All the grey ink lives at the end of the form. If c could sit defaultless after b=1, Python could not tell whether f(1, 2) meant to fill b or skip to c. It is a syntax error precisely to keep calls unambiguous.

3. Why do experienced authors write def add_task(task, queue=None) instead of queue=[]?

The default object is built one time, when the def line runs. Every defaulted call then sticks the same name on the same list — names on objects, chapter 2. Defaulting to None and creating the fresh list inside the body sidesteps the shared object entirely.

4. config = {"temperature": 0.2, "max_tokens": 500} — what does call_model("hi", **config) do?

Direction matters. Two stars in a def line collect extras into a dict; two stars at a call site do the reverse — explode a dict into individual named arguments. This is the config-file-to-function-call bridge you will see in every AI codebase.

5. A function body contains print(result) and no return statement. What does a caller receive?

The clerk shouted your details across the room and shredded the ticket. print is for humans watching a screen; return is how a value travels back into the program. A mysterious None where a result should be almost always means a missing return.

Key takeaways

  • A function is a named, reusable block: def defines it, the signature is its order form, return hands exactly one thing back through the bead curtain.
  • Parameters are the blanks on the form (in the def line); arguments are the ink a caller writes into them. Error messages use the two words precisely.
  • print shows, return gives back. A function without return returns None — the usual culprit when a mysterious None appears.
  • Defaults are grey ink: created once at def time (hence the mutable-default trap and the tools=None convention) and always grouped at the end of the parameter list.
  • Keyword arguments make call sites self-documenting; production code prefers retrain(data, test_split=0.2) over a bare row of positional values.
  • The stars work both directions: in a def line *args/**kwargs collect extras into a tuple/dict; at a call site */** unpack containers into arguments — the YAML-config-to-API-call bridge.
  • Docstrings are the laminated card — and in agent frameworks they double as the tool description the model reads when choosing tools, so a lazy docstring genuinely weakens an agent.
