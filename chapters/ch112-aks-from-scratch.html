AKS from scratch: your first cluster

Fourteen commands separate an empty Azure subscription from your agent answering requests inside a Kubernetes pod. This chapter is every one of those commands — what each does, what you should see, and why the order matters.

~35 minChapter 113 of 135
The story

You are opening a cloud kitchen. First you sign a lease on one unit in a food park — a named space where everything you build will live, and which you can hand back in one stroke when you are done. Then you stock a private warehouse with your ingredients; nobody cooks yet, but everything the kitchen will need is on a shelf with your name on it. Then the big call: a facilities company delivers a fully staffed kitchen — a shift manager who runs the floor, and two cooking stations ready for work. You hand the manager a keycard to your warehouse, because a kitchen that cannot fetch ingredients cooks nothing.

And here is the part that makes it a modern kitchen: you never shout instructions at the cooks. You pin one ticket on the board — "keep one biryani station running at all times" — and the manager makes it true, and keeps it true, even if a station catches fire at 2 a.m. Finally, before opening to the public, you taste the dish yourself through the service hatch.

Lease, warehouse, kitchen, keycard, ticket, taste. That is the exact sequence of this chapter — except the lease is a resource group, the warehouse is a container registry, and the kitchen is AKS.

Where you're standing

Quick recap of the last three chapters, because this one cashes them all in. In the Azure mental model you learned the hierarchy — a subscription holds resource groups, and resource groups hold resources — and you ran az login to prove the CLI could act on your behalf. In containers and images you packaged your agent with docker build: the agent's code, its Python, its dependencies, frozen into one shippable image. And in Kubernetes, plainly you met the cast — pods, nodes, Deployments, and the control plane that relentlessly makes reality match your declared wishes.

So far, though, it has all been theory and your laptop. Today you rent real machines in a real data centre and put your agent on them. AKS — Azure Kubernetes Service — is Kubernetes where Azure runs the control plane for you, so you get the whole machinery from chapter 112 without ever installing it. Everything below is copy-paste runnable in a sandbox subscription; that is precisely the point. Read it once for understanding, then run it for keeps.

Six words to keep straight before the commands start flying. Flip until they're boring:

Step 1: sign the lease — a resource group

Make sure you are logged in and pointed at your sandbox subscription, then create the group:

terminal — login and lease
az login                       # opens a browser; pick your sandbox account
az account show --output table # confirm WHICH subscription you're about to bill
az group create --name agent-rg --location centralindia

Line by line: az login gets the CLI a token, exactly as in chapter 110. az account show is the two-second habit that saves careers — confirm you are in the sandbox, not production, before creating anything. Then az group create takes just two things: a --name you'll type constantly (keep it short) and a --location, the Azure region whose data centre will host your resources — centralindia here; any region near you works. It returns almost instantly with a JSON receipt. The only line you care about:

expected output (trimmed)
{
  "id": "/subscriptions/…/resourceGroups/agent-rg",
  "location": "centralindia",
  "name": "agent-rg",
  "properties": { "provisioningState": "Succeeded" }
}

"provisioningState": "Succeeded" is Azure's way of saying "lease signed". Almost every az command replies with a JSON blob like this; you will learn to skim for that one field.

Step 2: stock the warehouse — registry, build, push

Your cluster-to-be will need to fetch the agent's image from somewhere it can reach. Your laptop is not that place — a private registry in the same resource group is:

terminal — create the registry
az acr create --resource-group agent-rg --name agentacr7421 --sku Basic

--resource-group agent-rg files it under your lease. --name is the one annoying flag: ACR names must be globally unique across all of Azure (they become a public DNS name), lowercase letters and digits only — so suffix yours with some digits and expect agentacr7421 to be taken by the time you read this. --sku Basic picks the cheapest tier, plenty for one image. In the JSON receipt, hunt for one field:

expected output (the field that matters)
{ "loginServer": "agentacr7421.azurecr.io", "provisioningState": "Succeeded" }

That loginServer is your registry's address, and it becomes part of your image's full name. Now the chapter 111 muscle memory — build, tag, push — with one new twist: you log in to your registry first.

terminal — build and push (recall ch110)
az acr login --name agentacr7421            # => Login Succeeded
docker build -t support-agent:v1 .          # bake the agent into an image
docker tag support-agent:v1 agentacr7421.azurecr.io/support-agent:v1
docker push agentacr7421.azurecr.io/support-agent:v1

Walking through it: az acr login wires your local Docker to the registry using your Azure identity — no separate password. docker build is unchanged from chapter 111: Dockerfile in, image out. docker tag gives the image a second name that starts with the registry's address — that prefix is literally how Docker knows where to push. And docker push uploads it layer by layer; you'll see a progress line per layer, ending with a v1: digest: sha256:… receipt. Your agent is now on the shelf, reachable from inside Azure.

Step 3: the big one — az aks create

Everything so far took seconds. This one is the real construction job:

terminal — commission the kitchen
az aks create \
  --resource-group agent-rg \
  --name agent-cluster \
  --node-count 2 \
  --generate-ssh-keys

Flag by flag: --resource-group and --name you know — the lease and the cluster's own name. --node-count 2 is the consequential one: it asks for two worker VMs in the default node pool. Two, not one, so that later chapters can show you pods rescheduling when a node dies. --generate-ssh-keys quietly creates SSH keys if your machine has none — a box-ticking requirement you'll almost never use.

What does this actually provision? Three things. A control plane — the API server, scheduler, and reconciliation brain from chapter 112 — which Azure hosts, patches, and (on the free tier) doesn't even charge you for. A node pool — two real VMs that will run your pods, and that do bill by the hour. And the plumbing between them: a virtual network, an identity for the cluster, monitoring hooks. Azure tucks the node VMs into a second, auto-created resource group named MC_agent-rg_agent-cluster_centralindia — don't touch that one; it's the cluster's engine room, managed for you.

Note

This command takes five to ten minutes. The terminal shows a spinner with Running… and it is genuinely working: creating VMs, wiring networks, booting Kubernetes. Make chai. When it finally returns a huge JSON blob, the field to skim for is the same old "provisioningState": "Succeeded".

Step 4: wire up kubectl — get-credentials

The cluster exists, but kubectl — your Kubernetes remote control from chapter 112 — has never heard of it. One command fixes that:

terminal — point kubectl at the cluster
az aks get-credentials --resource-group agent-rg --name agent-cluster
# => Merged "agent-cluster" as current context in /Users/you/.kube/config

kubectl get nodes

az aks get-credentials downloads the cluster's address and access credentials and merges them into your kubeconfig — the file at ~/.kube/config where kubectl keeps a phone book of every cluster you can talk to. "Merged … as current context" means: from now on, plain kubectl commands go to this cluster. Then kubectl get nodes is the handshake — your first real Kubernetes command against your own cluster:

expected output
NAME                                STATUS   ROLES   AGE     VERSION
aks-nodepool1-24816504-vmss000000   Ready    agent   3m12s   v1.32.6
aks-nodepool1-24816504-vmss000001   Ready    agent   3m18s   v1.32.6

Two nodes, both Ready. (Your random suffixes and version numbers will differ — that's fine.) Pause on this moment: that command left your laptop, crossed the internet to a control plane Azure runs in a data centre, and came back with the state of two machines you now rent. The kitchen is real.

Step 5: the keycard — --attach-acr

Here is the step everyone forgets once and never again. Your cluster and your registry live in the same resource group, made by the same you — and still, by default, the cluster is not allowed to pull your images. Azure trusts nothing implicitly. Grant it:

terminal — let the cluster pull from your registry
az aks update --resource-group agent-rg --name agent-cluster \
  --attach-acr agentacr7421

az aks update modifies an existing cluster; --attach-acr names the registry to trust. Under the hood it grants the cluster's identity the AcrPull role on your registry — permission to read images, nothing more. If you skip this, everything ahead still appears to work: the manifest applies cleanly, the pod is scheduled — and then it sticks forever in ImagePullBackOff, the pod state that means "the node tried to fetch the image and was refused or found nothing". You met that state in chapter 112; this flag is its most common cure on AKS.

Step 6: the order ticket — your first manifest

Now the Kubernetes half begins, and the style of conversation changes. With az you gave orders; with Kubernetes you declare desired state in a YAML file and let the control plane do the ordering. Here is the smallest useful ticket — save it as deploy.yaml. Click through the annotated lines:

deploy.yaml — "keep one copy of my agent running"
apiVersion: apps/v1
Which dialect of the Kubernetes API this object speaks. Deployments live in the apps/v1 group — this line is boilerplate you will type a thousand times and question never.
kind: Deployment
What we are declaring. A Deployment is the "keep N copies running" object from chapter 112 — not the pod itself, but the standing order that pods should exist.
metadata:
name: support-agent
The Deployment's name — how you will refer to it in every kubectl command from now on.
spec:
replicas: 1
How many identical pods to keep alive. One, for now. Scaling to three later is changing this number and re-applying — that is the entire procedure.
selector:
matchLabels:
app: support-agent
How the Deployment recognises which pods are "its" pods: by label. This MUST match the labels in the template below — if they differ, Kubernetes rejects the manifest outright. The classic first-day YAML error.
template:
Everything under here is the recipe for each pod the Deployment stamps out — a pod template, cookie-cutter style.
metadata:
labels:
app: support-agent
The label each stamped pod wears — the other half of the selector handshake above. Same string, or nothing works.
spec:
containers:
- name: agent
A pod can hold several containers (that dash starts a list) — sidecars come in the next chapter. Today: one.
image: agentacr7421.azurecr.io/support-agent:v1
The full image name you pushed in step 2 — registry address included, so the node knows exactly which warehouse shelf to pull from. A typo here is the other classic route to ImagePullBackOff.
ports:
- containerPort: 8000
Documents the port your agent listens on inside the container. Port-forwarding and Services (the next chapter) will aim at this.

Pin the ticket to the board:

terminal — apply the manifest
kubectl apply -f deploy.yaml
# => deployment.apps/support-agent created

apply means "make reality match this file"; -f points at the file. The response is instant — but read it carefully: it says created, not running. You have registered a wish. The control plane is only now springing into action.

Step 7: watch the pod come alive

This is the best free show in DevOps. The -w flag means watch — the command keeps streaming status changes instead of returning:

terminal — watch the status board
kubectl get pods -w

NAME                             READY   STATUS              RESTARTS   AGE
support-agent-6d5f7c9b4-qx2lp    0/1     Pending             0          0s
support-agent-6d5f7c9b4-qx2lp    0/1     ContainerCreating   0          1s
support-agent-6d5f7c9b4-qx2lp    1/1     Running             0          14s

Three states, three phases of the machinery. Pending: the pod exists only as a record in the control plane; the scheduler is choosing which of your two nodes has room. ContainerCreating: a node has been chosen and is pulling support-agent:v1 from your registry — the step that --attach-acr made possible — then starting the container. Running: your agent's process is alive on a VM in a data centre, READY 1/1. The whole journey usually takes seconds (the first pull is the slow part). Press Ctrl+C to stop watching — the pod keeps running; you were only observing.

Tip

The two failures you just simulated are the two you will actually meet this week. A pod parked in Pending means no node has room — count your nodes and their sizes. ImagePullBackOff means the image could not be fetched — check the image name and tag in the manifest, then check the registry attachment. kubectl describe pod <name> tells you which story you're in; that command gets the full treatment in chapter 115.

Step 8: taste through the hatch — port-forward

Your agent is running, but nothing outside the cluster can reach it yet — no public address, no open door. Deliberate and good: exposure should be a decision, not a default. For a private taste, kubectl can tunnel a port from your laptop straight to the pod:

terminal — private tunnel to the pod
kubectl port-forward deployment/support-agent 8000:8000
# => Forwarding from 127.0.0.1:8000 -> 8000

# in a SECOND terminal:
curl http://localhost:8000/health
# => {"status": "ok", "agent": "support-agent", "version": "v1"}

Read the port pair as local:remote — connections to port 8000 on your laptop are piped into port 8000 of the pod. The command occupies its terminal while the tunnel is open, so the curl happens in a second one. And that unremarkable JSON reply deserves a moment of ceremony: your laptop just spoke, through an encrypted tunnel, to your agent, in a container, in a pod, on a VM, in a cluster you built from nothing with fourteen commands.

Be clear about what port-forward is: a debugging hatch, alive only while the command runs, reaching only your laptop. Giving the agent a real, stable front door — a Service, and the rest of production wiring — is exactly where the next chapter picks up.

The whole sequence, on one page

Here is the complete ritual, start to finish. This is the block to keep; every line is annotated above. Run it in a sandbox subscription — it was written to be run, not admired:

from nothing to a running agent — the full sequence
az login
az group create --name agent-rg --location centralindia
az acr create --resource-group agent-rg --name agentacr7421 --sku Basic

Careful

Those two nodes are real VMs billing real money by the hour, whether or not pods are running on them. When you finish experimenting, return the lease — one command deletes the cluster, the registry, and everything else in the group: az group delete --name agent-rg --yes --no-wait. This is why step 1 put everything in one resource group. Make it a reflex in sandboxes: build, learn, delete.

In the wild

You will rarely see a team member type az aks create by hand twice — real teams freeze this exact sequence into infrastructure-as-code (Terraform or Bicep files in the repo), and the CI pipeline replays it. But read one of those files and you will recognise every field: resource group, registry, node count, the ACR attachment — the same ten commands wearing a suit. The kubectl half never goes away, though: kubectl get pods is the first command typed in virtually every production incident, and "it's stuck in ImagePullBackOff" is a sentence you will hear in standups for the rest of your career. As of the current AKS docs this command set is stable; if a flag ever moans at you, az aks create --help shows the current spelling.

Check yourself

1. What does az aks get-credentials actually do?

It merges an entry into your kubeconfig and sets it as the current context — from then on, plain kubectl commands go to that cluster. (Option c is what --attach-acr does.)

2. What did --node-count 2 buy you, exactly?

Node count is worker VMs only. Azure hosts the control plane for you, and pod copies are set by replicas: in the Deployment — three different dials.

3. You ran this chapter's sequence but skipped one command, and now the pod sits in ImagePullBackOff. Which command did you most likely skip?

ImagePullBackOff means the node tried to fetch the image and failed — on a fresh AKS-plus-ACR setup, a missing registry attachment (or a typo in the image name) is almost always the cause. kubectl get nodes only reads state, and without the resource group nothing else would exist at all.

4. In the Deployment manifest, what must selector.matchLabels agree with?

The selector is how the Deployment recognises its own pods, and the template's labels are what the pods wear. Kubernetes refuses the manifest if the two halves of that handshake don't match — a rite-of-passage YAML error.

5. Why is kubectl port-forward not how real users will reach your agent?

Port-forward pipes one local port to the pod for as long as you keep the command running — perfect for a private taste-test. Real exposure uses a Service (and friends), which is where the next chapter begins.

Key takeaways

  • The from-zero ritual is short and ordered: az group create → az acr create → build and push → az aks create --node-count 2 → az aks get-credentials → az aks update --attach-acr → kubectl apply -f → watch → port-forward.
  • az aks create provisions a control plane (hosted by Azure, invisible to you) plus a node pool of real, hourly-billed VMs — and it legitimately takes minutes.
  • get-credentials writes the cluster into your kubeconfig; --attach-acr grants it pull rights on your registry. Forget the latter and ImagePullBackOff will remind you.
  • The az half of the sequence is imperative (do this now); the kubectl half is declarative — you apply a Deployment stating desired state, and the control plane makes it true and keeps it true.
  • Pending → ContainerCreating → Running is the healthy pod journey: scheduler finds a node, node pulls your image, container starts. Each stuck state points at a specific culprit.
  • Everything lives in one resource group on purpose: az group delete --name agent-rg returns the lease and stops the meter. In sandboxes: build, learn, delete.
