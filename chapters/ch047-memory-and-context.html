Memory and the context problem

Your agent's memory is a Python list you keep appending to. This chapter is about the day that list outgrows the model's world — and the three tricks every harness uses to survive it.

~30 minChapter 48 of 135
The story

At eight every evening, the day doctor on a busy hospital ward hands over to the night doctor. The night doctor was not there for anything that happened earlier — she has no memory of the morning rounds, the test results, the family's anxious questions. Everything she will ever know about a patient arrives on one sheet of paper: the handover sheet. Early in a patient's stay this works beautifully. But by week three the notes would fill a novel, and the sheet is still one page. So the ward has rewriting rules: today's vitals stay word for word, the last three days stay detailed, and everything older is squeezed into two lines at the top — "Admitted 14 May, appendix removed, recovery normal until Tuesday." Whatever the rewrite drops, the night doctor simply does not know. As far as she is concerned, it never happened.

Your agent lives on this ward. The model is a brilliant doctor with total amnesia between shifts. The conversation history is the handover sheet. The context window is the size of the paper. And the rewriting rules — what to keep verbatim, what to squeeze, what to drop — are what this chapter teaches you to write.

The transcript is the world

In the ReAct chapter you built a complete agent: a history list that started with the task, a loop that sent the whole list to the model every turn, and two append calls — one for whatever the model said, one for each tool observation. That list was the agent's entire memory. Nothing else existed. If a fact was in the list, the agent "knew" it; if it was not, the agent had never heard of it.

The reason is a property worth naming properly. An LLM is stateless — between one API call and the next, it retains absolutely nothing, like our night doctor between shifts. There is no hidden account where the provider stores "what this model has discussed with you". Every call starts from a blank brain plus whatever messages you send. So when an operator says "the agent remembered the client's budget", the mechanical truth is: the budget was still present in the messages that got sent. When they say "the agent forgot", the truth is equally mechanical: it was not.

the whole memory system of ch045, three lines
history = [f"TASK: {task}"]      # the world begins
reply = fake_llm(history)        # the ENTIRE world is re-sent, every turn
history.append(observation)      # the world grows, one append at a time

Hold on to that mental picture, because everything in this chapter — and half of Act X — is downstream of it. There is no memory. There is only a transcript, and the discipline of deciding what stays in it.

It grows every turn — and you pay for all of it, again

Now do the arithmetic that every agent operator eventually does on the back of an invoice. Each turn of the loop adds a model reply and an observation — say 500 tokens together (a token, remember, is roughly three-quarters of a word). But the model is stateless, so each new call must re-send everything so far. Turn one sends 500 tokens. Turn ten sends about 5,000. Turn forty sends about 20,000 — per call. And your bill is the sum of all those calls, which grows roughly with the square of the number of turns. A conversation ten times longer does not cost ten times more in input tokens; it costs closer to a hundred times more. This is the mechanism behind the harmless-looking loops that became bills in the token-economics chapter.

Providers soften the blow with prompt caching — the unchanged prefix of your messages is recognised and billed at a discount — but caching only makes re-sending cheaper. It does not make the transcript smaller. And the transcript's size has a hard ceiling of its own.

The wall: when the world is full

You met the context window in chapter 30 — the maximum number of tokens a model can attend to in a single call. One refinement matters enormously for agents: the window covers the input and the output together. If the window is 200,000 tokens and your transcript occupies 199,500, the model has 500 tokens of room left in which to write its entire reply. The window is not a mailbox for your messages; it is the whole desk, and the model needs space on that desk to work.

Two things go wrong as you approach the wall, and only one of them is polite enough to announce itself. The loud one: exceed the window and the API simply rejects the call with a "prompt is too long" style error — mid-task, your agent falls over. The quiet one: well before the hard limit, models get measurably worse at using what they were given. Facts buried in the middle of a very long context get recalled less reliably than facts near the start or the end — researchers named the effect "lost in the middle". So a transcript at 95% of the window is not a triumph of thrift; it is an agent that is both about to crash and already mildly concussed. Serious harnesses act long before 100%.

Careful

Compaction failures are the sneakiest, because nothing errors. The agent carries on fluently — minus the one constraint that got summarised away ("keep it under 2,000 rupees", "never email the client directly"). If an agent's behaviour degrades specifically after a long session, suspect the trim before you suspect the model. The next chapter, How agents fail, files this under its rogues' gallery.

The trim, line by line

The summarize_middle function is the smallest honest compaction there is, and it is worth reading the way a reviewer would — because every harness you operate contains a bigger sibling of it.

summarize_middle — compaction in seven lines
def summarize_middle(msgs, budget=WINDOW):
Message list in, smaller message list out. Every harness owns a function of exactly this shape; Claude Code calls its version compaction and runs it automatically when the window gets tight.
if total(msgs) <= budget:
The cheap check first: under budget, do nothing at all. Trimming a healthy transcript throws away information for no benefit — good harnesses trim as late as safely possible.
return msgs
head, tail = msgs[:2], msgs[-4:]
This is pinning, done with two list slices. The first two messages (system prompt + original task) always survive; the four newest stay verbatim, because "recent" is usually "relevant". Note the design decision hiding in plain sight: who survives is hard-coded here.
dropped = msgs[2:-4]
The middle — everything that is neither pinned nor recent — is what gets sacrificed. In a long run this slice can be 90% of the transcript.
note = {"role": "user", "content": ...}
In production, this line is an LLM call: "summarise these messages in a few sentences". Our canned string keeps the demo browser-safe, but the trade is identical — the summary is lossy. Whatever it fails to mention has ceased to exist.
return head + [note] + tail
The new, shorter world, stitched together with list concatenation. Next turn, the model reads this and cannot tell anything was ever removed. Run the function again later and the old summary itself lands in the middle and gets re-summarised — summaries of summaries, fading like a photocopy of a photocopy.

Token budgeting: spend the window like money

Once you accept that the window is finite, the professional move is to budget it — deliberately, the way you would a purse. Because the transcript is not the only tenant in there. The system prompt pays rent on every single call. Every tool you registered in the tools chapter ships its name, description, and schema along too — a fat toolbox can quietly eat thousands of tokens per call before the conversation says a word. If the agent does retrieval (the RAG pattern), the fetched documents land in the window as well. And you must leave headroom for the model's own reply — the max_tokens you set on the call has to fit inside whatever is left.

A budgeting harness does three things on every turn. It counts before sending — even the crude four-characters-per-token estimate from the runner above catches most overflows, and providers offer exact token-counting endpoints when you need precision. It trims to budget, not to the hard limit — triggering compaction at, say, 80% of the window, so there is never a turn where the call simply fails. And it reserves output headroom first, because a reply that gets cut off mid-sentence is a failure the user sees, while a slightly shorter history is a failure nobody notices.

What trimming cannot do

Be clear-eyed about what you have learned: every trick in this chapter is a way of forgetting gracefully. Truncation forgets bluntly, the sliding window forgets the middle, compaction forgets the details — but they all forget. None of them gives the agent what an operator really wants: memory that survives the end of the session, knowledge shared across conversations, the ability to recall a fact from three weeks ago without carrying three weeks of transcript.

That requires a different move entirely: taking information out of the transcript and putting it somewhere durable — a file, a database, a vector store — then pulling back only the relevant slice when it is needed. You have already seen a hint of it: deep-agent frameworks give the model file tools precisely so it can write big findings to a scratch file instead of hauling them around in context, and some offer a persistent memories area that outlives the session. Designing that flow — what goes out, what comes back, and when — is a discipline with its own name, context engineering, and it gets a full treatment in Act X: context engineering and long-term memory. When you get there, remember this chapter — everything in Act X is a response to the wall you just hit.

In the wild

You have already met every idea in this chapter, wearing production clothes. When Claude Code announces it is compacting the conversation (or you run its compact command yourself), that is summarise-the-middle, live. When a chat UI says "this conversation is too long, please start a new one", that is the window wall with a polite face. LangChain ships a message-trimming helper for exactly the sliding-window move; LangGraph's checkpointers (InMemorySaver and friends) persist a thread's state so it survives a restart — note the distinction: persistence saves the transcript, it does not shrink it. And when a teammate reports "the agent forgot what we agreed an hour ago", you now know the diagnostic question to ask first: what is your trim strategy, and what did it eat?

Check yourself

1. An agent followed an instruction perfectly for an hour, then abruptly stopped honouring it. The most likely mechanical cause?

Models are stateless and their weights never change during use. If behaviour changes after a long session, the messages changed — a trim, a slide, or a compaction removed or blurred the instruction. Check what the transcript actually contained on the failing turn.

2. Each turn re-sends the whole history. If every turn adds roughly the same number of tokens, the total input tokens billed across a run grow…

Turn N re-sends everything from turns 1 to N-1, so the per-call cost climbs linearly and the run total is the sum of a growing series — roughly quadratic. Prompt caching discounts the repeats but does not change the shape, and the window limit still arrives on schedule.

3. A sliding window keeps only the most recent N messages. What is the classic danger, and the standard fix?

The oldest messages are the first to go, and the system prompt and original brief are the oldest of all. Pinning — always keeping the head of the transcript regardless of age — is why our trim function protected msgs[:2] before anything else.

4. Why do serious harnesses compact at around 80% of the window instead of waiting until it is completely full?

The window covers input and output together — a transcript at 99% leaves the model almost no room to answer, and the call can fail outright mid-task. On top of that, recall of facts buried in very long contexts gets worse ("lost in the middle"), so running at the edge hurts quality before it hurts availability.

5. Compaction never throws an error, yet it is the sneakiest of the three tricks. Why?

A summary is an opinion about what mattered. Anything it omits — a constraint, a number, a "never do X" — ceases to exist for the stateless model, which keeps working confidently on the blurred record. Nothing crashes; the behaviour just quietly bends.

Key takeaways

  • The model is stateless: its entire world is the message list you send on each call. "Remembering" means "still in the transcript"; "forgetting" means "no longer there".
  • History grows every turn and is re-sent every turn — per-call cost climbs linearly and the run total roughly with the square of the turn count. Caching discounts the repeats; it does not shrink the transcript.
  • The context window is a hard ceiling shared by input and output, and quality sags before you hit it — so harnesses trim early, at a budget threshold, not at the wall.
  • There are three honest trims: truncation (chop oldest), sliding window (keep recent + pinned head), and compaction (summarise the middle). Each forgets differently, and each has a signature failure you can now recognise.
  • Budget the window like money: system prompt, tool schemas, retrieved documents, history, and reserved reply headroom all share the same fixed purse.
  • Every trick here is graceful forgetting. Real memory — surviving across sessions — means moving information out of the transcript into files and stores, which is Act X's whole subject.
