What makes an agent 'deep'

Planning, a file system, subagents, and a detailed prompt — four upgrades that turn the fifty-line loop you built in Act V into a worker that stays sharp for hours.

~25 minChapter 67 of 133
The story

Your cousin Rohan once volunteered to run his sister's wedding single-handed. No notebook — he was proud of his memory. Guest list, caterer quotes, the florist's number, the priest's availability, which uncle is currently not speaking to which aunt: all of it in his head. Day one went brilliantly. By day ten, every new decision required him to mentally replay everything that had happened so far, and the things that mattered — the venue deposit deadline — were being shouted down by trivia about napkin colours. By week three he had phoned the same caterer twice with the same question and missed the deposit entirely.

The professional planner the family hired to rescue him had, measurably, no better memory than Rohan. What she had was a method. A whiteboard with the plan, rewritten every morning. A binder where every quote and contract was filed the moment it arrived — out of her head, retrievable in seconds. Two assistants who each spent days chasing venues and vendors and came back with a one-page shortlist apiece. And a dog-eared company playbook that said exactly how weddings are run here.

Same brain. Four habits. That is the entire difference between a shallow agent and a deep one — hold the picture, because this chapter is just naming its parts.

Where we stand: one loop, one transcript

Quick recap, because everything here builds on it. In Build a ReAct loop by hand you built the atom of all agents: an LLM, a dict of tools, and a loop that alternates think, act, observe until the model declares it is done. Every tool result gets appended to the history, and the entire history is re-sent to the model on every turn — because, as the memory chapter hammered home, the model itself remembers nothing between calls. LangChain's create_agent packaged that loop with nicer plumbing, and in the last chapter you read a full LangGraph app and found the same heartbeat drawn as a graph. The loop never changed shape; only the wrapping did.

This act calls that style of agent shallow — a single loop that keeps everything it knows in one ever-growing transcript. Shallow is not an insult. For a task that takes five tool calls — look up the invoice, compute the GST, draft the email — shallow is exactly right, and anything fancier is pure overhead. The trouble begins when the task stops fitting inside one sitting.

The long-horizon problem: shallow loops drown

A long-horizon task is one that needs dozens or hundreds of steps over minutes or hours — "research these five competitors and write a market brief", "migrate this codebase", "plan the wedding". Point a shallow loop at one and watch its context.

Every web page fetched, every file read, every command's output lands in the history verbatim, forever. A single search result can be thousands of tokens (tokens and context windows, if you want the refresher), and after forty tool calls the transcript is enormous. Three things now go wrong at once.

It gets expensive. The whole transcript is re-sent on every turn, so the total tokens billed grow far faster than the number of steps — the meter is always running, and Act VIII has a whole chapter on the resulting bill. It gets dumber. Even models with huge context windows attend less reliably to a crowded transcript; the goal, stated once at the top, is now buried under a landslide of raw tool output. Practitioners call this decay context rot — the model still technically has the information but can no longer find and weight what matters. It fails in familiar ways. Repeating tool calls it already made, forgetting constraints, drifting off-goal — the greatest hits from how agents fail, every one of them more likely as the transcript bloats. Rohan, week three, is a shallow loop at turn forty.

Note

A bigger context window does not solve this. A million-token window makes the drowning slower and pricier, not smarter. The fix is not a bigger head — it is the whiteboard, the binder, the assistants, and the playbook.

Four additions, four habits

A deep agent is the same ReAct loop upgraded with four things that let it work far beyond one comfortable context: explicit planning, a file system, subagents, and a detailed system prompt. The name comes from the deepagents library — which the next chapter dissects line by line — and the pattern itself was distilled from the applications that made long-running agents real: coding agents like Claude Code, and the Deep Research features in the major chat products. None of the four is exotic. Each one maps to a habit of the wedding planner, and each one exists to keep the main context small and the goal fresh. Flip the cards until the vocabulary feels like yours:

Pillar one: a plan the agent writes for itself

In deepagents the planning tool is called write_todos, and here is the surprise: mechanically, it barely does anything. It stores a list and replies "ok". No scheduler, no executor, no enforcement — nothing runs because an item is on the list. Its power is attentional. Every time the agent writes or rewrites its to-do list, the current plan lands in the most recent part of the transcript, exactly where the model attends best. Fifty turns in, the goal is not a faint memory from the top of a huge transcript; it was restated twelve messages ago, with item three marked in-progress.

You have watched this happen. When Claude Code prints a neat little task list before touching your code and ticks items off as it goes, that display is not for you — or not only for you. It is the agent reciting its plan back into its own context so it cannot lose the thread. A note-to-self that happens to be visible. Chapter 69 takes this tool apart properly.

Pillar two: a file system to think outside its own head

The second addition gives the agent file tools — in deepagents: ls, read_file, write_file, and edit_file. By default these files are virtual: they live in the agent's state object, not on your disk, so the agent can scribble freely with zero blast radius. (Real-disk and database-backed variants exist; chapter 70 covers the backends.)

What matters is the move the tools enable: offloading — moving bulky information out of the transcript into storage, leaving only a pointer behind. When a search returns fourteen thousand tokens of raw text, a shallow loop swallows it into the history forever. A deep agent writes it to research_notes.md, and the observation that stays in context is one line: "wrote research_notes.md (14,203 chars)". Nothing is lost — it is filed. Later the agent reads back only the section it needs, only when it needs it. The transcript carries pointers, not payloads, and that single habit is most of why the deep bar in the chart above stays low.

Pillar three: subagents — hire help, quarantine the mess

Some jobs are messy no matter how well you file. Researching one competitor might take twenty searches and sixty thousand tokens of reading, and the final useful yield is a paragraph. Do that inside the main loop and the mess of getting there poisons the main context for the rest of the run.

So the deep agent gets one more tool — in deepagents it is called task — that spawns a subagent: a brand-new agent loop with its own empty history, its own tools, and exactly one job. The subagent burns whatever tokens the job takes, privately. When it finishes, every page it read and every dead end it hit is thrown away, and precisely one thing crosses back to the parent: the final summary. That isolation is called context quarantine, and it is the whole trick — the mess dies with the messenger; the insight survives. As a bonus, several subagents can run in parallel: five competitors, five researchers, one afternoon. You met the multi-agent idea in subgraphs and multi-agent patterns; the deep-agent version is that same idea put to work purely for context hygiene. Chapter 71 dissects it.

Pillar four: the playbook — a system prompt that reads like a manual

The quietest pillar, and the one people underrate. The toy agents of Act V ran on three lines of system prompt. The agents that work for hours run on thousands of words: what the agent is, how to judge when a task deserves a written plan, worked examples for every tool, how to structure output, what to do when stuck, what never to do. Claude Code's system prompt is famously long for exactly this reason. The other three pillars are just tools sitting in a dict — they create the possibility of deep behaviour. The prompt is what teaches the model to actually plan first, file early, and delegate the messy parts. Hand a novice the whiteboard, the binder, and two assistants, and you get chaos with better furniture. The playbook is what makes it a system.

Tip

Now sabotage it — that is where the learning is. Make the subagent lazy: have run_subagent return the raw web_search output instead of the summary, run it again, and watch the main-context number explode on the next turn. Then delete the write_todos step from the script and notice the run still works — planning changes what the model attends to, and a scripted model has no attention to lose. That difference is exactly why pillar four matters for real models.

Keep that last row honest in your head: deep is not better, it is longer. For a five-step task, the planning and filing ceremony is pure waste — the "start simple, add complexity only when it earns its keep" rule from building effective agents applies with full force. Reach for depth when the horizon demands it, not because it sounds impressive.

You have been supervising deep agents for months

Here is the pleasant twist: none of this is new to you as an operator. When Claude Code opens your request by printing a task list and ticking items off one by one — that is pillar one, a planning tool reciting the plan back into context. When it announces it is sending a subtask off to an agent and returns with a tidy summary of what the explorer found — pillar three, context quarantine, live on your screen. When it writes scratch notes and intermediate files instead of holding everything in the conversation — pillar two. And the reason it behaves this consistently on a million different machines is pillar four, the playbook-length system prompt you never see.

Same story with the Deep Research features in the big chat products: you ask one question, the agent visibly writes a plan, fans out to read dozens of sources over several minutes — each fan-out effectively a quarantined sub-task — and comes back with a cited report. That entire product category exists because a shallow loop physically cannot do the job. You have been supervising the four pillars all along; now you can name them. The next chapter shows you the code that implements them, and chapter 73 closes the circle by comparing the library to the product you use daily.

In the wild

In the deepagents library, all four pillars arrive in a single call: from deepagents import create_deep_agent gives you write_todos, the virtual file tools (ls, read_file, write_file, edit_file), a task tool for subagents, and a long built-in system prompt. In agent logs, look for a planning-tool call near the start of every long run and task(...) calls wherever the work gets messy. And in merge requests the pillars appear as fixes: "moved report generation into a subagent to keep the main context small", "write scraped pages to files instead of returning them inline", "expanded the system prompt with tool-usage examples". A team writing those sentences is practising context engineering — the discipline this whole act teaches.

Check yourself

1. Why does a shallow ReAct loop fail on an hours-long task, even with a huge context window?

The transcript is re-sent on every turn and grows with every tool result — spending balloons and the goal drowns under raw output. A bigger window makes the drowning slower and pricier, not smarter.

2. Mechanically, what does the write_todos planning tool actually do?

There is no executor behind it. The plan is a note-to-self: every rewrite places the current goals in the freshest part of the transcript, exactly where the model attends best.

3. A subagent spends 60,000 tokens researching and returns a one-paragraph summary. What happens to the 60,000 tokens of context it built up?

That disposal is the whole point — context quarantine. The mess of doing the job dies with the subagent's context; the parent pays only for the distilled answer.

4. What is the deep agent's file system primarily for?

Files turn payloads into pointers: a 14,000-token search result becomes a one-line "wrote notes.md" observation, and the agent reads back only the part it needs, when it needs it.

5. Your task takes five tool calls: fetch an invoice, compute tax, draft an email. Which architecture should you reach for?

Deep is not better, it is longer. On a five-step task the plan, files, and subagents add cost and latency without preventing any failure — the shallow loop remains exactly the right tool.

Key takeaways

  • A deep agent is the same ReAct loop you built in chapter 46, plus four additions: explicit planning, a file system, subagents, and a detailed system prompt.
  • Shallow loops drown on long-horizon tasks because every tool result lives in the transcript forever — cost balloons and context rot sets in. Bigger windows do not fix it.
  • Planning tools like write_todos execute nothing; rewriting the plan keeps the goal in recent, high-attention context — the whiteboard.
  • File tools turn payloads into pointers: bulky results are filed and read back on demand instead of being re-sent every turn — the binder.
  • Subagents are context quarantine: they burn tokens in a disposable private context and return only a summary — the assistants.
  • The playbook-length system prompt is what makes the model actually use the other three pillars — tools create possibility; the prompt creates behaviour.
  • Depth is for long horizons only. For a five-step task, shallow remains the right answer — and you have been watching all four pillars at work every day in Claude Code and Deep Research.
