Agent evals

An agent that aced every demo can still be quietly getting worse every week. Evals are how you find out before your users do — datasets of tasks, graders, trajectory scores, an LLM examiner with a printed rubric, and a CI gate that blocks the merge.

~30 minChapter 105 of 135
The story

The driving examiner slides into the passenger seat carrying two clipboards. The first has a single line: did the car arrive at the finish point? The second is dense: mirror checks, signalling, speed control, that dreaded parallel park by the market. Forty minutes later you arrive — destination reached, clipboard one says pass. But clipboard two tells the real story: you rolled through a stop sign, and you circled the same roundabout twice because you missed your exit. The examiner fails you, politely. Arriving was never the point; arriving reliably was, and the drive says you got lucky.

Now notice the rest of the exam. Every candidate drives the same standard routes, so results compare fairly across days and examiners. The examiner scores against a printed rubric, not a mood. And the transport office re-runs those routes on every new batch — if this month's cohort fails at twice the usual rate, somebody inspects the driving school. Hold that picture: it is exactly how serious teams measure whether an agent actually works.

Tests pass. Is the agent any good?

In the last chapter you learned why tests gate merges: a test feeds a function a known input, asserts the exact expected output, and the pipeline turns red or green with no room for debate. That works because ordinary code is deterministic — same input, same output, every single time. Pytest's whole edifice rests on that promise.

Agents break the promise. You know from the chapter on LLMs versus deterministic software that a model is a probability machine, and from the sampling chapter that even at low temperature its wording wobbles. Ask your invoice agent the same question three times and you might get "GST comes to 360 rupees", "That works out to Rs 360", and "The GST amount is 360.00" — three phrasings, all correct, and assert answer == "GST comes to 360 rupees" fails on two of them. Worse, the failure can run the other way: you tweak one line of the system prompt, your single happy-path test stays green, and the agent quietly gets worse at thirty other tasks nobody was checking.

So the industry built a different instrument. An eval is a test for a nondeterministic system: instead of one input and one exact assertion, you run the agent across a whole dataset of tasks, score each result with graders that check meaning rather than exact strings, and track the rate — 84 per cent of tasks passed, up from 79 last week. One red test tells you a function broke. A falling pass rate tells you your agent is degrading. Different instrument, same job: catching regressions before they ship.

The anatomy of an eval

Every eval system — from a fifty-line script to an enterprise platform — has the same three organs:

A dataset. A collection of representative tasks, each paired with what "right" looks like: a reference answer, facts that must appear, or a rule to check. Teams call a trusted, human-reviewed dataset their golden dataset — the standard routes of the driving exam. It lives in the repo as a JSONL or CSV file, versioned like any other code, and it grows every time production surprises you: yesterday's incident becomes tomorrow's eval case.

A runner. Code that executes the agent on every task and captures everything — not just the final answer but every tool call and intermediate step. If you set up tracing in the observability chapter, this will feel familiar: the runner replays your agent under a microscope.

Graders. Functions that look at what the agent did and produce a score. A grader can be dumb and cheap (does the answer contain the number 360?), clever and cheap (extract all numbers, compare within a tolerance), or clever and expensive (ask another LLM to judge against a rubric — more on that shortly). One task can have several graders — the examiner's two clipboards.

Destination versus the drive: final-answer and trajectory evals

The first clipboard is the final-answer eval: did the agent produce the right result? For many tasks it is enough. But you learned in the ReAct chapter that an agent is a loop, and the history list that loop builds — thought, action, observation, repeated — is a complete record of how the answer was reached. That record is the trajectory, and grading it is the second clipboard: a trajectory eval asks whether the agent took sane steps, regardless of where it ended up.

Why grade the drive when the destination was right? Three reasons that show up in real merge requests:

Luck rots. An agent that answers a legal question correctly without ever calling the retrieval tool answered from its training memory. Today that memory happened to be right. Next week, on a slightly different clause, it will confidently be wrong. The final-answer grader says pass; only the trajectory grader sees the skipped lookup and raises the flag.

Cost compounds. An agent that takes forty steps to finish a three-step task passes the answer check and fails your budget. You saw in the token bill chapter that every loop turn re-sends the whole history — a step-count grader is a cost guardrail wearing an eval costume.

Safety is a process property. "Never call the payments tool for a read-only question", "never issue the same tool call twice in a row" (the classic stuck-loop smell), "always cite the document you retrieved" — none of these are visible in the final answer. All of them are one-line checks against the trajectory.

Note

Where do trajectories come from in production? From your tracing setup — the spans and tool-call logs from the observability chapter. Teams harvest production traces, scrub them, and promote them into the golden dataset. Observability feeds evals; evals justify observability.

The graders, line by line

The runner is the ch45 loop you already own. The graders and the gate are the new muscles — read them the way a reviewer would:

graders and gate — the heart of the eval
def grade_final(case, answer):
A grader is just a function: the task's expectations plus the agent's output go in, a verdict comes out. Nothing mystical — every eval platform ultimately lets you write plain Python here.
nums = [float(n) for n in re.findall(r"\d+(?:\.\d+)?", answer)]
Pull every number out of the answer text with a regex, using a comprehension. This is meaning-aware grading on a budget: we do not care whether the agent said "Rs 360", "360.00", or "GST comes to 360 rupees".
return any(abs(n - case["expected"]) < 0.01 for n in nums)
Numeric tolerance instead of string equality. A naive "does the text contain 300" check would pass the wrong answer "3000 rupees". Graders are code; they need the same care as the agent.

A threshold, not perfection. Demanding 100 per cent from a nondeterministic system makes every pipeline red and teaches everyone to ignore the gate. Pick a bar, hold it, raise it deliberately.

LLM-as-judge: hiring an examiner made of the same stuff

Numbers are easy to grade with code. But suppose your agent drafts a polite payment-reminder email, summarises a rental agreement, or explains a refund decision. There is no single correct string, no number to extract — a regex cannot tell a good apology from a passive-aggressive one. For prose, teams use LLM-as-judge: a second model call whose only job is to grade the first one's output against an explicit rubric.

The rubric is the whole trick. A judge asked "is this answer good?" returns vibes. A judge given binary criteria, a reference answer, and a required output format returns something you can aggregate:

judge_prompt.py — a rubric-driven judge
JUDGE_PROMPT = """You are grading an AI assistant's reply.

Score each criterion 0 or 1. Be strict.
1. correctness: every figure and date matches the reference
2. grounding: no claims beyond the provided context
3. tone: plain, polite, no jargon
4. action: ends with one clear next step for the reader

Reference answer:
{reference}

Reply to grade:
{candidate}

Return JSON only:
{{"correctness": 0, "grounding": 0, "tone": 0,
  "action": 0, "reason": "one sentence"}}"""

Notice the choices baked in. Binary criteria, not marks out of ten — models are unreliable at fine-grained scales but decent at yes/no, and binary scores aggregate cleanly. A reference answer whenever one exists, so the judge compares rather than imagines. A required JSON shape (the structured output chapter at work) plus a one-sentence reason, because a score you cannot audit is a score you cannot trust. In the real call, pin the judge's model version and keep its temperature low — the examiner should be the most boring driver on the road.

Careful

LLM judges have documented biases. Verbosity bias: longer answers score higher, deserved or not. Position bias: in "which of these two is better?" comparisons, the answer shown first wins more often — run every pair both ways and keep only consistent verdicts. Self-preference: models rate their own family's outputs more kindly, so judge with a different model where you can. And the judge is itself nondeterministic. The standing fix: keep a small set of human-graded examples and measure how often the judge agrees with the humans — when agreement drops, recalibrate the rubric. You grade the examiner.

From notebook to merge gate: the regression suite in CI

An eval you run once, by hand, the week before launch is a demo. It becomes a regression suite when it is fixed, versioned, and re-run on every change — so any drop against the baseline is caught at the merge request, in the pipelines you learned to read in the pipelines chapter. The wiring is ordinary CI:

.gitlab-ci.yml — evals as a merge gate
agent-evals:
  stage: test
  script:
    - python evals/run_suite.py --suite smoke --min-pass-rate 0.90
  rules:
    - changes:
        - agents/**/*
        - prompts/**/*
        - evals/**/*
  artifacts:
    when: always
    paths:
      - evals/report.json

nightly-evals:
  stage: test
  script:
    - python evals/run_suite.py --suite full --report evals/report.json
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule"

Two design decisions in that file matter more than the syntax. First, the smoke/full split. Unlike pytest, evals call real models — every case costs tokens and seconds — so the per-merge-request job runs a small smoke suite covering the critical behaviours, while the full suite (hundreds of cases) runs nightly. Second, the rules: changes: guard: the eval job fires only when agent code, prompts, or the evals themselves changed. Re-grading an untouched agent after a README edit burns money, and you saw in the failing-pipelines chapter what teams do to jobs that annoy everyone.

The report artifact is the third quiet hero: scores from every run become a trend line, and the trend line is the payoff. One number means nothing; the same suite scored across versions tells you whether that prompt rewrite actually helped:

Tip

Keep the dataset in git and review edits to it like code. If both the agent and the exam change in one merge request, a moving score is uninterpretable. Resist chasing 100 per cent: investigate step-changes in the rate, shrug at single-case wobble. A regression suite is a smoke alarm, not a court verdict.

The tooling landscape, briefly

You will rarely build the whole apparatus from scratch. The ecosystem (as of mid-2026 — names churn, the shape does not) splits into hosted platforms and open-source libraries:

ToolWhat it is
LangSmithHosted platform from the LangChain team: tracing, datasets, evaluators, and annotation queues where humans grade sampled runs. The archetype of the "LangSmith-style platform".
LangfuseOpen-source tracing plus evals; self-hostable, popular where data cannot leave the building.
BraintrustHosted eval platform with strong CI integration — scores posted straight onto pull/merge requests.
Arize PhoenixOpen-source observability with built-in LLM evals.
DeepEvalOpen-source Python library; writes evals in a pytest-like style, which makes the CI wiring feel like chapter 104.
promptfooConfig-driven eval runner (YAML in, scores out); a favourite for quick CI gates.
RagasMetrics built for RAG pipelines: faithfulness, answer relevance, context precision.
InspectOpen-source eval framework from the UK's AI Security Institute; strong on agentic, multi-step tasks.

Whatever the logo, look for the same organs you built above: a dataset store, a runner that captures trajectories, pluggable graders including rubric-driven judges, and a way to post the rate into CI. One distinction to keep sharp: public benchmarks (SWE-bench, GAIA and friends) measure models on standardised tasks so the industry can compare them; your regression suite measures your agent on your tasks. A leaderboard-topping model can still fail your invoice workflow — only your own golden dataset will tell you.

In the wild

In a real repo, look for an evals/ folder holding JSONL datasets (often goldens.jsonl), grader code, and a runner script; an eval job in .gitlab-ci.yml beside the pytest job; and merge-request descriptions carrying a small table — "billing suite: 91% → 94%, no trajectory regressions". In review threads you will hear "this MR regresses the refunds suite", "the judge and the humans disagree on twelve cases — recalibrating the rubric", and "add yesterday's incident as an eval case before we close the ticket". That last one marks a serious team: every production failure becomes a permanent exam question, the same way bug-fix commits arrive carrying a new test.

Check yourself

1. Your test asserts answer == "GST comes to 360 rupees." and fails every third run even though the agent is always right. What is the eval-shaped fix?

Low temperature reduces variation but does not eliminate it, and retry-until-green hides real regressions. The instrument built for nondeterministic systems is a dataset plus meaning-aware graders plus a rate.

2. An agent answers a contract question correctly, but its trace shows it never called the retrieval tool — it answered from training memory. Which grader catches this, and why does it matter?

Nothing in the answer text reveals the skipped lookup; only the trajectory does. Final-answer evals grade the destination, trajectory evals grade the drive — and unreliable driving is what fails you later.

3. In pairwise LLM-as-judge comparisons, your judge prefers whichever answer it is shown first. What is this, and what is the standard mitigation?

Order-dependent preference is position bias. Swapping the order and requiring agreement is the standard fix; self-preference is the opposite problem (judge favouring its own family), mitigated by judging with a different model.

4. Why do teams run a small smoke suite on every merge request but the full eval suite only nightly?

It is purely an economics-and-latency split. The smoke suite keeps the merge gate quick and affordable; the nightly full suite provides the deep, comprehensive signal. Both run the same graders.

5. In the pyrun above, a naive "answer contains 300" grader would have passed the wrong answer "3000 rupees". What is the lesson?

The whole eval is only as trustworthy as its graders. Substring checks, sloppy regexes, and vague judge rubrics all produce false confidence — grader code deserves the same review discipline as the agent it grades.

Key takeaways

  • Evals are tests for nondeterministic systems: a dataset of tasks, a runner that captures everything, graders that score meaning, and a pass rate instead of a single red/green assert.
  • Grade both clipboards — the final answer (did it arrive?) and the trajectory (was the drive sane: right tools used, no repeated calls, within the step budget). Lucky answers pass the first and fail the second.
  • LLM-as-judge unlocks grading prose — but only with an explicit rubric, binary criteria, a reference answer, structured output, and regular calibration against human grades; it carries verbosity, position, and self-preference biases you must counter.
  • A regression suite is an eval made permanent: versioned dataset in git, smoke subset gating every merge request, full suite nightly, scores archived so the trend line — not any single run — tells the story.
  • Graders are code with bugs of their own; the "contains 300 passes 3000" trap is real. Review and test your graders like you review the agent.
  • Platforms and libraries (LangSmith-style hosted tools; open-source runners like DeepEval, promptfoo, Ragas, Inspect) package the same organs you built by hand — and public model benchmarks never substitute for your own golden dataset on your own tasks.
