Structured output

How to make the world's most creative text generator fill in a form — every field present, every type correct, every single time. Pydantic returns to the stage, this time as the gatekeeper of everything an agent does.

The story

Dr. Rao is the finest diagnostician in the hospital — and the chattiest. Ask her for a prescription and you get a small lecture: "Lovely case! Given the history, I'd say amoxicillin, perhaps five hundred milligrams, twice a day — oh, and tell them to finish the course." Brilliant. Also completely useless to the pharmacy robot downstairs, which dispenses pills by reading exactly four boxes on a printed form: patient, drug, dose in mg (a number), frequency.

So the hospital employs a clerk. The clerk hands Dr. Rao the form, takes it back, and checks every box against the rules. Dose says "five hundred" in words? The form goes straight back with a sticky note: "Box 3 must be a number. Fix only that." Dr. Rao sighs, writes 500, and now — and only now — the form goes to the robot.

The doctor is the LLM. The printed form is a schema. The clerk is Pydantic. The sticky-note round trip is the retry loop. This chapter is about that whole desk.

First, remember Pydantic

Back in the type hints and Pydantic chapter, you met Pydantic as Python's strict doorman: you declare a class with typed fields — class Invoice(BaseModel): total: float — and Pydantic checks incoming data against it at runtime, coercing what it reasonably can and raising a loud ValidationError for what it can't. Back then it guarded config files and API payloads written by humans and other programs. The promise at the end of that chapter was that Pydantic would return with a bigger job. This is that job: guarding your code against the most fluent, most confident, most occasionally wrong data source you will ever connect to — a large language model.

The problem: the model writes essays, your code needs data

Every LLM you have ever used produces exactly one thing: text. Ask it to extract a prescription from a doctor's note and, left to its own devices, it replies like Dr. Rao:

what the model wants to say
Sure! Based on the note, here is the prescription information:

The patient A. Sharma should take amoxicillin at a dose of
five hundred milligrams, twice daily. Hope that helps!

Now imagine the next line of your program is db.save(prescription) or dose_total = dose_mg * days. You cannot multiply "five hundred milligrams". You cannot save "Hope that helps!" into a database column. Code does not consume prose; code consumes data — the dictionaries-and-lists world you met in the YAML and JSON chapter, where every value has a name and a type.

The naive fix is to beg in the prompt: "Please respond in JSON only." And the model usually obliges... in the way a golden retriever obliges. You get JSON wrapped in friendly chatter ("Sure! Here's your JSON:"), or JSON with a trailing comma, or JSON where the field you called dose_mg comes back as dosage, or — the classic — a perfectly formed object where the number you needed is a string. Any of these crashes the very next line of your code. When it works ninety-five per cent of the time, that is not reliability; that is a production incident on a five per cent timer.

The real fix has two halves, and this chapter walks both: first, tell the model the exact shape you want, in a language machines agree on. Second, never trust the reply until a validator has stamped it — and when the stamp fails, send the form back with the sticky note.

JSON Schema: the printed form itself

The machine-readable language for "here is the shape I want" is JSON Schema — a standard way of describing, in JSON, what fields an object must have and what type each one is. It is the printed form from the story: box names, box types, which boxes are mandatory. You could write one by hand, but you never have to — this is where Pydantic earns its second salary. Define the shape once as a Python class, and Pydantic generates the schema for you:

prescription.py — one class, two jobs
from pydantic import BaseModel

class Prescription(BaseModel):
    patient: str
    drug: str
    dose_mg: int
    frequency: str

print(Prescription.model_json_schema())
what model_json_schema() produces
{
  "title": "Prescription",
  "type": "object",
  "properties": {
    "patient":   {"title": "Patient",   "type": "string"},
    "drug":      {"title": "Drug",      "type": "string"},
    "dose_mg":   {"title": "Dose Mg",   "type": "integer"},
    "frequency": {"title": "Frequency", "type": "string"}
  },
  "required": ["patient", "drug", "dose_mg", "frequency"]
}

Read that JSON slowly once — it is the last time it will look foreign. It says: the result is an object; it has these four properties; dose_mg must be an integer; all four are required. This is a contract — an agreement, checkable by a machine, between the side producing the data (the model) and the side consuming it (your code). The Pydantic class is the single source of truth: it generates the schema you send to the model, and it validates the JSON that comes back. One class, both directions.

A few words to have ready before we go further — flip until they feel like old friends:

Three ways to ask, from polite request to locked form

Providers give you three escalating levels of enforcement. Knowing which level you are on tells you exactly how paranoid your code needs to be afterwards.

ModeHow you askWhat is actually guaranteed
Prompt begging"Respond in JSON only" in the promptNothing. Works often, fails memorably.
JSON modee.g. OpenAI response_format={"type": "json_object"}The reply parses as JSON. Fields and types are still whatever the model felt like.
Structured outputsYour JSON Schema attached to the request, strict flag onThe reply parses as JSON and matches your schema — right fields, right types.

The third level — structured outputs, meaning schema-enforced generation — is the one worth building on, and every major provider now offers it. On OpenAI's API the schema rides in the response_format parameter; on Anthropic's it rides in output_config.format. Better still, both Python SDKs accept your Pydantic class directly and hand you back a validated instance, no JSON handling in sight:

OpenAI — schema-enforced, straight from the Pydantic class
completion = client.chat.completions.parse(
    model="gpt-5",
    messages=[{"role": "user", "content": note}],
    response_format=Prescription,      # the Pydantic class IS the contract
)
rx = completion.choices[0].message.parsed   # a Prescription instance
Anthropic — same idea, different spelling
response = client.messages.parse(
    model="claude-opus-4-8",
    max_tokens=1024,
    messages=[{"role": "user", "content": note}],
    output_format=Prescription,
)
rx = response.parsed_output                 # a Prescription instance

Under the hood both requests carry the generated JSON Schema, and the provider applies constrained decoding: at every step of generation, tokens that would break the schema are simply not on the menu. The model is physically unable to write "five hundred" into an integer field — like a form whose dose box only accepts digits.

Note

Helper names drift between SDK versions — parse methods move in and out of namespaces, parameter names get renamed (Anthropic's older top-level output_format parameter on create calls, for instance, gave way to output_config.format). The wire-level idea is the stable part: a JSON Schema attached to the request, a strictness flag, valid JSON back. Check the current docs for the exact spelling; recognise the shape on sight.

Tip

The same trick exists for tools. Remember from the tool-calling chapter that every tool carries an input_schema? Providers let you mark a tool definition strict: true, which schema-enforces the arguments the model generates for it. A tool call is structured output wearing a work uniform.

The safety net: validate, then retry

So why does Pydantic still matter if the provider guarantees the schema? Three reasons, and they are the difference between a demo and a system. First, not every model and mode supports strict enforcement — local models, older endpoints, and plain JSON mode all hand you unverified text. Second, schemas can only express so much: "the discount must not exceed the total" is a rule Pydantic can check and no decoder can. Third — the deepest one — a guarantee you don't verify is a rumour. Production code treats every model reply as a form arriving at the clerk's desk, whoever claims to have checked it upstream.

Why everything in an agent flows through this

Here is the quiet punchline of this chapter: structured output is not one feature among many. It is the load-bearing wall of the entire agent world you operate.

Tool arguments are structured output. When a model "calls a tool" (ch35), what actually happens is that it generates a JSON object of arguments constrained by that tool's input_schema — the same schemas, the same validation, the same strict flags. A tool call is a filled-in form whose recipient happens to be a function instead of a database.

Agent state is structured output. Frameworks like LangGraph define an agent's working memory as a typed state schema; deep-agent planning tools accept the plan as a structured list of task objects, not freeform musings; MCP servers advertise every tool with a JSON Schema for its inputs. Each time an agent records a decision, updates a plan, or hands work to another agent, that information crosses the boundary as schema-shaped data — because the receiving code, like the pharmacy robot, only reads boxes.

Reliability is structured output. The difference between an agent demo and an agent product is almost never the model. It is whether every model-to-code handoff has a contract, a gate, and a retry — the desk you just built. When your production agents run for hours without a human watching, this pattern is what they are standing on. And it composes with everything still to come: the ReAct loop later in the course is this same conversation-with-a-gate, running on repeat.

In the wild

You will meet this chapter everywhere. In Python codebases: a schemas.py or models.py full of BaseModel classes, model_validate_json inside a try/except ValidationError, or the instructor library wrapping a client. In LangChain code: model.with_structured_output(MySchema). In raw API calls: response_format (OpenAI) or output_config with "type": "json_schema" (Anthropic), and "strict": true on tool definitions. In merge requests: a teammate adding additionalProperties: false to a schema, or a commit message like "add retry on validation failure for the extraction step". Every one of these is the clerk's desk — form, gate, sticky note — and you now read them fluently.

Check yourself

1. You switch on JSON mode (response_format={"type": "json_object"}). What is actually guaranteed about the reply?

JSON mode guarantees syntax only: the text will parse. Whether it contains your dose_mg field, or a number where you need one, is still up to the model. For shape guarantees you need schema-enforced structured outputs — and a validator regardless.

2. Pydantic raises a ValidationError on the model's reply. What does the retry loop send back to the model?

The magic is in the specificity. ValidationError names the exact field and the exact problem, and models are much better at fixing a named mistake than at succeeding cold. Re-rolling without feedback wastes the information the validator just produced.

3. With strict schema enforcement (constrained decoding), what can still be wrong with the reply?

Constrained decoding makes off-schema output impossible to generate, so syntax and shape are safe. But a schema cannot know the truth: the model can confidently write 5000 where the note said 500. Schemas guarantee shape, never correctness — which is why cross-field checks and human review still exist.

4. When an agent framework runs a tool call, how do the tool's arguments relate to this chapter?

A tool call is a filled-in form addressed to a function. The tool's input_schema is the contract, the generated arguments are the structured output, and marking the tool strict: true applies the same constrained decoding. One mechanism, wearing two uniforms.

Key takeaways

  • LLMs produce prose; code consumes data. Structured output is the discipline of bridging that gap with a contract instead of hope.
  • A JSON Schema is the machine-readable form; your Pydantic class is the single source of truth that both generates the schema and validates the reply.
  • Enforcement comes in levels — prompt begging guarantees nothing, JSON mode guarantees syntax, schema-enforced structured outputs guarantee shape. Build on the third; verify anyway.
  • The production pattern is a loop: validate with Pydantic, and on ValidationError send the specific error back and retry — bounded, so failure ends in an exception, not a bill.
  • Schemas guarantee shape, never truth: a perfectly valid object can still contain a wrong number. Keep business-rule checks and review where stakes are high.
  • Tool arguments, agent state, plans, and MCP tool inputs are all schema-shaped handoffs — every reliable agent you operate is this chapter, running in a loop.
