The meter is always running

Every token in, every token out — and every turn, the whole conversation again. Learn how LLM pricing really works, why chats quietly get dearer as they get longer, and the three levers that cut the bill.

The story

Your family retains a brilliant but peculiar law firm. The lawyer answers any letter with flawless judgement — under two strict billing rules. Rule one: reading is billed per page, and writing is billed at five times the reading rate, because writing is the hard part. Rule two: the lawyer keeps no memory of your case. None. Every time you write in, the clerk photocopies the entire correspondence — every letter you ever sent, every reply you ever received — staples your new one-line question to the back, and the lawyer reads the whole bundle from page one. Billed per page, every time.

The clerk offers two mercies. Pages the lawyer read recently sit in a warm pile on the desk, and re-reading a warm page costs a tenth of the normal rate. And if you do not need the answer today, the overnight drop-box is half price.

That law firm is every LLM API on Earth. The pages are tokens, the warm pile is the prompt cache, the drop-box is the batch API — and the meter, as you are about to see, is always running.

Two prices on one meter

Everything you learned in the tokens chapter now gets a price tag. Vendors bill in tokens, quoted per million, and every request has exactly two line items. Input tokens are everything you send — the system prompt, the conversation history, the user's question, any documents or tool results stuffed in. Output tokens are everything the model generates back. They are priced very differently:

Model tierInput, per 1M tokensOutput, per 1M tokensOutput multiple
Small and fast$1.00$5.005x

Note

These are illustrative round numbers, deliberately shaped like real 2026 price lists — always check your vendor's current pricing page. The durable fact is the ratio: output tokens typically cost four to five times what input tokens cost, at every vendor, at every tier.

Why is writing dearer than reading? Because of how the machine works. When the model reads your prompt, it can process all of those tokens in one big parallel pass — cheap. But it generates its reply one token at a time, and every single output token requires another full pass over everything that came before it. Reading is a skim; writing is labour. The lawyer's five-times rate for drafting is not greed, it is physics.

This asymmetry gives you your first practical instinct: a verbose model is an expensive model. The single simplest cost control in the entire field is max_tokens — a hard cap on how many output tokens one response may contain. You met it in your first API call; now you know why every serious request sets it.

The multiplier nobody sees: every turn resends everything

Here is the part that surprises almost everyone, and it follows directly from something you already know: the API is stateless — the model remembers nothing between calls, so your code must re-send the entire conversation history with every single turn. In the law firm, the clerk photocopies the whole case file every time. In your code, the messages list — which only ever grows — goes over the wire again and again.

That means turn ten of a chat does not cost what turn one cost. Take a modest example: a 2,000-token system prompt, users writing about 200 tokens per message, the model replying with about 300. Turn one sends 2,200 input tokens. Turn two sends those again, plus the first reply, plus the new question — 2,700. Every turn adds the previous exchange to the pile, so each turn is billed more than the last. Watch the cumulative input bill:

Read that last bar twice. After ten turns you have paid for 44,500 input tokens to sustain a conversation containing only 7,000 tokens of actual text. Roughly six times over, you re-billed words that had already been billed. The conversation grew in a straight line; the bill grew in a curve. This is the history-resend multiplier — the single biggest force in LLM economics, and it is invisible until you plot it.

Careful

This is exactly how a harmless-looking loop becomes a bill. Remember from the tool-calling chapter that every tool call is a full turn: model call, tool result appended, model call again. Now imagine an agent that dumps a 30,000-token file into a tool result on step 2 of a 40-step run. That file is re-sent — and re-billed — on all 38 remaining steps: over a million input tokens for one file. Add a retry loop ("just run it again until the tests pass") and you multiply the whole curve. Nobody wrote a bug; someone wrote a loop, and the meter did the rest.

Prompt caching: the warm pile on the desk

Vendors know the resend multiplier is brutal, so they offer the same mercy as the clerk: prompt caching — recently seen input is kept warm on the vendor's servers, and when your next request starts with the exact same tokens, that prefix is re-read at a steep discount instead of full price. The going rates: writing to the cache costs a small premium over normal input (about 25% more for the default short-lived cache; longer-lived caches cost more to write), and reading from it costs about a tenth of normal input — roughly 90% off. The cache pays for itself from the second use onward.

The one rule that governs everything: caching is a prefix match on exact bytes. The vendor compares your new request to the cached one from the very first character. The moment a single byte differs, everything from that point onward is a cache miss. Step through one lifecycle:

1. Turn one — the cache gets written

Everything the cache has seen is ~90% off; only the genuinely new tail pays full rate. In a long chat, the warm prefix is almost the whole request — which is why caching roughly halves a ten-turn bill and saves even more on longer ones.

3. The silent invalidator

"Today is 2 July, 09:14"a timestamp at the top of the system prompt
→
Cachemiss — everything after byte one re-billed

Someone helpfully adds the current time to the top of the system prompt. Every request now differs from the last at byte one, so nothing ever hits the cache — with no error, no warning, just a quietly doubled bill. Timestamps, random request IDs, and unsorted JSON keys are the classic culprits. Volatile content belongs at the end of the prompt, never the start.

4. Trust, but verify

usagecache_read_input_tokens: 2200
usageinput_tokens: 500 (full price)

Every API response carries a usage block that splits the bill: full-price input, cache writes, cache reads, output. If the cache-read number is zero request after request, a silent invalidator is at work. This little block is your meter reading — log it.

Tip

Vendors differ in mechanics — some cache automatically, others ask you to mark cache points in the request — but the economics and the prefix rule are universal. The design habit that always works: put the stable stuff first (frozen system prompt, fixed tool list, big documents) and the changing stuff last (the user's question, the current date).

Batch: half price if you can wait

The overnight drop-box is real, and it is called the batch API — you submit a big list of requests, the vendor runs them on spare capacity, and you collect the results later, typically at 50% off everything. "Later" usually means under an hour, with a guarantee of within a day. Nobody can sit in a chat window waiting for that, so batch is useless for conversation — and perfect for everything nobody is waiting on: classifying last month's ten thousand support tickets overnight, re-summarising a document library, running an evaluation suite, generating embeddings for a new corpus.

The discipline is simply to ask, for every workload: is a human actually waiting for this answer? If not, it belongs in the drop-box. Teams routinely discover that half their token spend is bulk work that nobody was waiting on, running at full interactive price out of habit.

Budget guardrails: what to ask any vendor (or your own team)

You operate these systems, which means sooner or later you will sit in the meeting where the invoice gets explained. These are the questions that make that meeting short — each one maps to a lever from this chapter:

  • Is output capped? Does every request set max_tokens, and does every agent loop have a hard turn limit? Output is the 5x-priced token, and unbounded loops are how bills happen.
  • Is there a hard budget with alerts? Can spend be capped per API key, per team, per feature — and does something alert (or cut off) before the invoice, not after?
  • Can we see the split? Does the usage dashboard separate input, output, cache writes, and cache reads — per feature? A single total hides everything actionable.
  • What is our cache hit rate? If cache reads are near zero, what is invalidating the prefix — a timestamp, a per-user ID, an unstable tool list?
  • What could move to batch? Which workloads have no human waiting, and are they getting the 50% discount?
  • Are we tiering models? Is a frontier-priced model doing work a small model passes the quality bar on — classification, routing, extraction?
  • What do long conversations cost us? Is anything trimming, summarising, or capping history growth, or does every chat ride the resend curve to the end?
In the wild

You will meet this chapter everywhere. In response logs: the usage block with input_tokens, output_tokens, and the cache fields — the meter reading attached to every call. In merge requests: "set max_tokens on the summariser", "move the eval run to the batch API", "the system prompt change broke the cache". In vendor dashboards: cost split by input/output and a cache hit-rate chart. And in the incident channel, the classic post-mortem: a retry loop or an over-eager agent ran all weekend, and the fix is a turn cap plus a budget alert — both of which you can now propose by name.

Check yourself

1. Why do output tokens cost roughly five times more than input tokens?

Reading is a parallel skim; generating is sequential labour — every output token requires another pass over everything before it. That is why the 4-5x ratio shows up at every vendor, and why capping output with max_tokens is the first guardrail.

2. Your one-line question on turn 20 of a chat costs far more than the same question did on turn 1. Why?

The model remembers nothing between calls; your code resends the whole transcript every turn. That is the history-resend multiplier: the bill grows in a curve while the conversation grows in a line.

3. A teammate adds "Current time: 09:14:22" to the top of the system prompt. The cache hit rate drops to zero. Why?

One changed byte at the front and nothing downstream ever matches — no error, just full price on every request. Volatile content (times, IDs) belongs at the end of the prompt; the stable prefix goes first.

4. Which workload belongs on the batch API?

Batch trades speed for a ~50% discount: results usually within an hour, guaranteed within a day. Perfect for bulk work nobody is waiting on; useless for anything interactive or time-critical.

5. An agent dumps a 30,000-token file into a tool result on step 2 of a 40-step run. Roughly what happens to the bill?

Every tool call is a full turn, and every turn resends the whole history — the file included. This is how a harmless-looking loop becomes a bill; the defences are turn caps, trimming big tool results, and prompt caching.

Key takeaways

  • LLM APIs bill two things: input tokens (everything you send) and output tokens (everything generated) — and output costs 4-5x more because generation is sequential work.
  • The API is stateless, so every turn resends the entire history; the bill grows in a curve while the conversation grows in a line. Agent loops and fat tool results multiply this — that is how a harmless loop becomes a bill.
  • Prompt caching re-reads an unchanged prompt prefix at ~90% off (for a ~25% one-time write premium). It is an exact-byte prefix match: keep stable content first, volatile content last, and watch out for timestamps.
  • The batch API gives ~50% off anything nobody is waiting on — usually back within an hour, guaranteed within a day.
  • Every response carries a usage block splitting full-price input, cache activity, and output. It is the meter reading; log it and alert on it.
  • The guardrails to demand anywhere: max_tokens on every request, turn caps on every loop, hard budgets with alerts, a dashboard that splits the bill, and the cheapest model that passes the quality bar.
