Tools: giving the model hands

The model chooses tools by reading their labels, never their code. In this chapter you write a tool, label it so well the model cannot misuse it, and execute it so safely that a mistake becomes a printout instead of an incident.

~30 minChapter 47 of 117
The story

Down the road from your flat is a tool-lending library — a basement full of drills, saws, ladders and one glorious cement mixer. Members never walk the shelves. You stand at a counter, flip through a ring-binder of catalogue cards, and fill in a request slip. The card that just says "DRILL" gets requested for every job, including several it ruins. The card that says "Cordless drill — wood and drywall only, bits up to 10 mm, not for concrete" almost never comes back with a burnt-out motor. Same drill. Different card.

Behind the counter sits the librarian, and she is the other half of the system. She checks that the tool on your slip actually exists in the catalogue, that the slip is filled in properly, and that you have done the safety course before the chainsaw comes out. For anything truly destructive she hands you a practice piece first: "show me on this; the real one comes out after."

The borrower is the model. The catalogue cards are your tool descriptions. The librarian is the code you are about to write.

Where we left off — and where it was weakest

In the previous chapter you built a complete ReAct agent in fifty lines: a history list, a loop, a scripted stand-in model, and a TOOLS dictionary holding exactly one function — a calculator. The model wrote ACTION: calculate(...), your loop looked the name up in the dict and ran the real function, and the result went back into the history as an observation. That loop is the skeleton of every agent you operate.

This chapter zooms in on the part of that skeleton we rushed past: the tools themselves. Three questions, in order. What makes a function a good tool? How does the model — which never sees a single line of your code — decide which tool to call and what to pass it? And how do you execute the model's requests without letting a confused sentence become a real-world disaster? The answers turn out to be one part writing, one part typing discipline, and one part healthy paranoia.

Anatomy of a tool worth having

A tool is nothing exotic: a plain Python function (chapter 7) plus two pieces of packaging — a name the model can request it by, and a description telling the model what it does. Here is a second tool for our agent, a GST rate lookup:

tools.py — a small, honest tool
def gst_rate(category):
    """Look up the GST rate (in percent) for a spending
    category. Known categories: restaurant, electronics,
    gold, cinema. Returns just the number, e.g. '5'."""
    rates = {"restaurant": 5, "electronics": 18,
             "gold": 3, "cinema": 12}
    category = category.strip().lower()
    if category not in rates:
        return (f"ERROR: unknown category {category!r}. "
                f"Known: {sorted(rates)}")
    return str(rates[category])

Four habits make this a good citizen of an agent loop, and they generalise to every tool you will ever write:

One job. It looks up a rate. It does not also calculate the tax, format a message, or send an email. Small tools compose; Swiss-army tools confuse the model about when to reach for them.

An obvious name. gst_rate says what it returns. A model choosing between tools reads names first — helper2 and do_stuff are how agents end up calling the wrong thing.

Text in, text out. The result must travel back through the observation channel as text the model can read. Return strings, or short JSON-ish summaries — not live objects.

Errors are answers, not explosions. Notice the unknown-category branch returns a readable message instead of raising an exception. Remember from chapter 9 what an unhandled exception does — it tears down the whole program, loop included. But a returned ERROR: unknown category 'spa'. Known: [...] becomes an observation, the model reads it, and on the very next turn it can correct itself. A good error message is a teaching aid for the model. This one habit fixes more "the agent got stuck" tickets than any clever prompt.

The description is the real prompt engineering

Now the idea this chapter exists for. When your loop (or the API doing native tool calling) advertises tools to the model, it does not send your source code. It sends a menu: each tool's name, its description, and a note of what parameters it takes. That menu is pasted straight into the model's context, right next to the system prompt. The model chooses tools the way you chose a drill at the lending library — by reading the card.

Which means the description is not documentation. Documentation is for humans who read it optionally, later. The description is a prompt: instructions the model follows, literally, on every turn. Every word in it steers behaviour. Anthropic's published guidance on writing tools for agents makes exactly this point, and reports that refining tool descriptions alone — no model change, no new tools — lifted an earlier Claude model to state-of-the-art on a public software-engineering benchmark. The cheapest, highest-leverage tuning knob in your whole agent is a paragraph of prose.

The distilled guidance, card by card:

Write for a new hire. Describe the tool as you would to a smart colleague on their first day: what it does, when to use it, when not to use it, what the inputs look like, what comes back. All the context you carry implicitly — quirky formats, niche terms, which of two similar tools wins — must be made explicit, because the model has none of it.

Name parameters unambiguously. A parameter called user invites an email address, a name, or an id, depending on the model's mood. Call it user_id and say in the description "the numeric customer id, e.g. 48291", and the ambiguity dies.

Make results meaningful. What a tool returns lands in the context too — tool results are also prompts. Return human-readable summaries with the fields that matter, not a dump of raw ids and internal codes the model must guess at.

Spend tokens like money. A search tool that returns 400 results floods the context (and your bill). Default to the top few, paginate, truncate long fields, and say in the description how to ask for more.

Prefer a few workflow-shaped tools over many thin wrappers. One schedule_meeting tool beats five separate calendar-API endpoints the model must choreograph itself. Fewer, better-labelled cards; less to choose wrongly between.

Tip

When an agent keeps misusing a tool, resist the urge to rewrite the system prompt. Nine times out of ten the fix is on the card: sharpen the tool's description, rename an ambiguous parameter, or state explicitly which sibling tool to prefer. You are debugging prose, and it is the most cost-effective debugging you will ever do.

Schemas from type hints: the card writes itself

In our hand-rolled loop the "menu" is plain text we glue together from docstrings. Production systems use something sturdier: a schema — a machine-readable description of the tool's parameters, their types, and which are required, usually in JSON Schema format (the same idea you met with structured output). Here is a destructive tool written properly, and the schema it becomes:

refunds.py — a dangerous tool, dressed correctly
def issue_refund(order_id: str, amount_inr: float,
                 dry_run: bool = True) -> str:
    """Refund a customer order, capped at 5,000 INR.
    order_id looks like ORD-123456. Use only after
    lookup_order confirms the order exists. With
    dry_run=True (the default) nothing moves: it
    reports what WOULD happen for review."""
    ...
What the API is actually sent (Anthropic tool format)
{
  "name": "issue_refund",
  "description": "Refund a customer order, capped at 5,000 INR. order_id looks like ORD-123456. Use only after lookup_order confirms the order exists. With dry_run=True (the default) nothing moves: it reports what WOULD happen for review.",
  "input_schema": {
    "type": "object",
    "properties": {
      "order_id":   {"type": "string"},
      "amount_inr": {"type": "number"},
      "dry_run":    {"type": "boolean", "default": true}
    },
    "required": ["order_id", "amount_inr"]
  }
}

Look at where every part of that JSON came from. The parameter names and types came from the type hints — order_id: str became "type": "string". The required list came from which parameters lack defaults. And the description came word-for-word from the docstring, the string sitting directly under the def line. You do not write this JSON by hand: frameworks generate it. LangChain's @tool decorator and the MCP Python SDK's @mcp.tool() (from mcp.server.fastmcp) both read your signature and docstring and build the schema for you — a decorator quietly doing paperwork, exactly as chapter 17 promised they would.

This is why sloppy docstrings produce sloppy agents. The docstring you were told was "just good practice" back in Act I is now literally the interface: the one paragraph of your code the model actually reads.

Safe execution: the librarian's rules

The model has now chosen a tool and produced arguments. Everything the model sends you is best treated the way a web developer treats a form submitted from the internet: probably fine, occasionally malformed, and never, ever trusted blindly. Four rules, in the order they fire:

Rule 1 — the allow-list. The TOOLS dict from chapter 46 is not a convenience; it is the security boundary of the entire agent. The loop runs a function only if the requested name is a key in that dict (an allow-list — a closed list of what is permitted, everything else refused). Never resolve tool names dynamically with getattr or — please — eval: that would let the model reach any function in your process. Our chapter 46 loop actually had a bug here: an unknown name crashed the loop with a KeyError. The hardened version replies with an error observation instead, and the model recovers.

Rule 2 — validate the inputs. Check shape and range before anything runs: the regex for ORD-123456, the 1-to-5,000 cap on the amount, the character whitelist in our calculator. Reject bad input with a message that says what valid input looks like.

Rule 3 — bound the run with a timeout. A tool that calls a slow database or a dead network endpoint can hang forever, and with it the whole agent. Give every tool a time budget; when it blows, return "timed out after 10s" as the observation and let the model decide what to do next.

Rule 4 — default to harmless. Separate read tools from write tools, and make anything with a side effect — a change to the real world — default to dry run: report what would happen, do nothing. The real action requires an explicit, deliberate elevation, often by a human. This is precisely why your coding agent pauses to ask before running a shell command: that permission prompt is Rule 4, rendered as a dialog box, at exactly the boundary this chapter drew.

Careful

Never pass model-written text into eval, exec, or a shell without a strict whitelist. Our calculator's innocent-looking character check — only digits and arithmetic symbols allowed — is what stands between "compute 20% GST" and the model accidentally (or an attacker deliberately, via a poisoned document) running arbitrary code on your machine. That one line is load-bearing.

In the wild

Open any MCP server on GitHub and read the docstrings under its @mcp.tool() functions — every one of those paragraphs is shipped into a model's context, which is why the good ones read like briefings, not comments. In merge requests you will meet one-line diffs that change only a docstring, with a message like "agent kept using search for exact id lookups" — that is agent behaviour being engineered in prose, exactly as this chapter promised. And in your daily life as an operator: the permission dialog before a coding agent runs a shell command, the read-only mode on a database tool, the --dry-run flag in a deploy script — all of them are the librarian's rules, deployed at the same boundary you just built.

Check yourself

1. Your registry offers search_orders and lookup_order. What does the model use to decide which one to call?

The function bodies never leave your machine. The model sees only the menu — name, description, schema — which is exactly why the description is the real prompt engineering.

3. Mid-run, the model requests ACTION: drop_all_tables() — a tool you never registered. In a well-built loop, what happens?

The registry is the security boundary: nothing outside it can execute, however confidently the model asks. (The naive chapter 46 loop would have crashed with a KeyError here — returning a readable error instead lets the model recover.)

4. Why should a destructive tool like issue_refund default to dry_run=True?

Defaults decide what happens when nobody is paying attention. With dry-run as the default, the real action requires a deliberate opt-in — often by a human at a permission prompt.

5. A tool receives an invalid argument. What is the best behaviour?

Observations are prompts too. An exception kills or derails the loop and an empty string teaches nothing — but "ERROR: unknown category 'spa'. Known: [cinema, electronics, gold, restaurant]" lets the model fix itself on the very next turn.

Key takeaways

  • A tool is a plain function plus a name and a description — and the description, not the body, is what the model reads. Tool descriptions are prompts: the real prompt engineering.
  • Write each description like a briefing for a new hire: when to use it, when not to, what inputs look like, what comes back — with unambiguous parameter names (user_id, never user).
  • Type hints plus docstrings become the JSON schema the API advertises; framework decorators generate it automatically, which is why sloppy docstrings produce sloppy agents.
  • The tool registry is an allow-list and the agent's security boundary: the model can only ever reach functions you registered, and unknown requests become error observations, not executions.
  • Treat model-typed arguments as untrusted input — validate shape and range, bound every run with a timeout, and default anything destructive to dry-run or read-only.
  • Results and error messages are prompts too: return meaningful, concise text — a good error message is how an agent recovers instead of getting stuck.
