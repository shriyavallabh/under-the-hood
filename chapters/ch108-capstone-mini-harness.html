The parts tray: what you are about to assemble

A harness — the word this whole course is named after — is the machinery wrapped around a language model that turns raw text prediction into a working agent: the loop, the tools, the memory management, the persistence, the safety rails, and the tests. You met the idea properly in the harness chapter; today you build one. Not a toy diagram of one — an actual runnable one, in six stages, each stage adding one organ to a living body. Every stage below is a runnable Python box. Press Run, read the output, then break it and run it again.

Because a browser page cannot call a paid model API, each stage uses the trick you know from the ReAct chapter: a scripted fake_llm() stands in for the model. Everything else — the loop, the trimming, the compaction, the checkpointing, the gate, the grading — is exactly the code a production harness runs. Swap the fake for one API call and this is real.

Stage 1 — the loop and its tools

Quick recap of the chapter where you first built this, because it is the block everything else bolts onto. An agent is an LLM run in a loop with tools: the model replies with text that requests an action, your code parses the request, runs the matching Python function, and appends the result to the history as an observation. The model never executes anything itself — it writes wishes, and your loop is the only thing that grants them. The loop ends when the model writes a final-answer marker, or when a turn cap stops a confused model from dithering forever.

Stage 2 — memory that stops growing

Remember the uncomfortable truth from the memory chapter: the history list is the agent's entire mind, it is re-sent to the model on every turn, and it only ever grows. Growth means cost — the token bill — and eventually it means overflowing the context window entirely. The crudest cure is trimming: keeping the task line plus only the newest few messages, and deleting the rest outright.

The run below reads four bills, and the history cap is deliberately set too small. Watch the [trim] lines fire, then read the final history the model could see — and spot what went missing:

Careful

Trimming is honest about cost and dishonest about knowledge. When an agent in production suddenly "forgets" an instruction from early in a long session, this is very often the reason: the message that held it was trimmed out of the window. You diagnosed this pattern in the failure-modes chapter; now you have built the mechanism that causes it.

Stage 3 — compaction: summarize the old, keep the new

The fix, which you met as a design idea in context engineering, is compaction: instead of cutting old messages away, you fold them into a short summary and keep that. The task line stays sacred, the newest messages stay verbatim — because the model needs fine detail about what it is doing right now — and the middle gets condensed. In production the summarizer is another LLM call ("summarize these messages, keep every figure and decision"); here it is a tiny function so you can see the shape of the plumbing.

Now read the compactor the way a reviewer would. Click the annotated lines until each one is boring to you — boring means fluent:

mini_harness.py — the compact() function
def compact(history):
Takes the full message list, returns a smaller one. It runs every turn, just before the model call — cheap when there is nothing to do, decisive when there is.
if len(history) <= MAX_MESSAGES:
The budget check. Under the cap, the history passes through untouched. Real harnesses measure tokens rather than message count, but the shape of the check is identical.
return history
head = history[0]
The TASK line is sacred — it is never summarized away. Lose the task and the agent drifts: it keeps acting, but no longer towards anything.
old = history[1:-KEEP_RECENT]
A slice (ch4) of the middle: everything except the task and the newest messages. This is the material that gets condensed.
recent = history[-KEEP_RECENT:]
The newest messages stay verbatim. The model needs full detail for the step it is in the middle of — you summarize the past, never the present.
summary = fake_summarize(old)
In production this line is an LLM call: "summarize these messages, preserve every figure, file path, and decision." When a coding agent compacts a long session, this is the line doing it.
return [head, summary] + recent
The new history: same shape as before — a list of strings — but far fewer tokens. The loop never notices anything happened. That transparency is the whole trick.
Note

Notice that fake_summarize also keeps lines starting with SUMMARY. That is deliberate: on a very long run, compaction runs repeatedly and must fold the previous summary into the next one, or facts saved by the first compaction would be lost by the second. Summaries of summaries are normal life for long-running agents.

Stage 4 — the checkpoint tin

You built durability intuition in the checkpointing chapter and the durability chapter; here is the whole idea in fifteen lines. The agent's entire life is one dictionary — task, history, turn count. A checkpoint is that dictionary serialized (turned into a string, here with JSON — ch5) and written somewhere that survives the process dying. Save after every turn, and a crash costs you nothing: a new process loads the dict and carries on as if nothing happened.

This run crashes on purpose after turn 2. Then a "brand-new process" starts, finds the checkpoint, and finishes the job:

Look at one detail before moving on: fake_llm here counts observations in the history instead of keeping its own counter. That is not laziness — it is the point. The model is stateless; the history is the state. Because the checkpoint carries the history, the restored run resumes at exactly the right step with no extra bookkeeping. When LangGraph resumes a thread from its checkpointer, this is precisely why it works.

Stage 5 — the red lever: a permission gate

So far every tool has been harmless. Now add one that is not: delete_note. You know from the guardrails chapter and human-in-the-loop that the only enforceable safety boundary is the line in your code where a tool actually executes — a model can be argued, tricked, or prompt-injected out of an instruction, but it cannot argue with an if-statement. So the loop gets a permission gate: a check, before execution, that dangerous tools require the operator's approval.

A browser page cannot pause for keyboard input, so the operator's decision is a variable at the top. Run it once with POLICY = "deny", then change it to "allow" and run again — and watch the model take a completely different path both times:

Tip

That denied-then-report behaviour is not decoration — it is the polite failure you want. The refusal goes back into the history as an observation, so the model knows it was refused and can plan around it instead of retrying blindly. Every time your coding agent asks "allow this command?", you are standing at this exact gate, holding this exact lever.

Stage 6 — the exam sheet: a tiny eval harness

The final organ comes straight from this act's theme: shipping alongside data scientists as an equal. Before an agent change merges, someone must answer "did this make the agent better or worse?" — and the answer cannot be a vibe. An eval is the answer machine: a fixed set of tasks, an expected outcome for each, a grader — a function that scores the agent's answer, here a simple substring check — and a scoreboard. It is a unit test whose assertion tolerates fuzzy output.

This last stage is the full assembly. The loop carries the compactor from stage 3, the checkpoint saves from stage 4, and the gate from stage 5 — and the whole machine is wrapped in an examiner running three tasks. One of them fails on purpose: the scripted model ignores its own observation and misreports a number, which is exactly the kind of regression evals exist to catch. Run it and read the scoreboard:

Sit with that output for a moment. The failing task is the interesting one: the tool returned the right number, the observation went into the history, and the model still wrote the wrong figure in its final answer. No exception was raised. No log line went red. Only the eval caught it — which is why serious teams treat "the eval suite passed" as the definition of done for agent changes, and wire the suite into the merge-request pipeline so a red score blocks the merge.

In the wild

You will meet today's six organs by name in real repositories from now on. A file called loop.py, agent.py, or executor.py is stage 1. A merge request titled "summarize old turns when context exceeds budget" is stage 3. A checkpointer= argument in a LangGraph service, or a table called checkpoints in a schema migration, is stage 4. A YAML block listing tools under requires_approval: is stage 5. And an evals/ directory plus a pipeline stage that runs it on every merge request — with a data scientist arguing in the comments about whether the grader is too strict — is stage 6, and the argument is yours to join now.

Check yourself

1. Stage 2 (trimming) and stage 3 (compaction) both keep the history small. Why is compaction usually the safer choice?

You watched it happen: trimming silently discarded the rent figure, while compaction preserved every number in one summary line. The cost of compaction is an extra summarization call; the cost of trimming is amnesia.

2. In stage 4, why could the restored run resume at exactly the right step without any extra bookkeeping?

The model has no memory between calls; the history is the state. Save the history and you have saved the agent. That single fact is why checkpointers — LangGraph's included — work at all.

3. Why does the permission gate live in the loop code rather than in the system prompt ("never delete anything without asking")?

Instructions influence behaviour; code determines it. The only boundary an attacker cannot talk their way through is the point where your loop decides whether to call the function — which is why every serious harness enforces permissions there.

4. In stage 6, the failing task raised no exception and produced no error log. What does that teach about evals?

The tool was right, the plumbing was right, and the answer was still wrong. Crashes are the easy failures; quality drift is the hard one, and a graded task suite is the only instrument that measures it.

5. To turn your mini-harness into a production agent, what is the single essential replacement?

The fake model was the only simulation on the bench. Everything else — the loop, the memory management, the persistence, the gate, the evals — is structurally what production harnesses run, just hardened and deepened.

Key takeaways

  • A harness is six organs around a model: a loop with tools, memory trimming, compaction, checkpointing, a permission gate, and evals — and you have now built every one of them.
  • Trimming controls cost by deleting knowledge; compaction controls cost by condensing it. When an agent "forgets", suspect the first; when it stays sharp on hour-long runs, thank the second.
  • The model is stateless, so the history is the agent — serialize the state dict every turn and any crash becomes a resume instead of a loss.
  • Safety that must hold lives in code, not prompts: the permission gate is an if-statement at the one boundary the model cannot talk its way through.
  • Agents fail without crashing — only a graded eval suite, run on every change, catches quality regressions before they ship.
  • Frameworks add depth, not mystery: LangGraph's checkpointers, deepagents' backends, and your coding agent's compaction are the same organs you built today, engineered further.
