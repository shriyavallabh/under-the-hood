Kubernetes in plain terms

Five words — cluster, node, pod, deployment, service — and one big idea: you write down what you want, and a tireless control loop makes the world match. If you have ever run an agent loop, you already understand Kubernetes.

~30 minChapter 112 of 133
The story

Meera runs the busiest harbour on the coast. On her desk sits the standing orders book. It does not say "send ship 47 to berth 12 at 3 p.m." — it says things like "at all times, keep three grain ships of this class afloat and working". Her deputies walk the docks around the clock, comparing the book to the water. Storm sinks a grain ship at 2 a.m.? By dawn a replacement is crewed, fuelled, and moored — nobody phoned Meera, nobody panicked. The merchants never notice either, because they don't deal with individual ships. They deliver cargo to Pier 9, the grain pier — a fixed address that quietly hands the work to whichever grain ship is afloat today.

Meera's harbour is a Kubernetes cluster. The docks are nodes, the berths are pods, the standing orders book is a deployment, and Pier 9 is a service. And the deputies walking the docks, forever comparing the book to reality and fixing the difference? That is the control plane — and by the end of this chapter you will recognise it as something you have already built with your own hands.

Why this thing exists at all

Earlier in this act you put your agent inside a container — a sealed, shippable box that runs the same everywhere. One container on one machine is easy. Production asks harder questions: what if the machine dies at night? What if you need three copies for reliability, or thirty for load? What if you want to ship a new version without dropping a request? Answering those by hand — SSH-ing into machines, restarting crashed processes, updating load balancers — is a full-time job for a team of humans.

Kubernetes (often written K8s — K, eight letters, s) is the software that does that job instead. You hand it a fleet of machines and a written description of what you want running, and it keeps reality matching that description, forever, without being asked twice. Azure's managed flavour of it is AKS — Azure Kubernetes Service — which simply means Microsoft runs the harbour master's office for you, and you bring the ships.

The harbour, drawn: five words that carry everything

Kubernetes has a reputation for hostile vocabulary, but the core is only five nouns, and each one maps cleanly onto Meera's harbour. Step through the diagram — each step adds one layer.

The big idea: desired state, actual state, and the loop between them

Here is the one idea that makes Kubernetes Kubernetes, and it should feel like coming home. You do not give Kubernetes commands — "start a container", "restart that one". You give it a desired state: a document describing the world you want. Three replicas of this image, listening on this port, reachable at this name. The cluster's brain — the control plane, Meera's office — then runs a permanent loop: observe the actual state of the harbour, compare it with the book, act to close any gap. Then do it again. Forever. This is the reconciliation loop, and it never, ever clocks off.

Pause and enjoy the echo. In the ReAct chapter you built an agent: a loop that observes, reasons about the gap between where it is and where it wants to be, and acts. The Kubernetes control plane is exactly that shape — an agent whose goal is written in YAML, whose tools are "start pod" and "stop pod", and which never emits a FINAL ANSWER because the job is never finished. The watchdogs-and-self-healing chapter taught the same pattern at harness level: don't fix failures by hand, run a loop that notices and repairs. Kubernetes is that philosophy, industrialised.

Three consequences fall out of this one design choice:

  • Self-healing is free. A crashed pod is just a gap between desired (3) and actual (2), and the loop closes gaps. Nobody is paged for the routine case.
  • Individual pods stop mattering. Ships are replaceable; the standing order is the truth. You never repair a pod — you fix the desired state and let the loop act.
  • Every change is the same verb. Scale up, ship a new version, recover from failure: edit the file, re-submit, converge. One mental model covers everything.

Run the loop yourself. This little simulator is pure Python, but the logic — observe, compare, act, repeat — is faithfully what the control plane does thousands of times a day. Press Run, then try changing DESIRED or making the crash happen on a different pass.

Tip

Notice what you did not write: any code that restarts the dead pod by name, or any special case for scaling up. Both are just "close the gap". That is why operating Kubernetes feels calm once it clicks — every incident and every change funnels into the same loop.

The YAML that carries your wish

So how do you tell the harbour master what you want? You write the desired state in a manifest — a YAML file (old friend from the YAML chapter) describing one Kubernetes object. Here is a real deployment manifest for an agent; click the annotated lines.

deployment.yaml — the standing order, written down
apiVersion: apps/v1
Which dialect of the Kubernetes API this document speaks. Deployments live in the apps/v1 group — you will type this line from muscle memory within a week.
kind: Deployment
What kind of object this file declares. Same file format, different kind, gives you Services, ConfigMaps, Secrets — the whole catalogue.
metadata:
name: support-agent
The deployment's own name — how it appears in kubectl get deployments, and the prefix of every pod it creates.
spec:
spec is Kubernetes-speak for "desired state". Everything under this key is the wish; the control plane's job is to make it true.
replicas: 3
The heart of the standing order: three ships of this kind afloat at all times. Change this number, re-apply, and the fleet grows or shrinks to match.
selector:
matchLabels:
app: support-agent
How the deployment recognises its own pods: by label. This must match the labels in the template below — if they disagree, Kubernetes rejects the manifest. It is the classic first-week error, so check it before you apply.
template:
The blueprint for every pod this deployment launches — a pod description nested inside the deployment. Everything below this line answers "what does one ship look like?"
metadata:
labels:
app: support-agent
The label stamped on each pod at birth — the other half of the selector handshake above, and the hook the Service will use to find these pods.
spec:
containers:
- name: agent
image: myregistry.azurecr.io/support-agent:1.4.2
Which container image to run — pulled from your Azure Container Registry. Shipping a new version of your agent is a one-character-ish edit right here: bump the tag, re-apply, and the loop rolls the fleet over.
ports:
- containerPort: 8000
The port your agent listens on inside its container — the door the Service will knock on.

And here is the pier address — the Service that gives those three mortal pods one immortal name:

service.yaml
apiVersion: v1
kind: Service
metadata:
  name: support-agent
spec:
  selector:
    app: support-agent
  ports:
  - port: 80
    targetPort: 8000

Read it top to bottom: it is a v1 object of kind Service, named support-agent. Its selector says "route to any pod labelled app: support-agent" — the same label the deployment stamps on its pods, which is how the two files find each other without ever mentioning each other. The ports block says: accept traffic on port 80, forward it to port 8000 inside a pod. By default this creates a ClusterIP service — an address reachable only from inside the cluster, which is exactly right for agent-to-agent traffic; opening a door to the outside world is a later chapter's business.

Note

Pods created by a deployment get names like support-agent-6c9d55d8b-x4k2q — the deployment name, a hash, and random letters. The hash comes from an internal bookkeeping object (a ReplicaSet) the deployment manages for you; you will see it everywhere and almost never touch it.

Talking to the harbour: kubectl

kubectl (say "kube-control") is your radio to the control plane. On Azure, one command wires it up — az aks get-credentials hands kubectl the keys to your cluster — and from then on every conversation looks the same: ask questions, or submit desired state. First, look around:

terminal — asking questions
kubectl get nodes
kubectl get pods
kubectl describe pod support-agent-6c9d55d8b-x4k2q
kubectl logs support-agent-6c9d55d8b-x4k2q

Line by line: kubectl get nodes lists the docks — every machine and whether it is Ready. kubectl get pods lists the ships — status, restart count, age; the most-typed command in Kubernetes. kubectl describe pod is the dossier on one pod — its node, its labels, and crucially an Events section that narrates its life ("pulled image", "started container", "killed — out of memory"); when a pod misbehaves, the answer is usually written there. And kubectl logs streams what the container printed — your agent's own voice, the same lines you would see running it locally.

Now the other half: submitting orders. There is really only one verb, and it is beautifully honest about the desired-state model:

terminal — submitting desired state
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml
kubectl get pods -w

kubectl apply -f deployment.yaml does not mean "run this now" — it means "make this file the new desired state"; the control plane diffs your wish against reality and converges. The second apply submits the Service the same way. Then kubectl get pods -w — the -w is watch — prints a line every time a pod changes state, so you can literally watch three ships launch: Pending, ContainerCreating, Running. Scaling to five replicas or shipping a new version is the same ritual: edit the file, apply, watch. You can nudge things imperatively (there is a kubectl scale), but the culture prefers changing the file, because the file is the truth.

2 a.m. in the harbour: a pod dies

Now for the payoff — the moment that used to mean a pager going off. Step through what actually happens when a pod dies at night, and notice that every step is just the reconciliation loop doing its rounds.

1. Steady state: the book matches the water

Three replicas desired, three running. The Service is routing traffic across all three. Everyone is asleep, as it should be.

A VM crashes, or the agent process hits an unrecoverable error and exits. Either way, pod b is gone. The Service quietly stops sending it traffic — callers only ever reach the healthy two.

3. The loop compares: desired 3, actual 2

Control planegap detected: 3 wanted, 2 alive
→
pod a
pod c

Nobody reports the outage; the reconciliation loop observes it on its next pass. The book says three, the water shows two. A gap is a to-do item, not an emergency.

4. A new pod is scheduled on a healthy node

Control planeschedule replacement
→
pod aNode 1
pod cNode 3
pod dNode 1 — pulling image, starting

The control plane picks a dock with room — here, Node 1 takes a second ship — pulls the image if needed, and starts the container. The newcomer gets a brand-new name and address; it is a replacement, not a resurrection.

5. 02:05 — the book matches the water again

Servicesame pier address throughout
→
pod a
pod c
pod dthe new ship

The new pod passes its startup checks, the Service folds it into the rotation, and desired equals actual once more. Total human involvement: zero. In the morning, one pod is younger than the others — the only evidence anything happened.

You can stage this drama yourself, safely — the best five minutes you will spend building trust in the platform:

terminal — sink a ship on purpose
kubectl delete pod support-agent-6c9d55d8b-x4k2q
kubectl get pods

The first line deletes a pod by name — you are the storm now. The second lists pods again, and there it is: three pods still, one of them seconds old with a name you have never seen. The standing order outranks the storm.

Careful

The safety net is the deployment, not the pod. kubectl delete pod is a party trick; kubectl delete deployment support-agent revokes the standing order itself, and the loop dutifully converges on the new desired state — which is zero pods. Desired-state systems are perfectly obedient: make sure the state you declare is the one you actually desire.

Namespaces: districts of the harbour

One more word and the vocabulary is complete. A namespace is a named district within one cluster — separate berths, separate paperwork, same harbour. Teams use them to keep environments and projects apart: your agent might live in both an agents-dev and an agents-prod namespace on one cluster, since names only need to be unique within a district. Every kubectl command takes a -n flag to aim at one — kubectl get pods -n agents-prod — and if you ever see nothing where something should be, check which namespace you are pointed at before you panic. Everyone makes the "wrong district" mistake in week one.

In the wild

You will meet this chapter everywhere. Repositories grow a k8s/ or manifests/ folder full of YAML — you can now read every file in it. Merge requests say "bump replicas to 5" or "roll image to 1.4.3" — both one-line edits to desired state. Runbooks open with kubectl get pods. When a colleague shrugs "it'll get rescheduled" about a dead pod, that is the reconciliation loop they are trusting. And "we run GitOps" means the desired-state idea taken to its logical end: the YAML lives in git, and yet another loop keeps the cluster matching the repository.

Check yourself

1. A pod running your agent crashes at 2 a.m. Who restarts it?

Nobody issues a restart command. The deployment declares "three replicas"; the loop observes two, and closes the gap with a brand-new pod. Self-healing is a side effect of desired state, not a separate feature.

2. What exactly does a deployment declare?

A deployment is Meera's standing orders book, not a launch command: "always three ships of this kind afloat". The network address is the Service's job.

3. Why does a Service need to exist at all, if the pods already have network addresses?

Every replaced pod is a new ship with a new address. The Service is the fixed pier: callers use its stable name, and it forwards to whichever labelled pods are currently running.

4. You need five replicas instead of three. What is the Kubernetes-native move?

Every change is the same verb: edit the desired state, apply it, let the loop close the gap. You never manipulate individual pods or machines by hand.

5. In the harbour analogy, which pairing is correct?

Cluster is the harbour; nodes are docks (real machines); pods are berths for containers; the deployment keeps N afloat; the service is the pier address that stays put while ships churn.

Key takeaways

  • Five words carry Kubernetes: cluster (harbour), nodes (docks — VMs), pods (berths holding one or more containers), deployments (standing orders: "always N afloat"), services (fixed pier addresses while pods come and go).
  • The big idea is desired state: you declare the world you want in YAML, and the control plane's reconciliation loop — observe, compare, act, forever — makes reality match. It is an agent loop that never sleeps and never files a FINAL ANSWER.
  • Self-healing is not a feature bolted on; a dead pod is just a gap between desired and actual, and the loop closes gaps. Replacements arrive with new names — pods are cattle, not pets.
  • Every change is the same ritual: edit the manifest, kubectl apply, watch the cluster converge. Scaling, upgrading, and recovering all funnel into one mental model.
  • In the deployment manifest, selector.matchLabels must match the pod template's labels — and the Service finds its pods by those same labels. Labels are the glue of the whole system.
  • Namespaces are districts of one harbour: same cluster, separate paperwork — and half of all "where are my pods?!" moments are just a missing -n flag.
