The M-by-N problem

Every agent wired to every tool by hand: the cost grows multiplicatively, and through 2024 the whole industry was paying it. Then one open protocol — a USB-C port for AI — collapsed the mess, and every major lab adopted it within a year.

The story

Sometime this decade you moved house, and at the back of a cupboard you found the shoebox. Inside: eleven chargers. The fat-pinned one for a phone that died in 2013. A barrel plug for a camera you cannot locate. Two nearly identical laptop bricks that refuse to charge each other's laptops. Every gadget maker had solved "get power into the device" from scratch, with its own private plug — so every new device meant one more cable to buy, label, lose, and eventually bury in the box.

Then, quietly, the industry agreed on one oval connector. Today you charge your phone, your laptop, your headphones and your neighbour's e-reader with the same cable, and the shoebox stopped growing. Notice what actually happened: nobody's charger got better. No new physics was discovered. What changed was an agreement.

This chapter is about the moment the AI-agent world found its own shoebox — every agent needing hand-made wiring to every tool — and about the one-oval-connector agreement, the Model Context Protocol, that emptied it in roughly a year.

The wall every team hit

Quick recall before the new idea. In the ReAct chapter you built a TOOLS dict — a private menu of functions your loop was willing to run. In the tools chapter you learned what makes a tool good: a clear name, a description the model can act on, a schema for its arguments. And in the LangChain chapters you saw a framework's own tool objects, with schemas generated from type hints. Just last chapter, reading the deepagents source, you watched a whole framework's built-in tools get registered the same way — as code compiled into that one framework.

Now notice something all those chapters quietly stepped around: every one of those tool formats was different. LangChain has its tool objects. The Anthropic API takes tool schemas in one JSON shape; OpenAI's API takes a slightly different one. An enterprise harness team invents a fourth shape in-house. Meanwhile, every capability worth having — a code repository, a ticket system, a database, a browser — sits behind its own API with its own authentication, its own pagination, its own error style.

So through 2024, connecting agents to the world was artisanal work. For each pair of (agent, tool), somebody wrote glue code by hand: fetch the credentials this way, call the API that way, reshape the result into whichever tool format this particular agent expects. That is the M-by-N problem: with M agents and N tools, you need M × N bespoke integrations, because every pairing needs its own adapter. The cost grows multiplicatively — add one agent and you owe N new integrations; add one tool and you owe M.

And it was worse than the count suggests, because each of those 48 glue pieces was written by a different person, at a different time, with different care. The same code-search integration existed five times across the industry — five sets of bugs, five auth implementations to audit, five things that silently broke when the upstream API changed. Mature industries always escape this trap the same way: they stop agreeing on implementations and agree on an interface. Electrical plugs. Shipping containers. HTTP for the web. And for device charging: USB-C.

A new tool ships one server and is instantly usable by every agent in existence. A new agent ships one client and inherits every tool ever published. This flips the economics — and economics, not elegance, is what makes standards win.

A USB-C port for AI

The agreement that did this for agents is the Model Context Protocol (MCP) — an open standard, released by Anthropic in November 2024, that defines one common language for connecting AI applications to tools and data. The USB-C comparison is not marketing garnish; it is structurally exact. USB-C does not care whether the device is a phone or a monitor, or who manufactured the charger — if both ends implement the connector spec, power and data flow. MCP does not care which model, framework, or vendor sits on either side — if both ends speak the protocol, tools flow.

Three words carry the design. An MCP server is a small program that wraps some capability — a database, a browser, a file system — and exposes it in the standard format. An MCP client is the piece inside an agent application that connects to servers and relays tool calls. The host is the agent application itself — a chat app, an IDE assistant, an agent harness — which may hold many clients, one per server it is connected to. The conversation between client and server is always the same agreed handshake: connect, ask the server to list what it offers, call a tool by name with arguments, get a result back.

Two more facts, briefly, because the next chapter dissects the architecture properly. Servers can expose more than tools — the spec also covers resources (readable data) and prompts (reusable templates), and newer spec additions like elicitation and long-running tasks exist too. And the wire between client and server is called a transport: stdio for a local server running as a subprocess, streamable HTTP for a server across the network.

But the property that should make your operator instincts tingle is discovery. The TOOLS dict you built in the ReAct chapter was hardcoded at write time — the agent could only ever reach tools its author typed in. An MCP client instead asks the server what it offers at connect time, and receives names, descriptions, and argument schemas it has never seen before. That is why you can add one entry to a config file, restart nothing else, and your agent suddenly has twenty new abilities. You have done this — every MCP server you have ever wired into a harness worked exactly this way. Now you know what the config entry was actually doing.

If you build tools: build once

Here is what the protocol means for the tool side of the economy. Before MCP, a company with a useful product had to ship an integration per framework per vendor, forever. After MCP, they write one server and are done — every MCP-speaking agent on Earth is now a potential user. The official Python SDK makes the server side almost embarrassingly small. If you build harnesses: consume anything

Now the agent side. A harness that implements one MCP client inherits the entire catalogue of existing servers on day one — file systems, browsers, databases, ticket systems, search engines, thousands of community-built wrappers. That is why harness teams stopped writing tool integrations almost overnight: the build-versus-buy question collapsed when "buy" became "add three lines of config".

You have seen those three lines. This is the shape of an MCP entry in an agent's project config — the same shape whether the tool is a weather toy or a production browser:

.mcp.json — wiring a server into an agent host
{
  "mcpServers": {
    "weather": {
      "command": "python",
      "args": ["weather_server.py"]
    }
  }
}

The host reads this, launches the server as a subprocess, speaks the handshake over stdio, and merges whatever tools come back into the model's menu. No agent code changed. And the frameworks you met in earlier acts plugged in the same way: LangChain's adapter package, langchain-mcp-adapters, provides a MultiServerMCPClient that connects to several servers at once and hands their tools to create_agent as if they were ordinary LangChain tools. The same trick powers deep agents and every serious harness you operate: one client implementation, every server ever written.

How a plug becomes a standard: the adoption year

Standards usually take a decade and a committee. MCP took about a year, and the speed is worth understanding because it tells you the pain was real.

Anthropic released MCP in November 2024 — not as a product but as an open specification, with SDKs and a batch of reference servers anyone could run. The initial reception was polite scepticism: the industry had seen plugin systems before, and every previous one had been a walled garden that benefited its owner. Developer-tool companies wired it in first, because they felt the M-by-N pain most acutely — every editor and coding assistant wanted the same dozen integrations.

Then the dominoes went. In March 2025, OpenAI — Anthropic's most direct competitor — announced MCP support across its products. A rival adopting a rival's standard is the strongest possible signal that a spec has escaped its creator. Google confirmed MCP support for Gemini shortly after, and Microsoft wired it into its Copilot ecosystem and into Windows itself. Meanwhile the community had built thousands of servers — round number, still growing — covering practically every API, database, and SaaS product you can name.

Why so fast? Network effects — the same force that made USB-C inevitable. Every new server made every client more valuable; every new client made writing a server more worthwhile. Once the flywheel spun, not supporting MCP meant your agent shipped with fewer abilities than the competition, and your tool reached fewer agents. Openness mattered too: because the spec was public and the governance open, competitors could adopt it without strengthening a rival's product. The pattern to remember, since Act IX meets it again with agent-to-agent protocols: an open interface wins when the cost of not joining exceeds the cost of joining.

In the wild

You are surrounded by this already. Open any project you operate and look for .mcp.json, or an mcpServers block inside a settings file — each entry is one server, one plug into the port. Claude Code's claude mcp add command, an IDE's "add MCP server" panel, and a harness's MultiServerMCPClient config are all the same act. In merge requests, "expose this as an MCP server" means "stop writing per-framework adapters — build the tool side once". And when a vendor's sales page says "works with any MCP-compatible agent", you can now translate precisely: they implemented the server side of the handshake, so your client — whoever built it — can discover and call their tools with zero custom glue.

Check yourself

1. Your platform team runs 4 different agent harnesses, and the business wants each connected to 12 internal systems. Without a shared protocol, roughly how many bespoke integrations must someone write and maintain?

That is the M-by-N problem: every pairing needs its own glue, so the count is 4 × 12 = 48. With a shared protocol it drops to 4 clients + 12 servers = 16 — and each new addition costs one, not many.

2. Under MCP, who builds the integration for, say, a ticket system?

The tool side implements the server once; the agent side implements a client once. Neither side needs to know the other exists in advance — the protocol in the middle is the agreement that makes strangers compatible.

3. You add one entry to .mcp.json, restart the agent, and it suddenly has twenty-five new browser tools. Which property of MCP makes that possible with zero code changes?

Unlike a hardcoded TOOLS dict, an MCP client learns what a server offers when it connects. The config entry only says how to reach the server; everything else arrives through the standard list-tools handshake.

4. Why did MCP adoption snowball instead of growing slowly like most standards?

The flywheel is the story: more servers make clients more capable, more clients make servers more worth writing. Once OpenAI, Google, and Microsoft joined, an agent without MCP simply shipped with fewer abilities than its competitors.

5. In the USB-C analogy, what plays the role of the connector's physical shape?

The shape is the agreement, not any implementation. SDKs exist in many languages and any of them (or none) will do — what makes two strangers compatible is that both ends speak the same handshake, exactly as any USB-C plug fits any USB-C port.

Key takeaways

  • M agents × N tools = M × N bespoke integrations — the count grows multiplicatively, every adapter is duplicated effort, and the whole industry was paying this tax through 2024.
  • A shared protocol collapses M × N to M + N: each agent implements one client, each tool implements one server, and strangers become compatible.
  • MCP is that protocol — a USB-C port for AI: one agreed handshake (connect, list tools, call tool) over stdio or streamable HTTP, open for anyone to implement.
  • Discovery is the superpower: clients learn a server's tools at connect time, which is why one config entry gives your agent new abilities with zero code changes.
  • Tool builders build once and reach every MCP-speaking agent; harness builders implement one client and inherit thousands of existing servers on day one.
  • Adoption took roughly a year — launch in November 2024, then OpenAI, Google, and Microsoft in 2025 — because network effects made not joining more expensive than joining.
