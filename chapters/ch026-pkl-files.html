PKL files: models in a jar

A trained model is expensive to make and cheap to reuse — so it gets poured into a jar on disk. This chapter is about that jar: what goes in it, how it comes back out, and the one kind of jar you must never open.

~25 minChapter 26 of 133
The story

Every August a grandmother spends two full days turning a crate of tomatoes into sauce — roasting, reducing, tasting, adjusting. It is slow, skilled work, and she is not going to redo it every time someone wants pasta in February. So at the end of those two days she ladles the finished sauce into glass jars, seals the lids, writes the date on each, and lowers them into the cellar. In winter she just lifts a jar off the shelf, pops the seal, and two days of work are there again, exactly as she left them — no re-cooking required.

That is pickling, and it is almost exactly what happens to a trained machine-learning model. The "cooking" — training — is the slow, expensive part. Nobody wants to retrain from scratch every time the model needs to make a prediction. So the finished model is poured into a jar on disk, sealed, dated, and shelved. Later, a different program on a different machine lifts the jar down, pops the seal, and the trained model is back — ready to answer in milliseconds. There is just one catch, and it is the reason this chapter exists: a jar someone else hands you might not contain sauce.

The slow part and the fast part

In the last chapter, the classical toolkit, you watched an sklearn model learn from data with a single .fit() call — and even on a laptop that call can take seconds, minutes, or for large datasets, hours. And from what a model actually is, you already know the payoff of all that work: a bundle of learned numbers (the weights) plus the structure that says how to combine them. Once you have that bundle in memory, making a prediction is cheap — a few multiplications. Re-earning the bundle by training all over again is not.

So the natural move is to save the trained thing to a file once and reload it whenever you need it. But a trained model is not a spreadsheet or a paragraph of text — it is a live Python object sitting in memory, full of nested arrays and settings. To get it onto disk you need to flatten it. Serialization is the general name for that flattening: turning a living object in memory into a plain stream of bytes you can write to a file or send over a network. Deserialization is the reverse — rebuilding the live object from those bytes.

Python ships with a built-in serializer called the pickle module. A file full of pickled bytes conventionally ends in .pkl (you will also see .pickle and .pck — same thing). You met a friendlier, human-readable cousin of this idea back in YAML and JSON: those store simple data as text you can open and read. Pickle is different — it is a binary format, unreadable by eye, but it can save almost any Python object, not just plain data. That power is exactly what makes it both useful and, later in this chapter, dangerous.

What actually goes in the jar

When you pickle a trained sklearn model, you are not saving a table of predictions and you are not saving the training data. You are saving the object itself — the estimator instance, with every learned attribute (the coefficients, the tree splits, the fitted scaler ranges) and every setting it was built with. Recall from names and objects that a Python object is just state plus the class it belongs to. Pickle captures that state and, crucially, a reference to the class — its import path, like sklearn.linear_model.LinearRegression — but not the class's source code. Hold that detail; it explains both of the pain points later.

The core API is two pairs of functions. pickle.dump(obj, file) writes an object to an open file; pickle.load(file) reads one back. (There are also dumps and loads — the s is for "string of bytes" — which skip the file and hand you the bytes directly.)

saving_a_model.py
import pickle
from sklearn.linear_model import LinearRegression

model = LinearRegression().fit(X_train, y_train)   # the slow part

# Write the whole trained object to a jar on disk ("wb" = write bytes):
with open("model.pkl", "wb") as f:
    pickle.dump(model, f)

# Later, in a totally different program, open the jar ("rb" = read bytes):
with open("model.pkl", "rb") as f:
    model = pickle.load(f)

model.predict(X_new)      # milliseconds — no retraining

That with open(...) block is the context manager pattern from earlier — it opens the file and guarantees it closes again. The important line is the one in the middle: pickle.dump pours the living, trained model into model.pkl, and later pickle.load pours it back out intact. The prediction at the end runs on a fully trained model that this program never trained.

joblib: the same jar, built for big arrays

Open almost any real data-science project and you will see the model saved not with raw pickle but with joblib — a small library whose dump and load do the same job but are tuned for objects stuffed with large NumPy arrays (which is what most fitted models are). scikit-learn's own documentation points people to joblib for exactly this reason: it stores big numeric arrays more efficiently and can compress the file. The interface is deliberately familiar.

with_joblib.py
import joblib
from sklearn.ensemble import RandomForestClassifier

model = RandomForestClassifier().fit(X_train, y_train)

joblib.dump(model, "model.pkl", compress=3)   # compress is optional
model = joblib.load("model.pkl")

Two things to notice. First, joblib often produces a file still named .pkl — the extension is just convention, so a .pkl in a repo might have been written by either tool. Second, and this matters: joblib is the same technology under the hood, so it carries the same superpower and the same danger as pickle. Everything this chapter says about the security of pickles applies word-for-word to joblib files.

How a model actually gets served

Steps 4 and 5 are usually a handful of lines at the top of a web service. Read them the way a reviewer would — click the annotated lines until each one is boring.

score_service.py — a scoring endpoint
import joblib
Loading a model at serving time almost always uses joblib (or pickle). Seeing this import near the top of a web service is your signal: "a pre-trained model is about to be opened here."
model = joblib.load("model.pkl")
The jar is opened once, when the process starts — not on every request. From here on, model is the fully trained object, ready to use. This single line is the entire "load" step.
def score(features):
A tiny wrapper the web framework will call for each incoming request. The heavy lifting already happened at load time.
return model.predict([features])[0]
The cheap part: milliseconds. predict wants a list of rows, so we wrap the single row in a list and take element [0] back out. The two-day sauce is being ladled onto a plate.

The jar remembers the kitchen: version skew

Recall the detail from earlier: a pickle stores the object's state plus a reference to its class, not the class's code. On load, Python re-imports that class from whatever version of the library happens to be installed on this machine. If the model was trained with scikit-learn 1.3.2 but the serving machine has 1.5.0, the class it re-imports may have changed shape since — and you get version skew: a mismatch between the library version that created a pickle and the one loading it.

loading_across_versions.py
import pickle

# model.pkl was created with scikit-learn 1.3.2 ...
with open("model.pkl", "rb") as f:
    model = pickle.load(f)

# ... but this machine has 1.5.0, so sklearn warns:
# InconsistentVersionWarning: Trying to unpickle estimator LinearRegression
# from version 1.3.2 when using version 1.5.0. This might lead to breaking
# code or invalid results. Use at your own risk.

Sometimes it is a loud crash. Worse, sometimes it is a quiet warning and the model loads but predicts subtly wrong numbers. This is why serious teams do three things: pin the exact library versions next to the model (that innocent sklearn_version string in the demo is a real habit), keep the whole environment reproducible, and retrain and re-pickle when they upgrade rather than trusting an old jar in a new kitchen. When a merge request says "bump scikit-learn to 1.5", the right reflex is to ask: "and did we re-bake the models?"

Never open a stranger's jar

Now the part that matters most, and the part almost nobody tells beginners. Unpickling a file can run arbitrary code on your machine. This is not a bug — it is how pickle works. To rebuild complicated objects, the format is allowed to say "call this function with these arguments" via a special method named __reduce__, and pickle.load dutifully executes whatever it is told. A malicious file can put a command there, and that command runs the instant you load the file — before you ever call .predict().

why_it_is_dangerous.py (illustration — never run untrusted .pkl files)
import pickle, os

class Trap:
    def __reduce__(self):
        # pickle calls this to learn how to rebuild the object.
        # Whatever it returns is EXECUTED on load.
        return (os.system, ("echo a stranger just ran code on your box",))

payload = pickle.dumps(Trap())   # produces an innocent-looking .pkl

pickle.loads(payload)            # the damage happens HERE, on load,
                                 # not later during prediction
Careful

Picture a bank. A data scientist finds a "pre-trained fraud detector.pkl" on a public model hub, or a vendor emails one over, and loads it on a laptop that has database credentials sitting in its environment. That single joblib.load is enough: the pickle can read those secrets, quietly phone them home, or drop a backdoor — all before a single prediction is scored. The rule is absolute: never unpickle a file you did not create yourself or receive from a fully trusted source over a trusted channel. Treat a .pkl exactly like a downloaded .exe, because in terms of what it can do to your machine, it is one.

Safer jars: ONNX and safetensors

Because "loading is running" is such a sharp edge, the ecosystem built formats that store models as pure data with no code-execution path. ONNX (Open Neural Network Exchange) saves a model's structure and weights as a portable computation graph, so it loads without executing arbitrary Python and can move between frameworks and runtimes. safetensors — now the default for sharing large-language-model weights — goes further and stores only the raw tensor numbers as plain data, with no mechanism to run code at all, which is precisely why the model-sharing world moved to it. Both give up some of pickle's "save literally any Python object" flexibility in exchange for a guarantee that opening a downloaded file cannot execute anything. When a model has to cross a trust boundary, prefer one of these over a pickle.

In the wild

You will meet .pkl and .joblib files living in a models/ or artifacts/ folder — usually kept out of the code repo itself (in a cloud bucket, git-LFS, or a tool like DVC) because they are large binaries, exactly the sort of thing git is bad at storing. You will see a web service that does joblib.load("model.pkl") at startup and a requirements.txt pinning scikit-learn==1.3.2 right beside it — those two files belong together. And in a security review, "untrusted pickle deserialization" is a named, well-known vulnerability class; if you ever see a task that loads a .pkl fetched from the internet, that is the moment to stop and ask where the jar came from.

Check yourself

1. What problem does pickling a trained model actually solve?

Training is the slow, costly step. Pickling captures the finished model as bytes so any other program can reload it and predict in milliseconds — no re-cooking.

2. When you pickle a scikit-learn model, what is stored in the .pkl file?

Pickle saves the live object — its learned attributes and settings — along with a reference to its class by import path. Not the predictions, not the training data.

3. Why can loading a colleague's or a stranger's model.pkl be dangerous?

Pickle rebuilds objects by executing instructions embedded in the file (via __reduce__). A malicious pickle runs its payload on load — before any prediction. Treat a .pkl like an executable.

4. A model saved with scikit-learn 1.3 is loaded on a machine running 1.5 and warns it "might produce invalid results". What is this called?

Because a pickle references its class rather than storing its code, loading re-imports the installed version. A mismatch between the saving and loading versions is version skew — pin versions and retrain on upgrade.

5. You need to share model weights publicly and want no code-execution risk when others load them. Which format fits best?

safetensors stores only the raw tensor numbers as plain data, with no way to run code on load — which is why the LLM world adopted it. Pickle and joblib both carry the execution risk.

Key takeaways

  • Training is slow and reuse is cheap, so a trained model is serialized — flattened to bytes on disk — and reloaded later without retraining. That file usually ends in .pkl.
  • A pickle stores the whole live object: its learned weights, its state, and a reference to its class by import path — not the training data and not the class's source code.
  • pickle.dump/load is the built-in tool; joblib.dump/load is what data scientists actually use for models full of large arrays. Same idea, same risks.
  • Version skew is real: load a pickle with a different library version than it was saved with and you may crash — or silently get wrong answers. Pin versions and retrain on upgrade.
  • Unpickling can execute arbitrary code. Never load a .pkl or .joblib file from an untrusted source; treat it exactly like a downloaded executable. In a bank, that reflex is not optional.
  • When a model must cross a trust boundary, prefer a data-only format — ONNX (structure + weights) or safetensors (weights only) — which cannot execute code on load.
