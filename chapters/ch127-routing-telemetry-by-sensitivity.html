Routing telemetry by data sensitivity

Some traces carry a customer's private details; you cannot pour those into a dashboard the whole company can browse. The answer is not log-everything or log-nothing - it is routing.

~25 minChapter 127 of 133
The story

A hospital, again - but now think about who is allowed to read what. The billing office can see that a patient was treated and how long it took, but the actual diagnosis lives behind a stricter door that only the care team may open. Nobody argues that billing should be blind; they just should not see the sensitive part. So the hospital does not choose between "share everything" and "share nothing." It sorts records by sensitivity and routes each to a place with the right lock.

Your agent's telemetry has the same problem the moment it runs for a large bank. A trace of one conversation might contain a customer's prompt - an account number, a name, a balance. You still want operational data (how slow, how many, how often it failed) visible to everyone who keeps the system alive. But you cannot let that private prompt land on a dashboard the whole company can open. The instinct to just switch tracing off throws away the very data you built this act to get. The grown-up answer is to route: sort each signal by what it carries, and send it to a place with the matching lock.

Telemetry inherits the sensitivity of what it carries

Here is the rule that governs everything in this chapter: a span or a log is exactly as sensitive as the data you put inside it. A latency number is harmless. A span attribute holding the customer's full prompt is not. So the question is never "is telemetry sensitive?" - it is "what did we choose to record, and where should each kind go?"

That reframing gives you three levers, and real systems pull all three together.

Three strategies, best combined

The cleanest defence is not to record the sensitive thing at all: put an id on the span, not the content. If you must keep detail, redact it - scrub or hash the risky attributes in the Collector before they leave your process. And whatever remains sensitive, route it: send that stream to a restricted destination with its own access control, while plain logs and metrics flow to the shared dashboards everyone uses. Do all three and you get full operational visibility without ever exposing a customer's prompt to the whole company.

What the routing looks like in config

All of this lives in the Collector's configuration, not scattered through your agent code. You declare two exporters - two destinations - and a processor that redacts, then a pipeline that wires them together. Read it the way you would in a review:

otel-collector-config.yaml - two destinations, one redactor
processors:
redact/prompts:
A processor whose job is to strip sensitive attributes. Everything flowing through it loses the raw prompt before it can reach a shared store.
remove_keys: ["gen_ai.prompt", "user.account"]
The named attributes that get dropped. This is the redaction lever in action - the harmless timings survive, the risky fields do not.
exporters:
otlp/shared:
The first destination: the shared LGTM org everyone can read. It only ever receives the redacted stream.
endpoint: "shared-collector:4317"
otlp/restricted:
The second destination: a locked org for confidential detail. Same protocol (OTLP), different address and different access control - the per-signal, per-destination lever.
endpoint: "restricted-collector:4317"
service:
pipelines:
traces/safe: { processors: [redact/prompts], exporters: [otlp/shared] }
The safe pipeline: redact, then send to the shared org. This is the stream your whole team watches.
traces/full: { exporters: [otlp/restricted] }
The full pipeline: no redaction, sent only to the locked org. Both pipelines start from the same spans - routing is what makes them different.

Access control: orgs and roles

The lock on the restricted destination is real, not decorative. Dashboards gate visibility by identity plus role. A confidential stream lands in a dedicated org - a separated tenant inside the dashboard - and you can only see it if you have been onboarded and hold the right role; without it, the org does not even appear in your list. Public and internal-classified data flows to a shared default org that operational staff can open. This mirrors the data classification tiers your organisation already uses (public, internal, confidential, and stricter): the classification of what a stream carries decides which org, and therefore which lock, it is allowed to use.

Careful

The most common leak is not a hacked dashboard - it is a well-meaning engineer adding span.set_attribute("prompt", full_text) to debug something, and forgetting to remove it. Now every trace carries the raw prompt to wherever that pipeline goes. Prefer recording an id over content, and let redaction be the safety net, not the only net.

In the wild

You will meet this as an exporter block with two endpoints, a redaction or attributes processor in a Collector config, and a question in review like "which org does this dashboard live in?" A merge request titled "stop exporting prompt text to the shared org" is exactly this chapter. At a large bank, a data-classification label (public / internal / confidential) attached to a service decides where its telemetry may go - and onboarding to a confidential org is a real request with a real approval, not a checkbox.

Check yourself

1. A trace may contain a customer's private prompt, but you still need operational visibility. What is the right approach?

Neither log-nothing nor log-everything works. Sorting by sensitivity and routing each part to a place with the right lock keeps both confidentiality and visibility.

2. Where is redaction of sensitive span attributes best performed?

The Collector sits between your app and the backends, so a processor there can scrub or drop fields before they ever reach a store - the earliest safe point.

3. What is "dual export"?

Dual (or multi) export is the foundation of routing: the same stream can fan out to a shared store and a restricted one, letting you send different content to different locks.

4. What decides whether you can open a confidential org's dashboards?

Access is role-based. A dedicated org for confidential data is gated by role; without onboarding and the right role, it is invisible to you - that is the lock the routing relies on.

Key takeaways

  • Telemetry is exactly as sensitive as what you put in it - so the question is what you record and where each kind should go, not whether telemetry is sensitive.
  • The choice is never log-everything or log-nothing; it is routing: sort each stream by sensitivity and send it to a destination with the matching lock.
  • Three levers, used together: record an id not the content; redact risky attributes in the Collector; and route sensitive streams to a restricted destination.
  • Dual export and per-signal endpoints make routing possible - the same spans can feed a safe pipeline (redacted, shared) and a full pipeline (locked).
  • Access is role-based: confidential data lands in a dedicated org you must be onboarded into, while public/internal data goes to a shared org.
  • The classic leak is a debug attribute holding raw prompt text left in by accident - prefer ids over content and treat redaction as the safety net.
