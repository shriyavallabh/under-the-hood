Shipping a harness as a product

Deploying one agent is a task; shipping the whole harness so other teams can deploy it themselves is a product. Helm charts, managed namespaces, infrastructure-as-code, and golden paths.

~35 minChapter 132 of 135
The story

There is a difference between cooking a meal for a guest and selling a meal kit. The meal you plate once, to your own taste, in your own kitchen. The kit has to work in a stranger's kitchen, with a stranger's oven, following a card that leaves nothing to guess - measured packets, one clear sequence, no step that assumes you already know the trick. The kit is harder to design, but it feeds a thousand kitchens instead of one table.

Earlier in the course you deployed a single agent as a pod - you cooked the meal. This chapter is the meal kit: packaging the whole harness so that other teams can deploy it, in their environment, repeatably, without you in the room. That means turning "the way we run it" into an artifact with knobs, a card that describes the infrastructure, and a paved path that leaves as little as possible to guess. Ship it as a product, not as a favour.

Not one artifact - several

The first surprise: a harness is not a single binary you hand over. It is several pieces that must be deployed together - the runtime that runs agents, the sandbox for code execution, the storage for conversations. Shipping it means packaging all of them and giving a tenant a repeatable way to stand them up. The standard vehicle for that on Kubernetes is a Helm chart.

Where does it run: managed namespace vs your own cluster

A tenant deploys into one of two kinds of home, and the difference shapes what you can offer. Many enterprises hand each team a managed namespace - a reserved, centrally-run slice of a shared cluster. You request a namespace plus CPU, GPU, and quota; you do not manage the nodes underneath. It is simpler and safer, but you cannot add cluster-level things. That matters directly: the hardware-isolated sandbox from earlier in this act often needs a special node pool, and you cannot conjure a node pool inside a managed namespace. Your own cluster gives full control - any node pool you like - at the cost of far more operations work.

Who runs the nodesCentral platform teamYour team

The clever part is that a single Helm chart can deploy into either. The chart is agnostic; a managed namespace is really just a reserved slice on top of the same kind of cluster, so you comply with its specific settings and deploy the same package. Design once, ship to both.

The knobs live in values.yaml

Everything a tenant needs to tune is in one file. Read it the way they will:

values.yaml - the tenant fills this in
runtime:
replicas: 2
How many copies of the agent runtime to run. A resource knob - the tenant sizes it to their load without touching a manifest.
telemetry:
otlpEndpoint: "central-collector:4317"
Where to send OpenTelemetry - the address of the central Collector from earlier in this act. One line wires the whole observability pipeline to the shared LGTM.
sandbox:
enabled: true
Turn code execution on or off. On a managed namespace without the right node pool, a tenant may have to leave this off - the constraint from above, surfaced as a switch.
storage:
postgresUrl: "postgres://conversations"
Where conversations persist - the store behind the time machine. The chart wires the runtime to it; the tenant just supplies the address.
models:
credentialsSecret: "model-keys"
A reference to a secret, not the key itself - recall never baking credentials into an image. The chart mounts it at runtime.

You met this YAML fluency back in YAML and JSON; a values file is just that skill pointed at a deployment. Install with the chart and those knobs become a running harness.

Deploy the infrastructure first, in code

A harness needs infrastructure the chart alone does not create: namespaces, databases, node pools. Define these as infrastructure-as-code (Terraform), for the same reason you version anything - so it is repeatable and reviewable. But there is a sharp lesson here about IaC drift. If someone creates a resource out-of-band - a quick script, a manual CLI command - without updating the infrastructure code's state, then the next time the infrastructure team runs their code, it sees a resource it does not know about and deletes it. Your hand-made sandbox node pool vanishes overnight. The rule that falls out of this: deploy the infrastructure up front, through code, then deploy the harness onto it - never bolt runtime infrastructure on by hand.

Careful

"I'll just add the node pool with one CLI command to unblock myself" is how the drift bites. It works today and is deleted on the next infrastructure apply, taking your sandboxes with it. If it is not in the code, it does not really exist. Add it to the Terraform, apply, then build on top.

Golden paths: the paved road

The best way to ship a product to many teams is to remove choices they should not have to make. A golden path is a paved, opinionated self-service flow: a tenant picks "Python agent," and the platform scaffolds an agent that already uses the harness engine and generates the infrastructure-as-code for the supporting pieces they selected - a managed namespace, a Postgres, whatever they need - all wired together. Instead of a stack of decisions and glue, they get a working, blessed starting point. Walk the path:

Keep the scope honest

A product succeeds partly by what it refuses to do. The harness offering should be: get an agent up and running, and connect it to existing data sources and MCP servers. Creating those data sources and MCP servers - standing up the database a tenant queries, building their internal tool server - is the tenant's job, not the harness's. Blurring that line turns a focused platform into an everything-machine nobody can maintain. Ship the engine and the paved road to it; let tenants bring their own destinations.

In the wild

You will meet a Chart.yaml with a values.yaml beside it, a helm install in a deploy pipeline, and Terraform modules that a review comment insists must run "before the app." An internal developer platform with a "create new agent" button is a golden path. When a platform team at a large bank says "we only support the managed namespace for now," they are drawing the scope line - and the reason code execution might not ship everywhere is usually the missing node pool. "It worked yesterday and Terraform deleted it" is drift, learned the hard way.

Check yourself

1. Why is shipping a harness different from deploying one agent?

Deploying once is cooking a meal; shipping a product is a meal kit that works in a stranger's kitchen - packaged, parameterised, and repeatable, covering the whole multi-part harness.

2. Why might a tenant on a managed namespace be unable to enable the code-execution sandbox?

A managed namespace gives you a reserved slice with quota, not control of the nodes. The special node pool a Kata/microVM sandbox needs is cluster-level, so that feature can be constrained there.

3. What is IaC drift, and how do you avoid it biting your harness?

A resource made out-of-band is invisible to the infrastructure code, which deletes it on the next apply. Declare everything in code and deploy infra first, then the harness on top.

4. What does a golden path give a tenant?

A golden path removes choices teams should not have to make: pick a template and get a working agent plus its infrastructure-as-code, blessed and consistent, instead of a week of glue.

Key takeaways

  • Deploying one agent is a task; shipping the harness is a product - it must work repeatably in other teams' environments, so you package the whole thing with knobs.
  • A harness is several pieces (runtime, sandbox, storage) shipped as a Helm chart, with a values.yaml holding the tenant's knobs (telemetry, credentials, sizes).
  • Managed namespace vs your own cluster is the key fork: the namespace is simple but cannot add cluster-level node pools, which constrains features like hardware-isolated sandboxes.
  • Define infrastructure as code and deploy it up front, then the harness on top - out-of-band resources cause drift and get deleted on the next apply.
  • Golden paths pave the road: pick a template, get a scaffolded agent on the harness plus generated infrastructure, wired together and consistent.
  • Keep scope honest - ship the engine and connections to existing data sources and MCP servers; creating those destinations stays the tenant's job.
