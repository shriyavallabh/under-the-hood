Dunders: Python's secret handshakes

Every "magic" framework object — the model you call like a function, the config that prints beautifully, the session that cleans up after itself — is answering a fixed menu of double-underscore signals. Learn the menu once and the magic evaporates.

~25 minChapter 14 of 117
The story

You land in a city you have never visited and walk into a branch of a hotel chain you know well. Nothing here is familiar — different building, different staff, different language on the signage — and yet everything just works. You tap the front-desk bell and someone appears. You ask "how many rooms are free tonight?" and get a straight number. You hold up two loyalty cards and the receptionist can tell you instantly whether they belong to the same member. Check-in follows the exact ritual you know from every other branch. And no matter how chaotic your stay — even if you bolt at 4 a.m. mid fire-alarm — check-out still happens: the room is released, the minibar is tallied, the bill is settled.

None of this was negotiated on the spot. The chain trains every branch to answer the same small set of standard signals: the bell, the headcount question, the card comparison, check-in, check-out. Python objects work exactly like this. The language approaches your object with a fixed set of standard signals — print it, count it, compare it, call it, enter it — and your class answers by implementing methods with strange double-underscore names. Learn the handshake list once, and every "magic" framework object becomes a hotel you already know how to use.

From classes to handshakes

Quick recap of the classes chapter, because this chapter stands directly on it. A class is a blueprint for objects: it bundles data (attributes) with the functions that work on that data (methods). __init__ is the setup method that runs when a new object is born, and self is simply the object itself, passed into every method so the method knows whose data it is touching. You have, in other words, already met one method with a weird double-underscore name — you just did not know it belonged to a whole family.

That family has a nickname. A dunder — short for double underscore — is any method named like __something__, with two underscores on each side. And here is the single most important sentence in this chapter: you write dunders, but you almost never call them. Python calls them for you, whenever you use built-in syntax.

Each dunder is a hook into a protocol — an agreed signal-and-response, like the hotel chain's bell. When you write len(toolbox), Python quietly turns around and runs toolbox.__len__(). When you write agent(question), Python runs agent.__call__(question). The pretty syntax on the left is a costume; the dunder on the right is what actually executes. Here is the translation table this chapter will teach you to read fluently:

Note

The convention in real code is to use the syntax, never the dunder directly: write len(x), not x.__len__(). The built-in versions do extra safety checks, and reading x.__len__() in a merge request is the Python equivalent of reaching over the hotel desk to ring the staff-side buzzer yourself.

__repr__ and __str__: the two name tags

Every hotel employee has two identities. There is the friendly spoken introduction for guests — "Hi, I'm Asha, I'll be looking after you" — and there is the precise entry in the manager's logbook: employee ID, role, shift, branch. Same person, two audiences.

Python objects carry the same pair. __str__ is the guest-facing introduction: what print() shows, what an f-string stitches in (via a cousin hook called __format__, which defaults to the __str__ text — remember f-strings from chapter 3). __repr__ is the logbook entry: unambiguous, developer-facing, ideally looking like the code that would recreate the object. It is what debuggers show, what error tracebacks show, and — a detail that surprises everyone once — what lists and dicts use for their elements when you print the whole container.

If a class defines neither, Python falls back to a default that is almost aggressively useless: something like <Invoice object at 0x104f2b9d0> — the type and a memory address. If you have ever stared at a production log full of those, you have felt the absence of a five-line __repr__. The practical rule: if you only write one, write __repr__, because when __str__ is missing, everything falls back to __repr__ anyway.

Being equal versus being the same

One more handshake before we run code. Recall from names and objects that variables are name tags tied to objects living in memory. By default, == between two custom objects asks the least interesting question possible: are these two name tags tied to the very same object? Two invoices with identical numbers and identical amounts, created separately, compare as False — because they are two distinct objects in memory, like two printouts of the same document.

Defining __eq__ is how you tell Python what equality should mean for your class — usually "same values in the fields that matter", the way two loyalty cards can represent one member. Watch all three handshakes work, then break them:

__call__: objects that answer the phone like functions

Here is the dunder that makes framework code look most magical. You know from the functions chapter that def creates a function object. Now the curtain-pull: parentheses are not special to functions. something(args) is just syntax that triggers something.__call__(args) — on anything that implements it. Functions are simply objects whose type ships with __call__ built in.

Why do frameworks adore this? Because an object can carry configuration and memory — an API key, a retry budget, a running call count — and still be used with the clean one-word grammar of a function. When PyTorch users write model(batch), model is a full object with millions of parameters inside; __call__ is what lets it answer the phone that simply. The scripted fake LLM below uses the same trick — and it is exactly the pattern you will use in earnest when you build a ReAct loop by hand in Act V.

While we are here, two more handshakes ride along. __len__ lets len() count your object, and __getitem__ is what square brackets ring — the exact same signal that makes the dicts and lists of the collections chapter work. One bonus handshake worth knowing because you will meet it in every codebase: if messages: on your own object rings __bool__, and when that is not defined Python falls back to __len__ — empty means false. That is the entire mystery behind "why does if not results: work on this framework object".

__enter__ and __exit__: check-in with guaranteed check-out

The hotel's deepest promise is not the bell or the headcount — it is check-out. However your stay ends, in triumph or in a 4 a.m. fire-alarm sprint, the room gets released and the bill gets settled. Code needs the same promise constantly: files must be closed, database connections returned, locks released — especially when things go wrong halfway through.

That promise is the with statement, and the pair of dunders behind it. An object with __enter__ and __exit__ is called a context manager — an object that knows how to set something up and, crucially, is guaranteed the chance to tear it down. Step through what Python actually does:

1. The with line runs __enter__

with Session("db") as s:Python calls __enter__()
→
bodynot started yet
__exit__

Python calls the object's __enter__ method — open the connection, acquire the lock, start the timer. Whatever __enter__ returns is bound to the name after as.

2. The indented body runs

__enter__done
→
bodyyour code, using s
__exit__waiting

Your code does its work inside the block. The context manager waits in the wings, holding its promise.

3. Body finishes cleanly: __exit__ runs

bodyfinished
→
__exit__(None, None, None)clean check-out

On a clean finish, Python calls __exit__ with three Nones — "no incident to report". The connection closes, the lock releases.

4. Body crashes mid-stay

bodyraises ValueError!
→
__exit__(type, error, traceback)STILL runs

An exception in the body does not skip check-out. Python still calls __exit__, now handing it the details of what went wrong. Cleanup happens no matter what.

5. __exit__ decides: let the error travel, or swallow it

__exit__ returns Falseerror continues outward
__exit__ returns Trueerror suppressed

If __exit__ returns False (the usual choice), the exception continues on its way after cleanup. Returning True means "handled — nobody else needs to know". Cleanup either way: that is the whole point.

Framework magic, decoded line by line

Now cash in. Below is a screenful of the kind of code that used to read as sorcery — an agent harness doing its morning routine. Click each annotated line and name the handshake yourself before reading the note. Every "how does that even work?" moment is one of the eight signals you now know:

harness_startup.py — every line is a handshake
config = AgentConfig.from_file("agent.yaml")
Plain method call, no dunder — included so you notice the difference. The magic starts on the next line.
print(config)
Rings __str__ (falling back to __repr__). When a config object prints as a tidy summary instead of a memory address, it is because someone wrote that method on purpose. Debuggability is a feature somebody built.
if not config.tools:
Rings __bool__, falling back to __len__ — an empty toolbox counts as False. That is why "truthiness" works on framework collections that are not real lists.
raise SystemExit("no tools configured")
search = config.tools["search"]
Square brackets ring __getitem__. config.tools is not a dict — it is some custom registry class — but it joined the square-bracket protocol, so it feels like one.
agent = Agent(config)
Two handshakes in one: calling the class constructs the object (which runs __init__ from chapter 13). Classes themselves are callable — the parentheses protocol again, one level up.
answer = agent("Summarise this incident report")
The showstopper: agent is an object stuffed with configuration and history, used like a function. __call__. This single line is most of why framework code "looks like magic".
with open("trace.log", "w") as f:
__enter__ opens the door and hands f to the block; __exit__ guarantees the file closes even if the write below fails.
f.write(answer)
for step in agent.history:
The for loop rings __iter__ to ask the object for its items one by one. Custom history objects, result sets, and message stores all loop like lists because they joined that protocol.
print(step)
And each step prints readably — because its author wrote a __repr__. The circle closes.

Notice what changed in you between the top of this chapter and now. The line answer = agent(...) is no longer "somehow the object just works". It is: somewhere in that class there is a method named __call__, and I could go read it. That sentence — "there is a dunder, and I could go read it" — is the skill. Frameworks are not hiding anything; they are speaking a handshake dialect you now understand.

In the wild

Search any serious Python repo for def __ and watch the protocol light up: __repr__ methods on config and message classes (someone cared about readable logs), __call__ on tools and model wrappers, __enter__/__exit__ on database sessions and file-like objects. You will also meet the shortcut: a @dataclass line above a class means Python auto-writes __init__, __repr__ and __eq__ from the field list — a robot intern typing the boilerplate handshakes so humans do not have to. And in merge-request review comments, "can we add a repr here?" translates to: "when this object lands in a production log at 2 a.m., I would like to know what it was".

One review lap before the quiz — flip each card until the back feels obvious:

Check yourself

1. You write len(history) and get 7. What actually happened?

len() is a polite messenger: it turns around and rings the object's __len__ handshake. Any class that implements it — list or not — is countable. That is the whole protocol idea in one line.

2. A production log shows <Invoice object at 0x104f2b9d0> where you needed to see which invoice failed. What is missing?

That unhelpful angle-bracket string is the default __repr__. Five lines of custom __repr__ turn 2 a.m. log archaeology into reading. Readable logs are a feature someone writes.

3. In a framework you see answer = model(prompt), where model is an object holding an API key and settings — not a function. Which handshake makes that line work?

Parentheses are syntax, not a function-only privilege. Any object whose class defines __call__ can be invoked like a function while carrying state and configuration inside — exactly why frameworks use it everywhere.

4. Code inside with open("trace.log", "w") as f: raises an exception halfway through. What happens to the file?

Guaranteed check-out is the entire point of the with statement: Python promises to call __exit__ whether the block finishes cleanly or explodes. That is why files, locks, and database sessions all use it.

5. Two separately created Invoice("INV-2041", 15000) objects compare as False with ==. Why?

Default equality is identity — two printouts of the same document are still two printouts. Defining __eq__ is how you declare that invoices with the same number and amount count as equal.

Key takeaways

  • A dunder is a double-underscore method you write but almost never call — Python calls it for you whenever the matching built-in syntax runs. Each one is a hook into a protocol, a fixed handshake between the language and your object.
  • The core translation table: print(x) rings __str__, logs and containers ring __repr__, len(x) rings __len__, == rings __eq__, x(args) rings __call__, x[key] rings __getitem__, and with rings __enter__ then a guaranteed __exit__.
  • If you write only one name tag, write __repr__ — everything falls back to it, and it is the difference between readable production logs and memory addresses.
  • Default == compares identity, not values; __eq__ is where you define what equality means for your class.
  • __call__ is why framework objects — models, tools, agents — get used like functions while carrying configuration and state inside.
  • __exit__ runs even when the block crashes; that guarantee is why files, locks, and database sessions live inside with statements.
  • Framework code stops looking like magic the moment you can name the handshake: "there is a dunder behind this syntax, and I could go read it."
