Control flow: if, for, while

Every program you will ever read is a straight line, a fork, and a circle. This chapter gives you the fork (if), the walk (for), and the repeat-until (while) — and teaches you to trace a loop by hand, the way the interpreter does.

~25 minChapter 6 of 135
The story

It is 9 p.m. and the night watchman at a housing society clips on his torch and glances at the laminated card by the gate. The card has three rules. If the face at the gate is a resident, wave them through. Otherwise, if they carry a visitor pass, note the flat number and let them in. Otherwise, ring the flat and ask.

Then come the rounds: for every floor in the building, walk its corridor and check each door. The third floor is empty for renovation — he skips it and carries on. On the seventh floor he finds a door ajar, abandons the rest of the round on the spot, and deals with it. And underneath it all runs the shift itself: while the clock has not yet struck six, stay at your post and keep watching.

Nobody ever taught the watchman the phrase "control flow", but he is running all three of its shapes: a decision, a walk over a collection, and a repeat-until. Every program you will ever read — every agent, every pipeline, every script — is built from exactly these three moves. Tonight you learn to read them fluently.

Programs are mostly a straight line — until they are not

For five chapters you have read Python the way you read a recipe: top to bottom, one line after another. You stuck names on objects, glued text together with f-strings, filled the four containers, and in the previous chapter you unmasked YAML and JSON as dicts in costume. Every example so far ran in a straight line.

Real programs almost never do. They pause at forks and pick a path. They repeat the same few lines a thousand times, once per customer, once per document, once per agent turn. Control flow is the set of rules that decides which line runs next — the grammar of decisions and repetition. Python gives you three keywords for it, and they map exactly onto the watchman's evening:

The moveKeywordThe watchman
Decide between pathsif / elif / elseThe laminated gate rules
Do something once per itemforThe rounds — one pass per floor
Repeat until something changeswhileThe shift — keep going until 6 a.m.

Everything else in this chapter — break, continue, range, enumerate, zip — is an accessory to those three. Learn the three shapes and the accessories click into place on their own.

if / elif / else: the gate rules

Here is the watchman's card, written in Python:

gate_rules.py
if face == "resident":
    open_gate()
elif has_visitor_pass:
    log_visit()
    open_gate()
else:
    ring_the_flat()

Read the mechanics slowly, because they never change. After if comes a condition — an expression Python evaluates to a yes or a no. Then a colon, and below it an indented block: the lines that run only when the answer is yes. You already know from chapter 1 that indentation is not decoration in Python — it is the block. Everything pushed four spaces in belongs to that branch; the first line back at the left margin is outside it.

A branch is one of the paths the program might take — each indented block under an if, elif, or else is a branch. elif is simply "else, if": it is checked only when everything above it said no. You can chain as many as you like. else is the optional catch-all at the bottom with no condition at all.

The rule that matters most when reading real code: Python checks the chain top to bottom, and the first condition that says yes wins. Its block runs, and every other branch in the chain — including ones lower down that would also have said yes — is skipped entirely. Exactly one branch runs. If nothing matches and there is no else, none of them run and life continues at the next unindented line.

Tip

Real code is full of one-branch ifs at the top of a file or function, shaped like if not api_key: raise SystemExit("set the API key first"). This is called a guard clause — read it as "bounce early if the setup is wrong". When you see several of these stacked at the top of a function, that is a careful engineer checking the ground before walking on it.

Truthiness: what Python counts as a yes

The condition after if does not have to be a comparison. You will constantly meet lines like if findings: where findings is just a list. This works because every value in Python has a built-in truth value — a property called truthiness, meaning whether a value counts as yes or no when a condition asks. The rule of thumb is beautiful: empty and zero things are no; everything else is yes.

The complete "no" list — the falsy values — is short: False, None, the number zero (0, 0.0), and every empty container: "", [], {}, set(). Absolutely everything else is truthy. Flip these cards until none of them surprise you:

Careful

Truthiness has one famous trap. if count: treats a count of zero the same as "no count at all". If zero is a real, meaningful value — zero retries, zero rupees, row zero — you must ask the precise question instead: if count is not None:. When you see a reviewer comment "this breaks when the value is 0", this is almost always the bug they found.

for: once per item

The watchman does not think "let floor-counter equal one; while floor-counter is less than eleven…". He thinks: for each floor, walk it. Python's for works exactly like the watchman, and this surprises people coming from older languages: it is not a counter, it is a walk over a collection.

the shape to read aloud
for tool in tools:
    print("checking", tool)

Read it aloud as: "for each tool in the list tools, do the indented block". Each trip through the block is one iteration — one pass of the loop. The name tool is the loop variable: an ordinary name, exactly like the ones in chapter 2, that Python re-sticks onto the next item before each pass. You may loop over anything that holds items: a list gives you its elements, a string gives you its characters, a dict gives you its keys (use .items() to get key–value pairs, as you saw in chapter 4), a file gives you its lines.

Three small helpers cover almost every for-loop you will meet in real code:

range manufactures a sequence of numbers when there is no collection to walk — range(5) yields 0, 1, 2, 3, 4. Two things to lock in: counting starts at 0, and the stop value is excluded. range(1, 4) is 1, 2, 3 — it stops one before the number you wrote. That is why the pattern for "n turns, counted from one" is range(1, n + 1).

enumerate hands you each item and its position at the same time, so you never need to maintain a counter by hand. zip walks two (or more) lists in lockstep, pairing the first with the first, the second with the second — and quietly stops at the end of the shorter one. Run all three:

Note

Lines like for i, tool in enumerate(...) use two loop variables at once. That works because enumerate hands back little pairs, and Python unpacks each pair into the two names — the same tuple unpacking you met in chapter 4. Later you will meet comprehensions, which are these same loops folded flat into one line.

Trace a loop by hand

Here is the single most useful skill this chapter can give you: when a loop confuses you, become the interpreter. Take a pencil, track each variable, and play the loop one iteration at a time. Every senior engineer does this — usually in their head, sometimes genuinely on paper during a code review.

Let us trace this loop together. It scans a list of health-check results and stops at the first failure:

first_failure.py
checks = ["ok", "ok", "error", "ok"]
for i, status in enumerate(checks):
    if status == "error":
        print("problem at position", i)
        break
    print("check", i, "passed")
print("done")

Four lines of output from a four-item list — but not one line per item. Tracing by hand is how you predict exactly this, before you run anything. If your prediction and the real output ever differ, the difference is precisely where your misunderstanding lives.

break and continue: the two escape hatches

You just watched break in action; its sibling deserves equal billing. break means leave the loop entirely — no more items, jump to the first line after the loop. continue means abandon this pass only — skip the rest of the body and move straight to the next item. The watchman breaking off his round at the open door is break; the watchman skipping the empty renovation floor and carrying on with the rest is continue.

Both only make sense inside a loop, and both are usually guarded by an if. When you read if not line.strip(): continue in a file-processing script, translate it instantly: "blank line? skip it and move on."

while: repeat until the world changes

A for loop knows its itinerary in advance — these ten floors, these five turns, this list of documents. A while loop does not. It keeps going as long as a condition stays true, and it is the right tool when you cannot know up front how many repetitions you need: retry until the API stops being busy, poll until the job finishes, keep the agent going until it produces an answer.

The mechanics: before every pass — including the very first — Python checks the condition. True? Run the body once, then check again. False? Skip past the loop and carry on. Which reveals the one way while can hurt you: if nothing inside the body ever changes the condition, the check stays true forever, and you have written an infinite loop — a loop that never ends and hangs the program (or, with a paid API inside it, burns money until someone notices). Every honest while loop has a line inside it whose whole job is to move the world toward "false".

Here is the most common while loop in the AI world — retry with backoff — annotated line by line. Click the lines:

retry.py — the shape inside every SDK
def ask_with_retries(question, max_tries=4):
This wraps the loop in a reusable function — the whole topic of the next chapter. For now, just read on to the loop inside.
attempt = 0
The counter starts at zero, before the loop. This is the value the while-condition will watch. Forget this line and Python complains the name does not exist; put it inside the loop by mistake and the counter resets every pass — a classic bug.
while attempt < max_tries:
The condition is checked before every pass, including the first. As long as we have attempts left, go round again. The comparison reads exactly like English: "while attempt is less than max_tries".
attempt += 1
The most important line in any while loop: the one that moves the condition toward false. += means "add to what is already there". Delete this line and the loop never ends — that is the infinite loop.
reply = call_model(question)
The real work — in production this is a paid API call, which is exactly why it lives inside a bounded loop rather than an unbounded one.
if reply == "overloaded":
A fork inside the circle. Loops and ifs nest freely — most real control flow is exactly this sandwich: a loop on the outside, decisions on the inside.
wait(2 ** attempt)
2 ** attempt is 2 to the power of attempt: wait 2, then 4, then 8 seconds. This is exponential backoff — ease off a struggling service more with each failure. You will meet this exact line in nearly every API client ever written.
continue
Skip the rest of the body and jump back up to the while-condition for another try. The renovation floor, remember: skip this pass, keep the loop alive.
return reply
If we got here, the reply was good. Returning leaves the function — which also ends the loop. A successful pass never reaches a fifth attempt.
return "Model stayed busy -- giving up."
Back at the left margin of the function: this only runs if the while-condition went false — all four attempts failed. An honest, bounded surrender instead of an infinite wait.

Note

You will often see while True: in real code — a condition that is always yes, on purpose. It is not a mistake: it means "loop forever, and I will use break (or return) to get out from the inside". Chat interfaces, servers, and agent loops love this shape. And there is a fourth kind of control flow — the emergency exit called an exception — which gets a chapter of its own.

In the wild

Control flow is the most common thing you will ever see in a codebase, so learn its costumes. Guard clauses open most functions: if not api_key: raise .... Ingestion pipelines are for-loops: for doc in documents:. Every SDK hides a retry while-loop with exponential backoff, surfaced as a max_retries setting. Agent harnesses cap their loop with max_turns or a recursion limit — when someone says "the agent hit max iterations", they mean a loop guard fired. And control flow escapes Python entirely: the rules: section of a GitLab CI file is an if/elif chain written in YAML — conditions checked top to bottom, first match wins, deciding whether a job runs at all.

Check yourself

1. A script runs findings = [] and then if findings:. What happens to the if-block?

Every value has a truth value, and empty containers count as no. if findings: reads as "if we found anything" — with zero findings, the block is skipped.

2. A loop is scanning 100 log lines. On line 40 it hits continue. What happens next?

continue abandons the current pass only — the renovation floor. Ending the whole loop and jumping past the remaining 60 lines would be break.

3. What does for i in range(3): give you?

range starts at 0 and the stop value is excluded — three numbers, but not the number three. This off-by-one convention is why "n turns counted from one" is written range(1, n + 1).

4. You need to keep polling a job's status until it reports "finished", and you have no idea how long that will take. Which loop fits?

for walks a known collection; while repeats until a condition changes. "Until it finishes" is a condition, not an itinerary — so it is a while (with a sensible attempt cap, so it cannot spin forever).

5. You have a list of tool names and a separate list of their run times, and you want to print them side by side. Which helper is built for this?

zip(names, times) pairs first with first, second with second. enumerate pairs each item with its position, not with another list — and while the range-and-index approach works, zip says the same thing in half the ink.

Key takeaways

  • All control flow is three moves: fork (if/elif/else), walk a collection (for), repeat until a condition changes (while). Everything else is an accessory.
  • An if/elif chain is checked top to bottom and the first true condition wins — exactly one branch runs, and the rest are skipped.
  • Truthiness: empty and zero things count as no (None, 0, "", [], {}); everything else counts as yes. Use is not None when zero is a legitimate value.
  • for is "for each item", not a counter. range manufactures numbers (stop excluded), enumerate adds positions, zip pairs lists in lockstep.
  • break leaves the loop entirely; continue skips just the current pass. Both usually ride behind an if.
  • Every honest while loop contains a line that pushes its condition toward false — without one, you have an infinite loop, and with a paid API inside, an expensive one.
  • When a loop confuses you, trace it by hand: track each variable, one iteration at a time, and predict the output before running. The gap between prediction and reality is exactly where the misunderstanding lives.
