Strings and f-strings

Every prompt you have ever sent to a model was a string. By the end of this chapter you can slice one, scrub one, and assemble one from a template — which is to say, you can read the raw material of the entire AI industry.

~25 minChapter 3 of 117
The story

A print shop, Tuesday morning. Three hundred wedding invitations due by evening, and the master card sits on the counter: one ornate template with three blanks — the guest's name, the table number, the dinner choice. Nobody is going to hand-write three hundred cards. The operator loads the guest list into the machine, and it slides each name into its blank, prints, trims the ragged edges of the card stock, and drops a crisp invitation onto the pile. Some names arrive from the list wearing stray spaces — "  Priya Sharma " — and the trimming blade tidies them before they ever touch a card. The guest list itself came in as one long comma-separated line, so the first job of the morning was cutting it into individual names.

That print shop is this chapter. The master card with blanks is a template string. The machine that fills the blanks is an f-string. The trimming blade is strip, and the guillotine that cuts the guest list into names is split. And every time an agent sends a system prompt with the user's question slotted into it, this exact print shop runs — thousands of times a minute, in production, on your behalf.

Text is an object, like everything else

Quick recap from the variables chapter: in Python, all data lives as objects floating in memory, and a variable is just a name tag stuck onto one of those objects. Assignment moves the tag; it never copies the object. That picture is about to pay off, because a string — an object that holds a sequence of text characters — is simply one more kind of object to stick a name on.

You make one by wrapping text in quotes. Single and double quotes mean exactly the same thing; Python does not care which you pick, so people choose whichever avoids awkward escaping: 'she said "hi"' reads better than a backslash festival. There is a third form — triple quotes, """like this""" — that can span multiple lines. We are saving it for the finale, because it is the one your prompts live in.

Strings have one personality trait you must absorb on day one: they are immutable — once a string object exists, its characters can never be changed. There is no editing a string in place, ever. Every operation that looks like it modifies text actually manufactures a brand-new string object and hands it back, leaving the original untouched. Try to overwrite one character and Python refuses with a TypeError. This is not a limitation you will fight; it is a guarantee you will lean on — a string you pass to five different functions comes back exactly as you sent it. But it sets up the single most common beginner bug in text handling, which we will walk straight into (and out of) in a moment.

Every character has a numbered seat

Picture a string as a train: one character per seat, seats numbered from the front. Python starts counting at 0, not 1 — you met that convention in chapter 1, and here is where it becomes muscle memory. Asking for a seat by number is called indexing — s[0] is the first character, s[1] the second. Negative numbers count from the back of the train: s[-1] is the last character, s[-2] the one before it, no need to know how long the string is.

Cutting out a whole run of seats is called slicing — s[10:20] takes seats 10 up to but not including 20. The stop number is always excluded, which feels odd for about a day and then becomes deeply convenient. Think of a hotel stay: check in on the 10th, check out on the 20th, and you sleep ten nights — nights 10 through 19. Same arithmetic: 20 - 10 tells you exactly how many characters you got. Leave out a number and Python fills in the obvious default: s[:9] means "from the start", s[21:] means "to the end". A second colon adds a step — s[::2] takes every other character, and the famous party trick s[::-1] walks backwards, reversing the whole string.

The everyday toolkit: five verbs

Strings come with behaviour attached. A method is a function that lives on an object and is called with a dot — text.strip() means "text, perform your strip behaviour". Python strings ship with dozens of methods, but five of them do about ninety per cent of the real-world work, and all five appeared in the print shop:

  • strip() — the trimming blade. Removes whitespace (spaces, tabs, newlines) from both ends. The middle is untouched.
  • split(",") — the guillotine. Cuts one string into many pieces at each separator and returns them as a list, the container you will properly meet in the next chapter. Called with no argument, it splits on any stretch of whitespace — perfect for "give me the words".
  • join(...) — the guillotine in reverse, and the one everyone finds backwards at first: it is a method on the glue, not the pieces. ", ".join(names) reads as "comma-space, weld these together with yourself between each pair".
  • replace(old, new) — find-and-replace, every occurrence, no regrets.
  • lower() / upper() — case-flattening, the standard first move before comparing text, because "Billing" and "billing" are different strings to a computer.

Here is where immutability bites. Because a string can never change, every one of these methods returns a new string — and if you do not catch it in a variable, it evaporates. Writing reply.strip() on its own line does exactly nothing useful: the cleaned string is created, ignored, and thrown away, while reply still points at the messy original. The fix is always the same: reply = reply.strip() — make the new object, then move the name tag onto it, exactly the move you learned in chapter 2.

Careful

"I called .strip() but the spaces are still there" is the most-asked string question on the internet. The method worked perfectly — the result just was not assigned to anything. If a cleanup ever seems to have no effect, check whether the line reads text.strip() instead of text = text.strip().

raw = " Label: Billing \n"
What actually came back from the model: leading spaces, a "Label:" prefix nobody asked for, a trailing space and a newline. Real model output looks like this constantly — instructions are followed approximately.
text = raw.strip()
The trimming blade removes the whitespace from both ends, including that invisible \n. Note the assignment: a NEW string "Label: Billing" is created and the name text tagged onto it. raw still holds the messy original — useful when you need to log what the model really said.
text = text.lower()
Case-flatten to "label: billing" so the next step works whether the model wrote "Label:", "LABEL:" or "label:". Reassigning text again — the name hops to each new object in the pipeline.
text = text.replace("label:", "")
Find-and-replace the prefix with an empty string — which is how you delete text: replace it with nothing. We are left with " billing" (note the leftover space where the prefix used to end).
label = text.strip().upper()
Two methods chained: strip() returns a new string, and .upper() is called directly on that result — a pipeline reading left to right. Final answer: "BILLING", ready to compare against your label set.
menu = "billing,account,general".split(",")
The guillotine: one string becomes a list of three — ['billing', 'account', 'general']. Yes, you can call methods on a literal string directly; it is an object the moment it exists.
pretty = " | ".join(menu)
And back again — the glue string " | " welds the pieces into "billing | account | general". Split and join are perfect inverses; agents use this pair to build tool menus into prompts and to parse structured replies back out.
print(f"model chose {label} from: {pretty}")

f-strings: the fill-in-the-blanks machine

The old way to build a sentence out of pieces was gluing with +: "Hello, " + name + "!". It works, and it is miserable — the quotes multiply, every number needs converting to text first, and by five pieces you cannot see the sentence for the plus signs. Python's answer is the f-string — put the letter f immediately before the opening quote, and anything inside curly braces is treated as live Python, evaluated on the spot, converted to text, and stitched into place: f"Hello, {name}!". The template reads like the finished sentence, which is the whole point.

The part most people underestimate: the braces hold any expression, not just variable names. Method calls, arithmetic, len(), whole function calls — all fair game. f"Customer: {name.strip()}" trims and inserts in one motion. f"total: {price * qty}" does the maths mid-sentence. This is why f-strings took over: the print shop machine does not just copy names onto cards, it can tidy and compute while the card is in the press.

After the expression, a colon opens the format spec — a mini-language that controls how the value is rendered, without changing the value itself. You will meet these constantly in cost logs and dashboards:

Tip

When you are staring at a mysterious variable in any Python codebase, print(f"{thing=}") is the sixty-second diagnostic: it prints the variable's name and its value in one shot, so your log reads retries=3 instead of a bare 3 you cannot place. You will see this in debugging commits everywhere.

Triple quotes: where prompts are born

One piece left from the print shop: the master card itself. Real prompts are paragraphs, not one-liners, and cramming paragraphs into a single-quoted string means littering it with \n escape codes until it is unreadable. The fix is the triple-quoted string — open with """, write as many lines as you like, close with """. Every line break you type is preserved in the string, exactly as written. What you see is literally what the model gets.

Now combine the two endings of this chapter: put an f in front of a triple-quoted string and you have a multi-line template with live blanks — a prompt template, the load-bearing wall of every AI application you operate. The system prompts inside coding agents, the triage prompts in a support bot, the instructions handed to every agent in a pipeline: under the hood, each one is a triple-quoted template with values slotted into braces. Careful

One famous trap: braces are f-string syntax, so a literal brace must be doubled. The moment your prompt asks a model to reply in JSON — f"Reply as {{"label": "..."}}" — you need {{ and }} for every brace the model should actually see, or Python will try to evaluate your example JSON as code and crash with a confusing error. If an f-string with JSON in it will not run, count your braces first.

In the wild

Open any agent repository and head for prompts.py or a prompts/ folder — you will find triple-quoted templates, often screens long, with braces waiting for the day's values. Nearly every model reply in production passes through .strip() before anything else touches it, and parsing code leans on split the way you saw in the ReAct loop chapter's reply.split("FINAL ANSWER:"). Tool menus are assembled with ", ".join(tool_names); billing logs are full of f"{cost:.4f}". One nuance to recognise: LangChain's prompt templates use single braces like {question} in a plain string — no f prefix — because the framework fills the blanks later rather than immediately. Same print shop, delayed printing. When you can tell those two apart in a merge request, this chapter has done its job.

Check yourself

1. You run s = " hi " and then, on its own line, s.strip(). What does s contain afterwards?

Strings are immutable, so strip() cannot edit s — it returns a new string, and since nothing caught it, it vanished. The fix is s = s.strip(). This is the most common string bug in existence.

2. What does "agent"[1:3] give you?

Seats are numbered from 0 — a(0), g(1), e(2), n(3), t(4) — and the stop index is excluded. So [1:3] takes seats 1 and 2: "ge". The length is always stop minus start.

3. Which line correctly welds names (a list of strings) into one comma-separated string?

join is a method on the glue string, not on the list — read it as "comma-space, weld these together with yourself between each pair". Backwards-feeling at first, universal once it clicks.

4. In f"cost: {price:.2f}", what does the :.2f part do?

Everything after the colon is a format spec — it controls how the value is displayed, not the value itself. price keeps its full precision; only the printed text is shaped to two decimals.

5. Your f-string prompt asks the model to reply in JSON and includes an example object — and Python crashes before anything is sent. Most likely cause?

Inside an f-string, every { starts a live Python expression — so literal braces, like the ones in example JSON, must be written {{ and }}. Python was trying to evaluate your sample JSON as code.

Key takeaways

  • A string is an immutable object: no method ever edits it in place — each returns a new string you must assign, or it is silently lost (s = s.strip(), not s.strip()).
  • Indexing counts seats from 0, negatives count from the end, and slices take start up to but not including stop — s[10:20] is exactly ten characters.
  • Five verbs do most real text work: strip trims the ends, split cuts into a list, join (a method on the glue) welds one back, replace swaps text, lower/upper flatten case before comparing.
  • An f-string evaluates any expression inside its braces — method calls and arithmetic included — and a colon opens format specs like :.2f, :, and :.1% that shape the display without touching the value.
  • Triple-quoted strings keep their line breaks, and f"""...""" is the prompt template pattern under every agent you operate; literal braces inside one must be doubled ({{ }}) — the classic JSON-prompt crash.
  • Cleaning a model's scruffy reply is just the five verbs in a pipeline — you now read reply.strip().replace(...).upper() the way a reviewer does.
