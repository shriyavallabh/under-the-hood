Checkpointing at scale: embedded or a service

The time machine needs somewhere to keep its tapes. This is the machinery underneath - the saver interface you swap, the database it sets up, and why a fleet of agents needs a shared checkpoint service instead of a drawer under the counter.

~35 minChapter 134 of 135
The story

A single corner shop keeps its accounts in a ledger in a drawer under the counter. It works perfectly: one shopkeeper, one book, always in reach. Then the shop becomes a chain of forty branches, and the drawer stops working - branch twelve cannot see what branch three wrote, two clerks scribble on the same line at once, and the totals go to war. The fix is not a bigger drawer. It is a central records office that every branch phones, one place that holds the real book and knows how to take forty updates at the same moment without anyone's entry clobbering anyone else's.

Your agent's memory has the same two lives. Back when you built the conversation time machine, saving a checkpoint was easy because you imagined one process holding the state. But production agents run as many copies at once (you saw pods and replicas in the deployment act), and the drawer-under-the-counter approach - state living inside one process - breaks the moment a second copy needs to read it. This chapter is about the drawer and the records office: the embedded checkpointer that suits a single deployment, the shared checkpoint service that a fleet needs, and the one small interface that lets you swap between them without rewriting the agent.

The one interface underneath it all

Whatever stores your checkpoints, it hides behind a single small contract. In the framework you have been meeting, that contract is the checkpoint saver - a base class with a handful of methods: put a checkpoint, get one back, list what is there, and record the in-between pending writes. Some methods are mandatory, some optional. The point of an interface is freedom: write the agent against the contract, and you can slide any storage behind it - memory, a database, or a remote service - without touching a line of agent code. Everything else in this chapter is just different things plugged into that one socket.

The embedded saver: a connection URL and done

For a single deployment, persistence is almost anticlimactic. By default the framework uses the in-memory saver - fine for a test, but state vanishes on restart. To make it durable, you give it one thing: a Postgres or a Redis connection URL. That single config is the whole switch. The library sees a persistent store is configured and routes every checkpoint to it - Redis keeps them as key-value entries by thread id, Postgres keeps them in a small set of tables (checkpoints, their blobs, and their writes).

Two details make it production-ready. First, setup: the first time it connects, the library can run the migration scripts that create those tables for you; if your team pre-creates them, the setup call simply does nothing and moves on. Second, credentials. You never hardcode a database password. The URL carries a placeholder, and at runtime the value is resolved - from a cloud identity token (the deployment's own identity, granted access to the database) or from a password pulled out of a secrets vault and injected as an environment variable by your Helm chart. The secret lives in the vault; the image and the code never see it.

config for a persistent checkpointer
checkpointer:
postgres_url: "postgresql://agent:__VAULT__@db:5432/state"
The single switch. Provide a Postgres (or Redis) URL and the library stops using memory and starts persisting. The __VAULT__ placeholder is resolved at runtime - the real password is injected from a secrets vault, never written here.
run_setup: true
On first connect, run the migrations that create the checkpoint tables. Set it false if your team pre-creates the schema - then setup is a harmless no-op.
# no code change to switch stores - only this URL
Because everything sits behind the saver interface, moving from memory to Postgres to Redis is a config edit, not a rewrite. The agent never knows which store it is talking to.
Note

The migrations create the tables; they do not hand out database permissions. The user in your connection string must already have the right access, and it is good practice to separate the account that runs migrations from the account that runs day-to-day queries. Most platforms deliberately do not bundle database management - teams usually already have a database with roles, run the migration themselves, and hand the agent a connection string that can only read and write the checkpoint tables. Flexibility over a one-size-fits-all box.

Why a fleet needs a records office

Here is where the drawer breaks. An embedded saver that keeps state inside the process is invisible to every other copy of the process. Run one instance and it is fine. Run ten - which any real deployment does, for load and resilience - and instance seven cannot see the conversation instance two is holding, and two writers can race on the same checkpoint. A shared database helps, but some platforms go one step further and expose checkpointing as its own checkpoint service: a small REST API in front of the store that every agent instance calls. It is built to handle concurrent, asynchronous updates and to resolve the race conditions a shared store invites.

The elegant part is that this changes nothing for the agent. The service has a thin client - a checkpoint saver that implements the very same interface, but whose put/get/list methods send HTTP requests to the service instead of talking to a database directly. In the wild

You will see a checkpointer block in a config with a Postgres or Redis URL, a setup() call on first boot that creates the tables, and a vault reference where a password should be. When a colleague says "the in-memory checkpointer is fine for the demo but we need Postgres for staging", that is this switch. When an agent runs as several replicas and someone asks "how do they share conversation state?", the answer is a shared database or a checkpoint service with a REST saver client. And "who runs the migrations versus who runs the queries?" is the separate-accounts question every platform team eventually has.

Check yourself

1. What is the point of the checkpoint saver being an interface (a base class)?

Write the agent against the contract (put/get/list/pending-writes) and any storage can slide in behind it. Changing stores becomes a config edit, not a rewrite.

2. To make an embedded checkpointer durable instead of in-memory, what do you typically provide?

The library sees a persistent store is configured and routes checkpoints to it. Setup can create the tables on first run; credentials are resolved from a vault or cloud identity, never hardcoded.

3. Why does an embedded, in-process checkpointer break when an agent runs as several replicas?

State inside one process is invisible to the others. A shared database or a checkpoint service gives every copy one place to read and write, so any copy can continue any conversation.

4. What is the REST checkpoint saver (the service client)?

Same contract, different wire: the client honours the put/get/list interface but delegates each call to the REST service, so the agent code is identical whether state is local or remote.

Key takeaways

  • Persistence hides behind one small interface - the checkpoint saver (put/get/list + pending writes) - so you can swap storage without changing the agent.
  • The default is an in-memory saver (great for tests, lost on restart); a persistent saver just needs a Postgres or Redis connection URL.
  • Setup runs the migrations that create the checkpoint tables on first connect, or does nothing if you pre-create them; credentials come from a cloud identity or a vault, never hardcoded.
  • Migrations create tables, not permissions - keep the migration account separate from the query account, and most platforms leave database provisioning to the consuming team.
  • An embedded in-process saver cannot be shared across replicas; a fleet needs a shared database or a checkpoint service (a REST API that handles concurrent writes).
  • The checkpoint service has a thin client that implements the same saver interface over HTTP, so scaling from one instance to many changes the config, not the agent.
