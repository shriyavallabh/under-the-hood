Running the model's code without losing the box

When the agent writes code you must actually run, a normal container is not a safe place to run it. The shared-kernel problem, and the hardware-isolated sandboxes that solve it.

~30 minChapter 129 of 133
The story

A bank teller can do a great deal for you - but there is bulletproof glass between you and the vault, and a drawer that passes cash through a slot. Why the theatre? Because the bank does not know you. Most customers are honest; a few are not; and the bank cannot tell which is which at the counter. So it does not rely on trust. It relies on a wall, and hands things through a controlled opening. Every customer, honest or not, meets the same glass.

Now your agent writes a neat Python snippet to crunch a spreadsheet and asks you to run it. You could just exec() it - on your server. But the model can be wrong, the input can be hostile, and generated code is a stranger at the counter. Run it on your host and a mistake or an attack runs with your host's hands: your files, your network, your credentials. The question every coding agent must answer is the bank's question - where does the stranger's code run, and what wall stands between it and everything that matters?

Code execution is powerful - and untrusted

Letting an agent run code is one of the biggest upgrades you can give it. Real coding agents do it constantly: compute, transform files, test a fix, plot a chart. But the code is untrusted - not because the model is malicious, but because you cannot verify in advance that a generated program is safe, and the data it processes might be crafted to make it misbehave. The entire design problem reduces to one question: how strong is the wall between that code and your system? That strength has a name - the isolation boundary - and not all boundaries are equal.

Recall: a container shares the kernel

Back in containers and images, you learned that a container is not a little virtual machine - it is a process on the host, walled off with kernel features (namespaces and cgroups) that give it its own view of the filesystem, its own process list, its own resource limits. That is wonderful for packaging and for keeping well-behaved services out of each other's way. But notice the load-bearing detail: every container on a machine shares the same host kernel. The walls are drawn by that one kernel.

For hostile code, that is the whole problem. Namespaces and cgroups are isolation of convenience, not a security boundary designed to withstand an attacker. If generated code finds a kernel bug or a misconfiguration, it can climb out of the container and onto the host - and because the kernel is shared, escaping one container can mean reaching all of them. A plain Docker container is a fine box for your own service; it is not a safe place to run arbitrary code a model just wrote.

The ladder of isolation

Think of a ladder from "no wall" to "hardware wall," and match the rung to the trust you have in the code.

Running the code in your own process with exec() is no wall at all - the code has everything your program has. A separate OS process or user is a weak wall on the same kernel. A plain container is the shared-kernel box above. The top rung is hardware or VM isolation: give each sandbox its own kernel. microVMs like Firecracker boot a tiny VM in a fraction of a second; gVisor puts a user-space kernel between the code and the host so syscalls never hit the real kernel; Kata Containers run each container inside its own lightweight VM. The rule is blunt: for arbitrary model-generated code, use a hardware/kernel-isolated sandbox, never a plain container.

Defence in depth

Isolation is the wall, but a careful sandbox layers more on top: an egress guardrail that blocks outbound network so code cannot phone home or fetch more tools; a read-only base image; strict CPU, memory, and time limits so a runaway snippet cannot exhaust the machine; and ephemerality so each run starts clean. And before any of that, a policy can simply refuse obviously dangerous code. Here is that first, cheapest layer - a policy that inspects a snippet and declines the risky ones. Press Run.

Careful

A deny-list like the one above is a speed bump, not the wall - a determined attacker can dodge string matching. It reduces noise and catches accidents, but the real safety is the hardware-isolated sandbox with no network behind it. Never let a clever filter tempt you into running untrusted code on a plain container "because we check it first."

In the wild

Every hosted code-interpreter feature you have used runs your code in something like this - Firecracker microVMs power a lot of serverless and code-sandbox products, and services built for agents (e.g. e2b, Modal) exist precisely to offer safe execution as an API. In a design review you will hear "we run tool code in a gVisor sandbox," "the sandbox has no egress," and "why can't the agent pip install?" - the answer to the last being that the network is deliberately cut. When a platform team at a large bank refuses to run generated code on the shared cluster's normal nodes, the shared kernel is exactly why.

Check yourself

1. Why is a plain Docker container not a safe place to run arbitrary model-generated code?

A container's isolation is drawn by the shared host kernel with namespaces and cgroups - convenience, not a security wall. Break the kernel and you are on the host, and potentially in every other container.

2. What is the defining property of a microVM, gVisor, or Kata sandbox that a plain container lacks?

Hardware/VM isolation means the untrusted code does not share the host kernel, so even a successful escape is trapped inside the disposable sandbox.

3. Besides isolation, which of these is a standard extra guardrail on a code sandbox?

Defence in depth: cut egress, limit CPU/memory/time, use a read-only base, and destroy the sandbox after each run. Handing it credentials or host access would defeat the point.

4. A teammate suggests running generated code on a normal container "because we deny-list dangerous strings first." What is the right response?

A determined attacker dodges string matching. Filters reduce noise and catch accidents, but the security boundary has to be a kernel-isolated sandbox, not clever pre-checks on a shared-kernel box.

Key takeaways

  • Running code is a huge capability, but generated code is untrusted - the whole design reduces to how strong the isolation boundary is between it and your system.
  • A plain container shares the host kernel; its namespaces and cgroups are convenience, not a security wall, so an escape can reach the host and other containers.
  • For arbitrary generated code, use hardware/kernel isolation - a microVM (Firecracker), gVisor, or Kata - so each sandbox has its own kernel and an escape stays trapped.
  • Layer defence in depth: block outbound network, use a read-only base, cap CPU/memory/time, and make the sandbox ephemeral so every run starts clean.
  • A policy that refuses obviously dangerous code is a cheap first filter, but it is a speed bump - never a replacement for real isolation.
  • Hosted code interpreters run on exactly this - microVMs and sandbox services - which is why "the sandbox has no internet" and "you can't pip install" are features, not bugs.
