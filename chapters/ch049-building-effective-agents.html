Anthropic's playbook: workflows vs agents

In late 2024 Anthropic published a short essay that quietly became the most-cited text in agent engineering. Its entire message fits in one sentence — use the simplest pattern that works — and it names the five patterns. This chapter decodes all of it.

The story

There is a tailoring shop near a busy market. At the front counter, an assistant handles alterations. She works from a laminated card: measure, pin, cut, stitch, press. Five steps, always in that order, and after each step she checks the work against the card before moving on. She finishes forty jobs a day, every one costed to the rupee, and if a customer ever complains, the owner can point at the card and say exactly what happened at step three.

In the back room sits the master tailor. He gets the jobs nobody can write a card for: "My grandfather's wedding coat — make it wearable again." He opens the lining, discovers what six decades did to the seams, and decides his next move based on what he just found. Nobody can predict how long he will take or what it will cost. But no laminated card could do what he does.

The owner's hiring rule is the whole chapter: never give a card job to the craftsman, and never give a mystery job to the card. The card is a workflow. The craftsman is an agent. Everything else is detail.

The essay everyone cites

Quick recap before the new material. In the ReAct chapter you built an agent by hand: a model, a dict of tools, and a loop that runs until the model says it is done. In the last chapter you saw what that freedom costs — loops that wander, tools misused, budgets burned. Which leaves an uncomfortable question hanging: if agents are this powerful and this failure-prone, when should you actually build one?

In December 2024, Anthropic answered with a short essay called Building Effective Agents, distilled from a year of watching real teams ship LLM systems across industries. It has been cited, screenshotted, and re-taught more than any other text in this field, and its central observation was almost embarrassing: the most successful teams were not using clever frameworks. They were using simple, composable patterns — and most of them were not building agents at all.

The essay's first move is to split all "agentic systems" into two families:

  • A workflow is a system where LLM calls and tools are orchestrated through predefined code paths — your code decides the route, in advance, like the laminated card.
  • An agent is a system where the model directs its own process — it decides, at run time, which tool to use next based on what the last one returned, like the craftsman under the lining.

Notice what the split is really about: who owns the control flow. Not intelligence, not tool count, not model size. If a human wrote the route in code, it is a workflow — no matter how many LLM calls sit inside it. If the model picks the route as it goes, it is an agent — even if it is fifty lines of Python, which, as you proved in chapter 46, is all it takes.

The building block: the augmented LLM

Before naming any pattern, the essay names the atom they are all made of. The augmented LLM is a single model call enhanced with the three augmentations you have already met: retrieval (look relevant documents up and put them in the prompt), tools (let the call request real actions), and memory (carry the right history in, and useful facts out).

Every pattern in this chapter — and honestly, every system in the rest of this course — is just augmented LLM calls arranged differently. A workflow arranges them along a fixed route. An agent puts one inside a loop and hands it the steering wheel. Hold that picture; it makes the next section feel less like five new things and more like five arrangements of one thing.

The five workflow patterns, one tour

Here are the five named patterns from the essay. Step through them; each one is a shape you have almost certainly operated in production without knowing its name.

1. Prompt chaining — a fixed assembly line

Break the task into a fixed sequence of small calls, each consuming the previous call's output. Between calls, ordinary code can inspect the intermediate result — a gate — and stop the run early if it looks wrong. You trade a little latency for a lot of accuracy, because each call gets one easy job instead of one hard one. Classic uses: outline a document, check the outline, then write it; generate copy, then translate it.

2. Routing — a receptionist with three doors

A first call classifies the input; your code then dispatches to a specialised prompt — or a different model — for that category. Each downstream prompt stays clean because it only ever sees its own kind of input. Bonus economics: route easy questions to a small, fast model and save the expensive one for the hard cases. Customer-service triage is the canonical example.

Run several calls at the same time and combine the results in code. Two flavours: sectioning (independent subtasks — one call answers the user while a second screens the exchange for policy problems) and voting (the same question asked several ways — three reviewers scan code for vulnerabilities and anything any of them flags gets a human look). Faster than a chain, and multiple perspectives beat one.

A central model reads the task, decides what subtasks are needed this time, delegates each to a worker call, and stitches the results together. It looks like parallelization, but the fan-out is not written in your code — the orchestrator invents it per task. The essay's example: a code change that touches an unpredictable set of files. This is the pattern's tell: subtasks you cannot list in advance.

One model drafts; a second grades the draft against explicit criteria and returns feedback; the first revises. Loop until the evaluator passes it — or a budget runs out. Worth the cost only when the criteria are clear and revision measurably helps: literary translation with a critic, or search results scored for completeness. Yes, it is a loop — but your code owns it, with a fixed cast and a fixed exit rule, so it is still a workflow.

Note

The evaluator-optimizer surprise generalises: what makes something a workflow is not the absence of loops or of multiple models. It is that a human decided the structure in advance. The moment the model starts choosing which step comes next, you have crossed into agent territory.

Agents: the model takes the wheel

So what does the essay say an agent is? Almost word for word what you built in chapter 46: typically just an LLM using tools in a loop, guided by feedback from its environment. The model plans, acts, reads what actually happened — the tool result, the test output, the error message — and decides its next move from that ground truth rather than from a script. It pauses at checkpoints for human sign-off when the stakes demand it, and it stops when the task is done or a max_turns-style budget runs out.

That autonomy is a straight trade: agents buy flexibility with latency, cost, and variance. The same task might take four tool calls today and nineteen tomorrow. You cannot promise a finance director what a run will cost, and you cannot promise an auditor which path it took — you saw the full catalogue of consequences in the failure-modes chapter. The essay's advice is blunt: use agents for open-ended problems where you cannot script the path, make sure success is verifiable (tests that pass, a booking that exists), and always fit guardrails and budgets.

For the times you do build one, the essay closes with three design principles. Keep the design simple (a loop you can read in one screen). Keep it transparent (show the plan; log every thought and tool call — future-you debugging at 2 a.m. will be grateful). And craft the ACI, the agent-computer interface — the tool names, descriptions, and argument formats the model sees — with the same care a designer gives a human interface, because a confusing tool description is to an agent what a mislabelled button is to a user. You will watch these principles pay off when we read LangChain's packaged loop later.

The golden rule: the simplest thing that works

Here is the sentence the whole essay exists to deliver: find the simplest solution possible, and only add complexity when it demonstrably improves results. In practice that is a ladder, and you climb it one rung at a time, measuring at each rung:

  1. One good call. A single well-crafted prompt (chapter 38) with retrieval and worked examples solves more problems than anyone likes to admit.
  2. One augmented call. Add tools or memory to that call before adding more calls.
  3. A workflow. If the task has knowable steps, wire them with one of the five patterns. Your code keeps the wheel.
  4. An agent. Only when the path genuinely cannot be scripted — and success can be verified — hand the model the wheel.

Most teams, the essay observes, never need rung four. And the ladder explains a pattern you may have seen in real engineering organisations: mature teams often move down it. The prototype was an agent because nobody yet knew the steps; six months of logs revealed that ninety per cent of runs took the same path; someone froze that path into a workflow and the system got faster, cheaper, and dull. Dull is the goal. This is the same instinct as chapter 45's warning about the word "agent" being fashionable: the label is not the value. Boring reliability is the value.

When a workflow beats an agent in a bank

Make it concrete. A large bank needs a system that drafts the monthly suspicious-activity summary for each flagged account: pull the transactions, extract the flagged items, classify each against typology rules, draft the memo, run a compliance-language check, and file it. Six steps, known in advance, identical for every account. Should this be an agent?

No — and for reasons that have nothing to do with model quality:

  • Auditability. When the regulator asks "why did the system say this?", a workflow lets you replay the exact path: same steps, same order, gate results on file. An agent's answer is "the model chose differently that day", which is not a sentence anyone says to a regulator twice.
  • Predictable cost and latency. Ten thousand accounts a month times a fixed number of calls is a number a finance team can sign. Times "somewhere between four and nineteen" is not.
  • Contained failure. If classification misfires, the gate after it stops the run — one step's blast radius, as you saw in the pyrun above. A wandering agent can carry an early mistake through every later step.
  • No decision to delegate. The steps were fixed by policy before any code was written. There is nothing left for a model to decide — so paying an agent's variance for run-time freedom buys the bank nothing at all.

Now walk down the corridor to the same bank's engineering team, staring at a build that fails only on Tuesdays. Nobody can write those steps down in advance — the next step depends on what the logs reveal — and success is machine-verifiable: the tests pass. That is agent territory, in the same building, under the same compliance regime. The boundary is not the industry. It is the shape of the task.

Careful

The expensive mistake is rarely choosing the wrong pattern — it is skipping the ladder. Teams reach for an agent because "agent" is the exciting word, then spend months adding guardrails to claw back the predictability a three-call chain had for free on day one. Before you build the craftsman, check whether a laminated card would do.

In the wild

These five names are now the industry's shared vocabulary. LangGraph's own tutorials are organised around these exact patterns; "orchestrator-workers" is what Claude Code's subagents and deepagents' task tool implement; every "LLM-as-judge" pipeline you have operated is an evaluator-optimizer. In design reviews, listen for the ladder in disguise: "do we actually need an agent here?" is the golden rule being applied, and "we replaced the agent with a pipeline" means a team moved down a rung on purpose — usually to cheers from whoever pays the API bill. The essay's own flagship examples of agents that do earn their keep: customer-support agents (conversation plus refund tools plus a clear resolution signal) and coding agents (open-ended path, tests as ground truth).

Check yourself

1. What is the real dividing line between a workflow and an agent?

It is about who owns the control flow. A workflow can contain many calls, tools, even loops — as long as a human scripted the structure. The moment the model decides what happens next, it is an agent.

2. A task's subtasks cannot be listed in advance — a code change might touch two files or twelve, depending on the codebase. Which workflow pattern fits?

Orchestrator-workers exists precisely for a fan-out you cannot hard-code: a central model invents the subtasks per task, delegates to workers, and synthesises. Parallelization looks similar but its branches are fixed in your code.

3. What is the essay's golden rule?

Climb the ladder one rung at a time: one good call, then an augmented call, then a workflow, then — only for genuinely open-ended, verifiable tasks — an agent. Most production systems never need the last rung.

4. Why does the bank's suspicious-activity report belong in a workflow, not an agent?

The task has no run-time decisions left to delegate, and everything the bank must promise — auditability, quotable cost, contained failure — is exactly what a fixed path provides and a model-directed loop cannot.

5. The evaluator-optimizer pattern contains a loop with two models. Why does the essay still call it a workflow?

Loops do not make agents. The generator and evaluator play scripted roles inside a structure your code owns, with a budgeted exit. Agenthood begins only when the model itself starts directing the process.

Key takeaways

  • The split that matters is who owns the control flow: workflows run on routes your code fixed in advance; agents let the model choose each next step from environmental feedback.
  • Everything is built from one atom — the augmented LLM: a single call with retrieval, tools, and memory attached.
  • The five workflow patterns are prompt chaining (fixed steps plus gates), routing (classify then dispatch), parallelization (section or vote, then aggregate), orchestrator-workers (a model invents the fan-out), and evaluator-optimizer (draft, grade, revise, budgeted).
  • Agents trade predictability for flexibility: variable cost, variable latency, unscriptable paths — worth paying only when the task is open-ended and success is verifiable.
  • The golden rule: find the simplest thing that works, measure, and only climb to the next rung when the simpler one demonstrably falls short. Mature teams often climb down.
  • In regulated settings like a bank, auditability, quotable cost, and contained failure make a workflow the winning answer even when an agent could do the task.
