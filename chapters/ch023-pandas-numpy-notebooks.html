DataFrames and notebooks

The spreadsheet that lives inside Python, the idioms filling every data science merge request, and why a notebook can show you code that never ran.

~35 minChapter 23 of 135
The story

Month-end at a wholesale distributor's office. The owner walks over with three questions: which cities bought the most, which orders were above 500 rupees, and does every order have a matching entry in the customer register. The junior clerk opens the sales ledger — six hundred pages — and starts crawling down it row by row with a ruler and a pencil. He will be here until Thursday.

The senior clerk does something completely different. She unclips the ledger, deals the pages into one pile per city, totals each pile, and staples everything back together. Three minutes. Then she lays the ledger and the customer register side by side and matches them by customer number. She never reads rows one at a time; she operates on whole columns and whole piles. And every trick goes onto a worksheet — step, result, step, result — so anyone can retrace her thinking tomorrow.

The senior clerk is pandas. The pile trick is groupby. The side-by-side matching is merge. The whole-column habit is vectorisation. And her worksheet is a Jupyter notebook, complete with its one famous trap: reading the worksheet in the wrong order.

The other dialect of Python

In the last chapter you toured a full engineering package: modules, classes, entry points, the whole machine. That was the software engineer's dialect of Python — code organised around objects and functions. Act III introduces the second dialect, spoken by data scientists: code organised around tables. Same language, same rules from Act I, but the sentences read differently — fewer loops, fewer classes, long chains of operations on DataFrames.

Three tools define this world. pandas gives Python a spreadsheet-like table you drive with code; numpy is the fast number engine underneath it; Jupyter notebooks are where data scientists spend their day. You will recognise the imports atop nearly every data science file — two famous aliases, via the import ... as trick from the imports chapter:

the first two lines of almost every DS file on Earth
import pandas as pd
import numpy as np

Series and DataFrames: a spreadsheet that lives in Python

A DataFrame is pandas' table: rows and named columns, like one sheet of a spreadsheet, held in memory as a Python object. Each column is a Series — a single labelled column of values sharing one type. Under the hood a DataFrame is close to something from the collections chapter: a dict of equal-length columns sharing row labels. That shared set of labels is the index — the bold row numbers (or names) down the left edge.

Where do DataFrames come from? Usually from a file. pd.read_csv reads a CSV — a plain-text spreadsheet — and hands back a DataFrame in one line:

loading a table
import pandas as pd

orders = pd.read_csv("orders.csv")   # file -> DataFrame, one line

orders.head()      # peek at the first 5 rows
orders.shape       # (6, 3)  -> 6 rows, 3 columns
orders.dtypes      # the type of each column: int64, object (text)...
what orders.head() prints
   order_id    city  amount
0       101    Pune    1200
1       102  Mumbai     450
2       103    Pune    2300
3       104   Delhi     800
4       105  Mumbai    1500

Read that output like a native: the unnamed bold column on the left (0, 1, 2...) is the index; every other column is a Series; and each Series has a dtype — one data type shared by the whole column, such as int64 for whole numbers or object for text. One dtype per column is not a limitation — it is the source of all the speed, as the numpy section will show.

Flip these cards until each feels obvious:

.loc and .iloc: names versus positions

Every table question starts with "which rows, which columns?" — and pandas gives you two selectors that look like twins and behave like strangers. .loc selects by label (index value, column name); .iloc by integer position, counting from zero like a list. Think of a hotel corridor: .loc["301"] asks for the room named 301; .iloc[3] walks to the fourth door, whatever its number plate says. While the index is the default 0, 1, 2... the two agree — which is exactly why the difference bites later, once filtering or sorting makes labels drift away from positions.

the selection idioms
orders.loc[3]                  # the row whose index LABEL is 3
orders.iloc[3]                 # the 4th row by POSITION (from 0)

orders["amount"]               # one column, as a Series
orders.loc[:, ["city", "amount"]]      # all rows, two columns
orders.loc[2:4, "amount"]      # label slice: rows 2..4 INCLUSIVE
orders.iloc[0:3]               # position slice: rows 0,1,2 (end excluded)

Notice the shape of every .loc call: rows first, comma, columns second. Read it that way and dense selection lines stop looking like hieroglyphics. A small aha from the dunders chapter: all this square-bracket magic is just __getitem__ — the protocol you already know, on an object holding a million rows.

Careful

Two classic traps. First, .loc slices include the end label; .iloc slices exclude it like normal Python lists — same-looking code, off-by-one results. Second, to change values, write through .loc in one step (df.loc[mask, "amount"] = 0); chaining selections before assigning (df[mask]["amount"] = 0) may edit a temporary copy, and modern pandas rejects it outright.

Filtering: ask the whole column a question

Here is the mental shift that makes pandas pandas. To find big orders you do not loop over rows asking each "are you big?". You ask the entire column at once, and the answer comes back as a column of True and False — one verdict per row. That Series is a boolean mask, and putting it inside square brackets keeps only the True rows:

boolean masks
mask = orders["amount"] > 500    # a Series of True/False, one per row
orders[mask]                     # keep only the True rows

# usually written in one line:
big = orders[orders["amount"] > 500]

# combine conditions with & (and) and | (or) — parentheses REQUIRED
pune_big = orders[(orders["amount"] > 500) & (orders["city"] == "Pune")]
Tip

Masks need & and | instead of Python's and/or, and every condition needs its own parentheses. Forgetting them produces the "truth value of a Series is ambiguous" error — one of the most-googled messages in data science, now one you can diagnose on sight.

groupby: split, apply, combine

Back to the senior clerk dealing pages into piles. That is precisely what groupby does; the textbook name is split-apply-combine — split rows into groups by a column's value, apply a calculation to each group, combine the answers into a new table. Step through it:

1. Start: one flat table

orders — 6 rowsPune 1200 · Mumbai 450 · Pune 2300 · Delhi 800 · Mumbai 1500 · Pune 650

Six orders, three cities, interleaved. The question: revenue per city.

2. Split into one pile per city

Pune1200, 2300, 650
Mumbai450, 1500
Delhi800

orders.groupby("city") deals every row into a pile keyed by city. No calculation yet — just sorting.

3. Apply a calculation to each pile

Punesum -> 4150
Mumbaisum -> 1950
Delhisum -> 800

Picking the amount column and calling .sum() totals each pile independently. Swap in .mean(), .count(), or .max() for different questions.

4. Combine into a new table

resultPune 4150 · Mumbai 1950 · Delhi 800

The per-pile answers are stapled into a new Series, indexed by city. One line of code, three mental steps.

groupby in code
orders.groupby("city")["amount"].sum()
# city
# Delhi      800
# Mumbai    1950
# Pune      4150

# several answers at once, with named columns:
orders.groupby("city").agg(
    orders=("order_id", "count"),
    revenue=("amount", "sum"),
    biggest=("amount", "max"),
)

That second form — .agg() with named results — fills merge requests. Read it aloud: "group orders by city; per city, count order IDs into orders, sum amounts into revenue, take the max into biggest."

merge: matching two tables by a key

The clerk's second trick: the sales ledger says which customer ordered, the register says who that customer is, and the customer number appears in both. merge lines two tables up on a shared key column and stitches their columns together — SQL's JOIN, or a VLOOKUP that brings back every column at once:

merging two tables
full = orders.merge(customers, on="customer_id", how="left")

on= names the shared key. how= answers the awkward question every join faces: what about rows with no partner on the other side?

how=Which rows surviveFeels like
"inner"Only rows whose key exists in both tablesStrict intersection — unmatched rows vanish silently
"left"Every left-table row; right-side columns become NaN where unmatched"Keep all my orders, decorate where you can"
"right"Every right-table rowMirror image of left
"outer"Every row from both sides, NaN in the gapsFull union — nothing lost, lots of holes

The quiet danger with merges is not syntax, it is row count. If the key is duplicated on the right-hand side, each left row matches multiple partners and the table silently grows — the classic "why did my revenue double?" bug. Careful authors add validate="many_to_one" (or "one_to_one") so pandas errors out if the key relationship is not what they believed. Spotting validate= in a merge request means the author wore a seatbelt.

Vectorised operations: the end of the for-loop

By now you may have noticed what is missing from every example: loops. In the engineering dialect, processing six values means a for loop or a comprehension. In the data dialect, you write the operation once, against the whole column, and it applies to every row simultaneously — the style called vectorised:

whole-column arithmetic
orders["with_gst"] = orders["amount"] * 1.18          # every row at once
orders["big"] = orders["amount"] > 1000               # whole column of booleans
orders["rupees_lakh"] = orders["amount"] / 100_000

This is not just style — it is speed. A Python loop pays interpreter overhead on every row; a vectorised operation hands the whole column to compiled C in one call. The gap is often a factor of fifty, which is why the reviewer comment "vectorise this" is never pedantry:

numpy, in exactly two paragraphs

Where does that compiled speed come from? numpy, the library underneath pandas. Its one big idea is the array: a block of values sharing one type, packed into one contiguous strip of memory — unlike a Python list, a collection of pointers to scattered objects. Uniform and contiguous means numpy can run arithmetic in tight compiled C loops, and that is the engine every pandas column drives. A Series is, in essence, a numpy array wearing an index.

What you need is recognition, not mastery. You will see np.array([...]) building arrays; np.mean, np.where, np.round operating on whole columns; and df.to_numpy() unwrapping a DataFrame when a machine-learning library wants raw numbers. File it away: in the next chapter a "model" turns out to be largely numpy arrays of learned numbers, and neural-network tensors are the same idea with more dimensions.

Jupyter notebooks: the lab bench

Where does a data scientist write all this? Rarely in a plain .py file. They live in a Jupyter notebook — an interactive document made of cells, each holding a chunk of code (or prose) with its result shown underneath. Behind the page sits a kernel — one running Python process that executes whichever cell you run and remembers every variable between runs.

Why do your data science colleagues love them? Their work is a conversation with data: load the table — look at it; filter — look again; group, plot, raise an eyebrow, adjust, replot. A script is a machine you run end to end; a notebook is a lab bench where every intermediate result stays visible. For exploration, that feedback loop is unbeatable.

The lab bench has one famous hazard, born of "the kernel remembers everything" plus "cells run in any order". Step through the classic accident:

1. A fresh notebook, run top to bottom

In [1]price = 100
In [2]total = price * 3
In [3]print(total) -> 300

The bracket numbers are execution counts — the order the kernel ran things. 1, 2, 3: all is well; the kernel remembers price and total.

2. You edit a cell... and do not run it

In [1]price = 200 (edited on screen, NOT run)
In [2]total = price * 3
In [3]print(total) -> 300

The screen says 200. The kernel's memory still says 100. Document and process have quietly diverged.

3. You re-run only the last cell

In [1]price = 200
In [2]total = price * 3
In [4]print(total) -> 300 (!?)

The count jumps to [4] — a gap that betrays out-of-order running. Output 300 sits under code that plainly reads 200 x 3: the notebook is showing results its visible code never produced.

4. A colleague opens the saved file

reader"price is 200, so how is total 300?"
→
stale outputssaved inside the .ipynb

Outputs are saved inside the file, so the lie outlives the kernel. Non-sequential counts and outputs contradicting code are the fingerprints.

5. The professional habit: Restart and Run All

In [1]price = 200
In [2]total = price * 3
In [3]print(total) -> 600

"Restart kernel and Run All" wipes memory and executes every cell top to bottom. If the notebook survives, its story is true. Careful teams do this before trusting any result — and always before committing.

How a notebook shows up in a merge request

The part that matters to you as a reviewer: a notebook's file format, .ipynb, is not Python — it is JSON. Every cell is a JSON object holding its source code, its execution count, and its saved outputs — printed tables, even chart images as long base64 text strings:

what an .ipynb file really is (trimmed)
{
  "cells": [
    {
      "cell_type": "code",
      "execution_count": 3,
      "source": ["orders = pd.read_csv(\"orders.csv\")"],
      "outputs": []
    }
  ]
}

Now recall how git diffs work: line by line, on raw text. Diffing JSON that embeds outputs means a one-line code change can produce a thousand-line diff — and merely re-running a notebook churns the diff too, since counts and outputs change. What lands in the merge request:

a typical notebook diff — one real change, lots of noise
--- a/analysis.ipynb
+++ b/analysis.ipynb
@@ -47,11 +47,11 @@
-   "execution_count": 3,
+   "execution_count": 7,
    "outputs": [
     {
-     "text/plain": ["0    1200\n1     450\n..."]
+     "text/plain": ["0    1416.0\n1     531.0\n..."]
     }
    ],
    "source": [
-    "orders[\"amount\"].head()"
+    "(orders[\"amount\"] * 1.18).head()"
    ]

Reading that like a native: the source lines at the bottom are the only actual change — someone added the GST multiplication. Everything above is churn. Reviewer survival: hunt for the "source" chunks, ignore output noise, and ask "was this run clean, top to bottom?" — non-sequential execution counts answer that. Teams tame the noise by stripping outputs before commit (an nbstripout pre-commit hook), pairing notebooks with .py twins via jupytext, or using notebook-aware diff tools; modern GitLab and GitHub render friendlier notebook views, though raw JSON is still what git stores.

In the wild

Open any data science merge request at a large company and you will recognise everything in it: pd.read_csv at the top; masks stacked with & inside .loc; a merge(...) whose how= deserves a second look; a groupby(...).agg(...) chain carrying the business logic; an .ipynb diff that is 95 per cent output churn around three changed "source" lines. The highest-value review comments here are "this merge can duplicate rows", "vectorise this loop", and "does this notebook run clean top to bottom?" — you can now make all three.

Check yourself

1. After filtering and sorting, the first row of a DataFrame carries index label 42. What do df.iloc[0] and df.loc[0] each give you?

.iloc counts positions like a list; .loc looks up index labels like a dict. They coincide only while the index is untouched — hence the surprises after filtering or sorting.

2. What is orders["amount"] > 500, on its own, before you put it inside square brackets?

The comparison is vectorised: every row answers at once, a whole column of verdicts. Filtering is the separate second step — the mask in square brackets keeps the True rows.

3. In orders.merge(customers, on="customer_id", how="left"), what happens to an order whose customer_id is absent from customers?

A left join keeps every left-table row; missing partners leave NaN holes. Silent dropping is what how="inner" does — which is why reviewers always glance at the how=.

4. Why does a reviewer write "vectorise this" on a for-loop processing a DataFrame row by row?

Columns are uniform numpy arrays underneath, so the whole operation runs once in compiled code. On large tables the loop-versus-vector gap is seconds versus minutes, not taste.

5. A colleague's notebook shows a cell reading price = 200, yet a later output shows totals computed from price = 100. What most likely happened?

The kernel remembers variables across runs, cells can run in any order, and outputs are saved inside the file. Diverging code and outputs plus non-sequential execution counts are the fingerprint; the remedy is "Restart kernel and Run All" before every commit.

Key takeaways

  • A DataFrame is a dict of labelled columns (Series) sharing one index — a spreadsheet you drive with code, usually born from pd.read_csv.
  • .loc selects by label, .iloc by position — rows first, then columns; they diverge the moment an index is filtered or re-sorted.
  • Filtering is a boolean mask: ask the whole column a question, keep the True rows — with &, |, and mandatory parentheses to combine conditions.
  • groupby is split-apply-combine; merge is key-matching between tables, with how= and validate= as the spots where bugs hide.
  • Vectorised column ops run in compiled C via numpy — arrays of one dtype in contiguous memory — which is why idiomatic pandas code has almost no for-loops.
  • Notebooks are lab benches: cells plus a kernel that remembers everything, which enables the out-of-order execution hazard — "Restart and Run All" is the honesty check.
  • An .ipynb is JSON with outputs baked in, so its git diff is mostly noise; in review, find the "source" changes and ask whether it runs clean top to bottom.
