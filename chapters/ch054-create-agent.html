The story

For three weekends you built a washing machine in your garage. A plastic drum, a scavenged motor, a kitchen timer, and a tangle of wiring you soldered yourself. It worked — and you understood every wire, because you had bent every wire. Then last Sunday you walked into an appliance showroom. On the floor sat a gleaming machine: sealed white body, one dial, three buttons. The salesman listed features — auto-restart after a power cut, a child lock, a drum that adjusts its spin to the load. You asked him to open the back panel. And there it was: a drum, a motor, a timer. Bigger, better mounted, properly earthed — but the very same three parts you bolted together in your garage. You did not feel cheated. You felt fluent. You could read that machine, because you had built its ancestor.

In chapter 46 you built the drum, the motor, and the timer of an agent: a history list, a model call, a turn cap. LangChain's create_agent is the showroom machine. Today we open the back panel together.

Three arguments, one agent

Quick recap of what you hand-built in the ReAct-loop chapter, because this whole chapter is a mirror held up to it. You wrote a fake_llm() stand-in for the model, a TOOLS dict mapping names to functions, string parsing that fished ACTION: lines out of the reply, a history list that grew turn by turn, and a max_turns cap so a confused model could not loop forever. Fifty lines, and it worked. Then in the previous chapter you upgraded the tools: the @tool decorator turned plain functions into objects carrying a name, a description, and an argument schema the model can read.

Now LangChain collapses your remaining forty lines into one call:

billing_agent.py
from langchain.agents import create_agent

agent = create_agent(
    model="anthropic:claude-sonnet-4-5",
    tools=[calculate, lookup_invoice],
    system_prompt="You are a precise billing assistant.",
)

result = agent.invoke(
    {"messages": [{"role": "user",
                   "content": "What is 20% GST on invoice INV-42?"}]}
)

print(result["messages"][-1].content)

Read the three arguments and notice you already know all of them. The model is the brain — here named by a provider string, though you can also pass a model object built with init_chat_model from chapter 53. The tools are the @tool-decorated functions from chapter 54 — your TOOLS dict, reborn as a list. The system_prompt is the standing instruction sheet you used to teach the protocol by hand. Loop, parser, registry, stop condition, turn cap: all of it now lives inside create_agent.

Careful

Older tutorials — and there are thousands of them — teach initialize_agent or AgentExecutor. Those are the legacy pre-1.0 APIs and are not how agents are built any more. If a blog post imports either of those names, it is describing a previous era; close the tab and check the current docs. The modern entry point is create_agent, full stop.

What comes back: a compiled graph

Here is the one genuinely new idea in this chapter. create_agent does not return a function, and not quite an ordinary object either. It returns a compiled graph — a small flowchart of nodes and edges, pre-wired and ready to run, built on LangGraph, the engine you will meet properly in Act VII. You do not need graph theory today. You need exactly two facts.

First: the graph has two working nodes. A model node — the step that calls the LLM with the system prompt, the conversation so far, and the tool schemas — and a tools node — the step that executes whatever tool calls the model requested. An edge from the tools node loops back to the model node, and a conditional edge — a fork that picks its route by inspecting the model's reply — decides after every model turn: tool calls present, go to the tools node; no tool calls, finish. Think, act, observe. It is your loop, drawn as a diagram instead of written as a for statement.

Second: because it is a compiled graph, you talk to it through the graph interface. You invoke it with a dictionary holding a "messages" list, and you get back the same shape — the full transcript, with the agent's final reply as the last message. Watch one question flow through it:

Your TOOLS[name](arg) line lives here. Each result is wrapped as a ToolMessage — the observation — appended to state, and the edge carries everything back to the model node for another think.

The final state — every message from your question to the closing answer — comes back from invoke. Your code reads result["messages"][-1].content and moves on with its day.

Note

The word "compiled" here does not mean compiled like C code. It means the graph's nodes and edges have been checked and locked into a runnable object. You will build and compile graphs yourself in chapter 59 — at which point create_agent will feel like a factory that runs those chapters for you.

Here is the promised mapping — the exact table to hold in your head whenever framework code feels opaque. Every row is one part of your garage machine and where it now lives inside the showroom one.

One honest footnote on the recursion-limit row: the limit counts graph steps, and each visit to the model node or the tools node is one step. So the default of 25 buys you roughly a dozen think-act rounds, not twenty-five. When an agent exhausts it, the run raises a GraphRecursionError — the framework's politer version of your "Gave up: hit the max-turns limit" string, and the same guard against the runaway-cost incidents you met in chapter 49.

What you get free

If the packaged loop were only your fifty lines with nicer names, it would not be worth a chapter. The value is in the parts the showroom fits as standard — the child lock and the auto-restart.

Structured tool calls. No string surgery, ever. The model requests tools through the provider's native tool-calling channel (chapter 36), arguments arrive parsed against the schema your @tool decorator generated, and results travel back as typed ToolMessage objects. A whole class of "the parser missed the bracket" bugs simply does not exist here.

Retries and self-correction. Two layers. At the transport layer, the underlying model clients retry transient network and rate-limit failures, so a flaky connection does not kill a run. At the loop layer, when the model requests a tool with arguments that fail schema validation, the loop hands the error text back to the model as a tool message — so the model can read the failure, fix its arguments, and try again. Your hand-rolled loop would have crashed with a traceback; this one lets the conversation route around the pothole. (An exception raised inside the tool's own body still propagates by default — feeding those back to the model too is a small middleware, next chapter.)

Streaming. Because the agent is a graph, you can swap invoke for stream and watch the run unfold live — the state after each node, or token-by-token output for a typing-indicator UI, the craft you learned in chapter 41. You wrote none of the plumbing.

Bookkeeping and growth slots. Message history is managed as proper typed objects, appended correctly on every turn. And the factory has sockets waiting for parts you have not needed yet: a checkpointer argument that persists state so conversations survive restarts (chapter 62), structured final answers in the spirit of chapter 37, and a middleware list for intercepting the loop — which is the entire subject of the next chapter.

When to drop down to LangGraph

create_agent packages exactly one shape: a single model, a set of tools, looping until done. That shape covers a remarkable share of real work — but not all of it. The signals that you have outgrown the box are worth memorising, because you will feel them long before you can name them.

  • State beyond messages. You need the agent to carry a running total, a set of retrieved documents, or an approval flag alongside the conversation — not just a transcript.
  • Deterministic stages. Part of the job should always happen, in a fixed order — fetch, then summarise, then file — with no model discretion about the route.
  • More than one brain. A researcher agent handing off to a writer agent, or a supervisor routing between specialists.
  • Forks that are not "tool calls or not". Branching on a quality score, a category, a budget check — any condition of your own design.
  • Pauses for people. Stopping mid-run at a specific point for human sign-off before something irreversible happens.

When those appear, you graduate from buying the machine to laying out the drum and motor yourself — defining your own nodes, edges, and state with LangGraph, which is all of Act VII. And here is the comfort: it is not a migration. create_agent is already a LangGraph graph — same state shape, same invoke and stream, same checkpointing. Dropping down just means you draw the flowchart yourself instead of accepting the factory's default two-node one.

Tip

Before you reach for raw LangGraph, check the middle rung. Many "I need my own loop" urges — logging every model call, trimming old history, blocking a dangerous tool, swapping models mid-run — are hooks into the existing loop, not new graphs. That middle rung is middleware, and it is the very next chapter.

In the wild

In any modern LangChain repository, search for create_agent( — you will usually find it once, near the top of a file called agent.py or graph.py, with the three arguments you now know by heart. Nearby you will see the state shape {"messages": [...]} going into invoke or stream, and result["messages"][-1].content pulling the answer out. In merge requests, "bumped the recursion limit to 50" means someone's agent legitimately needed more think-act rounds — or is quietly misbehaving, and it is worth asking which. And when a teammate's stack trace ends in GraphRecursionError, you can now say the most useful sentence in agent debugging: "your model never produced a reply without tool calls — find out why it thinks it isn't done."

Check yourself

1. What does create_agent(model, tools, system_prompt) actually return?

It returns a compiled LangGraph graph — a pre-wired model-node/tools-node loop. You drive it through the graph interface: invoke (or stream) with a messages dictionary, and the full transcript comes back in the same shape.

2. In your ch045 loop, the model signalled completion by writing FINAL ANSWER. How does the packaged loop know the agent is done?

The stop signal is structural, not textual: a reply with an empty tool-calls list means the model has nothing left to request, so the fork after the model node routes the run to END. Same idea as your marker, minus the string matching.

3. What became of the TOOLS = {"calculate": calculate} dict from your hand-rolled loop?

The registry still exists; you just stopped writing it. The tools node maps each requested tool name to the matching decorated function and runs it — and the model still cannot reach anything outside that list, which is where agent security continues to live.

4. A run dies with GraphRecursionError. What is the closest ancestor of this moment in your ch045 code?

The recursion limit is max_turns in framework clothing: a hard budget of graph steps (default 25) so a model that never stops requesting tools cannot loop and bill forever. The error is the framework saying "gave up" out loud.

5. Which situation is a genuine reason to drop from create_agent to hand-built LangGraph?

Many tools and streaming are both standard equipment on the packaged loop. You outgrow the box when the job stops being "one model plus tools, looping": custom state fields, always-run pipeline stages, multiple cooperating agents, or forks of your own design. Even then, check middleware (next chapter) first.

Key takeaways

  • create_agent(model, tools, system_prompt) is your fifty-line ReAct loop, packaged: three arguments in, a compiled graph out, driven with invoke({"messages": [...]}).
  • Inside the box are two nodes and a fork: a model node (your fake_llm call, made real), a tools node (your TOOLS dict, dispatched for you), and a conditional edge that ends the run when a reply carries no tool calls (your FINAL ANSWER marker, made structural).
  • The recursion limit (default 25 graph steps) is max_turns reborn; exhausting it raises GraphRecursionError instead of billing you forever.
  • Standard equipment you no longer write: structured tool calls with validated arguments, transport retries, invalid-argument errors fed back so the model self-corrects, streaming, and typed message bookkeeping.
  • Drop to hand-built LangGraph only when the job stops being "one model plus tools in a loop" — custom state, deterministic stages, multiple agents, custom forks. It is a step down the same ladder, not a migration: create_agent already is LangGraph.
  • Most customisation urges are satisfied one rung earlier, by hooking the existing loop — that is middleware, next chapter.
