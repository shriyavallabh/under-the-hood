The four containers

Lists, dicts, tuples, sets — four ways to hold a bunch of things, each built for a different job. Learn to tell them apart on sight and half of every agent codebase stops being mysterious, because agent code really is dicts all the way down.

~30 minChapter 4 of 135
The story

It is 6 a.m. at a small courier depot, and the supervisor is setting up the day with four pieces of stationery. First, the run sheet: today's delivery stops written top to bottom, in the exact order the van will drive them — and the same society appears twice if two parcels are going there. Second, the parcel register: a fat book where every entry sits under a tracking number, so anyone can flip straight to PN-4471 without reading from page one. Third, the sealed manifest: once the depot stamps it, its contents are fixed — nobody edits a sealed manifest; if something changes, they issue a new one. Fourth, the gate board: the guard chalks up each vehicle number he has seen today. Writing the same number twice changes nothing, and one glance answers "has this van already been in?"

Four jobs, four shapes of stationery. Python ships exactly these four shapes as built-in containers — list, dict, tuple, set — and choosing between them is the first genuinely architectural decision you will learn to read in any agent codebase.

Four shapes, four jobs

Quick recap before the new material. In chapter 2 you learned that every value in Python is an object, and a variable is just a name stuck on one. In chapter 3 the object was a string — a sequence of characters you could index and slice. A container is simply an object whose job is to hold other objects: strings, numbers, booleans, even other containers. The name-on-object rule does not change; the object just got roomier.

Python gives you four built-in containers, and each answers a different question:

  • A list is an ordered, editable sequence — the run sheet. It answers: what comes first, second, third?
  • A dict (short for dictionary) is a collection of labelled values — the parcel register. It answers: what is filed under this name?
  • A tuple is an ordered sequence that cannot be changed after creation — the sealed manifest. It answers: what was fixed at the moment of writing?
  • A set is an unordered collection with no duplicates — the gate board. It answers: have I seen this before?

Flip these cards until each one feels obvious. This vocabulary carries the whole chapter — and most of the course.

The list: when order is the point

You create a list with square brackets, and Python keeps the items in exactly the order you wrote them:

the run sheet
run_sheet = ["Kothrud", "Aundh", "Baner", "Aundh"]

Notice "Aundh" appears twice — lists do not mind duplicates, because two parcels really are going to the same suburb. Reading works exactly like the string indexing you met in chapter 3: run_sheet[0] is the first stop, run_sheet[-1] is the last, and slices like run_sheet[1:3] carve out a portion. Positions start at zero, same as before — one rule, learned once, reused everywhere.

Updating comes in two flavours. You can replace an item by position — run_sheet[0] = "Karve Nagar" — or grow the list at the end with run_sheet.append("Hinjewadi"). That .append() is worth a pause, because it is the most consequential method in agent engineering: every time an agent "remembers" a new message, somewhere a line of code appended a message to a list. You saw it in spirit in the depot; you will see it literally in a few minutes.

Iterating — visiting each item in turn — looks like this:

visiting every stop
for stop in run_sheet:
    print("deliver to", stop)

Read for stop in run_sheet: as plain English: "for each stop in the run sheet, do the indented bit". Loops get their own full treatment in chapter 6; for now that reading is all you need.

The dict: when labels are the point

The run sheet is great until someone phones asking about parcel PN-4471. Scanning an ordered list for one entry is slow and awkward — what you want is the register, where everything is filed under a label. That is a dict. You create one with curly braces, writing each entry as key: value:

one parcel in the register
parcel = {"tracking": "PN-4471", "weight_kg": 2.5, "cod": True}

Reading uses square brackets again, but with the label instead of a position: parcel["tracking"] gives you "PN-4471". Updating is the same shape in reverse — parcel["weight_kg"] = 2.7 overwrites the value under an existing label, and parcel["fragile"] = False files a brand-new entry, no ceremony needed. One label, one value: if you write to the same key twice, the second write wins.

Iterating over a dict usually means walking its labelled pairs with .items():

reading the whole register entry
for key, value in parcel.items():
    print(key, "=", value)

There is also .keys() for just the labels and .values() for just the contents, but .items() is the one you will meet most in real code.

.get() and the KeyError you will meet in production

Here is the sharp edge. Ask a dict for a label it does not have — parcel["insured"] — and Python does not shrug and return nothing. It crashes, loudly, with a KeyError: an error meaning "you asked for a key that is not in this dict". If you have ever watched an agent run die with a traceback ending in something like KeyError: 'content', you have already met it — some code assumed a label would be present, and one day it was not.

The polite alternative is .get(). Where square brackets demand, .get() asks: parcel.get("insured") returns the value if the key exists and None (Python's "nothing here" object) if it does not — no crash. Better still, you can supply a fallback: parcel.get("insured", False) means "give me the insured flag, and if there is no such entry, just say False".

Careful

Neither spelling is "correct" — they encode different beliefs. parcel["tracking"] says this key must exist, and if it does not, I want the loud crash so I find out. parcel.get("insured", False) says this key is optional, and here is the sensible default. Reviewers read the choice as a statement of intent. Blindly replacing every bracket lookup with .get() does not fix bugs — it hides them, because the crash that would have pointed at the real problem becomes a silent None that explodes three functions later.

The tuple: sealed once written

A tuple looks like a list wearing round brackets — depot = (18.5074, 73.8077) — and reading it works identically: depot[0] is the latitude. The difference is a single, deliberate restriction: a tuple is immutable, meaning it cannot be changed after creation. Try depot[0] = 19.0 and Python refuses with an error. No append, no overwrite, no delete. Lists and dicts, by contrast, are mutable — editable in place.

Why would you ever want the handcuffs? For the same reason the depot seals a manifest: some values are facts, not workspaces. Coordinates, an RGB colour, a (host, port) pair, a date-of-birth triple — things that should never be quietly edited by a distant line of code. Using a tuple is how Python programmers write "this is fixed" in a way the language itself enforces. When you see round brackets in code, hear a stamp coming down.

The set: already seen it

The gate board has two defining behaviours: chalking up a number that is already there changes nothing, and checking whether a number is on the board takes one glance no matter how full it gets. A set gives you both. You write it with curly braces but no colons — seen = {"MH12AB1234", "MH14XY9876"} — and you get an unordered bag where duplicates simply cannot exist. seen.add("MH12AB1234") when that number is already present does nothing at all, silently. And the membership check "MH14XY9876" in seen answers True or False effectively instantly, even with a million entries — dramatically faster than searching a list item by item.

The trade-off: sets have no order and no positions. There is no seen[0], because "first" means nothing on the gate board. If you catch yourself deduplicating a list by hand, or checking in against a long list inside a loop, a set is almost always the answer someone will suggest in review.

Choosing the right container

Nobody memorises this as theory; you choose by the question your data needs to answer. This table is the whole decision:

You need to…Reach forBecause
keep things in order, add and remove freelylistordered, editable, duplicates allowed
look things up by name or labeldictdirect key-to-value access, no scanning
bundle values that must never changetupleimmutability is enforced by the language
deduplicate, or ask "have I seen this?" fastsetno duplicates, near-instant membership tests

Two honest observations from the field. First, lists and dicts do perhaps ninety-five percent of the work in agent code; tuples and sets are the specialist tools you are glad to recognise when they appear. Second, the container choice is rarely wrong in a way that crashes — it is wrong in a way that makes code slow or confusing, which is why reviewers care about it more than the interpreter does.

Nesting: containers inside containers

Here is the move that unlocks real-world code: containers hold objects, and containers are objects — so containers can hold containers. This is called nesting: putting a list inside a dict, a dict inside a dict, a dict inside a list, as deep as the data demands. The parcel register does not store one fact per tracking number; it stores a whole labelled bundle. In Python, that is a dict of dicts — and it is the default shape of every configuration file, every agent registry, every API response you will ever read.

Click through the annotated lines below. The last two lines — the chained lookups — are the skill that makes unfamiliar code readable.

Why every agent transcript is a list of dicts

Now the payoff — the reason this chapter sits this early in the course. Every chat model API on Earth accepts the conversation as the same data shape: a list of dicts. Why that pairing, and not something else? Because a conversation needs both of the superpowers you just learned. It needs order — who spoke first matters enormously, and the model reads the transcript top to bottom — so the outer container is a list. And each message needs labelled parts — at minimum a "role" saying who is speaking (system, user, or assistant) and a "content" carrying the words — so each item is a dict. Step through it:

1. The transcript starts as a list holding one dict

messages[0]role: system · "You are a billing assistant."

The standing instructions go in first. The dict's two labels do two jobs: "role" says who is speaking, "content" carries the words.

2. The user's question is appended

messages[0]role: system
messages[1]role: user · "What is 20% GST on 1500?"

One .append() call. The list guarantees this dict stays after the system message forever — order is exactly what lists are for.

3. The entire list goes to the model

messagesthe whole list, every turn
→
LLMreads top to bottom

The model keeps no memory between calls — whatever it should know must be in this list. That single fact drives most of agent engineering, and it starts right here.

4. The reply becomes a dict too

messages[0]role: system
messages[1]role: user
messages[2]role: assistant · "300 rupees."

The model's answer is appended with role "assistant". The transcript is now three dicts long, and the next user question will make it four.

Order is why the outside is a list; labelled parts is why each item is a dict. Every framework, every provider, every transcript you will ever inspect: this exact shape.

Look at the shape of one message access: messages[0]["content"]. A number in the first brackets — so the outer container is a list. A string in the second — so the inner one is a dict. You can now read the container structure of code you have never seen, purely from what goes inside the square brackets. That is fluency starting.

In the wild

Open any agent codebase and the four containers greet you by name. The call into every model API carries messages=[{"role": ..., "content": ...}] — the list of dicts you just built. Tool registries are dicts mapping a tool's name to its function, so the loop can look up what the model asked for. Config loaded from YAML or JSON arrives as nested dicts and lists (next chapter's whole story). In merge requests you will see reviewers ask "should this be .get() with a default, or do we want the KeyError?" — now you can follow, and even hold, that argument. And when a production log ends in KeyError: 'content', you will know precisely what happened: some code reached into a dict for a label that was not there.

Check yourself

1. You must make sure no tracking number is ever processed twice today, and the "already processed?" check has to stay fast at any volume. Which container holds the processed numbers?

The job is deduplication plus fast membership — the set's exact specialty. A list would allow duplicates and get slower to search as it grows; a tuple cannot even be added to.

2. parcel["insured"] runs, but the dict has no "insured" key. What happens?

Square-bracket lookup demands the key exist and crashes loudly when it does not. The None-and-carry-on behaviour is what parcel.get("insured") gives you — and .get("insured", False) lets you choose the fallback.

3. Why is a list the right outer container for an agent's message history?

Who said what, and in what sequence, is the conversation — so the container must preserve order and allow appending, which is the list's whole job. Each item is then a dict because a message has labelled parts: a role and its content.

4. A teammate stores the depot's coordinates as (18.5074, 73.8077) — a tuple, not a list. What are they telling you?

Immutability is a statement of intent that Python enforces: any attempt to assign into the tuple raises an error. Choosing a tuple over a list is how code says "this is a fact, not a workspace".

5. What does config["researcher"]["tools"][0] do, in reading order?

Chained lookups peel one layer at a time, left to right — dict key, dict key, list index. The bracket contents even tell you each layer's type: strings mean dicts, numbers mean lists.

Key takeaways

  • Four containers, four questions: a list answers "in what order?", a dict answers "what is filed under this label?", a tuple answers "what was fixed at writing time?", and a set answers "have I seen this before?"
  • Lists and dicts are editable in place (mutable) and do most of the work in agent code; tuples are sealed (immutable) and sets refuse duplicates while making membership checks near-instant.
  • d[key] means "this key must exist — crash loudly if not"; d.get(key, default) means "this key is optional — here is the fallback". The choice is a statement of intent, and a KeyError traceback means some code chose the first and lost the bet.
  • Containers nest freely — dicts of dicts of lists — and chained lookups like config["researcher"]["tools"][0] read left to right, one layer per bracket.
  • Agent message history is a list of dicts because a conversation needs both order (the list) and labelled parts per message — role and content (the dict). Every provider and framework uses this shape.
  • The bracket contents reveal the structure: a string inside [...] means you are looking at a dict, a number means a list. You can now read the skeleton of code you have never seen.
